The one idea (read this first)

SQL is declarative: you describe the shape of the answer, and the engine picks the algorithm. That is why it feels slippery — the order you write the clauses is not the order it runs them. Learn the real running order once (FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT) and the whole language snaps into place: you'll know why WHERE can't see a column alias, why COUNT(*) belongs in HAVING and not WHERE, and why a window function can never appear in a WHERE clause. Almost every Medium and Hard problem is then one of five moves: reshape with a join, collapse with GROUP BY, count conditionally with CASE inside an aggregate, rank or look sideways with a window function, or name the steps with a CTE.

01. Declarative: you describe, it decides

In Python you write how to get the answer. In SQL you write what the answer looks like, and the engine invents the how.

Write SELECT name FROM users WHERE age < 30 and you have not told the database to scan a table, or use an index, or read rows in any particular order. You have described a set: the names of users under thirty. The engine is then free to get there however it likes — a sequential scan, an index lookup, a parallel scan across eight cores — and it will pick based on statistics about your data. Two queries that describe the same set are interchangeable, so the optimiser can rewrite yours into something faster as long as the answer is identical.

This is the source of SQL's power and of every one of its surprises. The power: you get a query planner doing work you never wrote. The surprise: the order you write the clauses has almost nothing to do with the order they take effect. You write SELECT first and it runs nearly last. That single mismatch causes most of the "why doesn't this work?" moments a beginner hits, and it is the reason the next section is the most important one on this page.

The mindset shift that makes SQL click

Stop reading a query top-to-bottom like a program. Read it as a pipeline of table transformations: each clause takes a table in and hands a different table out. FROM builds a working table. WHERE hands back a smaller one. GROUP BY hands back a table with one row per group — at which point the original rows no longer exist, which is exactly why you can't select a column that isn't in the grouping key. Every rule you were told to memorise is a consequence of what shape the table has at that moment in the pipeline.

02. The logical order of operations

This is the master key. Almost every rule in SQL — and every confusing error message — falls out of this one diagram.

The database evaluates your clauses in a fixed logical order, no matter what order you typed them in. "Logical" means: the engine is allowed to physically do whatever it wants (that's the declarative bargain), but the result must be as if it had executed them in exactly this sequence. So for reasoning about correctness, this order is the truth.

Three consequences you will hit this week

You cannot use a SELECT alias in WHERE. The alias does not exist yet — WHERE ran at step 2, and the alias is created at step 5. Postgres will tell you column "x" does not exist, which sounds absurd until you remember the running order. MySQL is lenient in HAVING and ORDER BY, but never in WHERE.

You cannot put an aggregate in WHERE. WHERE COUNT(*) > 2 is meaningless: at step 2 there are no groups yet, so there is nothing to count. The clause that filters groups is HAVING, and it lives at step 4, safely after GROUP BY.

You cannot put a window function in WHERE or HAVING. Window functions are evaluated at step 5, as part of SELECT — after grouping and after group filtering. So if you need to filter on ROW_NUMBER(), you must compute it in one query and filter in an outer one. That is the single most common reason a CTE appears in a Hard problem, and we come back to it in §23.

sql
-- − every one of these is an error, and now you know exactly why
SELECT salary * 12 AS annual FROM emp WHERE annual > 100000;   -- alias not born yet (step 5 > step 2)
SELECT dept FROM emp WHERE COUNT(*) > 5 GROUP BY dept;         -- no groups exist at step 2
SELECT name FROM emp WHERE ROW_NUMBER() OVER (ORDER BY pay) = 1; -- windows run at step 5

-- + the same three intentions, expressed at the right stage
SELECT salary * 12 AS annual FROM emp WHERE salary * 12 > 100000; -- repeat the expression
SELECT dept FROM emp GROUP BY dept HAVING COUNT(*) > 5;          -- filter groups in HAVING
WITH r AS (SELECT name, ROW_NUMBER() OVER (ORDER BY pay) AS rn FROM emp)
SELECT name FROM r WHERE rn = 1;                                -- wrap it, then filter

The one-line version worth memorising

FROM → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT. When a query won't compile, don't stare at the syntax. Ask: "at the stage where I wrote this, does the thing I'm referring to exist yet?" Nine times out of ten, that is the whole bug.

03. SELECT: projection, aliases, DISTINCT

SELECT is the shape of your output: which columns, computed how, under what names.

In relational-algebra terms SELECT is projection (π) — choosing and computing columns — while WHERE is selection (σ) — choosing rows. The naming is a historical accident and it trips people up forever, so it is worth saying out loud once: the SQL keyword SELECT does the thing algebra calls projection.

The target list can hold far more than bare column names. Anything that produces one value per row belongs there: arithmetic, function calls, casts, CASE expressions, scalar subqueries, and window functions.

sql
SELECT
    e.name,                                       -- a plain column
    e.salary * 12                AS annual_pay,  -- arithmetic
    UPPER(e.dept)                AS dept,        -- a function call
    e.salary::numeric(10,2)                     -- a cast (Postgres); CAST(x AS ...) is portable
        AS salary_2dp,
    CASE WHEN e.salary > 100000
         THEN 'senior' ELSE 'standard' END AS band,   -- a CASE expression
    (SELECT AVG(salary) FROM emp) AS company_avg,     -- a scalar subquery
    RANK() OVER (ORDER BY e.salary DESC) AS pay_rank  -- a window function
FROM emp e;

Aliases: where they work and where they don't

An alias created in SELECT is visible only to clauses that run after step 5 — in practice, ORDER BY. It is not visible to WHERE, GROUP BY, or HAVING in strict ANSI SQL, because those all run earlier. MySQL and Postgres both bend this rule for GROUP BY and HAVING as a convenience, but relying on that will burn you the moment you switch dialects, and StrataScratch will happily hand you a Postgres console.

ClauseRuns atCan it see a SELECT alias?
WHEREstep 2No — never, in any dialect
GROUP BYstep 3No in ANSI; yes in MySQL & Postgres
HAVINGstep 4No in ANSI; yes in MySQL
ORDER BYstep 7Yes — everywhere

The portable escape hatch is always the same: compute it in a CTE, then refer to the alias in the outer query, where it is a real column and every clause can see it.

DISTINCT applies to the whole row

This is a small thing that produces big confusion. SELECT DISTINCT a, b does not mean "distinct values of a, plus b". It means distinct pairs (a, b). If you want one row per a, you want GROUP BY a — and then you must decide which b to keep, because SQL will not choose one for you.

sql
-- rows: (US, shipped) (US, pending) (EU, shipped)
SELECT DISTINCT region, status FROM orders;  -- 3 rows: every distinct PAIR
SELECT DISTINCT region FROM orders;          -- 2 rows: US, EU

-- Postgres only, and a genuinely useful trick: one row per region,
-- and YOU choose which one via ORDER BY. This is a top-1-per-group in one clause.
SELECT DISTINCT ON (region) region, status, amount
FROM orders
ORDER BY region, amount DESC;              -- the biggest order in each region

DISTINCT is often a bug in disguise

When someone sprinkles DISTINCT on a query to "get rid of the duplicates", the duplicates are usually not noise — they are the symptom of a join fan-out (§08). DISTINCT hides the duplicate rows but any SUM or COUNT you compute alongside is still counting the duplicated values, so the number stays wrong and now the evidence is gone. Before you reach for DISTINCT, work out why the rows multiplied.

04. WHERE: filtering rows

WHERE looks at one row at a time and asks a yes/no question. Rows that don't answer yes are dropped.

That framing — one row at a time, and only TRUE survives — is worth holding onto, because it explains both why aggregates are banned here (an aggregate needs many rows) and why NULL behaves the way it does (§05).

sql
-- comparison, combined with AND / OR / NOT
WHERE age >= 30 AND (country = 'US' OR country = 'CA')

-- IN: shorthand for a chain of ORs. The list can be a subquery.
WHERE country IN ('US', 'CA', 'MX')
WHERE user_id IN (SELECT user_id FROM premium_users)

-- BETWEEN is INCLUSIVE on both ends. This is a classic off-by-one.
WHERE age BETWEEN 30 AND 40      -- same as age >= 30 AND age <= 40

-- LIKE: % is "any run of characters", _ is "exactly one character"
WHERE name LIKE 'A%'            -- starts with A
WHERE name LIKE '%son'          -- ends with son
WHERE name LIKE '_a%'           -- 'a' is the second character
WHERE name ILIKE 'a%'           -- case-insensitive (Postgres). MySQL LIKE is already
                                 -- case-insensitive on default collations.

The BETWEEN-on-timestamps trap

WHERE ts BETWEEN '2024-01-01' AND '2024-01-31' silently loses almost a whole day. A bare date is read as midnight, so the upper bound is 2024-01-31 00:00:00 and every event on the 31st after midnight is excluded. Use a half-open range instead — WHERE ts >= '2024-01-01' AND ts < '2024-02-01' — which is correct, index-friendly, and never has an off-by-one at the boundary. Half-open ranges are the professional default for any time filter.

Keep your filters sargable

A predicate is sargable (from "Search ARGument-able") if the engine can use an index for it. The rule is simple: leave the column bare on one side of the comparison. The moment you wrap the column in a function, the index on it becomes unusable, because the index stores created_at, not YEAR(created_at), and the engine has no way to work backwards. It falls back to reading every row.

sql
-- − not sargable: the column is wrapped, so the index on created_at is dead
WHERE YEAR(created_at) = 2024
WHERE UPPER(email) = 'A@B.COM'
WHERE amount * 2 > 100

-- + sargable: the column stands alone, the index can be walked
WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01'
WHERE email = LOWER('A@B.COM')   -- move the function to the constant side
WHERE amount > 50

This rarely changes your score on a puzzle site, where tables are tiny. It changes everything on a fact table with 400 million rows, and it is the kind of thing an interviewer follows up on when they ask "and how would this perform?"

05. NULL and three-valued logic

NULL does not mean zero, and it does not mean empty string. It means "unknown" — and unknown is contagious.

Because NULL means we don't know, any comparison involving it must also be we don't know. Is an unknown age greater than 30? Unknown. Is an unknown age equal to another unknown age? Still unknown — they might be the same, they might not. So SQL does not have two truth values, it has three: TRUE, FALSE, and UNKNOWN. And the single rule that ties it all together is this: WHERE keeps a row only when the predicate is TRUE. Both FALSE and UNKNOWN are thrown out, which means a NULL row silently disappears from your result and never tells you it was there.

The NOT IN trap — the most expensive NULL bug there is

If the subquery inside a NOT IN returns even one NULL, the whole expression can never be TRUE, and your query returns zero rows. Here's the mechanics: x NOT IN (1, NULL) expands to x <> 1 AND x <> NULL. That second half is UNKNOWN for every possible x, and TRUE AND UNKNOWN is UNKNOWN — which WHERE discards. No error, no warning, just an empty result. Use NOT EXISTS, which is immune, or a LEFT JOIN ... WHERE right.key IS NULL.

sql
-- − returns NOTHING if any customer_id in orders is NULL
SELECT name FROM customers
WHERE id NOT IN (SELECT customer_id FROM orders);

-- + NOT EXISTS is NULL-safe: it asks "did the subquery produce any row?", not "is x equal to..."
SELECT c.name FROM customers c
WHERE NOT EXISTS (SELECT 1 FROM orders o WHERE o.customer_id = c.id);

-- + the anti-join: join, then keep only the rows that failed to match
SELECT c.name FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id
WHERE o.customer_id IS NULL;

The tools for taming NULL

sql
-- COALESCE: the first non-NULL argument. Your default-value workhorse.
COALESCE(phone, 'N/A')                          -- one fallback
COALESCE(work_phone, home_phone, mobile, 'none')  -- a chain, left to right

-- NULLIF(a, b): NULL when a = b, otherwise a. Its killer use is guarding division.
revenue / NULLIF(orders, 0)                      -- 0 orders → NULL, not a divide-by-zero crash

-- The two combined: turn 0 into NULL, then NULL into a label.
COALESCE(NULLIF(bonus, 0)::text, 'None')         -- 0 or NULL both display as 'None'

-- Watch the AVG: these two are NOT the same thing.
AVG(score)                -- mean of the rows that HAVE a score (NULLs excluded from the divisor)
AVG(COALESCE(score, 0))   -- mean where a missing score counts as a zero

Which AVG you want is a business question, not a syntax question, and interviewers ask it on purpose. "Average score of students who took the test" and "average score across all enrolled students" are different numbers, and only one of them is AVG(score).

06. The function toolbox: string, numeric, date

These are scalar functions: one row in, one value out. They live in SELECT, WHERE and ORDER BY, and they are the least glamorous, most-used part of the language.

You do not need to memorise them. You need to know what exists, so that when a problem says "extract the domain from the email" or "group revenue by month" you know a one-liner is available and don't reach for something baroque. What you do need to memorise is where the dialects disagree — because that is where interview answers break.

Know which dialect you are in — it changes the function names

Scalar functions are the least standardised part of SQL. LeetCode defaults to MySQL; StrataScratch defaults to PostgreSQL. They disagree on almost every date function, on string concatenation, and on rounding. Worse, a lot of SQL material online is T-SQL (SQL Server) — DATEPART, DATEADD, DATEDIFF(part, a, b), FORMAT, CONVERT, LEN, ISDATE, EOMONTH, GETDATE, TOP — and none of those exist in Postgres, and most don't exist in MySQL either. If you learned dates from a SQL Server tutorial, half of what you know will throw a syntax error on both of the sites you're practising on.

Dates: the three things you ever actually do

Date questions look varied and are not. Almost every one is asking you to do exactly one of three things to a timestamp: truncate it (to bucket rows together), extract a part of it (to label or filter), or subtract two of them (to measure elapsed time).

There are dozens of date functions. Seven of them cover essentially every date problem on LeetCode and StrataScratch. Learn this table and skip the rest until something forces you to care.

The top three are the ones that carry real weight — monthly bucketing, day differences, and interval arithmetic are the engine of nearly every date question that gets asked. The bottom four are conveniences.

Two traps hide in that table. MySQL's DATEDIFF(end, start) takes the end date first — backwards from T-SQL's DATEDIFF(part, start, end), and a guaranteed sign error if you switch between them. And Postgres date subtraction returns an integer number of days, but timestamp subtraction returns an interval, which is a different type — cast with ::date on both sides if you want a plain day count.

Safe to ignore — delete these from your notes

Every one of these is SQL Server only and will throw a syntax error on both LeetCode and StrataScratch: DATEPART, DATENAME, DATEADD, DATEDIFF(part, a, b), DATETRUNC (T-SQL spelling — Postgres wants DATE_TRUNC), EOMONTH, ISDATE, FORMAT, CONVERT and its style-code table (101, 112, 120…), GETDATE, LEN, and TOP. If a tutorial taught you dates using these, it taught you a dialect neither practice site runs. The CONVERT style codes in particular are pure memorisation with zero transfer — drop them entirely.

The four date patterns that actually come up

Nearly every date question on either site is one of these four. Recognise the shape and you already have the answer.

sql
══ 1. GROUP BY MONTH ═══ by far the most common. "Revenue per month", "signups per month".
   DATE_TRUNC collapses every March timestamp to '2024-03-01', so they bucket together.
SELECT DATE_TRUNC('month', order_date) AS mth, SUM(amount) AS revenue
FROM orders GROUP BY 1 ORDER BY 1;                       -- Postgres
   MySQL: SELECT DATE_FORMAT(order_date,'%Y-%m') AS mth, SUM(amount) ... GROUP BY 1

   TRAP: EXTRACT(MONTH FROM d) returns 3 — which merges March 2023 with March 2024.
   Only safe if you also group by the year, or you genuinely want seasonality.

══ 2. CONSECUTIVE DAYS ═══ "today vs yesterday". LeetCode's Rising Temperature is exactly this.
   Compare each row to the previous DAY — not the previous ROW, because a day may be missing.
SELECT a.id                                                   -- Postgres
FROM weather a
JOIN weather b ON a.record_date = b.record_date + INTERVAL '1 day'
WHERE a.temperature > b.temperature;
   MySQL: ON DATEDIFF(a.record_date, b.record_date) = 1

   Why not LAG? LAG gives the previous ROW, which is only the previous DAY if there
   are no gaps. If a day is missing from the table, LAG silently compares across the gap.

══ 3. A ROLLING WINDOW OF DAYS ═══ "active in the last 30 days", "within 7 days of signup".
WHERE activity_date > CURRENT_DATE - INTERVAL '30 days'   -- Postgres
   MySQL: WHERE activity_date > CURDATE() - INTERVAL 30 DAY

══ 4. RETENTION / "DID THEY COME BACK THE NEXT DAY?" ═══ LeetCode Game Play Analysis IV.
   Self-join the first event to an event exactly one day later, then count the hits.
WITH first_login AS (
    SELECT player_id, MIN(event_date) AS d0 FROM activity GROUP BY player_id
)
SELECT ROUND(
    COUNT(a.player_id)::numeric / (SELECT COUNT(*) FROM first_login), 2
) AS fraction
FROM first_login f
LEFT JOIN activity a
       ON a.player_id  = f.player_id
      AND a.event_date = f.d0 + INTERVAL '1 day';   -- LEFT JOIN + COUNT(a.…) so
                                                  -- non-returners count as 0, not 1 (§09)

══ AND ALWAYS: filter a period half-open, never with BETWEEN (§04).
WHERE order_date >= '2024-01-01' AND order_date < '2024-02-01'

The one date insight worth more than the function list

"The previous row" and "the previous day" are not the same thing. LAG gives you the row before — which is only yesterday if no day is missing from your data. If a sensor skipped a day, or a user didn't log in, LAG happily compares Monday to Thursday and tells you nothing broke. When a question says consecutive days, join on the date arithmetic (d = prev_d + INTERVAL '1 day') rather than trusting row order — or generate a complete date spine first (§16). This distinction is the whole difficulty of half the Medium date problems.

Strings: eight functions, three patterns

Strings are a smaller topic than dates. Eight functions cover it, and they show up in three recognisable problem shapes. Concatenation is the one that bites on dialect: Postgres uses ||, MySQL does not support || by default. CONCAT() works in both — always use CONCAT() and stop thinking about it.

The three string patterns that actually come up

Safe to ignore

T-SQL string functions that work on neither practice site: LEN, CHARINDEX, PATINDEX, STUFF, FORMAT. Also genuinely rare in problems, so don't drill them: LPAD/RPAD, REVERSE, ASCII/CHR, and regex (REGEXP / ~) — regex appears occasionally on email-validation questions, but LIKE solves the overwhelming majority.

Numbers: it really is just five

Numeric functions are the smallest topic on this page. The functions are trivial; the two traps are what actually cost you submissions, and both are below.

POWER, SQRT, EXP, LOG, SIGN, RAND and the trigonometric functions exist, but they essentially never appear in SQL interview problems — if a question needs them, it will be obvious and you can look them up. ROUND(x, 2) can fail outright in PostgreSQL

Postgres only defines two-argument ROUND for the numeric type. Hand it a double precision and you get ERROR: function round(double precision, integer) does not exist — which is a baffling message the first time you see it on StrataScratch. It bites whenever you round something that came out of a float column or a division. The fix is always the same: cast first — ROUND(x::numeric, 2). MySQL has no such restriction, so this error simply never appears on LeetCode.

The other numeric trap is integer division: in Postgres, SUM(flag) / COUNT(*) with two integers truncates to 0, silently zeroing out your percentage. Lead with 100.0 * to force floating point. It matters most when you compute rates, so it gets a full treatment in §12.

07. JOINs: the shapes

A join asks one question — which rows on the left pair up with which rows on the right? — and then the join type decides what to do with the rows that didn't pair up at all.

That is genuinely the whole idea. Every join first works out the matches using the ON condition. Where they differ is their policy on leftovers: an INNER JOIN discards them, a LEFT JOIN keeps the unmatched left-hand rows and pads the right-hand columns with NULL, and so on. Get the matching picture in your head and the six join types become one idea with six settings.

RIGHT JOIN is real but almost nobody writes it, because a RIGHT JOIN b is just b LEFT JOIN a with the tables swapped, and humans read left-to-right. Pick LEFT JOIN, put the table you want to preserve first, and keep the whole query pointing the same direction.

The four joins you actually write

sql
-- INNER: only customers who have orders
SELECT c.name, o.amount
FROM customers c
JOIN orders o ON o.cust_id = c.id;        -- "JOIN" alone always means INNER

-- LEFT: every customer, with NULLs where they have no orders.
-- This is the join for "show me all X, and their Y if any".
SELECT c.name, o.amount
FROM customers c
LEFT JOIN orders o ON o.cust_id = c.id;

-- ANTI-JOIN: customers with NO orders. LEFT JOIN, then keep only the misses.
SELECT c.name
FROM customers c
LEFT JOIN orders o ON o.cust_id = c.id
WHERE o.cust_id IS NULL;                   -- the NULL means "no match was found"

-- SELF-JOIN: the same table twice, under two aliases.
-- Classic LeetCode: employees who earn more than their manager.
SELECT e.name
FROM employees e
JOIN employees m ON e.manager_id = m.id
WHERE e.salary > m.salary;

ON versus WHERE — they are NOT the same for outer joins

This one is subtle, it is asked constantly, and it silently changes your answer. For an INNER JOIN, a condition in ON and the same condition in WHERE give identical results. For a LEFT JOIN they do completely different things, and the reason is — as always — the order of operations. ON runs during the join (step 1); WHERE runs after it (step 2).

sql
-- + A: filter DURING the join. Every customer survives. Non-2024 orders
--   simply never match, so those customers get NULL — like having no order at all.
SELECT c.name, o.amount
FROM customers c
LEFT JOIN orders o ON o.cust_id = c.id AND o.year = 2024;

-- − B: filter AFTER the join. The LEFT JOIN dutifully produces a NULL row for
--   Ann — and then WHERE tests NULL = 2024, gets UNKNOWN, and deletes her.
--   You have silently turned your LEFT JOIN back into an INNER JOIN.
SELECT c.name, o.amount
FROM customers c
LEFT JOIN orders o ON o.cust_id = c.id
WHERE o.year = 2024;

The rule that settles it

In a LEFT JOIN, conditions on the right-hand (optional) table belong in ON. Put one in WHERE and you cancel the outerness of the join — because WHERE runs after the NULL-padding and throws the padded rows straight back out. The one deliberate exception is the anti-join, WHERE o.id IS NULL, where killing the matched rows is precisely the point.

08. Join fan-out: why your SUM is wrong

This is the single most common way to get a Medium problem almost right — a query that runs, returns plausible rows, and reports a number that is quietly two or three times too big.

A join is not a lookup. If a left row matches three right rows, that left row is duplicated three times in the output, and every one of its columns is duplicated with it. So the instant you join one-to-many and then aggregate a column from the one side, you are summing the same value once per match.

How to spot it and how to fix it

The tell is a row count that grew when you didn't expect it to. If customers has 500 rows and SELECT COUNT(*) FROM customers c JOIN orders o ON ... gives you 2,000, then every customer column in that result now appears an average of four times. Before you aggregate anything after a join, ask: "is this join one-to-one, or one-to-many?"

There are three good fixes, and one bad one.

sql
-- FIX 1 (usually best): aggregate FIRST, then join. The subquery collapses orders
-- to one row per customer, so the join is one-to-one and nothing can fan out.
SELECT c.name, c.budget, COALESCE(o.total, 0) AS spent
FROM customers c
LEFT JOIN (
    SELECT cust_id, SUM(amount) AS total
    FROM orders GROUP BY cust_id
) o ON o.cust_id = c.id;

-- FIX 2: COUNT(DISTINCT ...) when you only need a count of the ONE side.
SELECT COUNT(DISTINCT c.id) AS customers, SUM(o.amount) AS revenue
FROM customers c JOIN orders o ON o.cust_id = c.id;

-- FIX 3: a scalar or correlated subquery in SELECT — no join, so no fan-out at all.
SELECT c.name, c.budget,
       (SELECT SUM(amount) FROM orders o WHERE o.cust_id = c.id) AS spent
FROM customers c;

-- − THE BAD FIX: SELECT DISTINCT. It de-duplicates the ROWS you can see,
-- but SUM(c.budget) already added 100 three times. The total is still 300.

Fan-out compounds

Join a customer to 3 orders and 4 support tickets in the same query and you don't get 3 + 4 = 7 rows. You get 3 × 4 = 12 — every order paired with every ticket, because the two joins have nothing to do with each other. Revenue is now 4× too high and ticket counts are 3× too high. Whenever you need to aggregate along two independent one-to-many paths, aggregate each one separately in its own CTE and join the summaries together. There is no clever way around this; there is only doing it properly.

09. GROUP BY and aggregation

GROUP BY is the one clause that changes what a "row" means. Before it, a row is a fact. After it, a row is a bucket.

You hand GROUP BY a list of expressions — the grouping key — and it sorts every incoming row into a bucket, one bucket per distinct key value. Then each bucket collapses down to exactly one output row. The original rows are gone. This is the entire reason for the rule everybody memorises without understanding.

The rule, and why it exists

The GROUP BY rule

Every column in your SELECT must be either (a) in the GROUP BY list, or (b) wrapped in an aggregate function. There is no third option — and the reason is not pedantry. If you group by dept and then ask for name, the bucket for "Engineering" contains forty different names. Which one did you want? The question has no answer, so the database refuses to guess.

sql
-- − error: which of the many names in each dept bucket should it show?
SELECT dept, name, AVG(salary) FROM emp GROUP BY dept;

-- + pick one deliberately with an aggregate...
SELECT dept, MAX(name) AS a_name, AVG(salary) FROM emp GROUP BY dept;

-- + ...or group more finely, so each bucket really does hold one name
SELECT dept, name, AVG(salary) FROM emp GROUP BY dept, name;

MySQL historically let you break this rule and returned an arbitrary name from the bucket — a notorious source of wrong reports. Modern MySQL enables ONLY_FULL_GROUP_BY by default and now errors like everyone else. Postgres has always enforced it, with one sensible exception: if you group by a table's primary key, you may select any column of that table, because the key already determines them all.

That COUNT(*) versus COUNT(col) difference is a favourite interview probe, and it also gives you a free trick: COUNT(some_nullable_col) counts exactly the rows where that column is not NULL — which is the basis of conditional counting in §12. And the SUM row is a real trap: SUM over a group with no matching rows gives you NULL, not zero, so a LEFT JOIN plus SUM almost always wants a COALESCE(SUM(x), 0) around it.

sql
-- The LEFT JOIN counting trap, in one query.
-- Ann has no orders. Her LEFT JOIN gives one row with every o.* column NULL.
SELECT c.name,
       COUNT(*)            AS wrong,   -- 1 for Ann — it counted her NULL-padded row!
       COUNT(o.id)         AS correct, -- 0 for Ann — o.id is NULL, so it isn't counted +
       COALESCE(SUM(o.amount), 0) AS spent  -- NULL → 0 for Ann +
FROM customers c
LEFT JOIN orders o ON o.cust_id = c.id
GROUP BY c.name;

Read that one twice. After a LEFT JOIN, COUNT(*) is almost always the wrong answer — it counts the phantom NULL row that the outer join manufactured. Count a column from the right-hand table instead, and the phantom vanishes on its own.

10. HAVING: filtering groups

WHERE throws away rows before they are grouped. HAVING throws away whole groups after they are formed.

Once you have the order of operations, this needs no memorising. HAVING runs at step 4, after GROUP BY, so aggregates exist and it can test them. WHERE runs at step 2, before any group exists, so it cannot. Everything else follows.

sql
-- Both clauses in one query, each doing its own job:
SELECT dept, COUNT(*) AS headcount, AVG(salary) AS avg_pay
FROM employees
WHERE  hire_date >= '2020-01-01'   -- ROW filter: ignore old hires entirely
GROUP BY dept
HAVING COUNT(*) >= 5               -- GROUP filter: drop small departments
   AND AVG(salary) > 80000;

Note that these two filters are not interchangeable and they are not merely a style choice — they produce different numbers. WHERE hire_date >= '2020-01-01' means old hires never enter a bucket at all, so they don't count toward COUNT(*) or drag on AVG. If you moved that condition into HAVING it would be an error (there's no single hire_date per department), and if you moved a COUNT(*) condition into WHERE that would be an error too.

Put filters in WHERE whenever you legally can

If a condition can live in WHERE, put it there rather than in HAVING. WHERE runs first and shrinks the data before the expensive grouping work happens; HAVING makes the engine build every group and then throw most of them away. Same answer, far less work. Reserve HAVING for what only it can do: conditions on aggregates.

"Rows where X, among groups where Y" — the sentence to listen for

Interview problems love to hide both filters in one sentence. "Find the departments hired since 2020 that have more than five people" contains a row filter (hire_date >= '2020-01-01' → WHERE) and a group filter (COUNT(*) > 5 → HAVING). When you read a problem, sort each condition into one of those two boxes before you write a line of SQL. It is the fastest way to get the skeleton right.

11. CASE WHEN

CASE is SQL's if/else. It is an expression, not a statement — it produces a value, so it can go anywhere a value can go.

That last point is the one that unlocks it. Because CASE is just an expression, it is legal in SELECT, inside ORDER BY, inside a GROUP BY, and — most powerfully — inside an aggregate function, which is the whole of §12.

sql
-- SEARCHED form: each WHEN is its own full boolean test. Use this by default.
CASE
    WHEN score >= 90 THEN 'A'
    WHEN score >= 80 THEN 'B'
    WHEN score >= 70 THEN 'C'
    ELSE 'F'
END

-- SIMPLE form: compare one expression against constants. Compact but limited
-- to equality — and note it can never match a NULL, since NULL = NULL is UNKNOWN.
CASE status
    WHEN 1 THEN 'pending'
    WHEN 2 THEN 'shipped'
    ELSE 'unknown'
END

Three things that catch people out

Order matters — the first match wins. CASE evaluates its branches top to bottom and stops at the first TRUE. Write the grade ladder above with score >= 70 first and everyone with 95 gets a C, because 95 >= 70 is true and nothing after it is ever consulted. Always go from the narrowest condition to the broadest.

No ELSE means ELSE NULL. If no branch matches and you omitted ELSE, the result is NULL. That is sometimes exactly what you want (§12 uses it deliberately), but if you meant zero, say zero.

Every branch must return the same type. CASE WHEN x THEN 1 ELSE 'none' END is a type error in Postgres. Cast if you must mix.

sql
-- CASE in ORDER BY: a custom sort order that no column gives you for free.
SELECT name, priority
FROM tickets
ORDER BY CASE priority
             WHEN 'critical' THEN 1
             WHEN 'high'     THEN 2
             WHEN 'medium'   THEN 3
             ELSE 4
         END, created_at DESC;

-- CASE in GROUP BY: invent a dimension that isn't in the table. Bucket ages
-- into bands and count them — the CASE must be repeated in both clauses.
SELECT CASE WHEN age < 25 THEN '18-24'
            WHEN age < 35 THEN '25-34'
            ELSE '35+' END AS age_band,
       COUNT(*) AS users
FROM users
GROUP BY CASE WHEN age < 25 THEN '18-24'
              WHEN age < 35 THEN '25-34'
              ELSE '35+' END;

Repeating that whole CASE block twice is ugly, and it is exactly the moment to reach for a CTE (§15): compute age_band once in a WITH, then GROUP BY age_band in the outer query like a normal column. Same result, half the code, and no risk of the two copies drifting apart.

12. CASE WHEN inside aggregates

This is the technique. If you learn one thing on this page beyond window functions, learn conditional aggregation — it turns rows into columns, and it is the answer to a startling number of Medium problems.

The idea is simple once you see it. An aggregate function consumes one value per row. A CASE produces one value per row. So put the CASE inside the aggregate, and you get to decide, row by row, whether that row contributes to this particular total. Run several of them side by side and each one becomes its own column — a pivot, in plain SQL, with no special syntax.

The two forms, and why COUNT needs no ELSE

sql
-- The pivot in full. Every one of these runs over the SAME rows —
-- each CASE just decides which rows it cares about.
SELECT
    region,
    COUNT(*)                                                    AS total,
    SUM(CASE WHEN status = 'shipped'   THEN 1 ELSE 0 END)      AS shipped,
    SUM(CASE WHEN status = 'cancelled' THEN 1 ELSE 0 END)      AS cancelled,
    SUM(CASE WHEN status = 'shipped'   THEN amount ELSE 0 END) AS shipped_revenue,

    -- COUNT with NO ELSE. Non-matching rows become NULL, and COUNT skips NULLs.
    -- Exactly equivalent to the SUM(...1 ELSE 0) above. Pick one and be consistent.
    COUNT(CASE WHEN status = 'shipped' THEN 1 END)              AS shipped_again,

    -- AVG with no ELSE: the average amount OF SHIPPED ORDERS ONLY.
    -- If you wrote ELSE 0 here you would drag every cancelled order into the
    -- divisor as a zero and get a completely different (wrong) number.
    AVG(CASE WHEN status = 'shipped' THEN amount END)          AS avg_shipped
FROM orders
GROUP BY region;

ELSE 0 or no ELSE? The rule that decides

For SUM and COUNT, ELSE 0 and no-ELSE give the same answer — adding zero changes nothing, and COUNT ignores the NULLs. For AVG, MIN and MAX they are wildly different, because a NULL is skipped entirely while a 0 is a real value that joins the calculation. Rule: if you want the aggregate computed over only the matching rows, omit ELSE and let the rest go NULL.

The percentage-of-total pattern

The most common real use is a rate: what fraction of each group did something? Two gotchas live here at once — integer division, and division by zero — and interviewers know it.

sql
SELECT
    region,
    ROUND(
        100.0                                              -- 100.0, not 100 — see the note below
        * SUM(CASE WHEN status = 'shipped' THEN 1 ELSE 0 END)
        / NULLIF(COUNT(*), 0)                                -- guard: empty group → NULL, not a crash
    , 2) AS pct_shipped
FROM orders
GROUP BY region;

-- Shorter, and it reads beautifully: the average of a 1/0 flag IS the rate.
SELECT region,
       ROUND(AVG(CASE WHEN status = 'shipped' THEN 1.0 ELSE 0 END) * 100, 2) AS pct_shipped
FROM orders
GROUP BY region;

Integer division — and it is dialect-specific, so know which box you're in

In PostgreSQL (and SQL Server, and SQLite), dividing an integer by an integer truncates: 1 / 2 is 0, not 0.5. In MySQL the / operator always returns a decimal, so 1 / 2 really is 0.5 and this trap doesn't bite (MySQL's truncating operator is DIV). Since StrataScratch usually hands you Postgres and LeetCode usually hands you MySQL, it is worth knowing which side of that line you're standing on.

Where it hurts most is the order of operations. SUM(...) / COUNT(*) * 100 divides first — so on Postgres 2/3 collapses to 0 and your whole percentage is 0. Writing 100 * SUM(...) / COUNT(*) is better but still truncates (you get 66, not 66.67). Leading with 100.0 forces the whole expression into floating point and is correct in every dialect — which is why every percentage on this page starts that way.

The same trick answers "which groups contain only X?" or "which contain no X?" without a subquery in sight — you just compare a conditional count against the total.

sql
-- Customers whose every order shipped (and who have at least one order)
SELECT cust_id
FROM orders
GROUP BY cust_id
HAVING COUNT(*) = SUM(CASE WHEN status = 'shipped' THEN 1 ELSE 0 END);

-- Customers who never cancelled anything
SELECT cust_id
FROM orders
GROUP BY cust_id
HAVING SUM(CASE WHEN status = 'cancelled' THEN 1 ELSE 0 END) = 0;

-- Customers who bought BOTH product A and product B (the "relational division" problem).
-- The 1990s answer was a double NOT EXISTS. This is the same thing, readable.
SELECT cust_id
FROM orders
GROUP BY cust_id
HAVING COUNT(DISTINCT CASE WHEN product IN ('A','B') THEN product END) = 2;

That last one is worth sitting with. "Find customers who bought every one of these N products" is a genuinely Hard-tier problem in most textbooks, and conditional aggregation reduces it to count the distinct wanted products in each group and check you got all N. This pattern — HAVING COUNT(DISTINCT CASE WHEN ... END) = N — comes up on StrataScratch more than almost anything else.

13. Subqueries: scalar, row, table

A subquery is a query inside a query. Which shape it returns decides where you're allowed to put it.

People find subqueries slippery because they are taught as one topic when they are really three, distinguished by the shape of the thing they hand back. Sort them by shape and the rules become obvious — a single value can go anywhere a value can go, a whole table can go anywhere a table can go.

Scalar subqueries

A scalar subquery is a single value with extra steps, so it slots in anywhere a literal would. Use it whenever you need to compare each row against a number computed over the whole table.

sql
-- In WHERE: everyone paid above the company-wide average.
SELECT name, salary
FROM employees
WHERE salary > (SELECT AVG(salary) FROM employees);

-- In SELECT: show the benchmark next to each row so the comparison is visible.
SELECT name, salary,
       salary - (SELECT AVG(salary) FROM employees) AS vs_avg
FROM employees;

-- − if the subquery returns MORE than one row, this is a runtime error:
--   "more than one row returned by a subquery used as an expression"
WHERE salary = (SELECT salary FROM employees WHERE dept = 'Eng');
-- + say what you mean: ANY of them, ALL of them, or the max of them.
WHERE salary = ANY (SELECT salary FROM employees WHERE dept = 'Eng');  -- same as IN
WHERE salary > ALL (SELECT salary FROM employees WHERE dept = 'Eng');  -- beats every one

A scalar subquery that finds no rows returns NULL rather than erroring — which then makes your comparison UNKNOWN and quietly drops the row. If that matters, wrap it in COALESCE.

Row subqueries: comparing a whole tuple at once

The forgotten one — and it is genuinely elegant. A row subquery returns several columns in one row, and you compare against it as a tuple: (a, b) IN (SELECT x, y ...). The point is that the columns stay paired. It solves "the maximum per group" with no window function and no join at all.

sql
-- + The highest-paid employee(s) in each department. Read it as:
--   "keep the rows whose (dept, salary) PAIR is one of the (dept, max-salary) pairs."
SELECT name, dept, salary
FROM employees
WHERE (dept, salary) IN (
    SELECT dept, MAX(salary) FROM employees GROUP BY dept
);
-- Ties are ALL kept — if two people share the top salary, both come back.
-- (ROW_NUMBER() = 1 would arbitrarily pick one. Read the question carefully.)

Why the tuple matters — splitting it apart gives a wrong answer

It is tempting to write this as two separate conditions — dept IN (...) AND salary IN (SELECT MAX(salary) ... GROUP BY dept) — and it is wrong. That asks "is this a valid dept?" and "is this salary somebody's maximum?" as two unrelated questions, so the pairing is lost. Take Ann (Eng, 120), Bob (Eng, 100), Cid (Sales, 100): Eng's max is 120 and Sales' max is 100. The split version wrongly returns Bob — he isn't Eng's top earner, but his salary of 100 happens to be Sales' maximum, and the query has no idea those two facts belong to different departments. The tuple form (dept, salary) IN (...) keeps them welded together and returns only Ann and Cid.

Row constructors work in PostgreSQL and MySQL, so both practice sites are fine. SQL Server does not support them — there you'd fall back to a join or a window function. And the same idea works with = when the subquery returns exactly one row: WHERE (a, b) = (SELECT x, y FROM t LIMIT 1).

Table subqueries: the derived table

Put a query in the FROM clause and it behaves as a table. It must be given an alias — the engine needs a name to refer to its columns by, and forgetting this is the single most common syntax error in the whole topic.

sql
-- Pre-aggregate, THEN filter. Remember: you cannot filter on an aggregate in
-- WHERE, so a common move is to compute it inside and filter outside.
SELECT d.dept, d.avg_sal
FROM (
    SELECT dept, AVG(salary) AS avg_sal
    FROM employees
    GROUP BY dept
) d                                   -- ← the alias is mandatory
WHERE d.avg_sal > 50000;

-- The fan-out cure from §08: collapse the many-side before joining.
SELECT c.name, COALESCE(o.total, 0) AS spent
FROM customers c
LEFT JOIN (
    SELECT cust_id, SUM(amount) AS total FROM orders GROUP BY cust_id
) o ON o.cust_id = c.id;

Aggregates cannot be nested — but this is how you fake it

MIN(AVG(salary)) is illegal: an aggregate takes rows, and AVG(salary) is not a row. To get "the department with the lowest average salary" you need two passes — compute the averages in a derived table or CTE, then aggregate those. Any time you catch yourself wanting to nest one aggregate inside another, that's the signal to add a layer.

sql
-- The department with the lowest average salary — two passes, one CTE.
WITH dept_avg AS (
    SELECT dept, AVG(salary) AS avg_sal FROM employees GROUP BY dept
)
SELECT dept, avg_sal
FROM dept_avg
WHERE avg_sal = (SELECT MIN(avg_sal) FROM dept_avg);   -- ties all survive, which is
                                                     -- usually what you want

Note the deliberate choice there: WHERE avg_sal = (SELECT MIN(avg_sal) ...) keeps every department tied for lowest, whereas ORDER BY avg_sal LIMIT 1 would pick one arbitrarily and hide the tie. Interview problems very often say "if there is a tie, return all of them" — and that sentence is telling you which of these two to write.

14. Correlated subqueries and EXISTS

A correlated subquery mentions a column from the outer query. That one reference changes everything: it can no longer be computed once — it has to be re-computed for every outer row.

sql
-- CORRELATED: the inner query mentions e1 (the outer row). It cannot run alone.
-- "Employees who earn more than their OWN department's average."
SELECT e1.name, e1.salary, e1.dept
FROM employees e1
WHERE e1.salary > (
    SELECT AVG(e2.salary)
    FROM employees e2
    WHERE e2.dept = e1.dept        -- ← the correlation. Remove it and the
);                                  --   meaning changes to "company average".

-- The same answer with a window function: ONE pass, no repeated work.
-- This is the rewrite an interviewer is hoping you know.
SELECT name, salary, dept
FROM (
    SELECT name, salary, dept,
           AVG(salary) OVER (PARTITION BY dept) AS dept_avg
    FROM employees
) t
WHERE salary > dept_avg;

EXISTS: the existence check

EXISTS asks one question — did the subquery produce at least one row? — and answers TRUE or FALSE. It never looks at the values, only at whether any row came back, which is why SELECT 1 is the idiomatic body: the engine can stop at the first hit and doesn't care what you selected.

sql
-- SEMI-JOIN: customers who have ordered at least once.
-- Note the result has ONE row per customer no matter how many orders they placed —
-- EXISTS cannot fan out, which is exactly why it's safer than a JOIN here.
SELECT c.name
FROM customers c
WHERE EXISTS (SELECT 1 FROM orders o WHERE o.cust_id = c.id);

-- ANTI-JOIN: customers who have never ordered. NULL-safe, unlike NOT IN.
SELECT c.name
FROM customers c
WHERE NOT EXISTS (SELECT 1 FROM orders o WHERE o.cust_id = c.id);

15. CTEs: naming your steps

A CTE (WITH clause) is a named subquery you define before you use it. It turns a nested, inside-out query into a readable top-to-bottom pipeline.

Functionally, a CTE and a derived table in FROM do the same job. The difference is entirely for humans, and it is enormous. A nested query has to be read from the innermost parenthesis outwards; a chain of CTEs reads like a paragraph, each step named for what it produces. On a Hard problem with four stages, that is the difference between a query you can debug and one you can't.

sql
-- The pipeline above, in full. This is the single most useful query template
-- on this page — "the top thing in each group" wears a hundred disguises.
WITH monthly AS (
    SELECT DATE_TRUNC('month', order_date) AS mth,
           user_id,
           SUM(amount) AS revenue
    FROM orders
    GROUP BY 1, 2                       -- ordinal refs: 1 = the DATE_TRUNC, 2 = user_id
),
ranked AS (
    SELECT mth, user_id, revenue,
           ROW_NUMBER() OVER (PARTITION BY mth ORDER BY revenue DESC) AS rn
    FROM monthly                        -- CTE 2 reads CTE 1 by name
)
SELECT mth, user_id, revenue
FROM ranked
WHERE rn = 1                            -- the filter that a window fn can't do itself
ORDER BY mth;

Why this shape keeps appearing

Look at what forced the CTE here. A window function is computed at step 5 (SELECT), so you cannot say WHERE ROW_NUMBER() OVER (...) = 1 — the number doesn't exist yet when WHERE runs. The only way to filter on it is to finish the query, then filter the result, and a CTE is how you say that. Compute the window in one layer, filter it in the next. Once you internalise that, "top N per group" stops being a puzzle and becomes a reflex.

How to debug a CTE chain

Because each CTE is a real named table, you can inspect any stage by swapping the final SELECT: SELECT * FROM monthly LIMIT 20; shows you exactly what stage one produced. Work forwards until the output stops matching what you expected — the bug is in that step. You cannot do that with a deeply nested subquery, and it is the main reason to prefer CTEs even when both would work.

Performance, briefly and honestly

A CTE is usually inlined — the engine expands it into the main query and optimises the whole thing together, so it costs nothing over a derived table. The historical exception is Postgres before version 12, where WITH was an optimisation fence: it always materialised the CTE and never pushed your outer WHERE down into it, which could be catastrophically slow. Postgres 12+ inlines by default, and you can still force either behaviour explicitly with WITH x AS MATERIALIZED (...) or AS NOT MATERIALIZED (...). Worth knowing because it is a genuine interview question, and because the folklore "CTEs are slow" comes from exactly this and is now out of date.

One real caveat remains: a CTE referenced twice may be computed twice. If a CTE is expensive and you use it in two places, check the plan.

16. Recursive CTEs

A recursive CTE lets a query refer to itself. It is how you walk a hierarchy of unknown depth, or generate a series of rows out of thin air.

The shape never varies. An anchor query produces the starting rows. A recursive query joins back to the CTE's own name, producing the next layer. The two are glued with UNION ALL, and the engine keeps re-running the recursive half on whatever the previous round produced, until a round produces nothing — at which point it stops and hands you everything it accumulated.

sql
WITH RECURSIVE t AS (
    -- 1. ANCHOR: the starting point. Runs exactly once.
    SELECT 1 AS n

    UNION ALL

    -- 2. RECURSIVE STEP: reads t — its own name. Runs over and over,
    --    each time seeing ONLY the rows the previous round added.
    SELECT n + 1 FROM t
    WHERE  n < 10          -- 3. TERMINATION. Without this it never stops.
)
SELECT n FROM t;          -- 1, 2, 3, ... 10

Always write the stopping condition first

Drop the WHERE n < 10 and the query runs until it exhausts memory or the server kills it. Write the termination clause before you write the recursive step, every time. And if you're walking a graph that might contain a cycle — an org chart with a management loop, say — you also need cycle protection: carry the path so far in an array and refuse to revisit a node (WHERE NOT id = ANY(path) in Postgres), or the recursion goes round forever.

The two things recursion is actually for

sql
-- USE 1: walk a hierarchy of unknown depth. Everyone under the CEO,
-- with how many levels down they sit. A plain JOIN can't do this —
-- you'd need to know the depth in advance to write the joins.
WITH RECURSIVE org AS (
    SELECT id, name, manager_id, 1 AS depth
    FROM employees
    WHERE manager_id IS NULL            -- anchor: the top of the tree

    UNION ALL

    SELECT e.id, e.name, e.manager_id, o.depth + 1
    FROM employees e
    JOIN org o ON e.manager_id = o.id   -- attach the next level down
)
SELECT depth, name FROM org ORDER BY depth, name;

-- USE 2: generate a calendar. This is the fix for "gaps" in a time series —
-- days with zero sales don't exist in the orders table, so you must
-- manufacture the days and LEFT JOIN the data onto them.
WITH RECURSIVE days AS (
    SELECT '2024-01-01'::date AS d
    UNION ALL
    SELECT d + 1 FROM days WHERE d < '2024-01-31'::date
)
SELECT days.d, COALESCE(SUM(o.amount), 0) AS revenue
FROM days
LEFT JOIN orders o ON o.order_date = days.d
GROUP BY days.d
ORDER BY days.d;                       -- every day appears, even the empty ones

That second one is worth flagging as a pattern in its own right. When a question asks for "revenue for every day in January, including days with none", no amount of clever grouping will invent a row that isn't in the table. You have to generate the date spine first and outer-join to it. In Postgres you would normally skip the recursion and use generate_series('2024-01-01'::date, '2024-01-31'::date, '1 day'), which does the same thing in one line; the recursive version is what you write when the dialect has no such function.

17. Views, materialized views, and CTAS

A CTE names a result for one query. A view names it forever. The only question that separates the four options is: are you storing the question, or the answer?

A VIEW is a saved SELECT statement. It holds no data — when you query it, the engine substitutes the stored query text into yours and runs the whole thing. So a view is always perfectly fresh (it re-reads the base tables every single time) and it costs you the full computation on every read. It is a name for a query, nothing more.

A materialized view flips that trade: it runs the query once and stores the resulting rows on disk. Reads become instant — but the data is a snapshot, and it goes stale the moment a base table changes. You have to refresh it. And CREATE TABLE AS SELECT (CTAS) goes furthest: it's just a plain table built from a query, frozen at the moment you ran it, with no link back to its sources at all.

sql
-- VIEW: a name for a query. No data stored. Always current.
CREATE VIEW active_customers AS
SELECT id, name, country
FROM customers
WHERE churned_at IS NULL;

SELECT * FROM active_customers WHERE country = 'US';   -- query it like a table
-- The engine rewrites this into the full query against `customers` and runs it.

-- MATERIALIZED VIEW: run it once, store the ROWS. Fast to read, but it goes stale.
CREATE MATERIALIZED VIEW daily_revenue AS
SELECT DATE_TRUNC('day', order_date) AS d, SUM(amount) AS revenue
FROM orders GROUP BY 1;

REFRESH MATERIALIZED VIEW daily_revenue;   -- ← you must do this. It is not automatic.

-- CTAS: a real table, built from a query, frozen at this instant.
-- No link back to `orders` — insert a new order and this table does NOT change.
CREATE TABLE revenue_2024 AS
SELECT cust_id, SUM(amount) AS total
FROM orders
WHERE order_date >= '2024-01-01'
GROUP BY cust_id;

What views are actually for

Three real reasons, and "it's tidier" is not one of them.

A stable interface over a changing table. Twenty dashboards query active_customers. You restructure the underlying customers table — rename a column, split it in two. You rewrite the view once, and all twenty dashboards keep working. The view is a contract; the tables behind it are an implementation detail.

Access control. This is the one people forget, and it's the strongest argument for views. You can grant someone SELECT on a view without granting it on the underlying table — so the view can hide columns (salaries, emails, anything sensitive) and hide rows (a WHERE restricting each team to its own region). The user literally cannot see what the view doesn't select.

Encoding business logic once. If "an active customer" means something specific and fiddly, define it in exactly one place rather than letting six analysts each write their own subtly different version.

CTE or view? The question is scope, not syntax

They do the same job at different scales. A CTE is a name that exists inside one query — perfect for structuring the query you are writing right now, and invisible to everyone else. A view is a name that exists in the database — for logic that many queries, people, or dashboards need to share. If you find yourself pasting the same CTE into query after query, that CTE wants to be a view.

Where this actually matters for you

Be clear-eyed: none of this is testable on LeetCode or StrataScratch — both give you a read-only console, so you can't create anything. It is not exam material. But CTAS is everywhere in the data-engineering job, and especially in Databricks, where CREATE TABLE AS SELECT is the standard way to materialise a Delta table and shows up constantly in the Data Engineer Professional material. The view-versus-materialized-view trade-off — freshness against read speed — is also the exact trade-off behind every incremental pipeline you will ever build, so understanding it here pays off far beyond SQL.

18. DDL: tables, types, and constraints

DDL defines the shape of the database. It is where you tell the engine what a valid row even is — and then let it enforce that for you, forever, against every bug you will ever write.

DDL or DML? Get the words right — it's a real interview question

DDL — Data Definition Language — changes the shape of the database: CREATE, ALTER, DROP, TRUNCATE. DML — Data Manipulation Language — changes the data inside it: SELECT, INSERT, UPDATE, DELETE (§19). The CREATE VIEW and CREATE TABLE AS from §17 are DDL too.

The practical difference that bites: DDL usually commits immediately and cannot be rolled back in most engines (Postgres is the generous exception), while DML sits inside your transaction until you say COMMIT. So a mistyped DROP TABLE is a very different kind of bad day from a mistyped DELETE.

CREATE TABLE: the whole language in one statement

sql
CREATE TABLE employees (
    id        INT          PRIMARY KEY,           -- unique + not null, one per table
    name      VARCHAR(100) NOT NULL,              -- must always have a value
    email     VARCHAR(255) UNIQUE,                -- no two rows may share one
    dept_id   INT          REFERENCES dept(id),   -- FK: must point at a real dept
    salary    NUMERIC(10,2),                        -- EXACT decimal — money goes here
    rating    INT          CHECK (rating BETWEEN 1 AND 10),
    active    BOOLEAN      DEFAULT TRUE,           -- used when you don't supply one
    hired_at  DATE         DEFAULT CURRENT_DATE
);

Every one of those constraints is the database refusing to let you corrupt it. Insert a duplicate id, a NULL name, a second row with the same email, a dept_id pointing at a department that doesn't exist, or a rating of 50 — and each is rejected with an error, not quietly written. That is the point. Your pipeline has a bug; the constraint means the bug fails loudly instead of silently poisoning the table.

Types: the only two decisions that really matter

There are dozens of types. Almost all of the time you want INT, VARCHAR/TEXT, BOOLEAN, DATE/TIMESTAMP, and NUMERIC. Two choices, though, are worth getting right, because both are genuinely expensive to undo later.

Never store money in a FLOAT

FLOAT, REAL and DOUBLE PRECISION are approximate — they are binary fractions, and they cannot represent 0.1 exactly. In a float, 0.1 + 0.2 is 0.30000000000000004, and 0.1 + 0.2 = 0.3 evaluates to FALSE. Sum a million of those and your revenue report is off by cents that nobody can explain and finance will absolutely notice.

Use NUMERIC(10,2) / DECIMAL for anything that is money — it stores exact decimal digits, and 0.1 + 0.2 really is 0.3. Floats are fine for measurements where a rounding error in the fifteenth digit is meaningless; they are never fine for currency.

The other choice is TIMESTAMP versus TIMESTAMPTZ. A plain TIMESTAMP stores a wall-clock reading with no timezone — so "2024-03-15 14:30" is ambiguous, and two servers in different regions will disagree about what moment it names. TIMESTAMPTZ pins it to an actual instant. Store TIMESTAMPTZ (or UTC everywhere, consistently) and convert to local time only when you display it. Getting this wrong is the classic "our daily job double-counted an hour when the clocks changed" incident.

ALTER and DROP

sql
ALTER TABLE employees ADD COLUMN manager_id INT;
ALTER TABLE employees DROP COLUMN rating;
ALTER TABLE employees ALTER COLUMN salary TYPE NUMERIC(12,2);
ALTER TABLE employees ADD CONSTRAINT fk_dept
      FOREIGN KEY (dept_id) REFERENCES dept(id);
ALTER TABLE employees RENAME COLUMN name TO full_name;

DROP TABLE employees;              -- gone: rows, structure, indexes, everything
DROP TABLE IF EXISTS employees;    -- no error if it was never there — use this in scripts

A quiet warning about ALTER on a big table: adding a column with a DEFAULT, or changing a column's type, may force the engine to rewrite every row and hold a lock while it does — which on a 500-million-row table means your "quick schema tweak" takes the table offline for an hour. Modern Postgres avoids the rewrite for a simple ADD COLUMN ... DEFAULT, but the general lesson stands: on a large table, check what the ALTER actually does before you run it in production.

19. DML: INSERT, UPDATE, DELETE

Everything so far has been reading. These three statements change the data — and they have no undo button unless you put one there yourself.

The three statements

sql
-- INSERT: add rows. Always name the columns — relying on table order
-- silently breaks the day someone adds a column in the middle.
INSERT INTO customers (name, country) VALUES ('Ann', 'US'), ('Bob', 'CA');

-- INSERT ... SELECT: the workhorse. Load a table from a query — this is how
-- most real pipeline steps actually write their output.
INSERT INTO revenue_summary (cust_id, total)
SELECT cust_id, SUM(amount) FROM orders GROUP BY cust_id;

-- UPDATE: change existing rows. The WHERE is the whole safety mechanism.
UPDATE customers SET country = 'USA' WHERE country = 'US';

-- DELETE: remove rows. Same story — WHERE or bust.
DELETE FROM orders WHERE order_date < '2020-01-01';

The one that ends careers: forgetting the WHERE

UPDATE customers SET country = 'USA'; is valid SQL. It sets every customer in the table to USA, runs without complaint, and reports how many million rows it changed. Likewise DELETE FROM orders; empties the table. There is no confirmation prompt.

The habit that saves you: write it as a SELECT first. Run SELECT * FROM customers WHERE country = 'US';, look at the rows, confirm they're the ones you meant — then swap SELECT * for UPDATE ... SET or DELETE, leaving the WHERE untouched. And on anything that matters, wrap it in a transaction so you have a way back.

sql
-- The safety net. Nothing is permanent until COMMIT.
BEGIN;
  DELETE FROM orders WHERE order_date < '2020-01-01';
  -- check the damage BEFORE you commit:
  SELECT COUNT(*) FROM orders;
ROLLBACK;   -- ← undo it all. Or COMMIT; to make it real.

Updating from another table, and upserts

Two things you'll write constantly in a real pipeline and never on a puzzle site.

sql
-- UPDATE using values from ANOTHER table. The syntax differs by dialect.
UPDATE customers c                          -- Postgres
SET    lifetime_value = o.total
FROM   (SELECT cust_id, SUM(amount) AS total FROM orders GROUP BY cust_id) o
WHERE  o.cust_id = c.id;

UPDATE customers c                          -- MySQL
JOIN   (SELECT cust_id, SUM(amount) AS total FROM orders GROUP BY cust_id) o
       ON o.cust_id = c.id
SET    c.lifetime_value = o.total;

-- UPSERT: "insert it, or update it if it's already there." The backbone of any
-- idempotent load — re-running the pipeline must not duplicate rows.
INSERT INTO customers (id, name)            -- Postgres
VALUES (1, 'Ann')
ON CONFLICT (id) DO UPDATE SET name = EXCLUDED.name;   -- EXCLUDED = the row you tried to insert

INSERT INTO customers (id, name)            -- MySQL
VALUES (1, 'Ann')
ON DUPLICATE KEY UPDATE name = VALUES(name);

The two write problems LeetCode actually asks

Writes are rare on the practice sites, but not absent — and both classics are really just a SELECT you already know how to write, with the verb swapped.

sql
══ SWAP SALARY ═══ flip every 'm' to 'f' and vice versa, in one pass.
   No WHERE, deliberately — every row should change. CASE does the work (§11).
UPDATE salary
SET sex = CASE WHEN sex = 'm' THEN 'f' ELSE 'm' END;
   (A naive two-step UPDATE would set everyone to 'f', then set them all back to 'm'.)

══ DELETE DUPLICATE EMAILS ═══ keep the lowest id per email, delete the rest.
   Self-join the table to itself and delete the rows that have a smaller-id twin.
DELETE FROM person p1                       -- Postgres
USING person p2
WHERE p1.email = p2.email AND p1.id > p2.id;

DELETE p1 FROM person p1                     -- MySQL
JOIN person p2 ON p1.email = p2.email AND p1.id > p2.id;

Notice the shape of that second one. "Delete the duplicates" is just "find the duplicates" — a self-join where a row with the same email has a smaller id — with SELECT swapped for DELETE. That is the general move: get the SELECT right first, confirm the rows, then change the verb. It is both how you solve the problem and how you avoid deleting your production table.

20. Window functions: the anatomy

A window function computes a value across a set of related rows — without collapsing them. That is the entire difference from GROUP BY, and it is what makes Hard problems tractable.

Run GROUP BY dept and your forty engineers become one row. You get the department average, but you have lost every individual. Run AVG(salary) OVER (PARTITION BY dept) and you still have forty rows — each one now carrying its department's average alongside its own salary. You can compare a row to its group on the same row, which is precisely the thing that used to require a correlated subquery.

sql
-- GROUP BY: 40 engineers → 1 row. The individuals are gone.
SELECT dept, AVG(salary) FROM emp GROUP BY dept;

-- OVER: 40 engineers → 40 rows, each carrying its department's average.
SELECT name, salary, dept,
       AVG(salary) OVER (PARTITION BY dept) AS dept_avg,
       salary - AVG(salary) OVER (PARTITION BY dept) AS vs_dept
FROM emp;

Everything else is detail on the OVER (...) clause, which has exactly three knobs.

KnobWhat it doesIf you omit it
PARTITION BYSplits rows into independent groups. The function restarts at each one.The whole result set is one single partition
ORDER BYSequences rows within a partition. Required by ranking and offset functions.No order — and, crucially, the frame defaults to the entire partition
frameWhich rows around the current one actually feed the aggregate.With an ORDER BY: RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. Without one: the whole partition

The default frame is the thing that surprises everyone

Adding an ORDER BY to an aggregate window silently changes what it computes. SUM(x) OVER (PARTITION BY d) is the partition total — the same number on every row. Add an ORDER BY and it becomes SUM(x) OVER (PARTITION BY d ORDER BY t), whose implicit frame is "everything up to and including me" — a running total, different on every row. Same function, same partition, completely different answer. If you want the plain total and an ordering, either drop the ORDER BY from that one window or write the frame out explicitly.

21. Window frames: ROWS vs RANGE

The frame is a sliding cut-out. For each row, it says: of my partition, which rows do I actually aggregate?

ROWS or RANGE?

ROWS counts physical rows: "the 2 rows before me" means exactly two rows, whatever they contain. RANGE counts logical peers: all rows with the same ORDER BY value are treated as one indivisible clump, and the frame either takes the whole clump or none of it.

This only matters when your ORDER BY column has ties — and when it does, it matters a lot. Suppose you order by date and two orders land on the same day. A running total with RANGE gives both rows the same value (the total including both), because they are peers. With ROWS they get different values, because row 2 is physically after row 1. The default frame is RANGE, which means the common "running total" query has a subtle behaviour most people never notice until a tie appears in the data.

sql
-- Be explicit whenever ties are possible. This is the safe running total:
SUM(amount) OVER (PARTITION BY user_id ORDER BY order_date
                     ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)

-- A 7-day moving average (7 rows: today and the 6 before it).
-- ROWS is right here: you want exactly 7 rows, not "all rows sharing a date".
AVG(revenue) OVER (ORDER BY day ROWS BETWEEN 6 PRECEDING AND CURRENT ROW)

-- Centred window: 1 before, me, 1 after.
AVG(x) OVER (ORDER BY t ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING)

-- The whole partition, even though we're ordering. Note the explicit frame
-- overriding the default — this is how you get "total" and "running total" side by side.
SELECT day, revenue,
  SUM(revenue) OVER (ORDER BY day) AS running,
  SUM(revenue) OVER (ORDER BY day
                       ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING) AS grand_total,
  ROUND(100.0 * revenue / SUM(revenue) OVER (), 2) AS pct_of_total  -- empty OVER() = everything
FROM daily;

That last line is a lovely idiom worth stealing: an empty OVER () means "the entire result set", so x / SUM(x) OVER () is each row's share of the grand total — no subquery, no join, no second pass.

22. The window function families

Four families. Learn what each is for and you'll reach for the right one without thinking.

1. Ranking — and the three-way tie-break question

The difference between ROW_NUMBER, RANK and DENSE_RANK shows up only when there are ties, and interviewers reliably construct data that has them. Here is all three over the same scores:

scoreROW_NUMBER()RANK()DENSE_RANK()
100111
90222
903 ← arbitrary!22

ROW_NUMBER always gives distinct integers, so it breaks ties arbitrarily — if two rows tie and you don't add a tie-breaker to the ORDER BY, which one gets 2 and which gets 3 is not deterministic between runs. RANK gives ties the same number and then skips (1, 2, 2, 4). DENSE_RANK gives ties the same number and doesn't skip (1, 2, 2, 3).

Which one does the question want?

"The top 3 salaries" (three distinct salary values, however many people earn them) → DENSE_RANK() <= 3. "The 3 highest-paid employees" (exactly three people) → ROW_NUMBER() <= 3, and you must add a tie-break column to make it deterministic. "The Nth highest salary" → DENSE_RANK() = N, almost always. Read the noun: values means DENSE_RANK, people means ROW_NUMBER.

sql
ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC, id)  -- id = deterministic tie-break
RANK()       OVER (PARTITION BY dept ORDER BY salary DESC)
DENSE_RANK() OVER (PARTITION BY dept ORDER BY salary DESC)
NTILE(4)     OVER (ORDER BY salary)      -- quartiles: 1,1,2,2,3,3,4,4
PERCENT_RANK() OVER (ORDER BY salary)   -- (rank-1)/(n-1), from 0.0 to 1.0
CUME_DIST()    OVER (ORDER BY salary)   -- fraction of rows ≤ me

2. Offset — looking at the neighbouring row

LAG reaches backwards, LEAD reaches forwards. These are what turn "compare each month to the one before it" from a self-join into a single expression, and they are the backbone of every growth, churn and time-between-events question.

sql
-- The third argument is the DEFAULT when there is no such row.
-- Without it, the first row of every partition gets NULL — which then
-- poisons any arithmetic you do with it.
LAG(revenue, 1, 0)  OVER (PARTITION BY user_id ORDER BY mth)  -- previous month, 0 if none
LEAD(login_at)     OVER (PARTITION BY user_id ORDER BY login_at) -- their next login

-- FIRST_VALUE / LAST_VALUE: the extremes of the frame, not of the partition!
-- LAST_VALUE is the classic trap. The default frame ends at CURRENT ROW,
--   so "the last value" is... the current row. You must widen the frame.
FIRST_VALUE(price) OVER (PARTITION BY product ORDER BY day)   -- + works: first row
LAST_VALUE(price)  OVER (PARTITION BY product ORDER BY day)   -- − returns the CURRENT row
LAST_VALUE(price)  OVER (PARTITION BY product ORDER BY day
                          ROWS BETWEEN UNBOUNDED PRECEDING
                                   AND UNBOUNDED FOLLOWING)     -- + now it's the real last

-- Month-over-month growth, the canonical use.
SELECT mth, revenue,
       LAG(revenue) OVER (ORDER BY mth) AS prev,
       ROUND(100.0 * (revenue - LAG(revenue) OVER (ORDER BY mth))
              / NULLIF(LAG(revenue) OVER (ORDER BY mth), 0), 1) AS pct_growth
FROM monthly_revenue;

That NULLIF(..., 0) is not decoration. The first month has no previous month, so LAG returns NULL, and if a month genuinely had zero revenue you would divide by zero and crash. NULLIF turns both into a clean NULL growth figure, which is the honest answer.

3. Aggregate windows — every aggregate you know, with an OVER

Any aggregate — SUM, AVG, COUNT, MIN, MAX — becomes a window function the moment you bolt OVER (...) onto it. This is the family that replaces correlated subqueries, and the one you'll use most.

sql
SELECT user_id, mth, revenue,
  SUM(revenue) OVER (PARTITION BY user_id)                    AS lifetime_total,
  SUM(revenue) OVER (PARTITION BY user_id ORDER BY mth)      AS running_total,
  AVG(revenue) OVER (PARTITION BY user_id ORDER BY mth
                       ROWS BETWEEN 2 PRECEDING AND CURRENT ROW)  AS moving_avg_3,
  COUNT(*)     OVER (PARTITION BY user_id)                    AS months_active,
  revenue - AVG(revenue) OVER (PARTITION BY user_id)        AS vs_own_average
FROM monthly;

Look at lines two and three: identical function, identical partition, and the only difference is the ORDER BY — which flips it from a fixed lifetime total to a running total. That is the default-frame rule from §20 doing its quiet work, and it is why writing frames explicitly is a habit worth building.

4. Distribution and the WINDOW clause

When the same OVER (...) appears three times in one query, name it once with a WINDOW clause. It's pure readability, it's standard SQL, and almost nobody knows it exists — which makes it a nice thing to show an interviewer.

sql
SELECT user_id, mth, revenue,
       SUM(revenue)  OVER w AS running,
       AVG(revenue)  OVER w AS running_avg,
       ROW_NUMBER()  OVER w AS seq
FROM monthly
WINDOW w AS (PARTITION BY user_id ORDER BY mth);   -- define once, use three times

23. Window patterns that crack Hard problems

Six templates. Between them they cover a startling share of everything tagged Hard on LeetCode and StrataScratch.

Pattern 1 · Top N per group

The most common Medium problem in existence, in every disguise: highest-paid employee per department, best-selling product per category, most recent order per customer. Rank inside the partition, then filter the rank in an outer layer — because §02 forbids filtering a window function in WHERE.

sql
WITH ranked AS (
    SELECT dept, name, salary,
           DENSE_RANK() OVER (PARTITION BY dept ORDER BY salary DESC) AS rnk
    FROM employees
)
SELECT dept, name, salary
FROM ranked
WHERE rnk <= 3                -- DENSE_RANK: top 3 SALARY LEVELS, ties all included
ORDER BY dept, salary DESC;   -- swap for ROW_NUMBER if you want exactly 3 PEOPLE

Pattern 2 · Deduplication — keep one row per key

Same skeleton, different intent: number the duplicates and keep the first.

sql
WITH d AS (
    SELECT *, ROW_NUMBER() OVER (PARTITION BY email ORDER BY created_at) AS rn
    FROM users             -- PARTITION BY = "what counts as a duplicate"
)                          -- ORDER BY   = "which copy do I keep"
SELECT * FROM d WHERE rn = 1;   -- keep the earliest signup per email

Pattern 3 · Running total and moving average

sql
SELECT day, revenue,
  SUM(revenue) OVER (ORDER BY day
                       ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS cumulative,
  AVG(revenue) OVER (ORDER BY day
                       ROWS BETWEEN 6 PRECEDING AND CURRENT ROW)        AS ma_7day
FROM daily_revenue;

One caveat that separates a correct answer from a plausible one: a 7-row moving average is only a 7-day moving average if there is a row for every day. If days with no revenue are simply missing from the table, ROWS 6 PRECEDING reaches back seven rows, which might span three weeks. Generate a date spine (§16) and LEFT JOIN to it first.

Pattern 4 · Month-over-month change with LAG

sql
WITH monthly AS (
    SELECT DATE_TRUNC('month', order_date) AS mth, SUM(amount) AS revenue
    FROM orders GROUP BY 1
)
SELECT mth, revenue,
       LAG(revenue) OVER (ORDER BY mth) AS prev_revenue,
       revenue - LAG(revenue) OVER (ORDER BY mth) AS change,
       ROUND(100.0 * (revenue - LAG(revenue) OVER (ORDER BY mth))
              / NULLIF(LAG(revenue) OVER (ORDER BY mth), 0), 1) AS pct_change
FROM monthly
ORDER BY mth;

Pattern 5 · Gaps and islands — consecutive runs

"Find users who logged in three days in a row." "Find the longest streak." "Group consecutive events into sessions." These all look fearsome and all yield to one trick: if a sequence is consecutive, then value − ROW_NUMBER() is constant across the run. Numbering advances by one; the value advances by one; their difference doesn't move. The moment there's a gap, the difference jumps — and that difference becomes a group key for the run.

sql
-- Longest login streak per user.
WITH numbered AS (
    SELECT user_id, login_day,
           ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY login_day) AS rn
    FROM (SELECT DISTINCT user_id, login_day FROM logins) u   -- dedupe first!
),
islands AS (
    SELECT user_id, login_day,
           login_day - rn * INTERVAL '1 day' AS island_key   -- the constant
    FROM numbered
)
SELECT user_id, COUNT(*) AS streak_length,
       MIN(login_day) AS streak_start, MAX(login_day) AS streak_end
FROM islands
GROUP BY user_id, island_key      -- one group per consecutive run
ORDER BY streak_length DESC;

Two details that decide whether this actually works. Deduplicate first — if a user logged in twice on one day, the row numbers get out of step with the dates and every island key is wrong. And on integer sequences (like consecutive IDs) it's simply id - rn; only dates need the INTERVAL arithmetic.

Pattern 6 · The median, and other percentiles

SQL has no MEDIAN aggregate in most dialects, and "find the median salary per department" is a beloved Hard problem. There are two good answers.

sql
-- + THE EASY WAY (Postgres, and most modern engines): an ordered-set aggregate.
-- If your dialect has this, use it and say so.
SELECT dept,
       PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY salary) AS median_salary
FROM employees
GROUP BY dept;

-- + THE PORTABLE WAY (works anywhere with window functions).
-- Number the rows from both ends. The median row(s) are where the two
-- numberings are within one of each other — which handles odd AND even counts:
--   odd  n: exactly one row satisfies it → that row IS the median
--   even n: exactly two rows satisfy it → AVG them
WITH r AS (
    SELECT dept, salary,
           ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary ASC)  AS asc_n,
           ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC) AS desc_n
    FROM employees
)
SELECT dept, AVG(salary) AS median_salary
FROM r
WHERE ABS(asc_n - desc_n) <= 1
GROUP BY dept;

24. Set operations

Joins glue tables together sideways — more columns. Set operations stack them vertically — more rows.

All of them require the two queries to be union-compatible: the same number of columns, in the same order, with compatible types. Column names come from the first query; the rest are ignored.

OperatorReturnsDuplicates
UNION ALLEverything from both, stackedKept — and this is faster
UNIONEverything from bothRemoved (an implicit global sort/hash)
INTERSECTRows present in bothRemoved (unless INTERSECT ALL)

Default to UNION ALL

UNION must de-duplicate, which means sorting or hashing every row of the combined result — real work, on data that often has no duplicates to begin with. UNION ALL just concatenates. Use UNION ALL unless you actively need duplicates gone, and if you're writing UNION to clean up after a bad join, fix the join instead.

sql
-- The most useful set-op pattern: unpivot. Turn columns back into rows.
SELECT user_id, 'signup' AS event, signup_at  AS ts FROM users WHERE signup_at  IS NOT NULL
UNION ALL
SELECT user_id, 'purchase',      purchase_at      FROM users WHERE purchase_at IS NOT NULL
UNION ALL
SELECT user_id, 'churn',         churned_at       FROM users WHERE churned_at  IS NOT NULL
ORDER BY user_id, ts;         -- ORDER BY goes at the very END, once, for the whole thing

-- EXCEPT as a data-quality check: which rows changed between two snapshots?
SELECT * FROM snapshot_today
EXCEPT
SELECT * FROM snapshot_yesterday;   -- new or modified rows, in one line

25. How to attack a Medium–Hard problem

The people who solve these fast are not writing SQL faster. They are deciding what to write faster, because they follow the same six steps every time.

1

Name the grain of the answer

Before anything else: one row of my output = one what? One customer? One (customer, month)? One product per category? This single sentence tells you your GROUP BY or your PARTITION BY, and half the query writes itself. Most wrong answers are wrong because the grain was never decided.

2

Sort every condition into WHERE or HAVING

Re-read the question and label each condition. Is it about a row ("orders over $100") → WHERE. Is it about a group ("customers with more than 3 orders") → HAVING. Is it about a rank ("the top 3") → a window function plus an outer filter. Three boxes, and everything goes in one of them.

3

Check the join for fan-out before you aggregate

For every join, ask out loud: one-to-one, or one-to-many? If it's one-to-many and you're about to SUM something from the one side, stop and pre-aggregate (§08). This is the most common silent wrong answer in the whole game.

4

Ask what the NULLs do

Any LEFT JOIN? Then COUNT(*) is probably wrong and SUM probably needs a COALESCE. Any NOT IN? Rewrite it as NOT EXISTS now, before it bites. Any division? Wrap the denominator in NULLIF(x, 0).

5

Build it in CTEs, one stage at a time

Write the first CTE. Run SELECT * FROM it LIMIT 10 and look at the rows. Then the next. Debugging four small named steps is easy; debugging one 40-line query with three nested subqueries is not. You are never penalised for CTEs — you are rewarded, because the interviewer can follow your thinking.

6

Say out loud what happens on ties and empties

What if two departments tie for highest average? What if a customer has zero orders — do they appear with a 0, or not at all? What if the table is empty? The question usually has an opinion, and voicing it is what separates a candidate who wrote a query from one who solved the problem.

The five-move repertoire

Almost every Medium–Hard problem is a combination of five moves. Reshape (join, or union), collapse (GROUP BY), count conditionally (CASE inside an aggregate), look sideways (a window function — rank, lag, running total), and name the steps (a CTE). When you're stuck, run down that list and ask which move gets you one step closer to the grain you named in step 1.

Glossary

Declarative
You describe what the answer looks like; the engine decides how to compute it. The reason a query planner can rewrite your SQL and still be correct.
Logical order of operations
FROM → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT. Not the order you write them in, and the source of most SQL errors.
Projection
Choosing and computing columns — what the SELECT clause does. Confusingly, relational algebra calls row-filtering "selection", which is what SQL's WHERE does.
Scalar function
One row in, one value out — UPPER, ROUND, DATE_TRUNC, COALESCE. Contrast with an aggregate (many rows in, one value out) and a window function (many rows in, one value out per row, with the rows preserved). Which of the three a function is determines where it may legally appear.
Predicate
An expression that evaluates to TRUE, FALSE or UNKNOWN. WHERE keeps a row only when its predicate is TRUE.
Three-valued logic
SQL's truth system: TRUE, FALSE, and UNKNOWN. Any comparison involving NULL yields UNKNOWN, and UNKNOWN rows are discarded exactly like FALSE ones.
Sargable
A predicate an index can be used for — which means the column is bare on one side. WHERE YEAR(d) = 2024 is not sargable; WHERE d >= '2024-01-01' AND d < '2025-01-01' is.
Grain
What one row of a table or result means: one order, one (user, month), one product per category. Naming the grain of your answer is the first step of every problem.
Fan-out
When a one-to-many join duplicates the rows of the "one" side, so any aggregate over its columns is multiplied. The most common silent wrong answer in SQL.
Semi-join
"Does a match exist?" — returns left rows only, never duplicating them. EXISTS.
Anti-join
"Is there no match?" — NOT EXISTS, or a LEFT JOIN with WHERE right.key IS NULL. Never NOT IN, which NULLs will break.
Correlated subquery
A subquery that references a column from the outer query, and so must be re-evaluated for every outer row. Usually replaceable by a join or a window function.
Scalar subquery
A subquery returning exactly one row and one column, so it can be used anywhere a single value can.
Derived table
A subquery in the FROM clause, used as if it were a table. Must be given an alias.
CTE
Common Table Expression — a named subquery defined in a WITH clause. Functionally like a derived table; vastly more readable, and inspectable one stage at a time.
Recursive CTE
A WITH RECURSIVE whose body refers to itself: an anchor query, UNION ALL, and a recursive step. For hierarchies of unknown depth and for generating series. Needs a termination condition or it never stops.
Window function
A function computed across a set of related rows without collapsing them. Unlike GROUP BY, the rows survive and each one carries its group's answer.
Partition
The PARTITION BY group a window function restarts at. Independent of, and unrelated to, table partitioning on disk.
Frame
Which rows within the partition actually feed the window aggregate — ROWS BETWEEN … AND …. Defaults to "start of partition through current row" whenever there is an ORDER BY, which is why adding one turns a total into a running total.
ROWS vs RANGE
ROWS counts physical rows. RANGE treats rows with equal ORDER BY values as one indivisible peer group. They differ only when there are ties — and then they differ a lot.
Conditional aggregation
SUM(CASE WHEN … THEN 1 ELSE 0 END) and friends: a CASE inside an aggregate, letting each row choose whether it contributes. Pivots rows into columns with no special syntax.
Gaps and islands
The family of "consecutive run" problems. The trick: for a consecutive sequence, value − ROW_NUMBER() is constant, so grouping by that difference gives one group per run.
Union-compatible
The requirement for UNION / INTERSECT / EXCEPT: same number of columns, same order, compatible types.
Ordered-set aggregate
PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY x) — an aggregate that needs its input sorted. The clean way to get a median where the dialect supports it.
Row subquery
A subquery returning one row of several columns, compared as a tuple: (dept, salary) IN (SELECT dept, MAX(salary) …). The columns stay paired — split them into two separate INs and you get rows matching some other group's maximum.
DDL
Data Definition Language — changes the shape of the database: CREATE, ALTER, DROP, TRUNCATE. Usually commits immediately and cannot be rolled back.
DML
Data Manipulation Language — changes the data inside it: SELECT, INSERT, UPDATE, DELETE. Sits inside your transaction until you COMMIT.
Constraint
A rule the database enforces on every write — PRIMARY KEY, FOREIGN KEY, NOT NULL, UNIQUE, CHECK. Bad data is rejected with an error rather than quietly stored, which is what makes it the last line of defence against a buggy pipeline.
Referential integrity
What a FOREIGN KEY guarantees: the value it points at really exists in the parent table. You cannot create an orphan row.
View
A saved SELECT. Stores the query, not the data — so it is always current and is recomputed on every read. Also the standard tool for access control: grant on the view, not the table, and it can hide columns and rows.
Materialized view
A view that stores the result rows on disk. Instant to read, but stale until you REFRESH it. The freshness-versus-speed trade-off in one object.
CTAS
CREATE TABLE AS SELECT — a real table built from a query and frozen at that moment, with no link back to its sources. The standard way to materialise a table in Databricks and Delta Lake.
Upsert
"Insert it, or update it if it's already there": ON CONFLICT … DO UPDATE (Postgres) or ON DUPLICATE KEY UPDATE (MySQL). The backbone of an idempotent load — re-running the pipeline must not duplicate rows.
Idempotent
Safe to run twice: the second run either does nothing or produces exactly the same result. The single most valuable property a scheduled data job can have.
Transaction
BEGIN … COMMIT (or ROLLBACK). Nothing you did is permanent until you commit — which is the only undo button UPDATE and DELETE will ever give you.
