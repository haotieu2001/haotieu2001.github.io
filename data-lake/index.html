Data Lake & Data Lakehouse, Simply Explained

A Data Lake gives you cheap, flexible storage — but with no rules and no safety net. This guide opens it up: how Parquet stores your data underneath, the six ways plain lakes break in production, and how Delta Lake turns them into a reliable source of truth.

~28 min read · Data Engineering · Jun 2026

❯ The one idea (read this first)

A Data Lake = cheap cloud storage (S3 / ADLS) full of Parquet files. It's cheap and flexible, but writing to it is like dropping files into a shared folder — no rules, no safety net. A Data Lakehouse = the same cheap storage, but with a smart transaction layer on top (Delta Lake). That layer adds the reliability of a database without giving up the flexibility or low cost of a lake.

01. What is a Data Lake?

Before Data Lakes existed, companies stored all their data in relational databases (like MySQL or Oracle). That worked fine when data was small — a few million rows. But by the mid-2000s, companies were generating billions of events every day: web clicks, app logs, sensor readings, payment transactions, ad impressions. Storing all of that in a traditional database became impossibly expensive.

For analytics specifically, the answer through the 1990s and 2000s was the Data Warehouse — a database purpose-built for reporting (Teradata first, later cloud versions like Amazon Redshift, Google BigQuery, and Snowflake). A warehouse takes your structured business data, stores it in a tightly optimised internal format, and answers complex questions like "revenue by region per quarter" extremely fast. For clean, structured data it is still excellent today, and it's what powers most BI dashboards.

But a Data Warehouse has two hard limits that matter at scale:

  • Cost. A warehouse bundles storage and compute into one expensive system and charges a premium per gigabyte — often 10–20× the price of raw cloud storage. Keeping years of raw clickstream logs in it is wildly uneconomical.
  • Rigidity. Data must be cleaned and fitted to a strict table schema before it can go in — this is called "schema on write." Semi-structured JSON, images, audio, video, and free-form logs don't fit neatly. You can't just dump data now and decide its shape later.

The Data Lake was the response to both limits. The core idea: separate storage from computing. Instead of one server that stores AND processes data, you store everything in cheap cloud storage, and only rent compute power when you actually need to process it.

That's a Data Lake: cheap cloud storage (specifically object storage) where you dump raw data — CSV exports from your database, JSON from your API, log files from your servers, images, audio files, anything — without worrying about structure. You load now, figure out the shape when you read it.

What is object storage?

Object storage (Amazon S3, Azure ADLS, Google GCS) is a cloud service that stores files at massive scale for very low cost — roughly $0.02 per GB per month. A traditional database server with the same raw storage capacity can cost 50–100× more, because the database server also has to pay for CPU, RAM, and software licences that are always running even when idle.

The key trade-off: object storage is immutable by design — you can write new files and delete old ones, but you cannot edit a file in-place (like changing one row). That limitation is exactly what causes most of the problems we'll discuss in Section 05.

What kind of data goes into a Data Lake?

Everything. Here's what a real company might store:

  • App events — every click, page view, and button press from your website or mobile app, as JSON files.
  • Database exports — nightly CSV or Parquet dumps of your orders table, customers table, products table.
  • Server logs — raw text logs from web servers, load balancers, and microservices.
  • Third-party data — files from payment processors, marketing platforms, or data vendors.
  • ML training data — images, audio, video, or labelled datasets for machine learning models.

All of this lands in one place, in whatever format it arrived in. The key insight is that you don't need to clean or restructure data to store it — you just need it somewhere safe and cheap.

02. Data Lake architecture — what it's actually made of

A Data Lake has three layers stacked on top of each other. Each layer builds on the one below it:

Layer 1 — Storage (the files)

The base layer is a folder structure in cloud object storage. A typical layout looks like this:

The sub-folders like year=2026/month=06/day=01/ are called partitions. Here is why they matter:

Imagine the orders table has 3 years of data across 10,000 Parquet files. Without partitioning, a query for "orders from June 1, 2026" has to open all 10,000 files, check each one, and discard the ones that don't match. With date partitions, the query engine knows that June 1 data is in year=2026/month=06/day=01/ and only reads the 2 files in that folder, skipping the other 9,998. This is called partition pruning and it's the single biggest performance lever in a Data Lake.

Layer 2 — Metadata (the catalogue)

The storage layer is just a pile of files on S3. By itself, Spark can't do SELECT * FROM orders because it doesn't know what "orders" is or where it lives. That's what the metadata layer is for.

The most common tool is the Hive Metastore — a service (a small running server, usually managed for you by Databricks, AWS Glue, or a standalone Hive install) that stores a catalogue of your tables. Think of it as a phonebook: instead of "person → phone number", it stores "table name → S3 location + column definitions".

What the Hive Metastore stores for one table
Table name : orders
Location   : s3://my-company-lake/raw/orders/
Columns    : order_id (INT), customer_id (INT), amount (DOUBLE), order_date (DATE)
Partitions : year, month, day

When you run SELECT * FROM orders WHERE order_date = '2026-06-01', Spark asks the metastore: "where is 'orders'?" The metastore replies with the S3 path and the column definitions. Spark then knows where to look and what the data looks like. The metastore is what turns a folder of files into a table you can query by name.

Hive Metastore vs Unity Catalog vs AWS Glue

These are all versions of the same idea — a catalogue that maps table names to data locations:

  • Hive Metastore — the original open-source catalogue, still widely used. You manage it yourself.
  • AWS Glue Data Catalog — Amazon's managed version. Works with Athena, EMR and Spark on AWS with no servers to manage.
  • Unity Catalog — Databricks' next-generation catalogue. Adds fine-grained access control (who can see which columns), data lineage (where did this data come from?), and works across clouds. If you're on Databricks, this is what you use.

They all do the same core job. The differences are in management overhead and extra features.

Layer 3 — Compute (the engine)

The compute layer is what reads the files, transforms them, and produces results. The most important thing about this layer is that it's separate from storage — you spin up a cluster only when you need it.

🖥 What is a "cluster"?

A cluster is a group of virtual machines — rented cloud servers — that work together. When you "spin up a Spark cluster" with 10 machines, you're renting 10 virtual servers from AWS/Azure/GCP. Spark splits your data processing job across all 10 machines so they work in parallel, finishing in roughly 1/10th the time it would take one machine alone. When the job is done, you release the machines and stop paying for them. The separation of compute and storage is the key cost advantage of a Data Lake: you only pay for compute while it's actually running, not 24/7.

03. How Parquet works inside

Most data in a modern Data Lake is stored as Parquet. Before we look at the structure, it helps to understand why Parquet is designed the way it is — because everything else follows from one simple insight about analytics queries.

The core insight: analytics queries only need a few columns

A typical analytics query looks like this:

SQL · typical analytics query
SELECT category, SUM(amount) AS total_revenue
FROM orders
WHERE order_date >= '2026-06-01'
GROUP BY category;

This query only needs two columns out of a table that might have 30: category, amount, and order_date for filtering. But how is the data stored on disk?

  • Row-oriented storage (CSV, traditional databases): stores all columns of row 1 together, then all columns of row 2, and so on. To read just 3 columns from a 30-column table, you have to read all 30 columns for every row — then throw away the 27 you don't need. You're reading ~10× more data than necessary.
  • Column-oriented storage (Parquet): stores all values of order_id together, then all values of customer_id together, and so on. To read 3 columns, you skip the other 27 entirely. For a 30-column table, you read roughly 1/10th the data.

This is why Parquet is so much faster for analytics. Less data read from disk = faster query.

The physical layout of a Parquet file

A single Parquet file is organised in three nested layers — row groups contain column chunks, which contain pages:

Row Groups — horizontal slices

A Parquet file is first split into Row Groups — horizontal slices of roughly 128 MB each (the size, not a fixed row count — a table with wide rows might fit fewer rows per group, a table with thin rows might fit more). Think of a row group as a chapter in a book: the footer (explained below) acts like the book's table of contents, telling you exactly which page each chapter starts on. You can jump straight to any chapter without reading from the beginning.

Row groups matter because the engine can skip entire row groups using statistics — without reading a single byte of actual data from them. More on this in the "footer" section below.

Column Chunks — vertical slices within a row group

Inside each row group, the data is split by column. All order_id values for that row group are stored together, all category values together, and so on. This is the "columnar" layout.

Because all values in a column chunk are the same data type, they compress extremely well. Parquet uses two techniques to compress column chunks before writing:

Encoding (shrinking with patterns)

  • Dictionary encoding — if a column has many repeated values (like category which only has "clothing", "food", "tech"), Parquet builds a small lookup table called a dictionary page: 0 = "clothing", 1 = "food", 2 = "tech". The actual data pages then store tiny integers (0, 1, 2) instead of full strings. A 10-character string like "electronics" becomes a 1-byte integer. The dictionary page is stored at the start of the column chunk, and the data pages store the codes. When you read the column, the engine loads the dictionary first, then decodes each code back to the original string.
  • Run-length encoding (RLE) — consecutive identical values are collapsed. Instead of storing NULL, NULL, NULL, NULL four times, Parquet stores "4 × NULL". Great for sparse columns or booleans.
  • Delta encoding — for columns that are sorted or nearly sorted (like auto-increment IDs or timestamps), only the differences between values are stored: 1000, 1001, 1002, 1003 becomes 1000, +1, +1, +1. This works best when values are in order — on unsorted data, the differences would be large and random, saving almost nothing.

Compression (shrinking what's left)

After encoding, Parquet applies a standard compression algorithm on top — similar to how you'd zip a folder. The two most common choices are:

  • Snappy — the historical default. Fast to compress and decompress, but doesn't shrink files as small as ZSTD.
  • ZSTD — the default since Databricks Runtime 16.0, and increasingly common elsewhere. Slower to compress but produces significantly smaller files (often 20–40% smaller than Snappy for the same data). Better for storage cost; Snappy is better if CPU is the bottleneck. (Earlier Databricks runtimes and vanilla Apache Spark still default to Snappy — check your tool's documentation.)

Think of encoding + compression as two passes: encoding uses the patterns in your specific data, compression handles what's left. Applied together, a column of repeat strings can shrink to less than 5% of its original size.

The Footer — the file's index and key to fast queries

The footer is a few kilobytes at the very end of every Parquet file. It's tiny but it's the most important part for query speed, because it tells the engine what's inside the file without reading the whole thing.

📌 Why the very end of the file?

The Parquet format specification requires the last 4 bytes of every file to be a "footer length" — an integer that says exactly how many bytes the footer occupies. When a query engine opens a Parquet file, it always seeks to the end, reads those 4 bytes, then seeks backwards by that many bytes to land at the start of the footer. It's like a book where the very last line tells you "the index starts on page X" — you flip straight to the index without reading the whole book. This is a deliberate design decision baked into the Parquet standard, so every engine knows exactly where to look.

The footer contains three things:

  1. Schema — column names and data types for this file.
  2. Row group locations (byte offsets) — the exact byte position in the file where each row group starts, so the engine can jump directly to it. "Offset 0" means the very start of the file; "Offset 67,108,864" means jump 64 MB in.
  3. Column statistics per row group — for each column in each row group: the minimum value, the maximum value, and how many nulls. This is the magic behind "data skipping."

How statistics let the engine skip data it doesn't need to read

Say you query WHERE amount > 500. The engine reads the footer (a few KB) and finds:

  • Row Group 1: amount min=10, max=200 → skip entirely (200 < 500, so no row here can match)
  • Row Group 2: amount min=100, max=9000 → must read (9000 > 500, some rows might match)
  • Row Group 3: amount min=1, max=50 → skip entirely (50 < 500)

Result: the engine reads only 1 out of 3 row groups. This is called predicate pushdown or data skipping, and it's why a well-organised Parquet lake can scan 10 TB of logical data by physically reading only 50 MB.

💬 Does this work for text columns like category?

Yes. Parquet records min/max/null-count for every column, not just numbers — for text, the min and max are taken in alphabetical order (so a row group holding "clothing", "food", "tech" stores min="clothing", max="tech"). A query like WHERE category = 'food' can then skip any row group whose range can't contain that value.

But there's a catch: skipping only helps if similar values are grouped into the same files. If every row group contains a random mix of all categories, then every group has min="clothing", max="tech" — so "food" falls inside every range and nothing gets skipped. The statistics exist, they're just not useful. This is exactly why sorting/clustering the data by a column (covered later with OPTIMIZE … ZORDER) makes filters on that column fast.

How a query reads a Parquet file — step by step

1
Seek to the last 4 bytes of the file — they tell the engine how large the footer is. Jump back to the footer start and read it. (This is a tiny read, a few KB at most.)
2
Check each row group's column statistics against the query's filters. Any row group where the min/max range makes a match impossible is skipped — zero bytes read from that group.
3
For matching row groups, use the byte offsets from the footer to jump directly to only the column chunks needed by the query — skipping all other columns.
4
Within each column chunk, read the dictionary page first (to build the decode map), then read only the data pages needed.
5
Decode and decompress the data (reverse the encoding and compression from write time) and return the values to the query engine.

04. Zones in a Data Lake: Bronze, Silver, Gold

Real-world Data Lakes divide storage into zones (also called layers). The most popular pattern is the Medallion Architecture:

  • Bronze — raw landing zone. Data arrives here exactly as it came from the source, even if it's broken or messy. Bronze is append-only; nothing is ever changed or deleted here. If something goes wrong in later processing, you can always re-run everything from Bronze. Think of it as your permanent backup of the original data.

  • Silver — cleaned zone. This is where data engineers spend most of their time. Typical Silver transformations include:

    • Type casting — e.g. the source sends order_date as the string "2026-06-01", but you need an actual DATE type for date arithmetic. Silver converts it.
    • Deduplication — your API retried a failed request and the same event arrived twice. Silver keeps only one copy, usually based on a unique ID and timestamp.
    • Null handling — replace missing values with defaults, or flag rows where critical fields are missing.
    • Joins across sources — combine the orders table (from your database) with the customers table (from your CRM) so downstream consumers don't have to. Silver is where you assemble a single clean view from many raw sources.
    • Renaming and standardising — one source calls it cust_id, another calls it customer_id. Silver picks one name and sticks to it everywhere.
  • Gold — business zone. Aggregations, KPIs, and joined business metrics that power dashboards and reports. Gold tables are small (one row per day, not one row per event) and fast to query. Business analysts and BI tools (Tableau, Power BI, Looker) read from Gold, not Silver or Bronze.

💡 Why keep Bronze if Silver already cleaned everything?

Bronze is your safety net. If a Silver job had a bug that silently dropped valid rows, you can fix the job and re-process from Bronze. If Bronze was overwritten with the same bug, that data is gone forever. Disk space on S3 is cheap enough that keeping the raw copy is almost always worth it.

05. The 6 big problems with a plain Data Lake

A plain Data Lake — just files in S3 with no extra software on top — works fine as cheap storage. But the moment you start using it as your primary, reliable source of truth for business data, you will hit these six problems. Almost every data team discovers them the hard way.

❌ Problem 1 — Broken files from crashed jobs (no atomic writes)

Imagine a Spark job that rewrites the orders table. It needs to write 50 new Parquet files to S3. It writes 30 files successfully, then the cluster runs out of memory and crashes. Now you have 30 new files and 20 old files mixed together. Any query that runs on this table reads a broken mix of new and old data.

In a real database, this would never happen because databases use transactions: all 50 writes succeed together, or none of them are visible to anyone — the database rolls back automatically on failure. S3 has no such concept. Each file write is an independent operation. There is no rollback, no undo. You have to manually identify which files are broken, delete them, and re-run the job — hoping no downstream pipeline already read the corrupted data.

❌ Problem 2 — Schema drift (columns change silently)

Your orders API is updated. The old API sent first_name and last_name as separate fields. The new API sends one combined customer_name field. Both old and new Parquet files land in the same S3 folder. Old files have two columns; new files have one. Nobody is notified automatically.

Now SELECT first_name FROM orders returns data from old files and NULLs from new ones. Your customer name report is silently half-empty. This problem is discovered three weeks later when a manager notices the data looks wrong. In a database, any write that doesn't match the table's defined schema would have been rejected immediately with a clear error message. In a plain Data Lake, any file with any shape can land without complaint.

❌ Problem 3 — No row-level updates or deletes

Parquet files on object storage are immutable — once written, they cannot be edited. You cannot open a Parquet file and change a single value, the same way you cannot "edit" a file that's already been zipped and uploaded to S3. The file is fixed.

This means: when a customer sends a GDPR deletion request ("delete all my data"), you cannot run DELETE FROM orders WHERE customer_id = 123. Instead, you must: read every Parquet file in the entire table into memory, filter out the rows for customer 123, then write the entire table back to S3 from scratch. On a 5 TB table, that's reading and writing 5 TB of data to change a handful of rows. In a database, this is one sub-millisecond operation.

❌ Problem 4 — Thousands of tiny files (the "small file problem")

Streaming pipelines write data constantly — a new Parquet file every minute, or every few seconds. Each file is tiny: maybe 50 KB. After one week, you have 100,000 tiny files instead of 100 right-sized ones (50 MB each).

Every file requires its own overhead: open a network connection to S3, authenticate, seek to the footer, parse the statistics, then close the connection. With 100 files, you do this 100 times. With 100,000 files, you do it 100,000 times — even if each file only has 10 rows that match your query. The majority of query time is spent on this file-opening overhead, not on actually reading data. A query that should take 10 seconds takes 40 minutes.

❌ Problem 5 — No history or "undo"

A Silver job has a bug that silently drops 20% of orders during a JOIN operation. The bad data runs through Gold tables and into dashboards. You fix the bug the next morning — but what was the Silver table supposed to look like before the bad run? In a plain Data Lake, once the old Parquet files are overwritten or deleted, they're gone permanently.

In a database with transaction history or point-in-time recovery, you would restore to yesterday's state in minutes. In a plain Data Lake, your only option is to re-run the entire pipeline from Bronze — which can take hours or days if there's a lot of data.

❌ Problem 6 — Two jobs writing at the same time (concurrent writes)

When would two jobs write to the same table at once? More often than you'd think. A nightly batch job runs at midnight. A data science cleanup job also runs at midnight. Or a streaming job is writing new events while a batch job is updating old ones. Or two engineers trigger re-runs at the same time.

Here's exactly what goes wrong. Job A reads the table and sees files [F1, F2]. It processes the data, removes F2, and creates new file F3. Job B also read [F1, F2] at the same time. It removes F1 and creates new file F4. Job A finishes first and writes its result: "F2 is gone, F3 is here." Now Job B finishes and writes its view: "F1 is gone, F4 is here." Job B's write overwrites the record of what Job A did. F3 — all of Job A's work — is no longer referenced anywhere. It's silently lost. The table now appears to have only [F2, F4], which is a mix of states that never existed at the same time.

06. Enter the Data Lakehouse

The Data Lakehouse is the answer to all six problems above. The key idea: keep the cheap cloud storage (S3 / ADLS / GCS) as the foundation, but add a thin software layer on top that gives you database-style reliability.

That software layer is Delta Lake (open source, from Databricks, now a Linux Foundation project). It's not a separate database server, a cloud service you pay for, or a different file format. It's an open-source library that sits between your query engine (Spark, Databricks, Trino) and your Parquet files, adding a transaction log that makes everything safe.

07. How Delta Lake fixes each problem

Delta Lake adds one thing to your Parquet files: a transaction log stored in a folder called _delta_log/. This folder is automatically created by Delta Lake the first time you write a Delta table — it lives in the same S3 prefix as your Parquet files. Everything else — ACID, time travel, safe deletes, safe concurrent writes — comes from this one idea.

What is the transaction log?

The transaction log is a sequence of numbered JSON files inside _delta_log/. Each JSON file is called a commit and records one change to the table: "these Parquet files were added, those Parquet files were removed." To know what the table looks like right now, Delta reads the log commits in order and keeps track of which files are currently "active."

It's like a bank ledger: your account balance isn't stored as a single number — it's computed from the full history of deposits and withdrawals. The current table state is computed from the full history of commits.

What about 10,000 commits? Does Delta replay all of them every query?

No. Every 10 commits, Delta automatically writes a checkpoint — a single Parquet file (not JSON) that summarises the current table state: the full list of active files. Reading the table means loading the latest checkpoint (skipping thousands of old JSON commits) and then replaying only the handful of commits that came after the checkpoint. It's like a save point in a video game: you don't replay from the very beginning, you load the most recent save and continue from there.

Fix 1 — Atomic writes (no more half-written tables)

Delta's write process has two steps:

  1. Write the new Parquet files to S3. But don't tell the log about them yet. Delta Lake readers only track files that appear in a commit. Since these new files aren't mentioned in any commit, they're invisible to readers — even though they physically exist on S3. It's not that S3 is hiding them; it's that no reader will look at them until a commit references them.
  2. Write a single new commit JSON file to _delta_log/. This is the switch — it lists all the new files as "added." From this moment, they're visible to readers.

If the job crashes between step 1 and step 2, the new files sit on S3 but are never referenced by any commit. Readers still see the previous version. There are no broken partial states. The table is always consistent, regardless of when you read it.

Fix 2 — Schema enforcement (no more silent schema drift)

Delta stores the table's schema inside the transaction log. When a new batch of data arrives, Delta checks it against the stored schema before writing any files. If the incoming data has a column with the wrong type, or is missing an expected column, the write is rejected immediately with a clear error. The table stays clean. Nothing bad lands silently.

PySpark · schema enforcement
# This works — schema matches the Delta table
df_good = spark.createDataFrame(
    [(1, "Alice", 99.0)], ["order_id", "name", "amount"]
)
df_good.write.format("delta").mode("append").save("/path/to/orders")

# This FAILS — 'amount' is a string here but schema says DOUBLE
df_bad = spark.createDataFrame(
    [(2, "Bob", "not-a-number")], ["order_id", "name", "amount"]
)
df_bad.write.format("delta").mode("append").save("/path/to/orders")
# AnalysisException: Failed to merge incompatible data types

Fix 3 — Efficient row-level updates and deletes

Delta supports UPDATE, DELETE, and MERGE (upsert). Under the hood, it first uses the column statistics recorded in the transaction log to find which Parquet files actually contain the target rows — it doesn't scan every file blindly. (These statistics — min value, max value, null count per column — are captured from each Parquet file's footer at write time and stored inside the commit JSON. At delete/update time, Delta reads the log to do this lookup, not the Parquet files themselves.) Then it uses one of two strategies to apply the change:

  • Copy-on-Write (CoW) — the default. Delta reads only the files that contain the target rows, rewrites those files without the deleted/changed rows, and logs the old files as removed and the new files as added. All other files are untouched. Reads remain fast because there's no extra processing needed. Writes are slower if the affected files are large — but they only touch the files that actually contain the target rows, not the whole table.
  • Merge-on-Read via Deletion Vectors — the faster alternative for deletes. Instead of rewriting a Parquet file, Delta writes a tiny deletion vector file alongside it — a bitmap (a compact list of bit flags, one per row in the file) where a "1" means "skip this row." Deleting row 42 out of a 1-million-row file doesn't require rewriting all 1 million rows — it just flips bit 42 to 1 in a tiny bitmap file (kilobytes, instead of rewriting the whole multi-hundred-MB Parquet file). When you later read that file, Delta combines the Parquet data with the deletion vector and skips the flagged rows automatically. Deleted rows still sit physically on S3 until you run VACUUM, which cleans up old files that are no longer referenced by any active version of the table.
SQL · update, delete, and upsert
-- Update one customer's email
UPDATE orders SET email = 'new@email.com' WHERE customer_id = 123;

-- GDPR delete: fast even on a 5 TB table
DELETE FROM orders WHERE customer_id = 123;

-- Upsert: update the row if it exists, insert if it doesn't
MERGE INTO orders AS target
USING new_orders AS source
ON target.order_id = source.order_id
WHEN MATCHED THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *;

-- Clean up physically deleted data from S3
VACUUM orders RETAIN 168 HOURS;  -- keep 7 days of history, delete the rest

Fix 4 — Solving the small file problem with OPTIMIZE

Delta has a single command — OPTIMIZE — that reads many small Parquet files and combines them into a few right-sized ones (typically 128–256 MB each). The swap is logged atomically: old small files are marked as removed, new large files are added — all in one commit. If OPTIMIZE is interrupted, the table stays consistent.

You can add ZORDER BY to sort the data at the same time. Z-Order is a technique that co-locates rows with similar values in multiple columns into the same files. Unlike a simple ORDER BY (date) which sorts only by one column, Z-Order interleaves multiple sort dimensions — so rows close in both order_date and category end up in the same Parquet files. The result: tighter min/max statistics per row group, so more row groups can be skipped when queries filter on both columns. Think of it as "sorting by multiple columns simultaneously rather than one at a time."

SQL · compact and sort
-- Combine small files into right-sized ones
OPTIMIZE orders;

-- Compact AND sort by date+category so future queries skip more row groups
OPTIMIZE orders ZORDER BY (order_date, category);

Fix 5 — Time travel (query any previous version of the table)

Because the transaction log records every commit ever made to the table, Delta can reconstruct what the table looked like at any past version — instantly, without restoring from a backup. A Silver job's bug corrupted data? Query the version from before the bug ran. A manager says "show me the numbers from last Tuesday"? Query the table as of that timestamp. Need to roll back? One command.

SQL · time travel
-- Read the table as it was at commit version 3
SELECT * FROM orders VERSION AS OF 3;

-- Read the table as it was last Tuesday
SELECT * FROM orders TIMESTAMP AS OF '2026-06-24 09:00:00';

-- Roll back the table to before the bad job ran (version 5)
RESTORE TABLE orders TO VERSION AS OF 5;

-- See the full history of changes
DESCRIBE HISTORY orders;

Fix 6 — Safe concurrent writes (Optimistic Concurrency Control)

Delta uses a protocol called Optimistic Concurrency Control (OCC). Here's exactly how it works when two jobs run at the same time:

  1. Both jobs read the current log and see the table at version 2.
  2. Both jobs write their new Parquet files to S3 (still invisible — not in the log yet).
  3. Both jobs try to write commit 000003.json. Delta needs exactly one job to "win" this race. The mechanism depends on where your data lives:
    • Azure ADLS Gen2 — has native atomic "create-if-not-exists" semantics. Only the first writer succeeds; the second gets an error immediately.
    • Amazon S3 — supports conditional writes (the HTTP If-None-Match header), giving the same guarantee. Delta on AWS relies on this.
    In all cases, the outcome is the same: only one job commits 000003.json, and the other job knows it lost.
  4. The losing job now checks: did the winning job's commit touch the same Parquet files I was going to modify? Delta does this check automatically — no extra code from you.
    • No conflict (the two jobs touched different data, e.g. different partitions): the losing job simply writes its own changes as the next commit 000004.json. Both jobs' changes end up safely in the table.
    • Conflict (both tried to modify the same file): the losing job retries — but it only retries the commit step, not the whole job. It re-reads the updated log, adjusts its file list to account for what the winning job did, and tries again. A 3-hour Spark job does not restart from scratch — only the final write coordination is retried.

No data is lost or doubled, and no special locking mechanism is needed beyond what cloud storage already guarantees.

08. Summary & what to read next

Here's what you've learned:

  1. A Data Lake was invented to store massive amounts of data cheaply in object storage (S3/ADLS), separated from compute. It works great as a raw archive but breaks as a reliable source of truth.
  2. A Data Lake has three layers: storage (Parquet files in S3), metadata (Hive Metastore / Unity Catalog mapping names to paths), and compute (Spark clusters you rent on demand).
  3. Parquet stores data column-by-column in row groups. The footer holds statistics (min, max per column per row group) that let query engines skip irrelevant data entirely — this is what makes analytics on a Data Lake fast when the data is well-organised.
  4. The Medallion Architecture (Bronze → Silver → Gold) protects you from data loss by always keeping the raw original.
  5. A plain Data Lake has 6 major weaknesses: no atomic writes, no schema enforcement, no row-level changes, the small file problem, no history, and unsafe concurrent writes.
  6. The Data Lakehouse (Delta Lake) keeps all the cheap storage, adds a _delta_log/ transaction log, and fixes all 6 problems. Every Delta Lake feature comes from that single log.

Glossary

Every term used in this article, defined in one place. Skim it first if you're new, or jump back here whenever a word feels rushed.

Object storage
A type of cloud file storage (Amazon S3, Azure ADLS, Google GCS) built for storing massive amounts of files cheaply. Unlike a hard drive, you can't edit a file in place — you write new files and delete old ones. Very cheap ($0.02/GB/month), infinitely scalable, but has no built-in query engine or transaction support.
Data Lake
A collection of raw files (CSV, JSON, Parquet, images, logs...) sitting in object storage. No fixed structure required — you store data as-is and figure out its shape when you read it.
Data Warehouse
A structured database optimised for analytics (BigQuery, Snowflake, Redshift). Fast and reliable, but expensive and rigid — data must be in a defined format before it goes in.
Data Lakehouse
A hybrid: cheap lake storage + database-style reliability (ACID, schema, fast queries) layered on top. Delta Lake, Apache Iceberg and Apache Hudi all build a Lakehouse.
Parquet
A popular file format for analytics. Stores data in columns (not rows), compresses well, and includes statistics that let query engines skip irrelevant data without reading it.
Immutable
Cannot be changed after it's written. Parquet files on object storage are immutable — once saved, you can't edit a single cell. To "change" data, you write a new file and delete the old one.
ACID
Four safety promises that databases give you: Atomic (a write fully completes or has no effect — no half-done states), Consistent (rules like "price must be positive" are always enforced), Isolated (readers never see a half-finished write), Durable (once committed, data survives crashes). A plain Data Lake gives you none of these.
Schema
The list of column names and their data types for a table. Like a contract: "this table will always have order_id (integer), customer_name (string), amount (decimal)."
Partition
Splitting data into sub-folders by a column value (e.g. by date). A query filtering on that date only reads the matching sub-folder and skips everything else.
Cluster
A group of virtual machines (rented cloud servers) that work together. When you "spin up a cluster" in Spark or Databricks, you're renting, say, 10 virtual machines and linking them so Spark can split work across all of them. When you're done, you release the machines and stop paying.
Delta Lake
An open-source software library that adds a transaction log on top of Parquet files, giving you ACID, time travel, safe updates/deletes, and schema enforcement — without moving your data out of cheap cloud storage.
ETL / ELT
Moving and transforming data. ETL = transform first, then load. ELT = load raw data first, then transform inside the lake.
Medallion (Bronze → Silver → Gold)
A common pattern: Bronze = raw data, Silver = cleaned data, Gold = business-ready aggregates.
