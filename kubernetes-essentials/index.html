01. What Kubernetes Is and Why It Exists

Kubernetes (often shortened to "K8s" — the 8 stands for the eight letters between the "K" and the "s") is an open-source platform for running, scheduling, and managing containerized applications across a fleet of machines. You hand it your apps packaged as containers, tell it what you want ("run five copies of this web service, keep them healthy, expose them on port 80"), and Kubernetes figures out how and where to make that happen — and keeps it that way.

If you have only ever run docker run on a single laptop or server, think of Kubernetes as the layer that takes you from "one container on one machine" to "hundreds of containers spread across many machines, self-healing, and load-balanced," without you manually deciding which container lands on which box.

The problem Kubernetes solves

Before containers and orchestration, running software in production was painful in predictable ways:

  • "Works on my machine." An app depends on a specific version of a language runtime, system libraries (like libc or libssl), and config. The developer's laptop has one set of versions; production has another. Things break in deployment that worked fine in development.
  • Coupling between apps. Putting several programs on one server means they fight over shared libraries and resources, and teams step on each other.
  • Manual operations. When a machine dies at 3 a.m., someone has to notice, find a healthy machine, and restart the affected apps by hand. Scaling up for a traffic spike is equally manual.

Containers fixed the first two problems. A container image is an immutable bundle of your program plus all its dependencies and config, so it runs the same everywhere (we cover this in depth in the next part). But containers alone don't tell you which machine to run on, what to do when a machine fails, how to roll out a new version safely, or how to give a stable network address to a group of identical containers. That coordination problem is called orchestration, and orchestration is exactly what Kubernetes provides.

Key idea: Docker (and containers in general) solves packaging and running one container. Kubernetes solves running many containers reliably across many machines — scheduling, healing, scaling, networking, and rollouts.

What "orchestration" actually means

A container orchestrator does the boring-but-critical jobs that a human would otherwise do by hand:

  • Scheduling — deciding which node (machine) each container runs on, based on available CPU/memory and other rules.
  • Self-healing — restarting containers that crash, replacing containers on a node that dies, and rescheduling work elsewhere.
  • Scaling — adding or removing copies of an app as load changes.
  • Service discovery & load balancing — giving a stable name/IP to a group of identical containers and spreading traffic across them, even as individual containers come and go.
  • Rollouts & rollbacks — releasing a new version gradually and backing out automatically if it goes wrong.

A very brief history

Kubernetes grew out of Google's internal cluster manager (called Borg), which the company used for over a decade to run its own services. Google open-sourced Kubernetes in 2014 and donated it to the CNCF (Cloud Native Computing Foundation), a vendor-neutral home under the Linux Foundation. That neutral governance is a big part of why Kubernetes "won": every major cloud (AWS, Google, Azure) and on-prem vendor supports it, so your skills and your manifests are portable.

Kubernetes ships roughly three minor releases a year (about one every four months). At the time of writing the current stable line is in the v1.3x range.

2026 Update: Older guides say "latest is ~1.28–1.30" and describe a tighter support window. As of 2026 the latest stable upstream release is v1.36, with v1.36, v1.35, and v1.34 actively maintained (this is the N-2 policy — the three newest minor versions get patches). Each minor is supported about 14 months (12 months of patches plus a 2-month upgrade window). Practical advice: upgrade at least once per release cycle (~every 4 months) to stay in support. Managed services (EKS/GKE/AKS) layer their own, usually longer, support windows on top.

The declarative model — the single most important mental shift

This is the idea that makes Kubernetes "click," so spend a minute on it.

  • Imperative means you give step-by-step commands: "install this, start that, then run this." Nobody records the end state; if something drifts, you have to notice and fix it manually. Running apt-get install foo is imperative.
  • Declarative means you describe the desired end state in a file ("I want 3 replicas of this app, version 2.0, reachable on port 8080"), and the system continuously works to make reality match that description.

In Kubernetes you write the desired state as YAML manifests and submit them with kubectl apply -f file.yaml. Kubernetes stores that desired state and then never stops trying to keep the actual cluster matching it. If a container dies, the desired state still says "3 replicas," so Kubernetes starts a replacement. You didn't write any "restart the container" logic — you just declared the goal. This is the foundation of self-healing, and we'll see the machinery behind it (the reconciliation loop) in Section 3.

Tip: Treat your manifests like source code: keep them in Git, review changes in pull requests, and apply from there. The declarative model means your Git repo is the description of your running system. This is the basis of GitOps (see the Operations section).

Note: "Desired state" just means "what you asked for." "Actual/current state" means "what's really running right now." Kubernetes' whole job is to shrink the gap between the two.

When to use Kubernetes — and when not to

Kubernetes is powerful but not free: it has real operational and conceptual overhead. Reach for it when the problem actually calls for it.

Good fits:

  • Multiple services (microservices) that need to scale independently and talk to each other.
  • Apps that must stay up through machine failures and need automatic restarts/rescheduling.
  • Workloads with variable load that benefit from autoscaling.
  • Teams wanting consistent, portable deploys across dev/test/prod and across clouds.
  • Batch/scheduled jobs alongside long-running services on shared infrastructure.

Common use cases: stateless web/API backends, microservice meshes, CI/CD build runners, data-processing pipelines and batch jobs, and (increasingly, with care) stateful services like databases.

Think twice if:

  • You have a single small app with steady, low traffic — a single VM, a container service, or a platform-as-a-service is simpler and cheaper.
  • Your team has no appetite for the learning curve and no one to operate a cluster (in that case, strongly prefer a managed offering — see Section 3).
  • You need hard real-time guarantees or very specialized hardware/latency behavior that a general orchestrator complicates.
  • The app is inherently single-machine and will never scale out.

Key idea: Kubernetes earns its complexity when you have many moving parts that must be scheduled, healed, scaled, and connected. For one tiny app, it's usually overkill.

02. Containers, Images, and Runtimes

Kubernetes runs containers, so before the cluster machinery you need a solid grip on what a container actually is, how images are built and stored, and what software actually starts the container on a node. If you've used Docker this will feel familiar — but there are important 2026 corrections about Docker's role inside Kubernetes.

Containers vs. virtual machines (and where Docker fits)

A container is an isolated process (or small group of processes) running on a host, packaged with its own filesystem and dependencies. Unlike a virtual machine, a container does not carry its own operating-system kernel — it shares the host's kernel. That makes containers far lighter than VMs: they start in milliseconds, use less memory, and pack more densely onto a machine.

Note: A VM virtualizes hardware and runs a whole guest OS on top. A container virtualizes the operating system — many containers share one kernel but each gets its own private view of the filesystem, processes, and network. That's why containers are so much smaller and faster.

Docker is the tool that made containers mainstream by wrapping the underlying Linux features in a friendly CLI (docker build, docker run, docker push). Crucially, Docker is not the only way to run a container, and inside Kubernetes it is no longer the runtime (more on that below). Docker remains a perfectly good tool for building images on your laptop or in CI.

The Linux primitives underneath: namespaces and cgroups

Containers aren't magic — they're built from two long-standing Linux kernel features:

  • Namespaces control what a process can see: its own view of the filesystem mounts, process list (PID), network interfaces, hostname, and inter-process communication. This is the isolation.
  • Control groups (cgroups) control how much a process can use: CPU, memory, and I/O limits. This is the resource accounting and limiting.

Both have been in the Linux kernel since ~2008. Containers became popular much later because Docker made them easy — and because microservices, the public cloud, and the need to deploy fast made consistent packaging valuable.

Key idea: A container = a process wrapped in namespaces (isolation: what it can see) + cgroups (limits: what it can use), running from an immutable image. When you set CPU/memory requests and limits on a Kubernetes Pod (see the Pods section), those map straight down to cgroups.

Container images and how layering works

A container image is an immutable binary package containing everything needed to run a program: the root filesystem (libraries, binaries, files) plus metadata describing how to start it (the command, environment variables, ports, user). You build an image once and run it identically anywhere.

Images are built from layers stacked on top of each other using an overlay filesystem. Each instruction in a build adds or modifies a layer, and each layer references its parent, forming a chain. For example:

  • Layer A: a base OS (e.g. Debian)
  • Layer B: adds a language runtime on top of A
  • Layer C: adds your application code on top of B

Layering matters for two practical reasons:

  1. Sharing & caching. Many images can share the same base layers, so they're downloaded and stored once. A rebuild only re-creates layers that changed.
  2. Order affects rebuild speed and size. Put rarely-changing things (dependency installs) before frequently-changing things (your source code), so the expensive layers stay cached.

Security note: Layers are additive — deleting a file in a later layer does not remove it from the earlier layer where it was added. So never bake secrets (passwords, keys) into any layer; they remain recoverable. Use Kubernetes Secrets or an external secrets manager (see the Security section). Also keep images minimal to shrink the attack surface, and scan them (e.g. Trivy).

OCI: the standards that make images and runtimes portable

Early on, "Docker image" and "Docker runtime" were the de-facto standards. To prevent lock-in, the industry formed the OCI (Open Container Initiative) in 2015 to define vendor-neutral specs. Two are essential to know:

  • OCI Image Specification — defines the on-disk/on-registry format of an image: a manifest that points to the config and the filesystem layers, each layer a TAR archive identified by a SHA256 digest, plus image indexes for multi-platform images (e.g. amd64 + arm64). This is why an image built by Docker can be run by a completely different runtime.
  • OCI Runtime Specification — defines how to actually instantiate and run a container from an unpacked image: the config schema (root filesystem, command, env vars, user, resource limits) and the lifecycle (create → created → running → stopped). The reference low-level implementation is runc, which Docker, containerd, and CRI-O all use under the hood.

Key idea: Because images and runtimes follow OCI specs, the image format is universal. An image you build with Docker runs unchanged on containerd or CRI-O. Switching the runtime in your cluster does not require rebuilding your images.

How Kubernetes talks to a runtime: the CRI

Kubernetes does not start containers itself. On each node there's an agent called the kubelet — the program that actually starts and stops your containers on that machine (we cover it fully in Section 3). The kubelet doesn't run containers directly either; it delegates to a container runtime through a standard contract called the CRI (Container Runtime Interface).

Note: The kubelet is just "the agent on every node that starts and stops containers." When the rest of this section talks about the CRI, picture the kubelet on a node asking the runtime to do the actual work.

The CRI is a gRPC API (introduced in Kubernetes v1.5) that decouples the kubelet from any specific runtime. It defines two services:

  • RuntimeService — Pod and container lifecycle: create the Pod sandbox, create/start/stop/delete containers, plus exec, attach, and portforward.
  • ImageService — pull, list, and remove images.

A simplified "start a Pod" flow over CRI:

  1. The Pod is scheduled to a node; the kubelet calls RunPodSandbox to create a sandbox with a shared Linux network namespace (this is why all containers in one Pod share an IP — see the Pods section).
  2. The kubelet checks for the image (ImageStatus) and pulls it if missing (PullImage).
  3. It calls CreateContainer, then StartContainer for each container.

containerd, CRI-O, and the removal of dockershim

In the early days the only supported runtime was Docker. But Docker Engine doesn't implement the CRI directly, so the kubelet shipped with an adapter called dockershim to translate CRI calls into Docker API calls. (Amusingly, Docker itself used containerd internally anyway, so calls went kubelet → dockershim → Docker → containerd → container — an extra hop for no benefit.)

2026 Update: dockershim was REMOVED in Kubernetes v1.24 (April 2022) — this is long done, not "coming soon" as older notes imply. Since v1.24 you must use a CRI runtime directly. The two standard choices are:

  • containerd — the de-facto default, used on all major clouds (default on EKS node AMIs since v1.24).
  • CRI-O — purpose-built for Kubernetes, the choice in Red Hat OpenShift and high-pod-churn environments.

Both use runc and consume OCI images, so your images are unaffected. From v1.26 only the CRI v1 API is supported (so containerd must be ≥ 1.6). If you really need Docker Engine as the runtime you must install the separate cri-dockerd adapter — but there's rarely a reason to. **Docker is still great for building images; it's just not the cluster runtime.**

Note: People still loosely say "Docker container" — that's fine, because the image format is OCI/Docker-compatible. What changed is the runtime that runs it inside a cluster: that's now containerd or CRI-O, reached through the CRI, with no dockershim and no Docker daemon on the node.

Registries and image references

A locally built image only exists on one machine. A Kubernetes cluster needs every node to be able to pull the image, so images live in a registry — a central store you push to and nodes pull from.

  • Public registries (e.g. Docker Hub) allow anonymous pulls — good for open-source images.
  • Private registries (Amazon ECR, Google Artifact Registry, Azure ACR, or self-hosted) require authentication — for proprietary code.

An image reference looks like registry/repository:tag, for example myregistry.example.com/team/api:1.4.2. The workflow is docker login → docker tag → docker push; nodes then pull it when a Pod references it. For private registries, you create an image-pull secret and reference it in the Pod spec (imagePullSecrets).

Tip: In production, reference images by digest (@sha256:…), not by a mutable tag like :latest. Tags can be overwritten, so :latest is not reproducible and is a supply-chain risk. Pinning the digest guarantees every node runs exactly the bytes you tested (see the Security section on supply-chain hardening).

Multistage builds: small, secure images

A naive Dockerfile builds and runs in one image, leaving compilers, build tools, and source code in the final artifact — bloated (hundreds of MB) and a bigger attack surface. A multistage build splits the work: an early stage does the heavy build, and a final, minimal stage copies only the finished artifact.

# STAGE 1: build (has compilers, source, dev tools)
FROM golang:1.23 AS build
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN go build -o /myapp ./cmd/server

# STAGE 2: runtime (tiny, no build tooling)
FROM gcr.io/distroless/base-debian12
COPY --from=build /myapp /myapp
USER nonroot:nonroot
ENTRYPOINT ["/myapp"]

The final image contains just the compiled binary on a minimal ("distroless") base — often tens of MB instead of hundreds. Benefits: faster pulls and deploys, less storage/bandwidth, and a much smaller attack surface (fewer libraries and no shell for an attacker to abuse).

Try it: Build both a single-stage and a multistage version of a small app and compare sizes: ``bash docker build -t myapp:fat -f Dockerfile.single . docker build -t myapp:slim -f Dockerfile.multistage . docker images | grep myapp ``

03. Cluster Architecture and the Reconciliation Loop

Now that you know what containers and images are, let's see the machine that runs them at scale. A Kubernetes cluster is a set of machines (nodes) split into two roles: the control plane (the brain that makes decisions and stores cluster state) and the worker nodes (the muscle that actually runs your containers). You talk to the cluster through one front door — the API server — using kubectl or any client.

The control plane

The control plane makes global decisions about the cluster (scheduling Pods, reacting to failures, provisioning cloud load balancers) and holds the source of truth. Its components:

  • API server (kube-apiserver) — the central hub and the only component that talks to the datastore. Every interaction — from kubectl, from other control-plane components, from the kubelet on each node — goes through the API server's RESTful API. It validates requests and reads/writes objects. If the API server is down, you can't change anything in the cluster.
  • etcd — a consistent, distributed key-value store that holds all cluster state and configuration: every object's desired and current status. It's the cluster's database. It typically runs as a multi-member cluster for high availability (quorum), and you must back it up — losing etcd means losing the cluster's state.
  • Scheduler (kube-scheduler) — watches for newly created Pods that have no node assigned, and picks the best node for each. It works in three steps: Filter (eliminate nodes that can't fit the Pod — not enough CPU/memory, failing taints/affinity rules), then Score (rank the feasible nodes), then Bind the Pod to the highest-scoring node. (Affinity, taints, and topology spread, which steer these decisions, are covered in the Scheduling section.)
  • Controller manager (kube-controller-manager) — a single binary bundling many controllers (the Deployment controller, ReplicaSet controller, Node controller, Job controller, and more). Each controller runs a reconciliation loop (below) to drive some part of the cluster toward its desired state.
  • Cloud controller manager (cloud-controller-manager) — the controllers that integrate with a specific cloud provider: creating cloud load balancers for LoadBalancer Services, attaching cloud disks, and managing node lifecycle against the cloud's API. Splitting these out lets cloud vendors ship and update their integration independently of core Kubernetes.

2026 Update: Modern clusters use the out-of-tree cloud-controller-manager (the provider's code lives in its own repo and is deployed separately), and storage runs through CSI drivers rather than the old built-in ("in-tree") cloud/volume plugins, which have been removed. If a guide shows in-tree volume plugins like awsElasticBlockStore baked into the core, that's outdated — use the provider's CSI driver instead (see the Storage section).

Key idea: The API server is the single front door; etcd is the source of truth. Every other component reads desired state from, and writes status back to, the API server — they never talk to etcd directly or to each other point-to-point.

What happens when you submit a request: the API request flow

Every change — kubectl apply, a controller creating a Pod, anything — flows through the API server in a fixed pipeline before it's persisted:

  1. Authentication — Who are you? The API server verifies your identity (client certificate, token, OIDC, etc.).
  2. Authorization — Are you allowed to do this? Checked via RBAC (Role-Based Access Control) — does this user/ServiceAccount have permission for this verb (get/create/delete) on this resource? (Details in the Security section.)
  3. Admission control — Is the request well-formed and policy-compliant? Mutating admission can modify the object (e.g. inject defaults or a sidecar), then validating admission can accept or reject it (e.g. enforce a policy).
  4. Persisted to etcd — only now is the object stored as the new desired state.

2026 Update: The old PodSecurityPolicy (PSP) admission mechanism was removed in v1.25. Pod security is now enforced by Pod Security Admission (PSA) using namespace labels at three levels (privileged / baseline / restricted). For richer rules, in-process ValidatingAdmissionPolicy using CEL expressions (GA in v1.30) or tools like Kyverno/OPA Gatekeeper are used. Don't write PSP examples (see the Security section).

The worker nodes

Worker nodes are where your containers actually run. Each node runs three things:

  • kubelet — the node's agent. It registers the node with the API server, watches for Pods assigned to its node, and tells the container runtime (over the CRI) to pull images and start/stop containers. It also runs the liveness/readiness/startup probes and reports Pod and node status back to the API server. The kubelet is the bridge between the control plane's decisions and the actual containers.
  • Container runtime — containerd or CRI-O (reached via the CRI, as covered in Section 2). This is what actually creates the containers using namespaces and cgroups.
  • kube-proxy — implements the Service abstraction on each node. It watches Services and their endpoints and programs the node's networking (iptables or IPVS rules) so a Service's stable virtual IP load-balances to the right backend Pods. It typically runs as a DaemonSet (one copy per node).

Note: A handy way to remember it: the control plane decides, the kubelet does. The scheduler picks a node; the kubelet on that node makes it real by driving the runtime.

2026 Update: Routing to Pods is tracked via EndpointSlices, not the older monolithic Endpoints object (the Endpoints API is deprecated as of v1.33). Many modern clusters also use eBPF-based CNIs (e.g. Cilium) that can replace kube-proxy entirely — see the Networking section.

The reconciliation loop — the heartbeat of Kubernetes

Recall the declarative model from Section 1: you declare desired state, and Kubernetes makes reality match. The mechanism that does this is the reconciliation loop (also called the control loop), and essentially every controller runs one. It repeats, forever:

  1. Observe the current state (via the API server).
  2. Compare it to the desired state (also stored via the API server, in etcd).
  3. Act to close the gap — create, update, or delete resources.
  4. Repeat.

Concretely: you create a Deployment that says "3 replicas." The ReplicaSet controller observes that only 2 Pods exist, sees the gap, and creates 1 more. If a node dies and a Pod vanishes, the next loop observes "2 of 3" again and creates a replacement — automatic self-healing, with no special recovery code from you. The same pattern drives rollouts, scaling, Jobs, DaemonSets, and custom resources (the Operator pattern in the Operations section is just "your own controller running its own reconcile loop").

Key idea: Kubernetes is level-triggered, not edge-triggered: controllers don't react to one-off events and forget — they continuously re-check "desired vs. actual" and converge. That's why Kubernetes keeps working even after restarts, missed events, or partial failures.

Note: "Level-triggered, not edge-triggered" in plain terms: Kubernetes doesn't rely on catching a single event the moment it happens — it keeps re-checking the current state, so it still fixes things even if it missed the original event.

The standard interfaces: CRI, CNI, CSI (and friends)

A huge reason Kubernetes is so flexible is that the hard, environment-specific jobs are pluggable behind standard interfaces. Kubernetes defines the contract; you plug in an implementation that suits your environment.

Key idea: "Pluggable interfaces" is the Kubernetes design philosophy. The core stays small and stable; runtimes, networking, and storage are swappable plugins. That's why the same Kubernetes runs identically on your laptop, on bare metal, and on every cloud.

Managed vs. self-managed clusters

You don't have to run the control plane yourself. There are two broad models:

  • Self-managed ("roll your own"). You install and operate everything — API server, etcd, scheduler, controller managers — typically with kubeadm. You get full control and avoid vendor restrictions, but you own backups, upgrades, availability, and 3 a.m. pages. For high availability you run multiple control-plane replicas behind a load balancer.
  • Managed. A cloud provider runs the control plane for you and you attach worker nodes. Amazon EKS, Google GKE, and Azure AKS are the big three. The control plane internals are hidden; you mostly manage your workloads and (optionally) the nodes. This dramatically lowers the operational burden and is the right default for most teams — especially smaller ones.

Tip: With managed Kubernetes, most of Section 3's control-plane components are hidden from you — you can't SSH to the API server or etcd. That's a feature: the provider handles their availability, patching, and backups. You focus on Deployments, Services, and the workloads. (AWS EKS specifics — Auto Mode, Karpenter, Pod Identity vs. IRSA — are in the EKS section.)

Note: Even tools like Minikube and kind (for local dev) are "clusters" — they just collapse the control plane and a worker node onto a single machine so you can learn without a cloud bill.

Recap

  • Kubernetes orchestrates containers across many machines: it schedules, heals, scales, networks, and rolls out apps so you don't do it by hand. Use it when you have many moving parts; skip it for one small steady app.
  • The declarative model is the core mental shift: you declare desired state in YAML, and Kubernetes continuously reconciles reality to match it — that's where self-healing comes from.
  • Images and runtimes are OCI-standardized and portable. Docker is for building images; inside a cluster the runtime is containerd or CRI-O via the CRI — dockershim was removed in v1.24, so your images are unaffected but Docker is no longer the node runtime.
  • A container = a process in namespaces (isolation) + cgroups (limits), run from an immutable, layered image. Use multistage builds and pin images by digest.
  • Cluster = control plane (API server + etcd + scheduler + controller managers) + worker nodes (kubelet + kube-proxy + runtime). The API server is the only front door; etcd is the source of truth; every request passes authn → authz (RBAC) → admission → etcd.
  • The reconciliation loop (observe → compare → act, forever) is what every controller runs, and CRI/CNI/CSI are the pluggable interfaces that keep the core small and the platform portable. For most teams, a managed cluster (EKS/GKE/AKS) is the right starting point.

04. Pods: The Smallest Deployable Unit

If you remember only one sentence about Kubernetes object types, make it this: you don't run containers in Kubernetes — you run Pods. A Pod is the smallest thing Kubernetes can schedule, create, and destroy. Everything else in this section (Deployments, Jobs, StatefulSets, and friends) ultimately exists to create and manage Pods for you.

Key idea: A Pod is a group of one or more containers that are always scheduled together onto the same node, share an IP address, and can share storage. Think of it as a "logical host" — the closest Kubernetes equivalent to "a machine running a few tightly-coupled processes".

4.1 What a Pod actually is

When you start a container with Docker, that container gets its own network, its own filesystem, its own process space. A Pod changes that for the containers inside it. Containers in the same Pod share:

  • One network namespace — they all have the same IP address and the same port range. Container A can reach container B at localhost:<port>. (This also means two containers in one Pod can't both bind the same port.)
  • The same hostname (UTS namespace).
  • Optional shared volumes — directories that any container in the Pod can mount and read/write.
  • Optionally, inter-process communication (IPC), so containers can use shared-memory primitives.

Containers in different Pods are fully isolated from each other, even when they happen to land on the same node. They talk over the network like any two separate machines would.

A Pod is also the unit of co-location and atomic scheduling: all containers in a Pod are guaranteed to run on the same node, and they're created and deleted as a single unit. Pods are also ephemeral and immobile — once scheduled to a node a Pod never moves. If the node dies, that exact Pod is gone; a controller (see Section 5) replaces it with a new Pod, with a new IP. Never treat a Pod's IP or name as permanent.

Note: "Namespace" is an overloaded word. Here, Linux namespaces (network, PID, IPC…) are the kernel feature that isolates containers. Later we'll meet Kubernetes Namespaces (Section 7), which are a totally different thing — a way to partition objects in the cluster. Same word, unrelated concepts.

4.2 Single-container vs multi-container Pods

The overwhelmingly common case is one container per Pod. Reach for multiple containers only when the containers are genuinely symbiotic — when they would break if forced onto different machines. The litmus-test question:

"Would these containers still work correctly if they ran on different machines?" If yes → use separate Pods. If no (they share a volume, or talk over localhost, or one cannot live without the other) → one Pod.

A classic anti-pattern is stuffing a WordPress front-end and its MySQL database into one Pod. They communicate fine over the network, and they scale differently (web horizontally, database vertically), so they belong in separate Pods.

The legitimate multi-container patterns are:

PatternWhat the helper container doesExample
SidecarAugments the main containerLog shipper; service-mesh proxy; metrics exporter
AmbassadorProxies the main container's outbound connections to the outside worldA local proxy that fronts a sharded database
AdapterReshapes the main container's output for the outside worldNormalizing log/metric formats

4.3 Init containers

Init containers are containers that run to completion, one after another, before the Pod's main (app) containers start. If an init container fails, Kubernetes restarts it (subject to the Pod's restart policy) and won't start the app containers until every init container has succeeded. Use them for one-off setup: waiting on a dependency, running a database migration, fetching config, or pre-populating a shared volume.

apiVersion: v1
kind: Pod
metadata:
  name: app-with-init
spec:
  initContainers:
  - name: wait-for-db
    image: busybox:1.36
    command: ['sh', '-c', 'until nc -z db 5432; do echo waiting; sleep 2; done']
  containers:
  - name: app
    image: my-app:1.0

4.4 Native sidecar containers (the modern way)

For years, a "sidecar" was just a second regular container in the Pod. That worked, but it had two long-standing pain points: the sidecar didn't reliably start before the app, and on a Job (Section 5) a never-exiting sidecar (like a log shipper) would keep the Pod from ever completing.

2026 Update: Native sidecar containers went GA (Generally Available) in Kubernetes v1.33. A native sidecar is declared as an init container with restartPolicy: Always. Kubernetes then guarantees it: (1) starts before the main containers, (2) stays running for the whole life of the Pod (unlike a normal init container, which runs once and exits), and (3) is terminated after the app containers during shutdown — and crucially, a native sidecar does not block a Job from completing. Prefer this over the old "just add another container" approach for any helper that must outlive startup but shouldn't keep the Pod alive on its own.

Note: This sounds like a contradiction with §4.3, but it isn't. Yes, a native sidecar lives in the initContainers: list, but setting restartPolicy: Always changes its behaviour so it starts before app containers AND keeps running alongside them — Kubernetes reused the init mechanism rather than adding a new field.

apiVersion: v1
kind: Pod
metadata:
  name: app-with-sidecar
spec:
  initContainers:
  - name: log-shipper          # a NATIVE sidecar...
    image: fluent/fluent-bit:3.1
    restartPolicy: Always      # ...because of this line
  containers:
  - name: app
    image: my-app:1.0

4.5 Pod lifecycle and phases

A Pod moves through a small set of phases (status.phase):

PhaseMeaning
PendingAccepted by the cluster, but not yet running — usually waiting to be scheduled or pulling images
RunningBound to a node; at least one container is running (or starting/restarting)
SucceededAll containers exited with code 0 and won't restart (typical for Jobs)
FailedAll containers terminated and at least one failed (non-zero exit)
UnknownThe node can't be reached to report status

The restartPolicy (set on the Pod) governs what happens when a container inside the Pod exits:

  • Always (default) — restart the container whenever it stops. Used by long-running services.
  • OnFailure — restart only on non-zero exit. Typical for Jobs.
  • Never — never restart; create a fresh Pod instead if needed.

Note: When you see a Pod stuck in CrashLoopBackOff, that's not a phase — it's a container state. The container keeps crashing, so Kubernetes restarts it with an ever-increasing back-off delay (10s, 20s, 40s…). kubectl logs <pod> --previous shows the logs of the crashed instance.

4.6 Health checks: liveness, readiness, and startup probes

Kubernetes restarts the container inside a Pod if its main process dies — but a process can be alive yet useless (deadlocked, overloaded, still warming a cache). Probes let Kubernetes detect those subtler states. There are three, each answering a different question:

ProbeQuestion it answersOn failure
livenessProbe"Is this container still healthy, or wedged?"Kubernetes restarts the container
readinessProbe"Can this container serve traffic right now?"Pod is removed from Service endpoints (a Service is the load-balancer in front of your Pods — see §9; failing readiness just means "stop sending me traffic for now"), but not restarted
startupProbe"Has a slow-starting app finished booting yet?"Holds off liveness/readiness until startup succeeds; once the startup probe passes, the liveness and readiness probes take over

Each probe can use one of three mechanisms:

  • httpGet — HTTP request; success = status code 200–399.
  • tcpSocket — success = the TCP port accepts a connection (good for databases).
  • exec — run a command inside the container; success = exit code 0.

A complete example wiring all three probes:

apiVersion: v1
kind: Pod
metadata:
  name: web
spec:
  containers:
  - name: web
    image: my-web:1.0
    ports:
    - containerPort: 8080
    startupProbe:           # give a slow app up to 30 x 2s = 60s to boot
      httpGet:  { path: /healthz, port: 8080 }
      failureThreshold: 30
      periodSeconds: 2
    livenessProbe:          # restart if it wedges
      httpGet:  { path: /healthz, port: 8080 }
      periodSeconds: 10
      failureThreshold: 3
    readinessProbe:         # stop sending traffic when not ready
      httpGet:  { path: /ready, port: 8080 }
      periodSeconds: 5
      failureThreshold: 3

Common knobs: initialDelaySeconds (wait before the first probe), periodSeconds (interval), timeoutSeconds (max wait for a reply), failureThreshold (consecutive failures before acting), successThreshold (consecutive successes to be "good" again).

Tip: Use separate endpoints for liveness and readiness, and never make a liveness probe depend on an external system (like a downstream database). If you do, one slow dependency can trigger a cluster-wide restart storm — a self-inflicted outage.

4.7 Resource requests, limits, and QoS

Two settings per container tell the scheduler and the kernel how much CPU and memory a container needs:

  • requests — the guaranteed minimum. The scheduler only places the Pod on a node that has at least this much free, and it's reserved for you. A Pod's total request is the sum of its containers' requests.
  • limits — the hard ceiling. The container may burst above its request up to the limit if the node has spare capacity, but no further.
resources:
  requests:
    cpu: "500m"      # 500 millicores = half a CPU core
    memory: "128Mi"
  limits:
    cpu: "1000m"     # 1 full core
    memory: "256Mi"

How the two resources behave differently matters:

  • **CPU is compressible.The key takeaway: when a container hits its CPU limit, it is throttled** (slowed down, not killed). Requests only kick in under contention; idle CPU is free to use. (Under the hood, requests map to proportional cgroup CPU shares, and the limit is enforced by the kernel's CFS — Completely Fair Scheduler — bandwidth control; you rarely need those internals.)
  • **Memory is non-compressible.Exceed the memory limitand the kernel OOMKills** the container (you'll see OOMKilled). Under node memory pressure, Pods using more than their request are candidates for eviction.

These settings also assign each Pod a Quality of Service (QoS) class, which decides who gets evicted first when a node runs out of memory:

QoS classConditionEviction priority
GuaranteedEvery container sets requests = limits for both CPU and memoryEvicted last
BurstableAt least one request set, but not GuaranteedMiddle
BestEffortNo requests or limits at allEvicted first

Tip: For latency-sensitive or production workloads, set requests = limits (Guaranteed). Otherwise at minimum set requests on everything so the scheduler can do its job and your Pods aren't first in line for eviction.

2026 Update: In-place Pod resize is beta and on-by-default since v1.33. You can now change a running container's CPU/memory requests and limits via the resize subresource without restarting the Pod (watch the PodResizePending / PodResizeInProgress conditions). Before this, every resize meant recreating the Pod.

4.8 Ephemeral containers (live debugging)

What do you do when a Pod misbehaves but its image is a distroless one with no shell, no curl, no ps? You can't kubectl exec a shell that isn't there. Ephemeral containers (GA since v1.25) let you attach a temporary debug container — with whatever tools you like — into a running Pod, sharing its namespaces:

kubectl debug -it mypod --image=busybox:1.36 --target=app

Ephemeral containers have no resource guarantees, can't be restarted, and vanish when you're done. They're a debugging aid, never part of a normal spec.

05. Workload Controllers

You can create a bare Pod, but you almost never should. A bare Pod has no safety net: delete it, crash its node, and it's gone forever. Workload controllers are higher-level objects that create and continuously manage Pods for you, using a reconciliation loop: you declare a desired state, and the controller keeps comparing the real world to it and acting to close the gap (see the Architecture section for the control-loop pattern in depth).

Note: Selectors in one minute. Every controller below finds "its" Pods through labels — short key/value tags stamped on each Pod (e.g. app: nginx). A controller's selector.matchLabels says "I own every Pod whose labels match these", and its Pod template stamps exactly those labels onto the Pods it creates. So selector.matchLabels and template.metadata.labels must agree. That's all you need for now — §7 covers labels and selectors in full.

5.1 ReplicaSet — keep N copies running

A ReplicaSet has one job: ensure that exactly replicas copies of a Pod are running at all times. If a Pod dies or its node fails, the ReplicaSet notices the mismatch and creates a replacement — that's self-healing. It finds "its" Pods using a label selector (more on labels in Section 7).

apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:               # the Pod template — what each replica looks like
    metadata:
      labels:
        app: nginx        # MUST be matched by the selector above
    spec:
      containers:
      - name: nginx
        image: nginx:1.26.0

Note: You will almost never create a ReplicaSet directly. You create a Deployment, and it creates ReplicaSets for you. Understanding ReplicaSets still matters, because that's what you'll see when you debug a Deployment.

5.2 Deployment — the everyday workhorse

A Deployment manages ReplicaSets, which manage Pods. Its superpower over a raw ReplicaSet is versioned rollouts and rollbacks: change the Pod template (e.g. bump the image tag) and the Deployment orchestrates a controlled transition from the old version to the new one, with the ability to undo. Use a Deployment for long-running, stateless applications — web servers, APIs, workers. This is your default choice.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 25%          # how many EXTRA pods may exist during rollout
      maxUnavailable: 25%    # how many may be missing during rollout
  minReadySeconds: 10        # a new pod must stay ready this long to "count"
  progressDeadlineSeconds: 600  # mark the rollout failed if no progress in 10 min
  revisionHistoryLimit: 10
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.26.0

Working with rollouts:

kubectl rollout status   deployment/nginx     # watch a rollout progress
kubectl rollout history  deployment/nginx     # list revisions
kubectl rollout undo     deployment/nginx     # roll back to the previous revision
kubectl rollout undo     deployment/nginx --to-revision=3
kubectl rollout pause    deployment/nginx     # pause mid-rollout (e.g. for a canary)
kubectl rollout resume   deployment/nginx

Tip: Add an annotation kubernetes.io/change-cause: "..." to the Pod template so each entry in rollout history carries a human-readable reason.

Rollout strategies

StrategyHow it worksDowntime?Trade-off

RecreateKills all old Pods, then creates new onesYesOnly one version ever runs at once; fine for dev, or when a single RWO volume can't be shared during the switch

Note: Kubernetes' built-in strategy field only offers RollingUpdate and Recreate. Blue/green and canary are patterns you build on top, using multiple Deployments plus traffic routing (Services, or the modern Gateway API — see the Networking section).

5.3 DaemonSet — one Pod per node

A DaemonSet ensures a copy of a Pod runs on every node (or every node matching a nodeSelector/tolerations). When a new node joins the cluster, the DaemonSet automatically places its Pod there. This is the pattern for node-level agents: log collectors (Fluent Bit), monitoring agents, CNI networking plugins, storage daemons. (kube-proxy itself ships as a DaemonSet.)

apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluentd
spec:
  selector:
    matchLabels:
      app: fluentd
  template:
    metadata:
      labels:
        app: fluentd
    spec:
      tolerations:                  # also run on tainted nodes, e.g. control-plane
      - effect: NoSchedule
        operator: Exists
      containers:
      - name: fluentd
        image: fluent/fluentd:v1.16-1
        resources:
          requests: { cpu: 100m, memory: 200Mi }
          limits:   { memory: 200Mi }
        volumeMounts:
        - name: varlog
          mountPath: /var/log
      volumes:
      - name: varlog
        hostPath:
          path: /var/log

Note: DaemonSet Pods aren't placed by the normal scheduler's count logic — there's no replicas field, because the count is simply "however many nodes there are." Tolerations (see the Scheduling section) let a DaemonSet run even on nodes that are tainted to repel ordinary Pods.

5.4 StatefulSet — stable identity for stateful apps

Deployments treat their Pods as interchangeable cattle: random names, no ordering, shared-or-no storage. That's wrong for databases, message brokers, and anything that forms a cluster where each member has a distinct identity. A StatefulSet gives each Pod:

  1. A stable, ordinal name — mongo-0, mongo-1, mongo-2 — that survives reschedules (a Deployment gives random suffixes like nginx-7d8f…).
  2. A stable network identity via a headless Service (a headless Service has no load-balancing IP — clusterIP: None — and instead gives each Pod its own stable DNS name; see §9): each Pod gets a predictable DNS name, mongo-0.mongo.default.svc.cluster.local, following the pattern $(pod-name).$(service-name).$(namespace).svc.cluster.local.
  3. Its own persistent storage through volumeClaimTemplates: Kubernetes creates a separate PersistentVolumeClaim per Pod (data-mongo-0, data-mongo-1…) and re-attaches the same volume to the same ordinal across restarts. (See the Storage section for PVCs.)
  4. Ordered, predictable lifecycle: Pods are created 0 → 1 → 2 (each waits for the previous to be Ready) and removed in reverse 2 → 1 → 0. (podManagementPolicy: Parallel opts out of the ordering if your app doesn't need it.)
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mongo
spec:
  serviceName: mongo          # must reference a headless Service
  replicas: 3
  selector:
    matchLabels:
      app: mongo
  template:
    metadata:
      labels:
        app: mongo
    spec:
      containers:
      - name: mongo
        image: mongo:7.0
        ports:
        - containerPort: 27017
          name: peer
        volumeMounts:
        - name: data
          mountPath: /data/db
  volumeClaimTemplates:        # one PVC PER pod, auto-created
  - metadata:
      name: data
    spec:
      accessModes: ["ReadWriteOnce"]
      storageClassName: standard
      resources:
        requests:
          storage: 10Gi

The companion headless Service that gives the Pods their DNS names:

apiVersion: v1
kind: Service
metadata:
  name: mongo
spec:
  clusterIP: None             # headless: no virtual IP, returns per-pod DNS records
  selector:
    app: mongo
  ports:
  - port: 27017
    name: peer

5.5 Job — run to completion

A Job runs one or more Pods until a specified number of them complete successfully, then stops. It's for finite, batch work: a database migration, a report, a one-off computation. Unlike a Deployment, a Job is done when the work is done.

  • completions — total number of successful Pod runs required.
  • parallelism — how many Pods may run at once.
  • backoffLimit — how many retries before the Job is marked failed.
  • restartPolicy must be OnFailure or Never (never Always — a Job is meant to end).

Three common shapes: one-shot (completions: 1, parallelism: 1), fixed parallel (completions: N, parallelism: M), and work queue (parallelism: M, Pods drain a shared queue and exit when it's empty).

apiVersion: batch/v1
kind: Job
metadata:
  name: report
spec:
  completions: 10
  parallelism: 5
  backoffLimit: 4
  template:
    spec:
      containers:
      - name: worker
        image: my-batch:1.0
      restartPolicy: OnFailure

2026 Update: Job reliability features graduated in v1.33. You can now set a backoffLimitPerIndex (per-index retry budget for indexed Jobs instead of one global limit), a successPolicy to declare an indexed Job successful once enough indexes finish (without requiring every single one), and a podFailurePolicy to decide — based on exit codes — whether a failure should count against the backoff limit or be ignored/retried. These make large batch Jobs far less brittle.

5.6 CronJob — Jobs on a schedule

A CronJob creates a Job on a recurring schedule, using standard cron syntax. Each fire-off produces a fresh Job (which produces Pods).

apiVersion: batch/v1
kind: CronJob
metadata:
  name: nightly-backup
spec:
  schedule: "0 2 * * *"        # every day at 02:00
  timeZone: "Asia/Ho_Chi_Minh" # interpret the schedule in this TZ (since v1.27)
  concurrencyPolicy: Forbid    # don't start a new run if the previous is still going
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: backup
            image: my-backup:1.0
          restartPolicy: OnFailure

Tip: Set concurrencyPolicy (Allow / Forbid / Replace) and successfulJobsHistoryLimit / failedJobsHistoryLimit so old Job/Pod objects don't pile up in your namespace.

06. Configuration: ConfigMaps, Secrets, and the Downward API

A core principle: build the image once, configure it per environment at runtime. Don't bake config files, hostnames, or passwords into your image — you'd need a different image per environment, and secrets baked into a layer are recoverable forever. Kubernetes gives you three tools to inject configuration into a Pod at deploy time.

6.1 ConfigMaps — non-sensitive configuration

A ConfigMap stores non-confidential configuration as key/value pairs, in two flavours:

  • Property-like — short scalar values (LOG_LEVEL=info), great for environment variables.
  • File-like — a whole file's contents under one key (nginx.conf: | …), great for mounting as a file.
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  LOG_LEVEL: "info"            # property-like
  app.properties: |           # file-like
    name=YourApp
    version=1.0.0

6.2 Secrets — sensitive data (base64 is NOT encryption)

A Secret looks almost identical to a ConfigMap but is intended for sensitive data — passwords, tokens, TLS keys, registry credentials. Being a separate resource type means you can lock it down with its own RBAC rules (see the Security section). You can supply values as plain stringData (Kubernetes encodes them for you) or pre-encoded data.

apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque
stringData:                   # plain text in, Kubernetes base64-encodes it
  username: appuser
  password: "S3cr3t!"

Key idea: Secret values are stored base64-encoded, which is NOT encryption — it's trivially reversible (echo <value> | base64 -d). Anyone who can read the Secret can read its contents. Base64 only exists so Secrets can carry binary data.

2026 Update: Treat the default Secret as "obfuscated, not secure." For real protection you must layer on: encryption at rest for etcd (a KMS provider), tight RBAC on Secret read access, and ideally an external system — the External Secrets Operator (ESO) to sync from AWS Secrets Manager / HashiCorp Vault, or the Secrets Store CSI Driver to mount secrets straight into the Pod's tmpfs and keep them out of etcd entirely.

6.3 Consuming config: environment variables vs mounted files

Both ConfigMaps and Secrets are consumed the same two ways, and the choice has real consequences.

As environment variables — pull individual keys, or inject every key at once with envFrom:

spec:
  containers:
  - name: app
    image: my-app:1.0
    envFrom:
    - configMapRef:
        name: app-config        # injects every key as an env var
    env:
    - name: DB_PASSWORD          # pull one specific key from a Secret
      valueFrom:
        secretKeyRef:
          name: db-credentials
          key: password

As mounted files — the ConfigMap/Secret appears as a directory of files:

spec:
  containers:
  - name: app
    image: my-app:1.0
    volumeMounts:
    - name: config-vol
      mountPath: /etc/app        # each key becomes a file here
  volumes:
  - name: config-vol
    configMap:
      name: app-config
Environment variablesMounted files (volume)
UpdatesFrozen at Pod start — you must restart the Pod to pick up changesUpdated live (after a short delay); the app must re-read the file

6.4 Immutable ConfigMaps and Secrets

For config that should never change after creation, set immutable: true. Kubernetes then rejects any edit (you must delete and recreate), which protects against accidental updates and lets the kubelet stop watching the object — a real performance win in clusters with thousands of them.

apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config-v2
immutable: true
data:
  LOG_LEVEL: "info"

Tip: Pair immutability with versioned names (app-config-v2) and point your Deployment's template at the new name. Because the Pod template changes, this also triggers a clean rollout — a tidy way to roll out config changes deliberately rather than mutating in place.

6.5 The Downward API — a Pod's info about itself

Sometimes an app needs to know things about its own Pod — its name, namespace, node, labels, or its CPU/memory limit — without talking to the Kubernetes API. The Downward API exposes that metadata as environment variables or files:

spec:
  containers:
  - name: app
    image: my-app:1.0
    env:
    - name: POD_NAME
      valueFrom:
        fieldRef:
          fieldPath: metadata.name
    - name: POD_NAMESPACE
      valueFrom:
        fieldRef:
          fieldPath: metadata.namespace
    - name: MEM_LIMIT            # let the app size its heap to its limit
      valueFrom:
        resourceFieldRef:
          containerName: app
          resource: limits.memory

A great real-world use: a JVM reading MEM_LIMIT to set -Xmx so it never gets OOM-killed for exceeding the container's memory limit.

07. Organizing Resources: Labels, Selectors, Annotations, Namespaces

A real cluster has thousands of objects. Kubernetes doesn't organize them in folders — it organizes them with labels and namespaces, and wires objects together with selectors. Getting this mental model right is what makes the rest of Kubernetes click.

7.1 Labels vs annotations

Both are key/value maps in an object's metadata, but they serve opposite purposes:

LabelsAnnotations
PurposeIdentify and group objectsAttach non-identifying metadata
Queryable?Yes — you select on themNo — never used to select
Value sizeShort (≤ 63 chars), restricted charsetFree-form, large (JSON, URLs, git SHAs)
Examplesapp=payments, env=prod, tier=backendkubernetes.io/change-cause, build info, tool config, icon URLs
metadata:
  labels:
    app: payments
    env: prod
    tier: backend
  annotations:
    kubernetes.io/change-cause: "Bump to v2.3.1"
    example.com/contact: "team-payments@vib.com.vn"

Key idea: Labels are for machines to query; annotations are for humans and tools to read. If you ever want to find, count, or operate on a set of objects by some property, that property must be a label.

7.2 Selectors

A selector is a query over labels. Two operator families:

  • Equality: env=prod, tier!=frontend
  • Set-based: env in (prod,staging), tier notin (frontend), app (key exists), !app (key absent)
kubectl get pods --selector="app=payments,env=prod"   # comma = logical AND
kubectl get pods -l "env in (prod,staging)"

In manifests you'll see selectors written two ways. matchLabels is the simple equality form; matchExpressions is the richer set-based form:

selector:
  matchLabels:
    app: payments
  matchExpressions:
  - key: env
    operator: In
    values: [prod, staging]

7.3 How selectors wire a Deployment to a Service

This is the single most important wiring to understand, because it's where beginners get stuck. Pods, Deployments, and Services are loosely coupled — none of them references another by name. They find each other entirely through labels.

  • A Deployment's spec.selector.matchLabels must match the labels it stamps onto its Pods (spec.template.metadata.labels). That's how the Deployment knows which Pods are "its own".
  • A Service's spec.selector matches those same Pod labels. That's how the Service knows which Pods to send traffic to — the Service doesn't know the Deployment exists. It just selects Pods.
# Deployment: stamps app=payments onto its Pods
apiVersion: apps/v1
kind: Deployment
metadata:
  name: payments
spec:
  selector:
    matchLabels:
      app: payments        # (1) Deployment selects its Pods by this label
  template:
    metadata:
      labels:
        app: payments      # (2) ...and stamps it on every Pod it creates
    spec:
      containers:
      - name: payments
        image: payments:2.3.1
        ports:
        - containerPort: 8080
---
# Service: selects the SAME label, sends traffic to those Pods
apiVersion: v1
kind: Service
metadata:
  name: payments
spec:
  selector:
    app: payments          # (3) matches the Pod label from (2)
  ports:
  - port: 80
    targetPort: 8080

Note: The #1 "my Service has no endpoints" bug is a label mismatch between the Service's selector and the Pods' labels. When traffic doesn't flow, run kubectl get endpoints <service> — empty endpoints means the selector matches zero Pods. (Behind the scenes, Kubernetes tracks the matched Pods in EndpointSlices; the older Endpoints API is deprecated as of v1.33 but you'll still see kubectl get endpoints work.)

7.4 Namespaces

A Namespace is a virtual partition of the cluster — a way to scope names and apply policy. Object names must be unique within a namespace, not across the whole cluster, so team-a and team-b can each have a web Deployment. Namespaces are the natural boundary for RBAC, ResourceQuotas, and NetworkPolicies (all covered in later sections).

kubectl create namespace team-a
kubectl get pods --namespace=team-a       # or -n team-a
kubectl get pods --all-namespaces         # or -A

A few things to internalize:

  • Every cluster ships with default (where unscoped objects land), kube-system (the control-plane's own components — kube-proxy, CoreDNS, etc.), and kube-public.
  • Cross-namespace DNS: a Service is reachable as <service> within its namespace, or fully-qualified as <service>.<namespace>.svc.cluster.local from anywhere.
  • Namespaces are not a hard security boundary by themselves — they're an organizational and policy boundary. Strong isolation needs NetworkPolicies, RBAC, and quotas layered on top.

2026 Update: Since the old PodSecurityPolicy was removed in v1.25, baseline Pod security is now enforced per namespace via Pod Security Admission labels, e.g. pod-security.kubernetes.io/enforce: restricted. So namespaces are also where you set the security posture for the Pods inside them (see the Security section).

Recap

  • A Pod is the smallest deployable unit: one-or-more containers sharing an IP and (optionally) volumes, scheduled together and ephemeral. Prefer one container per Pod; add init containers for setup and native sidecars (init container with restartPolicy: Always, GA in v1.33) for helpers that must outlive startup.
  • Use probes correctly: liveness restarts, readiness gates traffic, startup covers slow boots. Set requests/limits to control scheduling and QoS (Guaranteed > Burstable > BestEffort); v1.33 even allows in-place resize without a restart.
  • Don't run bare Pods — pick a controller: Deployment (stateless), StatefulSet (stable identity + per-Pod storage), DaemonSet (one per node), Job (run to completion), CronJob (scheduled). All use apps/v1 or batch/v1.
  • Inject config at runtime with ConfigMaps (non-secret) and Secrets (sensitive — but base64 is not encryption; add etcd encryption + RBAC + an external secrets manager). Use mounted files for live updates, env vars for boot-time config; mark unchanging config immutable.
  • Labels identify and group; selectors query them; annotations hold non-queryable metadata. Deployments and Services find their Pods only through matching labels — a label mismatch is the classic "Service has no endpoints" bug.
  • Namespaces partition the cluster for naming, RBAC, quotas, NetworkPolicies, and (post-PSP) Pod Security Admission.

08. The Kubernetes Network Model and CNI

Before we talk about Services, Ingress, or anything fancy, we need to understand the foundation everything else is built on: how Pods talk to each other on the network. If you get this mental model right, the rest of Kubernetes networking falls into place.

The flat network model — the one rule to remember

Kubernetes has one big, opinionated rule about networking:

Key idea: Every Pod gets its own unique IP address, and every Pod can reach every other Pod directly using that IP — no NAT (Network Address Translation), no port mapping — as if they were all on one big flat LAN.

This is called the flat pod network (or "IP-per-Pod" model). Compare it to plain Docker, where containers usually share the host's IP and you juggle port mappings (-p 8080:80). Kubernetes throws that complexity out. From any Pod's point of view, the whole cluster looks like one flat network where it can dial any other Pod's IP and reach it.

Note: "Flat" just means there's no hierarchy or address translation to reason about. Pod A on Node 1 talks to Pod B on Node 5 using B's IP, and the packet arrives unchanged. Who makes that actually work across physical machines? The CNI plugin (we'll get there).

A few consequences of this model:

  • Containers inside the same Pod share one IP and one network namespace. They talk to each other over localhost (e.g., 127.0.0.1:8080) — like two processes on the same machine. (This is why a sidecar can reach the main app on localhost.)
  • Pods are ephemeral, so their IPs are too. When a Pod dies and gets rescheduled, it comes back with a new IP. That's exactly why we never hardcode Pod IPs and instead put a stable Service in front (Section 9).

Pod CIDR: how IPs get handed out

Every cluster has a big block of IP addresses reserved for Pods, called the Pod network CIDR (Classless Inter-Domain Routing block — just a range of IPs written like 10.30.0.0/16). You typically set it once at cluster creation:

# Example: tell the cluster Pods live in 10.30.0.0/16
kubeadm init --pod-network-cidr 10.30.0.0/16

Each node then gets a slice of that range — commonly a /24 (256 addresses), e.g. Node 1 owns 10.30.0.0/24, Node 2 owns 10.30.1.0/24, and so on. When the kubelet starts a Pod on a node, the CNI plugin hands it an IP out of that node's slice. This per-node carving is what keeps IPAM (IP Address Management) simple and fast.

Tip: Size your Pod CIDR for the future. Roughly: nodes × pods-per-node must fit. A /24 per node gives ~256 Pod IPs per node; a /16 Pod CIDR gives 256 such nodes. Running out of Pod IPs mid-growth is painful — it often means rebuilding the cluster. Avoid overlapping the Pod CIDR with your host/VPC network ranges.

CNI: the Container Network Interface

Kubernetes itself does not implement pod networking. It defines an interface — the CNI (Container Network Interface) — and you plug in an implementation. This is the same "Kubernetes defines the contract, vendors provide the engine" pattern you saw with CRI (Container Runtime Interface) for runtimes and CSI (Container Storage Interface) for storage.

When the kubelet needs to wire up a Pod, it calls the CNI plugin, which implements a few simple operations:

CNI operationWhat it does
ADDAttach a new Pod to the network; allocate and return its IP/interface
DELETEDetach a Pod; free its IP
CHECKVerify a Pod's network is set up correctly
VERSIONReport supported CNI spec versions

In practice the CNI plugin runs as a DaemonSet (one Pod per node) that drops a binary into /opt/cni/bin/ and a config into /etc/cni/net.d/. After that, kubelet + runtime call it for every Pod.

Note: A brand-new cluster with no CNI plugin installed is broken — CoreDNS and other Pods get stuck and you'll see errors like NetworkPluginNotReady / cni config uninitialized. The CNI plugin is not optional; it's what makes the flat network real.

How CNIs actually move packets between nodes

There are two broad approaches:

  • Overlay / encapsulation — the Pod's packet is wrapped ("encapsulated") inside another packet addressed to the destination node's IP, sent across the real network, then unwrapped on arrival. Common encapsulation protocols are VXLAN and Geneve. Pros: works almost anywhere because the underlying ("underlay") network doesn't need to know Pod IPs exist. Cons: a little overhead, slightly smaller usable packet size (watch your MTU), and harder to troubleshoot with tcpdump.
  • Native / routed — no wrapping; the real network is taught how to reach Pod IPs directly, often via BGP (Border Gateway Protocol) advertising routes. Faster and easier to inspect, but the underlying network has to cooperate.

The CNI plugins you'll actually meet

Managed platforms ship eBPF dataplanes as first-class options (EKS Auto Mode, GKE Dataplane V2, AKS). For a plain cluster that just needs standard NetworkPolicy, Calico is still a perfectly good choice.

A note on AWS VPC CNI

On EKS the default is the AWS VPC CNI: each Pod gets a real IP from your VPC subnet, attached via secondary IPs on the node's ENIs. The upside is dead-simple routing and troubleshooting (Pods look like normal VPC hosts to security groups and flow logs). The catch: the number of Pods per node is capped by the instance type's ENI/IP limits, so a small instance can host surprisingly few Pods. (Many teams run Cilium in "ENI mode" or enable prefix delegation to raise that ceiling.)

IPv4, IPv6, and dual-stack

Most clusters still run IPv4 only, but dual-stack (IPv4 + IPv6 at the same time) is stable and well supported. A node advertises a Pod CIDR for each family:

spec:
  podCIDRs:
    - 10.30.0.0/24        # IPv4 range for this node
    - fd00:10:244::/64    # IPv6 range for this node (ULA; lab/private use)

Use dual-stack when you have IPv6-native workloads or you're running out of IPv4 space. It requires the CNI plugin to support it and dual-stack settings on the API server, controller-manager, and kube-proxy.

Encrypting pod-to-pod traffic

By default, Pod-to-Pod traffic inside the cluster is not encrypted. If you need it (e.g., for compliance), several CNIs can add transparent encryption — Cilium and Calico via WireGuard, Antrea via IPsec — without changing your app. Expect some throughput cost. For per-connection identity and mTLS between specific services, a service mesh (Section 11) is the other common answer.

09. Services and kube-proxy

We just said Pod IPs are ephemeral — they change every time a Pod is rescheduled. So how does one app reliably reach another? The answer is the Service: a stable front door for a moving set of Pods.

The problem Services solve

Imagine a web Deployment with 3 replicas. Each Pod has its own IP, and those IPs come and go. You don't want every client to track that churn. A Service gives you:

  • A stable virtual IP (the ClusterIP) and a stable DNS name that never change for the life of the Service.
  • Load balancing across all the healthy Pods that match its label selector.
  • Service discovery — clients look up the name, not the Pods.

Key idea: A Service is "a named, stable IP + DNS name that load-balances to a live set of Pods chosen by a label selector." The Pods behind it can be created, killed, and rescheduled all day; the Service IP stays put.

Here's a minimal ClusterIP Service in front of Pods labeled app: nginx:

apiVersion: v1
kind: Service
metadata:
  name: nginx
spec:
  type: ClusterIP            # the default; internal-only virtual IP
  selector:
    app: nginx               # picks Pods carrying label app=nginx
  ports:
    - protocol: TCP
      port: 80               # port clients hit on the Service
      targetPort: 8080       # port the Pods actually listen on

Note: port is what clients call on the Service; targetPort is the port inside the Pod. They're allowed to differ. The Service quietly maps one to the other.

Note: targetPort can be a number (8080) or the name of a port declared in the container's ports[].name. If you write a named targetPort (e.g. targetPort: http) but never gave the container a matching ports: [{name: http, containerPort: 8080}], the Service has nothing to map to — endpoints stay empty and you get the classic "Service won't route." Match the name on both sides, or just use the number.

The Service types

There are five flavors. The first three build on each other like layers of a cake.

A Service can also expose multiple named ports, and supports TCP, UDP, and SCTP:

apiVersion: v1
kind: Service
metadata:
  name: kube-dns
spec:
  # normally you don't set clusterIP yourself — Kubernetes assigns one
  ports:
    - name: dns
      port: 53
      protocol: UDP
      targetPort: 53
    - name: dns-tcp
      port: 53
      protocol: TCP
      targetPort: 53

EndpointSlices: how a Service knows its Pods

A Service is just the stable front door. The actual list of "which Pod IPs are alive and ready right now" lives in a separate object. Historically this was the Endpoints object; today it's EndpointSlices.

2026 Update: The old Endpoints API is deprecated (since v1.33) — learn EndpointSlices instead. EndpointSlices were created to fix scalability problems: one giant Endpoints object for a 10,000-Pod Service had to be re-sent to every node on any change. EndpointSlices chop that list into chunks (up to ~100 endpoints each), so a single Pod change updates one small slice instead of one huge object. They've been the default since v1.19 and are populated automatically by the EndpointSlice controller.

Each endpoint records the Pod's IP, the node it's on, the target port, and — crucially — whether the Pod is ready (passing its readiness probe). Only ready Pods receive traffic. This is the link between health checks and load balancing: a Pod that fails readiness is silently pulled out of rotation.

# See which Pods currently back a Service
kubectl get endpointslices -l kubernetes.io/service-name=nginx
kubectl describe service nginx        # also shows the live endpoints

Try it: Scale a Deployment up and down and watch endpoints react in real time: ``bash kubectl get endpointslices -l kubernetes.io/service-name=nginx --watch # in another terminal: kubectl scale deployment nginx --replicas=5 ``

Try it: Actually verify routing from inside the cluster — spin up a throwaway Pod and curl the Service by name: ``bash kubectl run tmp --rm -it --image=busybox --restart=Never -- wget -qO- http://my-svc ` If you get the backend's response, DNS + Service + endpoints all work end to end. If it hangs or fails, check the Service's endpoints (empty endpoints = selector/targetPort` mismatch).

kube-proxy: turning the ClusterIP into real routing

The ClusterIP is virtual — no single machine "owns" it. So what makes a packet sent to 10.96.0.10 actually land on a real Pod? That's kube-proxy, a small agent that runs on every node (as a DaemonSet). It watches the API for Services and EndpointSlices and programs the node's Linux networking so that traffic to a ClusterIP gets load-balanced to a backing Pod IP.

kube-proxy has two main modes:

  • iptables mode (default): installs NAT rules. A packet to the ClusterIP hits the KUBE-SERVICES chain, jumps to a per-Service chain that randomly picks an endpoint, then a DNAT (Destination NAT) rule rewrites the destination to the chosen Pod IP. Simple and battle-tested, but rule lookup is roughly O(n) — with thousands of Services, both packet processing and rule updates get slow.
  • IPVS mode: uses the kernel's IP Virtual Server (built for load balancing) with hash tables, giving better performance and more balancing algorithms at scale.
# Peek at the NAT rules kube-proxy installs (iptables mode)
iptables --list --table nat | grep KUBE

Note: Endpoint selection is not guaranteed round-robin — iptables mode picks an endpoint randomly (probabilistically), so consecutive requests from the same client can land on different Pods. There's also no session stickiness by default: each connection is balanced independently. If an app keeps state in-memory per session and needs the same client pinned to the same Pod, set spec.sessionAffinity: ClientIP on the Service (optionally with sessionAffinityConfig.clientIP.timeoutSeconds). The real fix for stateful apps, though, is to externalize session state.

2026 Update: Modern eBPF CNIs like Cilium can replace kube-proxy entirely, doing Service load balancing with in-kernel eBPF maps (O(1) lookups) instead of iptables/IPVS chains. On big clusters this noticeably cuts latency and speeds up Service updates. You'll still think about Services the same way — the plumbing underneath just changes.

Note: Two real-world gotchas. (1) You can't ping a ClusterIP — Services are TCP/UDP/SCTP constructs, not real hosts that answer ICMP. Test with curl or telnet to a real port instead. (2) During a rolling update there's a brief race: a Pod can be terminating while kube-proxy hasn't removed its rule yet, causing the odd dropped request. Readiness probes plus graceful shutdown (and preStop hooks) minimize this.

externalTrafficPolicy is worth knowing for NodePort/LoadBalancer: Cluster (default) spreads traffic to Pods on any node (extra hop, source IP hidden behind SNAT); Local only sends to Pods on the receiving node (no extra hop, preserves the client's real source IP). The trade-off: Local can unbalance traffic (a node with more Pods gets proportionally more), and a node with no local Pod fails its load-balancer health check and receives nothing — so spread Pods across nodes if you use it.

Service discovery: DNS, the API, and env vars

There are three ways a Pod can find a Service.

1. DNS (the standard, use this). Every cluster runs CoreDNS (a Deployment, fronted by its own Service at a well-known IP like 10.96.0.10). Every Service automatically gets a DNS name:

<service-name>.<namespace>.svc.cluster.local

So nginx in the default namespace is nginx.default.svc.cluster.local. Thanks to the search domains injected into each Pod's /etc/resolv.conf, a Pod in the same namespace can just use the short name nginx; cross-namespace it uses nginx.<other-namespace>.

# A Pod's /etc/resolv.conf (managed by Kubernetes)
search default.svc.cluster.local svc.cluster.local cluster.local
nameserver 10.96.0.10
options ndots:5

Tip: On large clusters, DNS can become a bottleneck. Two common fixes: NodeLocal DNSCache (a DaemonSet that caches DNS on each node — lower latency, less load on CoreDNS) and autoscaling CoreDNS with the Cluster Proportional Autoscaler (adds CoreDNS replicas as the cluster grows).

2. The Kubernetes API. Sophisticated apps (load balancers, meshes, custom controllers) can query and watch the API for live endpoint changes. Powerful, but it couples your app to Kubernetes — keep it out of business logic.

3. Environment variables (legacy, avoid). For each Service that exists when a Pod starts, Kubernetes injects vars like NGINX_SERVICE_HOST=10.110.34.183 and NGINX_SERVICE_PORT=80. The fatal limitations: the Service must exist before the Pod starts, and the vars never update without a Pod restart. Prefer DNS.

10. Exposing HTTP: Ingress, Gateway API, and Load Balancing

Services get you to Layer 4 (TCP/UDP — routing by IP and port). But web apps need Layer 7 (HTTP) smarts: route by hostname, route by URL path, terminate TLS, and do it all behind a single IP. That's where Ingress — and its modern successor, the Gateway API — come in.

L4 vs L7, and why a Service isn't enough

A Service forwards packets without looking inside them (L4). It can't say "send /api to one app and /shop to another," because that decision lives in the HTTP request. And if you give every web app its own LoadBalancer Service, you pay for a separate cloud load balancer each time. The fix is virtual hosting: one load balancer / reverse proxy with one IP, inspecting the HTTP Host header and URL path to route to the right backend.

Ingress: the original L7 API

Ingress is the Kubernetes-native way to declare L7 HTTP routing rules. But Ingress is just a spec — Kubernetes ships no controller by default. You install an Ingress controller (a reverse proxy like NGINX, Envoy-based Contour, HAProxy, or Traefik) that watches Ingress objects and reconfigures itself to match.

Here's modern, correct Ingress YAML routing two hostnames to two Services:

apiVersion: networking.k8s.io/v1     # the ONLY valid version today
kind: Ingress
metadata:
  name: shop-ingress
spec:
  ingressClassName: nginx            # which controller handles this
  rules:
    - host: shop.example.com
      http:
        paths:
          - path: /
            pathType: Prefix         # required in networking.k8s.io/v1
            backend:
              service:
                name: shop
                port:
                  number: 80
    - host: api.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: api
                port:
                  number: 80

2026 Update: Old tutorials show apiVersion: extensions/v1beta1 and a backend: { serviceName, servicePort } shape. Both are removed. Use networking.k8s.io/v1, the nested backend.service.{name,port} form, ingressClassName, and an explicit pathType (Prefix, Exact, or ImplementationSpecific). The Ingress API itself is now frozen — it still works, but gets no new features.

2026 Update — community ingress-nginx is END-OF-LIFE (2026-03-24). The popular kubernetes/ingress-nginx controller (used in roughly half of cloud-native environments) is retired and read-only — no more CVE patches. Continuing to run it is a growing security liability. Migrate to the Gateway API (below) using the ingress2gateway tool, or move to a maintained alternative: a Gateway API implementation (Istio, Kong, Cilium, Contour), the AWS Load Balancer Controller on EKS, Traefik, or commercial F5 NGINX. Don't confuse this with F5's separate commercial NGINX Ingress Controller, which is not affected.

Gateway API: the modern, recommended path

The Gateway API is the official successor to Ingress (GA since v1.0 in late 2023; v1.4 as of late 2025, with 20+ conformant implementations). It fixes Ingress's biggest pains: feature sprawl crammed into vendor-specific annotations, weak multi-team support, and HTTP-only thinking.

Its model is role-oriented — it splits responsibilities across three resource kinds owned by different people:

ResourceWho owns itWhat it does
GatewayClassInfra provider / platform teamNames an implementation (like Ingress's ingressClassName, but a real object)
GatewayCluster operatorA concrete entry point: which ports/protocols/listeners, and the TLS config
HTTPRouteApp developerThe actual routing rules: match hostnames/paths/headers → send to Services

This separation means platform teams manage the front door (Gateways) while app teams self-serve their routes (HTTPRoutes) without touching shared infra. It's also protocol-aware (HTTP, TCP, gRPC) and supports native traffic splitting (canary / blue-green) by weight — no annotations required.

# 1) The entry point: a Gateway listening for HTTPS
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: web-gateway
spec:
  gatewayClassName: cilium        # set by your platform team
  listeners:
    - name: https
      protocol: HTTPS
      port: 443
      tls:
        mode: Terminate
        certificateRefs:
          - name: example-com-tls # a kubernetes.io/tls Secret
---
# 2) The routing rules, owned by the app team
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: shop-route
spec:
  parentRefs:
    - name: web-gateway           # attach to the Gateway above
  hostnames:
    - "shop.example.com"
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /
      backendRefs:
        - name: shop
          port: 80
    # Canary: split traffic 90/10 between two versions, by weight
    - matches:
        - path:
            type: PathPrefix
            value: /checkout
      backendRefs:
        - name: checkout-v1
          port: 80
          weight: 90
        - name: checkout-v2
          port: 80
          weight: 10

When two rules share a PathPrefix (here / and /checkout), matching is longest-prefix-wins, so a request to /checkout is handled by the /checkout rule rather than the catch-all /.

Key idea: For new clusters in 2026, reach for the Gateway API first. Ingress still works and you'll see it everywhere, but it's frozen and the most popular controller is EOL. The Gateway API is where the ecosystem is going — and many CNIs (Cilium, Calico) and meshes (Istio) implement it natively, so it can double as your mesh ingress.

TLS and cert-manager

To serve HTTPS, the controller needs a TLS certificate. You store the cert + key as a kubernetes.io/tls Secret and reference it (in an Ingress's tls: block or a Gateway's certificateRefs). The controller does TLS termination at the edge — it decrypts incoming HTTPS and forwards to your backend (often over plain HTTP inside the cluster). If a backend needs encryption all the way, you use TLS passthrough (forward the encrypted connection untouched, letting the Pod do its own handshake).

Managing certs by hand is miserable, so almost everyone uses cert-manager — a controller that automatically obtains and renews certificates (from Let's Encrypt / any ACME CA, or HashiCorp Vault) and stores them as Secrets. You declare an Issuer/ClusterIssuer, then just annotate your Ingress (or reference it from a Gateway) and cert-manager does the rest:

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: example-com
  annotations:

11. Network Security and Service Mesh

By default Kubernetes is wide open: thanks to the flat network, any Pod can talk to any other Pod. That's convenient for getting started and dangerous for production. This section covers locking traffic down with NetworkPolicy and the heavier-weight option, a service mesh.

NetworkPolicy: a firewall for Pods

A NetworkPolicy is a declarative, label-based firewall describing which traffic is allowed into (ingress) and out of (egress) Pods. Key things to internalize:

  • Policies are namespace-scoped and select Pods by label (podSelector).
  • They are additive and allow-only: there are no "deny" rules. Once any policy selects a Pod for a direction, that Pod becomes default-deny for that direction, and only the listed traffic is permitted.
  • You need a CNI plugin that enforces them (Calico, Cilium, etc.). The API exists in every cluster, but with a plugin that ignores it, your policy does nothing — a classic trap.

Note: "No policy selects this Pod" means allow everything. "At least one policy selects this Pod for ingress" flips it to deny all ingress except what's explicitly allowed. So the way you "turn on" security is to start applying policies.

The most important pattern is default-deny: select all Pods in a namespace and allow nothing, then add narrow allow-policies on top.

# Default-deny ALL ingress and egress in this namespace
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: org-1
spec:
  podSelector: {}              # {} = every Pod in the namespace
  policyTypes:
    - Ingress
    - Egress
  # no ingress/egress rules listed = nothing is allowed
# Now allow exactly what's needed: let the frontend reach the API on 8080,
# and let every Pod do DNS lookups (UDP 53) so name resolution still works.
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-and-dns
  namespace: org-1
spec:
  podSelector:
    matchLabels:
      app: api
  policyTypes:
    - Ingress
    - Egress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: frontend     # only Pods labeled app=frontend
      ports:
        - protocol: TCP
          port: 8080
  egress:
    - to: []                    # to anywhere...
      ports:
        - protocol: UDP
          port: 53              # ...but only DNS (UDP, the common case)
        - protocol: TCP
          port: 53              # ...and DNS over TCP (large answers, fallback)

Tip: When you flip a namespace to default-deny, the #1 thing people forget is DNS egress (UDP/TCP 53 to CoreDNS). Suddenly nothing can resolve names and you get confusing timeouts. Always allow DNS egress in your baseline policy.

The built-in NetworkPolicy API is intentionally limited — it's L3/L4 (IPs, ports, labels), namespace-scoped, and can't express L7 rules ("allow GET /public but not POST /admin") or cluster-wide policy. CNIs fill the gap with richer CRDs: Calico's GlobalNetworkPolicy, Cilium's CiliumNetworkPolicy / CiliumClusterwideNetworkPolicy (with DNS- and L7-aware rules). The trade-off is portability — those are vendor-specific.

2026 Update: Network security is just one layer. Pod-level hardening moved from the removed PodSecurityPolicy to built-in Pod Security Admission (and tools like Kyverno/OPA Gatekeeper) — covered in the Security section. Think of NetworkPolicy (who can talk to whom) and Pod Security (what a Pod is allowed to do) as complementary.

Service Mesh: when Pod firewalls aren't enough

A service mesh is a dedicated infrastructure layer for service-to-service ("east-west") communication that adds traffic management, security, and observability — without changing your application code. Where NetworkPolicy answers "is this connection allowed?", a mesh answers "encrypt it, retry it, trace it, and shift 10% of it to the new version."

A mesh has two parts:

  • Data plane — a lightweight proxy (almost always Envoy, originally from Lyft) deployed alongside every app Pod, traditionally as a sidecar container. All of the Pod's traffic is transparently redirected through this proxy (in the classic sidecar model, an init container rewrites the Pod's iptables to do so; newer ambient mesh modes move this off the Pod onto a per-node proxy). The app thinks it's making a normal call; the proxy handles the magic.
  • Control plane — configures all those proxies. In Istio (the most popular mesh), the control plane (now consolidated into istiod, historically split into Pilot/Citadel/Galley) pushes routing config and identity certificates to the Envoy proxies via Envoy's xDS APIs.

A mesh gives you three big capability buckets:

  1. Traffic management & reliability — weighted traffic splitting (canary/blue-green), request mirroring, retries, timeouts, circuit breaking. Example: shift 30% of traffic to v2:

  1. Security — mutual TLS (mTLS) between services, automatically. The mesh issues each service a cryptographic identity and encrypts + authenticates every service-to-service call. With Istio you can enforce it cluster-wide:

``yaml apiVersion: security.istio.io/v1 kind: PeerAuthentication metadata: name: default namespace: istio-system spec: mtls: mode: STRICT # reject any non-mTLS traffic ``

  1. Observability — automatic, uniform metrics, distributed traces, and access logs for every call, since all traffic flows through the proxies. You get a service dependency map and golden-signal metrics without instrumenting code.

2026 Update: Two shifts worth knowing. (1) Sidecars are now a first-class Kubernetes concept — native sidecars (a restartable init container, GA in v1.33) start before the app and shut down after it, fixing old ordering/race bugs that plagued mesh injection. (2) Sidecar-less / eBPF meshes are rising: Istio's ambient mode and Cilium Service Mesh move much of the work into a shared per-node proxy or the kernel, cutting the per-Pod memory and latency overhead of sidecars.

Key idea — is a mesh worth it? A mesh is powerful but it is real operational complexity (extra proxies, a control plane, more failure modes, latency overhead). The honest guidance: don't add one early. Adopt a mesh once you (a) have solid Kubernetes operational experience, and (b) genuinely need its features — automatic mTLS for compliance, fine-grained canary releases across many services, or uniform observability across a large microservice estate. If you just need "Pod A can't talk to Pod B," NetworkPolicy is the right, far simpler tool. Many teams get encryption from a WireGuard-enabled CNI and never need a full mesh.

Recap

  • Flat network + CNI: every Pod gets a unique IP and can reach every other Pod with no NAT; the CNI plugin (Cilium/Calico/AWS-VPC-CNI) makes that real and assigns IPs from the per-node slice of the Pod CIDR. In 2026, Cilium/eBPF leads and can even replace kube-proxy.
  • Services give moving Pods a stable IP + DNS name and load-balance to them; pick ClusterIP (internal), NodePort/LoadBalancer (external), Headless (direct Pod IPs), or ExternalName. The live backends live in EndpointSlices (the old Endpoints API is deprecated).
  • kube-proxy programs iptables/IPVS to route ClusterIP traffic to Pods; CoreDNS is how Pods discover Services by name (svc.<ns>.svc.cluster.local). You can't ping a ClusterIP; env-var discovery is legacy.
  • Ingress/Gateway do L7 HTTP routing behind one IP. Community ingress-nginx is EOL (2026-03-24) — use the Gateway API (GatewayClass/Gateway/HTTPRoute) for new work, with cert-manager for automatic TLS. Always use networking.k8s.io/v1 for any Ingress YAML.
  • NetworkPolicy is a label-based, allow-only, default-deny firewall for Pods (requires an enforcing CNI; remember to allow DNS egress). A service mesh (Istio/Envoy) adds mTLS, traffic shifting, and observability — adopt it only when you truly need those features.

12. Storage: Volumes, PV, PVC, StorageClass, and CSI

Containers are designed to be ephemeral (short-lived and disposable). When a container restarts, its writable layer is wiped clean — anything the app saved to its own filesystem is gone. That is fine for stateless web servers, but databases, message queues, and file uploads need data that survives restarts, rescheduling, and even node failure. This section is about giving Pods durable storage in a clean, declarative way.

Key idea: A volume is just a directory that Kubernetes makes available inside one or more containers in a Pod. Different volume types decide where that directory's data actually lives (the node's disk, RAM, or a network storage system).

12.1 Ephemeral volumes (data that lives and dies with the Pod)

Some data only needs to last as long as the Pod. For that, you don't need any of the heavy machinery below — you use a volume baked into the Pod spec.

  • emptyDir — an empty directory created when the Pod is assigned to a node and deleted when the Pod is removed. It survives container restarts (handy for crash recovery) but not Pod deletion. Great for scratch space, caches, and sharing files between containers in the same Pod.
  • hostPath — mounts a directory from the node's filesystem into the Pod. Powerful but dangerous: it ties the Pod to a specific node and can expose the host. Avoid it for application data; it's mainly for system agents (e.g. a log collector reading /var/log).
apiVersion: v1
kind: Pod
metadata:
  name: cache-demo
spec:
  containers:
    - name: app
      image: nginx:1.27
      volumeMounts:
        - name: scratch
          mountPath: /cache      # where the dir appears inside the container
  volumes:
    - name: scratch
      emptyDir: {}               # created on scheduling, deleted with the Pod

Note: Two parts always work together: volumes (defined once at the Pod level — "here is the storage") and volumeMounts (per container — "mount that storage at this path inside me"). They are linked by the name field.

2026 Update: The old notes' "use NFS/EBS/GCE-PD directly in the Pod spec" approach (called in-tree volume plugins) is gone. The in-tree-to-CSI migration went GA in v1.25 and the in-tree provider code has been removed. The gitRepo volume type was removed in v1.33. Today, all real persistent storage flows through PV/PVC + a CSI driver (covered below).

12.2 PersistentVolumes and PersistentVolumeClaims

For durable storage, Kubernetes splits the concern in two so that app developers and cluster admins don't step on each other:

ObjectWho creates itWhat it meansAnalogy
PersistentVolume (PV)Cluster / a CSI driverA piece of actual storage in the cluster (a 30 GiB EBS disk, an NFS export)The physical hotel room
PersistentVolumeClaim (PVC)App developerA request for storage ("I need 20 GiB, ReadWriteOnce")The booking request

A Pod never references a PV directly — it references a PVC, and Kubernetes binds that PVC to a PV that satisfies it (right size, right access mode, right StorageClass). This decoupling means developers ask for what they need without knowing which disk or storage backend supplies it.

Here is the full chain — a PV, a PVC that binds to it, and a Pod that mounts the PVC. Note this example uses a local: volume with hard nodeAffinity and a custom storageClassName to make the moving parts explicit:

Note: This is an advanced static local-PV case — hand-creating the PV, pinning it to one node, and managing its lifecycle yourself. You almost never do this in practice. Normally you use dynamic provisioning (§12.4): you write only a small PVC and the cluster creates the PV for you. Read the YAML below to learn the objects, but reach for §12.4 in real work.

Binding is the act of matching a PVC to a PV. A PVC stays Pending until a suitable PV exists — which is exactly why a Pod can get stuck Pending (see the Troubleshooting section: "Pod is bound to a Pending PVC").

Reclaim policy decides what happens to the PV when the PVC is deleted:

  • Retain — keep the data (admin must clean up manually). Safest for important data.
  • Delete — delete the underlying storage too. Common for dynamically provisioned cloud disks.

Tip: When you let the cluster auto-create volumes (dynamic provisioning, below), the default reclaim policy is often Delete. Deleting the PVC can therefore destroy your data. For anything you care about, use a StorageClass whose reclaimPolicy is Retain.

12.3 Access modes: RWO, ROX, RWX, RWOP

Access modes describe how many nodes can mount a volume and whether they can write. This is one of the most common sources of "why won't my second replica start?" confusion.

ModeShortMeaning
ReadWriteOnceRWOMounted read-write by Pods on a single node. Most common (cloud block disks like EBS / Azure Disk).
ReadOnlyManyROXMounted read-only by Pods on many nodes.
ReadWriteManyRWXMounted read-write by Pods on many nodes (typically file storage like NFS / EFS).
ReadWriteOncePodRWOPMounted read-write by exactly one Pod in the whole cluster (stricter than RWO).

Note: Note the "Once" in ReadWriteOnce means once per node, not once per Pod. Several Pods on the same node can share an RWO volume; Pods on different nodes cannot. Use ReadWriteOncePod if you truly need a single-Pod guarantee.

This matters the moment you scale. A Deployment with replicas: 3 whose Pods all mount one RWO volume will usually land its replicas on different nodes — only one attaches. The other two don't simply "fail to start"; they get stuck in ContainerCreating and kubectl describe pod shows a Multi-Attach error for volume event (search for that exact string when you hit it). The volume is already attached read-write to the first Pod's node and cannot attach to a second.

Fixes: use RWX storage (NFS/EFS) so all replicas can mount the same volume, use the Recreate rollout strategy for single-replica workloads on RWO, or — far better for stateful apps — give each replica its own volume with a StatefulSet (Section 12.7). Even with RWX, multiple writers must coordinate (write to per-Pod subdirectories) to avoid clobbering each other's files.

12.4 StorageClass and dynamic provisioning

Pre-creating PVs by hand (static provisioning) doesn't scale. Dynamic provisioning lets the cluster create a PV on demand the instant a PVC asks for one. The bridge is the StorageClass: it names a provisioner (a CSI driver) and the parameters for the storage it should create.

apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: default-block
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"
provisioner: ebs.csi.aws.com        # the CSI driver that creates volumes
allowVolumeExpansion: true          # lets PVCs be grown later
volumeBindingMode: WaitForFirstConsumer
parameters:
  type: gp3
  fsType: ext4

Key fields:

  • provisioner — which CSI driver provisions volumes for this class (e.g. ebs.csi.aws.com).
  • parameters — backend-specific knobs (disk type, IOPS, filesystem).
  • allowVolumeExpansion: true — lets you grow a PVC later (you can edit a PVC to a larger size; shrinking is not supported).
  • volumeBindingMode: WaitForFirstConsumer — delays creating the volume until a Pod that uses the PVC is actually scheduled, so the disk is created in the same zone as the Pod. (The alternative, Immediate, can provision a disk in a zone where no node can use it.)

Note: With WaitForFirstConsumer, seeing the PVC sit in Pending (reason: WaitForFirstConsumer) until a Pod that uses it is scheduled is normal, not a failure. The volume is created the moment the first consuming Pod is scheduled; only then does the PVC bind.

  • default class — the is-default-class annotation means PVCs that omit storageClassName use this class automatically.

Now a developer just writes a tiny PVC — no PV, no admin involved:

apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: pvc0
spec:
  storageClassName: default-block      # omit to use the default class
  accessModes: ["ReadWriteOnce"]
  resources:
    requests:
      storage: 20Gi

The provisioner sees the new PVC, creates a 20 GiB volume in the backend, wraps it in a PV, and binds the two together — fully automatic.

2026 Update: Use the correct API version storage.k8s.io/v1 (the older storage.k8s.io/v1beta1 and the volume.beta.kubernetes.io/storage-class annotation seen in 2024-era notes are removed). Also note VolumeAttributesClass (GA v1.34) lets you change a volume's parameters — e.g. bump IOPS — without recreating the PVC.

12.5 The Container Storage Interface (CSI)

CSI (Container Storage Interface) is the standard plug-in contract that lets any storage vendor write a driver that Kubernetes can use — without that code living inside Kubernetes itself ("out-of-tree"). This is why support for EBS, EFS, Ceph, Portworx, vSAN, etc. ships and upgrades independently of Kubernetes releases.

2026 Update: CSI is no longer optional or "emerging" as older notes implied — it is the storage mechanism. The legacy in-tree plugins (awsElasticBlockStore, gcePersistentDisk, cinder, …) have been removed; you install a CSI driver instead.

A CSI driver has two halves, both usually deployed as Kubernetes workloads:

  • Controller plug-in — handles cluster-wide volume lifecycle (create, delete, attach, snapshot, resize). Runs as a Deployment (often with leader-election for HA).
  • Node plug-in — runs on every node to do the node-local work (formatting filesystems, mounting volumes into Pods). Runs as a DaemonSet (one copy per node — see Section 13).

The controller is paired with small helper containers called sidecars that the Kubernetes storage community maintains, so each vendor only writes the storage-specific logic:

SidecarJob
external-provisionerCreates a volume when a PVC is created
external-attacherAttaches/detaches volumes to/from nodes
external-resizerExpands volumes when a PVC grows
external-snapshotterCreates/manages volume snapshots
node-driver-registrarRegisters the driver with the kubelet (node side)
livenessprobeReports driver health

Note: A sidecar is a helper container that runs alongside the main one in the same Pod. (Kubernetes now has first-class native sidecars — restartable init containers, GA in v1.33 — covered in the Pods section.) You rarely write these CSI sidecars yourself; you install a driver bundle and they come along.

You can confirm a node's drivers are registered:

kubectl get csinode
kubectl get csidrivers

End-to-end flow: PVC created → external-provisioner asks the backend for a volume → external-attacher attaches it to the Pod's node → node plug-in formats and mounts it into the container. If the Pod is rescheduled to another node, the volume detaches and re-attaches there, so the data follows the app.

12.7 StatefulSets and volumeClaimTemplates

A Deployment treats its Pods as interchangeable cattle — random names, no stable storage, any Pod can replace any other. Stateful systems (databases, Kafka, ZooKeeper) need the opposite: stable identity and per-replica storage. That is what a StatefulSet provides:

  1. Stable network identity — Pods are named by ordinal: mongo-0, mongo-1, mongo-2 (not random suffixes). Combined with a headless Service (clusterIP: None), each gets a predictable DNS name like mongo-0.mongo.default.svc.cluster.local.
  2. Ordered, graceful rollout — Pods are created lowest-to-highest (mongo-0 ready before mongo-1 starts) and deleted highest-to-lowest. (podManagementPolicy: Parallel opts out of ordering.)
  3. Per-replica persistent storage via volumeClaimTemplates — the StatefulSet stamps out a separate PVC for each Pod (data-mongo-0, data-mongo-1, …), and a Pod always re-binds to its own PVC after rescheduling.
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mongo
spec:
  serviceName: "mongo"          # must match the headless Service name
  replicas: 3
  selector:
    matchLabels:
      app: mongo
  template:
    metadata:
      labels:
        app: mongo
    spec:
      containers:
        - name: mongodb
          image: mongo:7
          ports:
            - containerPort: 27017
              name: peer
          volumeMounts:
            - name: data
              mountPath: /data/db
  volumeClaimTemplates:          # one PVC PER Pod, created automatically
    - metadata:
        name: data
      spec:
        accessModes: ["ReadWriteOnce"]
        storageClassName: default-block
        resources:
          requests:
            storage: 10Gi
---
apiVersion: v1
kind: Service
metadata:
  name: mongo
spec:
  clusterIP: None               # headless = stable per-Pod DNS, no load balancing
  selector:
    app: mongo
  ports:
    - port: 27017
      name: peer

Key idea: volumeClaimTemplates is the elegant answer to the "each replica needs its own disk" problem you'd otherwise hack together with one Deployment + one PVC per instance. Each Pod gets dedicated RWO storage that follows it for the life of the StatefulSet — and deleting a StatefulSet does not delete its PVCs, so your data is safe by default.

Try it: Watch ordered creation and stable storage: ``bash kubectl apply -f mongo-statefulset.yaml kubectl get pods -w # mongo-0 → mongo-1 → mongo-2, in order kubectl get pvc # data-mongo-0, data-mongo-1, data-mongo-2 ``

Recap

  • Containers are ephemeral; use emptyDir for throwaway scratch and PV/PVC for anything durable. Avoid hostPath for app data.
  • A PVC is a request; a PV is the actual storage. Pods reference the PVC, and Kubernetes binds it to a suitable PV.
  • Access modes decide sharing: RWO = one node, RWX = many nodes read-write, ROX = many read-only, RWOP = exactly one Pod. RWO + multiple replicas is a classic scaling trap.
  • A StorageClass + a CSI driver give you dynamic provisioning — developers write a small PVC and the volume is created automatically (use storage.k8s.io/v1).
  • CSI is the standard, out-of-tree storage plug-in model (controller Deployment + node DaemonSet + sidecars); in-tree plugins are gone.
  • Volume snapshots (snapshot.storage.k8s.io/v1) back up and clone PVCs; StatefulSets give stable names and per-replica storage via volumeClaimTemplates.

13. Scheduling: How Pods Land on Nodes

When you create a Pod, it doesn't run anywhere yet — it's just a record in etcd with no node assigned. The scheduler (kube-scheduler, a control-plane component) watches for these unscheduled Pods and decides which node each should run on. Once it picks a node, it binds the Pod to it, and that node's kubelet actually starts the containers.

Key idea: Scheduling is a placement decision, made once per Pod. After a Pod is bound to a node it does not move; to "reschedule" it, Kubernetes deletes the Pod and a controller (Deployment/StatefulSet) creates a fresh one, which the scheduler places again.

13.1 The scheduling pipeline: Filter → Score → Bind

The scheduler runs each Pod through three phases:

  1. Filter — eliminate nodes the Pod cannot run on. A node is filtered out if it lacks enough free CPU/memory, doesn't match a required nodeSelector/node affinity, has a taint the Pod doesn't tolerate, or can't attach the Pod's volume (e.g. wrong zone). The survivors are the feasible nodes.
  2. Score — rank the feasible nodes 0–100 using scoring plugins (spread across zones, prefer nodes with the image already pulled, honor preferred affinity, balance resource usage, etc.).
  3. Bind — assign the Pod to the highest-scoring node by writing spec.nodeName. The kubelet there pulls images and starts the containers.

If no node survives filtering, the Pod stays Pending and an event explains why (e.g. 0/5 nodes are available: insufficient memory). This is the single most common reason for a stuck Pod.

Note: "Feasible" (filter) is a hard pass/fail; "score" is a soft beauty contest among the survivors. Requirements like node affinity required and taints act in the filter phase; soft preferences act in the score phase.

13.2 nodeSelector — the simplest constraint

nodeSelector is the most basic way to pin Pods to nodes that carry specific labels. First label some nodes, then ask for them:

kubectl label nodes node-3 disktype=ssd
spec:
  nodeSelector:
    disktype: ssd      # only schedule onto nodes labelled disktype=ssd

It's a hard, all-or-nothing match. For anything richer (OR logic, soft preferences, "anything but…"), use node affinity.

13.3 Node affinity: required vs preferred

Node affinity is the expressive successor to nodeSelector. It comes in two flavors whose long names tell you exactly how they behave:

  • requiredDuringSchedulingIgnoredDuringExecution — a hard rule applied at the filter phase. If no node matches, the Pod stays Pending.
  • preferredDuringSchedulingIgnoredDuringExecution — a soft rule applied at the score phase, each with a weight (must be in the range 1–100). The scheduler prefers matching nodes but will place the Pod elsewhere if needed.

(The IgnoredDuringExecution part means: once a Pod is running, changing a node's labels won't evict it.)

Required (hard):

spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
          - matchExpressions:
              - key: disktype
                operator: In
                values: ["ssd"]

Preferred (soft), with weights:

spec:
  affinity:
    nodeAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 50                 # higher weight wins when ranking
          preference:
            matchExpressions:
              - key: price
                operator: In
                values: ["spot"]
        - weight: 1
          preference:
            matchExpressions:
              - key: disktype
                operator: In
                values: ["ssd"]

Note — AND vs OR: Multiple matchExpressions inside one term are ANDed (a node must satisfy all). Multiple nodeSelectorTerms (separate - matchExpressions: blocks) are ORed (matching any one term is enough). Operators include In, NotIn, Exists, DoesNotExist, Gt, Lt.

13.4 Pod affinity and anti-affinity

Where node affinity is about which node, pod affinity is about which other Pods are nearby. It schedules a Pod relative to other Pods, measured by a topologyKey — a node label that defines the "domain" (e.g. kubernetes.io/hostname = same node, topology.kubernetes.io/zone = same availability zone).

  • Pod affinity — attract: place me near Pods matching this label (e.g. co-locate a web app with its cache for low latency).
  • Pod anti-affinity — repel: keep me away from Pods matching this label (e.g. spread replicas of the same app across nodes/zones for resilience).

Spread replicas across different nodes (high availability):

spec:
  affinity:
    podAntiAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        - labelSelector:
            matchExpressions:
              - key: app
                operator: In
                values: ["nginx"]
          topologyKey: "kubernetes.io/hostname"   # "different node"

Co-locate two cooperating apps on the same node:

spec:
  affinity:
    podAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        - labelSelector:
            matchExpressions:
              - key: app
                operator: In
                values: ["kafka", "zookeeper"]
          topologyKey: "kubernetes.io/hostname"

Tip: Required pod anti-affinity can leave Pods Pending if there aren't enough distinct nodes (you can't place 5 hard-anti-affinity replicas on 3 nodes). For best-effort spreading, prefer the preferred... form or topology spread constraints (below).

13.5 Taints and tolerations — nodes that repel Pods

Affinity is the Pod choosing nodes. Taints are the node rejecting Pods — the opposite direction. A taint on a node says "keep Pods off me unless they explicitly tolerate this." A matching toleration on a Pod is its permission slip.

# Reserve spot nodes: taint them so only tolerant Pods land there
kubectl taint nodes node-1 price=spot:NoSchedule
spec:
  tolerations:
    - key: "price"
      operator: "Equal"
      value: "spot"
      effect: "NoSchedule"     # I tolerate the NoSchedule taint price=spot

Taint effects:

  • NoSchedule — don't schedule new non-tolerating Pods here (existing ones stay).
  • PreferNoSchedule — soft version; avoid if possible.
  • NoExecute — also evict already-running Pods that don't tolerate it.

Note: A toleration only permits a Pod onto a tainted node — it doesn't force it there. To both reserve nodes (taint) and attract specific Pods to them, pair the taint with node affinity. Control-plane nodes are tainted by default, which is how DaemonSet system agents (with broad tolerations) run there while your apps don't.

13.6 Topology spread constraints

topologySpreadConstraints give fine-grained control over how evenly Pods spread across failure domains (zones, nodes) — more flexible than anti-affinity for the common "balance my replicas" need.

spec:
  topologySpreadConstraints:
    - maxSkew: 1                       # max imbalance allowed between domains
      topologyKey: topology.kubernetes.io/zone
      whenUnsatisfiable: DoNotSchedule # or ScheduleAnyway (soft)
      labelSelector:
        matchLabels:
          app: my-web

maxSkew: 1 means the difference between the most- and least-populated zone may not exceed 1, so replicas land roughly evenly across zones. whenUnsatisfiable: DoNotSchedule makes it a hard rule; ScheduleAnyway makes it a preference.

13.7 DaemonSet scheduling

A DaemonSet runs exactly one copy of a Pod on every node (or every node matching a nodeSelector) — used for per-node agents like log collectors, monitoring agents, and CNI/CSI node plug-ins. New nodes automatically get a copy; removed nodes have theirs cleaned up.

DaemonSet Pods are scheduled by the normal kube-scheduler (it respects their affinity/taints), but the DaemonSet controller adds tolerations so the agent reaches nodes ordinary Pods avoid. To run on tainted nodes (e.g. control-plane), give the DaemonSet broad tolerations:

spec:
  template:
    spec:
      tolerations:
        - operator: "Exists"      # tolerate every taint → run truly everywhere
      containers:
        - name: fluentd
          image: fluentd:v1.17

Recap

  • The scheduler runs Filter → Score → Bind: discard infeasible nodes, rank the rest, assign the winner. No feasible node ⇒ Pod stays Pending.
  • nodeSelector is the simple label match; node affinity adds required (hard, filter phase) vs preferred (soft, weighted, score phase), plus AND/OR logic.
  • Pod (anti-)affinity places Pods relative to other Pods using a topologyKey; anti-affinity spreads, affinity co-locates.
  • Taints repel Pods from nodes; matching tolerations are the permission to stay (effects NoSchedule/PreferNoSchedule/NoExecute).
  • Topology spread constraints (maxSkew) balance replicas across zones/nodes; DaemonSets put one agent Pod per node and tolerate taints to reach all of them.

14. Autoscaling

Autoscaling adjusts capacity automatically so you neither pay for idle resources nor fall over under load. There are two layers, and they work together:

  • Workload autoscaling — change the application (more Pods, or bigger Pods).
  • Cluster autoscaling — change the infrastructure (more or fewer nodes).

And two directions:

  • Horizontal — add/remove replicas (no restart needed).
  • Vertical — change CPU/memory per Pod (historically needs a restart).

14.1 metrics-server — the fuel for autoscaling

Before any CPU/memory-based scaling works, the cluster needs live resource metrics. metrics-server collects CPU and memory usage from each node's kubelet and exposes it through the metrics.k8s.io API. It powers both kubectl top and the HPA's CPU/memory decisions.

kubectl top nodes
kubectl top pods -A

2026 Update: Older notes mention Heapster — it is retired. Use metrics-server for core resource metrics, and kube-state-metrics (object state: replica counts, restarts) plus Prometheus for everything richer. (See the Observability section.)

14.2 Horizontal Pod Autoscaler (HPA)

The HPA watches a metric for a workload and adjusts its replica count to hit a target. The controller checks every ~15 seconds; if average CPU is above target it scales up, if below it scales down (within your min/max).

Note: CPU/memory-based HPA needs metrics-server (§14.1) installed. Without it, kubectl get hpa shows the target as <unknown>/70% and the HPA won't scale, and kubectl top errors with Metrics API not available. On kind/minikube, metrics-server often needs the --kubelet-insecure-tls flag to start working.

2026 Update: Use autoscaling/v2. The old autoscaling/v1 (single targetCPUUtilizationPercentage field) and the autoscaling/v2beta* versions are obsolete. v2 supports multiple metrics, custom/external metrics, and a behavior block to tune scale-up/scale-down rates.

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: web
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70        # aim for ~70% CPU per Pod
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80
  behavior:                              # control how fast it reacts
    scaleDown:
      stabilizationWindowSeconds: 300    # wait 5 min before scaling in
      policies:
        - type: Percent
          value: 50
          periodSeconds: 60
    scaleUp:
      policies:
        - type: Pods
          value: 4
          periodSeconds: 60              # add at most 4 Pods per minute

Key points:

  • The target workload's containers must declare resource requests (e.g. cpu: 100m) — utilization is measured against the request. No request ⇒ HPA can't compute a percentage.
  • minReplicas/maxReplicas bound the range; the behavior block tames thrashing (e.g. a stabilization window so brief dips don't scale you in).
  • HPA suits stateless, horizontally scalable services. It's a poor fit for leader-elected or single-writer stateful apps.

Note: Don't manually kubectl scale a Deployment that has an HPA — they will fight, each overwriting the other.

Try it: ``bash kubectl autoscale deployment web --min=2 --max=10 --cpu-percent=70 kubectl get hpa -w ``

For metrics beyond CPU/memory (requests-per-second, queue depth), point the HPA at custom or external metrics, typically exposed via Prometheus + the Prometheus Adapter.

14.3 Vertical Pod Autoscaler (VPA)

The HPA adds more Pods; the VPA makes each Pod the right size by tuning its CPU/memory requests (and limits). It's the cure for guessing requests wrong — over-requesting wastes money, under-requesting risks eviction.

The VPA runs three components: a Recommender (suggests values from usage history), an Admission controller (writes the values onto new Pods), and an Updater (evicts Pods so they're recreated with new sizes). Its updateMode controls aggressiveness:

ModeBehavior
OffOnly recommends (read with kubectl describe vpa). Safe; pair with Goldilocks to visualize.
InitialSets requests only at Pod creation — no evictions.
Auto / RecreateActively evicts and recreates Pods to apply new sizes.
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: web
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web
  updatePolicy:
    updateMode: "Off"         # recommend only — the safe starting point
  resourcePolicy:
    containerPolicies:
      - containerName: '*'
        minAllowed: { cpu: 100m, memory: 128Mi }
        maxAllowed: { cpu: "2",  memory: 2Gi }
        controlledResources: ["cpu", "memory"]

**Don't run VPA in Auto mode and HPA on the same metric (e.g. both on CPU) for one workload — they conflict.** Common patterns: VPA in Off (recommendation) mode to right-size requests, then HPA for horizontal scaling; or VPA for CPU and HPA on a custom metric.

2026 Update — in-place pod resize: Historically VPA had to restart (evict + recreate) Pods to change resources. As of v1.33, in-place Pod vertical scaling is beta and on by default — CPU/memory can be resized via the Pod resize subresource without a restart (watch the PodResizePending/PodResizeInProgress conditions). This dramatically reduces the disruption of vertical scaling.

14.4 Scaling the cluster: Cluster Autoscaler vs Karpenter

When Pods are Pending because no node has room, node autoscaling adds capacity; when nodes sit underused, it removes them.

Cluster Autoscaler (CA) is the classic tool. It works through cloud node groups / Auto Scaling Groups: when Pods can't schedule, it bumps the desired size of a group; when nodes are underutilized, it scales them down. It only adds nodes of the pre-defined types in those groups.

Karpenter is the modern, workload-aware alternative (originally AWS, now broader). Instead of scaling fixed node groups, it looks at the actual Pending Pods and provisions right-sized nodes directly — picking instance types/sizes/zones to fit, in ~45–60 seconds (vs several minutes for CA). It bin-packs aggressively, consolidates workloads onto fewer nodes when load drops, and natively handles Spot capacity.

Cluster AutoscalerKarpenter (preferred in 2026)
Scaling unitFixed node groups / ASGsProvisions individual right-sized nodes
Speed~3–4 min~45–60 s
Bin-packing / consolidationLimitedStrong (active consolidation)
Instance flexibilityPer-group, pre-definedChooses type/size/zone per workload
Best forTightly controlled AMIs, fixed GPU pools, non-AWS portabilityMost dynamic workloads, cost optimization, Spot

2026 Update: Newer notes' default of "Cluster Autoscaler" is dated. Karpenter is now the preferred node autoscaler for most teams, and on AWS, EKS Auto Mode runs Karpenter for you off-cluster (no node groups to manage). CA still wins when you need rigidly controlled AMIs/node groups or cloud portability. (See the EKS section.)

Note: HPA/VPA and node autoscaling are partners: HPA adds Pods → those Pods are Pending for lack of room → Karpenter/CA adds a node → the Pods schedule. Neither replaces the other.

14.5 KEDA — event-driven autoscaling and scale-to-zero

The HPA scales on resource/metric utilization. KEDA (Kubernetes Event-Driven Autoscaling) scales on the size of external work — queue length, Kafka consumer lag, pending HTTP requests, cron schedules, cloud-provider metrics, and 60+ other "scalers." Crucially, KEDA can scale to zero: run no Pods when there's no work, then spin up when an event arrives.

apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: worker-scaler
spec:
  scaleTargetRef:
    name: worker                 # the Deployment to scale
  minReplicaCount: 0             # scale to ZERO when the queue is empty
  maxReplicaCount: 30
  triggers:
    - type: aws-sqs-queue
      metadata:
        queueURL: https://sqs.ap-southeast-1.amazonaws.com/123456789012/jobs
        queueLength: "5"          # ~1 Pod per 5 messages
        awsRegion: ap-southeast-1

A queue-based scaler like aws-sqs-queue needs AWS permissions granted via IRSA or EKS Pod Identity (see §24), or KEDA gets AccessDenied when polling the queue.

KEDA works with the HPA (it creates and feeds an HPA under the hood for the active range), and adds the 0↔1 activation the HPA alone can't do. It's ideal for queue workers, batch consumers, and putting non-prod environments to "sleep" on a schedule to save cost.

2026 Update: KEDA isn't in older notes but is now the standard for event-driven and scale-to-zero scaling — add it whenever load is driven by a queue/stream rather than CPU.

14.6 Overprovisioning — capacity ready before you need it

Node provisioning takes time (even Karpenter's ~1 minute can be too slow for a sharp spike). Overprovisioning keeps spare nodes warm by running low-priority placeholder Pods that request resources but do nothing. When real workloads need room, the scheduler evicts the placeholders (they have a negative PriorityClass), real Pods schedule onto the now-free capacity immediately, and the node autoscaler provisions replacement standby nodes in the background.

apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: overprovisioning
value: -1                         # lower than default → evicted first
globalDefault: false

This trades a little steady-state cost for much lower scaling latency, and lets you avoid setting HPA targets uncomfortably low just to keep headroom.

Recap

  • Two axes: horizontal (more replicas/nodes) vs vertical (bigger Pods), across workload and cluster layers. metrics-server feeds CPU/memory autoscaling (Heapster is gone).
  • HPA (autoscaling/v2) scales replicas on CPU/memory/custom metrics; needs resource requests set, and use the behavior block to prevent thrashing.
  • VPA right-sizes requests; start in Off (recommend) mode, and note in-place resize (v1.33+) now changes Pod resources without a restart. Don't run VPA-Auto and HPA on the same metric.
  • For nodes, Karpenter is the modern, fast, workload-aware choice (preferred; EKS Auto Mode runs it for you); Cluster Autoscaler remains for fixed node groups / portability.
  • KEDA adds event-driven scaling and scale-to-zero; overprovisioning with negative-priority placeholder Pods buys instant burst capacity.

15. Authentication, Authorization, and RBAC

Every request that touches a Kubernetes cluster — whether it comes from you typing kubectl, from a CI pipeline, or from a Pod talking to the API — goes through the same security gauntlet inside the API server. Understanding that gauntlet is the foundation of everything else in this part of the guide.

There are three gates, and a request must pass all of them in order:

  1. Authentication (authn): Who are you? The API server verifies your identity.
  2. Authorization (authz): Are you allowed to do this? Kubernetes checks your permissions — this is where RBAC (Role-Based Access Control) lives.
  3. Admission control: Should this specific object be allowed, and does it need to be modified first? (Covered in Section 18.)

Key idea: Authentication answers "who are you?" Authorization answers "what can you do?" They are separate steps. A request can be perfectly authenticated and still be rejected by authorization.

Authentication: how Kubernetes learns who you are

Here is something that surprises most newcomers: Kubernetes has no built-in User or Group object. There is no database of usernames and passwords. Instead, the API server trusts an external source to vouch for identity, and it supports several mechanisms (called authenticators). The three you will meet most often:

MethodHow it worksGood forWatch out for
x509 client certificatesYou present a TLS client cert; the Common Name (CN) becomes your username and the Organization (O) fields become your groups.Cluster bootstrap, admins, kubeadm-provisioned clusters.K8s does no CRL/OCSP, so certs can't be revoked before they expire — cut access by removing the RBAC binding (see note below). High maintenance at scale.
OIDC (OpenID Connect) tokensThe cluster trusts an external identity provider (Okta, Auth0, Entra ID, Google). You log in there, get a signed JWT (JSON Web Token), and the API server validates it.Human users, SSO, large orgs. Recommended for people.Requires an external IdP and API-server flags.

A static-token / username-password file is also possible but is discouraged — tokens never expire, are stored in plaintext, and changing them requires restarting the API server.

Tip: For human users prefer OIDC; for in-cluster workloads use ServiceAccounts. Reserve x509 certs for break-glass admin access (and guard that admin.conf kubeconfig carefully).

x509 in practice (Certificate Signing Requests)

A user can be onboarded by generating a key + CSR (Certificate Signing Request) and asking the cluster to sign it:

# 1. Create a private key and a CSR with CN=john (the future username)
openssl req -new -key john.key -out john.csr -subj "/CN=john/O=ops-team"
# 2. Submit it as a CertificateSigningRequest object
apiVersion: certificates.k8s.io/v1
kind: CertificateSigningRequest
metadata:
  name: john
spec:
  request: <base64 of john.csr>
  signerName: kubernetes.io/kube-apiserver-client
  usages:
  - client auth
# 3. An admin approves it; the signed cert can then go into john's kubeconfig
kubectl certificate approve john

Note: The CN (john) becomes the username and each O (ops-team) becomes a group. Kubernetes never "creates" john — it just trusts the cert your CA signed. That's why there's no kubectl create user.

Security note: Kubernetes does not check certificate revocation lists (CRL) or OCSP for client certs — a signed cert is valid until it expires, and there is no built-in "revoke this cert" button. So to actually cut off a compromised cert's access: (a) remove its RBAC binding — that's the real lever, since identity without any binding can do nothing; (b) keep cert lifetimes short so the window is small; and (c) as the heavy, nuclear last resort, rotate the cluster CA (which invalidates every cert it signed and forces a re-issue across the cluster).

ServiceAccounts: identity for Pods

A ServiceAccount (SA) is the one identity object Kubernetes does store. Every namespace has a default SA, and every Pod runs as one (the default, unless you set serviceAccountName). The kubelet projects a short-lived, audience-scoped JWT into the Pod at /var/run/secrets/kubernetes.io/serviceaccount/token, which the app uses to call the API.

apiVersion: v1
kind: ServiceAccount
metadata:
  name: invoice-reader
  namespace: billing
automountServiceAccountToken: false   # ONLY if this Pod never calls the Kubernetes API
apiVersion: v1
kind: Pod
metadata:
  name: invoice-app
  namespace: billing
spec:
  serviceAccountName: invoice-reader   # run as our dedicated SA, not "default"
  containers:
  - name: app
    image: registry.example.com/invoice-app:1.4.0

Note: Only set automountServiceAccountToken: false when the Pod genuinely does not talk to the Kubernetes API — it's least privilege for API-silent workloads (most web apps and batch jobs). If the workload does need the API (a controller, an operator, anything using client-go/kubectl in-cluster), leave the token mounted — pairing a no-token SA with an API-calling workload just breaks it with confusing 401/403 errors.

2026 Update: Modern Kubernetes uses bound, time-limited, audience-scoped ServiceAccount tokens (sometimes called projected tokens) by default — they auto-rotate and expire (e.g. every hour) instead of being long-lived secrets sitting in etcd forever. If older notes describe a permanent *-token-xxxxx Secret auto-created for every SA, that legacy behavior is gone; create a token Secret explicitly only when you truly need a static one. Disable automountServiceAccountToken for any Pod that doesn't call the API — least privilege starts here.

Authorization with RBAC

Once you're authenticated, RBAC decides what you may do. RBAC is built from just four object kinds, and the mental model is a simple chain:

  • **A Role (or ClusterRole)** is a list of permissions — "may perform these verbs on these resources." A Role grants nothing on its own; it's just a permission set.
  • **A RoleBinding (or ClusterRoleBinding)** attaches that Role to one or more subjects (a User, a Group, or a ServiceAccount).

Permissions in RBAC are purely additive — there are no "deny" rules. You start with nothing and grant exactly what's needed.

Namespaced vs cluster-wide

This is the part that trips everyone up. The "Cluster" prefix controls scope:

ObjectScopeUse it for
RoleOne namespace"Read pods in billing."
ClusterRoleWhole cluster (or cluster-scoped resources like nodes, PVs, namespaces)"Read pods everywhere," or grant access to non-namespaced things.
RoleBindingOne namespaceGrants a Role (or even a ClusterRole) within a single namespace.
ClusterRoleBindingWhole clusterGrants a ClusterRole across all namespaces.

Key idea: A RoleBinding can reference a ClusterRole. This is a hugely useful pattern: define a reusable ClusterRole once (e.g. "read-only"), then bind it inside individual namespaces with RoleBindings. You write the permission set one time and apply it per-team.

A complete example

Grant a ServiceAccount read-only access to Pods and Services, but only in the dev namespace:

apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: dev
  name: viewer
rules:
- apiGroups: [""]                 # "" = the core API group
  resources: ["pods", "services"]
  verbs: ["get", "list", "watch"]
- apiGroups: [""]
  resources: ["pods/log"]         # sub-resources are addressed with a slash
  verbs: ["get"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  namespace: dev
  name: viewer-binding
subjects:
- kind: ServiceAccount
  name: myapp
  namespace: dev
roleRef:
  kind: Role
  name: viewer
  apiGroup: rbac.authorization.k8s.io

A few things to internalize from this manifest:

  • apiGroups identifies which part of the API the resource lives in. "" (empty string) is the core group (pods, services, configmaps, secrets, nodes…). Other resources name their group, e.g. apps for Deployments, rbac.authorization.k8s.io for RBAC itself, networking.k8s.io for NetworkPolicies.
  • verbs are the actions: get, list, watch, create, update, patch, delete, deletecollection. (list and watch let a subject read all objects of that type — be stingy with them on Secrets.)
  • roleRef is immutable; to change which Role a binding points at, delete and recreate it.

Try it: Check what a subject can do without granting anything, using the built-in access reviewer: ``bash kubectl auth can-i list secrets --namespace dev kubectl auth can-i create deployments --as=system:serviceaccount:dev:myapp -n dev ``

Built-in ClusterRoles

You don't have to write everything from scratch. Kubernetes ships four "user-facing" ClusterRoles you can bind directly:

ClusterRoleGrants
viewRead-only on most namespaced resources (no Secrets).
editRead/write on most resources (can't change RBAC or quotas).
adminFull control within a namespace, including RBAC for that namespace.
cluster-adminGod mode — everything, everywhere. Hand out almost never.

Least privilege and the impersonation pattern

The golden rule of cluster security is least privilege: give each subject the minimum permissions to do its job, and no more. Two practical habits make this real:

  1. One ServiceAccount per workload. Don't let ten apps share default. A compromised app should only be able to do what that app needs.
  2. Namespace as a blast radius. Scope Roles to namespaces so a mistake or breach is contained.

A powerful operational pattern is impersonation. Instead of giving the ops team permanent cluster-admin, give them read-only access by default plus the right to impersonate a privileged identity only when they consciously ask for it:

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: break-glass-impersonator
rules:
- apiGroups: [""]
  resources: ["users"]               # impersonating a User needs the "users" resource
  verbs: ["impersonate"]
  resourceNames: ["alice"]           # may ONLY impersonate this one User
# To impersonate a Group instead, add a rule with resources: ["groups"]
# and resourceNames: ["system:masters"]; ServiceAccounts use resources: ["serviceaccounts"].

You then bind a privileged identity to that real subject — e.g. a ClusterRoleBinding granting cluster-admin to User alice (the actual subject you'll impersonate), so that --as=alice carries those powers.

Day-to-day the engineer runs harmless read commands; for a dangerous change they explicitly elevate by impersonating that subject:

kubectl get configmaps                         # works (read-only)
kubectl create configmap x --from-literal=a=b  # Forbidden by default
kubectl --as=alice create configmap x --from-literal=a=b  # impersonate User alice (who is bound to cluster-admin)
# Impersonate a Group instead with: kubectl --as=ignored --as-group=system:masters ...

Note: --as= takes a subject you can actually authenticate as via impersonation (a User, Group, or ServiceAccount that RBAC binds), not a Role name. kubectl --as=cluster-admin ... does not work, because cluster-admin is a ClusterRole, not a User. Bind the ClusterRole to a real User (e.g. alice) and impersonate that. The impersonate ClusterRole must list the matching verb-resource: users to impersonate a User, groups for a Group, serviceaccounts for an SA.

Because impersonation is recorded in the audit log, you get safety (no accidental destructive commands), auditability (who elevated, when), and scalability — all without standing privileges.

Tip: On managed clusters, identity flows into RBAC from the cloud provider. On AWS EKS, IAM principals map to Kubernetes groups (via EKS Access Entries, the modern replacement for the old aws-auth ConfigMap), and those groups are bound to ClusterRoles. The RBAC objects are identical — only how the username/group is established differs. (See the EKS section.)

Recap

  • Every API request passes authentication → authorization → admission, in that order.
  • Kubernetes stores no users; identity comes from x509 certs, OIDC (best for humans), or ServiceAccount tokens (for workloads).
  • RBAC = Role/ClusterRole (permission sets) + RoleBinding/ClusterRoleBinding (attach them to subjects). Permissions are additive — there is no deny.
  • "Cluster" in the name means cluster-wide scope; a RoleBinding can reuse a ClusterRole within one namespace.
  • Practice least privilege: one SA per workload, scope to namespaces, and prefer impersonation over standing cluster-admin.

16. Pod Security: Standards, Admission, and SecurityContext

RBAC controls who can talk to the API. But once a Pod is running, a different question matters: what can the container do to the node and the kernel underneath it? Containers share the host's Linux kernel, so a container running as root with broad privileges is one kernel bug away from owning the whole machine. This section is about clamping that down.

There are two layers, and you want both:

  1. SecurityContext — per-Pod / per-container settings that harden an individual workload.
  2. Pod Security Admission (PSA) — a cluster guardrail that enforces a baseline across whole namespaces, so nobody can deploy an unsafe Pod in the first place.

2026 Update: If your notes mention PodSecurityPolicy (PSP) — forget it. PSP was deprecated in v1.21 and REMOVED in v1.25. Its replacement is Pod Security Admission (PSA) enforcing the Pod Security Standards, which has been stable since v1.25. Do not write, learn, or copy PSP manifests; they will not even be recognized by a current cluster.

SecurityContext: hardening one workload

securityContext can be set at the Pod level (applies to all containers) or the container level (overrides, finer control). The fields you should reach for almost every time:

allowPrivilegeEscalation: falseBlocks a process from gaining more privileges than its parent (e.g. via setuid).+ Always
readOnlyRootFilesystem: trueMounts the container's root filesystem read-only (write to mounted volumes instead).+

A well-hardened Pod looks like this:

apiVersion: v1
kind: Pod
metadata:
  name: hardened-app
spec:
  securityContext:                  # Pod-level: applies to every container
    runAsNonRoot: true
    runAsUser: 1000
    runAsGroup: 3000
    fsGroup: 2000                   # group that owns mounted volume files
    seccompProfile:
      type: RuntimeDefault          # use the runtime's default syscall filter
  containers:
  - name: app
    image: registry.example.com/app:1.4.0
    securityContext:                # container-level overrides
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      privileged: false
      capabilities:
        drop: ["ALL"]               # drop everything…
        add: ["NET_BIND_SERVICE"]   # …add back only ports <1024 if you truly need them

Note: Capabilities are slices of root's power — NET_BIND_SERVICE (bind low ports), SYS_TIME (set the clock), etc. The safe pattern is always drop ALL, then add back the one or two you genuinely need. Most apps need none.

seccomp, AppArmor, and SELinux

These are kernel-level confinement mechanisms the SecurityContext can hook into:

  • seccomp (secure computing) filters the set of syscalls a process can issue. RuntimeDefault is a sensible, broadly-compatible profile and should be your default.
  • AppArmor confines a program to a set of allowed file/capability/network operations via a profile.
  • SELinux does mandatory access control via labels (common on Red Hat / OpenShift).

2026 Update: AppArmor is now configured with a first-class field, securityContext.appArmorProfile, not the old container.apparmor.security.beta.kubernetes.io/... annotation, which is deprecated: ``yaml securityContext: appArmorProfile: type: RuntimeDefault ``

RuntimeClass: stronger isolation for risky workloads

When a normal container boundary isn't enough — multi-tenant clusters, untrusted code — you can run a Pod under a sandboxed runtime via RuntimeClass. The kubelet talks to runtimes through the CRI (Container Runtime Interface), and sandboxes like gVisor (syscall interception) or Kata Containers / Firecracker (lightweight micro-VMs) give each Pod a much harder wall.

apiVersion: v1
kind: Pod
metadata:
  name: untrusted-job
spec:
  runtimeClassName: gvisor    # run inside a sandbox instead of a plain container
  containers:
  - name: job
    image: registry.example.com/third-party-task:latest

Pod Security Admission (PSA): the cluster guardrail

SecurityContext is opt-in per Pod — which means someone will forget it. Pod Security Admission is a built-in admission controller (on by default since v1.25) that enforces a security floor on an entire namespace, so an unsafe Pod is rejected at submission time.

PSA enforces the three Pod Security Standards:

LevelMeaning
privilegedNo restrictions. For trusted, infrastructure-level workloads only.
baselineBlocks the well-known dangerous stuff (host namespaces, privileged containers, most host mounts). A reasonable default for general apps.
restrictedStrict, hardened best practice: must run as non-root, drop ALL capabilities, allowPrivilegeEscalation: false, seccomp RuntimeDefault, read-only-friendly. The target for sensitive workloads.

And it applies them in three modes, which you can mix on the same namespace:

  • enforce — reject Pods that violate the level.
  • audit — allow, but record a violation in the audit log.
  • warn — allow, but return a warning to the user (great for nudging teams).

You turn it on with namespace labels — no controller to install, no CRDs:

apiVersion: v1
kind: Namespace
metadata:
  name: payments
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/enforce-version: latest
    pod-security.kubernetes.io/warn: restricted
    pod-security.kubernetes.io/warn-version: latest
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/audit-version: latest

If a Pod that doesn't set its SecurityContext is submitted to a restricted namespace, the API server pushes back with exactly what's missing:

Error from server (Forbidden): pods "kuard" is forbidden: violates PodSecurity "restricted:latest":
  allowPrivilegeEscalation != false; unrestricted capabilities (must drop ["ALL"]);
  runAsNonRoot != true; seccompProfile (must be "RuntimeDefault" or "Localhost")

Tip: Roll this out safely by labeling namespaces with warn and audit first. See what would break, fix the workloads, then flip on enforce. You can dry-run a label change against existing Pods: ``bash kubectl label --dry-run=server --overwrite ns payments \ pod-security.kubernetes.io/enforce=restricted ` Any existing Pods that *would* violate the new level come back as **warnings on stderr** (e.g. Warning: existing pods in namespace "payments" violate the new PodSecurity enforce level "restricted:latest"`) — that's where to look to see what would break.

Key idea: PSA gives you exactly three built-in levels. If you need richer or custom rules (e.g. "all images must come from our registry," "every Pod needs a team label"), reach for a policy engine — Kyverno, OPA/Gatekeeper, or in-tree ValidatingAdmissionPolicy — covered in Section 18. PSA and a policy engine are complementary, not either/or.

Note: PSA is validation only — it can reject a Pod, but it will not fix it for you by injecting a SecurityContext. That's a deliberate design choice; mutating defaults is the job of a policy engine if you want it.

Recap

  • Containers share the host kernel, so a privileged/root container is a real escape risk — harden every workload.
  • SecurityContext is per-Pod/container hardening: runAsNonRoot, readOnlyRootFilesystem, allowPrivilegeEscalation: false, drop ALL capabilities, seccomp RuntimeDefault.
  • PodSecurityPolicy is gone (removed v1.25) — use Pod Security Admission, enabled by namespace labels at three levels (privileged/baseline/restricted) and three modes (enforce/audit/warn).
  • Roll PSA out with warn/audit before enforce. PSA validates only; it never mutates.
  • For risky/untrusted workloads, isolate harder with RuntimeClass (gVisor, Kata/Firecracker). For policy beyond the three levels, add Kyverno/Gatekeeper/ValidatingAdmissionPolicy.

17. Secrets Management

Apps need passwords, API keys, TLS private keys, and database credentials. Kubernetes has a dedicated object for this — the Secret — but there's a famous gotcha you must understand before you trust it with anything sensitive.

The Secret API (and the base64 trap)

A Secret is a small key/value object, namespace-scoped, that you can feed into Pods. You can supply values pre-encoded under data: (base64) or in plaintext under stringData: (Kubernetes encodes it for you):

apiVersion: v1
kind: Secret
metadata:
  name: db-creds
  namespace: billing
type: Opaque
stringData:                 # plaintext here; Kubernetes base64-encodes on write
  username: invoice_app
  password: s3cr3t-rotate-me

Key idea: base64 is encoding, NOT encryption. Anyone can run base64 -d and read the value instantly. The base64 in a Secret exists only so binary data survives being stored as text — it provides zero confidentiality. Never treat "it's base64'd" as "it's protected."

Consuming Secrets in a Pod

Two ways to get a Secret into a container:

# As environment variables — convenient, but riskier
env:
- name: DB_PASSWORD
  valueFrom:
    secretKeyRef:
      name: db-creds
      key: password
# As a mounted volume — preferred
volumeMounts:
- name: creds
  mountPath: /etc/db-creds
  readOnly: true
volumes:
- name: creds
  secret:
    secretName: db-creds
Environment variablesVolume (file mount)
Updates without restart− needs Pod restart+ files refresh in place (hot-reload)
Leak riskshows up in kubectl describe, crash dumps, child-process env, logs+ stays in an in-memory tmpfs file
RecommendationUse sparinglyPrefer this

Tip: Prefer volume mounts over env vars. Volumes hot-reload when the Secret changes (no Pod restart) and are far less likely to leak into logs, debug output, or a child process's environment.

Restrict access with RBAC and be especially careful with list/watch verbs on Secrets — they let a subject dump every Secret in a namespace, not just one.

Security note: Two blast-radius facts a bank engineer must internalize. (a) get reveals the full value — get on a Secret returns the complete data, not just metadata; there is no "see that it exists but not its contents." If a subject can get a Secret, it can read the password. (b) Pod creation = Secret read. Anyone who can create a Pod in a namespace can mount or reference any Secret in that namespace (via secretKeyRef/volume) and exfiltrate it through the Pod — even without direct get on the Secret. This makes the namespace a hard security boundary: design namespaces so that workloads which must not see each other's credentials live in separate namespaces, and treat "can create Pods here" as "can read every Secret here."

Encryption at rest: protect etcd

By default, Secrets are stored in etcd (the cluster's key-value database) unencrypted — only base64-encoded. Anyone with read access to etcd or a disk/backup snapshot can read every secret in the cluster. Fixing this is non-negotiable for production.

You enable encryption at rest with an EncryptionConfiguration on the API server. The strong, recommended form is envelope encryption via a KMS (Key Management Service):

apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
  - resources:
    - secrets
    providers:
    - kms:
        apiVersion: v2            # KMS v2 is the current provider
        name: myKmsPlugin
        endpoint: unix:///tmp/kms.sock
    - identity: {}                # fallback for reading old, unencrypted data

Note: Envelope encryption is "a key that encrypts other keys." A unique DEK (Data Encryption Key) encrypts each secret; a KEK (Key Encryption Key) that never leaves an external KMS (AWS KMS, GCP KMS, Azure Key Vault, Vault) encrypts the DEKs. So even someone with full etcd access only sees ciphertext, and the master key lives outside the cluster. Rotating the KEK is cheap.

2026 Update: Use the KMS v2 provider (apiVersion: v2) — it's GA and far more performant than v1 (it caches DEKs and rotates without re-encrypting everything). On managed clusters this is often a one-click toggle: e.g. EKS lets you turn on envelope encryption of Kubernetes Secrets with a KMS key at (or after) cluster creation.

Security note: For a regulated environment (e.g. a bank), etcd encryption at rest is effectively mandatory, not optional. Without it, every Secret — DB passwords, TLS keys, tokens — sits in etcd as plaintext-equivalent base64, so anyone with an etcd disk, volume snapshot, or backup tape can read them all. Enable KMS/envelope encryption so the data-encryption keys are wrapped by a key that never leaves an external KMS, and confirm it's on (kubectl get secret ... -o yaml against a raw etcd read should show ciphertext, not base64).

Beyond the built-in Secret: external secret managers

The Kubernetes Secret API is fine for basic needs, but it lacks enterprise features (rich rotation policies, HSM backing, central audit). The 2026 ecosystem leans on a dedicated secret manager (AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, GCP Secret Manager) as the source of truth, and bridges it into Kubernetes. Four common patterns:

ApproachHow it worksBest for
External Secrets Operator (ESO)An operator watches ExternalSecret custom resources and syncs values from an external manager into native Kubernetes Secrets, keeping them refreshed/rotated.The de-facto default in 2026; works with AWS/Vault/Azure/GCP.
Secrets Store CSI DriverA CSI driver mounts secrets straight from the external store into the Pod's tmpfs volume — they need never land in etcd at all.Avoiding etcd storage entirely; tight mount-time fetch.
Vault Agent InjectorA mutating webhook injects an init/sidecar container that authenticates to Vault and writes secrets into a shared in-Pod volume.Dynamic, short-lived secrets (e.g. DB creds issued per-Pod).
Sealed SecretsYou encrypt a Secret with the cluster controller's public key into a SealedSecret that is safe to commit to Git; only the in-cluster controller can decrypt it.GitOps — keeping (encrypted) secrets in your repo.

Security note: With Sealed Secrets, the sealing controller's private key is the single recovery point. Every SealedSecret in Git can only be decrypted by that key — so if you rebuild the cluster (or in a DR event) without restoring it, every committed SealedSecret becomes permanently undecryptable. Back up the controller's private key out-of-band (and treat it like the crown jewels). Also note SealedSecrets are, by default, scoped to a specific namespace + name — a sealed value can't be moved to a different namespace or renamed without re-sealing.

Here's the shape of an ExternalSecret (ESO) pulling a value from AWS Secrets Manager into a regular Secret:

apiVersion: external-secrets.io/v1
kind: ExternalSecret
metadata:
  name: db-creds
  namespace: billing
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: aws-secrets-manager        # a SecretStore/ClusterSecretStore you configured
    kind: SecretStore
  target:
    name: db-creds                   # the native Secret ESO will create/refresh
  data:
  - secretKey: password
    remoteRef:
      key: prod/billing/db
      property: password

Tip: On EKS, don't bake AWS credentials into ESO. Authenticate it to AWS Secrets Manager via EKS Pod Identity (or IRSA), and scope its RBAC tightly per namespace. Workload identity (covered in the Identity discussion) is what lets a Pod prove who it is without a static credential — solving the "secret to get the secrets" chicken-and-egg problem.

2026 Update: Older notes treat Vault sidecar injection or Sealed Secrets as the main answer. Today External Secrets Operator is the most common bridge, and Secrets Store CSI Driver is a strong choice when you want secrets to bypass etcd entirely. All of them still assume you've enabled etcd encryption at rest as the baseline — defense in depth.

Best practices

  • Never commit raw secrets to Git. Use Sealed Secrets or an external manager + ESO.
  • Never log secrets — logs flow through aggregators and live forever.
  • Prefer volume mounts over env vars (hot-reload, less leakage).
  • Enable etcd encryption at rest (KMS v2 envelope) on every real cluster.
  • One Secret per workload, tight RBAC, and watch out for list/watch.
  • Decouple your app from the provider: read secrets from a file/env, not by calling Vault's API directly — keeps the app portable.
  • Audit and rotate — turn on audit logging for Secret access and rotate both secrets and keys on a schedule.

Recap

  • A Secret's base64 is encoding, not encryption — by itself it protects nothing.
  • Secrets sit in etcd unencrypted by default; enable encryption at rest with KMS v2 envelope encryption in production.
  • Prefer volume mounts to env vars; lock down access with RBAC, especially list/watch.
  • For real workloads, source secrets from an external manager via External Secrets Operator (sync), Secrets Store CSI Driver (mount, skip etcd), Vault injector, or Sealed Secrets (GitOps).
  • Authenticate the bridge with workload identity (EKS Pod Identity/IRSA), never static keys, and never log or commit secrets.

18. Admission Control and Policy

We've reached the third gate from Section 15. After a request is authenticated and authorized, it passes through admission control before anything is written to etcd. This is where the cluster gets to say "yes, but first let me change this" or "no, that's not allowed here" — based on your organization's rules, not just basic RBAC.

The admission chain

Admission runs in a specific order, and the order matters:

  1. Mutating admission — controllers that may modify the incoming object (add a label, inject a sidecar, set a default).
  2. Schema validation — the API server checks the (now possibly mutated) object is structurally valid.
  3. Validating admission — controllers that accept or reject the final object, but cannot change it.

Key idea: Mutate first, validate last. Mutating controllers run before validating ones, so validation always sees the final object after all mutations. A single validating controller saying "no" rejects the whole request; admission is unanimous-to-allow.

Some admission controllers are built into the API server (e.g. the one that injects the default ServiceAccount token, or the NamespaceLifecycle controller that blocks creating objects in a namespace being deleted). The interesting power comes from extensible admission — your own policy.

Two ways to plug in policy

1. Admission webhooks. You register a MutatingWebhookConfiguration or ValidatingWebhookConfiguration telling the API server to call an external HTTPS service for certain operations. The service receives an AdmissionReview object and replies allow/deny (and, for mutating webhooks, a JSON patch).

apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingWebhookConfiguration
metadata:
  name: require-team-label
webhooks:
- name: require-team-label.example.com
  rules:
  - apiGroups: [""]
    apiVersions: ["v1"]
    operations: ["CREATE"]
    resources: ["pods"]
    scope: "Namespaced"
  clientConfig:
    service:
      namespace: policy-system
      name: policy-webhook
      path: /validate
    caBundle: <base64 CA cert>          # webhooks MUST be reachable over TLS
  admissionReviewVersions: ["v1"]
  sideEffects: None
  failurePolicy: Fail                    # if the webhook is down, reject (fail-closed)
  timeoutSeconds: 5

Two webhook design decisions to understand:

  • failurePolicy — Fail (reject the request if the webhook is unreachable — safe but can block the cluster if the webhook dies) vs Ignore (let the request through). Critical security webhooks usually want Fail, paired with high availability so they don't become a single point of failure.
  • Scope your rules tightly — every webhook adds latency to API calls, and a too-broad rule can accidentally intercept kube-system and break the cluster.
  • caBundle must be the CA that signed the webhook service's serving cert (so the API server trusts the TLS connection it makes to the webhook) — not just any CA. cert-manager's ca-injector commonly populates this field automatically. A wrong or missing caBundle is the #1 cause of webhook TLS failures (x509: certificate signed by unknown authority).

Avoid scoping admission webhooks to kube-system — a buggy or down webhook there can wedge core components.

2. Policy engines (no code). Writing and operating webhook servers is a lot of work. Policy engines let you express rules declaratively instead:

EngineLanguageNotes
KyvernoYAML policies (Kubernetes-native)No new language to learn; can validate, mutate, and generate resources. Very popular for K8s-specific rules.
OPA / GatekeeperRego + ConstraintTemplate CRDsPowerful, general-purpose policy engine; centralized audit; steeper learning curve (Rego). Validate-focused.

A Gatekeeper policy is split into a reusable ConstraintTemplate (the logic, in Rego) and a Constraint (an instance with parameters), so platform teams ship templates and app teams just parameterize them.

ValidatingAdmissionPolicy: the modern in-tree option

2026 Update: You no longer need an external webhook or a third-party engine for many validation rules. ValidatingAdmissionPolicy — which uses CEL (Common Expression Language) expressions evaluated inside the API server — went GA in v1.30. It's faster (no network hop), needs no extra components, and has no webhook to keep highly available. Reach for it first for straightforward validation; use Kyverno/Gatekeeper when you need mutation, generation, complex logic, or richer reporting. (A companion MutatingAdmissionPolicy, also CEL-based, is maturing for the mutation side.)

A policy that requires Pods to declare runAsNonRoot, with no webhook anywhere (a starting point — see the caveat below, it is not complete root-prevention; Pod Security Admission restricted is the real floor):

apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingAdmissionPolicy
metadata:
  name: "disallow-root-containers"
spec:
  matchConstraints:
    resourceRules:
    - apiGroups: [""]
      apiVersions: ["v1"]
      operations: ["CREATE", "UPDATE"]
      resources: ["pods"]
  validations:
  - expression: "!has(object.spec.securityContext) || !has(object.spec.securityContext.runAsNonRoot) || object.spec.securityContext.runAsNonRoot == true"
    message: "Pods must set spec.securityContext.runAsNonRoot=true"
---
apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingAdmissionPolicyBinding
metadata:
  name: "disallow-root-containers-binding"
spec:
  policyName: "disallow-root-containers"
  validationActions: ["Deny"]              # or Warn / Audit, just like PSA modes
  matchResources:
    namespaceSelector:
      matchLabels:
        environment: production

Note the !has(object.spec.securityContext) || guard: securityContext is optional, so reading object.spec.securityContext.runAsNonRoot directly throws at runtime when the field is absent — CEL must guard against missing fields with has(). Also remember that runAsNonRoot can be set per-container (spec.containers[*].securityContext.runAsNonRoot), not only at the Pod level, and a container can override the Pod default (e.g. with runAsUser: 0), so a thorough policy must check every container too. This example only checks the Pod level — treat it as a teaching skeleton, not a complete control. For real "no root" enforcement, label the namespace with Pod Security Admission restricted (§16), which covers these escape hatches for you.

Note: CEL is a small, safe expression language (think one-line boolean checks like object.spec.replicas <= 10). Because it runs in-process and can't do arbitrary I/O, it's fast and can't take your API server down the way a flaky webhook can.

Note: In CEL, accessing a field that isn't present raises an error and the whole evaluation fails — always wrap optional fields with has() (e.g. !has(object.spec.securityContext) || ...) before reading them.

How this fits with Pod Security Admission

PSA (Section 16) gives you three fixed levels. Admission policy is the layer for everything PSA can't express: "images only from registry.example.com," "every workload must carry a team label," "no LoadBalancer Services in dev." Use PSA for the universal pod-hardening floor, and a policy engine / ValidatingAdmissionPolicy for organization-specific rules on top.

Recap

  • Admission is the third gate: it runs after authn/authz and before persisting to etcd.
  • Order is mutating → schema validation → validating; one validating "no" rejects the request.
  • Extend it with webhooks (any language, but you operate a TLS service; mind failurePolicy and scope) or policy engines — Kyverno (YAML, can mutate) and OPA/Gatekeeper (Rego).
  • ValidatingAdmissionPolicy with CEL (GA 1.30) is the modern in-tree choice — no webhook, runs in the API server; prefer it for simple validation.
  • Layer it with PSA: PSA for the pod-security floor, admission policy for org-specific rules.

19. Software Supply Chain Security

Everything so far protects the running cluster. But attackers increasingly target the path before code runs — the software supply chain: your base images, dependencies, build pipeline, and registry. If a poisoned image gets built and deployed, RBAC and Pod Security won't help. Supply-chain security is about making sure that what you run is what you intended to run, built from trusted parts.

The chain has a few stages, and you can add a guard at each one: build small and clean → scan for vulnerabilities → sign and attest → verify at admission → deploy.

1. Build minimal, non-root images

A smaller image has a smaller attack surface — fewer packages means fewer vulnerabilities and less for an attacker to exploit.

  • Use minimal base images. Prefer distroless (no shell, no package manager, just your app + runtime libs) or scratch (literally empty — for static binaries). Avoid bloated full-OS bases.
  • Run as non-root — bake a USER into the image so it's self-documenting and consistent, then back it up with runAsNonRoot in the Pod (Section 16).
  • Use multi-stage builds so build tools (compilers, SDKs) never ship in the runtime image.
  • Pin dependencies (and ideally base images by digest, not tag) for reproducible, tamper-evident builds.
# Build stage — has the full Go toolchain
FROM golang:1.23 AS build
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -o /app ./cmd/server

# Runtime stage — distroless, non-root, no shell or build tools
FROM gcr.io/distroless/static:nonroot
USER nonroot:nonroot
COPY --from=build /app /app
ENTRYPOINT ["/app"]

Tip: Pin images by digest, not by tag. A tag like :1.4.0 can be repointed at new content; a digest (@sha256:abc123…) is immutable and is the bedrock of everything below — you can't meaningfully verify "the image" if "the image" can change underneath you.

2. Scan for vulnerabilities

Scanners compare your image's packages against CVE (Common Vulnerabilities and Exposures) databases:

  • Trivy — fast, popular, scans images, filesystems, and IaC; easy to drop into CI.
  • Clair — registry-integrated scanning (Harbor, Quay).
  • Falco / Aqua — runtime security: watch a running container for suspicious behavior, beyond static scanning.
trivy image --severity HIGH,CRITICAL registry.example.com/app@sha256:abc123...

Scan in CI (fail the build above a severity threshold), and re-scan on a schedule — a CVE disclosed after you built is still your problem. Prefer alerting and a controlled redeploy over silent auto-patching, which can break things or bypass your audit trail.

3. Sign images and attach provenance

Scanning tells you the image is probably safe. Signing tells you the image is genuinely yours and hasn't been tampered with.

2026 Update: Older notes describe Docker Content Trust / Notary v1 (TUF) as the signing mechanism — operationally heavy and tied to a single registry. The modern standard is Sigstore / cosign, which supports keyless signing (you authenticate via OIDC; a short-lived cert is issued and the signature is logged in a public transparency log — no long-lived private key to guard). It's vastly simpler and is what the ecosystem has standardized on.

# Keyless sign with cosign (identity proven via OIDC, recorded in a transparency log)
cosign sign registry.example.com/app@sha256:abc123...

# Verify the signature came from your expected CI identity
cosign verify registry.example.com/app@sha256:abc123... \
  --certificate-identity-regexp='https://github.com/example/.*' \
  --certificate-oidc-issuer='https://token.actions.githubusercontent.com'

Two more artifacts you attach as signed attestations alongside the image:

  • SBOM (Software Bill of Materials) — a complete inventory of every package and version in the image, generated by a tool like syft. When the next big CVE drops, an SBOM lets you instantly answer "are we affected?"
  • SLSA provenance — SLSA (Supply-chain Levels for Software Artifacts) is a framework describing how an artifact was built (which source, which builder, which steps). Targeting SLSA Level 2–3 gives tamper-resistant, verifiable build provenance; GitHub's built-in artifact attestation makes Level 2 easy to reach.
syft registry.example.com/app@sha256:abc123... -o spdx-json > sbom.json
cosign attest --predicate sbom.json --type spdxjson registry.example.com/app@sha256:abc123...

Note: cosign's flags and type names drift between versions — --type spdx/spdxjson, predicate handling, and keyless defaults have all changed across releases. Pin your cosign version in CI so the flow doesn't silently break on upgrade, and check cosign attest --help for your exact version. The SBOM-attest flow is roughly cosign attest --type spdx --predicate sbom.json <image@digest>.

4. Verify at admission time

This is where supply-chain security meets the cluster: an admission policy (Section 18) that rejects any image that isn't signed by you, lacks an SBOM, or fails provenance checks. This closes the loop — even if someone pushes a rogue image to your registry, the cluster won't run it.

Kyverno (or the Sigstore policy-controller, or cloud-native gates like AWS Signer / Binary Authorization) can enforce this. A Kyverno rule, in spirit:

apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: verify-image-signatures
spec:
  validationFailureAction: Enforce
  rules:
  - name: check-cosign-signature
    match:
      any:
      - resources:
          kinds: [Pod]
    verifyImages:
    - imageReferences:
      - "registry.example.com/*"
      attestors:
      - entries:
        - keyless:
            issuer: "https://token.actions.githubusercontent.com"
            subject: "https://github.com/example/*"

This pairs naturally with a quarantine workflow: vendor/external images land in a holding registry, get scanned and tested, and only signed, approved images get promoted to the registry the cluster is allowed to pull from.

5. Deploy via GitOps

Finally, deploy declaratively from Git (ArgoCD, Flux). Git becomes the audited, reviewable source of truth for what is deployed, and because manifests pin images by digest, the signed-and-verified artifact is exactly what lands in the cluster. (See the GitOps discussion in the Operations section.)

Key idea: Defense in depth across the chain: a minimal, non-root image (small attack surface) that's scanned (no known CVEs), signed + attested (authentic, with an SBOM and provenance), verified at admission (the cluster refuses anything else), and deployed from Git (auditable). Each layer covers the others' gaps.

Recap

  • The supply chain — base images, deps, build, registry — is a prime attack target; secure what you run before it runs.
  • Build minimal (distroless/scratch), non-root, multi-stage images and pin by digest, not tag.
  • Scan with Trivy/Clair (and watch runtime with Falco); fail builds on high-severity CVEs and re-scan on a schedule.
  • Sign with cosign/Sigstore (keyless, transparency-logged — not legacy Notary v1), attach an SBOM (syft) and SLSA provenance as attestations.
  • Verify signatures/attestations at admission (Kyverno / policy-controller / Binary Authorization) so the cluster runs only trusted images, and deploy via GitOps.

20. Observability: Metrics, Logs, and Traces

Once your apps are running on Kubernetes, you need to see what they're doing. Observability is the practice of building enough signals into your systems that you can answer questions like "Is this service healthy?", "Why is it slow?", and "What changed at 3 a.m.?" — without SSH-ing into a box and squinting at it.

Observability rests on three pillars:

PillarQuestion it answersTypical tool
Metrics"How much / how many / how fast?" (numbers over time)Prometheus + Grafana
Logs"What exactly happened, in order?" (event messages)Fluent Bit → Loki / Elasticsearch / CloudWatch
Traces"Where did this one request spend its time across services?"OpenTelemetry + Jaeger / Tempo

Key idea: Metrics tell you that something is wrong, logs tell you what the error was, and traces tell you where across your microservices the time went. You usually need all three.

20.1 The three metrics components — and why they are different

This is the single most-confused topic for newcomers, so let's nail it. Kubernetes metrics come from three different components that are easy to mix up because they all deal with "metrics". They do not overlap — each does its own job.

1. metrics-server — the lightweight "live vitals" source. metrics-server collects current CPU and memory usage of pods and nodes by scraping each node's kubelet. It keeps only the latest reading in memory (no history). It exists to power exactly two things:

  • kubectl top nodes / kubectl top pods (a quick "who's hot right now" view)
  • the Horizontal Pod Autoscaler (HPA), which needs current CPU/memory to decide whether to add pods
kubectl top pods -n my-app
# NAME              CPU(cores)   MEMORY(bytes)
# web-7d9f...       142m         210Mi

2026 Update: The old Heapster project is retired (deprecated back in v1.11, long since end-of-life). If older notes tell you to install Heapster (or cAdvisor + Heapster) to get kubectl top or to feed the HPA — that's obsolete. Use metrics-server. It exposes data via the metrics.k8s.io API.

2. kube-state-metrics (KSM) — the "object state" source. KSM does not look at CPU or memory. Instead it reads the Kubernetes API and turns the state of objects into metrics: how many replicas a Deployment wants vs. has, how many times a container has restarted, whether a Pod is Pending, how much quota a namespace has used, the status of a Job, and so on. These are the metrics you alert on ("3 pods stuck Pending", "container restarting in a loop").

Note: Easy way to remember the split — metrics-server = resource usage (the "vital signs": CPU/memory), kube-state-metrics = object health (the "paperwork": replica counts, restarts, conditions). They are complementary, not alternatives. A production cluster runs both.

3. Prometheus — the time-series database (TSDB) and brain. Neither metrics-server nor KSM stores history. Prometheus is the engine that actually records metrics over time and lets you query them. It works on a pull/scrape model: Prometheus periodically fetches a plain-text /metrics page from each target (your apps, KSM, the Node Exporter, the API server) and stores the numbers in its built-in time-series database. You query it with PromQL (Prometheus Query Language).

A typical Prometheus stack includes:

  • Prometheus Operator — manages Prometheus via Kubernetes custom resources so you configure scraping declaratively rather than editing config files. Key resources: ServiceMonitor / PodMonitor (what to scrape), PrometheusRule (alerting + recording rules), Alertmanager.
  • kube-prometheus — a ready-made bundle of the whole stack (Prometheus, Grafana, Alertmanager, dashboards) to get started fast.
  • Node Exporter — runs as a DaemonSet (one pod per node) to expose machine-level metrics (disk, network, load).
  • kube-state-metrics — exposes the object-state metrics described above.
  • Prometheus Adapter — bridges Prometheus metrics into the Kubernetes metrics API so the HPA can autoscale on custom application metrics (e.g. requests-per-second), not just CPU/memory.

Tip: Application metrics are exposed in two ways: native instrumentation (your code uses a Prometheus client library to publish a /metrics endpoint) or an exporter (a sidecar/external process that translates something else into Prometheus format, e.g. Node Exporter for hardware, or a MySQL exporter). For short-lived batch jobs that finish before they can be scraped, push metrics to a Pushgateway, which Prometheus then scrapes.

Prometheus metric types you'll meet: Counter (only goes up — total requests), Gauge (up or down — current memory), Histogram (observations in buckets — request durations), Summary (observations plus quantiles — 95th-percentile latency).

Watch cardinality: Labels let you slice metrics flexibly (status="500", path="/login"), but every unique label combination is a separate time series. High-cardinality labels (user IDs, request IDs) can blow up Prometheus memory. Keep label values bounded.

20.2 Scaling and long-term storage

Prometheus is excellent but is not built for long-term storage or huge fleets. Two patterns solve this:

  • Federation — multiple Prometheus servers each scrape a slice of the cluster, and a central Prometheus pulls a curated subset up for a global view.
  • Thanos / Cortex — add durable object-storage backends (e.g. S3), deduplication, and long retention on top of Prometheus.

20.3 Grafana — turning numbers into dashboards

Grafana is the visualization layer. It queries Prometheus (and other sources) and draws dashboards and graphs. Single-source dashboards are simplest to manage; federation gives a consolidated view at the cost of more overhead.

20.4 Alerting

Prometheus evaluates alerting rules against metrics and fires alerts to Alertmanager, which deduplicates, groups, and routes them to humans (Slack, email, PagerDuty).

Tip: Alert on metrics, not on raw log lines — log text changes constantly and is noisy, while metrics are stable. Good alerts include a description and a link to a runbook. Avoid alert fatigue (too many false alarms → people ignore the pager). Add a Dead Man's Switch: an alert that always fires during normal operation, so silence tells you the alerting system itself has died.

20.5 What to measure — USE, RED, and the Four Golden Signals

Don't measure everything blindly. These three frameworks tell you what matters:

FrameworkFocusMeasures
USE (resources)InfrastructureUtilization, Saturation, Errors of a resource (CPU, memory, disk, queues)
RED (services)Request-driven servicesRate (requests/sec), Errors, Duration (latency)
Four Golden Signals (SRE)Any user-facing systemLatency, Traffic, Errors, Saturation

Use USE for hardware/resources, RED for online services, the Golden Signals as a general SRE checklist. Layer app-specific / business metrics on top (e.g. checkout completion time) — these map directly to what the business cares about.

20.6 Logs — the logging pipeline

In containers, the rule is simple: applications log to stdout / stderr, not to files inside the container. The container runtime captures those streams, and a per-node agent ships them off the node. There are three log-shipping strategies:

  1. Application forwarding — the app sends logs straight to the backend. Rarely recommended (tight coupling, maintenance burden).
  2. Sidecar processing — a sidecar container in the pod processes logs from shared storage. Used only when logging to stdout/stderr isn't possible.
  3. Node-agent forwarding (recommended) — a log agent runs on every node (as a DaemonSet), collects all container logs, tags them with metadata (namespace, pod, labels), and forwards to a backend.

Note: A DaemonSet ensures exactly one copy of a pod runs on each node — perfect for "one agent per machine" jobs like log collectors and the Node Exporter.

Common tools: Fluent Bit (lightweight, written in C — the usual node agent) optionally feeding Fluentd (heavier, plugin-rich — a cluster-wide aggregator). The backend store is typically Loki (lightweight, pairs with Grafana), Elasticsearch (with Kibana for search), or CloudWatch Logs on AWS.

Tip: Prefer structured logs (JSON with fields like level, userID, transactionID) over free-text — they're far easier to query in Loki/Elasticsearch. Never log secrets, passwords, or full account/card numbers — logs flow through many systems and could expose them. Mask or tokenize sensitive values.

20.7 Audit logs — who did what

Kubernetes audit logs record every request to the API server: who, what verb, which object, and the result. They're essential for security forensics and compliance. You enable them with an audit policy that controls how much is recorded per request:

apiVersion: audit.k8s.io/v1
kind: Policy
rules:
  - level: None                 # don't log noisy proxy watches
    users: ["system:kube-proxy"]
    verbs: ["watch"]
  - level: Metadata             # log who/what, but not bodies
    resources:
      - group: ""
        resources: ["secrets", "configmaps"]
  - level: RequestResponse      # full detail for these groups
    resources:
      - group: "apps"

Levels escalate from None → Metadata (who/what only) → Request (+ request body) → RequestResponse (+ response body). Forward audit logs the same way as app logs (node agent → backend). For compliance (e.g. a bank), audit logs need long retention and tamper-evident forwarding to an external SIEM / log store so the security record can't be altered or lost on the cluster.

Note: Don't confuse audit logs with Kubernetes Events (kubectl get events). Events are short operational notes about objects (e.g. "pulled image", "scheduling failed"), stored in etcd and kept only ~1 hour. Audit logs are the durable security record.

Note: On a self-managed cluster, etcd backup/restore and encryption-at-rest (so Secrets aren't stored in plaintext in etcd) are core ops/security concerns — etcd holds all cluster state, so losing or leaking it is catastrophic. On managed services like EKS the control plane and etcd are AWS-managed (backup, encryption, HA), so this burden shifts to the provider (see §24).

20.8 Traces — following a request across services

In a microservices system, one user click can fan out across a dozen services. Distributed tracing stitches that journey together:

  • Trace = the whole request as it travels through all services.
  • Span = one unit of work within a service (a function call, a SQL query). Spans nest into a tree.
  • Tags = metadata attached to spans for context and search.

The application must be instrumented to emit spans, and the trace context (an ID in HTTP headers) must be propagated between services so the spans can be linked. A modern tracing pipeline is straightforward: the app (using the OpenTelemetry SDK) exports spans to an OpenTelemetry Collector (it batches, processes, and routes), which forwards them to a backend for storage and a query UI — e.g. Tempo, a Jaeger storage backend (Cassandra/Elasticsearch), or a vendor service. (Older Jaeger deployments ran a per-node Jaeger Agent; that component is deprecated/removed — the OpenTelemetry Collector is the recommended collection point now, so don't deploy a Jaeger Agent for new setups.)

2026 Update: Older notes reference OpenTracing and OpenCensus as the instrumentation standards. Those two projects merged into OpenTelemetry (OTel), which is now the vendor-neutral standard for traces, metrics, and logs. Instrument new apps with the OpenTelemetry SDK and ship via the OpenTelemetry Collector (which can also scrape kube-state-metrics and export to Prometheus/Loki/Tempo). Modern Kubernetes components can even emit OTLP traces themselves.

Try it: kubectl top pods -A to see live usage (needs metrics-server). Then kubectl get events -A --sort-by=.lastTimestamp | tail to see what just happened in the cluster.

Recap

  • Three pillars: metrics (numbers over time), logs (event messages), traces (per-request path across services).
  • metrics-server = live CPU/memory for kubectl top + HPA; kube-state-metrics = object state (replicas, restarts); Prometheus = the TSDB that stores history and runs PromQL. They don't overlap. Heapster is retired.
  • Visualize with Grafana; alert from Prometheus via Alertmanager (alert on metrics, add a Dead Man's Switch).
  • Pick what to measure with USE / RED / Four Golden Signals.
  • Log to stdout/stderr; ship with a node agent (Fluent Bit) to Loki / Elasticsearch / CloudWatch; keep audit logs for security; never log secrets.
  • Use OpenTelemetry for tracing (it superseded OpenTracing/OpenCensus).

21. Multitenancy, Resource Quotas, and Fairness

A cluster is rarely used by one team. Multitenancy means running multiple tenants — teams, apps, or business units — on the same Kubernetes platform. Kubernetes gives you the building blocks (namespaces, RBAC, quotas, network policies), but it deliberately does not define what a "tenant" is — that's your design decision.

21.1 The isolation spectrum: hard vs. soft multitenancy

There's no single on/off switch for isolation; it's a spectrum:

ModelTrust assumptionIsolationCost
Single-tenant clustersn/a — each tenant gets its own clusterStrongestHighest (a control plane per tenant)
Soft multitenancyTenants somewhat trust each other (e.g. internal teams)Namespaces + RBAC + quotas + NetworkPolicyLow
Hard multitenancyNo trust (e.g. external customers)All of the above + stronger boundaries (separate node pools, sandboxed runtimes, strict network deny)Medium–High

Key idea: Inside one cluster, **a namespace is a soft boundary, not a security sandbox.** It separates names and lets you attach policies (RBAC, quotas, NetworkPolicy), but pods in different namespaces still share the same nodes and kernel. For genuinely untrusted (hard) tenants you add stronger layers — or give them separate clusters.

Isolation happens at three layers: the control plane (RBAC, admission, API fairness), the workload plane (resource quotas/limits, NetworkPolicy, sandboxing), and platform services (per-tenant or routed logging/monitoring so tenants can't see each other's data).

Namespace strategies: namespace per team (simple, but limits large teams), namespace per application (scales well, easy per-app policy — common in big orgs), or namespace per tier (rare — most people use separate clusters for dev/staging/prod instead).

21.2 ResourceQuota — capping a namespace's total consumption

A ResourceQuota sets hard caps on the total resources a namespace may use — total CPU/memory requested, total storage, or counts of objects (Pods, Services, etc.). The admission controller rejects any request that would push the namespace over its cap. This stops one team from eating the whole cluster.

apiVersion: v1
kind: ResourceQuota
metadata:
  name: cpu-mem
  namespace: app1
spec:
  hard:
    requests.cpu: "4"
    requests.memory: 8Gi
    limits.cpu: "8"
    limits.memory: 16Gi
    pods: "50"
kubectl describe resourcequota cpu-mem -n app1   # see used vs. hard

Note: Once a namespace has a ResourceQuota that limits CPU/memory, Kubernetes **requires every pod in it to declare requests and limits** — otherwise the pod is rejected. That's where LimitRange helps.

Gotcha: Adding a CPU/memory ResourceQuota to an existing namespace can silently break things. Every Pod must now declare requests and limits, so any existing Deployment whose Pods lack them will fail to create new Pods — the admission controller rejects them. The confusing symptom is a "rollout stopped creating pods" (the Deployment looks stuck, kubectl describe rs shows quota-rejection errors). Fix it by setting a LimitRange with defaults in the namespace, or by adding requests/limits to the Pod specs.

21.3 LimitRange — per-object defaults and bounds

A ResourceQuota caps the namespace total; a LimitRange governs individual objects. It can set default requests/limits for containers that didn't specify any (so quota-enforced namespaces don't reject pods), and enforce min/max bounds per container so nobody asks for an absurd 64 cores.

apiVersion: v1
kind: LimitRange
metadata:
  name: defaults
  namespace: app1
spec:
  limits:
    - type: Container
      default:          # applied as the limit if none given
        cpu: 500m
        memory: 256Mi
      defaultRequest:   # applied as the request if none given
        cpu: 100m
        memory: 128Mi
      max:
        cpu: "2"
        memory: 1Gi

Tip: Think ResourceQuota = the namespace's whole budget; LimitRange = the rules and defaults for each line item. Use them together.

21.4 QoS classes — who gets evicted first

When a node runs low on memory, the kubelet must evict pods. How a pod set its requests/limits determines its Quality of Service (QoS) class, which decides eviction order:

QoS classHow you get itEviction priority
GuaranteedEvery container sets requests == limits for both CPU and memoryLast to be evicted (safest)
BurstableAt least one request set, but not fully matched to limitsEvicted after BestEffort
BestEffortNo requests or limits at allFirst to be evicted

Key idea: Memory is non-compressible — if a container exceeds its memory limit, it's killed (OOMKilled); you can't "throttle" memory the way you can CPU. CPU is compressible — over-limit CPU is throttled (slowed via the Linux CFS scheduler), not killed. So for latency-sensitive workloads, set memory carefully and consider setting CPU limit == request for predictability. (See the Workloads section for requests/limits basics.)

21.5 API Priority and Fairness — protecting the API server

Quotas protect worker node resources. But the API server itself is a shared resource — a misbehaving controller hammering it with requests could starve everyone. API Priority and Fairness (APF) classifies incoming requests into priority levels and fair-queues them, so a flood of low-priority requests can't drown out critical traffic (like the scheduler or node heartbeats).

2026 Update: Older notes call APF an alpha feature to be cautious with. It has since graduated to stable (GA, flowcontrol.apiserver.k8s.io/v1) and is on by default in modern Kubernetes. You no longer need to "evaluate multiple clusters instead" to get fairness — it's built in. You can still tune FlowSchema and PriorityLevelConfiguration objects.

21.6 Enforcing policy: admission control & pod security

To keep tenants honest, pair quotas with admission webhooks that enforce rules — require an owner label, mandate readiness/liveness probes, disallow NodePorts, or block images from untrusted registries. Tools: Kyverno (YAML policies) or OPA Gatekeeper (Rego policies).

2026 Update: Older multitenancy notes restrict pods using PodSecurityPolicy (PSP). PSP was removed in v1.25. The replacement is built-in Pod Security Admission (PSA), which enforces the Pod Security Standards (privileged / baseline / restricted) per namespace via labels:

``yaml apiVersion: v1 kind: Namespace metadata: name: app1 labels: pod-security.kubernetes.io/enforce: restricted pod-security.kubernetes.io/warn: restricted ``

For policy beyond those three profiles, use Kyverno, OPA Gatekeeper, or in-process ValidatingAdmissionPolicy (CEL, GA in v1.30). (See the Security section for full detail.)

For network isolation between tenants, start with a default deny-all NetworkPolicy per namespace and let tenants open specific paths (covered in the Networking section).

Recap

  • Namespaces are soft tenancy — good for organizing and attaching policy, but not a hard security wall; untrusted tenants need stronger isolation or separate clusters.
  • ResourceQuota caps a namespace's total resources/objects; LimitRange sets per-container defaults and min/max.
  • QoS classes (Guaranteed > Burstable > BestEffort) decide eviction order; memory over-limit = killed, CPU over-limit = throttled.
  • API Priority and Fairness protects the shared API server and is now stable/on by default (no longer alpha).
  • Enforce rules with admission control (Kyverno/Gatekeeper) and Pod Security Admission — PSP is gone.

22. Extending Kubernetes: CRDs, Operators, Helm, and Kustomize

Kubernetes isn't just a container orchestrator — at its core it's an extensible API with a control loop: you declare a set of resources, and controllers observe and act to make reality match. That design lets you teach Kubernetes about brand-new kinds of objects and package/customize the YAML you ship. This section covers four ways to extend and manage Kubernetes: CRDs, Operators, Helm, and Kustomize.

22.1 CRDs — teaching the API new nouns

A Custom Resource Definition (CRD) registers a brand-new resource kind with the Kubernetes API server. After you apply a CRD, you can kubectl get, create, and apply your custom objects just like built-in ones — they get stored in etcd, validated against a schema, and protected by RBAC, all for free.

apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: backups.example.com
spec:
  group: example.com
  scope: Namespaced
  names:
    kind: Backup
    plural: backups
    singular: backup
  versions:
    - name: v1
      served: true
      storage: true
      schema:
        openAPIV3Schema:
          type: object
          properties:
            spec:
              type: object
              properties:
                schedule: { type: string }
                retentionDays: { type: integer }

Now kind: Backup is a real, first-class API object. But a CRD by itself is just data storage — nothing happens when you create a Backup. To make it do something, you need a controller.

22.2 Custom controllers & the Operator pattern

A custom controller is a program that watches your custom resources and runs a reconciliation loop: observe desired state → compare to actual state → act to close the gap → repeat (the same loop built-in controllers use; see the Architecture section).

An Operator is the pattern of CRD + custom controller that encodes human operational knowledge — the "how to run this software" expertise (provisioning, failover, backups, upgrades) baked into code. Instead of a runbook a human follows, the Operator does it automatically.

Real-world Operators you'll meet:

  • cert-manager — you declare a Certificate custom resource; its controller obtains the TLS cert (e.g. from Let's Encrypt), stores it in a Secret, and auto-renews it before expiry.
  • CloudNativePG — you declare a Cluster resource; the Operator provisions a PostgreSQL cluster, handles replication, failover, and backups.
  • Prometheus Operator — you declare Prometheus, ServiceMonitor, Alertmanager resources (we used these in the Observability section).
  • Crossplane — declare cloud infrastructure (databases, buckets) as Kubernetes resources and let controllers reconcile them with the cloud provider.

Key idea: The "expanding brain" of Kubernetes deployment: raw Pods → Deployments/StatefulSets → Helm charts → CRDs + Operators. CRDs + Operators turn Kubernetes from a container scheduler into a general-purpose control plane for anything you can describe and reconcile.

Note: A CRD is the vocabulary (the new noun). The controller/Operator is the brain (what to do when one appears). You almost always install both together — usually via a single Helm chart or manifest provided by the project.

22.3 Helm — the package manager for Kubernetes

Real apps are many objects: Deployment + Service + ConfigMap + Ingress + Secret + ServiceAccount... Copy-pasting and hand-editing that YAML per environment is error-prone. Helm is the de-facto package manager + templating engine for Kubernetes.

Helm's vocabulary:

  • Chart — a versioned package of templated manifests. Contains Chart.yaml (metadata), values.yaml (default config), and a templates/ folder of YAML with placeholders.
  • Values — the configuration knobs. values.yaml holds defaults; you override per environment with --values prod.yaml or --set replicas=5.
  • Release — one installation of a chart into a cluster. Installing the same chart twice (e.g. staging and prod) gives two releases. Helm tracks each release's history so you can roll back.
helm repo add hashicorp https://helm.releases.hashicorp.com   # add a chart repo
helm install vault hashicorp/vault --values prod-values.yaml   # create a release
helm upgrade vault hashicorp/vault --values prod-values.yaml   # change it
helm rollback vault 1                                          # revert to revision 1

Templates use Go templating with built-in objects (.Chart, .Release, .Values), conditionals, loops, and named templates:

apiVersion: v1
kind: ConfigMap
metadata:
  name: {{ .Release.Name }}-config
data:
  appVersion: {{ .Chart.AppVersion }}
  environment: {{ .Values.environment | quote }}
  {{- range .Values.configData }}
  {{- if .enabled }}
  {{ .key }}: {{ .value | quote }}
  {{- end }}
  {{- end }}

Tip: Charts can be stored in classic Helm repos or in any OCI registry (the same registries that hold container images). helm install against an OCI URL works just like a container pull.

22.4 Building a Helm chart

Using someone else's chart (22.3) is easy. The real skill is authoring a chart for your application so anyone on the team can deploy it to any environment with one command. Here is how a chart actually works and how to build one that won't bite you at 2 a.m.

Scaffold it: helm create

helm create myapp        # generates a complete, working starter chart

That gives you the canonical layout — learn these five things and you understand 90% of Helm:

myapp/
├── Chart.yaml          # metadata: name, version, appVersion, dependencies
├── values.yaml         # default configuration (the knobs users override)
├── values.schema.json  # OPTIONAL but recommended: JSON Schema to validate values
├── templates/
│   ├── _helpers.tpl    # named templates (reusable name/label snippets) — not rendered itself
│   ├── deployment.yaml # templated manifests
│   ├── service.yaml
│   ├── NOTES.txt       # post-install message printed to the user
│   └── tests/
│       └── test-connection.yaml   # a `helm test` Pod
└── charts/             # vendored dependencies (subcharts) land here

Key idea: A chart is values + templates → rendered manifests. The template engine merges your values.yaml (overridden by --set/--values) with built-in objects and produces plain Kubernetes YAML. helm install then sends that YAML to the API server and records it as a Release it can upgrade or roll back. Templating is just text substitution that happens before anything touches the cluster — helm template shows you the exact output.

Chart.yaml — metadata and dependencies

apiVersion: v2                 # v2 = Helm 3+. (v1 was Helm 2; dependencies lived in requirements.yaml)
name: myapp
description: The VIB widget API
type: application              # "application" (deployable) or "library" (helpers-only, see below)
version: 1.4.2                 # the CHART's version — SemVer, bump on every chart change
appVersion: "2.7.0"            # the APP's version (your image tag) — quoted; purely informational
dependencies:
  - name: postgresql
    version: "15.x.x"
    repository: oci://registry-1.docker.io/bitnamicharts
    condition: postgresql.enabled   # subchart toggled by a values flag
    alias: db                       # refer to it as .Values.db.* and release-db

2026 update: Two version fields confuse everyone. version is the chart version (bump it whenever you change the chart); appVersion tracks the software the chart ships (your container tag) and is informational. Version the chart with SemVer against its values interface: MAJOR for breaking changes (renamed/removed values, changed resource names), MINOR for new optional values, PATCH for fixes. Also note: since Helm v3.3.2 unknown top-level keys in Chart.yaml are rejected — put custom metadata under annotations:.

values.yaml + values.schema.json — the configuration contract

values.yaml holds production-safe defaults; users override per environment. The schema turns silent typos into fast failures:

# values.yaml
replicaCount: 2
image:
  repository: registry.vib/widget-api
  tag: ""            # default to .Chart.AppVersion when empty (see template below)
resources:
  requests: { cpu: 100m, memory: 128Mi }
  limits:   { cpu: "1",  memory: 512Mi }
postgresql:
  enabled: true
// values.schema.json — helm lint/install validate values against this (JSON Schema draft-07)
{
  "$schema": "https://json-schema.org/draft-07/schema#",
  "type": "object",
  "required": ["image"],
  "properties": {
    "replicaCount": { "type": "integer", "minimum": 1 },
    "image": {
      "type": "object",
      "required": ["repository"],
      "properties": { "repository": { "type": "string" }, "tag": { "type": "string" } }
    }
  }
}

Tip: If more than one person edits values.yaml, a schema is effectively mandatory — without it Helm silently ignores a mis-keyed replicaCnt: 5 and deploys the default. The schema also documents the chart's interface and blocks malformed input.

Templates: Go templating, helpers, and the standard labels

Templates are YAML with {{ ... }} actions. You get built-in objects — .Values (merged config), .Release (.Name, .Namespace, .Revision, .IsUpgrade), .Chart (from Chart.yaml), .Capabilities (cluster/API versions), and .Files (non-template files in the chart). Repeated names/labels go in _helpers.tpl as named templates:

{{/* templates/_helpers.tpl */}}
{{- define "myapp.fullname" -}}
{{- printf "%s-%s" .Release.Name .Chart.Name | trunc 63 | trimSuffix "-" -}}
{{- end -}}

{{- define "myapp.labels" -}}
helm.sh/chart: {{ printf "%s-%s" .Chart.Name .Chart.Version }}
app.kubernetes.io/name: {{ .Chart.Name }}
app.kubernetes.io/instance: {{ .Release.Name }}
app.kubernetes.io/version: {{ .Chart.AppVersion | quote }}
app.kubernetes.io/managed-by: {{ .Release.Service }}
{{- end -}}
# templates/deployment.yaml — pulling it together
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ include "myapp.fullname" . }}
  labels: {{- include "myapp.labels" . | nindent 4 }}
spec:
  replicas: {{ .Values.replicaCount }}
  template:
    spec:
      containers:
        - name: {{ .Chart.Name }}
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag | default .Chart.AppVersion }}"
          resources: {{- toYaml .Values.resources | nindent 12 }}

Note: Four functions do most of the work. include renders a named template and returns a string (use it, not template, so you can pipe the result); required "msg" .Values.x fails the render with a clear message if a value is missing; default supplies a fallback; and toYaml … | nindent N dumps a values block as correctly-indented YAML. The - in {{- / -}} trims surrounding whitespace so your output isn't full of blank lines. Use app.kubernetes.io/* labels on everything — dashboards, kubectl get -l, and network policies all rely on them.

Dependencies, subcharts, and library charts

Declare dependencies in Chart.yaml, then vendor them into charts/:

helm dependency update myapp     # reads Chart.yaml deps → downloads .tgz into charts/
  • condition toggles a subchart from a values flag (postgresql.enabled); alias lets you include the same chart twice under different names; global values are visible to parent and all subcharts; import-values lifts a subchart's values up to the parent.
  • A library chart (type: library) ships only _helpers.tpl-style templates, no manifests. It's how you stop copy-pasting the same Deployment/Service boilerplate across 20 microservice charts — extract the shared templates once and include them. Rule of thumb: the third time you paste the same YAML block, make it a helper or a library chart.

A hook is an ordinary manifest annotated to run at a specific moment (e.g. a DB migration Job before an upgrade):

    "helm.sh/hook": pre-upgrade,pre-install
    "helm.sh/hook-weight": "5"                       # lower runs first
    "helm.sh/hook-delete-policy": before-hook-creation

23. Delivering Software: GitOps and Deployment Strategies

You've built and packaged your app — now how does it actually reach the cluster, reliably and safely? This section covers how changes get applied (push vs. pull / GitOps) and how new versions roll out without taking down your users (deployment strategies).

23.1 Push vs. pull continuous delivery

There are two fundamentally different ways a change lands in your cluster:

Push-based CD (the traditional way). A central CI/CD pipeline (Jenkins, GitLab CI, GitHub Actions, Tekton) has cluster credentials and pushes the change in — typically kubectl apply or kubectl set image after building the new image.

sed -i 's#image:.*#image: myapp:v1.2.3#' deployment.yaml
kubectl apply -f deployment.yaml      # pipeline pushes to the cluster
  • Good for: a single cluster, reusing an existing CI/CD setup.
  • Painful at scale: the pipeline must hold credentials to every cluster (a security and management headache), and if a push half-fails, the cluster can drift from what you intended with nothing to correct it.

Pull-based CD / GitOps (the modern way). An agent inside the cluster continuously pulls the desired state from Git and reconciles the cluster to match. The pipeline never needs cluster credentials — it just commits to Git.

23.2 GitOps — Git as the single source of truth

GitOps treats a Git repository as the one source of truth for what should be running. All manifests (plain YAML, Helm charts, Kustomize overlays) live in Git. An in-cluster GitOps controller watches that repo and runs a reconciliation loop: observe Git → compare to the live cluster → apply the difference → repeat. It's the same declarative control loop Kubernetes uses internally, extended out to your whole delivery process.

Why teams love it:

  • Declarative & self-healing — if someone hand-edits the cluster (kubectl edit) or a resource is accidentally deleted, the controller notices the drift and restores the Git-declared state.
  • Git is the audit log — every change is a commit: who, what, when, reviewed via pull request. Instant history and easy rollback (git revert).
  • No cluster creds in CI — the pull agent lives in the cluster; your pipeline only writes to Git. Big security win, especially across many clusters.
  • Scales to many clusters — point each cluster's agent at the right repo/path.

Common tools:

  • Argo CD — popular, with a rich web UI, app grouping, sync status, and automated rollback.
  • Flux — lightweight, Kubernetes-native, integrates cleanly with Helm and Kustomize.

The secrets problem: You must never commit plaintext Secrets to Git. Use Sealed Secrets (encrypt with a public key; only the in-cluster controller can decrypt) or the External Secrets Operator (syncs from AWS Secrets Manager / Vault / etc. into real Secrets). Also watch out for dynamic resources created by in-cluster controllers/Operators — a strict GitOps sync may try to delete things that aren't in Git, so scope what the controller manages.

Note: GitOps doesn't replace your build pipeline — you still build and push images in CI. GitOps replaces the deploy step: instead of CI running kubectl apply, CI (or a bot) updates the image tag in Git, and the GitOps controller does the applying.

23.3 Deployment strategies — rolling out a new version safely

Getting a new version applied is one thing; switching live traffic to it without breaking users is another. The basic Kubernetes Deployment does a RollingUpdate (gradually replace old pods with new, controlled by maxSurge/maxUnavailable) or Recreate (kill all old, then start new — causes downtime). Beyond those, three progressive strategies reduce risk:

StrategyHow it worksBest forTrade-off
CanarySend a small % of traffic to the new version, watch metrics, gradually increaseValidating a release in prod with minimal blast radiusNeeds traffic-splitting + good metrics
Blue/GreenRun new version (Green) alongside old (Blue); flip all traffic at once after testing GreenInstant cutover + instant rollback (flip back to Blue)Needs double the infrastructure briefly
A/B testingRoute specific users (by header/geo/cookie) to the new version; compare behaviorFeature experiments, data-driven product decisionsRouting by attributes needs a service mesh

Key idea: Canary, blue/green, and A/B differ in how they split traffic. Canary splits by percentage, blue/green is an all-at-once switch, A/B splits by who the user is. All three let you validate before fully committing, and roll back gracefully.

23.4 Progressive delivery — automating the rollout

Doing canaries by hand is tedious and risky. Progressive delivery automates it: a controller gradually shifts traffic and watches health metrics (error rate, latency from Prometheus), automatically promoting the release if metrics stay healthy or rolling back if they degrade — no human babysitting the dashboard.

Tools:

  • Flagger — automates canary/blue-green/A-B by integrating with a service mesh (Istio, Linkerd) or ingress and analyzing metrics to decide promote-vs-rollback.
  • Argo Rollouts — a Kubernetes controller that replaces the standard Deployment with a Rollout resource supporting fine-grained canary steps, analysis, and automated rollback, pairing naturally with Argo CD.

Tip: Start simple. A small team on one cluster can get a long way with a plain RollingUpdate and push-based kubectl apply. Add GitOps when you have multiple clusters or strict change-management/compliance needs, and add progressive delivery (Flagger/Argo Rollouts) once you have a service mesh and solid metrics to drive automated promotion. Don't adopt all the machinery on day one.

Recap

  • Push CD: a central pipeline pushes changes (kubectl apply) — simple for one cluster, but holds creds everywhere and drifts silently.
  • GitOps (pull CD): an in-cluster agent (Argo CD / Flux) reconciles the cluster to Git as the source of truth — self-healing, auditable, no cluster creds in CI. Keep secrets out of Git (Sealed Secrets / External Secrets).
  • Rollout strategies: Canary (split by %), Blue/Green (instant switch + rollback), A/B (split by user) — plus the built-in RollingUpdate.
  • Progressive delivery (Flagger, Argo Rollouts) automates traffic shifting + metric analysis to promote or roll back without a human in the loop.
  • Adopt incrementally: simple deploys first, GitOps and progressive delivery as scale and risk grow.

24. Running Kubernetes on AWS with EKS

So far you've learned Kubernetes as if you were running it yourself. In the real world, most teams don't operate the control plane by hand — they let a cloud provider do it. On AWS, that managed service is EKS (Elastic Kubernetes Service). This section shows you how a Kubernetes cluster maps onto AWS building blocks: who runs what, how networking and load balancing work, how pods get AWS permissions, and how storage is wired up.

Key idea: EKS doesn't change Kubernetes. The API, kubectl, Pods, Deployments, Services — all identical to what you've already learned. EKS just runs the control plane for you and bolts Kubernetes onto AWS's networking, IAM, storage, and load balancers.

24.1 The shared-responsibility model

A Kubernetes cluster has two halves (see the Architecture section): the control plane (API server, etcd, scheduler, controller managers) and the worker nodes (kubelet, kube-proxy, container runtime, your pods). EKS splits ownership cleanly:

LayerWho runs itWhat it means for you
Control plane (API server, etcd, scheduler, controllers)AWSYou never SSH into it, patch it, or back up etcd. AWS runs it across 3 AZs and gives you a stable API endpoint. You pay a flat hourly fee per cluster.
Worker nodes / computeYou (unless you use EKS Auto Mode or Fargate)You choose instance types, scaling, AMIs, and pay EC2 prices.
Your workloads (Pods, Services, etc.)YouSame kubectl/YAML you already know.

Note: "Managed control plane" means AWS hides the master nodes from you entirely. You can't kubectl get pods -n kube-system and see the API server — it lives in an AWS-owned account. You only see and manage your worker nodes and workloads.

24.2 EKS networking: the VPC, subnets, and the magic tags

EKS runs inside a VPC (Virtual Private Cloud) — your own isolated network in AWS. A production EKS VPC typically spans at least two Availability Zones (AZs) for resilience, and each AZ has two subnets:

  • Public subnets — route to an Internet Gateway (IGW). They host internet-facing load balancers and a NAT Gateway.
  • Private subnets — route outbound traffic through the NAT Gateway (so nodes can pull images and reach AWS APIs) but are not directly reachable from the internet. Your worker nodes live here.

A typical route table layout:

Subnet typeDestinationTarget
Public10.0.0.0/16local
Public0.0.0.0/0Internet Gateway
Private10.0.0.0/16local
Private0.0.0.0/0NAT Gateway

By default, pods get real VPC IP addresses via the Amazon VPC CNI plugin — a pod is a first-class citizen on your VPC network, not hidden behind an overlay. (The CNI, Container Network Interface, is the pluggable standard for pod networking — see the Networking section.)

Subnet tags — the part everyone forgets

The AWS Load Balancer Controller (below) needs to know which subnets to place load balancers in. It learns this from subnet tags:

TagPut it onMeaning
kubernetes.io/role/elb = 1Public subnets"Place internet-facing load balancers here."
kubernetes.io/role/internal-elb = 1Private subnets"Place internal (private) load balancers here."

2026 Update: You should also tag subnets with kubernetes.io/cluster/<cluster-name> when sharing a VPC, but EKS no longer requires the per-cluster subnet tag for the cluster to function in newer versions — the ELB role tags above are the ones that bite people when load balancers fail to provision.

Tip: If a Service type: LoadBalancer or an Ingress sits forever with EXTERNAL-IP: <pending>, the number-one cause is missing or wrong kubernetes.io/role/elb subnet tags.

24.3 Where do pods run? Node groups vs Fargate vs Karpenter vs Auto Mode

EKS gives you several ways to provide compute. You'll meet all four:

OptionWhat it isWhen to use
Managed node groupsEC2 instances in an Auto Scaling Group, lifecycle managed by EKS (graceful drain on upgrade). You pick instance type + min/max/desired size.The classic, predictable choice. Good default.
Self-managed nodesYou build and manage the ASG / AMIs yourself.Niche — custom OS, special bootstrap.
AWS FargateServerless: each pod runs in its own tiny micro-VM. No nodes to manage.Bursty / CI / isolated workloads. 2026 Update — Karpenter has largely replaced the Cluster Autoscaler. The old Cluster Autoscaler scales fixed-size ASGs and takes ~3–4 minutes to add a node. Karpenter picks an optimal instance type for the pending pods and brings a node up in ~45–60 seconds, natively uses Spot, and consolidates underutilized nodes. Use Cluster Autoscaler only for tightly-pinned AMI control, fixed GPU pools, or non-AWS portability.

2026 Update — EKS Auto Mode (launched late 2024). Auto Mode is AWS running Karpenter + Bottlerocket nodes + core add-ons (EBS CSI, the AWS Load Balancer Controller) for you, off-cluster. There are no node groups or ASGs to manage; the only meaningful on-cluster system pod is metrics-server. You give up custom AMIs (run host agents as DaemonSets instead) but get near-zero node operations. For a small team, Auto Mode is the easiest way to get a production-grade cluster.

kubectl get pods --field-selector=status.phase=Pending

Your pods often need to call AWS — read an S3 bucket, write to DynamoDB, fetch a secret. You do not bake AWS access keys into images. Instead you attach an IAM role to a Kubernetes ServiceAccount, and pods using that ServiceAccount get short-lived AWS credentials automatically. There are two mechanisms:

# A ServiceAccount wired to an IAM role via IRSA (the annotation is the link):
apiVersion: v1
kind: ServiceAccount
metadata:
  name: s3-reader
  namespace: app

    eks.amazonaws.com/role-arn: arn:aws:iam::123456789012:role/s3-read-role

Note — debugging AccessDenied / sts:AssumeRoleWithWebIdentity failures. When a pod logs AccessDenied or is not authorized to perform sts:AssumeRoleWithWebIdentity (IRSA) / AssumeRoleForPodIdentity (Pod Identity), it almost always means the pod never got AWS creds. Check, in order: (1) the ServiceAccount annotation eks.amazonaws.com/role-arn is present and the pod actually uses that SA (kubectl get pod <p> -o jsonpath='{.spec.serviceAccountName}'); (2) for Pod Identity, an association exists (aws eks list-pod-identity-associations --cluster-name <c>); (3) the IAM role trust policy allows the right principal — IRSA trusts the cluster OIDC provider with the SA sub, Pod Identity trusts pods.eks.amazonaws.com; (4) the role's permission policy actually grants the API call being denied. A missing annotation/association means no creds at all; a present one with AccessDenied on a specific action means the permission policy is too narrow.

Separately, your human IAM users/roles are mapped to Kubernetes RBAC groups so that AWS identities can run kubectl. An IAM user needs eks:DescribeCluster to fetch kubeconfig; their RBAC group is then bound to a Role/ClusterRole inside the cluster (see the Security/RBAC section).

24.5 Load balancing: the AWS Load Balancer Controller (ALB & NLB)

Out of the box, the legacy in-tree cloud provider only knows how to create old Classic Load Balancers. For modern load balancers you install the separately-maintained AWS Load Balancer Controller, which:

  • Creates an ALB (Application Load Balancer, Layer 7) when you create an Ingress.
  • Creates an NLB (Network Load Balancer, Layer 4) when you create a Service type: LoadBalancer with the right annotation.
  • Registers pod IPs directly into target groups ("IP mode"), so traffic skips the extra NodePort hop.
  • Ships independently of the Kubernetes release cycle and implements new AWS features.

24.6 Storage: EBS (RWO) vs EFS (RWX) via CSI

EKS provisions storage through CSI (Container Storage Interface) drivers — the standard plug-in model (see the Storage section). The in-tree volume plugins are gone.

2026 Update: The old in-tree awsElasticBlockStore volume type is removed. CSI migration went GA in v1.25 — you must install the EBS CSI driver (and EFS CSI driver if needed), ideally as EKS-managed add-ons. The EBS CSI driver also needs IAM permissions, granted via IRSA/Pod Identity.

Note: "RWO" trips people up. An EBS volume can be mounted by only one node at a time, so a Deployment with 3 replicas spread across nodes will have 2 pods stuck unable to attach. Use a StatefulSet (one EBS volume per pod) or switch to EFS (RWX) if many pods must share one filesystem.

EFS gotcha #1 — the mount just hangs. If an EFS-backed pod sits forever in ContainerCreating with a mount timeout, the cause is almost always the mount-target security group not allowing NFS (TCP 2049) from the node security group. EFS is reached over NFSv4 on port 2049 via per-AZ mount targets; if the mount-target SG doesn't permit 2049 inbound from your nodes' SG, the mount blocks silently. Fix the SG rule first. Also note dynamic provisioning with the EFS CSI driver needs the EFS fileSystemId (and, for access point mode, the access-point config) in the StorageClass parameters — EFS isn't carved per-PVC like EBS, so the StorageClass points at an existing file system: ``yaml apiVersion: storage.k8s.io/v1 kind: StorageClass metadata: { name: efs-sc } provisioner: efs.csi.aws.com parameters: provisioningMode: efs-ap # dynamic access-point provisioning fileSystemId: fs-0123456789abcdef0 # the existing EFS file system directoryPerms: "700" ``

25. kubectl Mastery and Debugging

kubectl is your steering wheel for any cluster — EKS, GKE, AKS, or a laptop kind cluster. Get fluent with it and 80% of daily Kubernetes work becomes muscle memory. This section is the field manual.

25.1 The verb–noun mental model

Almost every kubectl command reads as verb + noun, optionally scoped by namespace and shaped by output format:

kubectl <VERB> <NOUN> -n <NAMESPACE> -o <FORMAT>
#         get        pods   -n app      -o wide
  • Verbs: get, describe, create, apply, delete, edit, logs, exec, scale, rollout, explain, diff, top, debug, port-forward.
  • Nouns (resource types): pods, deployments (deploy), services (svc), nodes, replicasets (rs), configmaps (cm), secrets, ingress, pvc, … most have short aliases.

Tip: Alias k=kubectl and enable shell completion. Together they cut keystrokes dramatically. ``bash alias k=kubectl source <(kubectl completion bash) # zsh: kubectl completion zsh ``

25.2 The everyday verbs

Inspect: get and describe

kubectl get pods                      # list pods in current namespace
kubectl get pods -A                   # all namespaces (alias for --all-namespaces)
kubectl get pods -o wide              # add node, pod IP columns
kubectl get pod my-pod -o yaml        # full manifest as stored
kubectl get pods --watch              # stream live changes
kubectl get pods,svc                  # multiple types at once
kubectl describe pod my-pod           # details + Events (your #1 debugging command)

get gives you the summary table; describe gives you the Events at the bottom — scheduling decisions, image pulls, probe failures. When something is wrong, describe is almost always your first stop.

Read logs: logs

kubectl logs my-pod                          # current logs
kubectl logs my-pod -f                        # follow (stream)
kubectl logs my-pod -c sidecar                # a specific container
kubectl logs my-pod --previous                # the crashed previous instance
kubectl logs deployment/my-app                # logs from a deployment's pods
kubectl logs -l app=foo --max-log-requests=10 # logs from ALL pods matching a label

Tip — logs across replicas. kubectl logs -l app=foo tails every matching pod; raise --max-log-requests (default 5) when you have many replicas, e.g. --max-log-requests=10. For serious multi-pod tailing with colour-coded pod names and live follow, install stern (stern app=foo) — it's the field standard.

Tip: --previous is gold for CrashLoopBackOff — it shows why the last container died, which the current (restarting) container's logs won't.

Get inside: exec, port-forward, cp

kubectl exec -it my-pod -- bash               # shell into the (default) container
kubectl exec -it my-pod -c app -- sh          # a specific container
kubectl port-forward my-pod 8080:80           # tunnel localhost:8080 -> pod:80
kubectl port-forward svc/my-svc 8080:80       # via a service (still hits one pod)
kubectl cp my-pod:/var/log/app.log ./app.log  # copy a file out

Apply, diff, dry-run: the GitOps-friendly verbs

kubectl apply -f app.yaml                      # create or update (declarative)
kubectl diff -f app.yaml                        # preview what apply would change
kubectl apply -f app.yaml --dry-run=server      # validate without persisting
kubectl delete -f app.yaml                      # delete what the file declares

Key idea: Prefer apply (declarative — describe the end state, let Kubernetes reconcile) over imperative create/run/edit. apply only changes the fields you changed, plays nicely with version control, and is the foundation of GitOps. Use kubectl diff before every apply to see exactly what will change.

Discover fields: explain

You don't have to memorize YAML schemas — ask the cluster:

kubectl explain pod.spec.containers
kubectl explain pod.spec.containers.resources.limits
kubectl explain deployment.spec.strategy --recursive

25.3 Contexts, kubectx, and kubens

A context bundles cluster + user + default namespace, stored in ~/.kube/config. Switching contexts is how you move between dev/prod or between clouds.

kubectl config get-contexts                     # list known contexts
kubectl config current-context                  # which am I on?
kubectl config use-context prod-eks             # switch cluster
kubectl config set-context --current --namespace=app   # change default namespace

Tip — install kubectx and kubens. They make switching effortless and are universal in the field: ``bash kubectx prod-eks # switch context (cluster) kubens app # switch default namespace ` Running the wrong command in the wrong cluster is a classic incident. kubectx/kubens` (and showing the context in your shell prompt) save you from it.

25.4 Live debugging: kubectl debug and ephemeral containers

Sometimes you need to poke at a running pod that has no shell — a distroless or scratch image with no bash, no curl, nothing. You can't exec into a shell that doesn't exist. The answer is an ephemeral container: a temporary, throwaway container that the API server injects into the running pod, sharing its namespaces, without restarting it.

2026 Update: Ephemeral containers are GA (since v1.25) and kubectl debug is the standard live-debugging tool. This is newer than many tutorials — learn it; it replaces awkward kubectl cp + image rebuild workflows.

# Attach a debug container (with tools) to a running, shell-less pod:
kubectl debug -it my-pod --image=busybox --target=app -- sh
# --target=app shares process namespace with container "app"

# Debug a crashing pod by making a COPY with a changed command:
kubectl debug my-pod -it --copy-to=my-pod-debug --container=app -- sh

# Debug a NODE by launching a host pod on it (sysadmin profile = host PID/network/privileges):
kubectl debug node/ip-10-0-1-23 -it --image=ubuntu --profile=sysadmin

Note: Pass --profile=sysadmin for node debugging — without it the debug pod can't see the host's processes, network, or filesystem the way you need. The profiles are general (default), netadmin (host networking / NET_ADMIN for network debugging), and sysadmin (host PID + network + privileged). This matters especially in PSA-restricted namespaces, where the default profile is blocked and you must use a profile (and often a namespace that permits it) for the privileged debug pod to even start.

25.5 jsonpath: extract exactly what you need

-o jsonpath pulls a single field out of an object — perfect for scripts and one-liners:

# A pod's IP
kubectl get pod my-pod -o jsonpath='{.status.podIP}'

# Name of the first pod matching a label
kubectl get pods -l app=alpaca -o jsonpath='{.items[0].metadata.name}'

# Every node's name, one per line
kubectl get nodes -o jsonpath='{range .items[*]}{.metadata.name}{"\n"}{end}'

# All image names running in the namespace
kubectl get pods -o jsonpath='{.items[*].spec.containers[*].image}'

25.7 K9s: a terminal UI for navigating and monitoring clusters

kubectl is precise but verbose — every check is a fresh command, and watching anything means re-running it or piping to watch. K9s is a full-screen terminal UI that sits on top of the same Kubernetes API: it gives you a live, auto-refreshing, keyboard-driven view of your resources, with one-keystroke access to logs, describe, YAML, shell, and port-forward. For monitoring and day-to-day operations it is dramatically faster than typing kubectl repeatedly.

Key idea: K9s is just a client. It reads your kubeconfig and acts entirely within your RBAC — it can do nothing your credentials couldn't already do via kubectl. It adds speed and visibility, not privileges. So "is K9s safe?" reduces to "what does my identity allow?" plus "is the binary genuine?" (verify the checksum from the official release).

Install (single static binary, no daemon):

# Linux amd64 — pin a version and verify the checksum (don't blindly curl|latest)
curl -fsSL https://github.com/derailed/k9s/releases/download/v0.51.0/k9s_Linux_amd64.tar.gz -o /tmp/k9s.tgz
echo "<published-sha256>  /tmp/k9s.tgz" | sha256sum -c -
mkdir -p ~/bin && tar -xzf /tmp/k9s.tgz -C ~/bin k9s     # ~/bin keeps it per-user on shared hosts
k9s version

Launch:

k9s                       # uses current context from ~/.kube/config
k9s -n my-app             # start scoped to a namespace
k9s --context prod-eks    # pick a context explicitly (skips the picker)
k9s --readonly            # disable all edit/delete/exec/scale — recommended for prod & shared boxes
k9s --kubeconfig ~/.kube/config-om   # use a specific kubeconfig

26. Troubleshooting Playbook

When a workload misbehaves, resist the urge to randomly restart things. Kubernetes problems almost always sit in one of a few layers, so debug bottom-up: Pod → Node → Service → Ingress. Confirm the pods are actually running and ready first (and that their node is healthy); only then check whether the Service routes to them; only then check the Ingress.

26.1 The decision tree

START
  │
  ↓
Are the Pods Running AND Ready (e.g. 1/1)? ──No──→ Diagnose the POD → 26.2

26.2 Pod-level problems

Pending — the pod can't be scheduled

kubectl get pods shows 0/1 Pending. The scheduler can't find a node. Run kubectl describe pod <name> and read the Events. Common causes:

CauseHow to confirmFix
Not enough CPU/memory in clusterEvent: 0/3 nodes available: insufficient cpuLower requests, add nodes (Karpenter/Auto Mode helps), or right-size.
ResourceQuota in the namespacekubectl describe quota -n <ns>Raise quota or reduce requests.
Unbound PVCkubectl get pvc shows PendingFix StorageClass / provisioner (e.g. EBS CSI not installed).
Affinity / taints / nodeSelectorEvent mentions taints/selectorsAdd tolerations, fix labels, or relax affinity.
(EKS) No free VPC IPs — subnet/ENI exhaustedPod stuck ContainerCreating; event failed to assign an IP address to containerUse a bigger/extra subnet, enable prefix delegation on the VPC CNI, or set WARM_IP_TARGET/MINIMUM_IP_TARGET.

ImagePullBackOff / ErrImagePull — kubelet can't pull the image

kubectl get pods shows 0/1 ImagePullBackOff. describe shows the pull error. Causes:

  1. Typo in the image name (ngnix not nginx).
  2. Tag doesn't exist (:v9.9.9).
  3. Private registry without credentials — needs an imagePullSecret (or, on EKS, an IAM/ECR permission via Pod Identity/IRSA — if the app itself gets AccessDenied/sts:AssumeRoleWithWebIdentity once running, that's a credentials wiring problem, see the IRSA/Pod Identity troubleshooting note in §24.4).
kubectl describe pod my-pod | grep -A3 Events
# ...Failed to pull image "myrepo/app:v2": not found / unauthorized

Tip: "ImagePullBackOff" vs "ErrImagePull": the first means kubelet has tried and is now backing off (waiting longer between retries); same root causes.

CrashLoopBackOff — the container starts then dies, repeatedly

0/1 CrashLoopBackOff. The container keeps exiting, so kubelet restarts it with growing delays. Causes:

  1. Bug / crash in the app (bad config, missing env var, can't reach a dependency).
  2. Container misconfiguration (wrong command/args, missing file).
  3. Liveness probe too aggressive — killing a healthy-but-slow-starting app.
kubectl logs my-pod --previous     # WHY the last instance died — read this first
kubectl describe pod my-pod        # check restart count + probe failures

describe shows the container's Last State: Terminated exit code — decode it:

Exit codeMeaningWhere to look
137SIGKILL (128+9) — almost always OOMKilled, or killed during evictiondescribe → Reason: OOMKilled; see OOMKilled below
143SIGTERM (128+15) — graceful stop the app didn't finish in its grace periodApp shutdown handling; raise terminationGracePeriodSeconds
1 / 2Generic application error (unhandled exception, bad config, missing env var)kubectl logs --previous

If kubectl logs --previous is EMPTY, the container never started — it's not a crash, it's a CreateContainerError / RunContainerError / entrypoint problem (bad command, missing ConfigMap/Secret, image with no such binary). There are no logs because the process never ran. Look at kubectl describe pod Events, not logs (see RunContainerError below).

Tip: If a slow app gets killed before it boots, add a startup probe (or raise initialDelaySeconds on the liveness probe). See the Pods section on probes.

RunContainerError / CreateContainerConfigError — container can't even start

The kubelet fails before the app runs. Almost always a config wiring problem:

  • A referenced ConfigMap or Secret doesn't exist (CreateContainerConfigError).
  • A bad command/entrypoint, mount, or permission.
kubectl describe pod my-pod    # Events name the missing ConfigMap/Secret or bad mount

OOMKilled — the container exceeded its memory limit

describe shows Last State: Terminated, Reason: OOMKilled, Exit Code: 137. The container used more memory than its limits.memory, so the kernel killed it.

kubectl describe pod my-pod | grep -A2 "Last State"
kubectl top pod my-pod                 # see live usage (needs metrics-server)

Fix: raise resources.limits.memory, fix a memory leak, or right-size with VPA recommendations.

Two very different "out of memory" failures — don't confuse them:

Container OOMKillNode memory-pressure eviction
What happenedOne container exceeded its own limits.memory; the kernel killed that containerThe whole node ran low on memory; kubelet evicted pods to reclaim it
Pod showsStill there; Last State: Terminated, Reason: OOMKilled, Exit Code: 137, restart count climbsPod status Evicted; describe Event reason: Evicted, message: The node was low on resource: memory
ScopeJust that pod/containerSeveral pods on the same node may be evicted at once
FixRight-size the pod — raise limits.memory / fix the leakNode-level: add capacity (Karpenter/Auto Mode), set sane requests so the scheduler stops overpacking, evict less by reserving system memory
# Container OOMKill — look at the one pod:
kubectl describe pod my-pod | grep -A2 "Last State"

# Node memory pressure — look at the node and evicted pods:
kubectl get pods -A --field-selector=status.phase=Failed   # Evicted pods land here
kubectl describe node <node>            # Conditions: MemoryPressure=True?
kubectl top nodes                       # which node is hot
kubectl get events -A | grep -i evict   # eviction events

2026 Update: With in-place pod resize (beta and on by default in v1.33) you can adjust CPU/memory on a running pod via the resize subresource without a full restart — handy when tuning limits.

Running but NOT Ready (0/1 Running) — readiness probe failing

The container is up (Running) but reports 0/1 Ready. Its readiness probe hasn't passed, so the Service deliberately keeps it out of rotation — it receives no traffic. Causes: app still warming up, dependency not reachable, or a wrong probe path/port.

kubectl describe pod my-pod   # Events: "Readiness probe failed: HTTP 503 ..."

Note: Liveness vs readiness: a failed liveness probe restarts the container; a failed readiness probe just removes it from the Service (no restart). A pod stuck 0/1 Running is a readiness problem, not a crash.

26.3 Node-level problems — when the node, not the pod, is the fault

Sometimes the pod is fine and the node is broken. Symptoms cluster around two states:

Node NotReady

kubectl get nodes shows a node NotReady. The kubelet has stopped reporting healthy — its pods will get rescheduled (eventually) but won't run there now.

kubectl get nodes
kubectl describe node <node>     # read the Conditions block — this is the answer

Read the node Conditions:

Condition = TrueMeaningTypical fix
Ready = False/Unknownkubelet down or not heartbeating (crash, network, expired creds)Check kubelet/agent; on EKS check the node joined (aws-auth/access entry, SG, IAM)
MemoryPressureNode low on memory → evictionsAdd nodes / fix requests (see OOMKilled above)
DiskPressureNode disk/inode full (image bloat, logs)Free disk, grow volume, garbage-collect images
PIDPressureToo many processesLower pod density / fix fork-bombing app

Pods stuck Terminating or Unknown

When a node goes unreachable, its pods sit in Terminating or Unknown because the API server can't confirm they're gone. Kubernetes adds a node.kubernetes.io/unreachable taint and waits before rescheduling.

kubectl get pods -o wide               # which pods, on which (dead) node
kubectl describe node <node>           # confirm NotReady / unreachable

If the node is genuinely gone and a pod won't clear, force-delete it as a last resort (only once you're sure the node is dead — otherwise you risk two copies running):

kubectl delete pod <pod> --grace-period=0 --force

Note: --grace-period=0 --force does not fix the node — it just tells the API server "trust me, that pod is gone" so a replacement can schedule. Never use it on a healthy node to "speed things up"; for StatefulSets it can cause two pods with the same identity to run at once.

26.4 Service problems — "the pods are fine but I can't reach them"

If pods are Running and Ready but traffic doesn't arrive, debug the Service. The classic failure is a Service with no endpoints:

kubectl get endpointslices -l kubernetes.io/service-name=my-svc
# No addresses?  -> the Service selector matches NO ready pods.
# Faster to type (still works): the short Endpoints form
kubectl get endpoints my-svc           # Endpoints is deprecated, not yet removed

Checklist:

  1. Selector mismatch — the Service spec.selector labels must match the pod template labels exactly. This is the #1 cause of empty endpoints.
  2. targetPort mismatch — the Service targetPort must equal the container's containerPort (by number or name). The Service port (front door) can be anything.
  3. No ready pods — readiness failing means pods are excluded from endpoints (see 26.2).
kubectl describe svc my-svc                       # confirm selector + ports
kubectl get pods -l app=myapp --show-labels       # do labels match the selector?
# Quick reachability test from inside the cluster:
kubectl run tmp --rm -it --image=busybox -- wget -qO- my-svc:80

Key idea — the alignment rules. Service selector → matches pod labels; Service targetPort → matches container port; Service port → arbitrary; Ingress service.name/service.port → match the Service's name/port. When connectivity breaks, these four lines are almost always where the typo is.

26.5 Ingress problems — 404, 502 and 504

Pods Ready, Service has endpoints, but the external URL fails. Now check the Ingress/Gateway layer.

Try it: Break a Service on purpose: deploy nginx + a Service, then edit the Service selector to app: wrong. Run kubectl get endpointslices -l kubernetes.io/service-name=<svc> and watch the endpoints disappear; kubectl describe svc confirms the mismatch. Fix the selector and watch them return.

Recap — Troubleshooting

  • Debug bottom-up: Pod → Node → Service → Ingress. Don't jump to the top layer.
  • describe (Events) + logs --previous solve most pod issues; an empty --previous means the container never started (entrypoint/config problem — read Events, not logs).
  • Memorize the pod states: Pending (scheduling), ImagePullBackOff (image/creds), CrashLoopBackOff (app/probe — decode exit codes 137/143/1/2), OOMKilled (container memory limit, exit 137) vs Evicted (node memory/disk/PID pressure), 0/1 Running (readiness).
  • Node issues: NotReady → describe node Conditions; pods stuck Terminating/Unknown on a dead node → --grace-period=0 --force only as a last resort.
  • A Service with no endpoints is almost always a selector or readiness problem (kubectl get endpoints <svc> is the quick check).
  • 404 = Ingress rule/class/backend mismatch; 502 = no healthy endpoints / ALB target-group unhealthy (health-check path, SG, ip-vs-instance); 504 = slow backend / ALB idle-timeout; <pending> LB = subnet tags / controller.
  • AWS creds failures (AccessDenied / sts:AssumeRoleWithWebIdentity) are an IRSA/Pod Identity wiring problem — see §24.4.

  • Annotation — Non-identifying key/value metadata on an object, for tools and humans (not used for selection).
  • API server — The front door of the control plane; all reads/writes go through it into etcd.
  • CNI (Container Network Interface) — Pluggable standard for pod networking (Cilium, Calico, AWS VPC CNI).
  • ClusterIP — Default Service type; a stable virtual IP reachable only inside the cluster.
  • ConfigMap — Non-secret configuration (env vars or files) decoupled from the image.
  • Context — A kubeconfig entry bundling cluster + user + default namespace.
  • Control plane — API server, etcd, scheduler, controller managers; the cluster's brain (AWS-managed on EKS).
  • Controller — A loop that drives actual state toward desired state (reconciliation).
  • cordon — Mark a node unschedulable so no new pods land on it (existing pods stay). kubectl cordon <node>; reverse with uncordon.
  • CRD (CustomResourceDefinition) — Extends the API with your own resource types; watched by an Operator.
  • CRI (Container Runtime Interface) — Standard between kubelet and the runtime (containerd, CRI-O).
  • CrashLoopBackOff — A container that keeps starting and dying; kubelet backs off between restarts.
  • CSI (Container Storage Interface) — Pluggable storage standard; the in-tree volume plugins are gone.
  • DaemonSet — Runs one pod copy on every (or selected) node; used for agents (logging, monitoring).
  • Deployment — Manages ReplicaSets for stateless apps; enables rolling updates and rollbacks.
  • drain — Cordon a node and evict its pods (respecting PodDisruptionBudgets) before maintenance. kubectl drain <node> --ignore-daemonsets --delete-emptydir-data.
  • EBS — AWS block storage; ReadWriteOnce (one node at a time).
  • EFS — AWS elastic file system; ReadWriteMany (shared across pods/nodes).
  • EKS — Amazon's managed Kubernetes; AWS runs the control plane.
  • EKS Auto Mode — AWS-managed Karpenter + nodes + core add-ons; minimal node ops.
  • EndpointSlice — Scalable list of a Service's backing pod IPs/ports (replaces the deprecated Endpoints API; kubectl get endpoints <svc> still works as a short form, not yet removed).
  • Ephemeral container — A temporary container injected into a running pod for debugging (kubectl debug).
  • Evicted — Pod status when the node reclaimed resources (memory/disk/PID pressure) and killed it — distinct from a container OOMKill. Reason appears in node/pod Events.
  • etcd — The distributed key/value store holding all cluster state.
  • Fargate — Serverless per-pod compute on EKS; no nodes to manage.
  • Gateway API — Modern, role-oriented successor to Ingress (GatewayClass/Gateway/HTTPRoute).
  • HPA (HorizontalPodAutoscaler) — Scales pod replicas by metrics; API autoscaling/v2.
  • Ingress — L7 HTTP routing into the cluster (frozen API; Gateway API is the successor).
  • Internet Gateway (IGW) — VPC component giving public subnets direct two-way internet access (e.g. for internet-facing load balancers).
  • IRSA (IAM Roles for Service Accounts) — Maps an IAM role to a ServiceAccount via an OIDC provider.
  • Karpenter — Workload-aware node autoscaler that provisions right-sized EC2 nodes fast.
  • kubeconfig — The file (default ~/.kube/config) holding clusters, users/credentials, and contexts that kubectl uses to reach a cluster; on EKS, populated by aws eks update-kubeconfig.
  • kubectl — The Kubernetes command-line client (verb–noun model).
  • kubelet — Node agent that runs and supervises pods via the CRI.
  • kube-proxy — Programs node networking rules so ClusterIPs reach pods.
  • kube-system — The built-in namespace holding core add-ons (CoreDNS, kube-proxy, CNI, metrics-server); on EKS the control-plane components live in an AWS account, not here.
  • Label — Identifying key/value used by selectors (Services, ReplicaSets, etc.).
  • LoadBalancer (Service) — Provisions an external L4 load balancer (NLB on EKS).
  • Liveness probe — Health check; failing it restarts the container.
  • Namespace — Logical partition of cluster resources (not a security boundary by default).
  • NAT Gateway — AWS managed component in a public subnet that lets private-subnet nodes make outbound internet calls (pull images, reach AWS APIs) while staying unreachable from the internet.
  • NodeNotReady (NotReady) — Node condition where the kubelet has stopped reporting healthy (crash, network loss, disk/PID/memory pressure); its pods won't run there and may be rescheduled. Diagnose with kubectl describe node → Conditions.
  • NodePort — Opens a high port (30000–32767) on every node; mostly for demos.
  • OOMKilled — Container terminated for exceeding its memory limit (exit 137) — a container kill, distinct from a node-pressure Evicted.
  • Pod — Smallest deployable unit; one or more containers sharing network + storage.
  • EKS Pod Identity — Newer agent-based way to give pods IAM roles (no per-cluster OIDC).
  • PodDisruptionBudget (PDB) — Caps how many pods of an app can be voluntarily down at once (e.g. minAvailable: 2); honoured by kubectl drain, node upgrades, and Spot consolidation. API policy/v1.
  • Pod Security Admission (PSA) — Enforces Pod Security Standards (privileged/baseline/restricted) via namespace labels (replaced PodSecurityPolicy).
  • PV / PVC — PersistentVolume (the storage) and PersistentVolumeClaim (the request for it).
  • RBAC — Role-Based Access Control: Roles/ClusterRoles bound to subjects.
  • Readiness probe — Health check; failing it removes the pod from the Service (no restart).
  • ReplicaSet — Maintains N identical pod replicas (managed by a Deployment).
  • Reconciliation loop — Observe actual state → diff against desired → act; repeat.
  • ServiceAccount — An identity for pods to authenticate to the API (and to AWS via IRSA/Pod Identity).
  • Spot interruption — AWS reclaiming a Spot EC2 instance with a 2-minute warning; the node is cordoned/drained and pods reschedule. Use PDBs, multiple instance types, and Karpenter's Spot handling to absorb it gracefully.
  • StatefulSet — Pods with stable identity, ordered rollout, per-pod volumes (databases, queues).
  • StorageClass — Names a provisioner (e.g. ebs.csi.aws.com) for dynamic PV creation.
  • Taint / Toleration — Nodes repel pods (taint) unless the pod tolerates it.
  • VPC — AWS virtual network that hosts the EKS cluster.

28. Why Data Engineers Run on Kubernetes (and the Operator Pattern for Data)

If you've reached this part of the guide, you can already create a Pod, run a Job, mount a volume, and read a Service's logs. Good — that's the whole foundation. This part is the payoff: it shows how the exact same primitives become a serious platform for data pipelines — the batch ETL, the Spark jobs, the Kafka topics, the Airflow DAGs you already know.

You don't need new Kubernetes magic to do data work. You need to recognise that a Spark executor is "just a Pod", a nightly ingest is "just a CronJob", and a Kafka broker is "just a StatefulSet with a volume". The rest is wiring, and increasingly, Operators do that wiring for you.

Key idea: Kubernetes is a general-purpose scheduler for containers. A data pipeline is just a particular shape of containers — short-lived batch Pods, long-lived stateful brokers, and bursty fleets of compute workers. Everything you learned about Pods, Jobs, StatefulSets, storage, RBAC, and autoscaling applies directly.

28.1 Why move data workloads onto Kubernetes at all?

For years the default answer to "where does my Spark job run?" was a dedicated cluster — a long-lived EMR cluster, a Hadoop/YARN cluster, or a pool of VMs you SSH into. That model has three chronic problems, and Kubernetes addresses each one.

Problem 1 — idle, expensive iron. A dedicated Spark/EMR cluster sized for your peak (the 2 a.m. month-end batch) sits mostly idle the rest of the day, but you pay for it 24/7. On Kubernetes, a batch job's Pods are created when the work starts and deleted when it finishes (see §11 Jobs). With cluster autoscaling (§14), the nodes themselves scale to zero when there's nothing to run.

Problem 2 — one workload, one cluster. Teams end up with a Spark cluster, a separate Airflow box, a separate Kafka cluster, a separate Trino cluster — each its own snowflake to patch and secure. Kubernetes is a multi-tenant shared runtime: Spark executors, Airflow tasks, Kafka brokers, and a Trino query engine can all run on the same cluster, isolated by Namespaces, ResourceQuotas, and RBAC (see §7 config/namespaces and §21 RBAC), bin-packed onto shared nodes.

Problem 3 — bursty, heterogeneous demand. Data work is spiky: nothing for an hour, then a 200-executor shuffle. Kubernetes' scheduler (§13) bin-packs many Pods onto each node, and Karpenter (§14, §24) provisions right-sized nodes in ~45–60 seconds — including cheap Spot instances for the throwaway compute that makes up most of a Spark job.

Tip: The single biggest cost lever in data-on-Kubernetes is spot + autoscaler. Stateless, retryable compute (Spark executors, Flink task managers, queue consumers) tolerates being evicted, so it's a perfect fit for Spot. Keep the stateful and coordinator pieces (Spark driver, Kafka brokers, Airflow scheduler) on stable on-demand nodes. We'll come back to this pattern in §30.

Note: "Bin-packing" just means the scheduler tries to fill each node efficiently — like packing a suitcase — rather than spreading Pods thinly across many half-empty machines. It's how you get high utilisation (and lower cost) from a shared cluster.

28.2 The Operator pattern, recapped for data

You met Custom Resource Definitions (CRDs) and Operators in §22. Quick refresher, because the entire data-on-Kubernetes ecosystem is built on them:

  • A CRD teaches the Kubernetes API server a new kind of object. After you install one, kind: SparkApplication or kind: Kafka becomes as real to kubectl as kind: Pod.
  • An Operator is a CRD + a controller (the controller is itself just a Pod running in the cluster). The controller watches your custom resources and runs a reconciliation loop (§2): "the user declared a SparkApplication; let me create the driver Pod, then the executor Pods, then clean up when it's done."

The win for data engineers is enormous. Without an Operator, running Spark on Kubernetes means hand-crafting a driver Pod, a service account, RBAC, a headless Service, executor specs, and lifecycle cleanup — dozens of lines of fiddly YAML you must keep correct. With the Spark Operator, you write one declarative SparkApplication describing what you want, and the controller builds all of that for you and tears it down when the job ends.

Key idea: The Operator pattern turns "a runbook a human follows" into "a controller the cluster runs". You stop wiring Pods by hand and start declaring a higher-level intent — SparkApplication, FlinkDeployment, Kafka — and let the Operator reconcile reality to match. It's the same declarative philosophy as a Deployment, just for a data engine.

28.3 When not to reach for Kubernetes

Kubernetes is powerful, but it is not free of cost — the cost is operational complexity. Be honest about when a simpler tool wins:

  • A single small batch job that runs occasionally and finishes in minutes. A plain container on AWS Fargate, a Lambda, an ECS task, or even a cron entry on one VM may be far simpler than standing up cluster, Operator, RBAC, and storage classes.
  • You want zero infrastructure to manage. Fully-managed services — Databricks, EMR Serverless, AWS Glue, Snowflake, MSK (managed Kafka), RDS/Aurora — hand you the engine with no cluster to operate. If your organisation already runs Databricks for ETL (as many do), don't rebuild it on raw Kubernetes for its own sake.
  • Your team has no Kubernetes operational capacity. A data platform on Kubernetes needs someone who understands nodes, autoscaling, upgrades (§1 cadence: ~3 releases/year), and CSI storage. Without that, a managed service trades money for sanity.

Note: "Managed vs self-run on Kubernetes" is the exact same trade-off you already know from "RDS vs running Postgres yourself" or "Databricks vs running Spark yourself". Kubernetes gives you control and cost-efficiency at the price of operating it. Pick the level of control your team can actually sustain.

The sweet spot for Kubernetes is a multi-team data platform with continuous, varied, bursty workloads — exactly the situation where a shared, elastic, bin-packed runtime pays for its complexity many times over.

Recap

  • Kubernetes is a shared, elastic runtime for data pipelines: batch Pods appear and vanish, stateful systems live as StatefulSets, and Karpenter + Spot drive cost down — replacing always-on dedicated clusters/VMs.
  • Everything reuses primitives you already know: a Spark executor is a Pod, a nightly ingest is a CronJob (§11), a Kafka broker is a StatefulSet with a volume (§12).
  • The Operator pattern (CRD + controller, §22) lets you declare a SparkApplication / FlinkDeployment / Kafka instead of hand-wiring Pods; the controller reconciles reality to match.
  • Mature data Operators exist for Spark, Flink, Kafka (Strimzi), Postgres (CloudNativePG), Ray (KubeRay), Trino — prefer them over raw Pod YAML.
  • Don't force Kubernetes onto a single small job, or when a fully-managed service (Databricks, EMR Serverless, MSK, RDS) is simpler and your team lacks Kubernetes operational capacity.

29. Orchestrating Pipelines: Jobs, CronJobs, Airflow, and Argo Workflows

Most data pipelines boil down to "run this step, and when it succeeds, run the next step." That's orchestration. Kubernetes gives you primitive batch building blocks (Job, CronJob); on top of those, full orchestrators (Airflow, Argo Workflows) manage the DAG — the dependency graph of steps. This section connects what you know from Airflow to what runs underneath on Kubernetes.

29.1 Batch as a first-class citizen: Job and CronJob for ETL

You met Job and CronJob in §11. Here's how they map directly onto data-pipeline patterns. (We won't re-explain the fields — we'll show the data-engineering shapes.)

A Job runs Pods to completion: it retries on failure and stops when the work is done. That's exactly a one-off batch step — "load yesterday's partition into the warehouse."

apiVersion: batch/v1
kind: Job
metadata:
  name: ingest-trades-20260608
spec:
  backoffLimit: 4              # retry the Pod up to 4 times on failure
  activeDeadlineSeconds: 3600  # hard kill the whole Job after 1 hour
  ttlSecondsAfterFinished: 86400  # auto-delete Job + its Pods 24h after it finishes
  template:
    spec:
      restartPolicy: OnFailure
      containers:
      - name: ingest
        image: my-registry/ingest:1.4.2     # pin by version/digest, never :latest
        args: ["--date", "2026-06-08", "--source", "trades"]
        resources:
          requests: { cpu: "1", memory: "2Gi" }
          limits:   { memory: "2Gi" }

The four fields that matter most for data jobs:

FieldWhat it doesWhy a data engineer cares
backoffLimitMax Pod retries before the Job is marked failedTolerate transient errors (a flaky S3 read) without alerting a human
activeDeadlineSecondsHard wall-clock cap on the whole JobStops a hung query from running (and billing) forever
ttlSecondsAfterFinishedAuto-delete finished Job + Pods after N secondsWithout it, finished Jobs pile up and clutter the namespace
restartPolicyOnFailure or Never for batchAlways is for services, not batch — don't use it on a Job

A CronJob is a Job on a schedule — the Kubernetes-native equivalent of a crontab line or a simple Airflow schedule:

apiVersion: batch/v1
kind: CronJob
metadata:
  name: nightly-warehouse-load
spec:
  schedule: "0 2 * * *"          # 02:00 every day
  timeZone: "Asia/Ho_Chi_Minh"   # GA since v1.27 — no more UTC-only headaches
  concurrencyPolicy: Forbid      # don't start tonight's run if last night's still running
  startingDeadlineSeconds: 300   # if a run is missed (controller was down), only start if <5min late
  successfulJobsHistoryLimit: 3
  failedJobsHistoryLimit: 3
  jobTemplate:
    spec:
      backoffLimit: 2
      template:
        spec:
          restartPolicy: OnFailure
          containers:
          - name: load
            image: my-registry/warehouse-load:2.0.0

2026 Update: CronJob is stable as batch/v1, and .spec.timeZone has been GA since v1.27 — so you can schedule in Asia/Ho_Chi_Minh instead of mentally converting from UTC. Older notes that say "CronJobs always run in UTC" are out of date.

Parallel batch: parallelism, completions, indexed Jobs, and work queues

Plain Jobs also do parallel batch, which is bread-and-butter data work. Two key fields:

  • completions — how many successful Pod runs make the Job "done".
  • parallelism — how many Pods run at once.

Two common patterns:

Indexed Job (completionMode: Indexed, GA since v1.24) — Kubernetes hands each Pod a fixed index via the JOB_COMPLETION_INDEX environment variable. Perfect for static sharding: "process partition N", where N is the index.

apiVersion: batch/v1
kind: Job
metadata:
  name: backfill-30-days
spec:
  completionMode: Indexed
  completions: 30        # 30 partitions (indices 0..29)
  parallelism: 6         # run 6 at a time
  template:
    spec:
      restartPolicy: OnFailure
      containers:
      - name: backfill
        image: my-registry/backfill:1.0.0
        command: ["sh", "-c", "process-partition --index $JOB_COMPLETION_INDEX"]

Work-queue Job — set completions to nothing (or equal to parallelism) and have each Pod pull units of work from a shared queue (SQS, Redis, RabbitMQ) until the queue is empty, then exit 0. The Job finishes when the queue is drained. This is the natural pattern for dynamic, uneven workloads.

2026 Update: Job reliability got a big boost in v1.33: per-index backoff limit (backoffLimitPerIndex) and success policy for indexed Jobs (successPolicy) are now GA, alongside the older Pod failure policy. For a 30-partition backfill you can now say "give up on a single bad partition without failing the whole Job" — exactly what you want for a wide backfill where one corrupt file shouldn't sink 29 good partitions.

Note: If you're coming from Airflow, you can think of a Job as one task instance and a CronJob as a scheduled DAG with one task. For multi-step DAGs with real dependencies, you graduate to Airflow or Argo Workflows (below) — but the leaf work still ultimately runs in Pods, often as Jobs.

29.2 Airflow on Kubernetes

Airflow is the most common orchestrator in data teams, and it runs beautifully on Kubernetes. The thing to understand is how Airflow runs your task code — that's the executor.

KubernetesExecutor — one Pod per task. With the KubernetesExecutor, the Airflow scheduler launches a brand-new Pod for every task instance, the task runs, and the Pod is deleted. There are no long-lived worker processes sitting idle.

  • Pros: strong isolation (each task gets its own Pod, its own image, its own filesystem); per-task resources (give the heavy Spark-submit task 8Gi and the trivial sensor task 256Mi); resources are consumed only while a task runs; a poisoned task can't corrupt a shared worker.
  • Cons: Pod startup latency (a few seconds per task) makes it less ideal for thousands of tiny, sub-second tasks.

**KubernetesPodOperator (KPO) — run any container as a task.** This is an operator you use inside your DAG, independent of which executor you run. It launches an arbitrary container image as a Pod and treats it as one DAG task. This is how data teams run polyglot pipelines: a dbt task, a Spark-submit task, a custom Rust binary — each in its own image — all orchestrated from one Airflow DAG.

from airflow.providers.cncf.kubernetes.operators.pod import KubernetesPodOperator

run_dbt = KubernetesPodOperator(
    task_id="dbt_run_marts",
    name="dbt-run-marts",
    namespace="data-pipelines",
    image="my-registry/dbt-project:2026.06.08",
    cmds=["dbt"],
    arguments=["run", "--select", "marts.*", "--target", "prod"],
    container_resources={"requests": {"cpu": "1", "memory": "2Gi"}},
    service_account_name="dbt-runner",   # for IRSA/Pod Identity S3 access (see §24)
    get_logs=True,
)

Tip: Versus the older CeleryExecutor (a fixed pool of always-on worker processes pulling from a Redis/RabbitMQ broker): Celery has lower per-task latency and is great for many small homogeneous tasks, but you pay for idle workers and share their environment across tasks. KubernetesExecutor wins on isolation, per-task right-sizing, and scale-to-near-zero. Many teams run CeleryKubernetesExecutor (a hybrid) to get both. Pick based on your task profile: lots of tiny tasks → Celery leans better; heterogeneous heavy tasks → Kubernetes.

Note: Don't confuse Airflow's operators (Python classes in your DAG, like KubernetesPodOperator) with Kubernetes Operators (CRD + controller, §22). Same word, different worlds. Airflow operators define DAG tasks; Kubernetes Operators manage cluster resources.

29.3 Argo Workflows: DAGs as Kubernetes custom resources

Argo Workflows is a Kubernetes-native orchestrator (a CNCF project). Where Airflow keeps the DAG in Python and a database, Argo expresses the whole workflow as a Workflow custom resource — each step is a container, and Argo's controller runs the DAG by creating Pods. It's GitOps-friendly (the DAG is a manifest) and has no separate scheduler/metadata DB to operate beyond the controller.

apiVersion: argoproj.io/v1alpha1
kind: Workflow
metadata:
  generateName: etl-
spec:
  entrypoint: pipeline
  templates:
  - name: pipeline
    dag:
      tasks:
      - name: extract
        template: run-step
        arguments: { parameters: [{name: cmd, value: "extract.py"}] }
      - name: transform
        template: run-step
        dependencies: [extract]          # transform waits for extract
        arguments: { parameters: [{name: cmd, value: "transform.py"}] }
      - name: load
        template: run-step
        dependencies: [transform]
        arguments: { parameters: [{name: cmd, value: "load.py"}] }
  - name: run-step
    inputs:
      parameters: [{name: cmd}]
    container:
      image: my-registry/etl:1.0.0
      command: ["python", "{{inputs.parameters.cmd}}"]

Tip: Airflow vs Argo Workflows — Airflow shines when your team thinks in Python, needs a rich provider/operator ecosystem, calendar scheduling, and a mature UI/backfill story. Argo shines when you want everything declared as Kubernetes manifests under GitOps (§23), per-step containers with no shared Python environment, and tight integration with Argo CD. They coexist happily; many platforms use Airflow for scheduling and KPO/Argo for execution.

A note on dbt. dbt itself isn't an orchestrator — it's a transformation tool that runs dbt run/dbt build. On Kubernetes you run it as a container in a Job/Pod, triggered however you like: as an Airflow KubernetesPodOperator task (shown above), as a step in an Argo Workflow, or as a plain CronJob. Package your dbt project into an image, mount or bake in profiles.yml, and give the Pod a service account with object-storage/warehouse credentials (next section).

29.4 The four things every data Pod must get right

Whatever orchestrator you use, the task code inside the Pod must respect Kubernetes reality. Pods are ephemeral and may be retried or evicted (especially on Spot — §30). Four rules:

  1. Idempotency. A task may run more than once (a retry after a node eviction, a duplicate trigger). Writing the same partition twice must not double-count. Use overwrite-by-partition writes (INSERT OVERWRITE / Iceberg/Delta MERGE on a partition key) rather than blind appends. (In this workspace, partition key ds_dt is the conventional handle for exactly this — see the workspace conventions.)
  2. Retries. Let the platform retry: backoffLimit on Jobs, retries on Airflow tasks, retryStrategy on Argo. Make the code safe to retry (rule 1) and let the orchestrator do its job.
  3. Checkpointing. Long-running streaming jobs (Spark Structured Streaming, Flink) must checkpoint progress to durable storage (S3), not the Pod's local disk — because the Pod can disappear. More on this in §30.
  4. Read/write object storage with proper identity. Pods reach S3 using a Kubernetes ServiceAccount mapped to an AWS IAM role via IRSA or EKS Pod Identity (see §24). Never bake long-lived AWS keys into an image or env var.
# A ServiceAccount annotated for IRSA: pods using it assume this IAM role for S3 access.
apiVersion: v1
kind: ServiceAccount
metadata:
  name: dbt-runner
  namespace: data-pipelines
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::022499042553:role/nedp-dbt-runner

2026 Update: On EKS, prefer EKS Pod Identity (agent-based, no per-cluster OIDC provider, one IAM role reusable across clusters) over classic IRSA for new setups — though IRSA is still required on Fargate and for portability. Either way, the principle is the same: the Pod's ServiceAccount, not embedded credentials, grants S3 access. See §24.

Try it: Watch an orchestrator turn a DAG into Pods in real time: ``bash kubectl get pods -n data-pipelines -w # In another terminal, trigger your CronJob immediately instead of waiting for the schedule: kubectl create job --from=cronjob/nightly-warehouse-load adhoc-run-001 -n data-pipelines ``

Recap

  • Job = run-to-completion batch step; CronJob = scheduled Job (with GA timeZone since v1.27). Master backoffLimit, activeDeadlineSeconds, and ttlSecondsAfterFinished for clean, self-bounding data jobs.
  • For parallel batch use parallelism/completions, Indexed Jobs for static sharding, and work-queue Jobs for dynamic workloads — now with per-index backoff and success policy (GA v1.33).
  • Airflow: KubernetesExecutor runs one Pod per task (isolation + per-task resources); KubernetesPodOperator runs any container image as a task. Both beat the always-on CeleryExecutor for heterogeneous heavy workloads.
  • Argo Workflows expresses the DAG as a Kubernetes Workflow CR — GitOps-native, per-step containers. dbt runs as a container in a Job/KPO/Argo step, not as an orchestrator itself.
  • Inside every data Pod: be idempotent, lean on retries, checkpoint to S3, and reach object storage via a ServiceAccount + IRSA/Pod Identity (§24) — never embedded keys.

30. Distributed Engines &amp; Streaming: Spark, Flink, Kafka, and Data Storage on K8s

This is where the heavy lifting lives: the distributed compute engines (Spark, Flink, Ray) and the streaming backbone (Kafka), plus the storage realities that make or break them on Kubernetes. Everything here is built on §11 (Jobs), §12 (StatefulSets), §10 (storage), §13 (scheduling), and §14/§24 (autoscaling, EKS) — go back to those if a primitive is unfamiliar.

30.1 Spark on Kubernetes

Spark's architecture maps cleanly onto Pods:

  • The driver — the coordinator that holds the SparkContext and the DAG of your job — runs as one Pod.
  • Each executor — a worker that runs tasks and holds data partitions in memory — runs as its own Pod. A job with 50 executors is 50 Pods, plus the driver.

The driver Pod actually talks to the Kubernetes API server to create and delete executor Pods as the job progresses. Kubernetes is, in effect, Spark's cluster manager (replacing YARN/Mesos/standalone).

Two ways to run Spark: spark-submit vs the Spark Operator

spark-submit --master k8s://... — the built-in, imperative way. You point spark-submit at the Kubernetes API server and it creates the driver Pod, which then creates executors.

spark-submit \
  --master k8s://https://<api-server>:443 \
  --deploy-mode cluster \
  --name trades-aggregation \
  --conf spark.kubernetes.container.image=my-registry/spark:3.5.1 \
  --conf spark.kubernetes.namespace=data-pipelines \
  --conf spark.kubernetes.authenticate.driver.serviceAccountName=spark-runner \
  --conf spark.executor.instances=20 \
  local:///opt/app/aggregate_trades.py

This works, but it's imperative — there's no Kubernetes object that represents "this Spark job" declaratively, no easy kubectl get of your jobs, no GitOps story.

The Spark Operator (SparkApplication CR) — the declarative way. You install the Kubeflow Spark Operator (formerly Google's spark-on-k8s-operator), then describe the job as a custom resource. The Operator's controller does the spark-submit for you and tracks the job's state as Kubernetes status.

apiVersion: sparkoperator.k8s.io/v1beta2
kind: SparkApplication
metadata:
  name: trades-aggregation
  namespace: data-pipelines
spec:
  type: Python
  mode: cluster
  image: my-registry/spark:3.5.1
  mainApplicationFile: local:///opt/app/aggregate_trades.py
  sparkVersion: "3.5.1"
  driver:
    cores: 1
    memory: "2g"
    serviceAccount: spark-runner       # IRSA/Pod Identity for S3 (see §24)
  executor:
    instances: 20
    cores: 2
    memory: "4g"
    # schedule executors onto cheap Spot nodes (see §13 affinity/§14 Karpenter)
    nodeSelector:
      karpenter.sh/capacity-type: spot
  dynamicAllocation:
    enabled: true
    minExecutors: 2
    maxExecutors: 50

Now kubectl get sparkapplications lists your jobs, and you can put that YAML under GitOps (§23). For anything beyond a one-off, prefer the Operator.

Tip: Dynamic allocation lets Spark add executors when there's backlog and release them when idle — instead of fixing a static count. On Kubernetes this means executor Pods appear and disappear during the job, and (with Karpenter, §14) nodes follow. It's the elasticity that makes Spark-on-Kubernetes cheaper than a fixed cluster. (Spark's external shuffle service story on Kubernetes has historically been the catch — see shuffle, below.)

Shuffle, ephemeral storage, and Spot

A Spark shuffle — the all-to-all data exchange between stages (after a groupBy, join, or repartition) — writes large intermediate files to local disk. On Kubernetes this is the trickiest part:

  • Give executors fast local scratch space. The best option is local NVMe SSD (e.g. EC2 instance-store on r5d/i4i families), surfaced as an emptyDir (optionally emptyDir.medium: Memory or backed by the instance store). Local NVMe is far faster than a network EBS volume for shuffle.
  • Spilled shuffle data is ephemeral scratch, not durable data — it's fine for it to die with the Pod, as long as the job can recompute lost partitions.
  • Spot + shuffle is a real tension. If a Spot executor is reclaimed mid-shuffle, Spark must recompute its lost shuffle blocks, which can cascade. Mitigations in 2026: keep the driver on on-demand (losing the driver kills the whole job), put executors on Spot, and consider Spark's decommissioning feature (graceful Spot-interruption handling that migrates shuffle/cache blocks off a doomed executor before it dies).

Note: "Ephemeral" means "temporary, will be lost when the Pod ends." emptyDir is an ephemeral volume scoped to the Pod's lifetime — perfect for shuffle scratch, useless for anything you need to keep. Durable data goes to S3 (below).

30.2 Flink on Kubernetes (and a word on Ray)

Apache Flink is the streaming-first engine (true event-at-a-time processing with strong state and exactly-once semantics). On Kubernetes you run it via the Flink Kubernetes Operator, declaring a FlinkDeployment:

apiVersion: flink.apache.org/v1beta1
kind: FlinkDeployment
metadata:
  name: fraud-detection
  namespace: streaming
spec:
  image: my-registry/flink:1.19
  flinkVersion: v1_19
  flinkConfiguration:
    taskmanager.numberOfTaskSlots: "2"
    state.backend: rocksdb
    state.checkpoints.dir: s3://aws-sg-nedp-prod-checkpoints/fraud-detection/cp
    state.savepoints.dir: s3://aws-sg-nedp-prod-checkpoints/fraud-detection/sp
    execution.checkpointing.interval: "30s"
  serviceAccount: flink-runner          # IRSA/Pod Identity for S3 (§24)
  jobManager:
    resource: { cpu: 1, memory: "2048m" }
  taskManager:
    resource: { cpu: 2, memory: "4096m" }
  job:
    jarURI: local:///opt/flink/usrlib/fraud.jar
    parallelism: 4
    upgradeMode: savepoint              # take a savepoint, then redeploy on the new version

The crucial point for streaming: checkpoints and savepoints go to S3, not the Pod. A checkpoint is Flink's periodic snapshot of its in-flight state; a savepoint is a manual snapshot for upgrades. Because Pods can die or be rescheduled at any time, durable state must live in object storage so a restarted job resumes exactly where it left off. The Operator handles the savepoint-then-upgrade dance (upgradeMode: savepoint) so you can deploy new code without losing state.

Ray (KubeRay), briefly. Ray is the distributed-Python framework popular for ML and large-scale Python workloads. KubeRay provides RayCluster (a long-lived cluster of head + worker Pods), RayJob (run a job, then tear the cluster down — like a Spark SparkApplication), and RayService (serve a model). Same pattern: an Operator turns a CR into Pods, workers scale on Karpenter, durable artifacts go to S3.

30.3 Kafka on Kubernetes with Strimzi

Kafka is stateful — each broker owns log segments on disk that must survive restarts. That's a textbook StatefulSet job (§12): stable network identity, ordered rollout, and a persistent volume per broker via volumeClaimTemplates. You could hand-write all that, but you shouldn't — you use the Strimzi Operator.

Strimzi installs CRDs so you declare Kafka clusters, topics, and users as Kubernetes objects:

apiVersion: kafka.strimzi.io/v1beta2
kind: Kafka
metadata:
  name: events
  namespace: streaming
spec:
  kafka:
    replicas: 3                          # 3 broker Pods (a StatefulSet under the hood)
    listeners:
      - name: tls
        port: 9093
        type: internal
        tls: true
    storage:
      type: persistent-claim             # one PV per broker via volumeClaimTemplates
      size: 500Gi
      class: ebs-gp3                      # an EBS-backed StorageClass (RWO) — see §10
  # KRaft mode (no ZooKeeper) is the default in current Strimzi
---
apiVersion: kafka.strimzi.io/v1beta2
kind: KafkaTopic
metadata:
  name: trades
  namespace: streaming
  labels:
    strimzi.io/cluster: events
spec:
  partitions: 12
  replicas: 3
  config:
    retention.ms: "604800000"            # 7 days

Strimzi also offers KafkaConnect / KafkaConnector CRs to run Kafka Connect (source/sink connectors — e.g. S3 sink, JDBC source) declaratively, and KafkaUser for per-app credentials and ACLs.

Note: Each Kafka broker gets its own PersistentVolume, created from the StatefulSet's volumeClaimTemplates. That's why brokers need stable identities (events-kafka-0, events-kafka-1, …) — broker 0 must always reattach to broker 0's disk after a restart. This is exactly why Kafka is a StatefulSet and not a Deployment. (See §12 for the StatefulSet mechanics.)

Tip: Running Kafka yourself with Strimzi gives full control and avoids vendor lock-in, but you own broker upgrades, rebalancing, and disk capacity. Many teams instead use managed Kafka (AWS MSK) and only run clients (producers/consumers/Connect) on Kubernetes. Same "managed vs self-run" trade-off as §28.3 — choose by team capacity.

30.4 Data storage realities on Kubernetes

This is the single most important mental model for data on Kubernetes:

Key idea: Pods are stateless compute; the data lake lives in object storage (S3). Compute Pods come and go; the durable truth sits in S3 as Parquet/Iceberg/Delta. Pod-attached volumes (EBS/EFS/local disk) are for operational state (a broker's log, a database's data dir) and scratch (shuffle/spill) — not for the lake itself.

Your storage choices (from §10) map to data use-cases like this:

StorageAccess modeSpeedUse it for
EBS (block, via EBS CSI)RWO — one node at a timeFastA single broker/database Pod's data dir (Kafka, Postgres). The default for StatefulSet PVs.
EFS (NFS, via EFS CSI)RWX — many Pods read/writeSlower, networkShared config/artifacts across Pods; rarely the hot path for analytics
Local NVMe (emptyDir / instance store)Pod-local, ephemeralFastestSpark shuffle/spill, scratch — recomputable, dies with the Pod
S3 (object, via SDK/s3a/connector)Many readers/writersHigh-throughput, high-latencyThe data lake — Iceberg/Delta/Parquet tables, checkpoints, savepoints

Note: RWO vs RWX. "ReadWriteOnce" (RWO) = the volume can be mounted by Pods on one node at a time — that's EBS, and it's why an EBS-backed Pod can't freely move across nodes. "ReadWriteMany" (RWX) = many nodes/Pods can mount it at once — that's EFS. For analytics, you mostly bypass both and read/write S3 directly from your engine (s3a:// in Spark, the S3 sink in Kafka Connect), because S3 scales to any number of readers/writers and is where the lake actually lives. (See §10 for the PV/PVC/StorageClass/CSI mechanics.)

# A StatefulSet volumeClaimTemplate (the §12 pattern) giving each replica its own EBS PV:
  volumeClaimTemplates:
  - metadata:
      name: data
    spec:
      accessModes: ["ReadWriteOnce"]       # EBS = RWO
      storageClassName: ebs-gp3
      resources:
        requests:
          storage: 500Gi

2026 Update: In-tree volume plugins (awsElasticBlockStore, etc.) are gone — storage on Kubernetes is CSI-only now (CSI migration went GA in v1.25). Use the EBS CSI and EFS CSI drivers (EKS-managed add-ons). The newer VolumeAttributesClass (GA v1.34) even lets you change a volume's IOPS/throughput without recreating the PVC — handy when a broker outgrows its provisioned EBS throughput. See §10.

30.5 Resources, cost, and event-driven scaling

Data jobs are resource-hungry and bursty, so requests/limits and autoscaling matter more here than almost anywhere else.

Requests, limits, and QoS (recap from §8/§13/§14). Each container declares requests (what the scheduler reserves) and limits (the hard cap). For data Pods:

  • Memory limit = the OOMKiller trigger. If a Spark executor's JVM + off-heap memory exceeds its memory limit, the kernel OOMKills the container — the classic "executor lost" failure on a big shuffle or a skewed join. Size memory limits with headroom for Spark off-heap/overhead, and watch for data skew (one partition far larger than the rest) as the usual root cause.
  • QoS classes flow from requests/limits: set requests == limits to get Guaranteed QoS for your critical stateful Pods (Kafka brokers, the Spark driver) so they're the last to be evicted under node pressure. Best-effort/burstable Pods are evicted first.

2026 Update: In-place Pod resize (CPU/memory adjusted without restarting the Pod) is beta and on-by-default since v1.33 (via the resize subresource). For a long-running streaming Pod that's bumping its memory limit, you can now nudge resources up without a disruptive restart — previously any resource change meant recreating the Pod. See §14.

Karpenter + Spot for cost. As established in §28 and §24: put stateless, retryable compute (Spark executors, Flink task managers, queue consumers, Ray workers) on Spot via Karpenter, which provisions right-sized nodes in ~45–60s and consolidates them away when idle. Keep stateful/coordinator Pods (Kafka brokers, Spark driver, Airflow scheduler) on on-demand nodes. This split is the core cost pattern for data-on-Kubernetes.

KEDA — scale on the metric that actually matters. Plain HPA (§14) scales on CPU/memory. But a Kafka consumer or a queue worker should scale on lag — how far behind the data it is — not CPU. KEDA (Kubernetes Event-Driven Autoscaling) adds scalers that watch external systems (Kafka consumer-group lag, SQS queue depth, Kafka topic offset, Prometheus queries) and drive a Deployment's replica count — including scale-to-zero when there's no work.

apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: trades-consumer
  namespace: streaming
spec:
  scaleTargetRef:
    name: trades-consumer          # the Deployment running your consumer
  minReplicaCount: 0               # scale to zero when the topic is idle
  maxReplicaCount: 30
  triggers:
  - type: kafka
    metadata:
      bootstrapServers: events-kafka-bootstrap.streaming:9092
      consumerGroup: trades-consumer
      topic: trades
      lagThreshold: "1000"         # add a replica per ~1000 messages of lag

Tip: KEDA is the cleanest answer to "my consumer can't keep up at peak but wastes money at night." It scales consumers up with lag and down to zero when the topic is quiet — far better than guessing a fixed replica count or scaling on CPU, which doesn't correlate with backlog. KEDA generates an HPA under the hood; it doesn't replace the HPA, it feeds it event metrics. See §14.

Try it: Diagnose the most common data-Pod failure — the OOMKill — and watch KEDA react to lag: ``bash # Was the executor OOMKilled? Check the last terminated state: kubectl get pod <executor-pod> -o jsonpath='{.status.containerStatuses[0].lastState.terminated.reason}' # Watch KEDA scale the consumer as lag rises and falls: kubectl get scaledobject,hpa,deploy -n streaming -w ``

Recap

  • Spark on Kubernetes: driver Pod + executor Pods, with the driver creating executors via the API server. Use the Spark Operator (SparkApplication) over raw spark-submit for declarative, GitOps-friendly jobs; enable dynamic allocation; put executors on Spot and the driver on on-demand.
  • Shuffle is fast, ephemeral scratch — use local NVMe emptyDir, and handle Spot reclamation (decommissioning) so lost shuffle blocks don't cascade.
  • Flink (FlinkDeployment) and Ray (RayCluster/RayJob) follow the same Operator-to-Pods pattern; checkpoints/savepoints go to S3, never the Pod.
  • Kafka runs via Strimzi as a StatefulSet with one EBS (RWO) PV per broker through volumeClaimTemplates; declare clusters/topics/users as CRs.
  • Storage model: Pods are stateless compute; S3 is the data lake. EBS (RWO) for a single Pod's data dir, EFS (RWX) for sharing, local NVMe for shuffle. Storage is CSI-only in 2026.
  • Cost & scaling: mind memory limits (OOMKilled on big shuffles), use Guaranteed QoS for critical stateful Pods, Karpenter + Spot for throwaway compute, in-place resize (v1.33) to avoid restarts, and KEDA to scale consumers on Kafka/queue lag — to zero when idle.
