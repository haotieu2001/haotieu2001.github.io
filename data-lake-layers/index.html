The Three Layers of a Data Lake, In Depth

Every Data Lake is built from three layers stacked on top of each other: Storage (the files), Metadata (the catalogue that turns files into tables), and Compute (the engines that read and process them). This guide opens each layer up — how it works inside, why it is kept separate from the others, the problems each one runs into, and how a single query travels down through all three to give you an answer.

~26 min read · Data Engineering · Jun 2026

❯ The one idea (read this first)

A Data Lake is not one thing — it's three independent layers that snap together. Storage holds the raw files cheaply but knows nothing about tables. Metadata is a small catalogue that says "the table called orders lives at this path and has these columns" — it turns a pile of files into something you can query by name. Compute is the rented engine that actually reads the files and computes your answer. Because the three are separate, you can swap or scale any one of them without touching the others — and that flexibility is the whole point of a Data Lake.

01. The three layers at a glance

Picture a Data Lake as three shelves stacked on top of each other. The bottom shelf holds the actual data. The middle shelf holds a small index that explains what's on the bottom shelf. The top shelf is the worker who reads the index, then reaches down for exactly the data it needs.

LayerWhat it isWhat it knowsExamples
3 · ComputeThe engines that read files and compute resultsHow to run a query in parallel across many machinesSpark, Trino/Presto, Databricks, Athena
2 · MetadataA catalogue mapping table names → locations + columnsWhat tables exist, where their files are, their schemaHive Metastore, AWS Glue, Unity Catalog
1 · StorageRaw files sitting in cheap object storageNothing about "tables" — just bytes in keysAmazon S3, Azure ADLS, Google GCS

The single most important idea is that these three layers are decoupled — each one can be replaced or scaled on its own:

  • You can point two different compute engines (say Spark for heavy jobs and Trino for fast dashboards) at the same files, at the same time.
  • You can turn all compute off overnight and pay only for storage — the data just sits there.
  • You can swap the catalogue (e.g. migrate from Hive Metastore to Unity Catalog) without moving a single data file.

In a traditional database, all three layers are fused into one product you can't pull apart. That fusion makes it fast — compute sits right next to the data with no network round-trip out to a separate storage service — but also expensive (you pay for that tightly-coupled box even when idle) and rigid (you can't scale or swap one part alone). The Data Lake's whole design philosophy is the opposite: separate the layers so each can be chosen, scaled, and paid for independently, accepting a little extra read latency in exchange. Keeping the three layers independent like this is the through-line of the whole article — it's what lets you scale, swap, and pay for each one on its own. We'll go shelf by shelf, bottom to top.

02. Layer 1 — Storage, in depth

The storage layer is the foundation: it's where the actual bytes live. Almost always, that means cloud object storage — Amazon S3, Azure Data Lake Storage (ADLS), or Google Cloud Storage (GCS). To understand everything built on top, you have to understand how object storage really behaves, because it is not a normal hard drive.

Object storage is a key → bytes dictionary, not a folder tree

On your laptop, files live in a tree of real folders. Object storage is different: it's essentially one giant dictionary (a key-value lookup). The key is the full path string, and the value is the file's bytes.

How object storage actually stores things
KEY (just a string)                                VALUE (bytes)
"orders/order_date=2026-06-01/part-001.parquet"  →  [ 128 MB of data ]
"orders/order_date=2026-06-01/part-002.parquet"  →  [ 128 MB of data ]
"customers/2026-06-01.csv"                       →  [  4 MB of data  ]

The slashes in the key look like folders, but there are no real folders underneath. orders/order_date=2026-06-01/ is not a directory — it's just a common prefix (a shared start) shared by many keys. This sounds like a technicality, but it explains several things that surprise newcomers:

  • There is no "rename folder" operation. To "rename" a prefix, the system must copy every object to new keys and delete the originals. Renaming a path with a million files is a million copy-and-delete operations — slow and not atomic (it can be caught half-done: a reader might see some files renamed and others not).
  • Listing is a search, not a folder open. "List everything under orders/" scans the dictionary for keys starting with that prefix. With millions of keys this is slow and is billed per request.

The three rules of object storage that shape everything above

1
Objects are immutable. Once written, you cannot change one byte of an object. To "edit" data you write a brand-new object and delete the old one. (This single rule is why plain Data Lakes struggle with updates and deletes — and why table formats like Delta Lake exist.)
2
Operations are whole-object. The basic verbs are PUT (write a whole object), GET (read an object, or a byte-range of it), DELETE, and LIST (find keys by prefix). There is no "update row 7." There is, however, a ranged GET — you can ask for "bytes 1000–2000 of this object" — and that one capability is what makes Parquet's skip-to-the-footer trick possible (Parquet and its footer are defined in the callout just below).
3
It's cheap and effectively infinite, but has latency. Storage is a few cents per GB per month and you never run out of space. But each request has real latency (tens of milliseconds) and a cost per request. So the game is: read fewer, bigger objects rather than many tiny ones.

🧪 Ranged GET, concretely

First, three terms we'll lean on (all defined in the Glossary too): Parquet is the columnar file format almost every lake uses; inside one Parquet file the rows are stored in blocks called row groups; and a small index at the very end of the file, the footer, lists every row group, where it sits, and the min/max value of each column in it.

Now the trick: when an engine reads a 512 MB Parquet file, it does not download all 512 MB. It first issues a tiny ranged GET for just the footer. From that index it learns exactly which byte ranges hold the rows it wants, then issues more ranged GETs for only those ranges. A single object gets read in a few small, targeted pieces. Keep this in mind — it's the bridge between the Storage layer and the Compute layer.

Physical layout: why folder structure is a performance decision

Recall one limitation from the rules above: object storage can't search inside files. Its only fast lookup is by prefix — the leading part of a path, like a folder name: "give me every file whose path starts with X."

That's why where you put a file decides query speed. Put the rows a query wants under one prefix and the engine reads just that prefix; scatter them and it must open everything, then throw most of it away. Compare two ways to store the same orders data:

The difference is night and day:

  • Flat — all 10,000 files share the one prefix orders/. "June 1 orders" has no narrower prefix to ask for, so the engine opens all 10,000 files and discards the 9,998 that don't match.
  • Partitioned — each day gets its own sub-folder. "June 1 orders" reads just the order_date=2026-06-01 folder: 2 files.

Same data, same query — but layout alone turned 10,000 file-opens into 2. Splitting files into sub-folders by a column's value (here, the date) is called partitioning; skipping the folders a query doesn't need is partition pruning. It's the biggest speed-up you get from storage layout alone. (Strictly, storage only makes pruning possible through how files are arranged; the engine — using the catalogue from Layer 2 — is what actually decides which folders to skip.)

🐝 What "Hive-style partitioning" means

You'll constantly see partition folders named column=value — like order_date=2026-06-01/ instead of plain 2026-06-01/. That name=value style is called Hive-style partitioning (after Apache Hive, the engine that made it standard).

The point of writing the column name into the folder: an engine can read order_date=2026-06-01 and instantly know "every file in here has order_date = 2026-06-01" — without opening a single file. With plain 2026-06-01/ it would just see anonymous folders and couldn't tell which column the value belongs to. So this naming is what lets engines recognise partitions automatically and prune them.

⚠️ One caution: don't over-partition

Partitioning on a column with millions of values (like user_id) makes millions of tiny folders, each holding a few rows — and now every query has to wade through millions of tiny files, slower than not partitioning at all. Pick columns that are low-cardinality and commonly filtered, like date or region, so each partition is big enough to hold at least a few sizable files.

Partitioning is the storage layer's main layout lever, but not the only way to make queries read less. More advanced layout techniques — such as Z-Order and clustering — are a deeper, table-format-specific topic beyond this post's scope.

The small-file problem

Because each object read costs a request and some latency, many tiny files are far slower to read than a few big ones, even for the exact same total data.

The pain

A streaming job writes a small file every second or two. After a month you have ~2 million files of ~20 KB each (≈40 GB total). A query that should take seconds now spends minutes just opening 2 million objects — the per-request overhead dominates, and the engine's task scheduler drowns in tiny tasks.

The fix

Periodically compact small files into larger ones (a target of 128 MB–1 GB per file is typical). Plain lakes do this with a manual rewrite job; Lakehouse table formats automate it (e.g. Delta's OPTIMIZE). The goal: fewer, fatter files so reads are dominated by streaming bytes, not by opening objects.

Storage classes & lifecycle (a quick cost lever)

Object storage offers tiers at different price/speed trade-offs: a hot/standard tier for frequently read data, and cheaper cold/archive tiers (e.g. S3 Glacier) for data you rarely touch but must keep. You set lifecycle rules like "move anything older than 90 days to archive, delete after 7 years." Archive tiers cost a fraction of standard storage but take minutes to hours to retrieve — fine for compliance backups, not for live queries.

Layer 1 in one line

Storage is a cheap, infinite, immutable key→bytes dictionary. It knows nothing about tables — it just hands back bytes for a key, or a byte-range of one. Everything clever (tables, skipping, updates) is added by the layers above.

03. Layer 2 — Metadata, in depth

The storage layer is just files. If you handed Spark a bucket and typed SELECT * FROM orders, it would have no idea what "orders" is, where its files live, or what columns they have. The metadata layer is what fills that gap — it's the bridge that turns a pile of files into a table you can name.

What a metastore actually stores

The classic metadata service is the Hive Metastore. Despite the scary name, it is just a small ordinary database (usually MySQL or PostgreSQL running behind the scenes) holding a catalogue. For each table it records, in plain rows:

What the metastore holds for one table
Table name : orders
Location   : s3://my-company-lake/orders/
Format     : parquet
Schema     : order_id    INT
             customer_id INT
             category    STRING
             amount      DOUBLE
             order_date  DATE
Partitions : order_date
Partition values known:
   order_date=2026-06-01  →  s3://…/orders/order_date=2026-06-01/
   order_date=2026-06-02  →  s3://…/orders/order_date=2026-06-02/
   …

That's it — a phonebook. Instead of "person → phone number," it stores "table name → location + columns + known partitions." When you run a query, the engine asks the metastore three questions and gets instant answers without touching storage:

  1. Where does orders live? → an S3 prefix.
  2. What columns and types does it have? → the schema, so the engine can plan the query and check that amount exists.
  3. What partitions exist? → so for WHERE order_date = '2026-06-01' the engine knows to read only the order_date=2026-06-01 prefix. This is partition pruning: the engine does the skipping, using the partition list the catalogue hands back — and it happens before a single file is opened.

💡 Why a separate layer at all?

Because storage is "dumb" by design (just keys and bytes), something has to remember what those bytes mean. Putting that knowledge in one shared catalogue means every engine — Spark, Trino, Athena — agrees on what orders is. Without it, each tool would need its own private notion of your tables, and they'd disagree.

The flavours: Hive Metastore vs Glue vs Unity Catalog

CatalogueWhat it isYou manage?Extra powers
Hive MetastoreThe original open-source catalogue (a service backed by MySQL/Postgres)Yes — you run the server & its databaseJust the core name→location mapping
AWS Glue Data CatalogAmazon's fully-managed, serverless metastoreNo servers to runAuto-discovery "crawlers," tight AWS integration
Unity CatalogDatabricks' next-gen catalogueManaged for youColumn-level access control, data lineage, works across clouds

All three do the same core job — map a table name to its files and schema — but they differ a lot in who runs them and what they add on top. Here's each in more detail:

Hive Metastore — the original

Born with Apache Hive in the Hadoop era, it's the catalogue everything else is compatible with. Under the hood it's just a small relational database (usually MySQL or PostgreSQL) holding the table records, fronted by a service that engines talk to over a standard API. Its big strength is being open and engine-neutral: Spark, Trino, Presto, and Flink can all read the same metastore, which is why it became the de-facto standard. The downsides: you operate it yourself (run and scale that database and service), it has no built-in security or governance (access control has to be bolted on by the surrounding system), and it slows down once a table has hundreds of thousands of partitions.

AWS Glue Data Catalog — the managed, serverless version

Amazon's drop-in replacement for the Hive Metastore: same concepts and API, but nothing to run — AWS hosts it and you pay per object stored and per request. It's the default catalogue across AWS analytics (Athena, EMR, Redshift Spectrum, Glue ETL), and it adds crawlers to fill itself in automatically. A crawler is a job you point at an S3 location (or a database, JDBC source, etc.): it samples the files it finds, works out the file format (Parquet, CSV, JSON…), infers each column's name and type, detects Hive-style partition folders, and then creates or updates the matching table definition in the catalogue — so nobody has to hand-write the schema. Put it on a schedule and it keeps the catalogue current as data lands: new partitions and new columns get registered on the next run. (The catch: its type-guessing is heuristic and can occasionally mislabel a column — e.g. read a ZIP code as an integer — so inferred schemas are worth a sanity check.) The trade-off overall is lock-in: it lives inside AWS, and governance comes from AWS IAM / Lake Formation rather than from the catalogue itself.

Unity Catalog — the governance-first catalogue

Databricks' newer catalogue, built for the things the Hive Metastore never handled. On top of the basic name→location mapping it adds fine-grained access control (grant permissions down to a single column or row), data lineage (what produced a table and what reads it), auditing, and a three-level namespace (catalog.schema.table) that works across workspaces and clouds. It also governs more than tables — files, ML models, even dashboards — from one place, and it's now open-sourced. If you're on Databricks it's the default; the cost is that it's centred on the Databricks ecosystem.

In short: Hive Metastore if you want open and self-managed, Glue if you're on AWS and want zero servers, Unity Catalog if you need real governance (security, lineage, auditing) across an organisation.

🗂 The three-level name

Modern catalogues organise tables as catalog.schema.table (e.g. prod.sales.orders) — three levels, like country → city → street. The first level (catalog) is the newer addition that lets one organisation cleanly separate, say, prod from dev data under a single governed system.

Heads-up on a name clash: "schema" here means a grouping of tables (the "city" level) — a totally different use from the other "schema" in this article (a table's column list). Same word, two meanings; the data world overloads it and you'll see both.

Where the classic metastore strains — and what was added on top

The Hive Metastore has one structural gap: it tracks tables and partitions, but not individual files. It knows "the order_date=2026-06-01 partition lives at this prefix," but to find out which files are actually in that prefix right now, the engine falls back to a live LIST on storage. And because that answer comes straight from storage (which, as we saw in Section 02, has no atomic "official file set"), real problems follow:

The pain

  • Slow listing. A table with hundreds of thousands of partitions means huge, slow LIST calls just to plan a query.
  • No atomic changes. Because the file set comes from a storage LIST (not the catalogue), a reader running while a write job is half-finished sees a mix of old and new files — there's no clean "this is the official set of files" marker.
  • No file-level skipping. The catalogue has no per-file min/max stats, so it can't help skip individual files (only whole partitions).

The fix: table formats (Delta Lake, Iceberg, Hudi)

A table format writes an extra metadata file alongside your data that lists the exact set of files making up the table right now, plus per-file statistics. Instead of LISTing storage and guessing, the engine reads this manifest and knows precisely which files to read — atomically and with file-level skipping. This is the leap from a plain Data Lake to a Lakehouse.

🧭 Two metadata "sub-layers," don't mix them up

It helps to see metadata as two cooperating things:

  • The catalogue (Hive/Glue/Unity) answers "what is the table named orders and where does it start?"
  • The table format's manifest (Delta/Iceberg/Hudi, stored next to the data) answers "which exact files are in it right now, and what are their stats?"

A modern Lakehouse uses both: the catalogue points to the table, the table's manifest points to the files. The deep dive on the manifest lives in the Delta Lake, Under the Hood post.

Layer 2 in one line

Metadata is the catalogue that turns files into named tables — it answers where, what columns, and which partitions, so the engine can plan a query and prune before reading. Table formats extend it down to the individual file.

04. Layer 3 — Compute, in depth

The compute layer is the worker on the top shelf: it takes your query, figures out a plan, then reads the right files from storage and computes the answer. The defining feature of this layer in a Data Lake is that it is completely separate from storage — you rent it only while a query runs, then give it back.

Separation of compute and storage — the payoff of decoupling

As Section 01 noted, a traditional database fuses storage and compute into one always-on box; a Data Lake splits them, and the top layer is where that split actually pays off. The two halves behave completely differently:

  • Storage stays on always (it's cheap — a few cents/GB/month) and holds your data permanently.
  • Compute is rented on demand. Spin up 10 machines for a 20-minute job, then shut them off and stop paying. Need a huge one-off job? Spin up 200 machines for an hour. Nothing about your stored data changes.

This is why two teams can run two different engines on the same data, and why your nightly bill can drop to near-zero compute while the data sits safely in storage.

What a cluster is made of: driver + executors

A compute engine like Spark runs as a cluster — a group of rented virtual machines with two roles:

  • The driver (1 machine) — the foreman. It talks to the metastore, builds the query plan, splits the work into tasks, and hands tasks out. It does not read the bulk data itself.
  • The executors (many machines) — the workers. Each executor reads some of the files from storage and processes them in parallel with the others.

Because the files are independent objects in storage, the driver can hand each executor a different set of files and they all read at once. Ten executors reading ten files finish the reading in roughly the time it takes one executor to read one — that's the parallelism a Data Lake is built to exploit. (This is near-linear speedup for the read phase only; the final combine step — e.g. merging partial GROUP BY sums — plus scheduling overhead and the occasional slow machine mean real end-to-end speedup is a bit below perfect.)

How the engine plans a query (the four steps inside the driver)

1
Parse. Read your SQL and check it's valid — turn the text into a tree the engine can reason about.
2
Resolve against the catalogue. Ask the metastore (Layer 2) where orders lives and what its columns are. Confirm category, amount, and order_date really exist.
3
Optimise. Rewrite the plan to do the least work — drop unused columns, reorder joins, and bake in three distinct read-reducing tricks: partition pruning (skip whole prefixes the filter rules out, using the catalogue's partition list), predicate pushdown (push the row filter amount > 500 down into the file scan so it runs during the read, not after), and data skipping (use Parquet's per-file min/max stats to skip row groups that can't match).
4
Schedule & execute. Split the surviving files into tasks, send them to executors, and combine their partial results into the final answer.

🎯 The engine skips data twice — coarse, then fine

The engine reads as little as possible by skipping in two stages: (a) using the metadata layer's partition list, it prunes whole partitions (keeps only the order_date=2026-06-01 prefix), then (b) inside each surviving file, the storage layer's Parquet footers let each executor skip row groups that can't match ("amount max is 200, skip this group"). Coarse filtering up top, fine filtering down below — and in both cases it's the engine doing the skipping, the layers below just hand it the information to do it with.

Different engines for different jobs

EngineBest atStyle
Apache SparkHeavy batch jobs & transformations (ETL — extract/transform/load), ML pipelinesBig clusters, throughput-focused
Trino / PrestoFast interactive SQL across sources, dashboardsAlways-on, low-latency queries
Amazon AthenaOccasional SQL with zero servers to manageServerless, pay-per-query (Trino under the hood)
Databricks SQLLakehouse BI (business-intelligence dashboards) & SQL on Delta tablesManaged, optimised for Delta

All of them read the same files through the same catalogue. That's the decoupling paying off: the right tool for each job, no data copies.

Autoscaling & the compute cost model

Because compute is rented, scale becomes a dial. Autoscaling adds executors when a job is big and removes them when it's idle. Serverless options (Athena, serverless Spark/SQL) take this further: you don't manage machines at all and pay per query or per second. The mental model that keeps bills sane:

  • Storage cost = how much data you keep (always on, but cheap).
  • Compute cost = machines × time they're running (expensive, so keep it short and shut it off).
  • Reading less data (good partitioning + Parquet skipping) makes compute finish faster, which directly lowers the compute bill. The layers are linked: smart Storage and Metadata make Compute cheap.

Layer 3 in one line

Compute is the rented, parallel engine that plans a query (driver), reads files in parallel (executors), and is billed only while running — so the cheaper you make its reads, the cheaper the whole lake.

05. One query through all three layers

Let's tie everything together by following one real query from the moment you press run to the rows coming back. This is the payoff — watch how each layer does exactly its job and nothing more.

SQL · the query we'll trace
SELECT category, SUM(amount) AS revenue
FROM orders
WHERE order_date = '2026-06-01'
  AND amount > 500
GROUP BY category;

We'll assume the common plain-lake setup: Parquet files plus a Hive-style metastore (no table format yet). That means one extra LIST on storage to discover the files in a partition — exactly the step Section 03 flagged. With a Lakehouse table format, that LIST disappears (the manifest already names the files); everything else is identical.

1
Compute parses your SQL into a plan and sees it needs the orders table, columns category, amount, order_date.
2
Compute asks Metadata: "where is orders, and what are its columns?"
3
Metadata answers with the S3 prefix and the schema. It confirms category, amount, and order_date exist and that the table is partitioned by order_date.
4
Compute prunes partitions using that list: because of WHERE order_date = '2026-06-01', only the order_date=2026-06-01/ prefix is relevant, so the engine drops every other day before any file is opened. (Metadata supplied the partition list; the engine did the skipping.)
5
Storage serves bytes: the engine first LISTs that one order_date=2026-06-01 prefix to discover which files are in it — here, 2 files. (This LIST is the plain-lake step from Section 03; with a table format the manifest would already name the files and no LIST is needed.) Executors then issue ranged GETs for those files. They read each Parquet footer, use the amount min/max stats to skip row groups that can't contain amount > 500, and read only the category and amount column chunks — ignoring every other column.
6
Compute finishes: each executor sums its slice, the driver combines the partial sums per category, and the final rows travel back up to you.

The big picture

A query on a 10,000-file table physically read just two files (the only two in the matching day), and inside them just two columns of a few row groups. Metadata did the coarse cut (which files), Storage's Parquet footers did the fine cut (which parts of those files), and Compute ran the whole thing in parallel and only while needed. Three layers, each doing one job — that's a Data Lake working as designed.

Glossary

Every term used in this post, defined in one place. Skim it first if you're new, or jump back here whenever a word feels rushed.

Object storage
Cloud file storage built for storing huge numbers of files cheaply (Amazon S3, Azure ADLS, Google GCS). You write whole files and delete whole files; you cannot edit one in place. Pricing is single-digit cents per GB per month (e.g. S3 Standard is ~$0.023/GB/month as of 2026, and varies by tier and region) — we round it to "a few cents" throughout.
Object / key
In object storage, one file is called an object, and its full path (e.g. s3://lake/orders/part-001.parquet) is its key. The "folders" in that path are not real folders — just text in the key (see Section 02).
Parquet
A columnar file format for analytics: stores data column-by-column and compresses well. Internally a Parquet file is split into row groups (horizontal blocks of rows — say a few hundred thousand each); within each row group there is one column chunk per column. Every column chunk carries min/max statistics, and a small footer at the end of the file indexes all of this — so an engine can skip whole row groups it doesn't need.
Footer
A few-KB index at the very end of a Parquet file. It lists every row group, where each one sits (its byte offset), and each column's min/max stats — so a query reads the footer first, then jumps straight to only the bytes it needs.
Atomic
All-or-nothing. An atomic change is never observed half-done — a reader either sees the state before it or fully after it, never a partial mix.
Lakehouse
A Data Lake plus a table format, which adds warehouse-like guarantees (atomic changes, file-level skipping, schema rules) on top of cheap object storage.
ETL / ELT
Pipelines that Extract data, Transform it, and Load it (ELT just reorders: load first, transform in place). The "transformations" a compute engine runs.
BI
Business Intelligence — the reports and dashboards that business users read (think charts of "revenue by region").
Metastore / Catalogue
A small database that maps a table name to its storage location and columns. Examples: Hive Metastore, AWS Glue Data Catalog, Databricks Unity Catalog.
Partition
Splitting a table's files into sub-paths by a column value (e.g. one folder per day). Queries that filter on that column read only the matching sub-paths.
Cluster
A group of rented virtual machines that work together. Compute engines split a job across the machines so they run in parallel, then release them when done.
Driver & executors
In Spark, the driver is the one machine that plans the work; executors are the worker machines that read files and crunch the data in parallel.
Partition pruning
Skipping whole partition folders that a query's filter rules out — e.g. WHERE order_date = '2026-06-01' reads only that one prefix. The engine decides what to skip using the partition list from the catalogue (Metadata layer); it's the coarsest cut and happens before any file is opened.
Predicate pushdown
Pushing a row-level filter (like amount > 500) down into the file scan, so rows are filtered while the data is being read rather than loaded first and filtered after. Less data flows up into the engine.
Data skipping
Skipping blocks inside a file — Parquet row groups — by checking the footer's per-block min/max stats against the filter (e.g. "this row group's max amount is 200, so it can't contain amount > 500 — skip it"). The fine-grained cut, done at the Storage layer once a file is already in play.
Schema
The list of a table's column names and their data types — e.g. order_id INT, amount DOUBLE.
Table format
A spec (Delta Lake, Apache Iceberg, Apache Hudi) that adds an extra metadata file next to your data files to track exactly which files make up a table right now. It works with the catalogue (it doesn't replace it) and extends the metadata layer down to the individual file — covered in Section 03.
