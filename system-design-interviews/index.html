Data Engineering System Design Interviews: The Complete Framework

A from-zero reference for the data engineering system design round: a repeatable 6-step framework — requirements, pipeline design, data modeling, storage & formats, data quality, and scalability — plus every sub-topic that comes up under it (Lambda/Kappa, medallion architecture, star schema vs. OBT, SCD, partitioning, Parquet/Avro, Delta/Iceberg/Hudi, data contracts, idempotency, backfills, schema evolution), closing with a full worked interview walkthrough in the style of a real Uber Eats / Zomato / Swiggy-scale question.

~25 min read · Reference · System Design

❯ The one idea (read this first)

Data engineering system design interviews aren't hard because you don't know Kafka or Spark — they're hard because they're open-ended. Ask ten senior engineers to design the same pipeline and you get ten valid, different architectures, each with different trade-offs. The interviewer isn't grading whether you land on "the" answer — there isn't one — they're grading whether your reasoning survives when a constraint changes (volume goes 10x, the SLA moves from a day to ten minutes, a source becomes unreliable). A repeatable 6-step framework — requirements → pipeline design → data modeling → storage & formats → data quality → scalability — turns that open-endedness into a structured conversation you can walk through under pressure, on any prompt, in any domain.

01. Why data engineering system design interviews are hard

Most candidates walking into this round have done the reading — books, videos, plenty of terminology. And yet a surprising number still fail. Not because they lack knowledge (below a certain bar you fail by default — that's table stakes), but because the round is testing something books don't teach well: how you navigate ambiguity.

Compare it to a coding interview. Even the hardest DSA problem has a destination — a two-pointer problem has a correct answer, a dynamic programming problem has an optimal substructure you can converge on by breaking it into subproblems. System design has no such floor. It is open-ended in a way coding questions simply aren't: there is no single correct architecture. Ask ten senior data engineers to design the same ingestion pipeline and you'll get ten valid designs, each trading off cost, latency, complexity, and failure handling differently.

Which means the interviewer isn't checking whether you "arrived" anywhere. They're checking whether your reasoning holds up when they start pulling on the constraints you assumed — what happens when volume grows 10x overnight, when the freshness requirement moves from "once a day" to "under ten minutes," when a source system starts sending malformed events. That pressure-testing is the actual interview; the initial design is just the opening move.

️ Software system design prep doesn't transfer

If your prep has been "design a URL shortener" / "design Twitter's feed," you've trained the wrong muscle. Those questions are about API design, sharding, and caching for request/response traffic. Data engineering design is about data — its volume, its shape, its correctness, its staleness — moving through pipelines, not services answering requests. The vocabulary, the failure modes, and the trade-offs are genuinely different.

The good news: despite the topic list looking huge — ingestion patterns, data modeling, batch vs. streaming, schema evolution, idempotency, backfills, cost and scale — a single framework applies to essentially any prompt in this space, whether it's a lakehouse for e-commerce, a recommendation pipeline, fraud detection, or a customer-360 platform. That framework is the spine of this guide, and the closing section (§09) walks it end-to-end against a real-style interview question.

02. The 6-step framework

Bring this into every interview as your mental map. Interviewers often use a "wide, then zoom" pattern: they open with a broad, almost vague prompt ("design a pipeline to track user activity for an e-commerce site"), then drill into two or three of these six steps based on what they want to probe. Walking through all six — even briefly — before diving deep on any one is what signals structure.

1
Requirements gathering — who is this for, what do they need, how fresh does it have to be, how much data is there. Skip this and everything downstream is guessing.
2
Pipeline design — how data physically moves from source to destination: batch vs. streaming, Lambda vs. Kappa vs. a lakehouse blend, which orchestrator (Airflow, Prefect, Dagster, Mage) coordinates it.
3
Data modeling — the shape of the data at rest: medallion layers, star schema vs. one-big-table, slowly changing dimensions, partitioning strategy.
4
Storage & file formats — Parquet vs. Avro, and why a lakehouse table format (Delta Lake, Iceberg, Hudi) sits on top of raw files; relational vs. non-relational stores for the serving layer.
5
Data quality & observability — how you know the pipeline is producing correct data, not just a result: quality dimensions, contracts, monitoring.
6
Scalability, backfills & DataOps — can it take 10x load, can you safely reprocess history, is a retry ever going to duplicate data.

The framework is a checklist, not a script

You won't spend equal time on all six steps in every interview — a 45-minute session might spend 10 on requirements, 20 on pipeline design + modeling, and 15 on quality/scale. The value of the framework is that you always know where you are and what's left to cover, so you never freeze staring at a blank whiteboard.

03. Step 1 — Requirements gathering (functional + non-functional)

The single most common mistake in this round: candidates hear the prompt and immediately start drawing boxes labeled Kafka, Spark, Airflow. Resist that for the first five minutes. The prompt is deliberately underspecified — that's the point — and the questions you ask are themselves part of what's being graded.

Functional requirements — what the system must do

Three questions pin this down:

  • Who is the end user? A marketing analyst running ad-hoc SQL, an ML team feeding a feature store, and a product team watching a live metric all imply completely different architectures — this one answer alone can flip batch vs. streaming.
  • What do they actually need? Aggregated metrics, raw event access, or historical snapshots are three different deliverables with three different modeling approaches.
  • How will they access it? SQL against a warehouse, a BI tool like Tableau or Power BI, or a REST API each constrain the serving layer differently.

Worked example: a marketing team wants to analyze an e-commerce purchase funnel — home page → search → product view → add to cart → purchase — segmented by user type and device. A concrete funnel might look like:

That single exercise — computing the drop-off and slicing it by device — is what turns a vague ask into a concrete functional requirement: daily funnel metrics, segmented by user type and device type, delivered to Tableau.

Non-functional requirements — how the system must behave

Four questions, and the answers to these are what actually pick your tools:

The same vague prompt — "design a pipeline to track user activity" — resolves into two entirely different systems depending on the answers:

Scenario A — marketing, daily report, 1-hour refresh tolerance

→ a batch pipeline: Spark, Airflow, Delta tables, a conventional fact/dimension model, served to Tableau.

Scenario B — ML team, near-real-time recommendation features

Think "trending near you right now" or a feed re-ranked using clicks from the last 30 seconds. → a streaming pipeline: Kafka, Flink or Spark Structured Streaming, a Delta table for history plus a low-latency online feature store for serving.

Back-of-the-envelope calculations

This is the step candidates skip most, and the one that instantly elevates an answer — it shows you think in numbers, and numbers are what actually justify tool choices instead of name-dropping them. Treat your assumptions as negotiable: state them, and let the interviewer correct you if they had different numbers in mind.

Five concrete decisions fall directly out of that math:

  • Spark over Pandas/Polars — the volume rules out single-node tools.
  • Parquet/Delta as the storage format — compression brings a year of raw events down to a very manageable footprint.
  • Airflow for orchestration — nothing here needs sub-hour freshness.
  • No streaming infrastructure — you can justify not building something, which is just as valuable a signal as justifying what you do build.
  • Partition bronze by event date (yyyymmdd) — so the daily/hourly batch job only scans the partitions it needs.

️ The trap: asserting numbers instead of negotiating them

Don't silently pick "5 million DAU" and run with it. Say it out loud — "let's assume roughly 5M daily active users, does that sound right for this business?" — and adjust live. The interviewer is watching how you handle uncertainty, not whether you memorized a number.

04. Step 2 — Pipeline design (batch vs. streaming)

With requirements pinned down, the first real architectural fork is batch vs. streaming — and by extension, whether you need both at once (Lambda) or can get away with one unified path (Kappa).

Batch — the default for most real workloads

Batch is the default for the large majority of real-world pipelines. It's the right call whenever data doesn't need to be fresh to the second — hourly reporting, daily analytics — and cost efficiency matters more than shaving off minutes of latency. The standard stack: Spark for processing, Airflow / Prefect / Dagster for orchestration, landing data through bronze → silver → gold in object storage (S3 / ADLS).

Concrete shape: an e-commerce daily sales report. An Airflow DAG kicks off at 2:00am, pulls the previous day's orders from Postgres into bronze; a Spark job cleans and types it into silver; another aggregates by product and region into gold; by 6:00am the numbers are sitting in a Tableau dashboard.

Streaming — when latency outranks cost

Streaming earns its complexity when the SLA is under a minute — real-time dashboards, fraud detection, event-driven systems where staleness itself is the failure mode. Stack: Kafka for ingestion/buffering, Spark Structured Streaming or Flink for processing, Delta Lake for the historical record, and a low-latency store (Redis, DynamoDB, Cassandra) for serving. Kafka is what makes this decoupled: it retains events for a configurable window and lets each consumer read at its own pace via offsets, so a slow or crashed consumer never blocks the producer.

Concrete shape: a ride-sharing driver's live earnings. A "ride completed" event (fare, surge multiplier, driver ID) lands on Kafka; Spark Structured Streaming enriches it with driver metadata from a lookup table, updates a running total, and writes the result to Redis — the driver's app reads it back instantly.

Most real products need both — a live view and an accurate historical view — over the same underlying events. That need is exactly what Lambda and Kappa architectures answer, in two different ways.

Lambda architecture — two paths, Uber-style

Lambda runs the same incoming data down two parallel paths: a batch layer for historical accuracy and a speed layer for real-time response — both eventually feeding a serving layer that a client reads from.

Two consumer groups read the same Kafka topic independently. The speed layer (Flink) computes surge pricing and driver ETA within milliseconds and writes to a cache — "fare ₹330, driver arriving in 4 minutes." On the other path, Firehose just buffers the same events straight into S3, with zero awareness of what Flink is doing — if Flink goes down, Firehose keeps writing uninterrupted. That decoupling is the whole point: each layer fails independently. The batch layer proper is what runs on top of that landed data: nightly Airflow-triggered EMR jobs that walk bronze → silver → gold to compute things that need history — which pickup zones spike hardest on Friday evenings, which routes have 3x higher cancellation rates, where to pre-position drivers ahead of a stadium event letting out.

The serving layer blends both because neither is sufficient alone: real-time data is noisy and lacks context (47 requests, 12 drivers available right now → roughly 1.8x surge), while batch data is accurate but already stale by the time you read it (historically, this zone averages 2.1x surge on Friday 7–9pm). Combined, you get a grounded estimate instead of either a jumpy real-time number or a lagging historical one.

️ Lambda's real cost: two codebases

The batch and speed layers usually implement similar business logic twice, in two different frameworks (Flink vs. Spark batch jobs, say). Every change to that logic has to be made — and tested — in both places, and they can silently drift out of sync. This is the exact pain Kappa architecture is a reaction to.

Kappa architecture — one path, replay instead of a second layer

Kappa's pitch: why maintain two systems? Run everything through one streaming pipeline, and when you need to reprocess history, replay it through that same pipeline instead of maintaining a separate batch codebase.

The structural shift from Lambda: Kafka's retention is configured for days, weeks, or months, so Kafka is the historical store, not just a transient buffer. If business logic changes and the last two weeks need reprocessing, you spin up a new consumer group pointed at the offset from two weeks ago, replay events through the (updated) processor, and once it catches up to present, swap the new output in for the old.

️ Kappa's limit: replaying deep history is slow and expensive

Reprocessing six months of data means either Kafka has retained six months (costly to keep on a log built for streaming) or the data's simply gone. And replaying millions of events through a processor sized for small continuous batches is far slower than a Spark job reading that same data in parallel off files. It works, but it doesn't scale gracefully.

Which is why, in practice, most real systems don't run pure Kappa. They land the same events in both Kafka (for the live path) and S3/ADLS (for cheap, parallel historical reprocessing) — in effect a lakehouse hybrid that keeps Lambda's decoupling but avoids maintaining two separate business-logic codebases. This is the pattern worth naming out loud in an interview: "pure Kappa" is a clean idea, but the pragmatic default is this hybrid.

05. Step 3 — Data modeling

This is where interviewers push hardest with follow-ups: "why not denormalize?", "what happens when this dimension changes?", "how does this hold up at 100x the rows?" Four sub-topics carry almost all of the weight: the medallion architecture, star schema vs. one-big-table, slowly changing dimensions, and partitioning strategy.

Medallion architecture — Bronze, Silver, Gold

A layered way to organize data inside a lakehouse, where each layer answers a different question about trust and usability.

  • Bronze — raw. Ingested exactly as the source sent it (JSON dumps, CDC logs, raw API payloads), no cleaning, no transformation. The philosophy: if anything breaks downstream, you can always reprocess from bronze, because nothing was thrown away.
  • Silver — cleaned & structured. JSON is parsed, schemas enforced, records deduplicated and correctly typed. This is typically where the dimensional model (fact/dimension tables) lives, and it's the layer analysts and engineers treat as the trusted source of truth.
  • Gold — aggregated. Pre-joined, pre-aggregated metrics tables that analysts query directly, with the expensive joins already done upstream.

Concretely: bronze holds a "song played" event (timestamp, user ID, song ID, duration played, device, app version) — and across app versions you should expect schema drift, nulls, and duplicates here, by design. Silver splits that into fact_stream (user, song, played-at, duration, device type) and dim_song (song ID, title, artist, genre, release year). Gold rolls those up into daily_artist_streams — one row per artist per day, with total streams and unique listeners — so a dashboard query never has to join fact and dimension tables live.

Star schema vs. One Big Table (OBT)

Inside silver/gold, you still choose how normalized the tables are.

Star schema keeps facts and dimensions separate (fact_orders, dim_customer, dim_product). BI tools are built around this shape, dimension tables stay small and cheap to update, and joins on well-indexed keys are fast. OBT pre-joins everything into one wide table, which pays off when there's a specific, high-frequency query pattern — e.g. a dashboard where many users repeatedly ask "revenue by region and category over the last 30 days" — avoiding a repeated 5-table join, shuffle, and all.

️ The OBT trap: cheap reads, expensive updates

Rename "clothes" to "apparel" in a star schema and you update one row in dim_product. In an OBT, that same rename may be duplicated across millions of order rows — and it compounds when several dimensions can change independently. Pick OBT only when you can name the specific query pattern it serves; don't default to it "for simplicity."

Slowly changing dimensions — SCD Type 1 & Type 2

A user on a music app upgrades from free to premium. Do you overwrite the old record, or keep the history?

  • SCD Type 1 — overwrite. Simple, but you lose the ability to answer "how many users were on the free tier six months ago" — the past state is gone.
  • SCD Type 2 — versioned rows. Keep both rows, tracked with validity dates: the old row's valid_to gets set to the change date, and a new row is inserted with valid_to = NULL (meaning "currently active").

One gotcha worth knowing before you write this on a whiteboard: a single MERGE statement can't both close the old row and insert the new one for the same changed user — a source row either matches the target (and takes the WHEN MATCHED branch) or it doesn't (and takes WHEN NOT MATCHED), never both. So SCD2 upserts are done in two explicit steps: close whatever changed, then insert the new current rows.

sql — SCD2 upsert, step 1: close changed rows
UPDATE dim_user AS target
SET valid_to = source.change_date, is_current = false
FROM staged_user_changes AS source
WHERE target.user_id = source.user_id
  AND target.is_current = true
  AND target.plan <> source.plan
sql — SCD2 upsert, step 2: insert new current rows
INSERT INTO dim_user (user_id, plan, valid_from, valid_to, is_current)
SELECT source.user_id, source.plan, source.change_date, NULL, true
FROM staged_user_changes AS source
LEFT JOIN dim_user AS target
  ON target.user_id = source.user_id AND target.is_current = true
WHERE target.user_id IS NULL  -- brand-new users, and users just closed in step 1

SCD2 is the standard choice whenever a dimension's history genuinely matters for analysis — subscription tier over time, a customer's region, an account's status. (You'll also see SCD Type 3 named alongside 1 and 2 — it stores a limited history as extra columns, e.g. previous_plan next to current_plan, a middle ground that only keeps the immediately-prior value rather than full history.)

Partitioning, bucketing & liquid clustering

The default that's almost always right: partition by event date. Time-series data is filtered by date in the overwhelming majority of queries, so this one choice prunes most of the table before a query even starts scanning. If a second filter is heavily used — say 80% of queries also filter by country or region — add it as a secondary partition key alongside date.

️ The trap: partitioning is a bet on query patterns that stay fixed

Fixed partition columns require committing upfront. If usage shifts — say from date-first filtering to region-first — the old layout stops helping and fixing it means rewriting the whole dataset. Liquid clustering (a Databricks Delta Lake feature) exists precisely for this: instead of rigid physical partitions, it dynamically reclusters data around chosen columns, and you can change which columns it clusters on later without a full rewrite.

Bucketing is a separate, complementary technique: hashing a high-cardinality join key (e.g. user_id) into a fixed number of files so rows that will be joined together end up co-located. That reduces shuffle — the expensive redistribution of data across a cluster's nodes that a join normally requires — on large joins, and it's orthogonal to date partitioning; you can partition by date and bucket by a join key at the same time.

06. Step 4 — Storage & file formats

Once the logical model is fixed — bronze/silver/gold, star schema or OBT, how dimensions version themselves — the physical files backing it raise their own set of decisions: what format they're written in, and what sits on top of them to make them safe to write to concurrently.

Motivating example to hold in mind for this whole section: a table with 200 columns and a billion rows, where analysts routinely query only 3 of those columns. A row-based format has to read every column of every row just to hand you the 3 you asked for. A columnar format reads only those 3 columns and ignores the rest — same table, same query, but one engine touches under 2% of the data the other one touches.

Parquet — column-based, built for reads

Parquet stores data column-by-column: every value of one column sits together in memory/on disk, rather than a row's values sitting together.

That selective read is called column pruning. Parquet also keeps per-file statistics (min/max values, row counts), so a query engine can skip entire files that can't possibly match a filter — file skipping — before it even opens them. This is why Parquet is the default choice for analytical, read-heavy workloads: its performance advantage comes exactly from never touching irrelevant columns or files.

Avro — row-based, built for writes

Avro is Parquet's mirror image: row-based, and optimized for write-heavy workloads. In a streaming system, every event — a ride completed, a button clicked, an order placed — has to be written as one complete record the instant it arrives. Avro serializes each event as one row with all its fields packed together, written in a single shot. A columnar format, by contrast, needs to collect a batch of rows first and reorganize them column-by-column before writing — fine for an hourly batch job, but a poor fit for 10,000 events/sec landing on Kafka that each need to be durably written immediately. That's why Avro (often paired with a schema registry) is the standard format inside Kafka topics, while Parquet takes over once that data is batched and landed for analytics.

Why Delta Lake / Iceberg / Hudi matter

Early data lakes were just raw Parquet files dumped into S3 with nothing managing them — and that created four concrete problems:

No ACID guarantees

Two concurrent Spark jobs writing to the same table can silently overwrite each other's output — records corrupted or missing, with no error raised.

No schema enforcement

Someone writes a string into a column meant to hold an integer, and it silently succeeds — you only find out when a downstream job breaks.

The small-file problem

A streaming job flushing thousands of tiny Parquet files means every query pays the I/O overhead of opening and closing all of them — much slower than reading one large file with the same data.

No historical versioning

Once a file is overwritten, the previous state of the table is simply gone — there's no way to see what it looked like before yesterday's job ran, or to recover from a bad write.

Delta Lake, Iceberg, and Hudi are table-format layers on top of Parquet that solve exactly these four problems, one-to-one: ACID transactions (safe concurrent writers), schema evolution (add a column without breaking existing readers), compaction commands (e.g. Delta's OPTIMIZE) that merge small files into larger, more efficient ones, and time travel (query the table as it looked yesterday, last week, or last month, or roll back to it).

07. Step 5 — Data quality & observability

Everything so far assumes the data flowing through is correct. In production it often isn't: sources change without warning, fields go null, row counts silently drop. Without explicit checks, a pipeline can finish "successfully" while quietly writing bad data — and stakeholders make decisions on it before anyone notices.

The 5 data quality dimensions

DimensionQuestionExample failure
CompletenessIs all the data there?Orders table normally gets 10M rows/day, today only 7.5M land; or a customer email column is 40% null.
AccuracyIs the data correct?An order amount of -500; a delivery timestamp earlier than the order timestamp.
ConsistencyDoes it agree across systems?Orders table shows $2M March revenue, payments table shows $1.8M.
FreshnessIs it current enough?SLA promises a daily refresh; the dashboard actually last updated 3 days ago.
UniquenessAny duplicates?A retried job after a timeout writes the same batch twice, inflating every downstream metric.

Map every check to a dimension

Don't say "we'll add some data quality checks." Say "we'll add a completeness check on daily row counts, an accuracy check on non-negative order amounts, and a uniqueness check on (order_id, event_type)." Naming the dimension turns a vague gesture into a concrete, reviewable design.

Data contracts — shifting quality left

The five dimensions above catch problems after bad data has already entered the system. A data contract catches structural problems at the point of ingestion instead: a formal agreement between a producer (e.g. a microservice team) and a consumer (your pipeline) that pins down schema, types, required fields, and allowed values before data ever lands.

yaml — a minimal data contract
contract: orders.order_placed.v1
owner: checkout-service-team
fields:
  order_id:   { type: string,  required: true }
  amount:     { type: decimal, required: true, min: 0 }
  placed_at:  { type: timestamp, format: iso8601, required: true }
breaking_change_policy: versioned migration, 2 weeks notice, no silent field removal

If the contract is violated, the write is flagged before the record enters bronze — not three layers downstream when a dashboard number looks wrong. Enforcement mechanisms in practice: a schema registry for streaming (e.g. Avro + Confluent Schema Registry) and schema-validation frameworks (e.g. Great Expectations-style expectation suites) for batch. The underlying principle is "shift-left" — prevent bad data at the source, don't just detect it downstream.

Pipeline-level observability

Distinct from data quality: this is about whether the process producing the data is healthy, not whether the data itself looks right. Is the DAG running on schedule? Did it fail? Did it take 3 hours when it normally takes 40 minutes (a signal worth investigating — a volume spike, or an undersized cluster)? Orchestrators (Airflow, Prefect, Dagster) expose this natively; external tools like Datadog or CloudWatch layer on top for alerting.

08. Step 6 — Scalability, backfills & DataOps

The closing set of questions: what happens when things break — because they will; how do you rerun a pipeline without duplicating data; how do you reprocess years of history when the business logic changes; how do you ship changes without taking production down?

Idempotency — the golden rule

Running the pipeline twice — or five times — on the same input must produce the exact same output: no duplicates, nothing missing. Simple to state, easy to violate by accident.

sql — idempotent incremental load
MERGE INTO fact_orders AS target
USING incoming_batch AS source
ON target.order_id = source.order_id
WHEN MATCHED THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *

Backfill strategies

A backfill reruns the pipeline over data that's already been processed — and this happens often, not as a rare emergency. Two common triggers:

  • A bug fix. Delivery duration was computed as delivered_at - created_at instead of delivered_at - dispatched_at for three months, quietly corrupting every downstream report. After the code fix, three months of history need reprocessing.
  • A changed business definition. "Total orders" used to count every order placed; the business now wants it to exclude cancellations within 24 hours. New logic applies going forward, but every existing report overstates the old numbers until they're recomputed.

If the pipeline is idempotent and built on Delta Lake or Iceberg, the clean approach is a direct partition overwrite for the affected date range — because these table formats support ACID transactions, the overwrite is atomic: either the whole replace succeeds, or nothing changes and it rolls back. Downstream consumers never see a half-rewritten table.

sql — atomic backfill
INSERT OVERWRITE TABLE fact_orders
PARTITION (event_date)
SELECT * FROM silver.orders_recomputed
WHERE event_date BETWEEN '2026-04-01' AND '2026-06-30'
-- atomic: either all 91 partitions swap, or none do

Schema evolution — flexible at bronze, strict at silver/gold

Upstream sources will change eventually — not daily, but inevitably: a payments team adds a discount_code column, a mobile team renames user_location to device_location, an API silently starts sending an integer field as a string. None of these look dramatic in isolation, but any one can break a Spark job or quietly corrupt a downstream number.

The design principle: absorb changes at the bottom, enforce consistency at the top. Bronze is the pipeline's safety net — Delta Lake's mergeSchema option lets it automatically accommodate new fields without failing. Silver and gold stay strict because analysts, dashboards, and ML pipelines all depend on stable types — an order amount that's always decimal, a user ID that's never missing.

09. Interview walkthrough — a food delivery platform

An illustrative walkthrough in the style of a real interview question, applying all six steps end-to-end. The prompt:

❯ The prompt

"Design a real-time analytics pipeline for a food delivery platform, like Uber Eats, Zomato, or Swiggy."

Step 1 — clarifying questions

  • Who are the users? Two very different groups: restaurant partners needing a live operational dashboard (today's earnings, orders delivered, pending orders), and an executive/finance team needing daily aggregated business reports.
  • Latency SLA? Restaurant dashboard: under 2 minutes end-to-end. Executive dashboard: next-day batch is fine.
  • Volume? ~5 million orders/day; peak during dinner rush (7–9pm) is ~500K orders/hour.
  • What events are tracked? Six: order placed → restaurant confirmed → driver assigned → driver picked up → driver delivered → order cancelled.
  • Retention? 2 years for analytics, 90 days for the real-time serving layer.

Step 1 (continued) — back-of-the-envelope math

back-of-envelope
# throughput
events/day   = 5,000,000 orders × 6 events = 30,000,000
avg / sec    = 30M / 86,400s               ≈ 350 events/sec
peak / sec   = (500K/hr × 6) / 3,600s        ≈ 833 events/sec
# → very manageable for Kafka: a modest 3-broker cluster, 6-12 partitions

# storage
event size   ≈ 1 KB  (order/restaurant/driver/customer ids, coords, timestamp)
daily volume = 30M × 1KB                  ≈ 30 GB/day
2yr raw      = 30GB × 365 × 2               ≈ ~22 TB
2yr, Parquet compressed (2-5x)              ≈ 4-11 TB # manageable on S3/ADLS

# serving layer (Redis) for the restaurant dashboard
per restaurant ≈ 1 KB (active orders, avg delivery time, completed count)
50,000 active restaurants                   ≈ ~50 MB total — one Redis instance, easily

Conclusion stated out loud to the interviewer: this is a moderate-scale problem, not a massive one — which matters, because it justifies a lean design (a small Kafka cluster, a single Redis instance) instead of over-engineering for load that isn't there.

Step 2 — architecture: Lambda-style, split by consumer

Partitioning the Kafka topic by restaurant_id keeps all of one restaurant's events in order on one partition — important since the dashboard's running totals are per-restaurant. Worth flagging to the interviewer: they may drill into Kafka internals next (delivery guarantees, retention and log compaction, how late-arriving events are handled) or into Spark internals (query planning, shuffle behavior, join strategy) depending on what they want to probe — these are advanced follow-ups beyond what this guide covers in depth, so steer toward whichever tool you personally know most deeply.

Step 3 — data model

  • Bronze: raw order-event JSON straight from Kafka — immutable, unprocessed, source of truth.
  • Silver: fact_orders (order, restaurant, driver, customer ids; placed/delivered/cancelled timestamps; amount; delivery duration — partitioned or liquid-clustered by order date), dim_restaurants, dim_drivers. (Additional tables like fact_payments or dim_food_items would exist in a full build but are out of scope for the walkthrough.)
  • Gold: a live aggregate table feeding the restaurant dashboard (built by the streaming path) and a daily-aggregated restaurant metrics table feeding the executive dashboard (built by the batch path).

Step 4 — storage & file formats

Raw ingestion lands in Parquet; everything from bronze onward moves to Delta, for the ACID guarantees and schema evolution §06 covered — this is what makes the Step-6 backfill and schema-evolution answers below actually work in practice, not just in theory.

Step 5 — data quality, layer by layer

LayerChecks
IngestionData contract / schema registry rejects malformed events before bronze.
Bronzeevent_type must be one of the 6 valid values; duplicate (order_id, event_type) pairs are logged, not silently dropped; customer_id must not be null.
SilverNull checks on key attributes; referential integrity (every restaurant_id in fact_orders must exist in dim_restaurants); business rules (delivery duration between 1–180 minutes, anything past 3 hours flagged); time ordering (picked_up_at after order_placed_at, delivered_at after picked_up_at); volume anomaly checks against a 7-day rolling average.
GoldCancellation rate must fall within 0–100%; average delivery minutes must be positive and within a business-defined bound.

Step 6 — closing the loop

Three things worth raising proactively before the interviewer asks: a backfill strategy for cleanly rerunning gold-layer aggregates over a date range when logic changes; idempotency as non-negotiable — the pipeline must produce the same output no matter how many times it's re-run; and schema evolution — bronze stays schema-on-read, silver/gold enforce a strict schema with Delta's merge-schema option handling new incoming fields safely.

+ The actual takeaway

Candidates rarely fail this round for not knowing Kafka or Spark deeply enough. They fail for lacking structure — jumping straight to tool names, skipping requirements gathering, designing only for the happy path. What separates a strong answer is slowing down, asking clarifying questions first, and making your reasoning about trade-offs visible as you go — that visibility is the actual signal being graded.

＊ Glossary

Batch processing
Processing data in scheduled chunks (hourly/daily) rather than continuously; optimizes for cost/throughput over latency.
Streaming
Processing events continuously as they arrive; optimizes for latency, at higher operational complexity.
Lambda architecture
Two parallel paths — a batch layer and a speed layer — over the same data, merged at a serving layer.
Kappa architecture
A single streaming pipeline for both real-time and historical processing; history is handled by replaying events.
Medallion architecture
Bronze (raw) → Silver (cleaned/structured) → Gold (aggregated) layering of a lakehouse.
Star schema
A normalized model with separate fact and dimension tables joined at query time.
One Big Table (OBT)
A denormalized model with dimension attributes pre-joined onto fact rows, trading update cost for read simplicity.
Slowly changing dimension (SCD)
A pattern for handling dimension attribute changes over time — Type 1 overwrites, Type 2 versions with validity dates.
Partitioning
Physically splitting a table's files by a column (typically date) so queries scan only relevant partitions.
Bucketing
Hashing a high-cardinality key into a fixed number of files to reduce shuffle on large joins.
Liquid clustering
A Delta Lake feature that dynamically reclusters data by chosen columns without requiring a full table rewrite.
Parquet
A columnar file format optimized for analytical, read-heavy queries via column pruning and file skipping.
Avro
A row-based file format optimized for write-heavy, streaming workloads; commonly paired with a schema registry.
Delta Lake / Iceberg / Hudi
Table-format layers over Parquet adding ACID transactions, time travel, schema evolution, and file compaction.
Data contract
An upfront agreement between a producer and consumer on schema, types, and change policy — enforced at ingestion.
Idempotency
The property that re-running a pipeline on the same input yields the same output, with no duplicates.
Backfill
Reprocessing already-processed historical data after a bug fix or a change in business logic.
Schema evolution
The practice of handling upstream schema changes gracefully — permissive at bronze, strict at silver/gold.
CDC (Change Data Capture)
Capturing row-level inserts/updates/deletes from a source database as a stream of events.
SLA vs. SLO
An SLA (service-level agreement) is the external commitment made to a stakeholder (e.g. "the dashboard is refreshed by 6am"); an SLO (service-level objective) is the internal target a team holds itself to in order to reliably meet that SLA (e.g. "the batch job finishes by 5:30am").
Shuffle
The redistribution of data across a cluster's nodes/partitions that a distributed join or aggregation requires — an expensive operation that partitioning and bucketing exist partly to reduce.
OLTP
Online transaction processing — a database optimized for many small, fast reads/writes (e.g. the operational database behind an app), as opposed to OLAP systems optimized for large analytical scans.

That's the full framework: requirements, pipeline design, data modeling, storage, quality, and scale — plus a worked walkthrough tying all six together. The through-line to carry into any interview: ask before you architect, justify every choice with a number or a trade-off, and design for the pipeline breaking, not just for it working.
