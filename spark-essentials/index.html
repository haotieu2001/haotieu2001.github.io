A job that should take a minute takes an hour. The same query is fast on a small file and painfully slow on a big one. You add more machines and it gets slower. That is the experience that makes Spark feel like black magic — and it is almost always one of a handful of root causes: a shuffle you didn't expect, a partition that swallowed all the data, a join that exploded, or memory configured wrong.

This guide builds the mental model that dissolves the magic. We start from why Spark was invented (what was so bad about MapReduce), walk through the engine layer by layer, trace exactly what happens when you submit a SQL query or a PySpark command, then end with the practical tuning playbook you reach for in real projects. Read the short version first; everything after it fills in the detail.

❯ Short version (read this first)

Spark is a distributed data engine. It splits your data into partitions, sends each piece to a worker (executor), and a driver coordinates them. The two ideas that make it fast:

• In-memory — it keeps intermediate results in RAM instead of writing every step to disk like MapReduce did.

• DAG (directed acyclic graph) — it builds a full plan of your whole pipeline before running, so it can fuse steps, skip work, and retry only what failed.

• Lazy — transformations (filter, join…) only build the plan; nothing runs until an action (count, write, show) triggers execution.

• Stages are split by shuffles — a stage is a stretch of work that needs no data movement; a shuffle (re-partitioning data across the network) is the expensive wall between stages.

One-line tuning instinct: most slow Spark jobs are slow because of shuffles, data skew, or too-few/too-many partitions. The Spark UI tells you which. Skip to tuning ↓

01. Why Spark exists — the MapReduce pain

Why a 2014 engine replaced a 2004 idea

In the early 2000s, Google published MapReduce, and Hadoop made it open-source. It solved a real problem: process data too big for one machine by splitting it across many. Each machine runs a Map on its slice, the system shuffles and sorts the output by key, then each machine runs a Reduce. It was a revolution — and it was painfully slow for anything iterative.

The killer was the disk. Between every step, MapReduce wrote the full intermediate result to disk, then read it back for the next step. A two-stage job was two rounds of disk writes. A multi-step pipeline (read → filter → join → group → write) was a chain of separate MapReduce jobs, each reading and writing the full dataset from HDFS in between. No step could see the next, so there was no way to fuse them.

MapReduce's two structural limits

  • Rigid two-step model. Every job is exactly Map → Shuffle/Sort → Reduce. Want filter → join → group → filter? That is four separate MapReduce jobs, each a full read-from-disk and write-to-disk of the entire dataset. You cannot express a complex pipeline as one optimized plan.
  • Disk-bound. Intermediate data always landed on disk. For iterative work — machine learning algorithms that re-read the same data dozens of times, or interactive analysis where you re-query the same data — this was brutal. Each iteration re-read everything from HDFS.

So the stage was set. The world needed an engine that (a) could hold data in memory between steps, and (b) could plan a whole multi-step pipeline as one graph and optimize it end-to-end. That engine was Spark.

02. How Spark fixed it — in-memory + the DAG

Two ideas, and why they change everything

Spark's 2010 paper (Matei Zaharia, UC Berkeley) had a deceptively simple claim: keep intermediate data in memory, and represent the whole computation as a DAG — a directed acyclic graph of operations. The result was order-of-magnitude faster on iterative and multi-step workloads, with one engine instead of a chain of jobs.

The two ideas, precisely

In-memory computing
Intermediate results live in RAM (as cached partitions) instead of being forced to disk after each step. For iterative algorithms (ML, graph), the same dataset is re-read from memory across iterations — 10–100× faster than re-reading from HDFS each time. Caveat: "in-memory" does not mean "never touches disk." Spark still spills to disk when data exceeds memory; it just isn't forced to between every step.
DAG execution (lineage)
Your whole pipeline is one directed acyclic graph of transformations. Spark knows the lineage (how each partition was derived). Two payoffs: (1) optimization — it sees the full plan, so it can pipeline narrow steps, reorder, push predicates down, and prune; (2) fault tolerance — if a node dies, Spark rebuilds the lost partitions from lineage instead of replicating everything.

That second point is subtle and important. MapReduce got fault tolerance by replicating data on HDFS (3×). Spark gets it by remembering how data was made (lineage). Lose a partition? Re-run the transformations that produced it. This is why Spark can afford to keep less on disk — the recipe to rebuild anything is in the DAG.

Myth check: "Spark is always faster than MapReduce."

− Not for a single, simple, one-pass job. A plain Map → Reduce with no iteration can be similar, because Spark has overhead (planning, JVM, serialization). Spark wins decisively on multi-step and iterative workloads — which is almost everything real pipelines do.

03. The mental model — transformations, actions, jobs, stages, tasks

Five words that make every Spark job make sense

If you only remember one section, remember this one. Five nested concepts explain what happens when you run anything in Spark:

1. Transformation — builds the plan, runs nothing
Operations like select, filter, join, groupBy, withColumn. They take a DataFrame and return a new DataFrame that is just "the old one plus one more node in the plan." No data is read, no compute happens. Example: df.filter("age > 18") returns instantly even on a petabyte — it only recorded the intent.
2. Action — the trigger that starts a job
Operations like count(), collect(), show(), write.save(), take(n). The moment you call one, Spark takes the plan built by all prior transformations, optimizes it, and actually runs it. One action = one job. Example: df.count() is the moment the filter, the scan, everything finally executes.
3. Job — the whole computation for one action
When an action fires, Spark submits a job to run the optimized plan for that action. If your code calls three actions, you get three jobs (and unless you cache, the source data is re-read for each).
4. Stage — a stretch of work with no shuffle in the middle
Spark cuts a job into stages at every shuffle boundary (any operation that must move data between partitions — a groupBy, a join on non-co-partitioned data, a repartition). Within a stage, tasks run as a single in-memory pipeline (a "map stage" / narrow work). Stages run in sequence, because stage N+1 needs stage N's shuffled output.
5. Task — the smallest unit of work, one per partition per stage
A stage is parallelized into tasks: one task per partition of that stage's input. Each task runs on one core of one executor. If a stage has 200 partitions, it launches 200 tasks; with 50 cores in the cluster they run in ~4 waves. This is why partition count matters: too few = poor parallelism; too many = scheduling overhead.

The lazy rule, stated plainly:

Transformations are lazy (free — they only describe work). Actions are eager (they pay for everything). A common bug: a notebook cell with only transformations "does nothing" and looks instant — the work all happens in the next cell that calls an action, which then looks mysteriously slow. The slowness was always there; the action just billed it.

04. Narrow vs wide transformations & the shuffle

The single most important performance distinction in Spark

Every transformation is either narrow or wide. This one distinction predicts most Spark performance:

Narrow transformation
Each output partition depends on exactly one input partition. filter, select, map, withColumn, union. No data leaves its partition, so these pipeline together inside one stage — a single pass over the data, no network, no disk.
Wide transformation (a shuffle dependency)
Each output partition depends on many input partitions — data with the same key must end up on the same partition. groupBy, join (when not co-partitioned), distinct, repartition, orderBy (global sort). This forces a shuffle: each executor writes its output to local shuffle files sorted by key, then reducers pull the pieces they need over the network.

What a shuffle actually costs

A shuffle is not "free data movement." Concretely it is: (1) serialize + write every record to local disk on the map side (a shuffle write), (2) transfer those bytes across the network to other executors, (3) deserialize + read them on the reduce side (a shuffle read). It touches disk and network, the two slowest things in a cluster. A job with 3 shuffles does this 3 times. This is why "reduce the number of shuffles" is the first lever of Spark tuning.

Read this if you read nothing else about performance:

• A "stage" in the UI is literally the gap between two shuffles. Count the stages; each boundary is a shuffle you pay for.

• join is wide unless the two sides are already co-partitioned (same partitioner on the join key) or one side is broadcast (small enough to send to every executor). Both turn a shuffle-join into a narrow one.

• groupBy().agg() always shuffles. But map-side (partial) aggregation can dramatically shrink what gets shuffled by pre-combining per partition before the network step — in DataFrames, groupBy().agg() already does this for you via Catalyst.

05. Spark architecture — every layer in detail

Driver, cluster manager, executors, tasks, blocks — and what each one actually does

A Spark application is one driver plus a set of executors, placed on a cluster by a cluster manager. Everything else — jobs, stages, tasks, blocks, shuffle services — is a consequence of those three. Here is the whole picture, then each layer in turn.

Layer by layer

Driver the brain — one process

Runs your program and coordinates the whole application. There is exactly one driver per Spark application.

SparkSession / SparkContext — the entry point. Holds your configuration, the connection to the cluster manager, and the master copy of the block-manager registry. (In modern Spark you use SparkSession; SparkContext is the older lower-level object inside it.)

Spark UI — served from the driver (port 4040 by default). This is where jobs/stages/tasks, shuffle bytes, memory, and the query plan live. If the driver dies, the application dies.

Two schedulers inside the driver:

• DAGScheduler — takes the optimized logical/physical plan and cuts it into stages at shuffle boundaries, building a DAG of stages. It also figures out data locality (prefer to run a task where its data already is).

• TaskScheduler — takes each stage's tasks and actually sends them to executors via the SchedulerBackend (the RPC layer). It handles task assignment, retries (a few attempts per task, with blacklist tracking of bad nodes), and locality waits.

BlockManagerMaster — tracks which executor holds which block (cached RDD/DataFrame partition or broadcast piece), so other tasks can fetch cached data instead of recomputing it (blocks are unpacked in the Task & block layer below).

Cluster manager the resource broker

A separate system that owns the machines. Spark does not manage nodes itself; it asks the cluster manager for containers.

YARN (Hadoop) — the classic enterprise choice. Spark runs as YARN applications; YARN negotiates containers for the AM (driver) and executors.

Kubernetes — Spark submits a driver pod and executor pods; the dominant choice for cloud-native stacks.

Standalone — Spark's own built-in master/worker manager. Simplest; fine for small clusters or testing.

Local[n] — not really a cluster: one JVM with n threads. What you use in a notebook or unit test.

The key point: the cluster manager only allocates and tracks processes. It has no idea what a task or a DataFrame is. Spark does all the data work; the cluster manager just hands out boxes.

Executors the muscle — one JVM each

Each executor is a long-lived JVM process running on a cluster node, allocated for the lifetime of the application.

Task slots — an executor runs as many tasks in parallel as it has cores (spark.executor.cores). 5 cores = 5 concurrent tasks in that JVM.

BlockManager — each executor's local store: in-memory cache for DataFrame/RDD partitions and broadcast variables, plus on-disk shuffle spill files. It reports its blocks up to the driver's BlockManagerMaster.

Shuffle files — when a task in a "map stage" (a stage that produces shuffle output for the next stage) writes its shuffle data, it writes sorted buckets to the executor's local disk; the next stage's tasks fetch from these via the shuffle service.

Executor memory — one heap (spark.executor.memory) divided internally into reserved, user, execution, and storage regions (see §09). This is the single most-misconfigured knob in Spark.

Executors are allocated once and reused across all jobs in the application — that reuse is a big part of why Spark beats MapReduce (no per-job JVM startup).

Task & block the unit of work & the unit of data

The two atomic things Spark moves around.

Task — one unit of computation applied to one partition, on one core of one executor. A stage with 200 partitions launches 200 tasks. If a task is slow, you see it as a straggler in the UI (often = data skew).

Block — a chunk of data identified by an RDD/DataFrame ID + partition index, or a broadcast variable. Blocks live in an executor's BlockManager (memory → disk spill). Cache = "keep these blocks"; broadcast = "copy this block to every executor."

Shuffle service the data mover

How reduce-side tasks fetch map-side output.

By default, reduce tasks pull shuffle files directly from the producing executor's BlockManager. An external shuffle service (a daemon running on each node, e.g. YARN's aux-service) serves shuffle files outside the executor JVM, so reduce tasks can fetch even after the map executor has exited — this decouples map and reduce lifecycles and reduces load on executor JVMs. It also helps when executors get killed dynamically.

Mental shortcut — who does what:

• Driver = thinks (plan, split into stages, assign tasks, collect results, host the UI).

• Cluster manager = provides boxes (allocates JVMs on nodes; knows nothing about your data).

• Executors = do the lifting (run tasks, cache blocks, serve & fetch shuffle data).

06. Deployment modes — cluster vs client

Same engine; the only question is which machine runs the driver

"Deployment mode" answers one question: where does the driver JVM run? Everything else is identical. The choice matters because the driver is the single point of coordination and the host of the Spark UI.

Cluster mode
The driver runs inside the cluster, as a container allocated by the cluster manager. You submit, then can close your laptop — the job keeps running. Use for: production batch jobs, scheduled pipelines, anything long-running. Trade-off: the Spark UI is on a cluster node (reach it via the resource manager's proxy / tracking URL), and collect() brings data to the driver in the cluster, not to your laptop.
Client mode
The driver runs on the machine you launched from (your laptop, a gateway, a notebook server). Executors still run in the cluster. Use for: interactive analysis, notebooks (Databricks, Jupyter), ad-hoc REPL work where you want the UI on localhost and direct collect() to your process. Trade-off: if your machine dies or disconnects, the whole application dies.

Note a third, rarer option: local mode (--master local[*]) runs driver and executors in one JVM — great for tests and small data, not for real clusters.

07. How Spark runs your code — the plan path

From a SQL string or PySpark call all the way to tasks on executors

This is the "black magic" question, made concrete. When you run a SQL query or a PySpark command, Spark walks a fixed pipeline of plans. Each plan is a tree of operators; each phase refines it. Understanding this pipeline is how you read the explain() output and find why a job is slow.

Each phase, with a running example

Suppose you run this PySpark:

from pyspark.sql import SparkSession
import pyspark.sql.functions as F

spark = SparkSession.builder.getOrCreate()
orders = spark.read.parquet("s3://bucket/orders")        # transformation (lazy)
cust   = spark.read.parquet("s3://bucket/customers")    # transformation (lazy)

joined = (orders
    .filter(F.col("amount") > 100)            # filter BEFORE join (good)
    .join(cust, "customer_id")
    .groupBy("country")
    .agg(F.sum("amount").alias("total")))

joined.write.mode("overwrite").parquet("s3://bucket/out")  # ACTION → triggers the pipeline

The four read/filter/join/groupBy lines do nothing. The .write(...) on the last line is the action that fires the whole pipeline below.

1 · Unresolved logical plan — "what you wrote, literally"
A tree of relational operators (Filter → Join → Aggregate → Project → Relation) with column names as raw unresolved attributes. Spark does not yet know whether amount exists or what type it is. explain() shows this as UnresolvedRelation / 'amount.
2 · Analyzed (resolved) logical plan — bind to the catalog
The Analyzer uses the session catalog (table & column metadata) to resolve names to real columns and types. Now amount is known as double, customer_id as long. If you typed a column that doesn't exist, you fail here — before any data is read. This is why a bad column errors fast.
3 · Optimized logical plan — Catalyst rewrites it
A set of rules rewrites the tree for efficiency. The big ones you'll recognize: predicate pushdown (move filter below the join so it runs first, on less data), column pruning (drop columns nobody selects), constant folding, projection collapsing. This is why writing filter "after" a join in code can still run "before" it — the optimizer moves it.
4 · Physical plan — pick the actual algorithms
Spark generates several candidate physical plans and picks the cheapest by cost. The most visible decision is the join strategy: BroadcastHashJoin if one side is small (≤ spark.sql.autoBroadcastJoinThreshold, default 10 MB) — no shuffle; otherwise SortMergeJoin — both sides sorted & shuffled on the key. Also decides partitioning, and applies Tungsten whole-stage codegen (fuses a whole stage into one generated Java function to avoid per-row virtual calls).
5 · RDDs → DAGScheduler → tasks
The physical plan maps to RDDs. The DAGScheduler cuts it into stages at shuffle boundaries (here: the join may shuffle, the groupBy shuffles). The TaskScheduler launches one task per partition per stage onto executor cores. Results flow back to the driver; for an action like write, the "result" is just success + metrics.

How to actually see this:

• df.explain(True) prints all four plans (parsed, analyzed, optimized, physical) — the single best debugging tool in Spark SQL.

• Look for PushedFilters: [amount > 100] near a FileScan — that means the filter was pushed into the Parquet reader (skips whole row groups). If your filter is not pushed down, you're reading more than necessary.

• Look for BroadcastHashJoin vs SortMergeJoin. A SortMergeJoin on a tiny dimension table is a missed broadcast — fix it with a broadcast hint.

08. Using Spark SQL & DataFrames (with PySpark)

The high-level APIs you should actually use — and how they map to each other

Modern Spark has three high-level APIs. You should use DataFrame / Spark SQL for almost everything; they share the same Catalyst optimizer. The low-level RDD API is for cases where you truly need raw control (and even then, Datasets usually cover it).

APIWhat it isUse when
Spark SQLWrite SQL strings; Spark parses them through the same Catalyst optimizer as DataFramesYou like SQL, or handing queries to analysts / BI tools
DataFrameTables with named columns; programmatic chainable methods (the de facto default in PySpark/Scala)Most ETL, pipelines, anything programmatic
Dataset[T]Typed DataFrame (compile-time type safety); Scala/Java only — no real equivalent in PySparkScala apps wanting types + encoder perf
RDDLow-level distributed collection of arbitrary objects; no Catalyst, you control everythingLegacy code, custom partitioners, fine-grained control — rarely needed now

Spark SQL and DataFrame are the same engine

The crucial fact: a SQL string and the equivalent DataFrame code compile to the same optimized plan. Pick whichever reads better. You can even mix them — register a DataFrame as a temp view and query it in SQL.

# --- DataFrame API ---
df = (spark.read.parquet("s3://b/orders")
        .filter(F.col("amount") > 100)
        .groupBy("country")
        .agg(F.sum("amount").alias("total")))

# --- identical plan, written as SQL ---
df = spark.sql("""
    SELECT country, SUM(amount) AS total
    FROM parquet.`s3://b/orders`
    WHERE amount > 100
    GROUP BY country
""")

# --- mix them: register a DF, query in SQL ---
orders.createOrReplaceTempView("orders")
top = spark.sql("SELECT country, total FROM orders WHERE total > 1000 ORDER BY total DESC")

The day-to-day vocabulary

# read / write
df = spark.read.parquet(path)              # also: csv, json, orc, jdbc, delta
df.write.mode("overwrite").parquet(out)   # append | overwrite | ignore | error

# shape & schema
df.printSchema(); df.columns; df.count(); df.show(20, truncate=False)

# select / transform
df.select("country", "amount")
df.withColumn("eur", F.col("amount") * 0.93)
df.filter((F.col("amount") > 100) & (F.col("country") == "VN"))

# joins (prefer broadcast for small dims)
from pyspark.sql.functions import broadcast
joined = big.join(broadcast(small), "id", "left")

# aggregations & windows
df.groupBy("country").agg(F.sum("amount"), F.countDistinct("cust"))
w = Window.partitionBy("country").orderBy(F.col("dt").desc())
df.withColumn("rnk", F.row_number().over(w)).filter("rnk = 1")

How Spark reads a SQL query vs a PySpark command — the detail

Both end in the same pipeline (§07), but the entry differs:

  • spark.sql("...") — the string is parsed by Spark's SQL parser (ANTLR grammar) into an unresolved logical plan. Column refs are unresolved attributes. Then Analyzer → Optimizer → Physical, identical to below.
  • DataFrame methods (.filter().join()...) — each method call directly appends a node to the unresolved logical plan (no SQL parsing step). F.col("amount") > 100 builds an expression tree in the plan.
  • From the resolved plan onward, the two paths are indistinguishable. That is why df.explain() on a SQL-derived DataFrame and a code-derived one can produce byte-identical physical plans.

The practical upshot: write whichever is clearer. SQL is better for declarative set logic and handing to analysts; DataFrame methods are better when logic is conditional/programmatic (loops, config-driven columns). They interoperate freely.

Patterns that pay off in real code:

• Filter early, select only needed columns. Catalyst pushes filters, but column pruning is most effective when you don't drag unused columns through a join.

• Broadcast small dimensions. A 50 MB country lookup joined to a billion-row fact table should be broadcast() — turns a shuffle-join into a map-side join.

• In the DataFrame API, groupBy().agg() already does map-side partial aggregation (Catalyst's HashAggregateExec) to shrink what gets shuffled. Avoid dropping to .rdd.reduceByKey() just to pre-combine — you'd lose the optimizer. Tune shuffle output size with spark.sql.adaptive.advisoryPartitionSizeInBytes.

• Cache only what you reuse ≥ 2×. Caching is expensive and eats memory; uncached reused data recomputes from source every action.

• Use Delta/Parquet with partitioning + predicate pushdown so filters skip whole files/partitions.

09. Executor memory deep dive

On-heap, off-heap, unified memory — and the sizing math that fixes 80% of OOMs

Most Spark memory pain is a config mistake, not a data problem. Here is exactly how one executor's memory is carved up, then the sizing example from the video.

The regions, defined

Reserved memory
A fixed ~300 MB that Spark holds back for its own internals and the JVM. Rule of thumb: usable heap ≈ M − 0.3 GB. On a tiny executor (e.g. 1 GB) this eats a lot proportionally — one reason not to use very small executors.
User memory
~40% of (M − Reserved). Space for your objects that Spark doesn't track: UDF internals, accumulators, large Python-side data (in PySpark the Python worker is a separate process with its own memory). OOMs here usually mean a UDF or collect() holding too much on the driver/executor.
Unified (Execution + Storage) memory
~60% of (M − Reserved), introduced in Spark 1.6. Execution = transient memory for shuffles, joins, sorts, aggregations (spills to disk when full). Storage = cached blocks & broadcast variables (evicted/dropped when full, not spilled). The two share the pool: execution can borrow from storage and vice-versa, with a soft boundary controlled by spark.memory.storageFraction (default 0.5).
Off-heap memory
A separate, non-JVM-heap allocation (spark.memory.offHeap.enabled=true, spark.memory.offHeap.size) used by Tungsten for its own binary rows. Bypasses GC pressure — useful for heavy shuffle/sort workloads. Off by default; enable when you see GC stalls.

The sizing example (from the video)

You have a node with ~64 GB RAM and 16 cores. A naive config might ask for one 64 GB / 16-core executor. That is usually wrong — one giant JVM has terrible GC pauses and no parallelism gain beyond ~5 cores per executor. The standard reasoning:

# Per-node: 64 GB, 16 cores. Reserve ~1 core + ~1 GB for OS / YARN NM / shuffle service.
# Pick ~5 cores per executor (good GC + HDFS throughput sweet spot).
# → 3 executors × 5 cores = 15 cores used; leaves headroom.

spark.executor.cores   = 5
spark.executor.memory  = 19g        # (64 − overhead) / 3 executors ≈ 19–20 GB each
spark.executor.instances = 3
spark.memory.fraction        = 0.6   # 60% of usable heap → unified
spark.memory.storageFraction = 0.5   # half of unified reserved for storage
spark.sql.shuffle.partitions  = 200 # raise for big shuffles (≈ 2× cores across cluster)

The fractions in action on a 19 GB executor:

  • Reserved ≈ 0.3 GB → usable ≈ 18.7 GB.
  • Unified ≈ 60% × 18.7 ≈ 11.2 GB (Execution ⇄ Storage).
  • User ≈ 40% × 18.7 ≈ 7.5 GB.
  • Within unified, Storage is protected up to 50% (≈ 5.6 GB) before it must give way to Execution.

Common memory mistakes:

− One fat executor per node (e.g. 64 GB / 16 core). GC pauses dominate. Prefer ~5 cores × several executors per node.

− collect() on a big DataFrame — pulls everything to the driver; OOMs the driver, not the executors. Use take(), toPandas() on a sample, or write to storage.

− Too few shuffle partitions → each task gets a huge partition → execution memory spills constantly. Raise spark.sql.shuffle.partitions or enable AQE.

− Caching everything → Storage fills, evictions thrash. Cache only reused DataFrames, and unpersist() when done.

10. Monitoring & optimizing a real cluster

A repeatable method: read the UI → find the bottleneck → apply the right lever

Tuning is not a checklist of flags; it's a diagnostic loop. The Spark UI tells you what is slow; this section maps each symptom to its lever. Learn the UI first — it's the instrument panel.

The Spark UI — what each tab tells you

UI tabAnswersWhat to look for
JobsIs the whole job on track? Which action fired it?Many jobs = many actions (maybe re-reading source). One job = one action.
StagesWhere is time spent? Is there a shuffle wall?Stage duration, shuffle read/write bytes, task skew (one task taking 100× the median).
Tasks (inside a stage)Skew, spill, stragglers, localityMin/median/max task time; Shuffle spill (memory/disk); GC time; locality level.
StorageWhat's cached, how big, hit rateFraction cached in memory vs disk; evictions; is the cache even being used?
ExecutorsResource use, dead executors, GCPer-executor task count, GC time, shuffle bytes, max/used heap, failed tasks.
SQLThe query plan & per-operator metricsexplain tree with real row counts & time per node — the fastest way to find a bad join.

The levers — symptom → fix

Symptom: one straggler task (skew)
Cause: a hot key — e.g. one customer_id has millions of rows, so its partition is huge. Fix: salt the key (add a random 0–N suffix to break the hot key into N partitions, join, then strip the salt); enable AQE skew join handling (spark.sql.adaptive.skewJoin.enabled=true); or pre-aggregate the hot key.
Symptom: huge shuffle read/write, stages slow at boundaries
Cause: shuffling more data than necessary. Fix: filter & project before the shuffle; broadcast small sides; use map-side (partial) aggregation; raise spark.sql.shuffle.partitions so each task's partition is smaller (less spill); enable AQE partition coalescing.
Symptom: lots of small tasks, scheduling overhead, slow start
Cause: too many tiny partitions. Fix: coalesce(N) to merge (no shuffle) or re-tune spark.sql.shuffle.partitions downward; enable AQE to auto-coalesce small partitions post-shuffle.
Symptom: high GC time, executor pauses
Cause: oversized executor heaps or many objects. Fix: smaller executors (~5 cores); enable off-heap; avoid Python UDFs that materialize big objects (use native SQL functions); consider spark.sql.execution.arrow.pyspark.enabled=true for Pandas conversion.
Symptom: re-reading source data across actions
Cause: a DataFrame used by multiple actions with no cache. Fix: df.cache() (or persist(MEMORY_AND_DISK)) before the first action; unpersist() after the last. Cache only if reused ≥ 2×.
Symptom: SortMergeJoin on a tiny table
Cause: table just above the 10 MB broadcast threshold, or stats missing. Fix: broadcast(small) hint; raise spark.sql.autoBroadcastJoinThreshold; run ANALYZE TABLE so the planner knows the real size.

Adaptive Query Execution (AQE) — the modern auto-tuner

AQE (Spark 3.0+, on by default since 3.2) re-optimizes the plan at runtime, after shuffle stages produce real statistics. It fixes three things you used to tune by hand:

  • Coalescing partitions — merges tiny post-shuffle partitions into right-sized ones (kills the "200 partitions but 5 have data" problem).
  • Switching join strategies — discovers at runtime that one side is small enough to broadcast, and converts a SortMergeJoin to BroadcastHashJoin.
  • Skew join splitting — detects a skewed partition and splits it into multiple tasks so no single task is a straggler.
spark.sql.adaptive.enabled            = true   # master switch (on by default in 3.2+)
spark.sql.adaptive.coalescePartitions.enabled              = true
spark.sql.adaptive.skewJoin.enabled                         = true
spark.sql.adaptive.autoBroadcastJoinThreshold              = 100MB
spark.sql.adaptive.advisoryPartitionSizeInBytes            = 64MB

Caching & persistence — when and how

cache() = persist(MEMORY_AND_DISK) with the default storage level. Caching stores block copies in the Storage region. Rules:

  • Cache a DataFrame only if it's used by ≥ 2 actions. One action + cache = pure overhead (you materialize once for the cache, then once to use it).
  • In PySpark, prefer df.persist(StorageLevel.MEMORY_AND_DISK) for large data; use MEMORY_ONLY only for small, hot data (it drops to recompute, not spill, if full).
  • Always unpersist() when done — the cluster has finite Storage memory, and stale caches cause confusing OOMs.

A real-cluster checklist (in priority order)

  1. Read the SQL/Stages tab first. Don't tune blind. Find the slow operator / stage / straggler before touching config.
  2. Right-size executors. ~5 cores each, several per node, leave OS/YARN headroom. Avoid one fat JVM.
  3. Turn on AQE. It removes a lot of manual partition tuning for free.
  4. Filter & project early; broadcast small tables. Cheap, high-impact, often the whole fix.
  5. Handle skew. Salt hot keys or rely on AQE skew splitting — the difference between a 2.5-min straggler and a 5-second stage.
  6. Tune shuffle partitions to the data size (≈ 2–4× total cores, or let AQE coalesce).
  7. Cache reused DataFrames; unpersist after.
  8. Watch GC & spill in the Executors tab; enable off-heap / smaller executors if GC dominates.
  9. Collect table stats (ANALYZE TABLE) so the optimizer picks good join strategies and skips files.
  10. Use the right file format — Parquet/Delta with partitioning + predicate pushdown, avoid many tiny files (compact).

The 80/20 of Spark performance:

1. Filter + project before shuffles and joins.

2. Broadcast small tables; salt or split skewed keys.

3. Right-size executors (~5 cores) and turn on AQE.

4. Read the UI before you change a single config flag.

11 Lazy eval — nothing runs until an action.

14 Task — one partition, one core, one executor.
15 Partition — the unit of data parallelism.

17 Wide transform — many inputs → 1 output; shuffle.
18 Shuffle — disk write + network + disk read.

23 DAGScheduler — cuts plan into stages.

25 Deployment modes — cluster vs client (where driver runs).
26 Plans — unresolved→analyzed→optimized→physical.
27 Catalyst — logical optimizer (rules).
28 Tungsten — physical: codegen + off-heap memory.
29 Unified memory — Execution ⇄ Storage pool.

∎ The takeaway

Spark exists because MapReduce forced every step through disk and could only see one job at a time. Spark keeps data in memory and plans the whole pipeline as a DAG. The mental model is five nested words: transformations build a lazy plan, an action fires a job, the job splits into stages at shuffles, each stage runs one task per partition on an executor, and a driver coordinates it all. Once you can read the plan (explain) and the UI (jobs/stages/tasks), tuning stops being guesswork: every slowness is a shuffle, a skew, a bad join, or memory — and each has a known lever.

Run explain(True) on your slowest job today. The plan will tell you exactly which of the 30 concepts is in your way.
