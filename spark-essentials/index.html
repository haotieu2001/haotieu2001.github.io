01. Why Spark Exists: MapReduce → In-Memory + DAG

Apache Spark is a distributed compute engine: it lets you process data that is far too big for one machine by splitting the work across a cluster of machines and coordinating them. It exists because the previous standard for doing that — Hadoop MapReduce — was correct but painfully slow, and slow means expensive.

Before Spark, the way you processed a terabyte of logs was to write a Hadoop MapReduce job. It worked, and it was fault-tolerant, but every step wrote its results to disk before the next step could read them. For a single pass that's tolerable; for the multi-step pipelines and iterative algorithms that real analytics and machine learning need, it was death by a thousand disk writes. Spark's founding insight (from the 2010–2012 AMPLab work at UC Berkeley) was simple: keep the working data in memory across steps, and only touch disk when you truly must. That one change made many workloads 10–100× faster.

Definition Apache Spark is an open-source, distributed, in-memory data processing engine. You write high-level code (in Python, SQL, Scala, Java, or R); Spark turns it into a graph of parallel tasks and runs them across a cluster, keeping intermediate data in RAM wherever possible.

What Spark Is Used For

  • Batch ETL / ELT — cleaning, joining, and aggregating huge datasets on a schedule (the bread and butter of data engineering).
  • SQL analytics — running interactive queries over data lakes with Spark SQL.
  • Streaming — near-real-time pipelines with Structured Streaming.
  • Machine learning — iterative training over large data with MLlib (iterative algorithms are exactly where in-memory caching shines).
  • Graph & unified workloads — one engine and one API surface for many data problems, instead of a separate tool per task.
The one-line mental model You describe what result you want; Spark figures out how to compute it in parallel, in memory, and re-computes only what it must if a machine dies. Everything else in this guide is detail hanging off that sentence.

How Hadoop MapReduce Worked — and Why It Was Slow

To understand why Spark is fast, you have to see what it replaced. Hadoop MapReduce forces every computation into a rigid two-step shape — Map then Reduce — and it writes to disk between and around every step.

The Canonical Example: Word Count

Count how many times each word appears across a huge collection of text files. In MapReduce this becomes three phases:

  1. Map: each machine reads its chunk of files and emits a (word, 1) pair for every word it sees.
  2. Shuffle & Sort: the framework groups all pairs by key so that every (the, 1) from every machine ends up together — this requires writing map output to local disk, transferring it across the network, and sorting it.
  3. Reduce: each reducer receives one word's full list of 1s and sums them to a final (word, total).

Why This Is Slow

CostWhat happensWhy it hurts
Disk I/O between stepsMap output is spilled to local disk; the final result is written to HDFS; the next job re-reads it from HDFS.Disk is orders of magnitude slower than RAM. A pipeline of N MapReduce jobs pays this tax N times.
Rigid 2-step modelEverything must be expressed as Map → Reduce. Anything more complex becomes a chain of separate MapReduce jobs.Each job in the chain launches fresh JVMs and re-reads data from HDFS — no reuse of in-memory data across jobs.
No in-memory reuseIterative algorithms (e.g. ML training, graph traversal) re-read the same dataset from disk on every pass.An algorithm with 100 iterations reads the full dataset from disk 100 times.
JVM startup overheadEach job spins up new mapper/reducer processes.Latency dominates for small or interactive queries.
The recap in one line MapReduce is reliable but writes to disk at every boundary and can only think in two steps — so real multi-step pipelines become long chains of disk-bound jobs. That is exactly the bottleneck Spark set out to remove.

How Spark Fixed It: In-Memory + DAG

Spark keeps two promises MapReduce couldn't: intermediate data stays in memory across steps, and you're not forced into two steps — you describe an arbitrarily deep pipeline that Spark plans as a single graph.

Fix 1 — In-Memory Processing

Spark's original core abstraction is the RDD (Resilient Distributed Dataset): a distributed collection of records partitioned across the cluster that Spark can hold in RAM. When you chain operations, the intermediate results stay in memory instead of being written to disk, so the next step reads straight from RAM. For the 100-iteration ML job above, Spark reads the dataset from disk once, caches it, and iterates in memory.

Definition — RDD A Resilient Distributed Dataset is the fundamental unit of data in Spark: an immutable, partitioned collection that can be rebuilt from its lineage if a partition is lost. Today you mostly work with the higher-level DataFrame API, but DataFrames compile down to RDD operations underneath.

"Resilient" — fault tolerance without disk checkpoints

How can you be fault-tolerant if you're not constantly writing to disk? Through lineage. Every RDD remembers the exact sequence of transformations that built it from its source. If a machine dies and takes a partition of data with it, Spark doesn't fail the job — it simply re-computes that one lost partition by replaying its lineage from the last available data. Resilience comes from knowing how to rebuild, not from having saved a copy.

Fix 2 — The DAG (instead of a fixed Map→Reduce)

Rather than forcing your logic into two steps, Spark records your whole chain of operations as a DAG — a Directed Acyclic Graph of transformations. Because it can see the entire graph before running anything, Spark's engine can optimize globally: pipeline operations together, avoid unnecessary passes, and only insert an expensive shuffle where the logic genuinely requires one.

MapReduce

Fixed 2-step shape · disk write at every boundary · a deep pipeline = a chain of separate jobs · no cross-step memory reuse.

Spark

Arbitrary DAG · data kept in memory across steps · whole pipeline planned & optimized as one graph · disk touched only at shuffles / on spill.

Result The combination of in-memory data and whole-graph planning is what delivers the famous 10–100× speedups — largest on iterative and multi-stage workloads, smallest on a single one-pass scan (where there's little intermediate data to keep in memory anyway).

The DAG Execution Model — Vocabulary

Five words carry most of Spark's execution vocabulary: transformation, action, job, stage, task. Get these straight now and the rest of the guide clicks into place; each gets a full section later.

TermOne-line meaning
TransformationA lazy instruction that describes a new dataset from an old one (e.g. filter, select, groupBy). Nothing runs yet — it just extends the DAG.
ActionA command that actually needs a result (e.g. count, collect, write). This is what triggers execution of the DAG built up so far.
JobAll the work triggered by a single action. One action → one job.
StageA slice of a job that can run without a shuffle. Stage boundaries are drawn exactly where a wide transformation forces a shuffle.
TaskThe smallest unit of execution: one stage's work applied to one partition of data. Tasks are what actually run on executors, in parallel.
The chain to memorize Your code → a DAG of transformations → an action triggers a job → the job is cut into stages at shuffle boundaries → each stage runs as parallel tasks, one per partition, on the executors.

02. Spark Architecture: Driver, Cluster Manager, Executors & Tasks

A running Spark application is a small distributed system of its own: one driver that plans and coordinates, a cluster manager that hands out machines, and many executors that do the actual work. Everything else in this part is a zoom-in on one of those three boxes.

The Handshake, in Order

  1. You spark-submit an application (or run a notebook cell). The driver starts and creates a SparkSession.
  2. The driver asks the cluster manager for resources (CPU + memory).
  3. The cluster manager launches executor JVMs on worker nodes and registers them back with the driver.
  4. The driver turns your code into a DAG, splits it into stages and tasks, and ships tasks straight to the executors.
  5. Executors run tasks on their cores, cache data in their memory, and report results and status back to the driver.
  6. When the app finishes (or you call spark.stop()), executors are released back to the cluster manager.
Key distinction The cluster manager is only a landlord: it allocates machines. Once executors exist, the driver is the brain that decides what runs where. The cluster manager is not in the loop for individual tasks.

Layer 1 — The Driver & SparkSession

The driver is the process that runs your program's main() function (or your notebook). It is the single coordinator of the whole application — the one place that holds the plan, the schedule, and the metadata. If the driver dies, the application dies.

What the Driver Actually Does

  • Hosts the SparkSession — your single entry point to all Spark functionality.
  • Builds the logical plan from your transformations and hands it to Catalyst (Spark's query optimizer, covered in detail later) to optimize.
  • Runs the DAG Scheduler — splits each job into stages at shuffle boundaries and decides their order.
  • Runs the Task Scheduler — turns each stage into a set of tasks and assigns them to executor cores, trying to place a task where its data already lives (data locality).
  • Tracks everything — task status, retries on failure, accumulator values, and the metadata that powers the Spark UI.
  • Collects results — when you call collect() or show(), results flow back to the driver's memory (which is why collecting a huge dataset can crash the driver).
SparkSession vs SparkContext SparkContext is the original low-level connection to the cluster (RDD world). SparkSession, introduced in Spark 2.0, is the modern unified entry point that wraps SparkContext and also gives you the DataFrame/SQL APIs. In PySpark you almost always start with spark = SparkSession.builder.getOrCreate() and reach for spark.sparkContext only rarely. 
from pyspark.sql import SparkSession

spark = (SparkSession.builder
         .appName("orders_etl")
         .config("spark.sql.shuffle.partitions", 200)
         .getOrCreate())

# everything flows from this one object:
df = spark.read.parquet("s3://lake/orders/")   # DataFrame API
spark.sql("SELECT * FROM orders LIMIT 10")       # SQL API
sc  = spark.sparkContext                          # low-level RDD access
Sizing the driver The driver needs enough memory for the plan, task bookkeeping, broadcast variables, and anything you collect(). It does not need to be as large as an executor unless you pull big results back to it. A common production mistake is df.collect() on a large DataFrame — it streams every row into driver memory and triggers an out-of-memory crash.

Layer 2 — The Cluster Manager

The cluster manager is the resource landlord. Spark itself is agnostic to which one you use — the same application code runs unchanged on any of them; you only change how you launch it.

Cluster managerWhat it isWhen you see it
StandaloneSpark's own built-in, simple manager (a Master + Workers).Small clusters, dev/test, dedicated Spark-only clusters.
YARNHadoop's resource manager.Still the most common in on-prem / Hadoop-heritage enterprises and managed services like EMR.

The Manager's Only Job

Whichever one you pick, the cluster manager does exactly one thing for Spark: it grants containers (a slice of CPU + memory on some node) in which executors run, and it can take them back when the app ends. It knows nothing about your DAG, your stages, or your tasks — that is all the driver's business. This clean separation is why the same PySpark script runs identically whether you submit it to YARN in a data center or to Kubernetes in the cloud.

You choose the manager at submit time via --master, not in your code — the application source is untouched. Inside the app you can read back which one you landed on:

# The SAME app.py — only the launch line changes which manager grants resources:
#   spark-submit --master yarn                    app.py   # YARN (Hadoop)
#   spark-submit --master k8s://https://HOST:6443 app.py   # Kubernetes
#   spark-submit --master spark://HOST:7077       app.py   # Standalone
#   spark-submit --master local[*]                app.py   # no manager (dev, one JVM)

from pyspark.sql import SparkSession
spark = SparkSession.builder.getOrCreate()

print(spark.sparkContext.master)                    # e.g. "yarn" — which manager am I on?
print(spark.conf.get("spark.submit.deployMode"))    # "client" or "cluster" (see next section)
On Databricks / EMR / Dataproc Managed platforms hide the cluster manager behind a friendly "create cluster" button, but underneath it is still one of the above (Databricks and modern cloud services increasingly use Kubernetes or a proprietary equivalent). The mental model is identical: something grants machines; the driver uses them.

Layer 3 — Executors, Cores & Tasks

Executors are where the real work happens. An executor is a JVM process launched on a worker node that lives for the whole application, runs the tasks the driver sends it, and holds cached data in its memory.

Executor, Core, Slot, Task — untangled

TermWhat it is
Worker nodeA physical or virtual machine in the cluster. Can host one or more executors.
ExecutorA JVM process on a worker node, sized by spark.executor.memory and spark.executor.cores. Runs tasks and stores cached partitions.
Core / slotOne thread of execution inside an executor. An executor with 4 cores has 4 slots and can run 4 tasks at the same time.
TaskOne unit of work = one stage's logic applied to one partition. A task occupies one slot for its duration.
The parallelism equation Total parallel tasks = number of executors × cores per executor. If you have 10 executors with 4 cores each, Spark can run 40 tasks simultaneously. If your stage has 200 partitions, they run in 5 waves of 40.

What Lives Inside an Executor

  • Task threads — one per core, each processing a partition.
  • A block manager — stores cached/persisted RDD & DataFrame partitions and serves shuffle data to other executors.
  • Execution & storage memory — the unified memory region (covered in depth in Part 5) used for shuffles, joins, sorts, and caching.
from pyspark.sql import SparkSession

spark = (SparkSession.builder
         .config("spark.executor.instances", 10)   # how many executor JVMs
         .config("spark.executor.cores",     4)   # slots = parallel tasks per executor
         .config("spark.executor.memory", "8g")    # JVM heap per executor
         .getOrCreate())

sc = spark.sparkContext
# total task slots = executors × cores = 10 × 4 = 40 tasks at once
print(sc.defaultParallelism)                       # default #partitions ≈ total cores (40)
print(sc.getConf().get("spark.executor.cores"))   # confirm the setting took effect
Fat vs thin executors Too many cores per executor (e.g. 16) causes heavy garbage collection and HDFS I/O contention; too few (e.g. 1) wastes memory overhead and loses the benefit of shared broadcast/cache. A widely used rule of thumb is ~5 cores per executor for good HDFS throughput, then size executor memory and count around that. There is no universal answer — it depends on your workload — but "5 cores" is the standard starting point.

03. Deployment Modes: Client vs Cluster

There is exactly one question that separates the two deployment modes: where does the driver run? Everything else follows from that.

Client mode

Driver runs on the machine that submitted the job. You see logs/output live in your terminal. Good for interactive work, notebooks, and spark-shell. Fragile for production: if your laptop/edge node dies or disconnects, the whole app dies, and it's a network bottleneck if far from the cluster.

Cluster mode

Driver runs on a worker node inside the cluster. The submitting machine can disconnect once the job is accepted. The standard for scheduled production jobs — resilient, low-latency driver↔executor communication, and managed by the cluster manager.

Rule of thumb Interactive / exploratory → client mode (notebooks, shells run this way). Scheduled production pipelines → cluster mode. In a Databricks-style notebook you're effectively in a client-like setup where the driver is a managed node in the cluster.

04. Transformations, Actions & Lazy Evaluation

Spark is lazy: transformations don't run when you write them. They only add a node to the DAG. Nothing computes until you call an action — and that laziness is a feature, because it lets Spark see the whole plan before optimizing it.

Transformations (lazy)

select, filter/where, withColumn, groupBy, join, orderBy, distinct, union, drop, repartition. Each returns a new DataFrame and runs nothing — it just records intent.

Actions (eager)

count, collect, show, take, first, foreach, write/save, toPandas. Each forces Spark to execute the accumulated DAG and produce a real result.

df  = spark.read.parquet("s3://lake/orders/")   # lazy: nothing runs
df2 = df.filter(df.amount > 100)                # lazy: DAG grows
df3 = df2.groupBy("country").count()            # lazy: DAG grows
# ---- still nothing has executed on the cluster ----
df3.show()                                       # ACTION: now the whole DAG runs

Why Laziness Wins

  • Whole-plan optimization — Catalyst can rearrange, combine, and prune operations because it sees them all at once (e.g. push a filter down to the scan so less data is ever read).
  • No wasted work — if a branch of the DAG is never needed by any action, it never runs.
  • Pipelining — narrow operations get fused so each record flows through several transformations in one pass, without materializing between them.
The hidden cost of laziness Every action re-runs the DAG from the start (or from the last cached point). If you call three actions on the same expensive df3, Spark recomputes it three times — unless you df3.cache() it first. This is the #1 reason to understand caching (Part 5).

05. Narrow vs Wide Transformations & the Shuffle

This is the single most important performance distinction in Spark. A narrow transformation needs data from only one input partition to produce one output partition. A wide transformation needs data from many partitions — which forces a shuffle: moving data across the network to regroup it.

NarrowWide
Data movementNone — stays on the same executorShuffle — data written to disk, sent over network, re-read
Examplesselect, filter, withColumn, cast, uniongroupBy, join, distinct, orderBy, repartition
Stage impactFused into the current stageEnds the stage; a new stage starts after the shuffle
CostCheapExpensive — usually the slowest part of a job
 
df = spark.read.parquet("s3://lake/orders/")
narrow = df.filter(F.col("amount") > 100).select("country", "amount")  # narrow: no shuffle
wide   = narrow.groupBy("country").count()                              # wide: forces a shuffle
wide.show()
What a shuffle actually does Map-side tasks partition their output by the grouping key and write it to local disk (shuffle write). Reduce-side tasks then pull the pieces they need from every executor over the network (shuffle read) and merge them. It's disk + network + serialization all at once — which is why "minimize shuffles" is the first rule of Spark tuning.

06. Jobs, Stages & Tasks

Let's trace one concrete PySpark snippet all the way from code to tasks, so the job/stage/task boundaries stop being abstract.

df = spark.read.parquet("s3://lake/orders/")     # 8 files → 8 partitions
result = (df
    .filter(df.status == "PAID")               # narrow
    .groupBy("country")                          # WIDE → shuffle
    .sum("amount")
    .orderBy("country"))                         # WIDE → shuffle
result.write.parquet("s3://lake/out/")          # ACTION → triggers the job

How Spark Breaks This Down

1

One action → one job

The write is the only action, so this is one job. (If you'd also called result.count(), that would be a second job re-running the same lineage.)

2

Two shuffles → three stages

Stage boundaries fall at each wide transformation:

  • Stage 1: read parquet + filter (all narrow) — then a shuffle write partitioned by country.
  • Stage 2: read the shuffle, do the groupBy/sum aggregation — then a shuffle write for the ordering.
  • Stage 3: read that shuffle, apply orderBy, and write the output.

(AQE and Spark's optimizer can sometimes combine or reshape stages, but this is the logical breakdown.)

3

Partitions → tasks

Each stage runs one task per partition:

  • Stage 1 reads 8 files → ~8 partitions → 8 tasks.
  • Stage 2 reads shuffle output. The number of partitions is controlled by spark.sql.shuffle.partitions (default 200) → up to 200 tasks (AQE will coalesce these down if the data is small).
  • Stage 3 similarly starts from the shuffle partition count.

If your cluster offers 40 slots (10 executors × 4 cores), Stage 1's 8 tasks run in one wave; Stage 2's 200 tasks run in ~5 waves.

Reading this in the Spark UI The Jobs tab shows one row for the write. Click it and the Stages view shows the three stages with their task counts and a DAG visualization; the shuffle read/write columns tell you exactly how much data moved between them. This is the primary way you diagnose a slow job.

07. The PySpark DataFrame API

A DataFrame is a distributed table: rows partitioned across the cluster, organized into named, typed columns. It's the API you'll write 95% of the time in Python. (We skip the typed Dataset API deliberately — it's a Scala/Java feature; in Python the DataFrame is the untyped Dataset.)

DataFrame = distributed, schema-aware table Under the hood a DataFrame is an RDD of rows plus a schema. The schema is what lets Catalyst optimize your query — it knows the columns and types ahead of time, so it can push filters down and generate tight code. That's why you should prefer DataFrame operations over raw RDD map/lambda code: RDD lambdas are opaque to Catalyst and can't be optimized.

Reading Data

# Columnar formats (Parquet is the default & recommended) carry their own schema
df = spark.read.parquet("s3://lake/orders/")

# CSV/JSON: give a schema explicitly in prod (inferSchema triggers an extra full scan)
from pyspark.sql.types import StructType, StructField, StringType, DoubleType
schema = StructType([
    StructField("order_id", StringType()),
    StructField("country",  StringType()),
    StructField("amount",   DoubleType()),
])
df = spark.read.schema(schema).option("header", True).csv("s3://lake/raw/")

# Read a governed table by name
df = spark.read.table("prod.sales.orders")

The Core Transformations

from pyspark.sql import functions as F

result = (df
    .select("order_id", "country", "amount")        # pick columns (column pruning)
    .filter(F.col("amount") > 100)                       # row filter (pushdown)
    .withColumn("amount_usd", F.col("amount") * 1.0)     # derived column
    .groupBy("country")                                  # wide → shuffle
    .agg(F.sum("amount_usd").alias("total"),
         F.count("*").alias("n_orders"))
    .orderBy(F.desc("total")))                           # wide → shuffle

result.show()                                          # action

UDFs: use built-ins first Prefer pyspark.sql.functions over Python UDFs — built-ins run inside the JVM and are fully optimizable by Catalyst, while a Python UDF ships each row to a Python process and back. When you must write one, Spark 4.0 turns on Arrow-optimized Python UDFs by default (spark.sql.execution.pythonUDF.arrow.enabled=true), which cuts that serialization cost dramatically. Pandas UDFs (vectorized) remain the fastest UDF option for heavy Python logic.

08. Spark SQL

Spark SQL lets you query distributed data with ordinary ANSI SQL. It is not a bolt-on — it's the same Catalyst engine described in Part 3, exposed through a SQL string instead of Python method calls. You can mix it freely with the DataFrame API in the same script.

Three Ways to Run SQL

# 1. Query a registered table / view directly
spark.sql("""
    SELECT country, SUM(amount) AS total
    FROM orders
    WHERE status = 'PAID'
    GROUP BY country
    ORDER BY total DESC
""").show()

# 2. Turn any DataFrame into a temporary view, then query it
df.createOrReplaceTempView("orders")
spark.sql("SELECT * FROM orders WHERE amount > 100")

# 3. Query files directly, no table needed
spark.sql("SELECT * FROM parquet.`s3://lake/orders/`")

Tables, Views & the Catalog

  • Temp view (createOrReplaceTempView) — session-scoped, disappears when the session ends. Just a name for a DataFrame.
  • Global temp view (createGlobalTempView, queried via global_temp.name) — shared across sessions in the same application.
  • Managed / external table — persisted in the metastore (Hive metastore or, on modern platforms, Unity Catalog), survives across applications. Referenced by catalog.schema.table.
ANSI mode is now the default (Spark 4.0) As of Spark 4.0, spark.sql.ansi.enabled defaults to true. That means operations like divide-by-zero, numeric overflow, and invalid casts now raise an error instead of silently returning NULL — safer, but a behavioral change if you're migrating from Spark 3.x. (Spark 4.0 also brings the semi-structured VARIANT type and locale-aware string collation — see What's New in Spark 4.0.)
SQL or DataFrame? It doesn't matter for speed Because both compile to the same optimized Catalyst plan, choose by readability and team preference, not performance. Complex multi-join business logic often reads better as SQL; programmatic, parameterized, or dynamically-built transformations read better as DataFrame code. Many teams mix both.

09. What's New in Spark 4.0

Apache Spark 4.0.0 shipped as GA on May 23, 2025 — the first release of the 4.x line (with 4.1 following in December 2025, and 4.2 since). The core execution model in this guide is unchanged, but several defaults flipped and some genuinely new capabilities arrived. If you're interviewing or upgrading, these are the ones to know.

The Headline Features

Spark Connect

A decoupled thin-client architecture (gRPC + Protobuf): your client talks to a remote Spark server instead of embedding the driver. 4.0 ships a pure-Python pyspark-client (~1.5 MB, no JVM) and a new spark.api.mode toggle. Lets apps, IDEs, and services drive Spark remotely at near-full parity with classic Spark.

ANSI mode ON by default

spark.sql.ansi.enabled=true is now the default (was false in 3.x). Divide-by-zero, overflow, and invalid casts raise errors instead of returning NULL. Safer, stricter — and the biggest breaking change when upgrading.

VARIANT data type

A new semi-structured type that stores JSON/nested data in a single column with efficient nested-field querying — schema-on-read, and far faster than parsing JSON-as-string on every query.

String collation

Locale- and case-aware string comparison and sorting via a COLLATE property (e.g. UNICODE_CI for case-insensitive, UNICODE_CI_AI for case- and accent-insensitive) — without rewriting data.

Python Data Source API

Write custom batch and streaming data sources (read and write) in pure Python — no Scala/Java needed to integrate a new system.

Arrow Python UDFs by default

spark.sql.execution.pythonUDF.arrow.enabled=true now defaults on, cutting the serialization cost of Python UDFs by using Arrow to move data between the JVM and Python.

Defaults & Platform Changes to Know

10. The Catalyst Pipeline: From Code to RDDs

When you write df.filter(...).groupBy(...) or a SQL string, Spark does not run it line by line. It first compiles your intent through the Catalyst optimizer — a fixed pipeline of plan transformations: parsed → analyzed → optimized logical plans, then physical plan candidates, a cost model that picks one, code generation to JVM bytecode, and finally a DAG of RDDs that actually runs. This is the machinery that makes both the SQL and DataFrame APIs fast, and understanding it is what separates people who use Spark from people who tune it.

Stage by Stage

#PlanWhat happens
1Unresolved Logical PlanYour query is parsed into a tree. Spark knows the shape (there's a filter, then a group-by) but hasn't confirmed that orders exists or that amount is a valid, numeric column.
2Analyzed / Resolved Logical PlanThe Analyzer consults the Catalog (table & column metadata) to resolve every name, attach a unique ID and data type to each column, and check the query is valid. A wrong column name fails here (AnalysisException).
3Optimized Logical PlanRule-based optimization rewrites the tree into an equivalent but cheaper one: predicate pushdown (filter before read), column pruning (read only needed columns), constant folding, boolean simplification, and more.
4Physical PlansThe optimized logical plan is turned into one or more executable physical plans — each a concrete strategy (which join algorithm, which aggregation approach).
5Selected Physical PlanSize-based heuristics and thresholds pick the concrete strategy (e.g. "the right table is below the broadcast threshold → broadcast it instead of shuffling"). Note this is not the cost-based optimizer — see the callout below.
6Code GenerationTungsten whole-stage code generation collapses an entire stage into a single generated Java function — no per-row virtual calls — compiled to bytecode and run as RDD tasks on the executors.
Logical vs Physical, in one line A logical plan describes what result you want, in terms of relational operations. A physical plan describes how to produce it, in terms of concrete algorithms and data movement. Catalyst's whole job is to get from the first to the best possible version of the second.
Under the hood: one pipeline for SQL and DataFrames Whether you write SQL text or chain DataFrame methods, both feed the same Catalyst pipeline. A SQL string is turned into an abstract syntax tree by an ANTLR4 grammar; a DataFrame chain builds the identical unresolved logical tree directly. The analysis phase resolves it against the Catalog, logical optimization applies rule batches (pushdown, pruning, constant folding, boolean simplification), physical planning is the one cost-based phase, and code generation emits Java via quasiquotes compiled by the Janino compiler. Four phases, three of them purely rule-based.

Tungsten: the engine under Catalyst

Catalyst decides the plan; Tungsten makes it run close to the metal. It manages memory in a compact off-heap binary format (bypassing slow JVM object overhead and garbage collection), processes data in cache-friendly columnar batches (vectorization), and generates the fused per-stage Java code mentioned above. Catalyst + Tungsten together are why a well-written DataFrame job can rival hand-tuned code.

11. Reading Query Plans: How to Read an explain() Tree

A physical plan is a tree printed as indented text. The trick: read it bottom-up. The leaves (scans) run first and data flows upward to the root, which is the final operation. Indentation and the +- / :- markers show the parent–child structure. Learn to hunt three things — the Exchange nodes, the pushed-down filters, and the join/aggregate strategy — and any plan turns from a wall of text into a map of where your job spends its time.

The three questions to ask of any plan (1) Where are the Exchange nodes? Each is a shuffle — the expensive part. (2) Did my filters push down? Look for PushedFilters in the scan. (3) Which join/aggregate strategy? The operator names tell you. Everything in the rest of this guide is learning to answer these for each kind of transformation.
Two optimizations to watch for Predicate (filter) pushdown moves WHERE conditions down to the data source, so fewer rows are read. Projection (column) pushdown reads only the columns you actually use. Both appear in the physical plan's scan node (as PushedFilters and the reduced column list / ReadSchema) — and both mean "read less," the cheapest possible speedup.

:- vs +-: reading a two-branch operator

Unary operators (Filter, Project) have one child, always marked +-. A binary operator — a join or a union — has two, and the markers distinguish them: :- is the first (left) child and the trailing +- is the second (right) child. The leading : characters are just guide rails that keep the left branch visually connected while you scan down past it to the right branch.

orders.join(customers, "cust_id").explain()
# == Physical Plan ==
# *(5) SortMergeJoin [cust_id#2], [cust_id#20], Inner
# :- *(2) Sort [cust_id#2 ASC NULLS FIRST], false, 0      <- LEFT branch (:-)
# :  +- Exchange hashpartitioning(cust_id#2, 200)          <-   its rail (:)
# :     +- FileScan parquet orders[cust_id#2,...]
# +- *(4) Sort [cust_id#20 ASC NULLS FIRST], false, 0     <- RIGHT branch (+-)
#    +- Exchange hashpartitioning(cust_id#20, 200)
#       +- FileScan parquet customers[cust_id#20,...]
Rule of thumb Read a binary node as: "this operator combines a :- branch and a +- branch." Follow the : rail straight down to find where the left branch ends and the right (+-) begins. The whole-stage-codegen numbers (*(2), *(4), *(5)) also climb as you move up and across — each shuffle starts a new group.

Whole-stage codegen: the *(n) prefix

The first thing worth noticing once you read plans regularly is the *(n) prefix: whole-stage code generation fuses consecutive operators into one compiled Java function, and a new group with the next (higher) number begins at each shuffle.

ReusedExchange

If the same shuffled result feeds two branches, the plan shows ReusedExchange — Spark computed the shuffle once and reused it. Seeing it means an optimization worked.

Column order is free

select("a","b") vs select("b","a") produce the same physical cost — ordering lives in a Project and is essentially free; it never adds a shuffle.

AQE wraps the plan

With AQE on you'll see an AdaptiveSparkPlan root and AQEShuffleRead nodes — and the plan you read after the run (in the SQL tab) may differ from explain()'s static one.

Reading an Adaptive (AQE) plan

Adaptive Query Execution (on by default since Spark 3.2) re-optimizes the plan at runtime using real shuffle statistics. Because explain() never runs the query, the static text always shows isFinalPlan=false — the true adaptive plan appears only in the Spark UI SQL tab after the query finishes. Three AQE rewrites are worth recognising: dynamic partition coalescing, a sort-merge join swapped to broadcast, and skew-join splitting.

When predicate pushdown fails

Pushdown is the biggest free win — but it doesn't always happen. When a filter can't be evaluated at the data source, it stays as a Filter node above the scan, and the scan reads everything. The plan shows this clearly: no PushedFilters, and a Filter sitting higher up.

from pyspark.sql.types import BooleanType
over100 = F.udf(lambda a: a > 100, BooleanType())
df.filter(over100("amount")).explain()    # no PushedFilters — Filter above scan, reads all rows
df.filter("amount > 100").explain()      # PushedFilters:[GreaterThan(amount,100)]
Common pushdown blockers
  • Filtering on a UDF result or any computed/derived column (filter the base column first, or before the UDF).
  • Non-deterministic expressions (e.g. rand(), monotonically_increasing_id()).
  • Filtering on an aggregate or window output (it doesn't exist at the source yet).
  • Sources that don't support the predicate (some formats/connectors, or complex expressions on nested types).
Fix: push the filter earlier in the chain — filter base columns before UDFs, joins, and aggregations so Catalyst can reach the scan.

12. Reading the DAG & the Spark UI

A DAG (Directed Acyclic Graph) is the picture of your computation: the operators, wired together, that Spark runs to produce a result. An action triggers a job; the job is cut into stages at every shuffle; each stage fans out into tasks (one per partition). The Spark UI draws all of this — and learning to read it is how you see, at a glance, where a job spends its time.

Job → Stages → Tasks, in one line An action (e.g. write, count, show) starts a job. Spark cuts the job into stages at each shuffle boundary. Each stage runs as parallel tasks, one per partition. The DAG shows the stages and the operators inside them; the shuffles are the seams between stages.

Jobs tab vs SQL / DataFrame tab — the same run at two zoom levels

These are two views of one execution, and knowing which to open saves time. The Jobs tab is the scheduler's-eye view: one row per action, an event-timeline, and a click-through to each job's coarse DAG of stage boxes. The SQL / DataFrame tab is the query planner's view: one row per query, and a click-through to the fine, operator-level DAG (Scan, Filter, Exchange…) with live per-node metrics. A single action can even span several jobs (Spark runs a probe job to size a broadcast, for instance) — but on the SQL tab it's still one query, which is why that view is easier to reason about.

Input partitions

On read, the partition count comes from file splits sized by spark.sql.files.maxPartitionBytes (~128 MB). One task per partition.

Input batches

The Parquet reader is vectorized — it reads columnar batches of rows (Batched: true), which ColumnarToRow then feeds into codegen. Fewer, bigger batches = less overhead.

Metadata matters

The scan reads Parquet footer metadata (row counts, min/max) — enabling skipping, and letting some queries (e.g. a bare count(*)) be answered from metadata with almost no data read.

Post-shuffle partitions: the 200 you'll always see — and why it's a ceiling Any stage after a shuffle defaults to 200 tasks (spark.sql.shuffle.partitions). For small data that's 200 near-empty tasks (overhead); for huge data it may be too few (spill). With spark.sql.adaptive.enabled and …coalescePartitions.enabled on, AQE measures each shuffle's real output and merges small post-shuffle partitions toward spark.sql.adaptive.advisoryPartitionSizeInBytes (~64 MB), dropping the count below 200. The trap: AQE can only go below the ceiling — it never raises it. On big jobs set the ceiling high (e.g. 2000) and let AQE coalesce down.

13. Narrow & Shuffle Operations: Plan + DAG

Now we read the operations themselves, each in both views at once: the explain() tree and the Spark-UI DAG. Start with the cheap end — reading a file and the narrow transforms (select, filter, withColumn) that need no shuffle — then the wide ones (repartition, groupBy, joins) whose Exchange node is where one stage ends and the next begins.

Reading a file — the simplest DAG

Reading a Parquet file and showing it is narrow — one job, one stage, no Exchange. The SQL-tab DAG is just a Scan parquet feeding a WholeStageCodegen box, and the interesting part is the metrics on the scan.

# a plain read + action: one job, one stage, no Exchange
df = spark.read.parquet("orders")
df.count()
df.show(5)
“8 files → 8 tasks” is a simplification — here's the real rule Input partitions aren't one-per-file. Spark packs file splits into partitions of at most spark.sql.files.maxPartitionBytes (128 MB), adding spark.sql.files.openCostInBytes (~4 MB) of “padding” per file so many tiny files don't each become a task. So one 1 GB file can split into ~8 partitions, and a big file only splits where a Parquet row-group boundary allows it. One tidy 200 MB file → 2 tasks; a thousand 1 KB files → a handful of tasks, not a thousand.
The scan that reads almost nothing: count(*) from the footer A bare df.count() over Parquet often shows a scan with tiny bytes-read: Spark answers row counts straight from each file's footer metadata instead of reading the columns. Likewise, min/max statistics in the footer let Spark skip whole row groups (data skipping) — which is exactly what the scan's rows-output metric confirms when it's far below the file's total.

Narrow transformations — plan and DAG

select, filter/where, and withColumn are narrow — each output partition depends on one input partition, so there's no shuffle. In the plan they appear as Project and Filter, fused into a single codegen stage, and Catalyst pushes both the filter and the column list down into the scan. In the DAG they all sit inside the same WholeStageCodegen box — still one stage.

df.select("id", "amount").filter("amount > 100").explain()
# narrow: select + filter + withColumn fuse into one stage
df = (spark.read.parquet("orders")
        .select("id", "amount")
        .filter("amount > 100")
        .withColumn("fee", F.col("amount") * 0.02))
Read the Scan node, not just the Filter The optimizer pushes work down into the scan, so the Filter node in the DAG may already be redundant. On the Scan parquet node look for three lines: PushedFilters: [GreaterThan(amount,100)] (predicate pushdown), ReadSchema: struct<id,amount> (column pruning — only two columns decoded), and, for partitioned tables, PartitionFilters (whole directories skipped). If rows output at the scan is already close to the post-filter count, pushdown did its job before a single row hit the Filter. withColumn reads the same way — the new expression just appears inside a Project; still narrow, unless the expression itself triggers a wide operation (e.g. a window).

Wide transformations & the shuffle

A wide transformation (join, groupBy, distinct, repartition) needs data regrouped across partitions — a shuffle. In the DAG that shows up as an Exchange node, and the Exchange is exactly where one stage ends and the next begins. This one shape explains every multi-stage DAG in the rest of the guide.

# the Exchange subtitle tells you HOW it partitions:
df.repartition(8, "country").explain()   # Exchange hashpartitioning(country, 8)
df.orderBy("country").explain()          # Exchange rangepartitioning(country ASC, 200)
df.agg(F.sum("amount")).explain()        # Exchange SinglePartition  (everything to one task)
Read the Exchange's subtitle — it names the shuffle Every Exchange is labelled with its partitioning scheme: hashpartitioning(key, N) (joins, groupBy, keyed repartition), rangepartitioning(key, N) (a global orderBy — it runs a sampling job first to pick range boundaries), SinglePartition (a global aggregate or coalesce(1) funnels everything into one task — a classic accidental bottleneck), and RoundRobinPartitioning (a plain repartition(N) with no key).

Repartition: the three Exchange flavours

repartition always introduces an Exchange — a full shuffle — but which kind depends on how you call it. With just a number you get round-robin partitioning (rows spread evenly, no key); with a column you get hash partitioning (same key → same partition).

df.repartition(8).explain()                # Exchange RoundRobinPartitioning(8)
df.repartition(8, "country").explain()     # Exchange hashpartitioning(country, 8)
Every Exchange costs A repartition is a deliberate, full shuffle. It's worth it to fix skew or raise parallelism, but a stray repartition you forgot about is pure overhead — if you see an Exchange RoundRobinPartitioning you didn't intend, that's a free win to remove.

Coalesce: fewer partitions with no shuffle

coalesce also reduces the partition count, but the plan looks completely different: there's no Exchange. Instead you see a Coalesce node, because it merges adjacent partitions on the same executor without a shuffle. That single difference — Exchange vs Coalesce in the plan — is the whole point.

df.coalesce(2).explain()   # Coalesce 2 — no Exchange (contrast repartition)
One shuffle, reused: ReusedExchange If the same shuffled dataset feeds two branches (a self-join, or a DataFrame used twice), Spark computes the Exchange once and drops a ReusedExchange node on the second branch — and the Jobs tab shows that stage as skipped. Seeing ReusedExchange is money saved; not seeing it where you expected reuse means you're shuffling the same data twice (often fixed with a cache()).

14. Joins in Plans & DAGs

A join is the clearest example of a two-branch tree, and its shape in both the plan and the DAG tells you immediately which strategy Spark chose — and therefore how much it will cost. A large-to-large equi-join becomes a sort-merge join (two shuffles, three stages); when one side is small enough, a broadcast hash join avoids shuffling the big side entirely.

Sort-merge join — plan and DAG

For a large-to-large equi-join you'll see a SortMergeJoin at the top, and beneath each side an Exchange hashpartitioning (the shuffle) and a Sort. The :- marks the first child, +- the second. In the DAG the same thing appears as two symmetric Scan → Exchange → Sort branches feeding the join — two Exchange nodes, so three stages.

orders.join(customers, "cust_id").explain()   # SortMergeJoin + 2× Exchange + 2× Sort
Where the SMJ actually hurts: the Sort, and its spill The two Exchange nodes get the blame, but the Sort above each shuffle is often the real cost — sorting a large partition that doesn't fit in memory spills to disk, and that shows up as spill (memory) / spill (disk) on the join stage in the Stages tab. Both Exchangees use the same hashpartitioning(cust_id, 200), which is what lets matching keys meet in the same task. Skew on cust_id (one hot key) makes one task's sort huge — the classic SMJ straggler. If both inputs are already bucketed and sorted on the key, the Exchange and Sort disappear and the DAG drops straight into SortMergeJoin.

The three equi-join operators — and their signatures

Spark's JoinSelection strategy picks one of three physical operators for an equi-join. You read which straight off the operator name, and its arguments tell you the keys, the join type, and (for hash joins) which side became the hash table.

# force each strategy to see all three in a plan:
big.join(F.broadcast(small), "cust_id").explain()          # BroadcastHashJoin
big.join(other.hint("shuffle_hash"), "cust_id").explain()  # ShuffledHashJoin
big.join(other, "cust_id").explain()                        # SortMergeJoin (default)

Why SortMergeJoin is the default, not the faster ShuffledHashJoin ShuffledHashJoin skips the sort, so it can be faster — but it builds an in-memory hash table per partition, and if a partition doesn't fit in memory the task fails with an OOM. SortMergeJoin can spill its sort to disk, so it degrades gracefully instead of failing. That robustness is why spark.sql.join.preferSortMergeJoin is true by default. Seeing a SortMergeJoin where one side is clearly small is the classic cue to add broadcast().

Broadcast join — the DAG changes shape

When one side is small, Spark broadcasts it and joins locally — so the big side is never shuffled. The DAG shows a BroadcastHashJoin with the small side coming through a BroadcastExchange (a broadcast, not a shuffle) — and no Exchange on the big side. Compare it directly with the sort-merge DAG above: the big branch flows straight into the join.

from pyspark.sql.functions import broadcast
# small side broadcast -> big side is never shuffled
orders    = spark.read.parquet("orders")
customers = spark.read.parquet("customers")
orders.join(broadcast(customers), "cust_id").show()

# Spark auto-broadcasts a side under this threshold (default 10 MB); -1 disables it
spark.conf.get("spark.sql.autoBroadcastJoinThreshold")   # 10485760
When Spark picks BHJ for you — and when it won't Spark auto-chooses a broadcast join when it estimates one side is under spark.sql.autoBroadcastJoinThreshold (10 MB by default). The estimate comes from stats, so a small table that Spark thinks is large (no stats, or bloated after a transform) gets a SortMergeJoin instead — that's when an explicit broadcast() hint earns its keep. Under AQE, Spark can also switch a planned SMJ to a BHJ at runtime once it sees the real post-shuffle size.
The hidden cost + the gotcha A broadcast isn't free: the small side is collected to the driver, then shipped to every executor — look for a small extra job feeding the BroadcastExchange. Broadcast a side that's actually large and you get a driver OOM or a spark.driver.maxResultSize error. Rule of thumb: broadcast the side that stays comfortably under the threshold, and never a side that grows after its filters.

15. Group-By Aggregations in Plans & DAGs: count, sum, count-distinct

Aggregations reveal one of Spark's smartest tricks: two-phase (partial) aggregation. A groupBy(...).count() or .sum() has two HashAggregate nodes around a single Exchange — a partial aggregate that pre-aggregates within each partition before the shuffle, and a final one that combines the partials after. countDistinct is the exception: it must deduplicate first, so it grows a second shuffle.

Group by count — plan and DAG

The partial_count node below the Exchange pre-counts within each partition; the shuffle then carries per-key partials, not raw rows; the final count combines them. Two HashAggregates straddling one Exchange = two stages.

df.groupBy("country").count().explain()   # partial_count → Exchange → count
When the partial aggregate can't save you: high-cardinality keys Pre-aggregation only shrinks the shuffle when there are few groups. Group by something near-unique (a user_id, a timestamp) and the partial HashAggregate can't collapse much — its in-memory hash map overflows and you'll see spill on that node, plus a shuffle that's nearly as big as the input. The DAG shape is identical; it's the metrics (spill > 0, huge shuffle write) that reveal the difference. Healthy: the partial's rows output is already tiny and the Exchange's shuffle write is kilobytes. Unhealthy: the partial passes through nearly as many rows as it read.

Group by sum — the same shape

groupBy(...).sum() produces the exact same plan and DAG as count — two stages around one Exchange. Only the aggregate labels change: the lower node is partial_sum and the upper is sum. Recognising that the shape is universal is the point.

df.groupBy("country").sum("amount").explain()
# == Physical Plan ==
# HashAggregate(keys=[country], functions=[sum(amount)])          <- final
# +- Exchange hashpartitioning(country, 200)                       <- shuffle
#    +- HashAggregate(keys=[country], functions=[partial_sum(amount)])  <- partial
#       +- FileScan parquet [country, amount]

Same two stages

WholeStageCodegen (1) with the partial aggregate → Exchange → WholeStageCodegen (2) with the final aggregate. Byte-for-byte the layout of Fig. 19.

Only the function differs

Lower node: HashAggregate ... partial_sum(amount). Upper node: HashAggregate ... sum(amount). Same story for min, max, avg.

Why it matters

All these aggregates run in a two-phase (partial/final) form, so Spark can pre-aggregate before the shuffle. The one that can't is count distinct.

The partial/final pattern is universal count → partial_count then count; sum → partial_sum then sum; likewise min, max, and avg (which carries a partial sum and count). All are associative, so Spark can pre-aggregate each partition and merge later — shuffling a handful of partials per key instead of every row. The exception is next.

Group by count distinct — two shuffles

countDistinct is the expensive aggregate. To count distinct values Spark must first deduplicate them — so the plan (and DAG) grows a second aggregation round and a second Exchange. Two shuffles instead of one, three stages, is the tell-tale that a distinct is costing you.

df.groupBy("country").agg(F.countDistinct("amount")).explain()   # two Exchanges: dedup then count
# approximate answer -> HyperLogLog -> back to ONE shuffle, two stages
df.groupBy("country").agg(F.approx_count_distinct("amount")).show()
# multiple distincts in one query multiply the Exchanges — watch the plan explode
df.agg(F.countDistinct("amount"), F.countDistinct("cust_id")).explain()

Two or more distincts → the Expand node

One distinct needs two shuffles. Two different distinct columns can't share a single dedup grouping, so Catalyst inserts an Expand: it replicates every input row once per distinct expression, tagging each copy with a gid (grouping id) that says which distinct it belongs to. The plan then dedups and counts each group by that tag. More distincts → a wider Expand → more rows flowing into the shuffle.

(df.groupBy("country")
   .agg(F.countDistinct("amount"), F.countDistinct("user_id"))
   .explain())
# == Physical Plan == (abridged)
# HashAggregate(keys=[country], count(if gid=1 amount), count(if gid=2 user_id))  <- final
# +- Exchange hashpartitioning(country)
#    +- HashAggregate(keys=[country, amount, user_id, gid])   <- dedup per group
#       +- Exchange hashpartitioning(country, amount, user_id, gid)
#          +- Expand [[country, amount, null, 1], [country, null, user_id, 2]]  <- ROW FAN-OUT
#             +- FileScan parquet [country, amount, user_id]
The Expand is the cost multiplier With two distincts the Expand doubles the row count feeding the shuffle; three triples it, and so on. That is why several countDistincts in one query can be dramatically slower than the same number of plain aggregates. If exactness is negotiable, approx_count_distinct (HyperLogLog) needs no dedup shuffle and no Expand — it merges small sketches, collapsing the whole shape back to a single-shuffle two-phase aggregate.

The same class of query as a full DAG — four stages

Add a join and a write and the DAG gets long: read two tables, filter, sort-merge join them, then groupBy aggregate and write. Three Exchange nodes → four stages — and every shape here came from the sections above.

How to read a long DAG fast Don't read every node — find the seams. Count the three Exchange nodes: that's 3 shuffles and 4 stages. The same 30-second routine works no matter how long the DAG gets: locate the shuffles, find the biggest one, and check the joins are broadcasting when they can.

Worked example: diagnose a real job across the UI tabs

Everything so far, on one real job. Run a filter → join → groupBy → write, then walk the three tabs in order — Jobs (count the stages), SQL / DataFrame (find the Exchangees), Stages (open the slow one and read its tasks) — and end with a diagnosis.

from pyspark.sql import functions as F

orders    = spark.read.parquet("orders")      # ~1B rows
customers = spark.read.parquet("customers")   # ~2M rows (too big to auto-broadcast)

result = (orders
            .filter("amount > 100")
            .join(customers, "cust_id")         # large + large -> SortMergeJoin
            .groupBy("country")
            .agg(F.sum("amount").alias("total")))

result.write.mode("overwrite").parquet("out/by_country")   # the action that starts it all

Step 1 — Jobs tab: how many stages? The write is the action. Click into the job: it's a handful of stage boxes. You'll see four compute stages — two parallel scan-and-shuffle stages (one per join side), a join+partial-aggregate stage, and a final-aggregate+write stage. Four stages means three shuffles happened.

Step 2 — SQL / DataFrame tab: find the Exchanges. Read the operator DAG bottom-up (Fig. 23's shape). Scan parquet orders shows PushedFilters:[amount>100]; the join is a SortMergeJoin [cust_id], not a broadcast — so two Exchange hashpartitioning(cust_id, 200) nodes; the groupBy adds a third by country. Three Exchange nodes → four stages, matching Step 1. Hover the biggest Exchange: the join shuffles move the whole billion-row orders table.

First diagnosis, from the SQL tab alone customers is 2M rows — over the 10 MB auto-broadcast threshold, so Spark chose a SortMergeJoin and is shuffling both billion-row and two-million-row sides. If customers can be trimmed (project fewer columns, pre-filter) under ~10 MB, a broadcast() hint would delete two of the three shuffles outright. Note it, then confirm the pain in the Stages tab.

Step 3 — Stages tab: open the slow stage and read its tasks. Here it's Stage 2 (read both join shuffles → sort → merge). Expand Summary Metrics and compare max against median across tasks.

The diagnosis The DAG shape was fine — three shuffles for a filter+join+aggregate is expected. The metrics told the real story: Step 2 flagged an avoidable SortMergeJoin (customers just over the broadcast threshold), and Step 3 exposed skew on cust_id — one hot key funnelling ~780 MB into a single task that spilled. Two independent problems, found in three clicks.
The fix — and how the DAG changes Two moves: (1) trim and broadcast(customers) so the two join Exchangees vanish — the big side stops shuffling (Fig. 24's shape); (2) turn on AQE skew handling (spark.sql.adaptive.skewJoin.enabled), which splits the hot partition into several tasks so no single one carries 780 MB. Re-run, reopen Stage 2: max-vs-median tightens, spill drops to 0. That before/after in the Stages tab is how you prove a tuning change worked. Read the DAG for the shape (how many shuffles, which join type), the Stages tab for the scale (which task, how much data, spill, skew). 
# the two fixes suggested by the DAG + Stages tab
customers_slim = customers.select("cust_id", "country")   # shrink under ~10 MB

spark.conf.set("spark.sql.adaptive.enabled", True)
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", True)  # split the hot partition

result2 = (orders.filter("amount > 100")
                 .join(F.broadcast(customers_slim), "cust_id")   # kills 2 of 3 shuffles
                 .groupBy("country")
                 .agg(F.sum("amount").alias("total")))
result2.explain("formatted")   # now: BroadcastHashJoin, one Exchange, two stages

17. Adaptive Query Execution (AQE)

Catalyst optimizes before the query runs, using estimates. But estimates are often wrong. Adaptive Query Execution re-optimizes the plan during execution, using the real statistics Spark observes after each shuffle. Introduced in Spark 3.0 and on by default since Spark 3.2 (spark.sql.adaptive.enabled=true), it's one of the biggest free wins in modern Spark.

The three things AQE does

Coalesce shuffle partitions

The static default of 200 shuffle partitions is often wrong. AQE merges small post-shuffle partitions into right-sized ones at runtime, so you stop getting hundreds of tiny tasks. Target size ~64 MB.

Switch join strategy

If a shuffle reveals one side is actually small enough to broadcast, AQE converts a planned sort-merge join into a broadcast hash join on the fly — skipping a huge shuffle.

Handle skew joins

If one partition is far larger than the others (data skew), AQE splits that oversized partition into smaller sub-partitions so one straggler task doesn't hold up the whole stage.

spark.conf.set("spark.sql.adaptive.enabled", True)                 # on by default since 3.2
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", True)
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", True)
Practical takeaway On Spark 3.2+ you get AQE for free. It's the reason the old advice to hand-tune spark.sql.shuffle.partitions for every job matters much less now — AQE reshapes partitions dynamically. Related: Dynamic Partition Pruning (spark.sql.optimizer.dynamicPartitionPruning.enabled, on by default since 3.0) prunes fact-table partitions at runtime using a dimension-table filter, a big win for star-schema joins.
Recognising AQE in the plan & UI Because explain() runs before execution, its static text always shows AdaptiveSparkPlan isFinalPlan=false. The final adaptive plan — with AQEShuffleRead coalesced nodes, a swapped BroadcastHashJoin, or a split skew partition — appears only in the Spark UI SQL tab after the query finishes. The skew-join branch of AQE (and its alternatives, like salting) is covered in depth in the Data Skew part; here it's enough to know AQE detects and splits a hot partition automatically.

19. Partitions: The Unit of Parallelism & Input Partitions

A partition is Spark's unit of parallelism: one partition becomes one task, run by one core. The first place partitions are created is on read. When Spark opens files, it doesn't hand a whole file to one task — it packs the bytes into chunks of at most spark.sql.files.maxPartitionBytes (default 128 MB) and each chunk becomes one input partition, which becomes one task. So this config directly sets how many tasks read your data, and therefore how parallel that read is.

# default is 128 MB; set it explicitly (bytes) before reading

Why partitions are rarely exactly 128 MB — openCostInBytes

# once openCostInBytes padding is folded in.

20. Shuffle Partitions: Sizing spark.sql.shuffle.partitions

A wide operation — a groupBy, a join, a distinct — needs every row with the same key in one place. To get them there Spark inserts a shuffle (an Exchange): each map task re-partitions its rows by key and writes them out, then reduce tasks pull the pieces that belong to them. The number of pieces the data is regrouped into is spark.sql.shuffle.partitions — and it defaults to 200. Each of those post-shuffle partitions becomes one task.

# the default number of post-shuffle partitions
spark.conf.get("spark.sql.shuffle.partitions")   # -> '200'

# any wide op inserts a shuffle (Exchange) and produces 200 partitions
orders.groupBy("product_id").agg(F.sum("amount")).explain()
# look for 'Exchange hashpartitioning(product_id, 200)' in the plan
Shuffle partition, in one line A shuffle partition is one of the N buckets the shuffled data is regrouped into by key. N = spark.sql.shuffle.partitions (default 200), and Spark runs one task per bucket — so this one number sets both your parallelism and how much data each task must chew through.

What actually happens: map write → network → reduce read

A shuffle is not one operation but a hand-off between two stages. The stage before the Exchange (the map side) finishes by writing its output; the stage after it (the reduce side) begins by reading that output back over the network. Three concrete steps sit between them, and each one shows up as a distinct number in the Spark UI:

PhaseWhat each task doesSpark UI metric
1. Map writeEvery map task partitions its rows by hash(key) % N, sorts them into N buckets, and writes those buckets to local disk as shuffle files (plus a tiny index).Shuffle Write (bytes / records)
2. Network fetchEach reduce task k asks every executor for its bucket k and pulls those blocks across the wire. All-to-all: M maps × N reduces block transfers.Shuffle Read Blocked Time
3. Reduce readThe reduce task merges the fetched blocks for its key range and runs the aggregation / join. If the merged data outgrows execution memory, it spills (Scenario 1).Shuffle Read (bytes / records)

Two consequences fall straight out of this. First, the map side always writes exactly N buckets per task regardless of how much data it has — so a huge N on a small dataset means many near-empty files. Second, the number you divide by the sizing formula is Shuffle Write, because that is the real volume crossing the Exchange.

Shuffle Write is the number that matters The bytes the map side writes (Shuffle Write) equal the bytes the reduce side reads (Shuffle Read) — that total, divided by your per-partition target, is the right partition count. Everything in this section comes back to that one figure.

Why shuffle partitions matter

The count does two jobs at once: it sets the post-shuffle parallelism (how many tasks run) and the per-partition data size (how much each task handles). The two pull against each other — more partitions means smaller tasks, fewer partitions means bigger tasks — and the fixed default of 200 only happens to be right for one particular data volume. For everything else it is either too many or too few. That is the whole game: find the count where each partition is a comfortable size.

The tension in one sentence Partition count trades task size against task count — halve the count and every task holds twice the data; double it and each task holds half but you pay twice the scheduling. Tuning is finding the count where per-partition size lands in the sweet spot (~128–200 MB) without exploding the task count.

The per-partition sweet spot

There is a well-worn target for how much shuffled data a single task should handle. Below it you are paying overhead for nothing; above it you risk memory pressure and spill. Roughly:

Per-partition sizeWhat happensVerdict
< ~10 MBTask overhead (schedule, launch, fetch, commit) rivals or beats the useful work.Too small — too many partitions
~128–200 MBFits comfortably in one core's execution memory; overhead amortised.Sweet spot

Why 200 is almost never the right answer

The default of 200 is a fixed constant baked in years ago; it has no knowledge of your data. It is exactly right only when your shuffle happens to be about 200 × 128 MB ≈ 25 GB. Push past that and 200 partitions are each too fat (spill); a 1 GB shuffle on 200 partitions gives ~5 MB tasks (all overhead). Worse, 200 is rarely an exact multiple of your core count, so the final wave runs a lopsided handful of tasks while the rest of the cluster sits idle. The number was never tuned to anything — which is the whole reason AQE now exists to override it at runtime.

Skew is a different disease Right-sizing the count assumes rows spread evenly across keys. If one key holds the lion's share (data skew), one partition stays huge no matter how high you set N — more partitions just means more small ones plus the same one giant straggler. That needs spark.sql.adaptive.skewJoin.enabled or a salting/repartition strategy, not a bigger partition count. Keep the two problems separate.

Scenario 1 — partitions too LARGE

With too few partitions for the data volume, each post-shuffle partition is enormous. When a task's data no longer fits in its slice of execution memory, Spark cannot hold it — so it sorts what it has and spills the overflow to disk, in one or more files, then reads them back and merges. Disk is orders of magnitude slower than memory, so those few fat tasks run long and drag the whole stage. This is the classic "why is my job so slow" cause.

# symptom: only 200 partitions for a large shuffle -> each is too big -> spill
# fix direction: MORE partitions so each fits in execution memory
spark.conf.set("spark.sql.shuffle.partitions", 800)

big_orders.groupBy("product_id").agg(F.sum("amount")).write.parquet("/out")
# check the stage in the Spark UI: Spill (memory)/(disk) should drop to 0
The cost of too few Too few partitions → each too big → execution memory overflows → sort + spill to disk (or, worse, an OutOfMemoryError). A few fat, slow tasks set the stage's wall-clock time. The default 200 is too small the moment your shuffle is more than ~25 GB.

Spill isn't the only tax — GC is the hidden one

Before a task spills, it first fills the JVM heap with the objects it is holding. A partition near or over the memory limit keeps the heap under constant pressure, so the garbage collector runs long, frequent pauses — sometimes so severe the executor stalls or is killed. This is why the fix for "too large" is often described as memory tuning: fewer bytes per task means less to GC and less to spill. Watch the GC Time column in the stage's task table; if it is a large fraction of task duration, the partitions are too big.

Reading the two spill numbers Spark reports spill twice. Spill (memory) is the deserialized size of what was evicted — how much data didn't fit. Spill (disk) is the smaller, serialized/compressed size actually written out. Both being > 0 across all tasks (not just one) says the whole stage is under-partitioned; if only one task spills, suspect skew instead.

Scenario 2 — partitions too SMALL

The opposite mistake: too many partitions for a modest amount of data. Now every task holds only a trickle of rows — kilobytes, not megabytes — and the fixed overhead of each task (scheduling it, launching it, reading its shuffle blocks, writing its output) dwarfs the actual work. You end up with thousands of near-empty tasks, and the driver spends its time bookkeeping rather than computing. A 5 GB job on the default 200 partitions is often fine; a 200 MB job on 200 partitions is death by a thousand tasks.

# symptom: a small ~200 MB shuffle spread over 200 partitions = 200 tiny tasks
# fix direction: FEWER partitions so each task does meaningful work
spark.conf.set("spark.sql.shuffle.partitions", 8)

small_dim.groupBy("region").count().show()
# the stage now runs a handful of healthy tasks instead of 200 near-empty ones
The cost of too many Too many partitions → tiny tasks → scheduling and startup overhead dominate. Symptoms: a stage with hundreds/thousands of tasks each running in milliseconds, a busy driver, and total time far larger than the sum of useful work. Common after filtering a big dataset down small but keeping 200 partitions.
How this looks in the Spark UI Open the stage and check three things: a very high task count, a median Duration in the tens of milliseconds, and a median Shuffle Read in kilobytes rather than megabytes. When per-task read is far below the ~128 MB target and there is no spill, you are over-partitioned — lower shuffle.partitions, or just let AQE coalescePartitions merge them.

How to tune slow jobs

Both failure modes have the same cure: make each post-shuffle partition a comfortable size. There is a simple sizing formula, a rule for rounding it to your cluster, and — since Spark 3.2 — an automatic option (AQE) that right-sizes partitions at runtime so you often don't have to.

The sizing formula

Target roughly 128–200 MB of shuffled data per partition, then round to a multiple of your total core count so no core sits idle on the last wave:

from pyspark.sql import functions as F

# sizing: partitions ~= total shuffle size / target (~128-200 MB)
#   100 GB shuffle / 128 MB  ~= 800 partitions
#   round to a multiple of total cores (e.g. 160 cores -> 800 = 5 waves)
spark.conf.set("spark.sql.shuffle.partitions", 800)

# set it PER QUERY, right before the wide op, sized to THAT shuffle
result = (orders.join(customers, "cust_id")
                .groupBy("region")
                .agg(F.sum("amount")))

Let AQE right-size it for you (Spark 3.2+)

Adaptive Query Execution knows the actual shuffle sizes at runtime, so it can coalesce the partitions for you — it starts from a high partition count and merges adjacent small ones until each is about advisoryPartitionSizeInBytes (~64 MB). This kills the "too many tiny tasks" problem automatically and is on by default since 3.2:

# AQE right-sizes post-shuffle partitions automatically (default on in 3.2+)
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")
spark.conf.set("spark.sql.adaptive.advisoryPartitionSizeInBytes", "64m")

# with AQE on, you can set a generous shuffle.partitions and let it coalesce down
spark.conf.set("spark.sql.shuffle.partitions", 2000)
Gotcha #1 — AQE only coalesces DOWN, never up Coalescing starts from initialPartitionNum (which defaults to shuffle.partitions) and only merges partitions — it never splits or adds. So if you leave shuffle.partitions=200 on a big shuffle, AQE is stuck at 200 fat partitions and still spills. The idiom is the reverse of hand-tuning: set the count deliberately high (or set initialPartitionNum) and let AQE shrink it.
Gotcha #2 — parallelismFirst ignores your 64 MB target By default spark.sql.adaptive.coalescePartitions.parallelismFirst=true, which tells Spark to maximise parallelism and pack partitions down to minPartitionSize (default 1 MB) rather than honour advisoryPartitionSizeInBytes. If you actually want ~64–128 MB partitions, set parallelismFirst=false so the advisory size is respected. 
# make AQE honour the advisory target instead of maximising parallelism
spark.conf.set("spark.sql.adaptive.coalescePartitions.parallelismFirst", "false")
spark.conf.set("spark.sql.adaptive.advisoryPartitionSizeInBytes", "128m")
spark.conf.set("spark.sql.adaptive.coalescePartitions.minPartitionSize", "16m")

# skew is a separate switch (on by default in 3.2+): splits the one giant partition
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")

Worked example — right-sizing a 12 GB shuffle

Let's walk the whole loop on one concrete job: a join then groupBy whose Exchange moves 12 GB of data on a cluster with 32 cores. First we run it on the default 200, read the real shuffle size off the Spark UI, apply the sizing formula, and re-run — then contrast what AQE would have done on its own.

Step 1 — run once, read the shuffle size

Run the query as-is, open the stage that writes the Exchange, and read its Shuffle Write total. Say it reads 12 GB = 12288 MB. That is the only measurement the formula needs.

Step 2 — apply the formula

Divide the shuffled bytes by a ~128 MB target, then round to a multiple of the 32 cores so every wave fills the cluster:

# measured from the Spark UI: Shuffle Write = 12288 MB, cluster = 32 cores
total_mb = 12288
target_mb = 128
raw = total_mb / target_mb                 # -> 96.0 partitions

# round to a multiple of total cores (32): 96 = 3 x 32 = 3 clean waves
cores = 32
partitions = round(raw / cores) * cores    # -> 96

So 12288 / 128 = 96, and 96 is already 3 × 32 — three full waves of tasks, no idle tail. Set it right before the wide op, sized to this shuffle:

from pyspark.sql import functions as F

# size the shuffle to ~128 MB/partition for THIS query
spark.conf.set("spark.sql.shuffle.partitions", 96)

result = (orders.join(customers, "cust_id")        # the Exchange (~12 GB)
                .groupBy("region")
                .agg(F.sum("amount").alias("revenue")))
result.write.mode("overwrite").parquet("/out/revenue")

Step 3 — contrast the default 200

Why not just leave it at 200? Because 12288 / 200 ≈ 61 MB per task — below the sweet spot, so the tasks are on the small side. And 200 is not a multiple of 32: 200 = 6×32 + 8, so after six full waves a ragged seventh wave of only 8 tasks trails behind while 24 cores sit idle. For this 12 GB shuffle, 200 is too many, not too few — a reminder that the default is arbitrary in both directions.

Step 4 — what AQE does on its own

With AQE you skip the hand-measurement: set a generous starting count and a 128 MB advisory, and let it coalesce down at runtime. Because the map stats say 12 GB, AQE merges the 400 planned partitions to about 12288 / 128 ≈ 96 — the same answer, computed for you. (Leave the advisory at its 64 MB default and you'd land near 192 instead; remember to set parallelismFirst=false so the target is honoured.)

# the hands-off version: high start + advisory target, AQE finds ~96
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")
spark.conf.set("spark.sql.adaptive.coalescePartitions.parallelismFirst", "false")
spark.conf.set("spark.sql.adaptive.advisoryPartitionSizeInBytes", "128m")
spark.conf.set("spark.sql.shuffle.partitions", 400)   # high upper bound; AQE coalesces to ~96
The takeaway Measure Shuffle Write (12288 MB), divide by the ~128 MB target (≈ 96), round to a multiple of cores (96 = 3×32), and set it per query — or hand the same job to AQE with a high start and a 128 MB advisory and get the same ~96. Either way the win is identical: tasks at the sweet spot, spill at zero, no idle tail, stage roughly 1.5× faster.

balanced = df.repartition(200, "cust_id")
# coalesce: cheap merge, no shuffle, decrease only (fewer output files after a filter)
compact = df.coalesce(10)

22. Disk Partitioning & Partition Pruning

Disk partitioning is a physical layout decision made at write time. When you call df.write.partitionBy("order_date"), Spark does not write one flat pile of files — it splits the rows by the distinct values of order_date and writes each group into its own directory named order_date=<value>. The column value becomes part of the path, so it isn't even stored inside the Parquet files any more; it's encoded in the folder name.

from pyspark.sql import functions as F

# write the table partitioned by order_date
(orders
    .write
    .mode("overwrite")
    .partitionBy("order_date")
    .parquet("/warehouse/orders"))

# resulting layout on disk:
#   /warehouse/orders/order_date=2026-07-15/part-00000.snappy.parquet
#   /warehouse/orders/order_date=2026-07-16/part-00000.snappy.parquet
Disk partitioning, in one line A physical arrangement of a table's files into one directory per value of the partition column(s) — chosen at write time with partitionBy. It does not change your data, only where the bytes live, so that later reads can touch fewer of them.
Not the same as shuffle partitions This is disk partitioning (the on-disk directory layout). It is unrelated to spark.sql.shuffle.partitions, which controls how many partitions a shuffle produces in memory during a job. Same word, different concept — keep them separate in your head.

How many files land in each directory

The layout is Hive-style: a directory named <col>=<value> per distinct value, and inside each leaf the actual part-*.parquet files. A subtlety that trips people up: the number of files written into each partition directory equals the number of in-memory (task) partitions that hold rows for that value at write time. If your DataFrame has 200 in-memory partitions and rows for order_date=2026-07-15 are scattered across 40 of them, you get 40 files in that one directory — potentially 40 tiny files. This is why you usually repartition on the partition column right before writing: it co-locates each value into one (or a few) task partitions, so each directory gets one healthy file instead of many slivers.

# repartition by the SAME column before writing to control file count
(orders
    .repartition("order_date")          # one in-memory partition per date
    .write
    .mode("overwrite")
    .partitionBy("order_date")          # one directory per date -> ~1 file each
    .parquet("/warehouse/orders"))
partitionBy vs repartition — don't confuse them partitionBy is a writer option shaping the on-disk directory tree. repartition reshuffles data in memory for parallelism. They are complementary: repartition("order_date") then partitionBy("order_date") gives you one tidy file per date directory.

The problem it solves — partition pruning

The point of laying data out this way is partition pruning: when a query filters on the partition column, Spark reads the directory names, works out which ones can satisfy the filter, and reads only those — skipping every other directory entirely. Less data touched means less I/O, fewer tasks, and a faster job. On a table partitioned by date, a single-day query can read one folder out of thousands.

# filtering on the partition column triggers pruning automatically
result = (spark.read.parquet("/warehouse/orders")
    .filter("order_date = '2026-07-15'"))

result.show()

# confirm pruning in the physical plan — look for PartitionFilters
result.explain()
# ... FileScan parquet ... PartitionFilters: [order_date = 2026-07-15],
#     ... will read 1 partition directory instead of all of them
How to verify it worked Run .explain() and look for PartitionFilters in the FileScan node (not just PushedFilters). If your filter shows up there, pruning is happening. The Spark UI's SQL tab also reports "number of partitions read" — a small number confirms the win.
Pruning only fires on the partition column A filter on a non-partition column (e.g. amount > 100) cannot prune directories — Spark must open every folder and filter rows inside. Pruning is the specific reward for filtering on the column you partitioned by, which is why choosing that column well matters so much.

Choosing a partition column

The right partition column is the single most important decision here, and it comes down to cardinality — how many distinct values the column has. You want low cardinality: something like order_date, region, or country, where each value groups a healthy chunk of data. Partitioning on a high-cardinality column like user_id is the classic mistake — it creates millions of directories, each holding a few tiny files. That is the small-files problem, and it makes everything slower.

# check cardinality BEFORE you partition
orders.select(F.countDistinct("order_date"),
              F.countDistinct("user_id")).show()
# order_date: ~1,000 distinct   |   user_id: ~50,000,000 distinct

# GOOD: low cardinality, groups data into big prunable chunks
orders.write.partitionBy("order_date").parquet("/warehouse/orders")

# BAD: high cardinality -> millions of tiny dirs (the small-files trap)
# orders.write.partitionBy("user_id").parquet("/warehouse/orders")

High-cardinality join key? Reach for bucketing instead When a column has too many distinct values to partition on but you still want data co-located — typically a high-cardinality join key like user_id — use bucketing. Rule of thumb: partition on low-cardinality columns you filter on; bucket on high-cardinality columns you join or aggregate on. 
# bucketing: fixed number of files by hash, no directory explosion

Not the same as shuffle partitions (again) Bucket count and partition directories are both physical-layout choices. Neither is spark.sql.shuffle.partitions, the in-memory shuffle width. Keep the three separate: directories (partitionBy) prune reads, buckets (bucketBy) remove join shuffles, shuffle partitions size intermediate in-memory stages.

Single vs multi-level partitioning

You can partition by more than one column. partitionBy("year") gives a single level of directories; partitionBy("year", "month") nests a second level inside the first, so each year folder contains twelve month folders. Multi-level partitioning enables finer pruning (filter on year and month and Spark descends only the branches it needs) — but each extra level multiplies the directory count, so it is easy to over-partition your way straight into the small-files trap.

# single-level: orders/year=2026/
orders.write.partitionBy("year").parquet("/warehouse/orders_y")

# multi-level: orders/year=2026/month=07/  (month nested inside year)
orders.write.partitionBy("year", "month").parquet("/warehouse/orders_ym")

# both levels prune when both are filtered:
(spark.read.parquet("/warehouse/orders_ym")
    .filter("year = 2026 AND month = 7")      # descends only year=2026/month=07/
    .show())
Over-partitioning is the risk Each level multiplies directory count. year × month × day × hour can explode into hundreds of thousands of leaf directories, each with a sliver of data — the small-files problem again, just reached from a different direction. Add a level only if you routinely filter on it and each leaf still holds ~1 GB. Order levels coarse-to-fine (year, then month) so the outer directories stay meaningful.
Practical ceilings Two field-tested guardrails: keep the total number of partition-value combinations comfortably under ~50K, and use at most two partition columns in most tables. Each extra column is another nested directory layer to list and track. If a finer grain (per-hour, per-user) is genuinely needed, prefer bucketing or a secondary sort inside the file over another partitionBy level.
Prefix pruning still helps Multi-level layouts prune on any prefix of the columns. On partitionBy("year","month"), a filter on year alone prunes to the matching year directories (then reads all their months); adding month narrows further. You do not have to filter on every level to benefit — but you must filter on the columns in outer-to-inner order for the outer prune to fire.

Worked example — partition a sales table by date and prune

Let's tie every idea together on one table. We build a year of synthetic orders, write it two ways — the right way partitioned by order_date, and the wrong way partitioned by order_id — then run the same query against each and read the plans. This is the whole guide in one runnable script.

Step 1 — build the data and write it partitioned by date

from pyspark.sql import SparkSession, functions as F

    .withColumn("order_id", F.col("id"))
    .withColumn("order_date",

The cost of the anti-pattern Partitioning by order_id creates one directory per row — 5,000,000 directories, each holding a single kilobyte-sized file. The driver must enumerate and track every one, file listing alone can take minutes to hours, and a read spawns a task per micro-file. You also gain nothing: nobody filters on a raw order_id across the whole table, so the pruning you paid so dearly for is never used. This is the small-files trap at its most extreme.
Takeaway Partition on the low-cardinality column your queries actually filter on (order_date), repartition on it first so each directory gets one healthy file, and verify pruning with explain() → PartitionFilters. Never partition on an identity/high-cardinality column — if you need co-location there, bucket instead. Do that, and a year of orders answers a one-day question by reading a single folder.

23. Bucketing: Joins, Aggregations & Bucket Pruning

Bucketing is a write-time layout. When you write a table with bucketBy(N, key), Spark hashes each row's key and pre-partitions the rows into a fixed number N of buckets, then persists that layout into the table on disk. The all-to-all shuffle that a join or group-by would normally do at query time is done once, here, at write time — a one-time cost that every later read gets to skip.

from pyspark.sql import functions as F

# bucketBy pre-hash-partitions rows into 64 buckets by cust_id, then persists it.
# This is a ONE-TIME write cost — the layout is baked into the table on disk.
(orders.write
    .mode("overwrite")
    .bucketBy(64, "cust_id")
    .sortBy("cust_id")
    .saveAsTable("orders_bkt"))
Bucketing, in one line bucketBy(N, key) writes the table already hash-partitioned into N buckets by key. Reads of that table know the layout — so operations keyed on key (join, group-by, filter) can skip the shuffle they would otherwise need.
Why "the one you're not doing" Bucketing needs foresight: you commit to a key and a bucket count when you write the table, before you know every query. That upfront thinking is why it's under-used — but for tables you join or aggregate repeatedly on a stable key, it's the single biggest shuffle win available.

How the write actually lays the data out

At write time Spark computes hash(cust_id) % N for every row (the same positive-hash-modulo Spark uses internally to decide a shuffle partition), and routes the row to that bucket. Each write task emits one file per bucket it produces, so the finished table is a set of files tagged part-XXXXX_0000k where k is the bucket id. A key can therefore land in exactly one bucket, and the table's effective partition count equals its bucket count — that invariant is what lets later reads reason about co-location without touching the data.

On by default — but only for metastore tables Reading and exploiting buckets is controlled by spark.sql.sources.bucketing.enabled, which defaults to true — you don't turn bucketing on, you just have to write the table so the layout is recorded. That recording only happens through saveAsTable (the bucket spec is stored as table metadata in the Hive/Glue metastore). A plain .parquet(path) / .save(path) write silently ignores bucketBy and gives you an ordinary, un-bucketed table.
Format & engine caveats Bucketing is a metastore/file-format feature, not a Delta feature — Delta Lake does not support Spark bucketing (it has its own layout tricks like Z-ordering and liquid clustering). Stick to Parquet/ORC managed tables when you want classic bucketing.

Bucketing for joins

This is the headline win. A normal sort-merge join of two big tables shuffles both sides so that equal keys meet on the same partition — two full Exchange stages across the network. But if both tables were bucketed on the same key with the same number of buckets, then bucket k of one table already holds exactly the keys that match bucket k of the other. Matching buckets are co-located, so the join runs bucket-against-bucket with no shuffle at all.

# Bucket BOTH tables the same way: same key, same number of buckets.
(orders.write.mode("overwrite")
    .bucketBy(64, "cust_id").sortBy("cust_id").saveAsTable("orders_bkt"))
(customers.write.mode("overwrite")
    .bucketBy(64, "cust_id").sortBy("cust_id").saveAsTable("customers_bkt"))

o = spark.table("orders_bkt")
c = spark.table("customers_bkt")

# Matching buckets are already co-located → the join needs NO shuffle.
joined = o.join(c, "cust_id")
joined.explain()   # look for SortMergeJoin with NO Exchange nodes below it
The two hard requirements The no-shuffle join only kicks in when both tables are bucketed on the same key and with the same bucket count. Differ on either and Spark falls back to a shuffle (see the join scenarios below). And it works only for tables created via saveAsTable — the layout must be recorded in the metastore.
Bucketing vs broadcast Broadcast join removes the shuffle when one side is small enough to fit in memory. Bucketing removes it when both sides are large — the case broadcast can't touch. They're complementary: broadcast for big-vs-small, bucketing for big-vs-big on a stable key.

Two things bucketing removes: the Exchange and the Sort

A sort-merge join needs its inputs both partitioned by the key (so equal keys share a task) and sorted by the key (so the merge can stream). Bucketing satisfies the first: matching buckets are already co-located, so the Exchange vanishes. Adding sortBy on the same key satisfies the second: each bucket file is written already sorted, so the plan's Sort nodes drop out too. Bucket without sorting and you still kill the shuffle, but Spark keeps a per-bucket Sort before the merge; bucket and sort and the scan feeds the SortMergeJoin directly.

o = spark.table("orders_bkt")
c = spark.table("customers_bkt")

# Both bucketed AND sortBy'd on cust_id → plan is Scan → SortMergeJoin,
# with NO Exchange and NO Sort in between.
o.join(c, "cust_id").explain(True)   # extended plan: confirm neither Exchange nor Sort appears
A bucketed table has exactly N partitions Because each bucket is read as one partition, a bucketed table's read parallelism is pinned to its bucket count — independent of spark.sql.shuffle.partitions (default 200). That is a feature for joins (both sides align at N tasks) but a reason to size N deliberately: N also caps how many cores the join can use.

Bucketing for aggregations

The same trick helps groupBy. A normal aggregation on a key shuffles rows so that every row of a group meets on one partition. But if the table is bucketed on that key, every row of a given key is already inside one bucket — so Spark can aggregate within each bucket independently and skip the shuffle. The group-by becomes a local per-bucket aggregate.

o = spark.table("orders_bkt")

# groupBy on the BUCKET KEY → each bucket is self-contained → no shuffle.
(o.groupBy("cust_id")
  .agg(F.sum("amount").alias("total"))
  .explain())        # no Exchange before the aggregate
Only when you group on the bucket key The no-shuffle aggregate applies only if the groupBy columns are the bucket key (or a prefix of it). Group on a different column and Spark still has to shuffle — bucketing on cust_id does nothing for a groupBy("product_id").

One aggregate instead of two

An ordinary group-by is a two-stage plan: a HashAggregate (partial) → Exchange → a second HashAggregate (final) that combines the partials after the shuffle. On a bucketed key every value of a group is already inside one bucket, so the partial and final stages collapse into a single per-bucket HashAggregate with no Exchange between them. The same holds for distinct and dropDuplicates on the bucket key.

o = spark.table("orders_bkt")

# distinct / dropDuplicates on the bucket key also skip the shuffle.
o.select("cust_id").distinct().explain()          # single HashAggregate, no Exchange

# Grouping on a NON-bucket key still shuffles — bucketing can't help here.
o.groupBy("product_id").count().explain()          # Exchange reappears

Filtering & bucket pruning

Bucketing helps reads even without a join. If you filter on the bucket key for a specific value, Spark can compute which bucket that value hashes to and read only that bucket file — skipping the other N−1 entirely. This is bucket pruning: less data scanned, faster query, for equality (and IN) filters on the bucket key.

o = spark.table("orders_bkt")

# A filter on the bucket key → Spark reads ONLY the bucket that value hashes to.
o.filter(F.col("cust_id") == 90210).explain()
# the scan reports a small number of buckets read, not all 64 (bucket pruning)

Read the plan: SelectedBucketsCount

Bucket pruning has been automatic since Spark 2.4 — there's no flag to flip beyond spark.sql.sources.bucketing.enabled. The proof is in the scan node of the physical plan: it reports SelectedBucketsCount: k out of N. An equality filter prunes to 1 out of 64; an IN (a, b, c) filter reads only the buckets those three values hash to, e.g. 3 out of 64.

o = spark.table("orders_bkt")

# IN prunes to the buckets the listed values hash to — not all 64.
o.filter(F.col("cust_id").isin(90210, 10001, 70808)).explain()
# scan shows e.g. "SelectedBucketsCount: 3 out of 64"
Range filters don't prune Pruning works for equality / IN on the bucket key, because a value maps to exactly one bucket. A range predicate (cust_id > 1000) can match any bucket, so no pruning happens — for range access you want partitioning or sorting, not bucketing.

Join scenarios with bucketed data

Bucketing pays off only when both sides line up. It's worth knowing the three cases you'll actually hit, because "I bucketed one table and nothing got faster" is the most common surprise. The shuffle disappears fully only in the first case.

# 1) both bucketed the same → no shuffle
spark.table("orders_bkt").join(spark.table("customers_bkt"), "cust_id")

# 2) only one side bucketed → the other side is shuffled to match (one-sided)
spark.table("orders_bkt").join(customers_raw, "cust_id")

# 3) mismatched bucket counts (64 vs 32) → Spark re-shuffles → benefit lost
spark.table("orders_bkt").join(spark.table("customers_bkt32"), "cust_id")
The mismatch traps Different key, different bucket count, or one side not bucketed at all — any of these can bring the shuffle back. Always confirm with .explain(): if you still see an Exchange, the bucketing isn't being used for that join.

Scenario 2, more precisely: the one-sided shuffle

When only one side is bucketed, Spark keeps the bucketed side in place and shuffles just the raw side to match — one Exchange instead of two. There is a subtlety: if the bucket count is smaller than spark.sql.shuffle.partitions (default 200), Spark may also re-partition the bucketed side to hit that partition target, so you can see an Exchange on the bucketed side too. It's still cheaper than a cold join, but "only one side bucketed" is not a guaranteed free lunch — bucket both sides when you can.

Scenario 3 rescued: coalesceBucketsInJoin (Spark 3.1+)

Since Spark 3.1.1 (SPARK-31350), if two bucketed tables share the bucket key but differ in count, Spark can coalesce the larger side down to the smaller count and still join without a shuffle — but only when the counts are in a clean factor relationship and the feature is switched on (it defaults to off).

# Off by default — opt in, and the counts must divide cleanly (64 / 32 = 2).
spark.conf.set("spark.sql.bucketing.coalesceBucketsInJoin.enabled", True)
# larger:smaller ratio must be ≤ maxBucketRatio (default 4)
spark.conf.set("spark.sql.bucketing.coalesceBucketsInJoin.maxBucketRatio", 4)

# Now 64-vs-32 can coalesce to 32 and join with no Exchange.
spark.table("orders_bkt").join(spark.table("customers_bkt32"), "cust_id").explain()

Choosing the number of buckets

The bucket count is the one real tuning knob. Aim for each bucket file to land in a healthy size range — roughly 128 MB to 1 GB — and to give enough buckets that the join has parallelism to fill your cluster. Too few buckets means huge files and little parallelism; too many means the small-files problem and wasted task overhead.

Why power-of-two, concretely Sticking to 32 / 64 / 128 keeps factor relationships clean, which is exactly what the coalesce path needs: 128-vs-64-vs-32 all divide evenly, so a big fact table and a smaller dimension can share the no-shuffle join even at different counts. Pick arbitrary counts (e.g. 100 vs 60) and neither an exact match nor coalescing is possible — you're back to a full shuffle.

The API is small — bucketBy + sortBy + saveAsTable — but there are sharp edges. The biggest is that bucketing is a metastore feature: it only works through saveAsTable, never a plain path write. And you should always verify, in the query plan, that the Exchange you expected to remove is actually gone.

When not to bucket

Bucketing is an amortised bet: you pay a full shuffle-and-sort at write time to skip shuffles on every later read. If there won't be many later reads, the bet loses.

  • One-off / exploratory jobs. Read once, join once, done — you'd pay the write shuffle and never recover it. Just let the query shuffle.
  • Small tables. If one side fits in memory, a broadcast join already removes the shuffle for free and needs no write-time layout. Bucketing earns its keep only when both sides are too big to broadcast.
  • Unstable or high-churn keys / changing join keys. The layout is baked in; if next quarter you join on a different key, the buckets are dead weight.
  • Delta tables. Not supported — reach for partitioning, Z-order, or liquid clustering instead.
from pyspark.sql.functions import broadcast

# Small dimension table? Don't bucket — broadcast it. No write cost, no shuffle.
big = spark.table("orders_bkt")
small = spark.table("dim_country")          # a few MB
big.join(broadcast(small), "country_code").explain()   # BroadcastHashJoin, no Exchange
Prove it in the UI The plan is the source of truth. In the SQL / DataFrame tab the DAG for a bucketed join goes straight from the scans into the join — with no Exchange node in between.

Worked example: bucketing two big tables for a repeated join

Put it all together. Two large tables — orders and customers — are joined on cust_id in a job that runs many times a day. We'll write both bucketed identically, join them, and read the plan. The payoff is visible in one line of explain(): the bucketed join has no Exchange, where the raw join has two Exchange stages and two Sorts.

Step 1 — the baseline: an un-bucketed join

# Raw tables, no bucketing. This is what most pipelines ship.
orders    = spark.read.parquet("/data/orders")       # ~40 GB
customers = spark.read.parquet("/data/customers")    # ~12 GB

orders.join(customers, "cust_id").explain()
# == Physical Plan ==
# *SortMergeJoin [cust_id], [cust_id], Inner
# :- *Sort [cust_id ASC], false, 0
# :  +- Exchange hashpartitioning(cust_id, 200)   ← shuffle #1
# :     +- *FileScan parquet [.../orders]
# +- *Sort [cust_id ASC], false, 0
#    +- Exchange hashpartitioning(cust_id, 200)    ← shuffle #2
#       +- *FileScan parquet [.../customers]
# Two Exchanges + two Sorts — paid on EVERY run.

Step 2 — write both tables bucketed the same way (one time)

# Same key, same count (64), same sortBy — the three things that must match.
(orders.write.mode("overwrite")
    .bucketBy(64, "cust_id").sortBy("cust_id")
    .saveAsTable("orders_bkt"))

(customers.write.mode("overwrite")
    .bucketBy(64, "cust_id").sortBy("cust_id")
    .saveAsTable("customers_bkt"))
# This write pays a full shuffle + sort ONCE. Every later read reuses the layout.

Step 3 — the same join, now shuffle-free

o = spark.table("orders_bkt")
c = spark.table("customers_bkt")

o.join(c, "cust_id").explain()

# :- *FileScan parquet [.../orders_bkt]  SelectedBucketsCount: 64 out of 64
# +- *FileScan parquet [.../customers_bkt] SelectedBucketsCount: 64 out of 64
# NO Exchange. NO Sort. Scans feed the join directly.
The takeaway The write in Step 2 is a one-time cost — a single shuffle-and-sort, run once. From then on every one of the job's daily runs skips two network shuffles and two sorts. Across many reads the amortised cost approaches zero, which is exactly when bucketing wins: a stable key, big-vs-big, joined again and again.

24. Dynamic Partition Pruning

Static partition pruning fires only on a literal filter against the partition column — but real analytics rarely filter the fact table directly; they filter a dimension table and join. Dynamic Partition Pruning (DPP) closes that gap: it evaluates the filtered dimension at runtime, broadcasts the surviving join-key values, and uses them to prune the fact scan — so a star-schema query reads only the fact partitions it actually needs. DPP is on by default since Spark 3.0.

Recap — static partition pruning

When a table is partitioned on disk by a column (say date), Spark stores each value in its own directory: date=2024-01-01/, date=2024-01-02/, and so on. If your query filters on that partition column directly, Spark can figure out at plan time exactly which directories it needs and read only those — skipping the rest entirely. That's static partition pruning, and it shows up in the physical plan as PartitionFilters.

# fact table written partitioned by date
fact.write.partitionBy("date").parquet("/warehouse/sales")

# a literal filter on the partition column → static pruning
spark.read.parquet("/warehouse/sales") \
     .filter(F.col("date") == "2024-01-03") \
     .explain()
# physical plan shows:  PartitionFilters: [isnotnull(date), (date = 2024-01-03)]
PartitionFilters vs. PushedFilters PartitionFilters operate on the directory layout — they decide which partition folders to even open. That is far cheaper than PushedFilters, which are applied to data inside files. Static pruning is the best case: Spark avoids the I/O completely.
The hard limit of static pruning Static pruning only fires on a literal (or a constant-foldable expression) directly against the partition column. The moment the value you want to filter by lives in another table — e.g. "the dates my dim_date table marks as holidays" — there is no literal for Spark to resolve at plan time. Static pruning has nothing to bind to, and that is exactly the gap Dynamic Partition Pruning closes.

The problem — filter on the dimension

Real analytics rarely filter the fact table directly. In a star schema you join a huge fact table (partitioned by date) to a small dimension table (a calendar, a store list) and put the filter on the dimension: "give me sales for the holiday week." Now the fact table has no literal on its partition column — so static pruning has nothing to work with, and without help Spark would scan the entire fact table and only then throw most of it away in the join.

# star-schema join: the filter is on the DIMENSION, not the fact partition column
fact  = spark.read.parquet("/warehouse/fact_sales")      # partitioned by date
dim   = spark.read.parquet("/warehouse/dim_date")        # small

result = (fact.join(dim, "date")
              .filter(dim.is_holiday == True))       # predicate on the dimension
# without DPP: fact is scanned in full, then the join drops the non-holiday rows
Why this is common — and expensive Star schemas are the default shape of a data warehouse. The fact table can be terabytes; the useful slice is often a few partitions. Scanning all of it to keep a sliver is exactly the pattern that makes "simple" dashboard queries slow.
Why filter pushdown doesn't save you A natural hope is that Spark "pushes" the dimension's is_holiday predicate down into the fact scan. It can't — that column doesn't exist on the fact table. The join key (date) does, but the surviving key values aren't known until the dimension has actually run. Ordinary predicate pushdown is a plan-time, single-table transform; it has no way to carry values across the join boundary.
The manual workaround people reach for Before DPP, teams hand-computed the key list — collect() the holiday dates, then fact.filter(col("date").isin(dates)) to force static pruning. It works, but it means an extra job, brittle glue code, and a driver-side collect. DPP does exactly this, automatically, inside the optimizer.

The mechanism

DPP bridges the gap at runtime. Spark first evaluates the small, filtered dimension, collects the set of join-key values that survive (the holiday dates), broadcasts that key set, and then uses it as a dynamic partition filter on the fact table — so the fact scan opens only the partitions whose key is in the set. The whole-table scan collapses to a handful of directories.

# DPP is on by default (Spark 3.0+); this is the same star-schema join
result = fact.join(F.broadcast(dim.filter(dim.is_holiday == True)), "date")

# at runtime Spark:
#   1. runs the filtered dimension  -> keys {2024-01-03, 2024-01-04}
#   2. broadcasts that key set
#   3. applies it as a dynamic PartitionFilter on the fact scan
#   -> only 2 of 5 date partitions are read
Why it must be dynamic The pruning value isn't known until the dimension has run — it depends on data, not on a literal in the query. So Spark defers the decision to runtime, plugging a subquery into the fact scan's partition filter. That is the difference between "static" (plan-time literal) and "dynamic" (runtime-computed) pruning.

Under the hood: two optimizer rules

DPP is not one step but two. During logical optimization, the PartitionPruning rule injects a DynamicPruningSubquery onto the pruning (fact) side — a duplicate of the filtered dimension wrapped in an IN expression on the join key. During physical planning, the PlanDynamicPruningFilters rule tries to reuse the broadcast that the join is already building, so the pruning subquery costs almost nothing extra. In the plan this appears as a SubqueryBroadcast node whose child is a ReusedExchange pointing at the join's BroadcastExchange.

Reuse is why the default is cheap With a broadcast hash join, the filtered dimension is already broadcast for the join itself. DPP reuses that same broadcast to build the partition filter — no second pass over the dimension. That is why reuseBroadcastOnly=true is a safe default: DPP only pays off when it can piggy-back on work the join already does. 
# the same join, viewed as the optimizer sees it (conceptual)
#   logical:  fact scan gains PartitionFilters:
#             date IN (SELECT date FROM dim WHERE is_holiday)   <- DynamicPruningSubquery
#   physical: that subquery is satisfied by REUSING the join's broadcast
#             SubqueryBroadcast -> ReusedExchange -> BroadcastExchange(dim)
result = fact.join(F.broadcast(dim.filter(dim.is_holiday == True)), "date")
result.explain("formatted")   # 'formatted' mode labels each node clearly

Caveats — when it doesn't work

DPP is powerful but conditional. It quietly does nothing unless several things line up: the fact table must be partitioned on the join key, the dimension side must be small enough to broadcast (so its keys can be pushed to every fact task cheaply), and the join must be an equi-join on that partition column. Miss any one and Spark falls back to a full fact scan — often without any error to tell you.

25. Why Distributed Joins Are Hard; Join Types vs Strategies

A join matches rows from two tables on a key. On one machine that's easy. Across a cluster it's the hard part of Spark, because the two rows you need to match almost never start on the same machine. To join them, Spark must first get every row with the same key onto the same executor — and that movement (a shuffle, or a broadcast) is what every join strategy is really about.

# a plain equi-join: Spark co-locates matching keys, then joins
df = l.join(r, "id")     # shuffles both sides by "id" first
Two ways to co-locate → two families of strategy Shuffle both sides so equal keys land in the same partition (Sort Merge Join, Shuffle Hash Join). Or broadcast the whole small side to every executor so no shuffle of the big side is needed (Broadcast Hash Join). Everything else — nested loop, cartesian — is a fallback for when there's no usable key.

Join Types vs Join Strategies

The single most useful distinction in the whole topic. A join type is which rows you want back (inner, left, full, semi…) — it's your business logic. A join strategy is how Spark executes it physically (broadcast hash, sort-merge…) — it's a performance decision the optimizer makes. They're independent axes: you pick the type; Spark picks the strategy.

Why the distinction matters when tuning When a join is slow, the fix is almost never to change the type (that changes your results) — it's to influence the strategy (broadcast a small side, fix skew, add a hint). Keep them separate in your head: correctness lives in the type, performance lives in the strategy.

Before the strategies, nail the types — which rows each one returns. Picture two tables L and R joined on a key: some keys are in both, some only in L, some only in R.

NULL keys never match — and they cost you A NULL join key matches nothing in an equi-join (NULL = NULL is unknown, not true). In an inner join those rows silently vanish; in an outer join they're kept but never paired. Worse, if one value (a default id, an empty string, or a huge block of NULLs) dominates the key, every one of those rows shuffles into the same partition — the classic skew trigger (covered in the data-skew part). Filter or bucket-out the nulls before the join when you can.

26. Join Strategies: Broadcast Hash, Sort-Merge, Shuffle Hash & Nested-Loop

Spark has five physical ways to run a join. Four cover equi-joins (an a = b key); one pair is the fallback for everything else. This section explains each strategy once, with its mechanics: how it moves data, when Spark picks it, and how to read and force it. They all return the same rows for a given type — only the network and CPU cost differ.

The Shuffle Mechanic: Hash Partitioning

Both shuffle-based strategies (Sort Merge and Shuffle Hash) rest on one idea: a shuffle decides each row's destination partition with partition = hash(key) mod numPartitions. That's what makes joins and groupBys work — equal keys always land in the same partition, so they end up co-located on the same executor. (It's also exactly what concentrates a hot key into one oversized partition; the resulting skew and its fixes are covered in the data-skew part.)

# join / groupBy / repartition(col) all hash-partition by the key
df.repartition(4, "cust_id")                       # partition = hash(cust_id) % 4
Why you can't just "add partitions" Raising spark.sql.shuffle.partitions doesn't rebalance a single hot key: no matter how many buckets you create, every row of key 7 still hashes to exactly one of them. That's why the shuffle count tunes healthy joins but never cures skew — the skew fixes (change the key, broadcast, or let AQE split the partition after the hash) live in the data-skew part.

Broadcast Hash Join

The fastest strategy, and the one to aim for whenever one side is small. Spark collects the small table to the driver, ships a full copy to every executor, and each executor builds a hash table from it and probes it with its slice of the big table. The big table never moves — no shuffle at all, and because the big side is never hash-partitioned by the join key, a hot key can't concentrate: no shuffle, no skew.

Under the hood, step by step.

  1. Estimate. The optimizer compares each side's stats.sizeInBytes against spark.sql.autoBroadcastJoinThreshold. If a side fits, it plans a BroadcastHashJoinExec and marks the build side BuildLeft or BuildRight.
  2. Collect to driver. The build side is executed and collect()-ed back to the driver as a HashedRelation (a serialized hash map keyed on the join keys). This is the step that can OOM the driver.
  3. Broadcast out. The driver ships that one HashedRelation to every executor via a BroadcastExchange (torrent-style, peer-to-peer so the driver isn't a bottleneck), so each executor holds a full read-only copy.
  4. Probe locally. Each executor streams its local partitions of the big (probe) side, looks each row up in the hash map, and emits matches. The big side is never shuffled — that's the whole win.

When Spark uses it.

  • One side's estimated size is ≤ spark.sql.autoBroadcastJoinThreshold (default 10 MB = 10485760 bytes), or you gave a BROADCAST hint (which ignores the threshold).
  • The join type allows broadcasting that side. Spark's canBuildRight allows a right build for inner, left outer, left semi, left anti; canBuildLeft allows a left build for inner, right outer. So for a left outer join only the right side may be broadcast, and a full outer join can never be a BHJ. Broadcast the wrong side of an outer join and Spark warns, then silently falls back to Sort Merge Join.
  • If both sides fit and both are hinted, Spark broadcasts the smaller (by stats).
from pyspark.sql.functions import broadcast
big.join(broadcast(small), "id")                # force it
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", 50*1024*1024)  # raise to 50 MB
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", -1)  # disable auto-broadcast
When it backfires — the driver bottleneck The small side is first collected to the driver, then shipped out. Broadcasting something too big (hundreds of MB) can OOM the driver — often as Total size … is bigger than spark.driver.maxResultSize (default 1 g) — make the broadcast slow, or blow up executor memory holding N copies. There's also a hard ceiling: a broadcast relation can hold at most ~512M rows / 8 GB, above which the broadcast fails outright. Spark's size estimate is also often wrong for compressed Parquet or post-filter data — so a join you expect to broadcast may silently become a Sort Merge Join, or vice-versa. Broadcast only genuinely small, lookup-style tables.
Reading BuildLeft / BuildRight In the plan you'll see e.g. BroadcastHashJoin [id], [id], Inner, BuildRight — BuildRight means the right child was hashed and broadcast, so it's the small side. The BroadcastExchange node sits under that same side. If you cache one side to speed a repeated broadcast, cache the one named by Build….

Sort Merge Join

Spark's default for joining two large tables. Three phases: shuffle both sides by the join key (hash-partitioning, above) so equal keys share a partition, sort each partition by key, then merge the two sorted streams in a single linear pass. Robust and scalable to huge↔huge joins — the price is two shuffles and two sorts.

spark.conf.set("spark.sql.autoBroadcastJoinThreshold", -1)  # no broadcast
df = l.join(r, "id")     # two large sides -> Sort Merge Join
df.explain()               # confirm: SortMergeJoin over Exchange + Sort, both sides

Why sort, and why it's the default.

  • Why sort: a merge join walks both sorted sides once with two pointers — O(n) after sorting — instead of building an in-memory hash table. Sorting can also spill to disk gracefully (via UnsafeExternalSorter), so it survives partitions too big to fit in memory.
  • Why default: it's the most robust choice for large↔large joins — it never needs a whole side to fit in memory, so spark.sql.join.preferSortMergeJoin defaults to true. That's also why Shuffle Hash Join (next) is rarely chosen automatically.
  • The requirement it enforces: both sides must satisfy HashClusteredDistribution on the keys (hence the Exchange hashpartitioning) and arrive sorted (hence the Sort on each side). Keys must be orderable — almost always true; when they aren't (e.g. some complex types) Spark can't use SMJ at all and reaches for Shuffle Hash Join instead.
The three costs, and what each depends on Shuffle — network + disk write of both sides, sized by the data volume and spark.sql.shuffle.partitions (default 200, produced regardless of data size). Sort — CPU, and disk if a partition can't fit memory (spill). Merge — nearly free, a single linear pass. So an SMJ that's slow is almost always paying in the shuffle (too much data, or skew) or the sort (partitions too big) — not the merge. That's what to attack.
Making an SMJ faster You can't make the shuffle free, but you can shrink it: filter both sides before the join, right-size spark.sql.shuffle.partitions so sorts don't spill, bucket the tables if you join them repeatedly (see the bucketing part — it removes the shuffle), and watch for skew on the join key (see the data-skew part) — skew turns a healthy SMJ into a job with one endless straggler, because every phase is proportional to the biggest partition.

Shuffle Hash Join

The hash-based cousin of sort-merge. It still shuffles both sides by key, but instead of sorting, each partition builds an in-memory hash table from the smaller side and probes it with the larger. No sort — so it can be faster than SMJ when the build side fits in memory. It's rarely picked automatically, because preferSortMergeJoin defaults to true.

# SHJ is rarely automatic (preferSortMergeJoin=true); force it with a hint
big.join(small.hint("shuffle_hash"), "id")   # build hash from small side, no sort
The exact conditions Spark requires Spark auto-selects Shuffle Hash Join only when all hold: (1) spark.sql.join.preferSortMergeJoin=false (or you gave a SHUFFLE_HASH hint); (2) the build side is small enough to build a local hash map (roughly, the whole build side < autoBroadcastJoinThreshold × shuffle.partitions — i.e. each partition averages under the broadcast threshold); and (3) the build side is much smaller than the probe side (about 3×). Because condition 1 is false by default, you'll usually only see SHJ when you force it with a hint.

The conditions above are two tiny functions in JoinSelection (in SparkStrategies.scala). Reading them removes all the hand-waving:

# canBuildLocalHashMap(plan): does ONE partition of the build side fit a hash map?
plan.stats.sizeInBytes < autoBroadcastJoinThreshold * numShufflePartitions

# muchSmaller(a, b): is the build side (a) at least 3x smaller than the probe (b)?
a.stats.sizeInBytes * 3 <= b.stats.sizeInBytes

The one time SHJ appears without you asking There's a hidden || !RowOrdering.isOrderable(leftKeys) branch: if the join keys aren't sortable, Sort Merge Join is impossible, so Spark picks Shuffle Hash Join even with preferSortMergeJoin=true. Also note SHJ gained full-outer support in Spark 3.1; before that, full outer had to be SMJ or Broadcast Nested Loop. AQE can likewise convert a planned SMJ to SHJ when every post-shuffle partition is under spark.sql.adaptive.maxShuffledHashJoinLocalMapThreshold (default 0, i.e. off).

Broadcast Nested Loop & Cartesian

The fallbacks for when there's no usable equality key — non-equi joins (a.x < b.y) and cross joins. With no key to shuffle or hash on, Spark compares every row of one side with every row of the other. It works, but it's O(n×m) — a genuine red flag if you see it where you expected a normal join.

# no equality key -> Broadcast Nested Loop / Cartesian
l.join(r, l.x < r.y)                       # non-equi join -> BroadcastNestedLoopJoin
l.crossJoin(r)                          # explicit cartesian product
l.join(r.hint("shuffle_replicate_nl"))     # force replicated nested loop (partitioned cross)

BNLJ vs Cartesian — the two fallbacks.

  • BroadcastNestedLoopJoin broadcasts one side (whichever fits, subject to the same join-type/side rules) and, on each executor, loops every local probe row against the whole broadcast side. It's used for non-equi conditions and for cross joins where one side is small. The plan shows a BuildLeft/BuildRight just like BHJ.
  • CartesianProduct is the pure cross product with no broadcast — used when neither side is small enough. It's O(n×m) rows and effort. The SHUFFLE_REPLICATE_NL hint requests this partition-replicated variant.
  • Neither can appear for an ordinary equi-join. Seeing one means Spark found no usable equality key.
The accidental Cartesian A missing or mistyped join condition silently becomes a cross join, which can try to emit billions of rows. If a "normal" join is inexplicably slow or its output explodes, open the plan and look for BroadcastNestedLoopJoin or CartesianProduct — it almost always means your join keys aren't actually matching (wrong column, type mismatch, or an OR in the condition that defeats the equi-join).

27. How Spark Chooses a Strategy; Join Hints; AQE at Runtime

Spark's JoinSelection rule decides the strategy by walking a fixed checklist and taking the first option that fits — first honouring any hint, then splitting on whether it's an equi-join, then on sizes and config. Learn this path, then learn the two ways to steer it: an explicit hint when the size estimate is wrong, and Adaptive Query Execution, which re-runs the choice at runtime on real data.

The order in words (1) A hint wins if given. (2) Non-equi joins can only be Broadcast Nested Loop or Cartesian. (3) For equi-joins: broadcast if a side is small enough → BHJ; else, only if preferSortMergeJoin=false and the build side fits & is ~3× smaller → SHJ; else → SMJ. Keys must be sortable for SMJ (almost always true).

For an equi-join, JoinSelection effectively runs this cascade — the first truthy branch wins. This is a faithful paraphrase of SparkStrategies.scala:

# 1. hint says broadcast, OR a side is under the threshold
if broadcast_hinted or canBroadcast(side):        # -> BroadcastHashJoin
# 2. hint says shuffle_hash, OR SMJ not preferred and build fits & 3x smaller
elif shuffle_hash_hinted or (
        not preferSortMergeJoin
        and canBuildLocalHashMap(build)
        and muchSmaller(build, probe)):        # -> ShuffledHashJoin
# 3. keys are orderable (the robust default)
elif RowOrdering.isOrderable(keys):            # -> SortMergeJoin
# 4. keys not orderable -> ShuffledHashJoin (SMJ impossible)
else:                                          # -> ShuffledHashJoin
Estimates, not truths Every size test above runs on the optimizer's estimated stats.sizeInBytes, computed before a single row is read. For compressed Parquet, deeply filtered scans, or after a UDF, that estimate is routinely off by an order of magnitude — which is exactly why AQE (below) re-runs the choice with real post-shuffle sizes, and why an explicit hint (next) is sometimes the only reliable fix.

A hint overrides the size test — but not the laws of physics A BROADCAST hint tells Spark to broadcast regardless of autoBroadcastJoinThreshold — but Spark still won't broadcast something it physically can't (an unsupported join-type side, or a relation past the ~8 GB ceiling); it warns and falls back. If both sides carry conflicting strategy hints, the higher-priority one in the table wins. Separately, partitioning hints (COALESCE, REPARTITION, REPARTITION_BY_RANGE, REBALANCE) don't pick a join algorithm but reshape the shuffle that feeds it.
Use hints sparingly A hint hard-codes a decision that was meant to adapt to data. It's the right fix for a persistent bad estimate, but a hint that's correct today can hurt when the data grows (e.g. a BROADCAST on a table that's no longer small). Prefer letting AQE (next) adjust at runtime; reach for hints when you've confirmed in the plan that the optimizer is repeatedly choosing wrong.

28. Reading Joins in the Query Plan

Run df.explain() and the physical plan tells you exactly which strategy you got. Each join node has a signature — and the operators beneath it (a BroadcastExchange, or an Exchange + Sort on each side) confirm how the data was moved. This is how you verify a join is doing what you expect.

df = big.join(small, "id")
df.explain()     # tree form; look for the join node + exchanges
# == Physical Plan ==
# *(2) BroadcastHashJoin [id], [id], Inner, BuildRight   <- broadcast! good
# :- *(2) Filter ...                                     <- big side, no exchange
# +- BroadcastExchange HashedRelationBroadcastMode ...   <- small side broadcast

explain() Modes Worth Knowing

df.explain("cost")           # adds sizeInBytes/rowCount estimates -> see WHY a side did/didn't broadcast
df.explain(True)             # parsed -> analyzed -> optimized -> physical (all four)

The "cost" mode is the one that settles broadcast arguments: it prints the estimated sizeInBytes the optimizer used, so you can see at a glance whether a side was judged over or under the 10 MB threshold.

What to check every time (1) Which join node, and which side is Build…? (2) Is there a BroadcastExchange where you expected one (or an unwanted SortMergeJoin on a small table)? (3) With AQE on, read the final plan in the Spark UI's SQL tab — AQE may have swapped the strategy at runtime, so explain()'s static plan can differ from what actually ran. Look for AdaptiveSparkPlan isFinalPlan=true and AQEShuffleRead nodes as the giveaway that AQE rewrote things.

Worked Example: Force & Compare Join Strategies

Everything above, on one small script. We join a big table to a small one, then read the physical plan three times — letting Spark auto-broadcast, then forcing Sort Merge Join, then forcing Shuffle Hash Join — and watch the Exchange, Sort, and BroadcastExchange nodes appear and disappear. This is the loop you'll run for real whenever a join is slow.

1 · Setup — a big fact table and a small dimension.

from pyspark.sql import SparkSession, functions as F

spark = SparkSession.builder.appName("join-strategies").getOrCreate()

# big: 20M rows of orders keyed by cust_id in [0, 100k)
big = (spark.range(0, 20_000_000)
         .withColumn("cust_id", (F.rand(1)*100_000).cast("int"))
         .withColumn("amount", (F.rand(2)*500).cast("int")))

# small: 100k customer dimension (a few MB) -> naturally broadcastable
small = (spark.range(0, 100_000)
           .withColumnRenamed("id", "cust_id")
           .withColumn("segment", F.concat(F.lit("seg_"), (F.col("cust_id")%5))))

2 · Auto — Spark broadcasts the small side (BHJ).

auto = big.join(small, "cust_id")
auto.explain()
# == Physical Plan ==
# *(2) Project [...]
# +- *(2) BroadcastHashJoin [cust_id], [cust_id], Inner, BuildRight   <- small side hashed
#    :- *(2) Filter isnotnull(cust_id)          <- big side: NO Exchange (never shuffled)
#    :  +- Scan ...
#    +- BroadcastExchange HashedRelationBroadcastMode(...)   <- the ONE network move
#       +- Scan small ...

Observe: one BroadcastExchange on the small side, and no Exchange under the big side. The 20M-row table stays put. This is the plan you want.

3 · Force Sort Merge Join — disable auto-broadcast.

spark.conf.set("spark.sql.autoBroadcastJoinThreshold", -1)   # nothing may auto-broadcast
smj = big.join(small, "cust_id")
smj.explain()
# +- *(5) SortMergeJoin [cust_id], [cust_id], Inner
#    :- *(2) Sort [cust_id ASC], false, 0             <- sort, left side
#    :  +- Exchange hashpartitioning(cust_id, 200)    <- shuffle, left side
#    +- *(4) Sort [cust_id ASC], false, 0             <- sort, right side
#       +- Exchange hashpartitioning(cust_id, 200)    <- shuffle, right side

Observe: the BroadcastExchange is gone; now both sides carry an Exchange (a full shuffle) and a Sort. That's two network shuffles plus two sorts for a join that needed neither — a concrete picture of what over-eagerly disabling broadcast costs you.

4 · Force Shuffle Hash Join — a hint.

# threshold still -1; add the hint so a hash join is legal without broadcast
shj = big.join(small.hint("shuffle_hash"), "cust_id")
shj.explain()
# +- ShuffledHashJoin [cust_id], [cust_id], Inner, BuildRight
#    :- Exchange hashpartitioning(cust_id, 200)   <- shuffle, left
#    +- Exchange hashpartitioning(cust_id, 200)   <- shuffle, right  (NO Sort under it)

Observe: both sides still shuffle (two Exchange nodes), but there is no Sort — SHJ builds a hash map from the small (build) side instead. Compared with SMJ it saves the sort; compared with BHJ it still pays two shuffles.

30. What Data Skew Is, Why It's So Bad & When It Happens

Data skew is when your data is spread unevenly across partitions. Spark's whole model assumes work divides roughly equally into tasks — but if one key is far more common than the rest, then after a shuffle all of its rows pile into a single partition, and the one task handling it runs long after every other task has finished. That straggler is the slowdown.

# a shuffle on a skewed key: every 'hot' cust_id row lands on one partition
transactions.groupBy("cust_id").count().orderBy(F.desc("count")).show()
# the top cust_id dwarfs the rest — that one giant group is the straggler
Skew, in one line A stage finishes only when its last task finishes. Skew makes one task enormous, so 199 tasks finish in a second and the 200th runs for minutes — you paid for a big cluster but are effectively using one core.

Why skew is so bad

Skew defeats the whole point of Spark: parallelism. Because a stage ends only when its last task finishes, one giant task makes the rest of the cluster wait — you provisioned dozens of cores and are effectively using one. And the fat partition is often too big to fit in memory, so it spills or OOMs, making the straggler even slower.

Stragglers

The stage waits for the one huge task; wall-clock time is set by the worst partition, not the average.

Spill & OOM

The fat partition may not fit in execution memory → it spills to disk, or the executor runs out of memory and the task fails/retries.

Wasted cluster & cost

Idle cores are money burned; and downstream stages can't start until the straggler's stage completes, so the delay cascades.

The four ways skew hurts, unpacked

  • The straggler sets the wall clock. A stage's duration is max(task_time), never the average. Doubling the cluster doesn't help — the hot partition still lands on one core and takes the same time, so the stage is as slow as its single worst task.
  • Utilisation collapses to ~1/N. With N cores, once the fast tasks finish you are running on one core while N−1 sit idle (Fig. 2). An 8-core executor drops toward 1/8 ≈ 12% utilisation; a 200-core cluster toward 0.5%. You pay for all of it and use a sliver.
  • Spill, then OOM, on the fat partition. The hot partition often can't fit in execution memory. Spark first spills it to disk (slow — extra serialise/write/read), and if even that isn't enough the executor throws an OutOfMemoryError: the task fails, is retried, and can take down the executor — making the straggler far slower still. High Max spill and GC in the Summary Metrics (see Detecting Skew) are this happening.
  • The delay cascades. Downstream stages can't begin until this one's last task lands, so a single skewed stage stalls the whole DAG — and in a shared cluster it holds executors other jobs are queued for.
The counter-intuitive cost Because runtime is set by the worst partition and utilisation is ~1/N, a skewed job on a big cluster can finish slower than a balanced job on a small one — you scaled the cores but the work still funnels through a single task.

When does skew happen?

Skew comes from the data, not the code. Real-world key distributions are almost never uniform — a few values dominate and the rest form a long tail (a power law). Shuffle on such a key and you get skew. The usual culprits are worth memorising because you can often see them coming.

Common source of skewExample
A dominant categoryOne country / customer / product accounts for most rows (the power law above).
NULL keysMillions of rows with a missing join key all collapse into one NULL group.
Default / placeholder valuesA sentinel like -1, 0, "unknown", or "N/A" used for "no value" becomes a mega-key.
Low-cardinality keysGrouping by a boolean or a handful of statuses puts huge counts in each of a few partitions.
Time hotspotsOne date/hour (a launch day, Black Friday) holds far more events than the rest.

Reading the five culprits

Each row above skews for a slightly different reason, and knowing which one you have points straight at the fix:

  • Power-law dominance. The default shape of real data. A "celebrity" customer, a flagship product, or the home country produces a partition orders of magnitude larger than the tail. Nothing is broken — this is simply what shuffling on a natural key does.
  • NULL keys. The most-missed cause. Every row whose key is NULL hashes to the same partition, so millions of unrelated "missing" rows collapse into one giant group — and in a join they usually can't match anything anyway, so the work is pure waste.
  • Sentinels / placeholders. A stand-in for "no value" — -1, 0, "", "unknown", "N/A" — behaves exactly like a NULL that Spark can't recognise as null, silently becoming a mega-key.
  • Low cardinality. Grouping by a boolean or a handful of statuses gives Spark only a few possible partitions, so even "balanced" data piles enormous counts into each.
  • Time hotspots. Partitioning or grouping by day/hour concentrates a launch or sale spike onto one value — a skew that moves with the calendar.
Sentinels are NULLs in disguise NULL and sentinel keys are the two skew sources you can usually see coming from the schema alone — before running anything. If a join/group key is nullable or has a documented "unknown" code, assume it will be hot and plan to isolate those rows.

You can see the shape before you shuffle by profiling the key — cardinality plus the heaviest values:

print("distinct keys:", transactions.select("cust_id").distinct().count())

(transactions
   .groupBy("cust_id").count()
   .orderBy(F.desc("count"))
   .show(10, truncate=False))
# watch for NULL / -1 / 'unknown' at the top, or one value dwarfing the rest

Operations that cause skew

Skew only bites at a shuffle. A shuffle sends every row with the same key to the same partition — so if a key is huge, all its rows converge onto one task. Narrow operations (filter, select) can't cause skew; the wide, key-based ones can.

# wide, key-based ops that shuffle — the hot cust_id converges on one task
transactions.join(customers, "cust_id")                    # shuffle join on the key
transactions.groupBy("cust_id").agg(F.sum("amount"))    # shuffle by the grouping key

Narrow ops are safe select, filter, withColumn and other narrow transformations never trigger a shuffle, so they can't create skew (though a filter can reveal it by leaving mostly one key). Skew is a property of what you shuffle on — so the fix is always about the key or the shuffle, never the narrow steps.

Wide vs. narrow, and why it decides everything

The wide/narrow split is the whole story. A narrow transformation maps each input partition to output using only that partition's own rows — no data crosses the network, so the partition layout you started with is the one you keep. A wide transformation must first redistribute rows by key so that all rows sharing a key sit together; that redistribution is the shuffle, and it is the only place a hot key can pile up. This is why you never fix skew by rewriting a filter: the damage is done at the exchange, on the key you chose to shuffle by.

Two shuffles that behave differently A broadcast join ships the small side to every executor and does no shuffle — so it can never be skewed (this is why "broadcast the small side" is the cleanest fix). And df.write.partitionBy(col) is not a shuffle at all: it skews the output files on disk (one huge file per hot value), a different problem from the task-level skew this guide is about.
Scope note for the automatic fix Spark's AQE skew-join optimisation only rescues shuffle joins (sort-merge / shuffle-hash). A skewed groupBy, a skewed window, or a skewed full outer join falls outside it — those need salting or an isolate-and-broadcast pass by hand.

31. Detecting Skew in the Spark UI & Simulating It

Skew has an unmistakable fingerprint in the Spark UI: in a stage's Summary Metrics, the Max task is many times the Median. One glance at that table — or at the stage's Event Timeline — tells you whether a slow stage is skewed. And to test a fix you need skewed data on demand, which is a one-recipe job.

How to get there in three clicks

Open the Spark UI (http://<driver>:4040 while a job runs, or the History Server for a finished one) → Stages tab → click the slow stage → scroll to Summary Metrics for Completed Tasks. That percentile table is the single most useful skew diagnostic in Spark. When work is balanced, every percentile column reads about the same; under skew the columns fan out sharply, and the Max cell is the tell.

Two quick thresholds

You don't need to eyeball it. Two rules of thumb turn "that looks lopsided" into a decision:

  • The Databricks 50% rule: if a stage's Max task duration is more than 50% larger than its 75th percentile, suspect skew. A healthy stage keeps 75th and Max close together.
  • The 10× heuristic: when task durations (or shuffle-read sizes) vary by 10× or more between median and Max, the skew is severe — e.g. most tasks finish in 30 s but a few run 20 min.

Which metrics to read — and the two that confirm it

Duration and Shuffle Read Size are the primary signals (Fig. 5). Two more rows in the same table confirm the memory pressure a fat partition creates, so check them before you conclude:

Summary metricBalanced stageSkewed stage (the fingerprint)
DurationMax ≈ 75th ≈ MedianMax » Median (e.g. 0.4 s median vs 92 s max, ≈230×)
Shuffle Read SizeMax ≈ MedianMax » Median (e.g. 12 MB vs 3.1 GB) — one partition holds almost everything
Spill (Disk / Memory)≈ 0 across all tasksMax spill huge vs ~0 median (e.g. 4.7 GB vs 36 MB) — the fat partition overflowed memory
GC TimeSmall fraction of durationMax GC > 20% of Max duration — the straggler is managing memory, not working

Two levels down, the stage's per-task table (below the summary) lets you sort by Input Size / Records or Shuffle Read: one or two rows sitting at the top, far above the rest, are the tasks that own the hot key. That's the per-task input distribution behind the percentile summary.

Spill and GC are corroboration, not the diagnosis High Max spill or GC time never appears on its own — it rides along with a Max duration and Shuffle Read that already dwarf the median. Read them as a second opinion confirming the fat partition ran out of memory, not as a separate problem to tune away (bumping executor memory only lets the skew "spill gracefully" — it doesn't remove the straggler).

You can also put a number on the gap the UI is showing, straight from a DataFrame:

           F.max("count").alias("max"))

# max >> median  ->  the same fingerprint the Spark UI draws, as a ratio you can log

Simulating a skewed dataset

To test skew fixes you need skewed data on demand. The recipe is simple: build a dataset where one key dominates — a big block of rows all carrying the hot key, plus a small, evenly-spread remainder.

from pyspark.sql import functions as F

# 90% of rows carry the hot key 'US'; the rest spread over a few others
skewed = (spark.range(10_000_000)
    .withColumn("country",
        F.when(F.rand() < 0.9, F.lit("US"))
         .otherwise(F.element_at(
             F.array(F.lit("UK"), F.lit("FR"), F.lit("DE")),
             (F.rand()*3 + 1).cast("int"))))
    .withColumn("amount", (F.rand()*100).cast("double")))

# confirm the skew
skewed.groupBy("country").count().orderBy(F.desc("count")).show()
# country | count      →  US ~9,000,000, UK/FR/DE ~333k each
Why simulate A reproducible skewed dataset lets you run a groupBy or self-join on country, watch the straggler appear in the Spark UI (Figs 5–6), then apply a fix and measure the difference. You can't tune what you can't reproduce.

Two knobs worth having

Keep the recipe honest by controlling the two things that matter: the severity of the skew (the hot-key fraction) and whether you are testing a join (you need a second table to join against). Turn the 0.9 up toward 0.99 to make the straggler brutal, or down to 0.6 to test whether AQE's default thresholds even fire.

# a small dimension to join against — one row per country
countries = spark.createDataFrame(
    [("US", "United States"), ("UK", "United Kingdom"),
     ("FR", "France"), ("DE", "Germany")],
    ["country", "name"])

# a shuffle join on the skewed key reproduces a skewed *join* stage
# (disable broadcast so it actually shuffles and you can see the straggler)
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", -1)

joined.write.format("noop").mode("overwrite").save()   # force execution, no output
Force the work to actually run A lazy DataFrame does nothing until an action. Use .write.format("noop") (a sink that computes everything but writes nothing) or .count() so the shuffle really executes and shows up as a stage in the Spark UI — show() alone may only compute a few partitions.

32. Fixing Skew with AQE (Skew-Join Handling)

Catalyst plans a query before it runs, using size estimates that are often wrong. Adaptive Query Execution (introduced in Spark 3.0, on by default since 3.2) re-optimises the plan during the run, using the real statistics Spark observes after each shuffle. It makes three runtime moves — and the third, splitting a skewed partition, is the one that fixes skew with zero code.

# AQE is on by default in Spark 3.2+ — check or (re-)enable it explicitly
spark.conf.get("spark.sql.adaptive.enabled")        # 'true'
spark.conf.set("spark.sql.adaptive.enabled", True)
Why runtime beats estimate A shuffle produces exact statistics — the true size of every partition. AQE waits for those, then reshapes the rest of the plan. That's why it can fix problems the up-front optimizer can't even see, like a single partition that turns out 200× larger than the others.

How AQE re-optimises: execute → re-plan → execute

AQE isn't a one-shot rewrite. Spark cuts the plan at every shuffle (an Exchange) into query stages and runs the leaf stages first. As each stage materialises, Spark records its exact statistics, feeds them back into the optimizer, re-plans what's left, and launches the next stage — a loop that repeats to the end of the query. That's how it reacts to sizes no compile-time estimate could know.

EXPLAIN hides AQE df.explain() prints the initial plan — before any AQE rewrite. To see what actually ran, open the SQL tab in the Spark UI (it shows initial vs final plan) or inspect the DAG after the query completes. A join that reads as SortMergeJoin in EXPLAIN may run as BroadcastHashJoin.

The skew-join move itself

Here's the move that beats skew. After the shuffle, AQE looks at each partition; if one is far bigger than the rest, it splits that partition into several smaller ones and matches them against the corresponding partition on the other side of the join. No single task is a straggler any more — all with no code change.

The properties to set

AQE and its skew handling are on by default in Spark 3.2+, but it's worth knowing (and, on older clusters, setting) the switches. Skew-join splitting needs the first two; partition coalescing is a companion optimisation worth leaving on:

Both conditions must hold A partition is only split if it's both 5× the median and over 256 MB. That "and" avoids over-splitting on small jobs (where 5× a tiny median is still tiny). If AQE isn't splitting a partition you think is skewed, it may simply be under 256 MB — lower the threshold, or use a broadcast/salt instead.
Split and replicate Splitting the big side into N sub-partitions isn't enough on its own — each sub-partition still needs the matching rows from the other side. So AQE also replicates the corresponding partition of the other side to every split, turning one giant task into N balanced ones that each see the full set of matches.

Coalescing partitions — the companion optimisation

Skew splitting makes a few partitions smaller; coalescing solves the opposite problem — the fixed 200 shuffle partitions (spark.sql.shuffle.partitions) leaving hundreds of tiny post-shuffle tasks, each dominated by scheduling overhead. Put it together: the skewed transactions × customers join is slow because the sort-merge join shuffles the big side by cust_id. If customers is small, force a broadcast — the shuffle disappears, and with it the skew.

# or raise the auto-broadcast threshold so Spark chooses it itself

34. Fixing Skew with Salting: Joins & Aggregations

Skew has a single root cause: a hot key whose rows, after a shuffle, all land on one partition and one task — the straggler that defines your runtime. Salting attacks that directly. You append a random integer 0..N-1 to the key, turning one over-loaded value like 'US' into N synthetic sub-keys (US_0, US_1, …). Because the shuffle now spreads those sub-keys over N partitions, the one giant task becomes N balanced ones.

from pyspark.sql import functions as F

N = 16  # number of salt buckets — how many tasks the hot key spreads across

# append a random 0..N-1 to the key: 'US' -> 'US_0' ... 'US_15'
salted = df.withColumn("salted_key",
    F.concat_ws("_", F.col("country"), (F.rand() * N).cast("int")))
Salting, in one line Salting trades one impossibly-large partition for N manageable ones by making the shuffle key artificially more granular. The rows are identical; only their routing changes.
Choosing N N is a dial. Too small and each sub-partition is still large; too big and you create needless partitions (and, in joins, needless replication — at the extreme, a Cartesian blow-up). A useful starting heuristic: N ≈ rows(hot key) / avg rows per key — enough buckets to shrink the hot key's slice down to the size of an ordinary partition. Start around 10–20, then read the Spark UI and adjust.
Confirm skew before you salt Never salt on a hunch — salting adds code and, in joins, replicates data, so pay that cost only against a real straggler. Two cheap checks tell you almost everything: a key-frequency count (df.groupBy(key).count().orderBy(desc("count"))) and the task-duration spread in the Spark UI, where a skewed stage's max dwarfs its 75th percentile and the straggling task's shuffle read is many times the median.

Salting in joins — concept

A join has two sides, and that creates a subtlety. If you salt the big (skewed) side with a random 0..N-1, its US rows now carry keys US_0..US_2 — but the matching US row on the small side still has the bare key US. It would match nothing. The fix: the small side must be replicated (exploded) across all N salt values, so there's a US_0, US_1 and US_2 on that side too. Then you join on the composite key (id, salt) and every sub-key finds its partner.

The replication is the cost Exploding the small side multiplies its rows by N. That is fine when the "small" side really is small, but it is the price of salting a join — and the reason you salt the skewed key, not every key. Only the small side is replicated; the big side just gets one extra column. Concretely: a 1M-row dimension at N = 16 becomes 16M rows — still trivial — but the same factor on a 500M-row side would be a disaster. Keep the replicated side genuinely small, or salt only the hot keys so the blow-up applies to a handful of values, not the whole table.

Why random on one side but every-value on the other?

The asymmetry is the whole trick, so it's worth stating plainly. The big side picks one random bucket per row — that's what spreads its rows across N tasks. The small side can't guess which bucket each big row chose, so it must be present in all of them: one physical row becomes N rows, one per salt. Join on (key, salt) and correctness is guaranteed — every big row finds exactly the one small copy that shares its bucket, so the result set is identical to the unsalted join, just shuffled evenly.

NULL keys are their own hot key A flood of NULL join keys skews exactly like a hot value — they all hash to one partition. But NULL never matches anything in an inner join, so the cleanest fix is usually to filter it out before the join (df.filter(F.col("id").isNotNull())) rather than salt it. If an outer join needs the NULL rows preserved, split them off, process them separately, and union the result back — don't waste salt buckets on rows that can't join.

Salting in joins — code

In PySpark it is three moves: give the big side a random salt, explode the small side across all N salts, then join on both the original key and the salt. After the join you drop the salt column and you're back to the original schema — but the shuffle was balanced.

from pyspark.sql import functions as F

N = 16  # salt buckets — the skewed key spreads over this many tasks

# 1) BIG (skewed) side: attach a random bucket 0..N-1
big_s = big.withColumn("salt", (F.rand() * N).cast("int"))

# 2) SMALL side: REPLICATE every row across ALL N salt values
small_s = small.withColumn(
    "salt", F.explode(F.array([F.lit(i) for i in range(N)])))

# 3) join on the composite key (id, salt) — every sub-key still matches
result = (big_s.join(small_s, ["id", "salt"])
                .drop("salt"))
Only salt the hot keys (optional refinement) You don't have to salt every row. A common optimisation is to salt only the known hot keys and leave the rest of the big side with a fixed salt = 0, replicating the small side only for those hot keys. That keeps replication minimal while still splitting the stragglers. 
# salt only when the key is hot; everyone else gets bucket 0
hot = ["US"]
big_s = big.withColumn("salt",
    F.when(F.col("id").isin(hot), (F.rand() * N).cast("int"))
     .otherwise(F.lit(0)))

# small side: hot keys explode to 0..N-1, others stay at salt 0
small_s = small.withColumn("salt",
    F.when(F.col("id").isin(hot),
           F.explode(F.array([F.lit(i) for i in range(N)])))
     .otherwise(F.lit(0)))
Gotchas that bite in practice
  • Same N on both sides. The big side's random range and the small side's explode range must match exactly, or some buckets never meet and you silently drop rows.
  • Cast the salt to int. F.rand() returns a double in [0,1); (F.rand()*N).cast("int") floors it to 0..N-1. Skip the cast and you're joining on floats.
  • Drop the salt after the join so downstream code sees the original schema.
  • Don't over-salt. Salting non-hot keys just inflates the replicated side for no benefit — scope it to the keys that actually straggle.

Salting in aggregations — concept

Aggregations have only one dataset, so there is nothing to replicate — the shape of the fix is different. Instead you split the work into two stages. Stage 1 salts the key and computes a partial aggregate per (key, salt): this is the expensive shuffle, and salting makes it balanced. Stage 2 strips the salt and computes the final aggregate per key — a cheap shuffle, because stage 1 already collapsed each hot key down to just N partial rows.

Why two stages, not replication A join needs both sides to line up on the same sub-key, so the smaller side is copied. An aggregation reduces a single dataset, so instead of copying you split the reduction: aggregate partially per salt, then combine. This works for associative aggregates (sum, count, min, max); for avg carry sum and count separately and divide at the end.
Put numbers on it Say 'US' has 10M rows and you want sum(amount). With N = 5, stage 1 splits those rows into US_0..US_4 — roughly 2M rows each, run in parallel — producing five partial sums, say 120, 95, 110, 130, 145. Stage 2 strips the salt and adds the five partials: 600. Same answer as the naive groupBy, but the heavy shuffle ran on five balanced tasks instead of one 10M-row straggler, and stage 2 only ever sees five rows for US.
countDistinct does NOT survive salting Two-stage salting is only correct for aggregates that recombine from partials. countDistinct is not one of them: counting distinct values per (key, salt) and then summing those counts double-counts any value that landed in more than one salt bucket, inflating the answer. If you must de-duplicate a skewed column, salt on a hash of that column instead of a random number — e.g. F.crc32(F.col("device_id")) % N — so identical values always share a bucket and stay countable. Random salt is for additive measures only.

Salting in aggregations — code

The code mirrors the diagram exactly: add a salt column, aggregate by (key, salt), then aggregate again by key alone — combining the partials and dropping the salt.

from pyspark.sql import functions as F

N = 16

# STAGE 1 — salt the key, partial aggregate per (key, salt): the balanced shuffle
stage1 = (df
    .withColumn("salt", (F.rand() * N).cast("int"))
    .groupBy("country", "salt")
    .agg(F.sum("amount").alias("partial")))

# STAGE 2 — strip the salt, final aggregate per key: only N rows per key left
result = (stage1
    .groupBy("country")
    .agg(F.sum("partial").alias("total")))
Correct because sum is associative sum(a, b, c, d) == sum(sum(a, b), sum(c, d)) — so splitting the rows by salt, summing each group, then summing the groups gives the identical answer. That algebraic property is what makes the two-stage rewrite safe.

avg: carry sum and count, divide at the very end

Averages aren't associative — you can't average the partial averages. Carry the raw sum and count through both stages and divide only once, in stage 2.

# STAGE 1 — partial sum AND partial count per (key, salt)
stage1 = (df
    .withColumn("salt", (F.rand() * N).cast("int"))
    .groupBy("country", "salt")
    .agg(F.sum("amount").alias("s"),
         F.count("amount").alias("c")))

# STAGE 2 — combine, THEN divide once for the true mean
result = (stage1
    .groupBy("country")
    .agg((F.sum("s") / F.sum("c")).alias("avg_amount")))

Targeted variant: isolate the hot keys, salt only those, union back

If only one or two keys straggle, you don't need to two-stage the whole dataset. Split the frame in two: process the cold keys with a plain groupBy, salt only the hot ones, and union the results. This keeps the extra shuffle scoped to the rows that actually need it.

hot = ["US"]

# cold keys: ordinary single-stage groupBy (no skew here)
cold = (df.filter(~F.col("country").isin(hot))
          .groupBy("country").agg(F.sum("amount").alias("total")))

# hot keys: two-stage salted aggregation, then align the schema
hot_df = (df.filter(F.col("country").isin(hot))
            .withColumn("salt", (F.rand() * N).cast("int"))
            .groupBy("country", "salt").agg(F.sum("amount").alias("p"))
            .groupBy("country").agg(F.sum("p").alias("total")))

result = cold.unionByName(hot_df)

Worked example — salt a skewed join end to end

One concrete run ties everything together: a transactions table where 90% of rows share a single cust_id = 'HOT', joined to a small customers dimension. We'll watch the naive join produce one straggler, apply N = 16 salting to flatten it, then narrow the fix with targeted salting.

Step 0 — build the skewed data

Ninety percent of transactions carry the hot key; the rest scatter over 100k cold customers. The dimension is tiny — one row per customer — which is exactly what makes it safe to replicate later.

from pyspark.sql import functions as F

# 100M transactions — 90% land on the single hot customer 'HOT'
tx = (spark.range(0, 100_000_000)
        .withColumn("cust_id",
            F.when(F.rand() < 0.9, F.lit("HOT"))
             .otherwise(F.concat(F.lit("c"), (F.rand() * 100_000).cast("int"))))
        .withColumn("amount", (F.rand() * 100).cast("int")))

# small customer dimension — one row per id, incl. 'HOT'
customers = (spark.range(0, 100_000)
        .select(F.concat(F.lit("c"), F.col("id").cast("int")).alias("cust_id"),
                (F.col("id") % 5).alias("segment"))
        .union(spark.createDataFrame([("HOT", 0)], ["cust_id", "segment"])))

Step 1 — the naive join, and the straggler it creates

A plain sort-merge join shuffles both sides by cust_id. Every 'HOT' row — 90M of them — hashes to the same partition, so one task does virtually all the work while the rest idle.

# all 90M 'HOT' rows shuffle to ONE partition → one giant task
naive = tx.join(customers, "cust_id")
naive.groupBy("segment").count().show()   # correct result, painful runtime

Step 2 — salt the join with N = 16

Attach a random bucket to every transaction, explode the tiny dimension across all 16 buckets, then join on [cust_id, salt]. The dimension grows from ~100k to ~1.6M rows — still nothing — and the 90M 'HOT' rows now fan out over 16 tasks.

N = 16

# big side: one random bucket per transaction

# small side: replicate every customer across ALL 16 buckets
cust_s = customers.withColumn(
    "salt", F.explode(F.array([F.lit(i) for i in range(N)])))

# join on (cust_id, salt); drop salt to restore the original schema
salted = (tx_s.join(cust_s, ["cust_id", "salt"]).drop("salt"))
salted.groupBy("segment").count().show()   # identical result, balanced shuffle

Salting every transaction replicated the whole dimension 16×. But only 'HOT' was ever skewed. Targeted salting confines the random bucket — and the replication — to that one value; every cold customer stays at salt = 0 and is replicated only once.

Takeaway Salting didn't change a single output row — it changed where the rows went. By turning one 90M-row partition into 16 balanced ones, a straggler-bound 4-minute stage becomes a ~20-second one. The whole cost is a modest replication of a small dimension, and targeted salting shrinks even that. Salt narrowly, on confirmed hot keys, and verify the win in the Spark UI.

35. Which Fix, When: AQE vs Broadcast vs Salting

Three fixes, one simple rule of thumb: if a side is small, broadcast; if not, let AQE split the skew; if AQE isn't enough (or it's a skewed aggregation AQE can't touch), salt by hand. They're complementary, not competing — and it all starts with correctly identifying the skew you're fixing.

1. Broadcast the small side

If the other table is small, a broadcast join removes the shuffle entirely — no shuffle, no skew. The first thing to reach for when a side fits.

2. Rely on AQE

AQE (on by default in Spark 3.2+) detects a skewed join partition and splits it automatically — often fixes skew with zero code. Confirm spark.sql.adaptive.skewJoin.enabled=true.

3. Salt the hot key

Add a random suffix to the skewed key so its rows spread across many tasks (and replicate the matching rows in a join, or two-stage an aggregation). The manual fix when AQE isn't enough.

4. Handle NULLs / sentinels

If the hot key is NULL or a placeholder, filter or process those rows separately — they usually can't match in a join anyway.

36. Executor Memory Layout: Execution vs Storage, Off-Heap & Overhead

When a Spark job dies with java.lang.OutOfMemoryError, or crawls because it keeps spilling to disk, the root cause is almost never "not enough RAM." It's that the executor's heap is carved into fixed regions with fixed rules, and your data landed in the wrong one — or the region it needed was capped smaller than you thought. Understand the layout and both problems become predictable.

An executor is a single JVM process running on a worker node; it runs the tasks of your job and holds their data in memory. The number you set with spark.executor.memory is not a free-for-all pool — the JVM heap it requests is split into four purposes before your first row is read. Two failure modes follow directly:

Spill to disk

A task needs more execution memory than is available, so Spark writes intermediate data to local disk and reads it back. Correct, but slow — disk is orders of magnitude slower than RAM.

OutOfMemory / killed

A region genuinely cannot grow (or the container exceeds its overhead), so the task fails and retries, or the cluster manager kills the whole executor.

There is a second trap hiding behind the word "memory": the figure you type into spark.executor.memory is only the JVM heap. The cluster manager (YARN, Kubernetes, Mesos, standalone) actually reserves a larger container made of up to four independent budgets. Confuse the heap for the container and you will either over-commit the node or watch the manager kill executors that never printed an OutOfMemoryError at all — they were killed from the outside for exceeding the physical container limit.

The full container equation container = spark.executor.memory + spark.executor.memoryOverhead + spark.memory.offHeap.size + spark.executor.pyspark.memory. Only the first term is the heap that gets carved into Reserved / Unified / User. The other three live outside the heap and are added on top — overhead is always present (default max(384MB, 0.10×heap)), while off-heap and the PySpark budget are zero unless you set them.

A third symptom deserves a name of its own: GC pressure. Long garbage-collection pauses on a large heap don't just waste CPU — they delay the very spills that would have relieved memory, so a job that "should" have spilled safely instead trips an OOM because Spark couldn't free space fast enough. GC time is the earliest warning sign in the Spark UI, and it is the reason off-heap memory exists at all.

# Memory settings are START-UP time: set them when the JVM launches,
# not with spark.conf.set(...) after the SparkSession exists.
from pyspark.sql import SparkSession

spark = (SparkSession.builder
    .appName("memory-demo")
    .config("spark.executor.memory", "4g")          # on-heap size per executor
    .config("spark.executor.memoryOverhead", "512m")  # off-heap container slack
    .getOrCreate())
# equivalently: spark-submit --executor-memory 4g --conf spark.executor.memoryOverhead=512m
The mental model Every OOM or spill question reduces to two sub-questions: which region ran out (execution, storage, user, or the container overhead), and could it have borrowed from a neighbour? The rest of this section answers both.

The executor memory layout

Take the on-heap size (spark.executor.memory) and Spark carves it up in a fixed order. First it sets aside a small Reserved block. What's left is the usable memory, and that splits into Unified Memory (the fraction Spark manages for your data) and User Memory (everything else your code needs).

The four on-heap regions

  • Reserved Memory — a fixed 300 MB Spark keeps for its own internal objects. You never touch it; it's simply subtracted before anything else.
  • Unified Memory — spark.memory.fraction (default 0.6) of the usable memory. This is the pool Spark actively manages, shared between Execution and Storage.
  • User Memory — the remaining 0.4 of usable memory. Holds data structures your code creates (UDF state, user objects, RDD lineage metadata). Spark does not track it, so runaway user objects here cause OOMs the framework can't prevent.
  • Off-Heap & Overhead — live outside the heap entirely (covered below).

spark.conf.get("spark.memory.fraction")         # -> '0.6'  (Unified share of usable)

Two details about the Reserved block matter in practice. First, the 300 MB is a hard-coded constant (RESERVED_SYSTEM_MEMORY_BYTES in UnifiedMemoryManager), not a fraction — so its relative cost shrinks as the heap grows (7% of a 4 GB heap, under 1% of a 32 GB one). Second, Spark refuses to start an executor whose heap is smaller than 1.5 × reserved = 450 MB; ask for less and you get an IllegalArgumentException before any task runs. That is also why spark.executor.memory has a documented minimum of 450m.

User Memory is the region people forget. Spark neither tracks nor caps it, so a broadcasted Python dictionary, an over-large accumulator, or a UDF that materialises a whole partition into a list will grow it until the JVM throws — and no amount of raising spark.memory.fraction helps, because that steals from User Memory rather than feeding it. When your own objects are the problem, the fix is smaller objects or a bigger heap, not a bigger managed fraction.

The default heap is tiny spark.executor.memory defaults to just 1g. After Reserved that leaves ~724 MB usable, ~434 MB Unified, ~290 MB User — enough for a demo, not for real data. Almost every production job overrides it. Set it explicitly and size the node around the container, not the heap.

Execution vs Storage — the sliding boundary

Inside Unified Memory the boundary between Execution and Storage is not fixed — it slides. That's what "unified" means: since Spark 1.6 the two share one pool. But the sliding is asymmetric, and understanding the asymmetry is the whole point.

Execution memory

Short-lived, for computation: shuffle buffers, join hash tables, sort buffers, aggregation state. Freed as soon as a task finishes. Spark protects it aggressively.

Storage memory

Longer-lived, for cached/persisted DataFrames and broadcast variables. Nice to have, but evictable — a dropped cache block can just be recomputed or re-read.

The rules, precisely

  • Storage floor R = unified × spark.memory.storageFraction (default 0.5). Storage is guaranteed at least R — execution can evict cached blocks only down to this line.
  • Execution wins. If a task needs execution memory and storage has borrowed the space, storage blocks are evicted (spilled/recomputed) to give it back — because computation cannot be paused, but a cache can always be rebuilt.
  • Storage never evicts execution. If execution is using the whole pool, a new .cache() simply doesn't fit — the blocks aren't stored (or are stored to disk if the storage level allows), and no execution memory is taken.
# Raise the storage floor if you cache heavily and keep losing cached blocks;
# lower it if shuffle-heavy jobs are starved of execution memory.
spark = (SparkSession.builder
    .config("spark.memory.fraction", "0.6")          # unified share of usable
    .config("spark.memory.storageFraction", "0.5")   # storage floor within unified
    .getOrCreate())

# A cache() only competes for Storage; a groupBy/join competes for Execution.
df.cache().count()                 # fills Storage (evictable)
df.groupBy("k").count().collect()  # demands Execution (protected)
Why the asymmetry Execution memory is in use right now by a running computation — take it away and the task fails. Storage memory holds a cache, which is by definition reproducible. So Spark makes the reversible thing (storage) yield to the irreversible thing (execution). That single principle explains every eviction decision.

The borrowing states

Because the boundary slides, at any instant the pool is in one of a few states. Reading them straight tells you what will happen next when either side asks for more.

SituationWhat Spark does
Execution needs more, free space exists in MGrants it immediately — no eviction needed.
Execution needs more, Storage has borrowed above its floor REvicts the borrowed Storage blocks (spilled to disk or dropped/recomputed), down to but never below R, then grants the space.
Execution needs more, Storage is already at or below RCannot evict further. Execution spills its own data to disk instead.
Storage needs more, free space exists in MBorrows it — even above R — until Execution asks for it back.
Storage needs more, Execution holds the spaceDenied. The new blocks aren't cached in memory (dropped, or written to disk if the storage level permits). Execution is never evicted.

Spill is not a separate feature — it is what Execution does when it has reached the last row and still can't fit. A shuffle, sort, or aggregation builds an in-memory buffer inside Execution memory; when the buffer can't grow (all of M is claimed and Storage is already at R), Spark serialises the buffer to local disk, frees the memory, and continues. The job stays correct but pays for the write and the later read. The two knobs that reduce spill both act on Execution's share: give it more absolute room (bigger heap, off-heap, or a lower storageFraction so it can borrow more of M), or give it less to hold at once (more/smaller partitions, so each task's working set fits).

On-heap and off-heap are two separate unified pools When spark.memory.offHeap.enabled=true, Spark maintains the Execution↔Storage split twice — once for the on-heap region and once for the off-heap region — each with its own boundary and its own storageFraction floor. A task acquires from whichever pool its operator targets; the two do not borrow across the heap boundary.

Off-heap, overhead & PySpark memory

Everything so far lived inside the JVM heap — and therefore under the JVM's garbage collector. Off-heap memory is different: Spark's Tungsten engine can store data in a compact binary format in native memory outside the heap, sidestepping GC entirely. It's off by default.

# Off-heap is opt-in and needs BOTH flags — enabled AND a non-zero size.
spark = (SparkSession.builder
    .config("spark.memory.offHeap.enabled", "true")  # default: false
    .config("spark.memory.offHeap.size", "2g")       # default: 0 — MUST be set
    .getOrCreate())
# Off-heap adds to the container footprint too — count it in your node budget.
When to enable off-heap Reach for it when GC pauses dominate (very large heaps, long GC stalls in the Spark UI) or when you want a bigger managed pool without inflating the heap. It's not a free win: you now hand-size offHeap.size, and getting it wrong OOMs the container just as surely. Most jobs run fine on-heap; enable off-heap deliberately, after seeing GC pain.
Tungsten Spark's execution engine that stores data as compact binary in memory it manages directly (on- or off-heap), avoiding JVM object overhead and reducing GC pressure. When off-heap is enabled, Execution and Storage memory can live out there in that binary form.

Memory Overhead — the always-on off-heap budget

Off-heap Tungsten is opt-in, but every executor already carries a mandatory slice of native memory: the memory overhead. It is not part of the heap and is never carved into the 300 / 0.6 / 0.4 split — it is added on top for everything the JVM allocates outside its managed heap: thread stacks, network and shuffle buffers (Netty direct buffers), interned strings, memory-mapped files, JNI code, and — crucially for PySpark — the Python worker processes.

PySpark memory — the process outside the JVM

A PySpark executor runs two processes: the JVM that executes Spark's shuffles, joins, and aggregations, and one or more Python worker processes that run your Python UDFs, mapPartitions, pandas UDFs, and anything the JVM can't. They talk over a Py4J / socket bridge. That Python heap lives entirely outside the JVM — so a Python UDF that builds a huge list, or a pandas UDF that pulls a whole partition into a DataFrame, consumes memory the JVM figures never see. By default it is charged against overhead, which is why PySpark jobs so often get "container killed" without a Java OutOfMemoryError.

    .config("spark.memory.offHeap.enabled", "true")

Raise overhead, or raise the heap? They fix different failures. If the executor is killed by the cluster manager (YARN "container ... exceeded physical memory", K8s OOMKilled) with no Java stack trace, the off-heap budget overflowed — raise memoryOverhead (or pyspark.memory for Python-heavy jobs). If you see java.lang.OutOfMemoryError: Java heap space in the executor log, or heavy spill, the heap is short — raise spark.executor.memory (or enable off-heap). Raising the wrong one wastes RAM and leaves the crash in place.

37. Memory Calculations & Diagnosing Spill

The layout only pays off when you can turn it into numbers — and read those numbers backwards off the Spark UI when a stage crawls. This section works the arithmetic for a real executor to the megabyte, then uses the same figures to diagnose and fix the one failure mode that matters most for tuning: spill.

A worked example — the 4 GB executor

Set spark.executor.memory = 4g (that's 4096 MB) and follow the arithmetic step by step. Every figure below is exact given the defaults — no hand-waving.

  1. Heap = 4g = 4096 MB.
  2. Subtract Reserved 300 MB → usable = 3796 MB.
  3. Unified M = 3796 × 0.6 = 2277.6 MB.
  4. Storage floor R = M × 0.5 = 1138.8 MB. Execution = M − R = 1138.8 MB.
  5. User = 3796 × 0.4 = 1518.4 MB.
  6. Overhead = max(384, 0.10 × 4096) = 409.6 MB (added on top of the heap).
  7. Total container requested ≈ 4096 + 409.6 = 4505.6 MB (plus off-heap size, if enabled).
# Reproduce the arithmetic in plain Python — no Spark needed.
executor_mb   = 4 * 1024                              # 4096
reserved_mb   = 300
usable_mb     = executor_mb - reserved_mb                # 3796
unified_mb    = usable_mb * 0.6                       # 2277.6  (spark.memory.fraction)
storage_floor = unified_mb * 0.5                      # 1138.8  (spark.memory.storageFraction)
execution_mb  = unified_mb - storage_floor               # 1138.8
user_mb       = usable_mb * 0.4                       # 1518.4
overhead_mb   = max(384, 0.10 * executor_mb)          # 409.6
container_mb  = executor_mb + overhead_mb                # 4505.6
print(usable_mb, unified_mb, storage_floor, user_mb, overhead_mb, container_mb)
The container is bigger than you asked for You requested 4 GB but the cluster manager (YARN / Kubernetes) reserves ~4.4 GB per executor because of overhead. Forget this and you over-commit the node — the manager can't fit the executors you planned, or worse, kills them for exceeding the physical container limit. Always budget heap + overhead + off-heap.
Watch the units, and don't over-trust the decimals Spark treats g and m as binary (1 g = 1024 MB = 1024² bytes), so 4g is 4096 MB, not 4000. The internal computation is in bytes and truncates, so the actual Unified pool is floor((4096−300)×1024² × 0.6) bytes — the "2277.6 MB" above is that value rounded for readability. You never need the exact byte figure to tune; you need the proportions. Confirm the real split for your build by launching with --conf spark.log.level=INFO and reading the MemoryStore / UnifiedMemoryManager lines the executor prints at start-up.

What spill actually is

For tuning you only need to internalise one failure mode: spill. When a task can't fit its working data in execution memory, Spark writes it to local disk and reads it back — turning a fast in-memory operation into a slow disk-bound one.

Diagnosing a spill from the Spark UI

The job runs but a stage crawls. Open the Spark UI → Stages tab → the slow stage, and look at two columns in the task summary: Spill (Memory) and Spill (Disk). "Spill (Memory)" is the in-memory size of data that was paged out; "Spill (Disk)" is the compressed bytes actually written. Any nonzero Disk spill means Execution memory ran out — the buffer couldn't fit in M with Storage already at its floor, so it went to local disk.

# Reproduce a spill: a wide aggregation over many groups on a big range.
from pyspark.sql.functions import col, count, sum as _sum

df = (spark.range(0, 400_000_000)
        .withColumn("k", (col("id") % 200000).cast("int")))
(df.groupBy("k")
   .agg(count("*").alias("n"), _sum("id").alias("s"))
   .write.format("noop").mode("overwrite").save())   # noop sink = measure compute, skip I/O

# WHAT TO WATCH in the UI for this stage:
#   Spill (Memory) / Spill (Disk)  -> nonzero Disk = execution ran short
#   GC Time as a % of Task Time    -> high % = heap pressure feeding the spill
#   Shuffle Read size per task     -> large = partitions too big, cut them

The fix must add to Execution, because that is where the shortfall lives. Three levers, cheapest first:

  1. Shrink the working set. Raise spark.sql.shuffle.partitions so each task holds less at once — often the whole fix, and it costs no extra RAM. Removing skew belongs here too: it stops any single task from getting an outsized partition.
  2. Let Execution borrow more of M. Lower spark.memory.storageFraction if the job caches little, so Execution can push the boundary further before spilling.
  3. Add real capacity. Raise spark.executor.memory, or enable off-heap to enlarge the managed pool without inflating the GC-scanned heap.
# Targeted fix for execution-side spill (leave overhead alone — it wasn't the problem).
spark = (SparkSession.builder
    .config("spark.executor.memory", "8g")
    .config("spark.sql.shuffle.partitions", "800")   # 1: smaller tasks => less per-task footprint
    .config("spark.memory.storageFraction", "0.3")   # 2: give Execution more of M to borrow
    .config("spark.memory.offHeap.enabled", "true")   # 3: enlarge the pool off the GC heap
    .config("spark.memory.offHeap.size", "3g")
    .getOrCreate())
Before → after Before: 200 default shuffle partitions, storageFraction 0.5 → each reduce task must hold ~1/200th of the shuffle in an Execution baseline of only ~2.4 GB; the UI shows GB of Disk spill and GC time above 20% of task time. After: 800 partitions quarter each task's footprint, storageFraction 0.3 lifts Execution's borrow ceiling, and 3 GB of off-heap absorbs the rest → Disk spill drops to zero, GC time falls, wall-clock time drops with it. Container now budgets 8g + 0.8g overhead + 3g off-heap ≈ 11.8 GB — re-check it fits the node.
A little spill is fine Spill shows up in the Stages tab as Spill (memory) and Spill (disk). A little spill is normal and not worth chasing; large or constant spill is a real bottleneck. Work the levers above in order — more partitions first (free), then borrow room, then real capacity — and re-measure each time.

38. Executor & Core Sizing: Fat vs Thin, the Rules & Sizing a Cluster

A Spark cluster is a set of worker nodes. On each node, Spark launches one or more executors — JVM processes that actually run your tasks and hold data in memory. Each executor is given a slice of the node's cores (its task slots) and a slice of its RAM. The number of tasks that can run at the same instant is simply executors × cores-per-executor. How you carve a node into executors has a large effect on shuffle throughput, garbage collection, and memory safety — and the same hardware can be split many ways.

The three knobs

Executor tuning is almost entirely about three numbers: --num-executors (how many), --executor-cores (slots each), and --executor-memory (heap each). The spark-submit flags are aliases for spark.* properties — the same settings you pass to SparkSession.builder.config(...), put in spark-defaults.conf, or set with --conf. Know both spellings and their defaults, because the defaults are almost never what you want on a real cluster.

The sizing problem, and where the driver runs

A node has C cores and R GB of RAM. Spark won't slice it for you — you decide how many executors to carve out and how many cores/GB each one gets. Before carving, you must know where the driver lives, because in one mode it eats into the very cores and RAM you are dividing up. The --deploy-mode flag decides this.

cluster mode

The driver runs inside the cluster (on YARN it is the ApplicationMaster). It consumes a slot on some node, so you must subtract its cores and memory from what's available for executors. Set them with --driver-cores / --driver-memory. Best for production / scheduled jobs.

client mode

The driver runs where you launched spark-submit (an edge node or your laptop). It does not take an executor slot — only a lightweight ApplicationMaster does (YARN default 1024 MB + 1 core). Best for interactive shells and notebooks.

# the SAME node (16 cores / 64 GB) expressed three ways at submit time:
# fat
spark-submit --num-executors 1  --executor-cores 16 --executor-memory 60g app.py
# thin
spark-submit --num-executors 16 --executor-cores 1  --executor-memory 3g  app.py
# balanced (what the rules derive)
spark-submit --num-executors 3  --executor-cores 5  --executor-memory 19g app.py
Why the driver shows up in the arithmetic The − 1 we subtract from the executor total later is exactly this: in cluster mode the driver (or AM) claims roughly one executor's worth of cores and RAM. In client mode it's cheaper, but the safe habit is to leave room for it either way.

Fat vs Thin — the two extremes

A fat executor takes almost the whole node: few executors, many cores each — the extreme being one executor with all 16 cores and nearly all the RAM. The opposite is the thin executor: many executors, one core each. Each fails for opposite reasons; the optimum sits deliberately between them.

Fat — sharing

A broadcast variable or cached dataset is held once and reused by all cores — no duplication across executors, less network for broadcasts.

Fat — GC pauses

A very large heap makes JVM garbage collection long and stop-the-world; the whole executor freezes while it collects.

Fat — I/O contention

Too many cores hammering HDFS from one JVM causes contention — throughput per core drops (the origin of the ~5-core rule).

Fat — blast radius

One bad task (OOM, a poison partition) can crash the single executor — and you lose all its cached data and in-flight work at once.

Thin — isolation

A failing or memory-hungry task only takes down its own tiny executor; the blast radius is minimal and GC per JVM is trivial.

Thin — no sharing

Broadcasts and cached data must be copied into every executor — 16 copies on one node — wasting memory and network.

Thin — overhead × many

Each executor reserves off-heap memoryOverhead (≥ ~384 MB). With one core per executor you pay that fixed cost dozens of times.

Thin — lost benefit

A single core per JVM means no multi-threaded reuse of one broadcast/cache and more scheduler bookkeeping — the multi-core advantage is gone.

Why exactly ~5 cores? (the HDFS reason) The number isn't arbitrary. The HDFS client doesn't handle many concurrent threads well: as you pile more reader/writer threads into one JVM, per-thread throughput drops from lock contention and connection churn. Field testing (Cloudera) found that ~5 concurrent tasks per executor is about the ceiling for full HDFS read/write throughput — beyond that, extra cores mostly contend rather than help. Hence spark.executor.cores in the 4–6 range, with 5 as the default pick.
The GC ceiling & the overhead tax Garbage-collection pause time grows with heap size, so a single ~60 GB executor can freeze for seconds in stop-the-world collections while every task on it waits — keep a single executor's heap at or below ~64 GB; past that, split into more executors rather than growing one. At the thin end, every executor pays memoryOverhead = max(384 MB, 0.10 × heap) plus the fixed cost of one more JVM to launch and track; sixteen 1-core executors pay that reserve 16 times, three 5-core executors pay it three times. On PySpark it's worse — the overhead factor defaults to 0.40 on Kubernetes to feed the Python workers, so a wall of tiny executors bleeds memory fast.

The number one submit-time failure is asking for a container the cluster manager can't grant: heap + memoryOverhead must fit inside the RAM you actually left per executor. Always add them up first.

# 19 GB heap + max(384m, 0.10 * 19g) = 19 + ~1.9 = ~21 GB  -> fits 21 GB/exec
heap_gb       = 19
overhead_gb   = max(0.384, 0.10 * heap_gb)   # ~1.9 GB
container_gb = heap_gb + overhead_gb           # ~21 GB  (must be <= RAM/exec)
assert container_gb <= 21, "executor won't fit — shrink the heap"

This is the right default for shared clusters and bursty pipelines — and on most managed platforms (Databricks, EMR, Dataproc) it is the standard way to run.

def size_executors(cores, ram_gb, nodes, cores_per_exec=5):
    exec_per_node = (cores - 1) // cores_per_exec
    ram_per_exec  = (ram_gb - 1) / exec_per_node

    return total_exec, cores_per_exec, heap_gb

39. Why Cache? Lazy Evaluation, Lineage & the Cost of Recomputation

A Spark DataFrame is not a table sitting in memory — it is a recipe: the chain of transformations (its lineage) needed to produce it from the source. Every time you call an action, Spark cooks that recipe from scratch. If you reuse the same DataFrame across several actions, you pay for the full scan-and-transform each time. Caching cooks it once, keeps the result, and lets later actions eat from the fridge.

# a DataFrame reused by two actions is recomputed from its lineage each time
df = spark.read.parquet("/data/events").filter(F.col("amount") > 0)

df.count()   # action 1 -> scans + filters the WHOLE source
df.count()   # action 2 -> scans + filters it ALL OVER AGAIN
Cache, in one line A DataFrame is a plan, not a result. Actions execute the plan and then throw the result away. Caching keeps the result so that the second, third and hundredth reuse is nearly free.

Lazy evaluation & the lineage

To understand caching you first have to understand why Spark recomputes. Transformations (select, filter, join, withColumn…) are lazy: calling them builds nothing but a DAG of dependencies — the lineage. Only an action (count, show, write, collect) triggers real work, and it runs the entire DAG from the source. Nothing is remembered afterwards, so the next action starts over from scratch. That is exactly the recomputation caching is designed to short-circuit.

from pyspark.sql import functions as F

# transformations are lazy -- nothing runs yet, only a DAG is built
clean = (spark.read.parquet("/data/events")
              .filter(F.col("status") == "OK")
              .withColumn("day", F.to_date("ts")))

clean.count()   # action -> runs read + filter + withColumn
clean.show()    # action -> runs the ENTIRE DAG again from scratch
The trap Because transformations are lazy and results are discarded, a DataFrame built from an expensive chain (joins, aggregations, wide shuffles) is silently re-executed on every action. This is the single most common source of "why is my job doing the same work five times?" — and caching is the direct answer.

Where does recomputation start from?

Every action rebuilds a DataFrame from the source — unless it can reach a cached ancestor along the way. Once a node in the lineage is cached and materialised, later actions short-circuit there: they read the cached rows and run only the transformations downstream of the cache. In other words, a cache truncates the lineage. Recomputation always starts from the nearest materialised point: the last cache if there is one, otherwise the raw source.

base = spark.read.parquet("/data/events").filter(F.col("status") == "OK")
base.cache(); base.count()          # materialise base once

# these two DataFrames branch off the SAME cached base
by_country = base.groupBy("country").count()   # reads cache, then groupBy only
by_day     = base.groupBy("day").sum("amount")  # reads cache, then groupBy only

by_country.show()   # recompute starts at the cache, NOT at the parquet scan
by_day.show()       # same -- read + filter are never re-run
Auto-persist for shuffles Spark already persists some intermediate shuffle output on its own (e.g. the map-side files behind a groupBy or join) so a failed node doesn't force a full recompute. That is not a substitute for caching: it's per-shuffle, transient, and gone after the job. If you reuse a DataFrame, still cache it explicitly.

Caching vs no caching — reading the Storage tab

The difference is visible in the Spark UI. Without cache, each action re-submits the same stages — you see the scan and shuffle repeated in the SQL / Jobs tabs. With cache, the first action materialises the dataset and the Storage tab lights up: it lists the cached RDD/DataFrame with its size in memory, its fraction cached and its partition count. Later actions skip the recompute and read that green, cached block.

df = spark.read.parquet("/data/events").filter(F.col("amount") > 0)

df.cache()     # mark for caching -- still LAZY, nothing cached yet
df.count()     # first action -> computes AND materialises into memory
df.count()     # second action -> served from cache, no re-scan
# Spark UI -> Storage tab now lists this dataset (the green row above)
Cache is lazy too df.cache() only marks the DataFrame; the data is written to the cache during the first action that reads it. If you want to force materialisation eagerly, trigger a cheap action like df.count() right after cache() — then confirm it in the Storage tab.

The Storage tab is your ground truth for what is actually cached. Only datasets you explicitly cache()/persist() (plus a materialised action) appear here — if a row is missing, nothing is cached, no matter what your code says. Each row reports:

40. cache() vs persist(), Storage Levels, When to Cache & unpersist

cache() and persist() are the same mechanism with different ergonomics. cache() is a shorthand with a fixed default; persist(level) lets you choose exactly where and how the data is kept via a StorageLevel. For a DataFrame/Dataset, cache() equals persist(MEMORY_AND_DISK) — stored as deserialized objects, which is why modern Spark also spells this level MEMORY_AND_DISK_DESER (the default returned by a bare df.persist() in PySpark). For a low-level RDD, cache() equals persist(MEMORY_ONLY). The storage level trades three things against each other: memory footprint, CPU cost, and fault-tolerance.

from pyspark import StorageLevel

# cache() is shorthand -- for a DataFrame it means MEMORY_AND_DISK
df.cache()                                    # == df.persist(StorageLevel.MEMORY_AND_DISK)

# persist(level) gives explicit control (call on a fresh, un-persisted df)
df.persist(StorageLevel.MEMORY_ONLY)          # fastest; evicted if it does not fit
df.persist(StorageLevel.DISK_ONLY)            # survives memory pressure, slow reads
df.persist(StorageLevel.MEMORY_AND_DISK_2)    # replicated x2 for fault-tolerance

PySpark gotcha: SER levels are (mostly) moot In PySpark, cached objects are always serialized with Pickle, so choosing MEMORY_ONLY vs MEMORY_ONLY_SER makes little difference — the practical Python levels are MEMORY_ONLY, MEMORY_AND_DISK, DISK_ONLY and their _2/_3 variants. The _SER distinction matters most on the JVM (Scala/Java) side. For DataFrames, Spark manages an efficient columnar in-memory format under the hood regardless of API.
Which one to pick Start with the default (MEMORY_AND_DISK via cache()). Reach for a _SER level only when memory is the bottleneck, DISK_ONLY for huge datasets you can't hold in RAM, and _2 replication only when losing the cache would be genuinely expensive to rebuild.

When to cache — and when not to

Caching is not free. The cached data occupies the same executor memory your computations need, so an over-eager cache can starve execution — forcing spills, extra GC, even OOMs — and make the job slower. The rule is narrow: cache only data that is genuinely reused (multiple actions) or iterated (ML loops, iterative graph algorithms). Everything else should not be cached. And whatever you cache, you must unpersist() when you're done with it.

df.cache()
df.count()          # materialise once

# ... reuse df across several actions / an iterative algorithm ...
result_a = df.groupBy("country").count()
result_b = df.filter(F.col("amount") > 100).count()

df.unpersist()      # ALWAYS free it -- caching competes with execution memory

Cache when

The DataFrame is read by multiple actions, or reused inside a loop (ML training, iterative graph/PageRank), or is expensive to rebuild (wide joins, heavy aggregations).

Don't cache when

The data is read once, is trivial to recompute, or is so large it evicts everything else. A one-shot pipeline gains nothing from caching.

Always unpersist

unpersist() returns the memory to the pool. Spark evicts LRU-style on its own, but relying on that leaves less room for execution — free it explicitly when done.

Cache is not free Every cached partition sits in the same memory that joins, aggregations and sorts need. Cache indiscriminately and you trade recomputation for spills, GC pressure and OOMs — a net loss. Cache the reused/iterated data, nothing more, and unpersist() the moment you're finished. Cache the smallest reused result — filter/select first, and prefer caching a narrow projection over a wide one.
Defaults worth remembering DataFrame.cache() → MEMORY_AND_DISK (spills, so it won't just vanish under pressure). RDD.cache() → MEMORY_ONLY (no disk fallback — partitions that don't fit are simply recomputed). Know which API you're on before you assume where the data lives.

The unified memory pool & storageFraction

Since Spark 1.6, execution and storage share one unified memory region (governed by spark.memory.fraction, default 0.6 of usable heap). Within it, spark.memory.storageFraction (default 0.5) is not a hard split — it is a floor. When execution needs room it may borrow from storage and evict cached blocks, but only down to the storageFraction boundary; cache below that line is protected. The reverse is not symmetric: storage can never push execution out. That asymmetry is exactly why a cache can shrink mid-job (Fraction Cached drops below 100%) when a heavy join or sort demands memory.

# unified memory knobs (defaults shown)
spark.conf.set("spark.memory.fraction", "0.6")          # exec+storage share of heap
spark.conf.set("spark.memory.storageFraction", "0.5")   # protected cache floor

# release caches
df.unpersist()                 # drop ONE dataset, return its memory
spark.catalog.clearCache()     # drop ALL cached tables/DataFrames at once
print(df.storageLevel)         # inspect: StorageLevel(True, True, False, True, 1)
Clearing everything df.unpersist() releases a single dataset (add blocking=True to wait for the eviction to finish). spark.catalog.clearCache() wipes every cached DataFrame/table in the session — handy at the end of a notebook or between pipeline stages to hand all storage memory back to execution.

41. Inside Parquet: File Layout, Row Groups, Pruning & How an Executor Reads It

This is the lowest level of the read path: what is actually inside one part-*.parquet file, how Spark narrows a whole table down to the handful of files a query needs, and what a single task does when it finally opens one of those files. Every layer here is designed to do one thing — read less — and it all rests on Parquet being a columnar format with statistics baked into each file.

The anatomy of a Parquet file

Parquet is columnar — instead of storing row after row, it stores each column's values together. A file is a stack of row groups; each row group holds one column chunk per column; each column chunk is a series of pages; and a footer at the end carries the schema plus per-row-group statistics. This layout is the whole reason column pruning and data skipping are possible.

# write plain Parquet, then read its footer stats to see the internal structure
orders.write.mode("overwrite").parquet("/tmp/orders_pq")
import pyarrow.parquet as pq
meta = pq.ParquetFile("/tmp/orders_pq/part-00000-....snappy.parquet").metadata
print(meta.num_row_groups)                        # row groups in this file
print(meta.row_group(0).column(0).statistics)   # min / max / null_count
# tune the row-group size Spark writes (default 128 MB = 134217728 bytes):
spark.conf.set("spark.sql.parquet.block.size", 128 * 1024 * 1024)
Two free wins baked into the format Column pruning — a query touching 3 of 50 columns reads only those 3 column chunks, ignoring the rest of every file. Predicate pushdown / row-group skipping — the footer's min/max per row group lets Spark skip entire row groups whose range can't satisfy a filter. Row-based formats (CSV, JSON) can do neither, which is why Parquet is the default for analytics.

From query to file list: the pruning funnel

Before any of that per-file work happens, Spark first decides which files to open at all. When you query a Delta table, Spark doesn't read all of it — it narrows a large set of files down to the few it actually needs through a funnel of pruning steps, each driven by metadata from the transaction log, before any data page is read. The surviving files are then split into read tasks.

q = (spark.read.format("delta").load("s3://lake/orders/")
      .where("order_date = '2026-07-15' AND amount > 100"))
# how many files actually feed the scan after pruning?
print(len(q.inputFiles()))     # far fewer than the table's total file count
# the scan node shows PartitionFilters + PushedFilters + the bytes/files read
q.explain("formatted")
Make skipping work for you Data skipping is only effective if the data is physically clustered so each file's min/max range is narrow. If a filter column's values are scattered across every file, every file's range matches and nothing is skipped. This is why Z-Ordering / liquid clustering (co-locating similar values into the same files) and sensible partitioning matter so much: they tighten the per-file stats that power skipping.

How an executor reads one file

Now the read path in full. A task is handed a file split (a byte range of one or more files — the row groups to read). It reads the footer, uses the pushed-down filter to skip row groups that can't match, reads only the column chunks for the columns you selected, decompresses and decodes their pages, and hands the result up as vectorized columnar batches — blocks of column values the engine processes together, which is what makes Spark SQL fast.

# the read path is governed by these (all on / sensible by default):
spark.conf.set("spark.sql.parquet.filterPushdown", True)           # row-group skipping
spark.conf.set("spark.sql.parquet.enableVectorizedReader", True)   # columnar batches
spark.conf.set("spark.sql.parquet.columnarReaderBatchSize", 4096)   # rows per batch
# fewer columns + a pushable filter = fewer bytes read off disk
(spark.read.parquet("s3://lake/orders/")
    .select("amount", "country").where("amount > 100")
    .explain("formatted"))   # scan shows ReadSchema of 2 cols + PushedFilters
The whole chain, bottom to top A filter you write → becomes a PushedFilter in the plan → the Delta log skips files → the Parquet footer skips row groups → the executor decodes only the surviving column chunks → vectorized batches flow into your operators. Every layer is designed to read less — the first and cheapest rule of tuning.

42. Delta Lake on Top of Parquet

A Delta table isn't a new file format — the data is plain Parquet files. What Delta adds is a transaction log, the _delta_log/ directory, that records which files currently belong to the table and carries statistics about them. Spark reads that log first, so it knows exactly which Parquet files to read without listing the directory — and can skip most of them.

# a Delta table is Parquet + _delta_log — read it like any other source
sales = spark.read.format("delta").load("s3://lake/sales/")
# inspect the files & stats the log tracks (no directory listing needed)
spark.sql("DESCRIBE DETAIL delta.`s3://lake/sales/`").show()    # numFiles, sizeInBytes
spark.sql("DESCRIBE HISTORY delta.`s3://lake/sales/`").show()   # commits / versions
# time travel — read an older snapshot straight from the log
old = spark.read.format("delta").option("versionAsOf", 3).load("s3://lake/sales/")
Why this matters for performance Because the log already knows every file's partition values and min/max stats, Spark can decide which files to read from metadata alone — before opening a single data file. On a cloud object store, where listing directories is slow and expensive, this is a big win: the read is planned from one small log instead of thousands of LIST calls.

43. The Performance-Tuning Mindset & the Query Lifecycle

Spark tuning is not a bag of magic configs. It's a loop: measure → find the bottleneck → remove it → measure again. Ninety percent of the wins come from attacking three specific enemies — and the only way to know which one is hurting you is to read the Spark UI, not to guess. Before you can tune a job, though, you need one mental model of what it actually does, end to end — so this capstone opens with the cost ladder and the full query lifecycle, then hands you the playbook and a worked walkthrough.

Why Spark Jobs Are Slow: a Cost Hierarchy

Every operation in Spark has a wildly different cost. Reading a value already in memory is essentially free; moving data across the network is thousands of times more expensive. Tuning is mostly about pushing work up this ladder — turning shuffles into local work, and disk spills into in-memory work.

The three enemies (the "3 S's") Almost every slow Spark job is losing time to one of these — this whole guide is organised around finding and fixing them:
  • Shuffle — data moved across the network to regroup by key (joins, aggregations, repartition). Expensive; minimise it.
  • Spill — a task ran out of execution memory and had to write intermediate data to disk. Wastes I/O and GC.
  • Skew — data unevenly distributed across partitions, so one "straggler" task runs far longer than the rest.
Measure first — always Never change a config based on a hunch. Run the job, open the Spark UI, find the stage that dominates wall-clock time, and read why it's slow (shuffle bytes? spill? one long task?). Only then pick the matching fix from this guide. Tuning blind usually makes things worse.

The End-to-End Query Lifecycle

Now assemble everything into one picture: what happens, in order, from the moment you call an action to the moment bytes are read off disk and a result comes back. The driver does all the thinking (planning, scheduling); the executors do all the work (reading files, computing, shuffling). Storage — Parquet files under a Delta table — is where the data actually lives.

Where each tuning lever fits the lifecycle Step 2 is the Catalyst plan and AQE; steps 3–4 are the DAG and tasks; step 5 is file reading (columnar formats, partition pruning, DPP); step 7 is the shuffle you spend most tuning effort on (join strategy, skew, partition sizing). The deep-dive parts cover each technique — this part is about knowing which lever to reach for, and in what order.

44. The Tuning Playbook: A Decision Checklist

Tuning is a loop, not a one-shot. Measure in the Spark UI, diagnose the single biggest bottleneck, change one thing, then re-measure. Below is the loop, the three rules everything reduces to, a symptom→cause→fix lookup for the whole guide, and a one-page config reference.

The Three Rules Everything Reduces To

Before the lookup table, keep the shape of the whole thing in mind: nearly every tuning move is a variation on read less data, shuffle less data, and keep partitions the right size.

  • Read less data. Use a columnar format (Parquet) so Spark reads only the columns you select (column pruning) and skips row groups that can't match your filter (predicate pushdown). Partition data on disk by a common filter column (partitionBy("event_date")) so queries hit only the relevant directories (partition pruning). Let dynamic partition pruning (on by default, 3.0+) prune fact-table partitions at runtime from a dimension filter in a star-schema join.
  • Shuffle less data. Broadcast small tables in joins to eliminate the shuffle entirely; filter early so less data crosses the network; pre-aggregate rather than shuffling raw rows (groupByKey-style); bucketBy large tables you join repeatedly on the same key so they're pre-shuffled on write.
  • Right-size partitions. Aim for ~128–200 MB each — enough to keep every core busy, small enough to avoid spill. repartition(n) does a full shuffle to produce evenly-sized partitions (and can raise the count); coalesce(n) only decreases the count and avoids a full shuffle by merging adjacent partitions — cheaper, but can leave uneven partitions. Use coalesce to trim output files, repartition to fix skew or add parallelism. Cache only a dataset that is genuinely reused, and unpersist() when done.

45. Worked Example: Tuning a Slow Job End to End

Time to run the whole loop on one real-shaped job. First we look at what a well-shaped pipeline and its physical plan should look like — the target you are aiming for. Then we take a slow job that misses that shape, diagnose it in the Spark UI, and apply four fixes in order, measuring after each, until an 18m 40s job runs in about 2.5 minutes. Every change comes straight from the playbook.

First, the shape to aim for: read the plan

Every concept in this guide shows up in one small pipeline: read a big table, filter it, join a tiny lookup table by broadcast, aggregate by group, and write the result. Learn to read its physical plan and you can debug most Spark jobs.

from pyspark.sql import functions as F
from pyspark.sql.functions import broadcast

# 1) READ — a big fact table (lazy: just a FileScan node in the plan)
orders = spark.read.parquet("s3://lake/orders/")          # ~2 TB, 2000 files

# 2) READ — a small dimension table we intend to broadcast
customers = spark.read.parquet("s3://lake/customers/")    # ~4 MB (customer_id, country)

# 3) FILTER (narrow) → JOIN by broadcast (no shuffle) → GROUP BY (one shuffle)
result = (orders
    .filter(F.col("status") == "PAID")                    # narrow — pushed into the scan
    .join(broadcast(customers), "customer_id")            # broadcast → NO shuffle of orders
    .groupBy("country")                                   # WIDE → the one Exchange
    .agg(F.sum("amount").alias("revenue")))

# 4) WRITE — the single ACTION that triggers the whole job
(result.write
    .mode("overwrite")
    .parquet("s3://lake/gold/revenue_by_country/"))

result.explain(mode="formatted")   # inspect the physical plan (below)

Read a Spark plan bottom-up: the leaves (the FileScans) run first, data flows upward, and the root is produced last. The labels (A)–(E) are annotations for the walkthrough that follows.

== Physical Plan ==
AdaptiveSparkPlan isFinalPlan=false
+- HashAggregate(keys=[country], functions=[sum(amount)])                # (E) FINAL aggregate
   +- Exchange hashpartitioning(country, 200), ENSURE_REQUIREMENTS      # (D) the ONE shuffle
      +- HashAggregate(keys=[country], functions=[partial_sum(amount)])   # (C) PARTIAL aggregate
         +- Project [country, amount]
            +- BroadcastHashJoin [customer_id], [customer_id], Inner, BuildRight   # (B) NO shuffle
               :- Filter (isnotnull(status) AND (status = PAID))          # (A) narrow, pushed down
               :  +- FileScan parquet orders[customer_id,amount,status]
               :        PushedFilters: [IsNotNull(status), EqualTo(status,PAID)]
               +- BroadcastExchange HashedRelationBroadcastMode           # small side collected + shipped
                  +- Filter isnotnull(customer_id)
                     +- FileScan parquet customers[customer_id,country]

Three things make this plan efficient, and they are exactly what we will be restoring in the slow job below:

  • (A) Narrow ops fuse into the scan. The Filter (status = PAID) is pushed into the FileScan (PushedFilters), so non-PAID row groups are skipped at read time and only three columns are read (column pruning) — all in one pipelined stage.
  • (B) The broadcast join moves no big data. Because customers is broadcast, Spark chose BroadcastHashJoin…BuildRight: the tiny table is shipped as a hash map to every executor while the huge orders table never moves. No shuffle on the big side — contrast a sort-merge join, which shuffles both tables.
  • (C–D–E) The groupBy is the one real shuffle, done in two phases. A partial aggregate pre-sums per country within each partition before the Exchange, so the network only carries a handful of partial sums per country; the final aggregate after the shuffle adds them up. Two-phase aggregation is why groupBy(...).agg(sum...) is far cheaper than shuffling raw rows.
UnitWhat you'll see in the Spark UI
JobsUsually two: a small preparatory job that reads customers and builds the broadcast, then the main job triggered by write. Every action is a job — the broadcast collect counts as one.
StagesThe main job splits into two stages at the one Exchange. Stage 1: scan + filter + broadcast-join probe + partial aggregate, ending in a shuffle write partitioned by country. Stage 2: shuffle read + final aggregate + write to Parquet.
TasksStage 1: one task per input partition — ~2000 files chunked by spark.sql.files.maxPartitionBytes (128 MB). Stage 2: up to 200 tasks (the shuffle-partition count), which AQE coalesces down to a handful since the aggregated output is tiny.

The slow job (baseline)

Now the same shape done wrong. We read a big orders fact table, join a small customers dimension, group by segment and sum revenue — and it takes 18m 40s.

from pyspark.sql import functions as F
orders    = spark.read.parquet("s3://lake/orders/")      # ~500 GB fact table
customers = spark.read.parquet("s3://lake/customers/")   # ~40 MB dimension

report = (orders.join(customers, "cust_id")               # join
                .groupBy("segment")                        # group
                .agg(F.sum("amount").alias("revenue")))    # aggregate
report.write.mode("overwrite").parquet("s3://lake/out/revenue_by_segment/")
# wall clock: 18m 40s

Step 1 — diagnose in the Spark UI

What the UI shows
  • SQL / DataFrame tab: a SortMergeJoin sitting over two Exchange nodes. The 40 MB dimension is just above the 10 MB autoBroadcastJoinThreshold, so Spark shuffles both sides — including the whole 500 GB fact table. That is the big shuffle.
  • Stages tab (aggregation stage): only 200 tasks (the default spark.sql.shuffle.partitions), each reading ~750 MB → Spill (disk) 40 GB.
  • Task summary: median task 22 s but max 6.2 min — a 17× straggler. One segment dominates the data, so its post-shuffle partition is huge. That is the straggler.

Step 2 — fix (a): broadcast the dimension

The plan told us the small side was shuffled because it was just over the threshold. Force it — now the 500 GB fact table never moves for the join, and the plan looks like the BroadcastHashJoin from the target above.

from pyspark.sql.functions import broadcast
report = (orders.join(broadcast(customers), "cust_id")
                .groupBy("segment").agg(F.sum("amount").alias("revenue")))
# plan now: BroadcastHashJoin — the Exchange on the 500 GB side is gone
# 18m 40s  ->  7m 10s   (one of the two shuffles eliminated)

Step 3 — fix (b): right-size shuffle partitions

The groupBy still shuffles ~150 GB, and 200 partitions makes each ~750 MB — far past execution memory, hence the 40 GB spill. Size them to ~128 MB: 150 GB / 128 MB ≈ 1200.

# ~150 GB shuffle / 128 MB target ≈ 1200 partitions (default 200 was far too few)
spark.conf.set("spark.sql.shuffle.partitions", 1200)
# 7m 10s  ->  4m 05s   (Stages tab: Spill (disk) now 0)

Step 4 — fix (c): enable AQE

1200 partitions is right for the average segment but wasteful for tiny ones, and it doesn't fix the one hot-segment straggler. AQE coalesces the small partitions and splits the skewed one — at runtime, no code change.

spark.conf.set("spark.sql.adaptive.enabled", True)                 # default in 3.2+
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", True)
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", True)
# AQE splits the hot-segment partition and coalesces the tiny ones
# 4m 05s  ->  2m 30s   (task max ≈ median now — the straggler is gone)

Step 5 — fix (d): cache the reused result

The pipeline actually feeds three downstream outputs (a parquet write, a high-value filter, and a top-segments display). Without caching, each recomputes the full 2m 30s lineage. Materialise report once, then reuse it from memory. (DataFrame.cache() defaults to MEMORY_AND_DISK, so it spills rather than recomputes.)

report = report.cache()
report.count()                                # 2m 30s — materialize once, fills the cache
report.write.parquet("s3://lake/out/revenue/")     # reads cache:  9s
report.filter("revenue > 1e6").write.parquet("s3://lake/out/big/")  # reads cache:  7s
report.orderBy(F.desc("revenue")).show(20)      # reads cache:  5s
report.unpersist()
# without cache the 3 reuses each recomputed 2m 30s (~7.5 min); with cache: ~21s total

The takeaway
  • Order matters. We attacked the biggest bottleneck first — the 500 GB shuffle — which alone gave a 2.6× speedup. Fixing partitions or AQE first would have helped less and hidden the real problem.
  • One change per loop, always measured. Each fix was verified in the UI before the next, so we know exactly what each was worth.
  • Every fix came from the playbook. SMJ where one side is small → broadcast; spill → more partitions; straggler → AQE skew; reuse → cache. Roughly 7× faster end-to-end, and far more on repeat runs — the slow job now has the same shape as the well-formed plan we started with.

Cache in RAM as deserialized objects; partitions that don't fit are recomputed, not spilled. Fastest, least safe. RDD default.
memoryOverheadFactor
spark.executor.memoryOverheadFactor (Spark 3.3+) — the fraction of heap used for overhead when no absolute value is set. Default 0.10 on YARN, 0.40 on Kubernetes (for PySpark workers).
min / max / initial executors
The bounds of dynamic allocation: minExecutors (floor, default 0), maxExecutors (ceiling, default infinity), initialExecutors (start count, defaults to min).
Multi-level partitioning
Partitioning by several columns, e.g. partitionBy("year","month"), nesting each column as a sub-directory; finer pruning but many more directories.
Narrow transformation
A transformation where each output partition depends on one input partition — no shuffle.
Off-heap memory
Memory managed by Spark outside the JVM heap (Tungsten binary format), avoiding GC. Opt-in.
OFF_HEAP
Cache outside the JVM heap (requires off-heap memory configured), avoiding garbage-collection pauses. Advanced / experimental.
One-sided shuffle
When only one table is bucketed, Spark shuffles just the other (raw) side to match the buckets — cheaper than two shuffles, but not free.
Page (Parquet)
The smallest unit inside a column chunk (dictionary page + data pages); the unit of encoding and compression.
parallelismFirst
AQE flag (default true) that packs coalesced partitions down to minPartitionSize for maximum parallelism; set false to respect the advisory size.
Parquet
The default columnar file format: data stored column-by-column in row groups, with a footer of statistics — enabling column pruning and row-group skipping.
Partition
A chunk of a distributed dataset; the unit a single task processes.
Partition column
The column(s) passed to partitionBy whose values name the directories. Pruning only works when you filter on these.
Partition pruning
The reader skipping partition directories that cannot satisfy a filter on the partition column, so only matching directories are read. Confirm via explain() → PartitionFilters.
PartitionFilters
The part of a scan's physical plan that decides which partition directories to open; with DPP it holds a dynamicpruning subquery instead of a literal.
persist()
The general form: persist(StorageLevel) lets you choose exactly where and how the cached data is stored.
Physical plan
The concrete, executable strategy (algorithms + data movement) Catalyst chooses to run a query.
Power law
The typical real-world key distribution — a few values dominate, the rest form a long tail; the root cause of skew.
Pre-shuffle
The one-time shuffle done at write time by bucketBy; it replaces the per-query shuffle a join or group-by would otherwise repeat.
Predicate pushdown
An optimization that moves row filters down to the data source so less data is ever read (e.g. skipping Parquet row groups that can't match).
preferSortMergeJoin
Config (default true) that makes Spark prefer Sort Merge Join over Shuffle Hash Join — why SHJ is rarely auto-chosen.
Projection pushdown
Reading only the columns a query uses; shown by a reduced ReadSchema in the scan.
PushedFilters
The scan-node field listing filters evaluated at the source — the proof predicate pushdown happened.
PushedFilters / ReadSchema
Lines on the Scan node proving predicate pushdown (rows skipped at read) and column pruning (only needed columns decoded); empty PushedFilters means the whole file is read.
PySpark memory
Memory used by the Python worker processes that run Python UDFs and pandas UDFs, outside the JVM. Optionally budgeted with spark.executor.pyspark.memory; otherwise it consumes overhead and is a frequent cause of container kills.
rangepartitioning
The Exchange variant from a global orderBy: rows split into sorted key ranges via sampling.
RDD
Resilient Distributed Dataset — Spark's low-level immutable, partitioned, fault-tolerant collection.
Repartition
Changing the partition count via a full shuffle, producing evenly-sized partitions (up or down).
Replication (_2)
A suffix (e.g. MEMORY_AND_DISK_2) that stores each partition on two nodes for fault-tolerance, at double the storage cost.
Reserved memory
A fixed 300 MB of the heap held back for Spark's internal objects; subtracted before any other split. Not configurable.
reuseBroadcastOnly
Config (default true) that limits DPP to cases where a broadcast can be reused. Set false to also allow DPP on non-broadcast (sort-merge) joins via an independent, cost-gated subquery.
ReusedExchange
A plan node showing a shuffle result computed once and reused by multiple branches.
Row group
A horizontal slice of rows within a Parquet file, holding one column chunk per column and its own min/max stats; the unit at which Spark skips or reads data.
Runtime pruning
Deferring the pruning decision until execution because the filter value depends on data (the dimension), not on a query literal.
Salt
The random integer 0..N-1 appended to a key. N is the number of buckets the hot key spreads across.
Salting
Adding a random value to a skewed key so its rows spread across tasks (replicating matches on the other join side); the manual skew fix.
saveAsTable / metastore
The only supported way to write buckets: the bucket layout is recorded as table metadata in the metastore, and readers use it via spark.table(…).
SelectedBucketsCount
The scan-node metric in the physical plan (e.g. 1 out of 64) that shows how many bucket files were read after pruning — the proof pruning fired.
Serialized
Storing partitions as compact byte arrays (the _SER levels) — smaller footprint, but CPU is spent (de)serializing on read.
Shuffle
The all-to-all data exchange (disk + network) that regroups data by key across the cluster; caused by wide transformations.
Shuffle Hash Join
A join that shuffles both sides by key and builds an in-memory hash map from the smaller side (no sort); rarely auto-chosen.
Shuffle Hash Join (SHJ)
Shuffles both sides, then builds an in-memory hash table from the smaller side and probes it — no sort. Rarely auto-selected.
Shuffle partition
One of the N buckets shuffled data is regrouped into by key; Spark runs one reduce task per bucket.
Shuffle partitions
spark.sql.shuffle.partitions (default 200) — the fixed post-shuffle partition count; the reason AQE coalescing exists. Raising it never cures skew.
Shuffle read / write
The bytes an Exchange sent (write) and pulled (read) across the network — the metric that flags an expensive shuffle.
Shuffle tracking
spark.dynamicAllocation.shuffleTracking.enabled (Spark 3.0+, experimental) — lets dynamic allocation release executors safely without an external shuffle service by not evicting executors that still hold live shuffle blocks.
Shuffle Write / Shuffle Read
The bytes the map side writes to local disk and the reduce side fetches back; they are equal, and their total ÷ the per-partition target is the right count.
ShuffledHashJoin
A shuffle join that builds a hash table instead of sorting: Exchange on both sides but no Sort. Faster than SMJ but can OOM, so it's off by default (preferSortMergeJoin=true).
Single-level partitioning
Partitioning by one column, e.g. partitionBy("year") — one flat level of directories.
Skew
Uneven data distribution where one key/partition is far larger than others, creating straggler tasks.
skewedPartitionFactor
AQE threshold (default 5): a partition is skewed if it exceeds this multiple of the median partition size…
skewedPartitionThresholdInBytes
…and exceeds this absolute size (default 256 MB). Both must hold for AQE to split it.
Skipped stage
A stage the UI marks as skipped because its output (a prior shuffle or a cached result) was reused — work Spark didn't have to redo.
Small-files problem
Too many tiny files/directories (often from partitioning on a high-cardinality column) so metadata and per-task overhead dominate, slowing jobs and straining the driver.
Sort Merge Join
The default large↔large join: hash-shuffle both sides by key, sort each, merge. Skews when the key is hot.
Sort Merge Join (SMJ)
Shuffles both sides by key, sorts each, then merges — the robust default for large↔large equi-joins.
Sort-Merge Join
Spark's default join for two large tables; bucketed inputs let it run per bucket with no Exchange (and, with sortBy, no re-sort).
sortBy(key)
Sorts rows within each bucket by the key at write time, so a sort-merge join can also skip its sort step.
SortMergeJoin
The default large↔large join operator: Exchange + Sort on both sides, then a merge. Robust — its sort can spill to disk rather than fail.
Spark Connect
A decoupled thin-client architecture (gRPC) that lets remote clients drive a Spark server; expanded in Spark 4.0.
spark.memory.fraction
Fraction of usable memory (heap − 300 MB) assigned to Unified Memory. Default 0.6; the rest becomes User Memory.
spark.memory.storageFraction
Fraction of Unified Memory guaranteed to Storage — the floor below which Execution cannot evict cached blocks. Default 0.5.
spark.sql.shuffle.partitions
The config setting N, the number of post-shuffle partitions; defaults to 200.
spark.sql.sources.bucketing.enabled
The master switch for using bucket metadata in query planning; defaults to true.
SparkSession
The unified entry point to Spark (since 2.0), wrapping SparkContext and exposing the DataFrame/SQL APIs.
Spill
When a task exceeds its execution memory and Spark writes intermediate data to disk — a common cause of slow stages, visible in the Spark UI.
Spill (memory / disk)
Data that overflowed executor memory during a sort/aggregate/join. Memory is its deserialized in-RAM size, disk the serialized bytes written out; any spill flags memory pressure.
Stage
A set of tasks that can run without a shuffle; stage boundaries fall at each wide transformation.
Star schema
A warehouse design with one large fact table joined to several small dimension tables via key columns.
Static partition pruning
Plan-time skipping of partition directories because the query has a literal filter directly on the partition column.
Storage level
How a cached dataset is stored (memory / disk, deserialized / serialized, on-/off-heap), e.g. MEMORY_AND_DISK.
Storage memory
Longer-lived memory for cached/persisted data and broadcasts. Evictable down to the storage floor when Execution needs room.
Storage tab
The Spark UI page listing cached datasets with their storage level, cached-partition count, fraction cached and size in memory / on disk.
StorageLevel
The object (in pyspark) describing a caching policy — whether to use memory, disk, off-heap, serialization, and replication.
Straggler
A single task running far longer than its peers (from skew), holding up the stage.
SubqueryBroadcast / ReusedExchange
Plan nodes that feed a DPP filter: SubqueryBroadcast supplies the pruning key set, typically via a ReusedExchange that reuses the join's existing BroadcastExchange (no extra pass over the dimension).
Summary Metrics
The Spark UI per-stage table of Min / 25th / Median / 75th / Max task metrics; Max » Median signals skew.
Targeted salting
Salting only the known hot key(s) — random salt for those rows, a fixed salt = 0 for the rest — then replicating (join) or two-staging (aggregation) only that slice. Minimises replication and extra shuffle by scoping the fix to the values that actually straggle.
Task
The smallest unit of execution: one stage's computation applied to one partition, run on one core.
Task Scheduler
The driver component that assigns individual tasks to executor cores and retries failures.
Thin executor
An executor with a single core and small heap (many per node). Great isolation, but overhead is paid many times and nothing is shared.
Time travel
Querying a Delta table as it existed at an earlier version or timestamp, using the history in the transaction log.
Tungsten
Spark's execution engine backend: off-heap binary memory, cache-aware processing, and whole-stage code generation.
Two-phase aggregation
Partial (per-partition) aggregation before the shuffle, then final aggregation after — minimises shuffled data.
Two-stage aggregation
The salted-aggregation pattern: stage 1 aggregates per (key, salt) (balanced), stage 2 strips the salt and combines the partials per key (cheap). Works for associative aggregates.
Unified memory
The shared memory pool (execution + storage) introduced in Spark 1.6, with a dynamic boundary between the two.
Unified memory / storageFraction
Execution and storage share one pool (spark.memory.fraction, 0.6). spark.memory.storageFraction (0.5) is the floor below which cached blocks are protected from eviction; execution can borrow/evict storage above it, never the reverse.
unpersist()
Removes a dataset from the cache and returns its memory to the pool. Should be called explicitly once the cached data is no longer needed (blocking=True waits for eviction).
User memory
The 0.4 of usable heap Spark does not manage — your own objects, UDF state, and metadata live here. A common hidden source of OOMs.
VARIANT
A semi-structured data type (Spark 4.0) for storing and efficiently querying JSON/nested data in one column.
Vectorized reader / batch
Reading and processing column values in batches (columnar chunks) rather than row-by-row — the fast path for Parquet in Spark SQL.
Whole-stage code generation
A Tungsten technique that compiles an entire query stage into a single Java function, removing per-row overhead.
Whole-stage codegen
Compiling a run of operators into one Java function (the *(n) prefix); a new group with the next number begins at each Exchange (the counter increments, it does not reset).
WholeStageCodegen (n)
A dashed box grouping operators fused into one generated Java function — effectively one stage's compute; the number increments at each shuffle.
Wide transformation
A transformation where output partitions depend on many input partitions — forces a shuffle.
Z-Ordering
A clustering technique that co-locates similar values into the same files, tightening per-file min/max ranges so data skipping drops far more files.
~5-core rule
The rule of thumb that ~5 cores per executor maximises HDFS read/write throughput without I/O contention.
