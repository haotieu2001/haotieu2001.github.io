01. Why Linux is the ground every pipeline stands on

You may develop on a Mac or a Windows laptop. Your job does not run there. It runs on Linux — and when it breaks at 3am, Linux is the only thing you get to talk to.

Every layer of a modern data platform bottoms out in Linux. The Airflow scheduler is a Linux process. Each Spark executor is a Linux process in a container, which is itself just a Linux process with a restricted view (that is the whole trick — Docker Essentials §05). Your dbt run, your Kafka broker, your Postgres instance, the CI runner that tested your pull request — all Linux. When a managed service hides this from you, it hides it right up until the moment something goes wrong, and then it hands you a log file and wishes you luck.

So the question is not whether you need Linux. It is how much. And the honest answer is: far less than a sysadmin, but far more fluently. A data engineer needs five things — to find things on a machine, to read text at scale, to run something on a schedule, to get data in and out over the network, and to work out why a job failed using nothing but a terminal. That is what this page covers, in that order, with the traps that actually catch people.

The mindset shift that makes Linux click

Linux is not an application you learn the menus of. It is a box of small, sharp tools that all speak the same language — lines of text — and a way of bolting them together into a chain. There is no "log analysis program." There is grep, which throws away lines you don't want; awk, which pulls out columns; sort and uniq, which count things. Bolt those four together with pipes and you have written a log analyser in one line, on a server with no software installed, over SSH, at 3am. That composability is the operating system's design.

02. The mental model: kernel, shell, process, file

Four words explain almost everything. Learn what each one actually is and most "weird Linux behaviour" stops being weird.

The kernel is the program that owns the hardware. It, and only it, can touch the disk, the network card, and memory. It decides which process runs on which CPU and when. You never speak to the kernel directly — you ask it for things through system calls: open(), read(), write(), fork(), execve(). There are a few hundred of them, and every single thing your program does that touches the outside world is one of them underneath.

The shell — usually bash — is not the operating system. It is an ordinary program whose whole job is to read a line you type, work out what you meant, and start other programs on your behalf. When you type ls -l *.csv, the shell expands *.csv into a list of filenames itself (ls never sees the star), finds the ls binary by searching the directories in $PATH, calls fork() to clone itself, then execve() to replace the clone with ls. Understanding that the shell expands things before the command runs explains half of all shell bugs.

A process is a running program: a copy of the code, its own memory, a numeric ID (PID), an owner, a current directory, and a table of open file descriptors. And a file — this is the famous part — is almost everything. Your keyboard is a file. A network socket is a file. A running process's own state is a set of files under /proc. Because they are all "files", they can all be read and written with the same handful of syscalls, which is why a pipe can shove the output of any program into the input of any other without either program knowing.

"Everything is a file" is not a slogan — it is why pipes work

A process gets its input from file descriptor 0 and writes to 1. It does not know or care what is on the other end. Redirect 0 to a file and it reads the file. Point 1 at another process's 0 — that is all a pipe is — and you have a chain. This is why grep can filter a log file, the output of curl, or a stream from a database dump, without a single line of special-case code in grep.

03. The filesystem: one tree, and what lives where

There are no drive letters. There is one tree, rooted at /, and every disk, USB stick, and network share is grafted into it at some directory.

This is the single biggest difference from Windows, and it is a good design: a program never has to care which device a path lives on. When you attach a new disk, you mount it somewhere — say /mnt/data — and from then on /mnt/data/events.parquet is just a path. The kernel's virtual filesystem layer figures out which physical device to talk to. This is exactly how a Docker volume, an EBS disk, and an NFS share all end up looking identical to your Python script.

One layer down, a filename is not the file. The file is an inode — a numbered record holding the permissions, the owner, the size, and the pointers to the actual blocks on disk. A directory is just a list of names mapped to inode numbers. That one fact explains a surprising amount:

1
A hard link is a second name for the same inode. ln a.csv b.csv does not copy anything. Both names point at one file; delete either and the data survives until the last name is gone.
2
A symlink is a signpost. ln -s /mnt/data/current latest stores a path, not an inode. If the target disappears you get a dangling link — which is why symlink-swapping is the classic atomic "publish" trick for a new dataset version.
3
mv within one filesystem is instant, because it only rewrites a directory entry — no bytes move. mv across filesystems (say /tmp to /mnt/data) is secretly a copy-then-delete, and can take an hour on a big file. This is why the "write to a temp file, then mv it into place" trick is only atomic when both are on the same filesystem.
4
Deleting a file that a process still has open frees nothing. The name is gone from the directory but the inode lives until the last file descriptor closes. This is the notorious "df says the disk is full, du says it is empty" — usually a rotated log that some process is still writing to. Find it with lsof +L1.

04. Navigating the filesystem

Where am I, what is here, how do I get there, and where on this machine is the file I actually want?

Every process has a current working directory, and every path you type is interpreted relative to it — unless it starts with /, in which case it starts from the root and your location is irrelevant. That distinction is the whole of "absolute vs relative", and it is the reason a script that works when you run it by hand fails under cron: cron starts you somewhere else.

bash · getting around
pwd                      # where am I?
ls -lah                  # long, all (incl. dotfiles), human-readable sizes
ls -lt | head           # newest files first — "what just changed?"
cd /var/log              # absolute
cd ../data               # relative: .. is up one
cd ~                     # home. ~ is your $HOME
cd -                     # back to the previous directory (surprisingly useful)
stat events.csv          # size, owner, inode, timestamps
file mystery.dat         # what IS this? (gzip? parquet? text?)
du -sh *                 # size of each thing here
du -sh --max-depth=1 /mnt/data | sort -h   # who is eating the disk?
df -h                    # how full is each filesystem?

Then there is find, which is the one navigation command worth learning properly. It walks a tree and tests every entry against your conditions. For a data engineer it answers "which partitions landed today", "what is over a gigabyte", "which files are older than the retention window".

bash · find, the useful flags
# parquet files under /mnt/data modified in the last 24h, bigger than 10 MB
find /mnt/data -name '*.parquet' -mtime -1 -size +10M

# directories only, exactly one level down — a quick partition listing
find /mnt/data -mindepth 1 -maxdepth 1 -type d

# delete extracts older than 30 days (print first. ALWAYS print first.)
find /mnt/extracts -type f -mtime +30 -print
find /mnt/extracts -type f -mtime +30 -delete

# run a command per match, safely, even with spaces in filenames
find . -name '*.csv' -print0 | xargs -0 -n1 gzip

Quote the pattern, or the shell eats it

find . -name *.csv often fails with a confusing error. Why? The shell expands *.csv before find ever starts — so if there happen to be two CSVs in your current directory, find receives -name a.csv b.csv and complains. Quoting it (-name '*.csv') passes the star through to find, which does its own matching at every level of the tree. This is the shell-expands-first rule from §02, biting.

05. Permissions: users, groups, and chmod

Three actors (owner, group, everyone else) and three verbs (read, write, execute). That is the entire model, and "Permission denied" is always one of the nine boxes it makes.

Every file has an owner and a group. When you try to touch it, the kernel asks one question in order: are you the owner? If yes, the owner's three bits apply and nothing else is consulted. Otherwise — are you in the group? If yes, the group bits apply. Otherwise the "others" bits apply. It never falls through to a more permissive set, which trips people up: if the owner bits say r-- and the others bits say rw-, then the owner cannot write the file, even though a stranger could.

bash · the four you will actually type
chmod +x ingest.sh          # make a script runnable (the #1 use)
chmod 600 ~/.aws/credentials # only I can read it. secrets ALWAYS 600.
chmod 644 events.csv         # I write, everyone reads. normal data file.
chmod 755 /opt/etl           # normal directory: I write, everyone can enter+list

chown airflow:airflow /opt/etl/run.py   # change owner:group (needs root)
chmod -R g+w /mnt/data/staging          # -R = recursive. think before you type it.

# symbolic form: who (u/g/o/a) + operation (+/-/=) + bits
chmod u+x,go-w deploy.sh

When something is denied and you cannot see why, do not guess — walk the whole path. Access to /mnt/data/raw/2026/events.csv requires x on /, /mnt, /mnt/data, /mnt/data/raw, /mnt/data/raw/2026, and then r on the file. One missing x three levels up produces exactly the same error message as a wrong bit on the file itself.

bash · debugging "Permission denied"
namei -l /mnt/data/raw/2026/events.csv   # shows the bits at EVERY level. start here.
id                                       # who am I, and which groups am I in?
ls -ld /mnt/data/raw                     # -d = the directory itself, not its contents
sudo -u airflow ls /mnt/data/raw          # does it work as the user that ACTUALLY runs the job?

The one that catches everyone: umask

New files do not get the permissions you might expect — they get the default (666 for files, 777 for directories) with the umask bits subtracted. A typical umask 022 means every file your pipeline writes lands as 644 and every directory as 755. So when your job writes an extract that a teammate's job then cannot overwrite, the culprit is usually the umask of the user that ran the first job — not a bug in either script.

06. Streams, redirection, and pipes

Three numbered channels. Redirect them into files, or plug one process's output into the next one's input. This is the mechanism the whole toolbox is built on.

Every process starts with three file descriptors already open: 0 = stdin (where it reads input), 1 = stdout (where it writes results), and 2 = stderr (where it writes complaints). The split between 1 and 2 is deliberate and load-bearing: it lets you pipe the results of a command into another program while the errors still land on your screen instead of silently corrupting the data.

bash · redirection
cmd > out.txt            # stdout to a file (overwrites!)
cmd >> out.log           # stdout appended
cmd 2> err.txt           # stderr only
cmd > out.txt 2>&1       # stdout to file, then stderr to the SAME place
cmd &> all.log           # bash shorthand for both
cmd < input.csv          # feed a file in as stdin
cmd > /dev/null 2>&1     # throw everything away (the bit bucket)

cmd | tee run.log         # see it AND save it — the cron-logging trick
cmd1 | cmd2 | cmd3       # a pipeline: 1's stdout becomes 2's stdin

2>&1 before the redirect does nothing useful

cmd > log 2>&1 works: first stdout is pointed at the file, then stderr is pointed "wherever stdout is" — the file. But cmd 2>&1 > log points stderr at the terminal (where stdout still is at that moment), and only afterwards moves stdout to the file. Result: your errors go to the screen and vanish, and your log looks clean while the job silently failed. Read redirections strictly left to right.

07. Reading and searching text: cat, less, grep

The first tools you reach for, and the ones you will use ten times a day. Mostly you are answering: is my thing in this file, and how many times?

cat dumps a whole file to stdout. It is perfect for small files and for feeding one into a pipeline, and it is the wrong tool for a 4 GB log — that will flood your terminal and, worse, load nothing useful into your brain. Reach for less (scrollable, searchable with /, quits with q) or take a slice with head and tail.

bash · look at a file without regretting it
cat small.csv                # fine for small things
less huge.log                # scroll, / to search, q to quit. never floods you.
head -5 events.csv           # first 5 lines — usually the header
tail -100 app.log            # last 100 lines — usually the crash
tail -f app.log              # FOLLOW it live. watch a job as it runs.
zcat events.csv.gz | head   # peek inside a gzip without unpacking it
wc -l events.csv             # how many rows? (lines, really)

grep keeps the lines that match a pattern and throws the rest away. That is all it does, and it is the most useful thing on the machine. Learn six flags and you have a log analysis tool.

bash · grep, the six flags that matter
grep 'ERROR' app.log              # the lines containing ERROR
grep -i 'error' app.log           # -i  case-insensitive
grep -c 'ERROR' app.log           # -c  just COUNT them
grep -v 'DEBUG' app.log           # -v  INVERT: everything that does not match
grep -n 'ERROR' app.log           # -n  show line numbers
grep -r 'api_key' /opt/etl         # -r  recurse a whole directory
grep -E 'ERROR|FATAL' app.log      # -E  extended regex (| = or)

# the two-line combo that starts most investigations:
grep -i error app.log | tail -20        # the most recent 20 errors
grep -c 'ERROR' app.log                 # ...and is it 3, or 300,000?

# context: 3 lines either side of the match. gold for stack traces.
grep -C 3 'Traceback' app.log

grep -F is dramatically faster on big files

By default grep compiles your pattern as a regular expression. If you are searching for a plain string — an order ID, a hostname — grep -F ("fixed string") skips the regex engine entirely and can be several times faster over a multi-gigabyte log. And if you have a whole list of IDs to look for, grep -F -f ids.txt events.log matches all of them in a single pass, which beats a shell loop calling grep 5,000 times by roughly the margin you would expect.

08. Parsing with awk (and a little sed)

grep chooses rows. awk chooses columns — and can do arithmetic on them while it streams. For anyone who works with CSVs and logs, this is the highest-leverage tool on the page.

awk has a model so simple it can be stated in one sentence: for every line, split it into fields, and run your program on it. The line is $0. The fields are $1, $2, $3… NF is the number of fields on this line and NR is the number of the record (line) we're on. A BEGIN block runs once before the first line, an END block runs once after the last — which is where totals get printed.

bash · awk for data work
# print columns 1 and 3 of a CSV
awk -F, '{ print $1, $3 }' events.csv

# filter: rows where the amount (col 3) exceeds 100
awk -F, '$3 > 100' sales.csv

# sum a column
awk -F, '{ s += $3 } END { print "total:", s }' sales.csv

# GROUP BY: total per region — an associative array does the work
awk -F, '{ sum[$2] += $3 } END { for (k in sum) print k, sum[k] }' sales.csv

# skip the header row (NR is the record number)
awk -F, 'NR > 1 { s += $3 } END { print s }' sales.csv

# find malformed rows: not exactly 5 fields — a 10-second data quality check
awk -F, 'NF != 5 { print NR": "$0 }' events.csv

# change the delimiter: CSV in, TSV out
awk -F, 'BEGIN { OFS="\t" } { $1=$1; print }' events.csv

sed is awk's blunter sibling: it edits a stream, line by line, and its one killer feature is substitution. You will use it to fix delimiters, strip characters, and patch config files in place.

bash · sed, realistically
sed 's/,/\t/g' in.csv > out.tsv     # s/find/replace/g — g = every occurrence, not just the first
sed '1d' events.csv                 # delete line 1 (drop the header)
sed -n '10,20p' huge.log            # print ONLY lines 10–20 (-n = don't auto-print)
sed -E 's/[0-9]{4}-[0-9]{2}-[0-9]{2}/DATE/g' app.log  # -E = modern regex
sed -i 's/localhost/db.prod/' config.ini  # -i = EDIT THE FILE IN PLACE. no undo.

awk and sed do not understand CSV

They split on a character. They have no idea that "Smith, John",42 is two fields, not three — a quoted comma will silently shift every column after it. For quick investigation of well-behaved data this is fine and fast. For anything you will load into a warehouse, use a real CSV parser (Python's csv, duckdb, or xsv/csvkit). Knowing which tool is "good enough for a look" versus "safe for production" is exactly the judgment being tested when someone hands you a broken file.

09. The pipeline toolkit: sort, uniq, cut, tr, wc

Five more small tools. Chained onto grep and awk, they answer most "what is going on in this file" questions in a single line.

bash · the rest of the toolkit
sort file            # lexical sort
sort -n              # NUMERIC ("10" < "9" lexically — this bites constantly)
sort -rn             # numeric, biggest first
sort -u              # sort and drop duplicates
sort -t, -k3 -n      # split on comma, sort by 3rd field, numerically
uniq -c             # count adjacent duplicates. MUST be sorted first.
uniq -d             # show only the duplicated values (dupe check!)
cut -d, -f1,3       # fields 1 and 3, comma-delimited (awk-lite)
tr -d '\r'          # delete carriage returns — fixes Windows CSVs
tr '[:upper:]' '[:lower:]'   # lowercase a stream
wc -l               # count lines
shuf -n 100         # random sample of 100 rows — a great sanity check
split -l 1000000 big.csv part_   # chop into 1M-line chunks
bash · one-liners worth stealing
# top 10 IPs hitting an nginx log
awk '{print $1}' access.log | sort | uniq -c | sort -rn | head -10

# is my join key unique? (prints only the keys that appear more than once)
cut -d, -f1 users.csv | sort | uniq -d | head

# how many rows landed per day, from the filenames
ls /mnt/data/raw | cut -d_ -f2 | sort | uniq -c

# does the row count match what the source promised?
zcat extract.csv.gz | wc -l

10. Processes, jobs, and signals

How to start something long, walk away from it, find it again, and stop it politely — plus what actually happens when you press Ctrl-C.

bash · finding and controlling processes
ps aux | grep python        # what python is running? (classic, if crude)
pgrep -af ingest             # cleaner: PIDs + full command line
top                          # live view. press 1 for per-CPU, q to quit
htop                         # the same, but civilised (if installed)

kill 4821                    # SIGTERM — ask nicely
kill -9 4821                 # SIGKILL — last resort
pkill -f 'ingest.py'        # kill by command line pattern (careful!)

jobs                         # background jobs of THIS shell
fg %1                        # pull job 1 back to the foreground
bg %1                        # resume a stopped job in the background
Ctrl-Z                       # suspend the foreground job

timeout 30m ./ingest.sh       # kill it if it runs over 30 minutes. use this in cron.
nice -n 10 ./heavy.sh         # be polite to other jobs on a shared box
echo $?                       # exit code of the last command. 0 = success.

The real answer for long jobs is tmux, not nohup

SSH into a box, start a six-hour backfill, and the moment your VPN blips the connection dies — and takes the job with it. nohup saves the job but you lose the ability to watch it. tmux gives you both: tmux new -s backfill, start the job, press Ctrl-b d to detach, close your laptop, and tomorrow tmux attach -t backfill puts you back in the same terminal with the job still printing. Every experienced engineer learned this the hard way exactly once. See §19.

11. Writing shell scripts

A shell script is just the commands you would have typed, in a file, with a line at the top saying which interpreter should read them. Everything else is learning the handful of rules bash applies before your command ever runs.

The first line — the shebang — tells the kernel what to run the file with. Write #!/usr/bin/env bash rather than #!/bin/bash: it finds bash via $PATH, which is more portable across distributions and containers. Then make the file executable (chmod +x, §05) and you have a new command on the machine.

bash · the three lines every script starts with
#!/usr/bin/env bash          # 1. which interpreter runs this file
# ingest.sh — pull yesterday's extract and stage it
set -euo pipefail            # 2. fail fast and loudly (§12 explains each flag)

Variables, and the one rule that causes every bash bug

Assignment has no spaces around the = — NAME="hao", never NAME = "hao" (that would try to run a command called NAME). Reading a variable uses $. So far, so ordinary. The part that bites is what bash does to the value after it substitutes it in.

Before any command runs, bash rewrites the line: it substitutes variables, then splits the result on whitespace into separate words, then expands any globs it finds. The command is only handed the final list of words. So an unquoted $FILE is not "the value of FILE" — it is the value, chopped up. Double quotes are what switch the chopping off.

bash · variables and expansion
NAME="hao"                      # no spaces around =
TODAY="$(date -u +%F)"          # $( ) runs a command, captures its stdout
ROWS=$(( $ROWS + 1 ))            # $(( )) does arithmetic

echo "$NAME"                     # ALWAYS quote. always.
echo "${NAME}_suffix"            # braces when the name touches other characters

# defaults and guards — enormously useful in scripts
DIR="${1:-/mnt/data}"            # use $1, or /mnt/data if it wasn't given
TOKEN="${API_TOKEN:?API_TOKEN is not set}"   # abort with a message if missing

# string surgery, without calling out to sed
F="events_2026-07-12.csv.gz"
echo "${F%.gz}"                  # events_2026-07-12.csv   (strip shortest from END)
echo "${F%%.*}"                  # events_2026-07-12       (strip longest from END)
echo "${F#events_}"              # 2026-07-12.csv.gz       (strip from the FRONT)
echo "${#F}"                     # 24 — the length

Arguments, conditionals, loops, functions

bash · arguments
# $1 $2 ...  the positional arguments
# $#         how many were given
# "$@"       ALL of them, each one still a separate word (quote it!)
# $0         the script's own path — dirname "$0" gives its directory

if [[ $# -lt 2 ]]; then
  echo "usage: $0 <source> <date>" >&2    # usage goes to STDERR
  exit 2                                # 2 = "you called me wrong"
fi
SOURCE="$1"
DAY="$2"

# proper flags, when a script grows up: -v verbose, -o FILE output
while getopts ":vo:" opt; do
  case "$opt" in
    v) VERBOSE=1 ;;
    o) OUTFILE="$OPTARG" ;;      # the colon after o means "takes a value"
    *) echo "bad flag" >&2; exit 2 ;;
  esac
done
shift $(( OPTIND - 1 ))                  # drop the flags; "$@" is now the real arguments
bash · conditionals — the test operators worth knowing
# files and directories
[[ -f "$F" ]]      # exists and is a regular file
[[ -d "$D" ]]      # exists and is a directory
[[ -s "$F" ]]      # exists and is NOT EMPTY  ← the one that catches bad downloads
[[ -r "$F" ]]      # I can read it
[[ -x "$F" ]]      # I can execute it

# strings
[[ -z "$S" ]]      # empty
[[ -n "$S" ]]      # not empty
[[ "$A" == "$B" ]]           # equal
[[ "$F" == *.csv ]]         # pattern match (do NOT quote the pattern)

# numbers — these are DIFFERENT from the string operators. mixing them is a classic bug.
[[ "$N" -eq 0 ]]   # equal        -ne  not equal
[[ "$N" -gt 100 ]] # greater than -lt  less than   -ge  -le

# combine, and use the exit code of a command directly
if [[ -f "$F" && -s "$F" ]]; then ... fi
if grep -q ERROR app.log; then ... fi   # "if grep FOUND something" (-q = quiet)

# the one-line idioms you will read in other people's scripts
[[ -f "$F" ]] || { echo "missing $F" >&2; exit 1; }   # || = "or else"
mkdir -p "$DIR" && cd "$DIR"                        # && = "and then, if that worked"
bash · loops and arrays
# over a fixed list
for table in users orders events; do
  echo "pulling $table"
done

# over files — glob directly, never "for f in $(ls)" (it breaks on spaces)
for f in /mnt/data/raw/*.csv; do
  [[ -e "$f" ]] || continue            # a glob that matches nothing stays literal!
  gzip "$f"
done

# over the lines of a file — this exact incantation, for a reason:
#   IFS= keeps leading/trailing spaces, -r stops backslash mangling,
#   [[ -n "$line" ]] catches a last line with no trailing newline
while IFS= read -r line || [[ -n "$line" ]]; do
  echo "row: $line"
done < events.csv

# arrays
TABLES=(users orders events)
echo "${TABLES[0]}"                  # users
echo "${#TABLES[@]}"                 # 3 — how many
for t in "${TABLES[@]}"; do echo "$t"; done   # quote it, or spaces split again
bash · functions and exit codes
# functions take arguments the same way the script does: $1, $2, "$@"
log() { echo "[$(date -u +%FT%TZ)] $*"; }
die() { echo "FATAL: $*" >&2; exit 1; }

row_count() {
  local file="$1"                  # local: don't leak into the rest of the script
  wc -l < "$file"                  # the function's "return value" is what it PRINTS
}

N=$(row_count events.csv)          # capture it with $( )
[[ "$N" -gt 0 ]] || die "events.csv is empty"

# `return` sets the exit CODE (0–255), not a value. use it for true/false:
is_gzip() { file "$1" | grep -q gzip; }      # grep's own exit code is the answer
if is_gzip extract.bin; then echo "it's gzipped"; fi

# EXIT CODES — the only thing cron, Airflow and CI actually look at
exit 0    # success
exit 1    # generic failure
exit 2    # called wrong (bad arguments) — a useful convention
echo $?   # the exit code of the last command

Quote your variables. Every time. This is the #1 bash bug.

The diagram above is the whole reason. rm $FILE where FILE="my report.csv" deletes two files. rm -rf $DIR/ where DIR is unset becomes rm -rf / — a legendary, career-defining outage. And for f in $(ls) silently breaks on every filename containing a space. The habit is simple: if it has a $, it goes in double quotes — including "$@" and "${ARRAY[@]}". Then run shellcheck yourscript.sh and let it find the ones you missed; it is free, it takes two seconds, and it catches real bugs before production does.

Example 1 · a CSV file you can trust

A vendor drops a file every morning. Before anything downstream touches it, you want to know: is it there, is it non-empty, does it have the columns we agreed, and does every row have the right number of fields? This script answers that and nothing else — and its exit code is the answer, so cron or Airflow can act on it.

check_csv.sh · validate a delivered file, or fail loudly
#!/usr/bin/env bash
#
# check_csv.sh — validate a delivered CSV before anything downstream reads it.
#
#   usage:  ./check_csv.sh [-d ,] [-m 100] <file> <col1,col2,...>
#   exit :  0 = good   1 = the data is bad   2 = you called it wrong

set -euo pipefail

DELIM=","
MIN_ROWS=1

log()  { echo "[$(date -u +%FT%TZ)] $*"; }
die()  { echo "FATAL: $*" >&2; exit 1; }
usage() { echo "usage: $0 [-d DELIM] [-m MIN_ROWS] <file> <expected,columns>" >&2; exit 2; }

# ── flags ─────────────────────────────────────────────────────────────────
while getopts ":d:m:" opt; do
  case "$opt" in
    d) DELIM="$OPTARG" ;;
    m) MIN_ROWS="$OPTARG" ;;
    *) usage ;;
  esac
done
shift $(( OPTIND - 1 ))

[[ $# -eq 2 ]] || usage
FILE="$1"
EXPECTED="$2"

# ── 1. does it exist, and is there anything in it? ────────────────────────
[[ -f "$FILE" ]] || die "$FILE does not exist"
[[ -r "$FILE" ]] || die "$FILE is not readable — check permissions (§05)"
[[ -s "$FILE" ]] || die "$FILE is empty — the vendor sent us nothing"

# ── 2. is the header the one we agreed? ──────────────────────────────────
HEADER=$(head -1 "$FILE" | tr -d '\r')      # strip Windows line endings
if [[ "$HEADER" != "$EXPECTED" ]]; then
  echo "  expected: $EXPECTED" >&2
  echo "  got     : $HEADER"   >&2
  die "header mismatch — the schema changed upstream"
fi

# ── 3. enough rows to be plausible? ──────────────────────────────────────
ROWS=$(( $(wc -l < "$FILE") - 1 ))          # minus the header
[[ "$ROWS" -ge "$MIN_ROWS" ]] \
  || die "only $ROWS rows, expected at least $MIN_ROWS"

# ── 4. does every row have the right number of fields? ───────────────────
NCOL=$(awk -F"$DELIM" 'NR == 1 { print NF }' "$FILE")
# awk stops itself after 5 — piping to `head` here would SIGPIPE awk and, under
# `set -o pipefail`, abort the whole script exactly when there ARE bad rows.
BAD=$(awk -F"$DELIM" -v n="$NCOL" 'NR > 1 && NF != n { print NR; if (++c == 5) exit }' "$FILE")
if [[ -n "$BAD" ]]; then
  echo "  first bad line numbers: $(echo "$BAD" | tr '\n' ' ')" >&2
  die "some rows do not have $NCOL fields"
fi

log "OK — $FILE: $ROWS rows, $NCOL columns, header matches"
exit 0
bash · using it
$ ./check_csv.sh -m 1000 /mnt/data/raw/events.csv 'event_id,user_id,occurred_at,amount'
[2026-07-13T02:31:07Z] OK — /mnt/data/raw/events.csv: 48,201 rows, 4 columns, header matches
$ echo $?
0

# and when the vendor quietly adds a column:
$ ./check_csv.sh /mnt/data/raw/events.csv 'event_id,user_id,occurred_at,amount'
  expected: event_id,user_id,occurred_at,amount
  got     : event_id,user_id,occurred_at,amount,currency
FATAL: header mismatch — the schema changed upstream
$ echo $?
1        # ← cron/Airflow sees this, and the run goes red instead of loading garbage

Example 2 · archive yesterday, prune last month

The other script every data platform ends up needing: compress what has landed, delete what is past its retention window, and never do either without being sure it worked. Note the --dry-run flag — anything that deletes files should be able to show you what it would delete before it does it.

rotate_extracts.sh · compress and prune, safely
#!/usr/bin/env bash
#
# rotate_extracts.sh — gzip extracts older than a day, delete those past retention.
#
#   usage:  ./rotate_extracts.sh [-n] [-k 30] <directory>
#           -n  dry run: print what WOULD happen, change nothing
#           -k  keep files for N days (default 30)
#   cron :  15 3 * * *  /opt/etl/rotate_extracts.sh -k 30 /mnt/data/extracts >> /var/log/etl/rotate.log 2>&1

set -euo pipefail

DRY_RUN=0
KEEP_DAYS=30

log() { echo "[$(date -u +%FT%TZ)] $*"; }
die() { echo "FATAL: $*" >&2; exit 1; }

while getopts ":nk:" opt; do
  case "$opt" in
    n) DRY_RUN=1 ;;
    k) KEEP_DAYS="$OPTARG" ;;
    *) echo "usage: $0 [-n] [-k DAYS] <dir>" >&2; exit 2 ;;
  esac
done
shift $(( OPTIND - 1 ))

[[ $# -eq 1 ]] || { echo "usage: $0 [-n] [-k DAYS] <dir>" >&2; exit 2; }
DIR="$1"
[[ -d "$DIR" ]] || die "$DIR is not a directory"     # NEVER rm -rf an unchecked path

[[ "$DRY_RUN" -eq 1 ]] && log "DRY RUN — nothing will be changed"

# ── 1. compress plain CSVs that are no longer being written to ───────────
COMPRESSED=0
while IFS= read -r -d '' f; do          # -d '' + -print0: safe with any filename
  if [[ "$DRY_RUN" -eq 1 ]]; then
    log "would gzip: $f"
  else
    gzip -- "$f"                          # -- : a filename starting with - is still a filename
    gzip -t -- "$f.gz" || die "$f.gz is corrupt after compression"
    log "gzipped: $f"
  fi
  COMPRESSED=$(( COMPRESSED + 1 ))
done < <(find "$DIR" -type f -name '*.csv' -mtime +0 -print0)

# ── 2. delete anything past the retention window ─────────────────────────
PRUNED=$(find "$DIR" -type f -name '*.csv.gz' -mtime +"$KEEP_DAYS" | wc -l)
if [[ "$PRUNED" -gt 0 ]]; then
  if [[ "$DRY_RUN" -eq 1 ]]; then
    log "would delete $PRUNED file(s) older than $KEEP_DAYS days:"
    find "$DIR" -type f -name '*.csv.gz' -mtime +"$KEEP_DAYS" -print   # SHOW them
  else
    find "$DIR" -type f -name '*.csv.gz' -mtime +"$KEEP_DAYS" -delete
    log "deleted $PRUNED file(s) older than $KEEP_DAYS days"
  fi
fi

log "done — compressed $COMPRESSED, pruned $PRUNED, $(df -h "$DIR" | awk 'NR==2 {print $4}') free"
exit 0

What to notice in both scripts

They are the same shape, and it is a shape worth copying. A usage line and an exit code convention (0 fine, 1 the data is bad, 2 you called it wrong) so a scheduler can act on the result. Every variable quoted, and every path checked before it is used — [[ -d "$DIR" ]] is the line standing between you and rm -rf on the wrong directory. Failures go to stderr and exit non-zero, successes are one timestamped line on stdout. Anything destructive has a dry run. And they do one thing: validating is not loading, rotating is not archiving. Small scripts that compose (§06) beat one big script with a --mode flag, every time.

12. Making a script safe to run unattended

A script you run by hand and watch is a different animal from one that fires at 02:30 with nobody looking. The difference is entirely in how it fails.

By default, bash is dangerously forgiving. If a command fails, it shrugs and runs the next one. If you reference a variable you never set, it substitutes an empty string and carries on. If the third command in a five-stage pipeline dies, the pipeline still reports success. Each of these defaults will, eventually, load an empty file into production and report a green run. Three flags turn all of it off.

bash · the line that goes at the top of every script
set -euo pipefail

# -e           exit immediately if any command fails (non-zero exit code)
# -u           treat an unset variable as an error, not as ""
# -o pipefail  a pipeline fails if ANY stage failed, not just the last one

# without pipefail this "succeeds", because sort exits 0:
#   curl -s https://api.example.com/data | sort > out.csv
# curl can die, sort happily sorts nothing, out.csv is empty, exit code 0.

Then there are four habits that separate a script that runs for a year from one that pages someone in month two.

1
Clean up on the way out, however you leave. trap 'rm -f "$TMP"' EXIT runs your cleanup whether the script finished, crashed, or was killed with Ctrl-C. Pair it with TMP=$(mktemp) so you never invent temp filenames that collide between two runs.
2
Write somewhere else, then move into place. Never write directly to the file that downstream jobs read. Write to events.csv.tmp, and only when it is complete and verified, mv it to events.csv. On the same filesystem a mv is atomic (§03) — so a reader either sees the whole old file or the whole new one, never a half-written one.
3
Only one of you at a time. If a run takes 40 minutes and cron fires every 30, you will eventually have two copies fighting over the same output. flock fixes this in one line and is the single most under-used command in scheduling.
4
Make failure loud. An unattended script must exit non-zero when it fails and say why on stderr. A job that fails quietly is far worse than one that crashes — it produces a green dashboard on top of stale data, and nobody looks for weeks.
bash · the patterns, in code
#!/usr/bin/env bash
set -euo pipefail

# 1. temp file + guaranteed cleanup
TMP="$(mktemp)"
trap 'rm -f "$TMP"' EXIT

# 2. atomic publish — readers never see a partial file
curl -fsS "$API" > "$TMP"
[[ -s "$TMP" ]] || { echo "empty response" >&2; exit 1; }   # -s = non-empty
mv "$TMP" /mnt/data/raw/events.json

# 3. single instance — the second run exits instead of colliding
#    put this in the CRON LINE, not the script:
#    flock -n /tmp/ingest.lock /opt/etl/ingest.sh   ( -n = fail rather than wait )

# 4. retry with backoff on a flaky API
for attempt in 1 2 3; do
  if curl -fsS --max-time 30 "$API" > "$TMP"; then
    break
  fi
  echo "attempt $attempt failed, retrying" >&2
  sleep $(( attempt * 10 ))          # $(( )) = arithmetic. 10s, 20s, 30s.
  [[ $attempt -eq 3 ]] && exit 1
done

13. Scheduling with cron

Cron is a daemon that wakes every minute, reads a table, and runs whatever is due. Five numbers describe when. Almost every cron bug is an environment bug, not a schedule bug.

bash · managing the crontab
crontab -e          # edit YOUR crontab
crontab -l          # list it
crontab -r          # DELETE IT ALL. no confirmation. never type this by accident.

# system-wide jobs (needs root) — these take an extra USER column:
#   /etc/cron.d/etl      30 2 * * 1-5  airflow  /opt/etl/ingest.sh

"It works when I run it, it fails in cron." Always the same four causes.

Cron does not give you a login shell. It does not source ~/.bashrc or ~/.profile. So: (1) your $PATH is a stub — typically just /usr/bin:/bin — so python, aws, dbt and anything in a virtualenv are simply "command not found". (2) Your working directory is $HOME, not the script's directory, so every relative path breaks (§04). (3) Environment variables you exported in your shell — API_TOKEN, PYTHONPATH — do not exist. (4) Output goes nowhere useful; if you do not redirect it, cron tries to email it, and on a server with no mailer it evaporates. The fix for all four is below.

crontab · how to write a cron line that actually works
# set the environment explicitly at the top of the crontab
SHELL=/bin/bash
PATH=/usr/local/bin:/usr/bin:/bin
MAILTO=""

# min hr dom mon dow   command
  30  2   *   *  1-5   flock -n /tmp/ingest.lock /opt/etl/ingest.sh >> /var/log/etl/ingest.log 2>&1

# what each piece buys you:
#   flock -n ......... never two runs at once; the second one just exits
#   /opt/etl/... ..... ABSOLUTE path. cron is not in your directory.
#   >> ...log ........ append stdout to a log you can actually read tomorrow
#   2>&1 ............. AND the errors. without this you keep only the good news.
bash · debugging a cron job in 60 seconds
# 1. did cron even try? (Debian/Ubuntu)
grep CRON /var/log/syslog | tail -20

# 2. reproduce cron's environment exactly — this catches ~80% of cases
env -i /bin/bash --noprofile --norc -c '/opt/etl/ingest.sh'

# 3. make the script self-sufficient: cd to its own directory, load its own env
cd "$(dirname "$0")"            # $0 is the script's own path
source /opt/etl/.env             # load secrets explicitly (file must be chmod 600)

Cron's real limits — and when to stop using it

Cron has no memory. It cannot express "run B after A succeeds", it will not retry a failure, it has no backfill, and if the box was down at 02:30 the run is simply lost forever. It also runs in the system timezone, which is why servers are set to UTC and why a job scheduled in local time will fire twice — or not at all — on daylight-saving night. Cron is perfect for one independent, idempotent job on one machine. The moment you need dependencies, retries, or backfills, that is what Airflow (or a systemd timer, for something simple with proper logging) exists for. Reaching for Airflow first is over-engineering; still being on cron when you have twelve interdependent jobs is the other mistake.

14. Downloading files with wget

wget is a file-fetcher. Point it at a URL and it puts the bytes on disk — reliably, resumably, and without you watching.

The division of labour is worth stating plainly, because interviewers ask it: wget downloads, curl converses. wget's defaults are built around "get this file, keep trying, save it, survive a bad connection" — it writes to disk by default, retries automatically, and can resume a half-finished 40 GB transfer. curl's defaults are built around "make this HTTP request and show me exactly what came back" — it prints to stdout, follows nothing unless told, and speaks two dozen protocols with total control over headers. Use wget for the nightly dump; use curl for the API.

bash · wget
wget https://example.com/data/events.csv.gz          # saves as events.csv.gz
wget -O daily.csv.gz "$URL"                          # -O choose the output name
wget -P /mnt/data/raw "$URL"                         # -P choose the directory
wget -q "$URL"                                       # -q quiet — use this in cron
wget -c "$URL"                                       # -c CONTINUE a partial download. the killer flag.
wget --tries=5 --timeout=30 "$URL"                   # retry a flaky server
wget --limit-rate=5m "$URL"                          # don't saturate the pipe on a shared box
wget -N "$URL"                                       # -N only re-download if the server's copy is NEWER
wget -i urls.txt                                     # -i fetch every URL listed in a file
wget --header="Authorization: Bearer $TOKEN" "$URL"  # auth, if you must
bash · the download you would actually put in a pipeline
#!/usr/bin/env bash
set -euo pipefail

URL="https://vendor.example.com/exports/daily.csv.gz"
DEST="/mnt/data/raw/daily_$(date -u +%F).csv.gz"

# -c resume, quiet, retry, and fail loudly if it still doesn't work
wget -c -q --tries=5 --timeout=30 -O "$DEST.part" "$URL"

# verify BEFORE you trust it — a truncated gzip is a silent data bug
gzip -t "$DEST.part"                       # -t = test integrity; fails non-zero if corrupt
sha256sum -c checksums.txt                # if the vendor publishes checksums, CHECK them

mv "$DEST.part" "$DEST"                   # atomic publish (§12)
echo "landed: $DEST ($(du -h "$DEST" | cut -f1))"

15. Querying an API with curl

curl builds an HTTP request, sends it, and hands you back exactly what the server said. Learn the anatomy once and every API on earth is the same shape.

bash · curl, the flags that matter
curl https://api.example.com/health          # GET, prints the body to stdout

curl -s "$URL"                              # -s silent: no progress bar (essential in scripts)
curl -sS "$URL"                             # -sS silent BUT still show real errors
curl -f "$URL"                              # -f FAIL (exit 22) on 4xx/5xx. see the trap above.
curl -L "$URL"                              # -L follow redirects (curl does NOT by default)
curl -o out.json "$URL"                     # -o write to a file instead of stdout
curl -i "$URL"                              # -i include the response headers
curl -I "$URL"                              # -I HEAD only: "is it there? how big?" without downloading

# headers and auth
curl -H "Authorization: Bearer $API_TOKEN" -H "Accept: application/json" "$URL"

# POST some JSON
curl -X POST "$URL" \
     -H "Content-Type: application/json" \
     -d '{"since":"2026-07-12","limit":1000}'

# timeouts and retries — never hit a third-party API without these
curl -fsS --max-time 60 --connect-timeout 10 \
     --retry 3 --retry-delay 5 --retry-all-errors "$URL"

# just the status code, for a health check
curl -s -o /dev/null -w '%{http_code}\n' "$URL"
bash · the shape of a real API pull, with pagination
set -euo pipefail
PAGE=1
rm -f events.ndjson

while :; do
  BODY=$(curl -fsS --max-time 60 --retry 3 \
            -H "Authorization: Bearer $API_TOKEN" \
            "https://api.example.com/v1/events?page=$PAGE&per_page=1000")

  COUNT=$(echo "$BODY" | jq '.items | length')
  [[ "$COUNT" -eq 0 ]] && break            # no more pages — stop

  echo "$BODY" | jq -c '.items[]' >> events.ndjson   # one JSON object per line
  echo "page $PAGE: $COUNT rows" >&2
  PAGE=$(( PAGE + 1 ))
  sleep 1                                    # be a good citizen. respect rate limits.
done

Never put a token on the command line

Anything you type as an argument is visible to every user on the box via ps aux, and it lands in ~/.bash_history forever. Keep secrets in environment variables (-H "Authorization: Bearer $API_TOKEN" — the shell substitutes it, so the token still ends up in the process's argv…) or, better, in a chmod 600 file that curl reads itself: curl -H @auth-header.txt, or use --netrc-file. The rule of thumb: if a secret can appear in ps, a screenshot, or a log line, treat it as leaked.

16. Shaping JSON with jq

curl gives you JSON. Your warehouse wants rows. jq is the missing link, and it is the difference between a one-line ingestion and a Python script.

jq is a filter language for JSON, exactly as awk is one for lines. It streams, so it handles files far bigger than memory, and its output can be raw text, compact one-object-per-line NDJSON, or straight CSV — which is precisely what a bulk loader wants.

bash · jq
curl -s "$URL" | jq .                       # pretty-print (the debugging default)
jq '.items' resp.json                       # one field
jq '.items[] | .id' resp.json               # iterate the array, pull a field
jq -r '.items[].email' resp.json            # -r RAW: no quotes. pipe-friendly.
jq '.items | length' resp.json              # how many rows did we get?
jq '.items[] | select(.status == "failed")' resp.json   # a WHERE clause
jq -c '.items[]' resp.json > events.ndjson   # -c compact: one object per line

# JSON straight to CSV — @csv quotes and escapes properly for you
jq -r '.items[] | [.id, .created_at, .amount] | @csv' resp.json > events.csv

# with a header row
{ echo 'id,created_at,amount'; \
  jq -r '.items[] | [.id, .created_at, .amount] | @csv' resp.json; } > events.csv

# flatten a nested field and give it a default when missing
jq -r '.items[] | [.id, (.user.country // "unknown")] | @csv' resp.json

17. Environment variables and secrets

Every process inherits a small dictionary of strings from its parent. That is the whole mechanism — and it is how configuration and credentials reach your job.

A shell variable (X=1) lives only in that shell. An exported variable (export X=1) is copied into every child process the shell starts afterwards. Inheritance is one-way and only forwards: a child can never change its parent's environment, which is why ./setenv.sh cannot set variables in your shell — you have to source setenv.sh so the commands run in the current shell instead of a child.

bash · environment
TABLE=users                     # shell variable — this shell only
export DBT_PROFILES_DIR=/opt/etl  # exported — children see it too
env | sort                      # everything in my environment
echo "$PATH"                     # where the shell looks for commands
which python3                    # which one will actually run?

source .env                      # run these lines IN my shell (also written as: . .env)
API_TOKEN=abc ./ingest.sh        # set a var for ONE command only
bash · handling secrets like an adult
# a .env file, readable by nobody else
chmod 600 /opt/etl/.env
cat /opt/etl/.env
#   export API_TOKEN=...
#   export PGPASSWORD=...

# load it explicitly at the top of the script — cron will NOT do this for you
set -a; source /opt/etl/.env; set +a   # -a auto-exports everything defined

# keep it out of git, forever
echo '.env' >> .gitignore

# keep a command out of your shell history: prefix it with a space
#   (needs HISTCONTROL=ignorespace, which is the default on most distros)
 curl -H "Authorization: Bearer sk-real-token" "$URL"

18. Disk, memory, and CPU: is the box the problem?

Data jobs die for four reasons: out of disk, out of memory, out of file descriptors, or out of patience. Each has a command that tells you in one second.

bash · the four checks, in the order you should run them
# 1. DISK — the most common cause of a 3am failure
df -h                          # which filesystem is full?
df -i                          # INODES. you can have free GBs and still fail:
                               # millions of tiny files exhaust the inode table.
du -sh --max-depth=1 /mnt/data | sort -h | tail   # who ate it?
lsof +L1                       # deleted-but-still-open files (see §03)

# 2. MEMORY
free -h                        # look at "available", not "free" — cache is not waste
dmesg -T | grep -i 'killed process'   # was my job OOM-killed? (exit code 137)

# 3. CPU / LOAD
uptime                         # load average: 1, 5, 15 min
                               # rule of thumb: load > number of cores = things are queuing
nproc                          # how many cores do I have?
top -o %MEM                    # sort by memory hog

# 4. LIMITS
ulimit -n                      # max open files. "Too many open files" = this is too low.

Exit code 137 means the kernel killed you

If a job dies with 137 and no stack trace, it was not a bug in your code — 137 = 128 + 9 = killed by SIGKILL, and the killer was almost certainly the kernel's OOM killer reclaiming memory (or a container hitting its memory limit — the exact same mechanism, which is why this shows up constantly in Kubernetes). Confirm with dmesg -T | grep -i kill. The fix is never "retry": it is to process the data in chunks, raise the limit, or stream it instead of loading it. Similarly 143 = 128 + 15 = SIGTERM, meaning something asked it to stop politely — usually a scheduler timeout.

19. Working on a remote box: ssh, rsync, tmux

The machine your pipeline runs on has no screen. Everything you do to it, you do through a pipe.

bash · ssh and moving files
ssh hao@10.0.4.12                       # log in
ssh-keygen -t ed25519                    # make a key pair (do this once)
ssh-copy-id hao@10.0.4.12                # install your public key — no more passwords
ssh etl-prod 'df -h /mnt/data'          # run ONE command and come straight back

# ~/.ssh/config turns the above into: ssh etl-prod
#   Host etl-prod
#     HostName 10.0.4.12
#     User hao
#     IdentityFile ~/.ssh/id_ed25519

scp report.csv etl-prod:/tmp/            # simple copy, one shot
rsync -avz --progress /local/data/ etl-prod:/mnt/data/   # the grown-up version:
#   -a archive (perms, times)  -v verbose  -z compress
#   only sends what CHANGED, and --partial resumes an interrupted transfer.
#   for anything over a few hundred MB, always rsync, never scp.

# port-forward a remote UI (e.g. Spark) to your laptop's browser
ssh -L 4040:localhost:4040 etl-prod      # then open http://localhost:4040
bash · tmux — so your six-hour backfill survives your wifi
tmux new -s backfill        # start a named session
./backfill.sh 2020-01 2026-07  # start the long job inside it
# press Ctrl-b then d  → detach. the job keeps running. close your laptop.

tmux ls                     # what sessions exist?
tmux attach -t backfill     # tomorrow: walk right back in, output still scrolling

20. Worked example: a nightly ingestion job

Everything on this page, in one file that you could genuinely put into production tonight.

/opt/etl/ingest.sh
#!/usr/bin/env bash
#
# ingest.sh — pull yesterday's events from the API, land them as CSV.
# cron: 30 2 * * 1-5  flock -n /tmp/ingest.lock /opt/etl/ingest.sh >> /var/log/etl/ingest.log 2>&1

set -euo pipefail                              # §12 — fail fast, fail loudly

# ── config ────────────────────────────────────────────────────────────────
cd "$(dirname "$0")"                            # §13 — never trust cron's cwd
set -a; source /opt/etl/.env; set +a             # §17 — $API_TOKEN (file is chmod 600)

DAY="$(date -u -d yesterday +%F)"              # always UTC on a server
OUT_DIR="/mnt/data/raw/events"
FINAL="$OUT_DIR/events_$DAY.csv"
API="https://api.example.com/v1/events?date=$DAY&per_page=1000"

# ── setup ─────────────────────────────────────────────────────────────────
TMP="$(mktemp)"
trap 'rm -f "$TMP"' EXIT                        # §12 — cleanup no matter how we leave
mkdir -p "$OUT_DIR"

log() { echo "[$(date -u +%FT%TZ)] $*"; }         # timestamped, so tomorrow's you can read it

# ── 0. skip if we already have it (idempotent re-runs) ───────────────────
if [[ -f "$FINAL" ]]; then
  log "$FINAL already exists — nothing to do"
  exit 0
fi

# ── 1. pull, with retries and a hard timeout ─────────────────────────────
log "pulling $DAY"
curl -fsS \
     --max-time 120 --connect-timeout 10 \
     --retry 3 --retry-delay 5 --retry-all-errors \
     -H "Authorization: Bearer $API_TOKEN" \
     "$API" > "$TMP.json"
#    -f is what makes a 500 an ERROR instead of a file full of HTML (§15)

# ── 2. reshape JSON → CSV ────────────────────────────────────────────────
{
  echo 'event_id,user_id,occurred_at,amount'
  jq -r '.items[] | [.id, .user_id, .occurred_at, .amount] | @csv' "$TMP.json"
} > "$TMP"

# ── 3. verify BEFORE publishing — this is the step people skip ───────────
ROWS=$(( $(wc -l < "$TMP") - 1 ))          # minus the header
if [[ "$ROWS" -lt 1 ]]; then
  log "FATAL: 0 rows for $DAY — refusing to publish an empty file" >&2
  exit 1                                       # non-zero → the alert fires
fi
awk -F, 'NR > 1 && NF != 4 { print "bad row at line " NR > "/dev/stderr"; bad++ }
         END { exit (bad > 0) }' "$TMP"      # §08 — cheap schema check

# ── 4. publish atomically — readers never see a half-written file ────────
mv "$TMP" "$FINAL"                             # §03 — same filesystem = atomic
chmod 644 "$FINAL"                             # §05 — the loader user must read it

log "OK — $ROWS rows → $FINAL"
exit 0

Why this script is interview-grade

It is idempotent (re-running it is safe and cheap — step 0). It cannot run twice at once (flock, in the cron line). It cannot half-publish (temp file plus atomic mv). It cannot publish garbage (row count and field count checked first). It cannot fail silently (set -e, non-zero exit, timestamped log with stderr folded in). And it cannot leak (token in a 600 file, never in argv). Every one of those is a question someone will ask you about a pipeline — and each answer here is one line of shell.

21. The troubleshooting playbook

Symptom on the left, actual cause in the middle, the command that proves it on the right. These ten cover most of what will happen to you.

1
"Permission denied" — but the file looks fine. You are missing x on a parent directory, not on the file. → namei -l /full/path/to/file
2
"command not found" — but only under cron. Cron's $PATH is a stub and it never sources your .bashrc (§13). → Use absolute paths, or set PATH= at the top of the crontab.
3
"No space left on device" — but du shows plenty free. Either a deleted-but-open file is still holding blocks, or you have run out of inodes. → lsof +L1 and df -i
4
The job "succeeded" but the output is empty. A pipeline stage failed and the last stage still exited 0. → Add set -o pipefail (§12), and check the row count before publishing.
5
The API call "worked" but the file contains HTML. You got a 404 or 500 and curl exited 0 anyway. → Always curl -f (§15).
6
Killed at 3am with no error, exit code 137. The OOM killer took it. → dmesg -T | grep -i 'killed process', then chunk or stream the data (§18).
7
bad interpreter: /bin/bash^M The script was saved with Windows line endings. → dos2unix ingest.sh, or sed -i 's/\r$//' ingest.sh.
8
"Argument list too long" Your glob matched 200,000 files and the shell cannot pass them all. → Pipe through xargs: find . -name '*.csv' -print0 | xargs -0 rm
9
Two copies of the job are fighting over the output. The run now takes longer than the schedule interval. → flock -n in the cron line (§12).
10
Counts are all 1 when you know there are duplicates. You ran uniq -c without sorting first — it only collapses adjacent lines (§09). → sort | uniq -c

Glossary

Kernel
The program that owns the hardware. Schedules processes, manages memory, and is the only thing that touches the disk or network. You reach it through system calls.
Shell
An ordinary program (usually bash) that reads your command line, expands globs and variables, finds the binary on $PATH, and starts it. Not the operating system.
Process
A running program: its own memory, a numeric PID, an owner, a working directory, and a table of open file descriptors.
File descriptor
A small integer a process uses to refer to something it has open. 0 = stdin, 1 = stdout, 2 = stderr; everything else is a file, socket, or pipe.
Inode
The actual file record on disk — permissions, owner, size, block pointers. A filename is only a directory entry pointing at an inode, which is why hard links and "deleted but still open" files behave as they do.
Symlink
A file whose contents are a path to another file. Breaks if the target moves. The basis of the atomic "swap the current pointer" publish trick.
Mount
Attaching a device (a disk, a volume, a network share) at a directory in the single tree. There are no drive letters; there is only /.
Permission bits
Nine bits: read/write/execute for owner, group, and others. Read as octal by adding r=4, w=2, x=1 per triad. On a directory, x means "may pass through".
umask
The mask subtracted from the default permissions of every new file your processes create. Usually 022, giving files 644 and directories 755.
Pipe
A kernel buffer connecting one process's stdout to the next one's stdin. Both run at the same time and data streams through, so a pipeline never has to hold the whole file in memory.
Glob
A wildcard pattern (*.csv) expanded by the shell before the command runs. Quote it if you want the command to see the star itself.
Shebang
The #!/usr/bin/env bash first line that tells the kernel which interpreter should execute this file.
Exit code
The number a process returns. 0 means success and everything else means failure. 137 = OOM-killed, 143 = terminated. Check it with $?.
Signal
A message sent to a process. SIGINT (2, Ctrl-C) and SIGTERM (15, kill) can be caught and cleaned up after; SIGKILL (9) cannot, which is why kill -9 leaves half-written files.
Environment variable
A string in the dictionary every process inherits from its parent. export puts it there. Cron gives you almost none of them — the cause of most "works for me" bugs.
Cron
A daemon that runs commands on a schedule described by five fields (minute, hour, day-of-month, month, day-of-week). No dependencies, no retries, no backfill, no memory of missed runs.
flock
A lock on a file used to guarantee only one copy of a job runs at a time. flock -n makes the second copy exit instead of queueing.
Idempotent
Safe to run twice: a second run either does nothing or produces exactly the same result. The single most valuable property a scheduled data job can have.
Atomic publish
Writing to a temporary file and only mv-ing it into place when it is complete and verified — so a reader can never see a half-written file. Only atomic within one filesystem.
Load average
The average number of processes wanting to run, over 1/5/15 minutes. Compare it to your core count (nproc): consistently higher means work is queuing.
OOM killer
The kernel's last resort when memory runs out: it picks a process and sends SIGKILL. Your job dies with exit code 137 and no stack trace.
