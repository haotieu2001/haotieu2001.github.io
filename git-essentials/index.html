Git Essentials for Data Engineers

Git is the tool that remembers every version of your code, lets a whole team work on the same project without stepping on each other, and makes "undo" possible even weeks later. This is a from-zero reference: it assumes no prior experience, builds Git's mental model one piece at a time, walks through every command you'll actually use, then covers the team workflows (Gitflow, GitHub Flow, trunk-based) and the parts that matter specifically for data engineering — dbt, notebooks, CI/CD, and versioning data itself.

~35 min read · Essentials · Reference

❯ The one idea (read this first)

Git is a time machine + collaboration tool for a folder of files. Every time you reach a good stopping point you take a commit — a labelled snapshot of the entire project at that moment. Git keeps every snapshot forever, linked in a chain, so you can look back, compare, or jump to any past version. A branch is just a lightweight pointer to one snapshot, which lets several people (or several ideas) move forward in parallel and later merge their work back together. Almost everything else in Git is a variation on those three words: commit, branch, merge.

01. What Git is & why it exists

Imagine writing an important document and saving copies named report_final.doc, report_final_v2.doc, report_final_REALLY_final.doc. That's version control done by hand — and it falls apart the moment a second person joins. Git is the tool that does this properly: it tracks every change to a set of files, remembers who changed what and why, and lets many people work on the same project at once without overwriting each other.

Concretely, Git gives you four superpowers over a plain folder:

  • History — every saved version is kept forever. You can see exactly what the project looked like last Tuesday, and what changed since.
  • Undo — broke something? Roll any file, or the whole project, back to a known-good version.
  • Collaboration — many people edit the same files in parallel; Git stitches their changes together and flags the rare spots where two people changed the same line.
  • Accountability — every change is stamped with an author, a timestamp, and a message explaining why — an invaluable record when you're debugging "why is this pipeline suddenly broken?" six months later.

"Distributed": the one word that defines Git

Older version-control tools (like SVN) were centralized: there was one server holding the history, and you needed a network connection to do almost anything. Git is distributed — when you clone a project, you get a complete copy of the entire history on your own machine. You can commit, branch, look at history, and compare versions entirely offline. You only need the network for the moments you deliberately share: pushing your work up, or pulling other people's work down.

🔑 Git ≠ GitHub

Git is the version-control tool that runs on your computer — free, open-source, works with no internet. GitHub (and GitLab, Bitbucket, Azure Repos) is a website that hosts a shared copy of your Git project in the cloud and adds team features on top: pull requests, code review, issue tracking, and automation. You can use Git with no GitHub at all; GitHub is just the most popular place to put the shared copy. The relationship is like a camera (Git) versus a photo-sharing site (GitHub).

02. How Git thinks (the mental model)

Most early confusion with Git comes from skipping its mental model and memorising commands instead. Spend five minutes here and the commands stop feeling arbitrary. There are two ideas: commits are snapshots, and your files pass through three areas on the way into one.

A commit is a snapshot, not a "change"

When you picture "a commit," it's tempting to imagine Git saving only your latest edit — the handful of lines you just changed. It actually does something simpler to reason about: every commit saves a complete snapshot of all your files, exactly as they stood at that instant, with a label attached (a message, your name, and the time). Think of each commit as a photograph of the whole project — not a note about what moved.

Each commit also stores a pointer back to the one right before it — its parent. Follow those pointers backwards and you walk the entire history one snapshot at a time, like stepping back through the frames of a film. That chain of linked snapshots is your project's history.

Every commit has a unique ID — a 40-character hash like 9f2c1ab… (you'll usually see just the first 7, e.g. 9f2c1ab). That hash is a fingerprint of the commit's exact contents, so it's impossible to alter history without it being obvious.

The three areas every change passes through

This is the diagram to burn into memory. A file moves through three places before it's safely in history:

1
Working directory — the real files on disk that you open and edit. Git sees them as "modified" but hasn't recorded anything yet.
2
Staging area (also called the index) — a holding pen. With git add you pick exactly which changes go into the next snapshot. This lets you commit some of your edits and leave others for a later commit — the key to making small, focused commits.
3
Repository — the .git folder. git commit takes everything in the staging area and writes it into history permanently as a new commit.

💡 Why a staging area at all?

It seems like an extra step, but it's what lets you craft clean history. Say you fixed a bug and renamed a variable in the same editing session. You can git add just the bug-fix files, commit them with the message "fix null check," then git add the rest and commit "rename for clarity." Two tidy commits instead of one muddled one — much easier for a reviewer (and future-you) to read.

HEAD — the "you are here" marker

HEAD is Git's name for "the commit you're currently sitting on." Normally HEAD points at the latest commit of your current branch. When you make a new commit, HEAD (and the branch) move forward to it. When you switch branches, HEAD moves to point at that other branch's latest commit. Whenever a command mentions HEAD, read it as "where I am right now." (One gotcha: if you check out a raw commit hash instead of a branch, you land in a "detached HEAD" — you're sitting on a commit that no branch points to, so new commits there can be lost. To keep them, make a branch: git switch -c new-branch.)

03. The everyday commands

Ninety percent of daily Git is a handful of commands. Here they are in the order you'd naturally use them, each with what it does and why.

Starting a project: init or clone

bash
# Option A: turn an existing folder into a Git project
git init                 # creates the hidden .git folder; history starts here

# Option B: copy an existing project (e.g. from GitHub) onto your machine
git clone https://github.com/acme/data-pipelines.git
# downloads the full project AND its entire history

Use init when starting fresh; use clone when joining a project that already lives somewhere. clone is what you'll do most often at work.

Configure who you are (once per machine)

bash
git config --global user.name  "Hao Tieu"
git config --global user.email "hao@example.com"
# every commit you make gets stamped with this name + email

The core loop: status → add → commit

This three-step cycle is the heartbeat of Git. You'll run it dozens of times a day.

bash
git status               # what's changed? what's staged? — run this constantly

git add transform.sql    # stage one file for the next snapshot
git add .               # …or stage everything that changed

git commit -m "Add revenue-by-region transform"
# take the snapshot. -m is the message explaining WHY.

🧭 git status is your best friend

When you're unsure what state things are in — what's modified, what's staged, what branch you're on, whether you're ahead of the server — run git status. It also suggests the next command for common situations. There's no such thing as running it too often.

Looking at history & differences

bash
git log                       # full history, newest first
git log --oneline --graph     # compact, with a visual branch graph

git diff                      # unstaged changes (working dir vs last commit)
git diff --staged             # what you've staged but not yet committed
git show 9f2c1ab               # everything a specific commit changed

git diff answers "what exactly did I change?" and git log answers "how did we get here?" — together they're how you understand any codebase's evolution.

.gitignore — telling Git what to not track

Some files should never go into Git: secrets, huge data files, and machine-generated junk. You list patterns for them in a file named .gitignore at the project root, and Git pretends those files don't exist.

.gitignore
# secrets — NEVER commit these
.env
*.key
credentials.json

# data — too big / not code (version it elsewhere — see §10)
*.csv
*.parquet
/data/

# machine-generated clutter
__pycache__/
.ipynb_checkpoints/
.DS_Store

⚠️ .gitignore only affects untracked files

A crucial catch: ignoring a file that Git is already tracking does nothing — Git keeps tracking it. So if you accidentally committed a big data.parquet (or a secret) and only added the pattern afterwards, you must stop tracking it explicitly: git rm --cached data.parquet, then commit. (The --cached flag removes it from Git but leaves the file on your disk.)

⚠️ The cardinal sin: committing secrets

Because Git keeps history forever, a password or API key committed once stays in the history even after you "delete" it in a later commit — anyone who clones the repo can dig it out. Add secrets to .gitignore before your first commit. If one slips through, treat it as leaked: rotate the credential immediately, then scrub history with a tool like git filter-repo.

04. Branches — just movable pointers

Branches sound advanced but are mechanically trivial — and that triviality is Git's secret weapon. A branch is simply a lightweight, movable label pointing at one commit. That's the whole thing. Creating one writes a tiny file with a commit ID in it; that's why Git branching is instant and cheap, where older tools made branching a heavyweight ordeal.

Why branch? So you can build a new feature, experiment, or fix a bug without touching the known-good main code. You branch off, do messy work in isolation, and only fold it back in when it's ready and reviewed. The main line stays stable the whole time.

The branch commands

bash
git branch                        # list branches; * marks the one you're on
git switch -c feature/revenue     # create a new branch AND move onto it
#  (older syntax, still everywhere: git checkout -b feature/revenue)

# …make commits on the branch…

git switch main                   # hop back to main (your work is safe on the branch)
git merge feature/revenue         # fold the branch's work into main
git branch -d feature/revenue     # delete the branch once it's merged

🔀 switch/restore vs the old checkout

For years one overloaded command, git checkout, did two unrelated jobs: switching branches and restoring files. Modern Git split them into git switch (change branch) and git restore (discard file changes) — clearer and safer. You'll still see checkout everywhere in older docs and tutorials; just know it's the same machinery.

What "switching branches" actually does

When you git switch to another branch, Git moves HEAD to that branch and rewrites the files in your working directory to match that branch's latest commit. Your other branch's work doesn't vanish — it's safely stored in commits, waiting where you left it. Switching feels like teleporting between parallel versions of the project.

05. Remotes — sharing with a team

Everything so far happens on your laptop. A remote is a shared copy of the project living somewhere else — usually on GitHub. It's the meeting point where the team's work comes together. The default remote is conventionally named origin.

bash
git remote -v                       # show the remote(s) this project is linked to

git push                            # upload your new commits to origin
git push -u origin feature/revenue  # first push of a new branch (-u sets the upstream, so later just `git push`)

git pull                            # download teammates' new commits AND merge them in
git fetch                           # download them but DON'T merge yet (just look)

💡 fetch vs pull

git fetch = "show me what's new on the server, but don't change my files yet." git pull = fetch plus immediately merge those changes into your current branch. pull is the convenient default; fetch is the cautious one when you want to inspect incoming changes before letting them touch your work. Fetched commits land in a read-only mirror called origin/main (your local snapshot of the server's branch), kept separate from your own main until you merge.

The typical day, end to end

bash
git switch main && git pull            # 1. start from the latest shared code
git switch -c feature/add-dq-checks    # 2. branch off for your task
# … edit files …
git add . && git commit -m "Add row-count check"  # 3. commit as you go
git push -u origin feature/add-dq-checks # 4. share your branch
# 5. open a Pull Request on GitHub → review → merge (see §09)

06. Merge vs rebase (& resolving conflicts)

Eventually two lines of work need to become one. Git has two ways to do that — merge and rebase — and understanding the difference (plus how to handle conflicts) is the thing that separates a nervous beginner from a confident user.

Merge: join two histories with a "knot" commit

A merge ties two branches together by creating a special merge commit that has two parents — one from each branch. Nothing is rewritten; both histories are preserved exactly as they happened, and the merge commit records where they reunited.

A special easy case is the fast-forward: if main hasn't moved at all since you branched off, Git can simply slide the main pointer forward to your branch's tip — no merge commit needed, because there's nothing to reconcile.

Rebase: replay your commits onto a new base

A rebase takes your branch's commits, sets them aside, moves your branch's starting point to the tip of main, and then re-applies your commits one by one on top. The result is a perfectly straight line of history — as if you'd started your work from the latest main all along. No merge commit, no fork in the graph.

MergeRebase
History shapeBranching, with merge commitsFlat, linear
TruthfulnessRecords exactly how it happenedRewrites into a tidier story
Commit IDsUnchangedChanged (commits are recreated)
Best forMerging finished work into a shared branchTidying your own branch before sharing

⚠️ The golden rule of rebase

Never rebase commits that other people have already pulled. Because rebase recreates commits with new IDs, rebasing shared history makes everyone else's copy disagree with the server — a painful mess. Rule of thumb: rebase freely on your own un-pushed branch to clean it up; once work is shared, merge instead.

💡 Rebased a branch you'd already pushed? You'll need to force-push

Rebasing or --amend-ing rewrites commit IDs, so a normal git push is rejected as "non-fast-forward." On your own feature/PR branch this is fine and expected — push with git push --force-with-lease. Always prefer --force-with-lease over a bare --force: it refuses to push if someone else has added commits to the branch since you last fetched, so you can't silently clobber a teammate's work.

# "non-fast-forward" = your history diverged from the remote, instead of just # adding on top of it — so Git won't push without an explicit force. you last saw the remote at C, then rebased C → C'; meanwhile a teammate pushed D: A ─ B ─ C ─ D origin/feature (you don't have D yet) A ─ B ─ C' your rebased local branch git push --force → remote becomes A ─ B ─ C' … D is gone 💥 git push --force-with-lease → "remote moved past C?" rejected ✓ — D is safe

Merge conflicts — what they are and how to fix them

A conflict happens when two branches changed the same lines of the same file in different ways. Git can't know which version is right, so it stops and asks you. This is normal — not an error you did wrong. Git marks the clashing spot inside the file like this:

transform.sql — a conflict
<<<<<<< HEAD            ← your version (current branch)
SELECT revenue, region
=======                  ← divider
SELECT revenue, country
>>>>>>> feature/geo      ← the incoming version

To resolve it, you edit the file by hand: keep whichever lines are correct (maybe a blend of both), and delete the <<<, ===, >>> marker lines entirely. Then:

bash
# 1. open each conflicted file, edit to the correct final version
git add transform.sql        # 2. mark it resolved by staging it
git commit                  # 3. finish the merge (or: git rebase --continue)

# escape hatch — bail out and go back to before the merge/rebase:
git merge --abort           # (or git rebase --abort)

Steering a paused merge or rebase

When Git can't auto-combine, it pauses mid-operation and hands control to you — you're neither finished nor reverted until you say so. The difference between merge and rebase explains the steering commands: a merge combines everything in one step, so it pauses at most once. A rebase replays your commits one at a time onto the new base, so it can pause at each commit — which is exactly why rebase has a --continue and merge doesn't.

  • git rebase --continue — after you've fixed the current conflict and git add-ed the files, this finishes re-creating that commit and moves on to replay the next one. If that one conflicts too, it pauses again — resolve, add, --continue, repeat until done. (git rebase --skip instead drops the conflicting commit entirely.)
  • git rebase --abort — the rebase escape hatch: cancel the whole thing and restore your branch to its original commits, exactly as before you started. Because rebase saved your starting point, nothing is lost.
  • git merge --abort — the same escape hatch for a merge: throw away the half-done merge and return to the pre-merge state, with the conflict markers removed.

Notice there's no --continue for finishing a merge: once you've resolved a conflicted merge you simply git add the files and git commit, and that commit becomes the merge commit. (Modern Git also accepts git merge --continue as an alias for that final commit.)

CommandWhenWhat it does
git rebase --continueAfter resolving a rebase conflictFinish this commit, replay the next one
git rebase --abortMid-rebase, any pointCancel the whole rebase; restore your original branch
git merge --abortMid-merge, conflictedCancel the merge; restore the pre-merge state
git add + git commitAfter resolving a merge conflictComplete the merge (creates the merge commit)

The reassuring part: a paused merge or rebase always offers two doors — resolve and proceed, or abort and land back exactly where you started. You can never get irreversibly stuck.

✅ Conflicts get smaller the more often you sync

Big, scary conflicts come from branches that drifted apart for weeks. Pull main into your branch (or rebase onto it) frequently — daily — and each reconciliation is tiny. Small, short-lived branches are the single best defence against conflict pain.

07. Undoing things safely

A huge part of Git's value is that mistakes are recoverable. The trick is knowing which undo to reach for — they differ in how much they touch, and whether they're safe on shared history. (In the commands below, HEAD~1 means "one commit before HEAD," HEAD~2 two before, and so on.)

You want to…CommandNotes
Throw away un-staged edits to a filegit restore fileReverts the file to the last commit. The edits are gone — use with care.
Un-stage a file (keep the edits)git restore --staged fileMoves it out of the staging area; your changes stay in the working dir.
Fix the last commit's message or contentsgit commit --amendReplaces the last commit. Don't amend commits you've already pushed.
Undo a commit but keep the changes to re-dogit reset --soft HEAD~1Moves the branch back one commit; changes return to staging.
Undo a commit and discard its changesgit reset --hard HEAD~1Destructive. Wipes the commit and its edits. Be sure.
Undo a commit that's already publicgit revert <hash>Creates a new commit that cancels the old one. Safe to share.
Stash work temporarily without committinggit stash / git stash popPockets your changes so you can switch tasks, then brings them back.

reset vs revert: the key safety distinction

Both "undo" a commit, but very differently:

  • reset rewrites history — it moves your branch pointer backwards as if the commit never happened. Great for cleaning up local, un-pushed work. Dangerous on shared branches (it's history-rewriting, same hazard as rebase).
  • revert adds history — it makes a brand-new commit that undoes the changes of an earlier one. The original stays in the record. Because it doesn't rewrite anything, it's the safe way to undo something already on the shared remote.

The reflog — Git's undo for your undos

Terrified of losing work to a bad reset --hard? Git quietly records every place HEAD has been in the reflog. Even "deleted" commits usually linger there for weeks, so you can almost always get back:

bash
git reflog                  # a timeline of everywhere HEAD has been
git reset --hard HEAD@{2}   # jump back to how things were 2 moves ago

✅ The reassuring truth

Once something is committed in Git, it's very hard to truly lose — even after a scary reset, the reflog usually has your back. The things that are easy to lose are changes you never committed (un-staged edits wiped by restore or reset --hard). Lesson: commit early, commit often, and your work is safe.

08. Git workflows & Gitflow

The commands so far are the mechanics. A workflow (or "branching strategy") is the team agreement about how to use them: which branches exist, what each is for, and how work flows from idea to production. Picking one keeps a team coordinated. Here are the four you'll actually meet, from simplest to most structured.

① Feature-branch workflow (the foundation)

The baseline every modern workflow builds on: main always holds working code; all new work happens on short-lived branches that merge back via review. Never commit straight to main. Everything below is a variation on this theme.

② GitHub Flow — simple & continuous

The lightweight favourite for web apps and services that deploy often. There's exactly one long-lived branch (main), which is always deployable. For any work you branch off main, open a pull request (a request to review and merge your branch — see §09), get it reviewed and CI-tested (automated tests run on every push), and merge back — then deploy. That's it.

  • Pros: easy to learn, fast, perfect for continuous deployment.
  • Cons: relies on strong automated tests — without them main can break. No built-in concept of "supported old releases."

③ Gitflow — structured, release-oriented

The classic heavyweight model (coined by Vincent Driessen in 2010), built for software shipped as versioned releases (v1.0, v1.1…) where multiple versions may be supported at once. It uses several long-lived and supporting branches, each with a strict role:

BranchRoleLifetime
mainProduction-ready code only; every commit is a tagged releasePermanent
developThe integration branch where finished features accumulatePermanent
feature/*One per new feature; branches off and back into developShort-lived
release/*Freeze a batch of finished features and stabilise it for shipping — bug-fixes only, no new featuresShort-lived
hotfix/*Emergency fix branched off main, merged to both main & developShort-lived

🔎 What a release/* branch is really for

It fills the gap between "all the features for this version are done" and "it's actually shipped." When develop has gathered enough for, say, v1.4, you cut a release/1.4 branch off it. From that moment the release is frozen to new features — only bug-fixes, version-number bumps, and last-minute polish are allowed in. That freeze is the point: testers can stabilise a fixed target instead of chasing a branch that keeps changing, while the rest of the team carries on adding new features to develop for the version after this one. When it's solid, release/1.4 is merged into main and tagged v1.4 (the shipped version) and merged back into develop so those bug-fixes aren't lost.

A hotfix/* branch is the same idea for an emergency: a fix to code that's already live, so it branches off main (not develop), then merges back into both main and develop.

  • Pros: strict control, clear separation of in-progress vs released code, room for parallel release maintenance. Great for desktop/mobile apps, libraries, and on-prem products with scheduled releases.
  • Cons: complex; the many branches and merge steps slow things down and fit poorly with continuous delivery. Often overkill for a web service or a data platform that deploys continuously.

④ Trunk-based development — fast & CI-native

The model behind most high-performing teams and a prerequisite for true continuous integration. Everyone commits to a single trunk (main) in small, frequent increments — often several times a day — through very short-lived branches (hours, not days). Unfinished features are hidden behind feature flags — runtime on/off switches that let half-built code ship to main while staying invisible to users — rather than long branches, so the trunk is always releasable.

  • Pros: minimal merge pain (everyone's always near the tip), continuous integration in the truest sense, fastest delivery.
  • Cons: demands excellent automated testing and CI discipline — without guardrails the trunk gets messy. Requires team maturity.

Which one should you use?

WorkflowLong-lived branchesBest when…Release style
GitHub Flow1 (main)Web apps/services, small-to-mid teams, deploy oftenContinuous
Gitflow2 (main+develop)Versioned products, scheduled releases, multiple supported versionsPeriodic / versioned
Trunk-based1 (main)Mature teams with strong CI/CD chasing maximum velocityContinuous
GitLab Flow1 + environment branchesYou want GitHub Flow but with explicit staging/production branchesContinuous w/ stages

🧭 The honest recommendation

For most data teams, start with GitHub Flow (or GitLab Flow if you want explicit dev/staging/prod branches that map to your data environments). It's simple and pairs naturally with PR review and CI. Reach for Gitflow only if you genuinely ship versioned releases that need parallel maintenance. Aim for trunk-based as your team's testing and automation mature — it's where velocity peaks. The biggest mistake is adopting Gitflow's complexity for a project that deploys continuously.

09. Pull requests & code review

A pull request (PR) — called a "merge request" on GitLab — is where collaboration actually happens. It's a GitHub feature (not a Git command) that says: "here's the work on my branch; please review it and merge it into main." It turns a raw git merge into a reviewed, discussed, tested gate.

The PR lifecycle, step by step:

1
Push your branch and open a PR on GitHub, targeting main. Write a clear title and a description of what changed and why.
2
Automated checks run (CI). Tests, linters, and builds fire automatically. A red ✗ blocks the merge until fixed.
3
Teammates review. They read the diff, leave inline comments, ask questions, and request changes. You push more commits to address them — the PR updates live.
4
Approve & merge. Once approved and green, the PR is merged into main. The feature branch is then deleted.

💡 Three merge buttons, briefly

GitHub offers three ways to merge a PR: a merge commit (keeps every commit + adds a knot), squash and merge (collapses the whole branch into one tidy commit on main — very popular for keeping history clean), and rebase and merge (replays your commits onto main for a linear history with no merge commit). When unsure, squash and merge is the friendliest default.

✅ What makes a PR easy to review

Keep it small (a few hundred lines beats a few thousand), focused on one logical change, well-described, and green on CI. A reviewer can give a thoughtful read to 200 lines; faced with 2,000 they'll rubber-stamp it. Small PRs get better review and merge faster — the rare win-win.

10. Git for data engineers

Everything above applies to any code. But data engineering has its own quirks — SQL and pipeline configs, notebooks, environments that mirror branches, and the big one: Git is for your code, not your data. This section covers what's different.

What you do version in a data project

  • Transformation code — dbt models, SQL, PySpark/Python jobs, Airflow/Dagster DAGs.
  • Configuration — pipeline YAML, schema definitions, environment configs, IaC (Terraform).
  • Tests & data-quality checks — dbt tests, Great Expectations suites.
  • Notebooks — but with care (see below).

dbt: the model citizen of Git in data

dbt made software engineering practices mainstream in analytics, and it's built around Git. A typical flow: each analyst branches off main, edits SQL models, opens a PR; CI spins up the models in a temporary schema and runs the tests; on merge, the changes deploy to production. Branches commonly map to environments — e.g. a dev branch builds into a dev schema, main into prod — so your Git branches commonly correspond to your data environments. (In practice it's looser than a strict 1:1: developers often share a dev target with per-user schemas, CI builds each PR into a throwaway schema, and prod is a scheduled job on main.) This is exactly where GitLab Flow's environment branches shine.

Notebooks: the awkward case

Jupyter/Databricks notebooks (.ipynb) are stored as JSON that mixes your code with cell outputs, images, and an execution counter that changes every run. That wrecks Git: diffs are unreadable, and merge conflicts are common and ugly — if two people just run the same notebook, every cell's execution count clashes, and Git's conflict markers can even make the JSON invalid so the notebook won't open.

The pain

You open a PR changing two lines of notebook code, and the diff shows hundreds of changed lines of output JSON and execution counts. Reviewers can't see what you actually did, and merging triggers conflicts in machine-generated noise.

The fixes

(a) Use nbstripout — a Git filter that strips outputs and execution counts before committing, so Git tracks only the code (note: each collaborator must enable it locally, and stripped outputs won't live in the repo). (b) Use a rich-diff tool like ReviewNB to review notebook PRs visually. (c) On Databricks, use Git folders / Repos, which store notebooks as source files and integrate Git directly into the workspace. (d) Best of all: move reusable logic out of notebooks into plain .py modules that version and review cleanly, keeping notebooks thin.

CI/CD for data pipelines

The same PR-and-CI gate from §09, applied to data. On a pull request, automation typically: lints SQL, compiles/parses the pipeline, runs unit tests and a sample data-quality run, and (for dbt) builds models in a throwaway schema. On merge to main, a CD step deploys — via the Databricks CLI / Asset Bundles, GitHub Actions calling the Databricks REST API, or your orchestrator picking up the new code. The branch you merge into decides which environment gets the deploy.

The big one: Git versions code, not data

New data engineers often ask "can I just put my datasets in Git?" — and the answer is a firm no. Git was designed for small text files. It chokes on large binary data: a multi-gigabyte Parquet file bloats the repo, every clone becomes enormous, and Git can't meaningfully diff or merge it. Data needs its own versioning tools built for scale:

ToolWhat it versionsHow it works
Delta Lake "time travel"A single table's historyIts transaction log lets you query a past version by number (VERSION AS OF 5) or time (TIMESTAMP AS OF '…'). Not unbounded, though — old versions expire once VACUUM / log retention removes them.
lakeFSAn entire data lake / object storeAdds Git-like branches, commits & merges over S3/ADLS/GCS — branch your whole lake, validate, then merge
DVCML datasets & models, alongside codeStores small pointer files in Git; the actual large files live in object storage, linked by hash

🔑 The mental split that ties it together

Code → Git. Data → Delta/lakeFS/DVC. Your transformation logic lives in Git and flows through branches, PRs, and CI like any software. The data those transforms produce is versioned separately by tools built for big files. Keeping that boundary clear is the core of doing version control right in data engineering — put *.parquet and *.csv in .gitignore and reach for the data tools instead.

11. Best practices & commit style

  • Commit small and often. One logical change per commit. Small commits are easy to review, easy to understand later, and easy to revert if one turns out wrong.
  • Write meaningful messages. The message explains why, not just what. "Fix bug" is useless; "Handle null customer_id in revenue join" is gold six months later.
  • Branch for everything; protect main. Never commit directly to main. Turn on branch protection so changes must arrive via reviewed PRs that pass CI.
  • Pull/sync frequently. Integrate others' work daily so conflicts stay tiny and you're never building on stale code.
  • Keep branches short-lived. Hours-to-days, not weeks. Long branches drift far from main and become merge nightmares.
  • Never commit secrets or big data. .gitignore them from day one (§03, §10).
  • Don't rewrite shared history. Rebase/amend/reset only on your own un-pushed work; use revert to undo anything public (§07).

Conventional Commits — a tidy message format

Many teams adopt the Conventional Commits convention: prefix each message with a type. It makes history scannable and can even auto-generate changelogs and version numbers.

commit messages
feat:     add revenue-by-region dbt model
fix:      handle null customer_id in orders join
docs:     document the ingestion DAG
refactor: split monolithic notebook into modules
test:     add row-count check to silver layer
chore:    bump dbt to 1.8

12. Quick-reference cheat sheet

The commands you'll reach for most, in one place.

everyday git — keep this handy
── start ─────────────────────────────────────────────
git init                      # new repo here
git clone <url>                # copy a repo + its history

── the core loop ─────────────────────────────────────
git status                    # what's going on (run constantly)
git add <file> / git add .     # stage changes
git commit -m "message"       # snapshot staged changes
git log --oneline --graph     # view history
git diff                      # see unstaged changes
git blame <file>              # who last changed each line, and when

── branching & tags ──────────────────────────────────
git switch -c <branch>         # create + switch to a branch
git switch <branch>            # move to an existing branch
git merge <branch>             # merge a branch into this one
git branch -d <branch>         # delete a merged branch
git tag -a v1.2 -m "release"   # mark this commit as a named version
git push origin v1.2          # tags aren't pushed by default

── syncing with the team ─────────────────────────────
git pull                      # fetch + merge others' work
git push                      # upload your commits
git push -u origin <branch>    # first push of a new branch
git push --force-with-lease   # after a rebase/amend on YOUR branch
git fetch                     # download without merging

── undo (see §07 for which to use) ───────────────────
git restore <file>            # discard unstaged edits
git restore --staged <file>   # un-stage, keep edits
git commit --amend           # fix the last commit
git reset --soft HEAD~1      # undo last commit, keep changes
git revert <hash>             # safe undo of a public commit
git stash / git stash pop    # shelve / restore work-in-progress
git reflog                   # find "lost" commits

＊ Glossary

Repository (repo)
A project tracked by Git — your files plus the hidden .git folder holding all history.
Commit
A saved snapshot of the whole project at one moment, with a message, author, time, and unique hash.
Hash / SHA
A commit's unique 40-character fingerprint (usually shortened to 7 chars, e.g. 9f2c1ab).
Working directory
The actual files on disk you edit, before staging or committing.
Staging area (index)
The shortlist of changes you've marked (via git add) to include in the next commit.
Branch
A lightweight movable pointer to a commit — a parallel line of development.
HEAD
"You are here" — the commit/branch you currently have checked out.
Detached HEAD
Sitting on a specific commit that no branch points to; commits made here can be lost unless you create a branch.
main / master
The default primary branch holding the canonical, production-ready code.
Tag
A permanent label on a specific commit, typically marking a released version (e.g. v1.2).
Remote / origin
A shared copy of the repo elsewhere (e.g. GitHub); origin is its conventional name.
Clone
A full local copy of a remote repository, including its complete history.
Push / Pull / Fetch
Send commits up to the remote / download-and-merge from it / download without merging.
Merge
Combine two branches, creating a two-parent merge commit (unless it can fast-forward).
Rebase
Replay your commits onto a new base for a linear history — rewrites commit IDs.
Conflict
When two branches change the same lines and Git needs you to choose the result.
Pull request (PR)
A hosted request to review and merge a branch; where code review and CI happen.
CI/CD
Automated testing (Continuous Integration) and deployment (Continuous Delivery) triggered by Git events.
Workflow / branching strategy
A team's agreed rules for branches and how work flows to production (e.g. GitHub Flow, Gitflow).
Stash
A temporary shelf for uncommitted changes so you can switch context and restore them later.
reflog
Git's private log of every position HEAD has held — your safety net for recovering "lost" commits.
.gitignore
A file listing patterns Git should never track (secrets, data, build artefacts).

That's the full toolkit: Git's mental model, the everyday commands, branching and merging, the team workflows, and the data-engineering specifics. Bookmark §12 as your day-to-day cheat sheet — and remember the through-line: commit small, branch freely, sync often, and keep code in Git but data in data tools.
