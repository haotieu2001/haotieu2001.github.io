CI/CD Essentials for Data Engineers

CI/CD is the automation that takes the code you just committed and — without anyone clicking through a checklist by hand — tests it, packages it, and ships it safely to production. It builds the mental model one piece at a time (what continuous integration, delivery and deployment each really mean), then makes it concrete with GitHub Actions — runnable workflows you can copy — before covering environments, deployment strategies, and the parts that matter specifically for data engineering: dbt, Airflow, Databricks, data-quality gates, and infrastructure-as-code. Every company wires CI/CD up with a different tool; the concepts here transfer to all of them.

~40 min read · Essentials · Reference

The one idea (read this first)

CI/CD is an automated assembly line for your code. The moment you push a commit, a pipeline springs to life and runs a fixed series of steps for you: it builds the code, tests it, and — if everything is green — ships it onward, first to a safe staging copy and then to production. CI (Continuous Integration) is the front half: every change is automatically merged, built and tested so problems surface in minutes, not weeks. CD is the back half: Continuous Delivery keeps a release always ready to ship at the push of a button, while Continuous Deployment goes one step further and ships every green change automatically. Replace "click through a manual checklist" with "a script the computer runs the same way every time" — that's the whole game.

01. What CI/CD is & why it exists

Picture shipping software the old way. Everyone works in isolation for weeks, then on release day someone merges all the branches together (the dreaded "integration"), manually runs the tests, copies files to a server, edits a few configs, restarts the service, and prays. It's slow, it's stressful, and it's where most bugs are born — because every step is done by a tired human, slightly differently each time. CI/CD replaces that ritual with automation that runs the same way, every single time.

The name bundles two related practices:

  • CI — Continuous Integration: developers merge their work into the shared branch frequently (many times a day), and every merge automatically triggers a build and a test run. Integration stops being a scary once-a-month event and becomes a quiet, constant background hum.
  • CD — Continuous Delivery / Deployment: once the code passes CI, automation takes it the rest of the way — packaging it and releasing it to your environments. "Delivery" keeps it one button-press away; "Deployment" presses the button for you.

Concretely, CI/CD buys you four things a manual process can't:

  • Fast feedback — you learn a change is broken within minutes of pushing it, while the code is still fresh in your head, not days later in a release window.
  • Repeatability — the steps live in a file, so the 500th deploy is identical to the first. No "it worked on my machine," no forgotten manual step.
  • Safety — automated tests and gates stand between a bad change and your users. Nothing reaches production without passing the same checks.
  • Speed — small changes flow to production continuously instead of piling up into risky big-bang releases.

CI/CD is a practice, not a product

"CI/CD" describes how a team works — integrate often, automate the path to production. Many tools implement that practice: GitHub Actions, GitLab CI, Jenkins, CircleCI, Azure Pipelines, Argo CD. They differ in syntax and features but share the same concepts — triggers, pipelines, stages, jobs, artifacts, environments. Learn the concepts once and you can read any of them. This guide uses GitHub Actions for every example because it's free for public repos, lives right next to your code, and is the most common starting point.

CI/CD sits on top of Git

Every pipeline is triggered by a Git event — a push, a pull request, a tag. CI/CD is the automation layer that reacts to the workflows you read about in Git Essentials: you open a PR, CI runs the checks, and on merge CD deploys. If branches, merges and pull requests feel shaky, skim that guide first — everything here builds on it.

02. The mental model: CI, CD & CD

The single biggest source of confusion is that "CD" stands for two different things that build on each other. Get this picture straight and the rest of the guide clicks into place. Think of three nested stages, each one wrapping the one before it: Continuous Deployment contains Continuous Delivery, which contains Continuous Integration. (In the diagram below there is only one pipeline, running left to right. The three brackets above it show how much of that pipeline each term covers — each bracket includes everything the one below it covers, and adds one more step.)

1
Continuous Integration (CI) — the foundation. On every push or pull request, automatically build the code and run the tests. The goal is a fast, honest answer to one question: "does this change break anything?" CI stops here — it doesn't deploy. Its product is confidence (and usually a tested, packaged build called an artifact).
2
Continuous Delivery (CD) — CI plus automatically getting that tested build ready to release. It's deployed to staging, verified, and parked at the door of production. The very last step — actually releasing to users — stays a manual button-press (often gated by an approval). You could ship at any moment; a human decides when.
3
Continuous Deployment (CD) — the same pipeline with the last manual gate removed. Every change that passes all the automated checks goes straight to production, automatically, no human in the loop. This is only safe with excellent automated testing — the tests are the gate now.

The one-line difference between the two CDs

Continuous Delivery = every change is always ready to deploy, but a person clicks the button. Continuous Deployment = every change that passes the tests deploys itself, no button. Delivery keeps a human gate; Deployment replaces that human with trust in the test suite. In practice the same labels can describe a single rung of the ladder, not just the whole pipeline: many data teams auto-ship all the way to staging (Continuous Deployment to staging) but keep a human approval on the final production step (Continuous Delivery to prod).

TermWhat's automatedThe last step to production is…Needs
Continuous IntegrationBuild + test on every changen/a — CI doesn't deployA test suite
Continuous DeliveryCI + deploy to staging, release-readyA human clicks "deploy"CI + automated deploy + an approval
Continuous DeploymentCI + deploy all the way to prodAutomatic, on green testsStrong tests + monitoring + easy rollback

03. Anatomy of a pipeline

Every CI/CD tool, whatever its syntax, is built from the same five nouns. Learn this vocabulary once and you can read a GitHub Actions workflow, a GitLab .gitlab-ci.yml, or a Jenkinsfile without re-learning anything.

The five nouns of every pipeline

Trigger (event)
The thing that starts a pipeline — a push, a pull request, a new tag, a schedule (cron — the standard "run on a schedule" syntax, whose five fields mean minute / hour / day-of-month / month / day-of-week), or a manual click. "When X happens, run this."
Pipeline (workflow)
The whole automated process that runs in response to a trigger — the full assembly line from start to finish. GitHub Actions calls one a workflow.
Stage
A logical phase of the pipeline — e.g. build → test → deploy. Stages run in order; everything in one stage must pass before the next begins. Important: a stage is a conceptual phase, not a structural level — that's why it isn't a box in the diagram above. GitHub Actions has no stage keyword between the pipeline and its jobs; you get the same effect by chaining jobs together with needs (some other tools, like GitLab CI, do have an explicit stages keyword).
Job
A group of steps that runs together on one fresh virtual machine (a runner). Jobs in the same pipeline run in parallel by default, unless you declare a dependency between them. Each job starts from a clean slate.
Step
A single command or pre-packaged action inside a job — "run pytest," "check out the repo." Steps in a job run top-to-bottom on the same machine, sharing its filesystem.

Two supporting concepts complete the picture:

  • Runner — the machine a job executes on. GitHub gives you fresh, throwaway cloud VMs (Linux/Windows/macOS) for free; you can also register your own (self-hosted runners) for special hardware or private-network access. Because each job gets a clean machine, nothing leaks between jobs — which is why you need artifacts.
  • Artifact — a file (or folder) produced by one job and handed to another, or kept for download. Since each job starts fresh, an artifact is how a built thing in the "build" job survives to the "deploy" job. Think compiled bundles, test reports, or a .whl Python package.

Each job is an island

A beginner's classic trap: job A installs something or writes a file, and job B can't find it. That's by design — jobs run on separate, clean machines. To pass data between jobs you must upload it as an artifact in A and download it in B (or use a cache, or job outputs for small values). Within a single job, though, steps share one filesystem, so file X written in step 1 is right there in step 4.

04. Continuous Integration in practice

CI is the half you'll set up first and rely on daily. Its job is to answer "is this change safe?" automatically, on every push and every pull request. A good CI run is a stack of cheap-to-expensive checks, ordered so the fastest, most common failures trip first.

The typical CI stages, fastest first

1
Lint & format — static checks that need no execution: code style, obvious mistakes, formatting. Seconds to run, so they go first and fail fast (ruff, black --check, sqlfluff for SQL).
2
Build / compile — turn source into a runnable artifact: compile the code, build a Python wheel (a packaged, installable bundle of your Python code) or a Docker image (your code plus everything it needs to run, packed into one portable unit), or dbt compile to confirm every model parses.
3
Unit tests — fast, isolated tests of individual functions with no external systems. The bulk of your safety net (pytest).
4
Integration / data tests — slower checks against a real (but disposable) database or a sample dataset: does the whole pipeline run end-to-end? Do the data-quality rules hold?

Here's that exact ladder as a real GitHub Actions workflow. Don't worry about the syntax yet — §06 breaks it down line by line. Read it for the shape: a trigger up top, then a job, then steps.

.github/workflows/ci.yml
# A minimal but real CI pipeline for a Python data project
name: CI

on:                              # ── the TRIGGER ──
  pull_request:                  # run on every PR…
  push:
    branches: [main]               # …and on every push to main

jobs:
  test:                           # ── one JOB ──
    runs-on: ubuntu-latest         # a fresh Linux RUNNER
    steps:                         # ── the STEPS, top to bottom ──
      - uses: actions/checkout@v4   # 1. pull the repo onto the runner
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: pip install -r requirements.txt -r requirements-dev.txt
      - run: ruff check .            # 2. lint   (fastest, fails first)
      - run: sqlfluff lint models/   # 3. lint the SQL too
      - run: pytest -q               # 4. unit tests

Two bits of YAML to know first

These files are written in YAML — a plain-text format for structured config (indentation shows nesting). Two shorthands trip up beginners: a key with nothing after the colon, like pull_request: above, means "use the defaults" (here: run on any pull request). And a block written over several indented lines is identical to its compact one-line form — with: then an indented python-version: "3.12" means exactly the same as with: { python-version: "3.12" } (you'll see both styles in this guide). Same data, two ways to write it.

The golden rule of CI: keep main green

The whole point of CI is that the shared branch always works. Wire CI to run on pull requests, then turn on branch protection (a GitHub setting) so a PR cannot be merged until its checks pass. Now a broken change physically can't reach main — the red − blocks the merge button. A green main means anyone can branch off it with confidence.

Why "fast" matters more than it sounds

CI is only useful if people wait for it. A 2-minute pipeline gets watched; a 40-minute one gets ignored and bypassed. So you optimise relentlessly: run the cheap checks first, run independent jobs in parallel, and cache slow things like dependency downloads (more on both in §07). The target most teams aim for is "PR feedback in under ten minutes."

05. Delivery vs Deployment — the back half

Once CI says "this change is good," CD takes over and moves it toward users. The mechanics are the same for both flavours of CD; the only difference is whether a human stands at the final gate. The key new idea here is the environment ladder.

The environment ladder: dev → staging → prod

You never deploy straight from a laptop to production. Instead a change climbs a ladder of progressively more production-like environments, proving itself at each rung before promotion to the next.

  • Dev — where you and your changes live first. Cheap, disposable, expected to break.
  • Staging (a.k.a. pre-prod, or UAT — User Acceptance Testing) — a faithful copy of production: same infrastructure, same config shape, ideally a realistic slice of data. The dress rehearsal. If it works here, it should work in prod.
  • Production — the real thing your users (or downstream data consumers) depend on. Changes arrive here only after passing the lower rungs, and usually behind an approval gate.

Build once, deploy many

A cornerstone principle: produce the deployable artifact a single time in CI, then promote that exact artifact through dev → staging → prod. Don't rebuild it for each environment — a rebuild can pull in a newer dependency or a different commit, so the thing you tested in staging isn't the thing that lands in prod. What differs between environments should be configuration (database URLs, credentials, scale), injected at deploy time — never the build itself.

A delivery pipeline, in GitHub Actions

This workflow shows the back half: after tests pass, deploy to staging automatically, then deploy to production only after a human approves (that's the environment: line, which we'll wire to a required reviewer in §08). This is Continuous Delivery — auto to staging, human-gated to prod.

.github/workflows/deploy.yml
name: Deploy
on:
  push:
    branches: [main]              # a merge to main starts a release

jobs:
  deploy-staging:
    runs-on: ubuntu-latest
    environment: staging         # auto-deploys, no gate
    steps:
      - uses: actions/checkout@v4
      - run: ./scripts/deploy.sh staging

  deploy-prod:
    needs: deploy-staging        # ← only after staging succeeds
    runs-on: ubuntu-latest
    environment: production      # ← this one REQUIRES a human approval
    steps:
      - uses: actions/checkout@v4
      - run: ./scripts/deploy.sh production

To turn this into Continuous Deployment, you'd simply remove the approval requirement on the production environment — then a green pipeline ships to prod with no human in the loop. Same file, one setting different. That's how close the two practices really are.

06. GitHub Actions, concretely

Time to make it real. GitHub Actions is GitHub's built-in CI/CD: you drop YAML files in a special folder, and GitHub runs them in response to repo events. Everything in §§02–05 maps onto a handful of keywords. Here's the whole vocabulary in one place.

Where workflows live, and the key words

A workflow is a .yml file in .github/workflows/. You can have many — one for CI, one for deploys, one for nightly jobs. Each maps onto the pipeline anatomy from §03:

KeywordMeansMaps to (§03)
nameA human label for the workflowthe pipeline's name
onWhich events run it (push, pull_request, schedule, workflow_dispatch…)trigger
jobsThe set of jobs in this workflowthe pipeline body
runs-onWhich runner OS the job usesthe runner
needs"Run this job only after that one passes"stages / ordering
stepsThe ordered commands inside a jobsteps
usesRun a pre-built, shareable actiona step (reused)
runRun a shell commanda step (custom)

uses vs run — the two kinds of step

A step either runs a shell command (run: pytest) or pulls in a reusable, community-published building block (uses: actions/checkout@v4). An action is packaged automation someone already wrote — checking out code, setting up Python, uploading artifacts, logging into a cloud. They live in the Marketplace, and you pin a version with @v4. Think of uses as importing a library and run as writing the code yourself.

A fully annotated workflow

This one example exercises most of what you'll use day to day — triggers, a build job, a dependent deploy job, an artifact handed between them, and a manual-run option. Read the comments; this is the Rosetta Stone for the rest of the guide.

.github/workflows/pipeline.yml
name: Build, Test & Deploy

on:
  push:
    branches: [main]            # merge to main → full pipeline
  pull_request:                # every PR → build + test (deploy job is skipped, see `if`)
  workflow_dispatch:           # adds a "Run workflow" button for manual runs

jobs:
  # ───────────── JOB 1: build & test ─────────────
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4         # clone the repo onto the runner
      - uses: actions/setup-python@v5
        with: { python-version: "3.12" }
      - run: pip install -r requirements.txt
      - run: pytest -q                      # test
      - run: python -m build                 # produce dist/*.whl
      - uses: actions/upload-artifact@v4    # save the build for the next job
        with:
          name: package
          path: dist/

  # ───────────── JOB 2: deploy (only on main) ─────────────
  deploy:
    needs: build                          # wait for build to pass
    if: github.ref == 'refs/heads/main'   # skip on PRs; deploy only from main
    runs-on: ubuntu-latest
    environment: production              # attaches approval + prod secrets (set up in Settings → §08)
    steps:
      - uses: actions/download-artifact@v4  # grab the package built in job 1
        with: { name: package, path: dist/ }
      - run: ./scripts/deploy.sh
        env:
          API_TOKEN: ${{ secrets.PROD_API_TOKEN }}  # injected, never hard-coded

Reading ${{ ... }}

The ${{ }} braces are GitHub Actions expressions — values filled in at run time. ${{ secrets.PROD_API_TOKEN }} pulls a stored secret; ${{ github.ref }} is the branch/tag that triggered the run; ${{ matrix.python }} (next section) is the current matrix value. Anywhere you see them, read "look this up when the pipeline runs."

One exception that looks like a typo but isn't: inside an if: you'll see the value written without braces — if: github.ref == 'refs/heads/main' above. That's because if: is always evaluated as an expression, so the ${{ }} is optional there. Everywhere else, keep the braces.

07. Core building blocks

Beyond the basic shape, a handful of features turn a toy workflow into a real one: secrets, variables, the build matrix, caching, artifacts, and reusable workflows. These same ideas exist in every CI/CD tool under different names — learn them here, recognise them everywhere.

Secrets & variables — never hard-code credentials

Pipelines need passwords, API tokens, and cloud keys — but those must never appear in your YAML (it's in Git, readable by all). Instead you store them in the repo's encrypted Secrets store (Settings → Secrets and variables) and reference them by name. GitHub masks them in logs automatically. Non-secret config (a region, a warehouse name) goes in Variables alongside.

using secrets & variables
steps:
  - run: dbt run --target prod
    env:
      DBT_PASSWORD: ${{ secrets.DBT_PROD_PASSWORD }}   # encrypted secret
      SNOWFLAKE_ACCOUNT: ${{ vars.SNOWFLAKE_ACCOUNT }}  # non-secret variable

The build matrix — test many versions at once

A matrix runs the same job across a grid of values — Python 3.11 and 3.12, on Linux and macOS — all in parallel, each on its own runner. One block of YAML fans out into many runs.

matrix strategy
jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, macos-latest]
        python: ["3.11", "3.12"]      # 2 × 2 = 4 parallel runs
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: ${{ matrix.python }} }
      - run: pytest -q

Caching — don't re-download the world every run

Installing dependencies from scratch on every run is slow. A cache saves a folder (your package downloads) keyed by a fingerprint (a "hash") of your dependency file, and restores it next time — cutting minutes off each run. The key is the clever bit: when requirements.txt changes, its fingerprint changes, so the cache misses and rebuilds; otherwise it's a fast hit. (Many teams skip this hand-rolled step and use setup-python's built-in cache: pip instead — same idea, less YAML.)

dependency caching
- uses: actions/cache@v4
  with:
    path: ~/.cache/pip
    key: pip-${{ hashFiles('requirements.txt') }}   # changes only when deps change

Reusable workflows — write the pipeline once

When ten repos need the same deploy steps, you don't copy-paste YAML ten times. A reusable workflow is a pipeline that other workflows call, passing in parameters — the same don't-repeat-yourself (DRY) instinct as a function. Define it once with workflow_call, then uses: it from anywhere.

calling a reusable workflow
jobs:
  deploy:
    uses: my-org/ci-templates/.github/workflows/deploy.yml@v1
    with:
      environment: production
    secrets: inherit

Notice this deploy job has no runs-on and no steps — that's the one case where a job breaks the usual shape. When a job uses: a whole reusable workflow, that called workflow becomes the job's body (it brings its own jobs and runners). with: passes parameters into it, and secrets: inherit forwards your secrets so the called workflow can use them. So uses: works at two levels: on a step (run one action) and, as here, on a job (run an entire workflow).

These map onto any CI/CD tool

Secrets, matrices, caching and reusable templates aren't GitHub inventions — GitLab CI has variables, parallel:matrix, cache: and include:; Jenkins has Credentials, matrix builds, and shared libraries. The keywords change; the concepts are identical. That's why understanding these blocks matters more than memorising one tool's syntax.

08. Environments, approvals & promotion

§05 introduced the dev → staging → prod ladder as a concept. Here's how you actually enforce it — so that production can't be touched without the right checks and the right people signing off.

GitHub Environments — gates around the dangerous steps

A GitHub Environment (Settings → Environments) is a named target — staging, production — that you attach to a job with environment:. Attaching it unlocks three powerful guardrails:

  • Required reviewers — the job pauses and waits for a named person/team to click "Approve" before it runs. This is the manual gate that makes Continuous Delivery what it is.
  • Environment secrets — secrets scoped to just that environment. Your production database password is only readable by jobs targeting production, so a staging job can't accidentally (or maliciously) touch prod.
  • Protection rules — extra conditions like "only the main branch may deploy here" or a wait timer before deploy.

("Smoke test staging" above means a quick sanity check that the deployment is alive and basically working — a query returns, the job starts — not the full test suite. The name comes from electronics: power it on and see if it smokes.)

Promotion = redeploy the same artifact, higher up

"Promoting" a build means taking the artifact that already passed in staging and deploying that identical artifact to production — only the configuration (which database, which secrets) changes. You're not rebuilding or re-testing the code; you're re-pointing the proven package at a higher environment. This is the practical payoff of "build once, deploy many" from §05.

Mapping branches to environments

Most teams tie which branch you merge into to which environment gets the deploy — exactly the GitLab-Flow idea from Git Essentials §08. A common shape:

Branch / eventTriggersDeploys to
open / update a PRCI: lint + test (no deploy)nothing (or a temporary throwaway "preview" environment spun up just for that PR)
merge to mainCD: build → staging → (approval) → prodstaging, then production
push a tag v1.4.0release pipelineproduction, tagged release

09. Deployment strategies & rollback

"Deploy to production" hides a choice: how do you swap the old version for the new one without breaking things for users mid-request? These strategies trade simplicity against safety. They matter most for always-on services and APIs; data pipelines borrow the same ideas (see §10).

Quick vocab for this section

This section borrows web-service language. An instance is one running copy of your service (you often run several at once to share the load). Traffic is the stream of incoming user requests. Cutting over means switching users from the old version to the new. Data pipelines don't serve live requests, but §10 shows how each idea still maps onto tables and partitions.

StrategyHow it worksTrade-off
RecreateStop the old version entirely, then start the new oneSimplest, but causes downtime during the swap
RollingReplace instances a few at a time until all run the new versionNo downtime; but old & new run side-by-side briefly
Blue-greenStand up a full second copy (green), switch all traffic at once, keep the old (blue) as instant rollbackZero-downtime & instant rollback; costs double infra during the switch
CanarySend a small % of traffic to the new version, watch metrics, then ramp to 100%Safest — limits the "blast radius" (how many users a bad change can hurt); needs good monitoring & traffic routing

The deploy isn't done until you can undo it

Every deploy strategy is really about one thing: making a bad release reversible, fast. Always know your rollback path before you ship — redeploy the previous artifact, flip blue-green back, or drain the canary (stop sending live traffic to the new version). Pair it with monitoring (error rates, latency, data-freshness) so you notice a bad release in minutes. A deploy you can't cleanly undo is a gamble, not a process.

10. CI/CD for data engineers

Everything so far applies to any software. But data work has its own shape — SQL and transformation logic instead of an app, data that can be wrong even when the code is right, orchestrators to deploy, and stateful schemas that can't just be "replaced." This section covers what's different, with the tools you'll actually meet.

What you're really deploying in a data project

Not a web server — but plenty that benefits from a pipeline: dbt models and SQL, Airflow/Dagster DAGs, Spark/Python jobs, Databricks notebooks & jobs, pipeline config (YAML), and the infrastructure itself (Terraform). CI tests that this code is correct; CD ships it to your orchestrator or warehouse.

The extra gate data adds: data-quality checks

Here's the twist unique to data: your code can be flawless and your output still wrong — because the data changed. A schema drifted, a source went stale, nulls crept into a key column. So data CI/CD adds a gate ordinary software doesn't have: data-quality (DQ) tests that assert things about the data, not just the code (row counts, uniqueness, freshness, accepted values). Tools: dbt tests, Great Expectations, Soda.

Two questions, two kinds of test

Code tests (unit tests, plus dbt parse to catch broken model references) ask "is the logic correct?" and run in CI on every PR. Data-quality tests (dbt tests, Great Expectations) ask "is the data correct?" and run both in CI (against sample/staging data) and in production after each pipeline run, because data can break long after the code stopped changing. A mature data pipeline gates on both.

dbt: the model citizen of data CI/CD

dbt was built for this workflow. On a PR, CI builds your changed models into a temporary schema and runs their tests; on merge, CD deploys to production. The trick that keeps CI fast on a big project is dbt's state comparison, usually called Slim CI: you hand dbt the manifest.json (a metadata file describing all your models) from the last production run via --state, and state:modified+ then selects only the models that changed since then, plus their downstream children. --defer tells dbt to point any unchanged upstream ref()s at the existing production tables, so it doesn't rebuild your whole project. The catch: you must fetch that production manifest before the run — without it, state: has nothing to compare against and dbt errors out.

.github/workflows/dbt-ci.yml
name: dbt CI
on: { pull_request: {} }       # `{}` = "with default settings"

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install dbt-snowflake
      - run: dbt deps
      - run: dbt parse                  # fast syntax/ref check, builds this branch's manifest
      # Slim CI: fetch the manifest from the last PRODUCTION run so dbt
      # knows what changed (from blob storage, or the prod job's artifact):
      - run: aws s3 cp s3://my-dbt-state/prod/manifest.json prod-state/manifest.json
      - run: >
          dbt build --select state:modified+
          --defer --state prod-state/      # run + TEST only changed models & children
        env:
          DBT_PASSWORD: ${{ secrets.DBT_CI_PASSWORD }}

dbt build runs models and their tests together, so a uniqueness or not-null test failing turns the PR red — a data bug caught before merge. (The throwaway per-PR schema is configured in your profiles.yml target, not by a magic environment variable; many teams also drop that schema automatically when the PR closes.)

Deploying orchestrators & Databricks

For the deploy half, CD pushes your code to wherever the pipelines run:

  • Airflow — CD syncs your DAG files to where the scheduler reads them: a helper process that continuously pulls them from Git (a "git-sync sidecar"), an S3/GCS bucket, or an API upload. (A DAG — Directed Acyclic Graph — is just the dependency graph that defines one scheduled pipeline.)
  • Dagster — works differently: CD deploys a code location (your pipelines packaged and run as a self-contained service that the Dagster instance connects to) — not loose files dropped in a folder. Either way, merge to main → the new pipelines go live.
  • Databricks — Asset Bundles (the databricks bundle CLI) package notebooks, jobs and config and deploy them per-environment from CI. It's the "build once, deploy many" pattern, Databricks-flavoured.
deploying a Databricks Asset Bundle
jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4
      - uses: databricks/setup-cli@v0.221.1   # pin to a release tag, never @main (see §11)
      - run: databricks bundle deploy -t prod    # push jobs & notebooks to the prod workspace
        env:
          DATABRICKS_TOKEN: ${{ secrets.DATABRICKS_PROD_TOKEN }}

Infrastructure-as-code & schema migrations

Two more data-flavoured deploys worth knowing:

  • Infrastructure-as-code (Terraform) — your warehouses, buckets and clusters defined as code. CI runs terraform plan on a PR (a dry-run preview of the change); CD runs terraform apply on merge. Now infra changes get reviewed like any code. (Fully unpacked in Terraform Essentials.)
  • Schema migrations — unlike app code, a database has state you can't just replace. Migration tools (Alembic, Flyway) apply ordered, incremental changes — "add column X," "backfill Y" — so prod tables evolve safely. CD runs the pending migrations as a deploy step. Always design them backward-compatible (add before you remove) so a rollback doesn't strand the data. (Note: dbt snapshots are a different thing — they track historical changes in your data, not changes to the table structure — i.e. DDL, the SQL that adds or alters tables — so they don't belong in this list.)

Rolling back data is not like rolling back code

Redeploying yesterday's code is easy. Un-writing the rows a bad pipeline already wrote is not — the side effects live in tables, not in the artifact. So data CD leans on extra safety nets: write to a staging table and swap only after DQ checks pass, use Delta Lake time travel to restore a table to a prior version, make jobs idempotent (re-running gives the same result), and keep migrations backward-compatible. Plan the data rollback, not just the code rollback.

This is how the §09 strategies map onto data: that staging-table-swap is your blue-green deploy (build the new table beside the old one, flip to it only once DQ passes), running a transform on yesterday's partition first is your canary, and an idempotent re-run of the previous version is your rollback. Same ideas as services — just expressed in tables and partitions instead of web traffic.

The data engineer's CI/CD checklist

On every PR: lint SQL, compile/parse the pipeline, run unit tests, and build + test changed models in a throwaway schema. On merge: deploy to the target environment and run DQ checks after the pipeline finishes. Across all of it: keep code in Git, keep data in data tools (see Git Essentials §10), and gate on both code and data correctness.

11. Security & best practices

A pipeline has the keys to your production systems, so it's a juicy target. A few habits keep it safe — and keep it fast and maintainable.

Security

  • Never hard-code secrets. Use the encrypted secrets store and ${{ secrets.* }}. A token pasted into YAML is a token leaked to everyone with repo access — and it's in Git history forever.
  • Prefer OIDC over long-lived keys. Instead of storing a permanent cloud key in your secrets, have the cloud hand the pipeline a temporary credential that's valid only for that one run and expires automatically — so there's nothing permanent sitting around to be stolen. The standard that makes this work is OIDC (OpenID Connect), which lets AWS/GCP/Azure verify "yes, this really is a run of your workflow" before issuing that short-lived credential. This is the modern best practice for cloud deploys.
  • Least privilege. Give each workflow the minimum it needs. Set permissions: explicitly, scope prod secrets to the production environment, and don't let PR pipelines touch prod.
  • Pin your actions. uses: actions/checkout@v4 (or a full commit SHA) — never an unpinned @main you don't control, which could change under you.
  • Be careful with PRs from forks. A fork is an outside copy of your repo that anyone can make to propose changes. Untrusted contributors opening PRs from their fork shouldn't get your secrets; GitHub withholds them from fork PRs by default — keep it that way.

General best practices

  • Keep pipelines fast. Fast feedback gets used; slow feedback gets bypassed. Parallelise jobs, cache dependencies, run only what changed.
  • Fail fast, fail loud. Cheapest checks first; surface failures clearly; notify the right channel on a broken main.
  • Make every run reproducible. Pin versions, build once, inject config. The pipeline should behave identically today and in six months.
  • Make jobs idempotent. Re-running a deploy (or a data job) should be safe and give the same result — vital when you retry a flaky step.
  • Keep workflows in version control & review them. Pipeline YAML is code: PR it, review it, and treat a change to your deploy process with the same care as a change to production logic.
  • Monitor what you ship. CD without monitoring is shipping blind. Watch error rates, latency, and (for data) freshness and volume, so you catch a bad release before your users do.

12. Quick-reference cheat sheet

The GitHub Actions keywords and patterns you'll reach for most, in one place.

github actions — keep this handy
── triggers (on:) ────────────────────────────────────
on: { push: { branches: [main] } }   # on push to a branch
on: { pull_request: {} }              # on every PR
on: { schedule: [{ cron: "0 6 * * *" }] }  # nightly at 06:00 UTC
on: { workflow_dispatch: {} }         # a manual "Run workflow" button
on: { push: { tags: ["v*"] } }       # on a version tag

── job structure ─────────────────────────────────────
jobs:
  my-job:
    runs-on: ubuntu-latest            # the runner
    needs: [other-job]                 # run after other-job passes
    if: github.ref == 'refs/heads/main'  # conditional run
    environment: production           # attach approval + scoped secrets
    steps: [ ... ]

── steps: uses (action) vs run (shell) ───────────────
- uses: actions/checkout@v4          # a reusable action, version-pinned
- run: pytest -q                      # a shell command

── common actions ────────────────────────────────────
actions/checkout@v4                   # clone the repo
actions/setup-python@v5               # install Python
actions/cache@v4                      # cache deps between runs
actions/upload-artifact@v4            # save a file for later jobs
actions/download-artifact@v4          # retrieve it

── secrets, vars & expressions ───────────────────────
${{ secrets.MY_TOKEN }}               # encrypted secret
${{ vars.MY_SETTING }}                # non-secret variable
${{ github.ref }}                     # the triggering branch/tag
${{ matrix.python }}                  # current matrix value

── matrix & caching ──────────────────────────────────
strategy: { matrix: { python: ["3.11", "3.12"] } }
uses: actions/cache@v4  with: { key: pip-${{ hashFiles('requirements.txt') }} }

── the data engineer's PR pipeline ───────────────────
ruff check . && sqlfluff lint models/   # lint code + SQL
pytest -q                               # unit tests
dbt build --select state:modified+ --defer --state prod/  # Slim CI: build+TEST changed models
# on merge → deploy (bundle/terraform/dag-sync) → run DQ checks

＊ Glossary

CI — Continuous Integration
Automatically building and testing every change as it's merged, so integration problems surface in minutes.
CD — Continuous Delivery
CI plus keeping a tested build always ready to release; a human clicks to deploy to production.
CD — Continuous Deployment
The same, but the final release to production is automatic on green tests — no human gate.
Pipeline / workflow
The whole automated process triggered by an event; in GitHub Actions, a .yml file in .github/workflows/.
Trigger / event
What starts a pipeline — a push, pull request, tag, schedule (cron), or manual run.
Stage
A logical phase (build → test → deploy); each must pass before the next begins.
Job
A group of steps running together on one fresh runner; jobs run in parallel unless linked by needs.
Step
A single command (run) or reusable action (uses) inside a job.
Runner
The machine a job executes on — GitHub-hosted (fresh cloud VM) or self-hosted.
Action
A pre-built, shareable step from the Marketplace, pinned to a version (e.g. actions/checkout@v4).
Artifact
A file produced by one job and passed to another or kept for download (a build, a test report).
Environment
A named deploy target (staging, production) with its own scoped secrets and approval rules.
Promotion
Deploying the same already-tested artifact to a higher environment, changing only config.
Secret
An encrypted credential referenced as ${{ secrets.NAME }}, masked in logs, never hard-coded.
Matrix
Running one job across a grid of values (OS, language version) in parallel.
Cache
A saved folder (e.g. dependency downloads) restored on later runs to save time.
Branch protection
A rule that blocks merging a PR until its required checks pass — how CI keeps main green.
Blue-green deploy
Run two full copies; switch all traffic to the new one at once, keep the old for instant rollback.
Canary deploy
Route a small % of traffic to the new version, watch metrics, then ramp to 100%.
Rollback
Reverting to the previous known-good version after a bad release.
OIDC
Short-lived, verifiable identity a pipeline presents to a cloud, replacing long-lived stored keys.
Idempotent
An operation that gives the same result no matter how many times it's run — safe to retry.
Data-quality (DQ) test
An assertion about the data (row counts, uniqueness, freshness), not the code — e.g. dbt tests, Great Expectations.
Infrastructure-as-code (IaC)
Defining infrastructure (warehouses, buckets) as reviewable code, e.g. Terraform; plan previews, apply ships.
Schema migration
An ordered, incremental change to a stateful database (add column, backfill), applied as a deploy step.
YAML
A plain-text format for structured config (indentation shows nesting); the language GitHub Actions workflows are written in.
Cron
The standard syntax for "run on a schedule"; its five fields are minute / hour / day-of-month / month / day-of-week.
Smoke test
A quick sanity check that a deployment is alive and basically working — not the full test suite.
Fork
An outside copy of a repo that anyone can make to propose changes; PRs from forks are withheld your secrets by default.
DAG
Directed Acyclic Graph — the dependency graph that defines a scheduled pipeline (Airflow/Dagster).
Slim CI (dbt)
Building & testing only the models that changed since production, using state:modified+ with --defer --state and the prod manifest.json.
DRY
"Don't Repeat Yourself" — the instinct to factor shared steps into one place (e.g. a reusable workflow).

That's the full toolkit: what CI/CD is, the three stages, the anatomy of a pipeline, GitHub Actions end to end, environments and promotion, deployment strategies, and the data-engineering specifics. Bookmark §12 as your day-to-day cheat sheet — and remember the through-line: integrate often, build once, gate on both code and data, and never ship what you can't roll back.
