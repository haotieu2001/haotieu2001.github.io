Delta Lake vs Iceberg vs Hudi

Three names you keep hearing. One job: turning a pile of cheap files into a real, safe table. This guide explains all three from zero — what they are, how they work inside, and which one to pick. No deep background needed.

~22 min read · updated June 2026 · written in plain English

It is 2 a.m. A data job just stopped halfway while writing a few million rows. Now the morning reports are reading broken, half-written data. Two other jobs wrote over each other. And nobody can answer one simple question: what did this table look like yesterday?

If you have had this problem before, you have met exactly what these three tools were built to fix. A plain data lake — millions of files sitting in cloud storage — is cheap and can grow forever. But on its own it has none of the safety a real database gives you. Over time it turns into what people call a data swamp: messy, slow, and not safe to trust.

Delta Lake, Apache Iceberg and Apache Hudi are the three tools that fix this. They are called open table formats. This guide is long on purpose: by the end you will understand the one big idea, the key words everyone uses, how each tool works inside, and how to choose. Let's start.

❯ Short version (read this first)

All three take cheap files and add a "logbook" on top that writes down every change. That logbook is what makes the files behave like a safe table. They mostly differ in how they keep that logbook — and that gives each one a personality:

• Delta Lake — the easy choice if you work on Databricks or Spark. Best for everyday reports and dashboards.

• Apache Iceberg — the open standard that works with almost any tool. Best when many tools must read the same data.

• Apache Hudi — the fast-update specialist. Best when data changes all the time (live, second by second).

Fast picker: On Databricks? → Delta. Many tools need one copy? → Iceberg. Constant live updates? → Hudi. See the full guide below ↓

New to the words? Spark, Flink, Trino, Snowflake, BigQuery are "engines" — programs that read and write the data. Databricks is a popular data platform built on Spark. S3 is Amazon's cloud file storage. Parquet is the common file type used here. Streaming means data that keeps arriving, live, all the time. JSON is a common text format for flexible data.

~/lakehouse ❯ the backstory

01. How we got here

Database → Warehouse → Lake → Lakehouse

To see why these tools exist, it helps to know the 40-year story. Each step below was invented to fix a weak point of the step before it.

  • A database runs the business live. Your bank balance drops the second you take out cash. It is fast and safe, but it costs a lot when you want to study many years of data at once.
  • A data warehouse is built to study data: big reports and trends. It is fast for that, but storage is expensive and it does not like messy or raw data.
  • A data lake made storage very cheap. You can drop any file in it, and it grows without limit. The downside: it threw away all the safety. This is where the "data swamp" comes from.
  • A lakehouse is the fix. Keep the cheap lake, and add the safety back with a thin table format layer on top. That is exactly the job of Delta, Iceberg and Hudi.

~/lakehouse ❯ the one big idea

02. So what is a table format?

Here is the whole idea in one line:

A table format keeps your data in normal files — and adds a logbook next to them that writes down every change.

That logbook is the magic. It is a record of every "add this file" and "remove that file". To read the table, a tool reads the logbook to learn which files count right now. The files themselves never change — the logbook decides what the table looks like.

If you have used Git (a tool that saves every version of your code), here is a shortcut: a table format is like Git for your data. Every change is a save point. You can look at old versions. You can undo a bad change.

Three words get mixed up all the time. Let's make them clear, because the rest of the guide depends on them:

  • File format — how one file stores its rows. The common one is Parquet. All three table formats keep their data in Parquet files.
  • Table format — the logbook (Delta / Iceberg / Hudi) that turns a folder of Parquet files into one safe table.
  • Catalog — the "phone book". It maps a table name (like sales.customers) to the place where its logbook lives. Examples: Unity Catalog, AWS Glue.

Three quick myths to clear up:

− "A lakehouse is a product I buy." → No. It is a design. The table format is what turns a lake into a lakehouse.

− "A table format is a database." → No. There is no running server. It is just a set of rules plus files that many tools can read.

− "Delta only works on Databricks." → No. Delta is open and free; Microsoft Fabric (another data platform) even uses it by default. (Some new features did arrive on Databricks first, that part is true.)

~/lakehouse ❯ the words everyone uses

03. The building blocks

Learn these ten ideas once, and every tool becomes easy.

People throw these words around in every blog and meeting. Here is each one in plain words, with a small example. You do not need to memorize them — just get the feel.

1. The logbook (transaction log)
A list of every change, in order. Reading the table means replaying the list to see which files are live now. (This logbook is also called the table's metadata — which just means "data about your data".) Example: "added file A, added file B, removed file A" → the table is now just file B.
2. ACID (all-or-nothing)
A fancy word for one promise: a write either fully happens or does not happen at all. Readers never see a half-done job. Example: a job loading 5 million rows crashes at row 3 million → readers still see the old, complete table, not the broken half.
3. Time travel and undo
Because the logbook keeps old versions, you can read the table as it was last week, or undo a bad load with one command. Example: a bad job deleted most rows → you run "go back to yesterday's version" and it is fixed in seconds.
4. Schema (the table's shape, and its rules)
The schema is the list of columns and their types (for example, balance must be a number). The format checks writes against it, so bad data is rejected before it is written, instead of quietly breaking the table. You can also change the shape later (add a column) without rewriting old data.
5. MERGE / upsert (and CDC)
Upsert = "update the row if it exists, add it if it does not". MERGE is the command that does this for a batch of changes. This is how you keep one table matching another. CDC (Change Data Capture) just means "grab what changed in the source and apply it here". Example: 50,000 balances changed overnight and 200 new accounts opened → one MERGE updates and adds them all. It can also delete one person's data when they ask you to under privacy law (the "right to be forgotten").
6. Partitions and skipping
A partition splits a big table into folders by some value, usually a date. When a query asks for one day, the tool reads only that one folder and skips the rest. The logbook also stores small facts about each file (like the smallest and largest value), so the tool can skip files that cannot match — for example, a file whose latest date is in May can be skipped by a query about June. Less data read = faster and cheaper. Example: a report for "1 June" reads one day's folder, not three years of data.
7. Small files and compaction
Frequent small writes create thousands of tiny files, and opening each one is slow. Compaction (Delta calls it OPTIMIZE) merges many tiny files into a few big ones. It is routine cleanup that keeps reads fast.
8. Many writers at once (concurrency)
Two jobs may write at the same time. The common trick (in Delta and Iceberg) is: let both jobs work, then at the very end each one checks "did someone else change the same files while I was working?". If they changed different files, both updates are kept. If they changed the same data, one job is asked to run again on the newest version — its work is not thrown away, it just redoes the change. Either way, no update is silently lost. So a dashboard can read while a job writes, with no broken data and no waiting.
9. Two update styles (rewrite now, or note it down)
When you change a few rows, the format must choose. Copy-on-Write rewrites the whole file now → reads stay fast, writes are heavy. Merge-on-Read writes the change to a small side-note and merges it later → writes are fast and light, reads do a bit more work. This choice is the heart of why Hudi is great for live data (more on this below).
10. Deletion vectors and the catalog
A deletion vector is a tiny note that says "rows 7 and 12 in this file are deleted" so the format does not have to rewrite the whole file just to drop a few rows. The catalog is the service that maps a table name to its logbook and controls who can see it (for example Unity Catalog or AWS Glue). All three formats now use deletion vectors; the catalog is becoming the most important piece of all.

~/lakehouse ❯ meet the three

04. Meet the three tools

They all do the logbook thing. The interesting part is how — and that one choice shapes everything about each tool.

A note on fairness: most comparisons online come from a company that sells one of these — Databricks made Delta, Netflix and others back Iceberg, Uber made Hudi. I am not selling any of them, so I have tried to stay fair.

Delta Lake

"The easy default — if you are on Databricks."

The story: built by Databricks in 2019, then made open and free (it is run by the Linux Foundation, a neutral home for open-source projects).

How it works inside

A Delta table is a folder of Parquet files plus a small _delta_log folder. Every change adds one numbered file to that log (commit 0, commit 1, commit 2…). Think of a bank passbook: the little book that lists each deposit and withdrawal in order. To know the balance, you read the lines. Every so often Delta writes a checkpoint — a summary — so tools do not have to read the whole history each time.

How a read and a write work

Read: load the latest checkpoint, then read the few newest log lines on top. Now you know exactly which files are live, and which files you can skip.

Write: write the new files, then try to add the next numbered log line. The storage lets only one job create that numbered file (a file name can be created once and only once). If two jobs try at the same time, one succeeds; the other is told "it already exists" and simply tries again with the next number. No data is lost.

Best at

Everyday reports, dashboards, and machine learning on Spark / Databricks. It has the simplest model and the smoothest tools.

Watch out

Its newest features usually arrive on Spark/Databricks first; other tools catch up a bit later.

New in 2025 (Delta 4.0): a flexible Variant type for JSON-like data (free-form data that does not fit neat columns), safe column type changes with no rewrite, and "coordinated commits" — this extends the "many writers" idea from block 8 so even different tools and clouds can write to one table safely.

Apache Iceberg

"The open standard that works with anything."

The story: built at Netflix around 2017 to fix old, slow table tools; now a neutral Apache project that no single company owns.

How it works inside

Iceberg keeps its logbook as layers of save points (called snapshots). The catalog points to the current top file (metadata.json). That file points to a list of files (Iceberg calls these lists manifests). That list points to more lists. The bottom holds the data files — think folders inside folders. Each step stores small facts (like the smallest and largest value in a file), so a tool can skip the parts it does not need.

How a read and a write work

Read: the tool follows the layers from the top and, at each step, skips the parts that cannot match the query. It never has to scan the whole folder — even with millions of files, planning stays fast.

Write: build a brand-new top file, then ask the catalog to switch the "current" pointer to it in one safe step. If two jobs race, only one switch wins; the other tries again. Old save points stay on disk — that is what makes time travel possible.

Best at

Being read and written by many tools — Spark, Flink, Trino, Snowflake, BigQuery, AWS — all on one copy of the data, and you are never stuck with one company's product (people call this "no lock-in"). It can also change how a table is split into folders without rewriting old data.

Watch out

A few more parts to manage than Delta's single log. And new features are still arriving across tools at different speeds.

New in 2025 (Iceberg v3): deletion vectors, row-level change tracking (so finding what changed is easy), and the same Variant type Delta uses — on purpose, so the two grow closer.

Apache Hudi

"The fast-update specialist for live data."

The story: built at Uber in 2016 to land a constant stream of changes into the lake within minutes. The name stands for Hadoop Upserts, Deletes, Incrementals (Hadoop was an older big-data system).

How it works inside

Hudi is different: every record has a key (like a customer id), and Hudi keeps an index (a simple lookup: key → which file) that remembers where each key lives. So when one record changes, Hudi jumps straight to the right file — it does not scan the whole table. It records all actions on a timeline (its logbook).

The choice that defines Hudi

Hudi lets you pick how updates are stored — the two styles from block 9. Copy-on-Write rewrites the file now (fast reads, heavier writes); Merge-on-Read notes the change in a small side-log and merges it later (fast writes, a bit more work to read). The diagram below shows both.

Best at

Data that changes all the time: live feeds, constant updates and deletes, and reading "only what changed since last time".

Watch out

It has the most parts to run and tune, and the weakest support on Databricks.

New in 2025 (Hudi 1.0): it now calls itself "a database on the lake" — a faster timeline, the ability to update only a few fields of a record, and writers that do not block each other (so a live feed and a cleanup job can run together).

~/lakehouse ❯ the recent change

05. What changed in 2025: they are growing alike

For years people argued about which one "wins". People argue about this much less now. In 2025 all three added the same main new features — deletion vectors for fast deletes, a shared Variant type for JSON-like data, and Iceberg even added the change-tracking that used to be only Hudi's feature.

On top of that, bridge tools now let one set of files be read as more than one format. Delta's UniForm lets Iceberg tools read a Delta table. Apache XTable translates between all three. So:

Picking a format no longer traps you. Choose for today; you can switch later using the bridge tools.

…but do not over-trust the bridges. They have real limits. UniForm only goes one way: other tools should use it to read the table, not write to it. If a tool ignores that and writes through it anyway, it can break the Delta table. And both bridges do not work well with the newest features (deletion vectors, Merge-on-Read). So "the format does not matter anymore" is an exaggeration. It matters least for plain tables, and most for the newest features.

Also, the argument did not end — it just changed topic. The main question in 2026 is no longer the file format. It is the catalog — the service that names and protects your tables (for example Unity Catalog or Apache Polaris). Choosing a format is now easy; choosing a catalog is the harder, separate decision. We will not cover it here. And if you ever pick wrong, switching means a one-time conversion (or a bridge for reads) — possible, but plan for it.

The part people forget to mention:

• You now do the cleanup. Those tiny files do not tidy themselves. You will run compaction and remove old versions, or reads get slow and storage grows.

• Company speed tests are marketing. The company that makes the test always makes itself look best. Trust a test on your own data.

~/lakehouse ❯ your decision

07. Which one should you pick?

Follow the branch that matches your biggest need.

The plain verdict:

→ Delta if you use Databricks/Spark and mostly do reports and dashboards.

→ Iceberg if many tools must read one copy, or you live on AWS data tools.

→ Hudi if your main job is constant, fast, live updates.

Not sure? Pick the one your tools already support best, and use the bridge tools if you ever need the others.

Starting fresh, with no tools pushing you one way? Iceberg has become the common first choice — but then think carefully about your catalog (see above).

~/lakehouse ❯ a real example

08. A real example: how a bank chose

Theory is tidy; real choices are more interesting. Here is how the trade-offs play out for a real company — a large bank in Vietnam building a modern data platform for its millions of customers. I will keep this at the "why" level, not the deep tech.

What shaped the choice

  • A strict regulator. A bank answers to a central bank that wants strong proof: who changed this number, where did it come from, and can we show what it was last quarter? This is the law, not a nice-to-have.
  • A mixed set of tools. Data is born in many source systems and first lands on AWS. The serious analysis, dashboards, and machine learning happen on Databricks.
  • The work is mostly overnight batch (one big run at night). Clean the day's data, improve it step by step, and serve reports — not second-by-second live updates.

The decision: not one winner, but the right tool per job

The useful lesson is that the bank did not choose a single winner. It runs two environments, and the format follows the environment:

  • The landing area (on AWS) uses Iceberg. When data first arrives from source systems — like the core banking system — it is written to AWS storage as Iceberg tables, processed by Spark on Amazon EMR with the AWS Glue catalog. Iceberg is the natural, open choice for that AWS + Spark stack, and it lets several AWS tools read the same data.
  • The refined layers (on Databricks) use Delta. From the landing area, the data is brought into Databricks and improved step by step — a raw copy, then a cleaned layer, then business-ready and report-ready layers. Every one of these layers is stored as Delta. Delta is the native, low-effort choice on Databricks, and Databricks' catalog gives the change-history (who changed what, and what a number was last quarter) and the access control the regulator needs.

One real detail worth seeing: a few source systems already hand the bank their data as Iceberg (for example, customer master data). Instead of always copying it, the team decides case by case whether to convert it to Delta or simply read it where it sits through a bridge (UniForm, or a read-only connection to the Iceberg tables). That is the convergence story from section 05 — happening in real production.

The lesson is not "Delta won". It is "let your platform, your workload, and your rules decide — not the marketing".

Why no Hudi?

Not because Hudi is weak — it is excellent at its job. It just does not match this bank's shape. The work is overnight batch, not constant live updates, so Hudi's special power would not be used. And it is not well supported on the bank's analysis platform. Adding it would mean more cost and more parts to run, for a benefit the work does not need. The best tool is the one your problem actually asks for.

What would change the answer?

One thing: a real need for live, second-by-second data — for example, catching fraud as it happens. That is the exact job that would make a live-first format worth considering again. Until then, "Iceberg for landing on AWS, Delta for everything inside Databricks" is a stable, low-risk setup that is easy to justify — and thanks to the bridge tools above, easy to change later too.

~/lakehouse ❯ getting started

09. What does using one look like?

Good news: you mostly write normal SQL, and choosing a format is usually one or two words in your CREATE TABLE. Here is the whole day-to-day life of a table on Spark / Databricks — four moves that cover almost everything.

1. Create a table. The only real difference between the three is the format name; everything after this point looks identical:

-- pick ONE format name — the rest of the workflow is the same CREATE TABLE sales USING delta AS SELECT * FROM raw_sales; CREATE TABLE sales USING iceberg AS SELECT * FROM raw_sales; CREATE TABLE sales USING hudi AS SELECT * FROM raw_sales; -- + a few extra options

2. Apply daily changes with MERGE — the upsert from building block 5. One statement updates rows that already exist, inserts brand-new ones, and can delete on request. This is the exact pattern behind keeping a table in sync with a source every night:

MERGE INTO sales t USING daily_changes s ON t.id = s.id WHEN MATCHED AND s.deleted THEN DELETE WHEN MATCHED THEN UPDATE SET * WHEN NOT MATCHED THEN INSERT *;

3. Look at the past (time travel). Because the logbook keeps old versions, reading last week's table — or undoing a bad load — is a single line:

-- read the table as it was at an older version, or a date SELECT * FROM sales VERSION AS OF 42; SELECT * FROM sales TIMESTAMP AS OF '2026-06-01'; -- undo a bad load: bring the live table back to a good version RESTORE TABLE sales TO VERSION AS OF 42;

4. Routine cleanup. Remember from section 06: nothing tidies itself. Two commands you will schedule — merge the tiny files, and drop old versions you no longer need:

OPTIMIZE sales; -- compact many small files into a few big ones VACUUM sales RETAIN 168 HOURS; -- remove unreferenced files older than 7 days

The exact spelling differs a little per engine — Hudi usually needs a few more settings, and Iceberg writes some of these as stored procedures (for example CALL system.rewrite_data_files instead of OPTIMIZE) — but the four moves are the same everywhere. Learn create → MERGE → time travel → clean up, and you can run any of the three.

What does it cost? The formats themselves are free and open. Your real costs are storage (usually cheap) and the engine you run to read and write (this is the bigger cost). On effort: Delta needs the least tuning, Iceberg a little more, and Hudi the most — so Hudi usually needs more skilled people. Keep that in mind when your team is small.

∎ The takeaway

Open table formats exist to do one beautiful thing: give a cheap pile of files the reliability and safety of a real database. Delta keeps the logbook simple. Iceberg keeps it open. Hudi keeps it fast for live data. They are growing alike, the bridges are real, and the "right" choice is usually whichever one your tools already support best.

So do not worry. Learn the one idea, match the tool to your situation, and build it.

Read more: the official sites — delta.io, iceberg.apache.org, hudi.apache.org — and the 2025 release notes for Delta 4.0, Iceberg v3, and Hudi 1.0.
