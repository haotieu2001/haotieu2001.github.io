Everything you need to actually understand dbt (data build tool) — from your first ref() to a governed, multi-project deployment running on a schedule in front of a real warehouse. ~60 min read · updated 2026 · written in plain English, no assumed background

Every data team eventually hits the same wall: a folder full of ad-hoc SQL scripts, nobody quite sure which one runs first, no tests, no documentation, and one person who "just knows" how the numbers get built. dbt (data build tool) exists to fix exactly that problem — not by replacing your warehouse's SQL engine, but by giving your SQL a project structure, a dependency graph, tests, documentation, and a repeatable way to deploy it, the same way software engineers already treat application code.

This guide is built around a real, working example: a dbt project that transforms raw MovieLens rating data (styled as a "Netflix" analytics project for a portfolio, from a widely-watched YouTube tutorial) into clean dimension and fact tables in Snowflake — staging models, dimension/fact tables, a snapshot, custom tests, and a mart. That concrete project shows up throughout this guide as a worked example, but the guide itself covers the full breadth of dbt: every core concept, dbt's newest 2024-2026 features, and — because "I understand dbt" and "I could run dbt in production" are very different skills — a dedicated look at how real companies actually deploy, orchestrate, test, and govern dbt once it's live.

❯ Short version (read this first)

dbt lets you write your data transformations as SELECT statements in .sql files. Each file is a model. Models can reference each other with ref() instead of hardcoding table names, and dbt uses those references to build a DAG (dependency graph) automatically, so it always runs things in the right order.

dbt does not move data and does not execute anything itself — it compiles your Jinja-templated SQL into plain SQL and pushes it to your warehouse (Snowflake, BigQuery, Databricks, Postgres, Redshift, and others) to actually run. dbt is the organizing layer on top of SQL your warehouse already understands.

• dbt Core — the free, open-source CLI that compiles and runs your project. Orchestration (dbt Cloud, Airflow, Dagster, or a simple cron job) is what actually triggers dbt run on a schedule in production.

• The warehouse — Snowflake, BigQuery, etc. — does 100% of the actual computation. dbt just tells it what SQL to run and in what order.

In production: raw data lands in the warehouse (via Fivetran, Airbyte, or a native loader) → dbt models transform it in layers (staging → intermediate → marts) → a scheduled job runs dbt build → tests run automatically → BI tools and analysts query the resulting marts.

New to the words? A model is one .sql file dbt turns into a view or table. The DAG is the graph of which models depend on which. A materialization is how a model gets physically built (view, table, incremental, etc). ref() is how one model points to another by name, not by hardcoded database/schema. A source is a pointer to a raw table dbt didn't create.

01. Why dbt Exists

From "a folder of SQL scripts" to a real, tested, documented codebase

Before dbt, most SQL-based transformation work looked the same everywhere: a pile of .sql scripts, run by hand or by a fragile cron job, in whatever order someone remembered to run them in. No one could easily answer "what depends on this table?" or "did this number change because of a bug?" — because nothing wrote that dependency down anywhere except in someone's head. dbt (data build tool) exists to fix exactly that: it turns your SQL transformations into a real, version-controlled project — with dependencies, tests, and documentation — while leaving the actual data movement and computation to tools that already do those jobs well.

That last point matters more than it sounds: dbt does not extract data from source systems, and it does not execute SQL itself. It compiles the SQL you write (with some templating help from Jinja, section 08) and hands it to your warehouse — Snowflake, BigQuery, Databricks, Redshift, Postgres — which does 100% of the actual computation. dbt's job is everything around the SQL: dependency ordering, testing, documentation, and environments.

ETL vs. ELT — why dbt showed up when it did

Older data platforms did ETL: Extract data from a source, Transform it (usually in a separate processing tool, before it ever reached the warehouse), then Load the finished result. That made sense when warehouses were expensive and slow. Cloud warehouses changed the economics — storage and compute got cheap and fast enough that it became easier to Load raw data first and Transform it afterward, directly inside the warehouse, using the warehouse's own SQL engine: ELT. Tools like Fivetran and Airbyte handle the "EL" (getting raw data into the warehouse, largely unmodified); dbt is built specifically for the "T" — transforming that raw data into clean, tested, analysis-ready tables, using nothing but SQL the warehouse already understands.

The role dbt helped create: analytics engineer

dbt didn't just ship a tool — it popularized a job title. An analytics engineer sits between a data engineer (who builds pipelines that move raw data into the warehouse) and a data analyst (who answers business questions with that data). They own the transformation layer: writing and testing the SQL that turns raw, messy source tables into clean, well-modeled, documented tables analysts can trust — using software-engineering practices (version control, code review, tests, CI) on SQL that used to live in nobody's repository at all.

❯ Short version

dbt is the "T" in ELT: it doesn't move data or execute anything itself — it compiles SQL and pushes it into your warehouse, which does all the real computation. It exists to bring version control, dependency tracking, tests, and documentation to the SQL transformation layer, and it's the tool most responsible for "analytics engineer" becoming a real job title.

02. Core Concepts: Models, ref(), and the DAG

The three ideas everything else in dbt is built on

A model is the basic unit of work in dbt: one .sql file containing a single SELECT statement. There's no CREATE TABLE, no INSERT, no DDL of any kind in the file you write — dbt wraps your SELECT in whatever DDL/DML is needed based on how you've configured it to be built (section 06). If a file is named fct_orders.sql, running it produces an object named fct_orders in your warehouse.

ref() — the single most important function in dbt

Instead of writing SELECT * FROM analytics.staging.stg_orders, you write SELECT * FROM {{ ref('stg_orders') }}. That one substitution does two things: dbt resolves it to the right, fully-qualified table name for whichever environment you're running in (dev schema, staging, prod — you never hardcode a database or schema again), and — because dbt now knows this model reads from that one — it can automatically compute the correct build order for every model in your project. Do this consistently, and the DAG (directed acyclic graph) of your entire transformation layer falls out for free, without you ever writing it down by hand.

-- models/marts/fct_orders.sql
WITH orders AS (
    SELECT * FROM {{ ref('stg_orders') }}
),
customers AS (
    SELECT * FROM {{ ref('dim_customers') }}
)

SELECT
    o.order_id,
    o.customer_id,
    c.customer_segment,
    o.order_total
FROM orders o
LEFT JOIN customers c ON o.customer_id = c.customer_id

The other half of this is source() (covered fully in section 04): a pointer to a raw table dbt did not create — something a loader like Fivetran or a native pipe wrote. Using source() instead of hardcoding the raw table name gives dbt the starting nodes of the DAG, plus a place to declare freshness checks on data it doesn't own. Between ref() and source(), every table your project touches — the ones it builds and the raw ones it reads from — is represented as one connected graph.

Two commands make the compile/run distinction concrete: dbt compile takes every model's Jinja-templated SQL and resolves it down to plain SQL your warehouse could run directly — writing the result to target/compiled/ without touching the warehouse at all. dbt run does that same compilation and then actually executes the result against the warehouse, writing what it ran to target/run/. If you ever want to know exactly what SQL dbt is about to run, or debug why a model behaves unexpectedly, reading its compiled file in target/compiled/ is the fastest way to find out — it's the literal SQL your warehouse receives, with every ref() and macro already expanded.

❯ Short version

A model is one .sql file with one SELECT. ref() points to another model by name instead of a hardcoded table path, which both makes your project environment-independent and lets dbt compute the DAG automatically. source() plays the same role for raw tables dbt didn't build. dbt compile resolves Jinja to plain SQL without running it; dbt run compiles and actually executes it.

03. Project Anatomy

dbt_project.yml, profiles.yml, and the standard folder layout

Every dbt project is a plain folder, almost always checked into git. Two YAML files control almost everything about how it behaves, and they intentionally hold very different kinds of information.

Installing dbt

dbt Core installs like any Python package: pip install dbt-core dbt-snowflake — swap dbt-snowflake for whichever adapter matches your warehouse (dbt-bigquery, dbt-databricks, dbt-postgres, and so on). The adapter is the piece that actually knows how to talk to that specific warehouse; dbt Core itself is warehouse-agnostic. dbt init <project_name> scaffolds the standard folder layout below and interactively walks you through your first profiles.yml entry. Run dbt debug immediately after — it confirms dbt can actually reach your warehouse before you write a single model.

dbt_project.yml vs. profiles.yml

dbt_project.yml lives inside the project, is committed to git, and configures the project itself: its name, which folders hold models/seeds/macros/tests/snapshots, and default materializations per folder. profiles.yml lives outside the project (by default in ~/.dbt/) and is never committed to git, because it holds warehouse connection details — account, user, and (for local dev) credentials. dbt Cloud sidesteps this file entirely by managing connections and secrets through its own UI and environment variables instead of a local file on disk.

# dbt_project.yml
name: 'analytics'
version: '1.0.0'
profile: 'analytics'          # which profile in profiles.yml to use

model-paths:    ["models"]
seed-paths:     ["seeds"]
macro-paths:    ["macros"]
test-paths:     ["tests"]
snapshot-paths: ["snapshots"]
clean-targets:  ["target", "dbt_packages"]

models:
  analytics:
    +materialized: view      # default for everything...
    marts:
      +materialized: table   # ...override per folder

One folder is easy to overlook: analyses/ holds .sql files that dbt will compile — resolving every ref(), source(), and Jinja block exactly like a model — but never actually run or materialize as a table or view. It's the right home for a one-off, ad-hoc analytical query you still want version-controlled and benefiting from dbt's templating (say, an investigation into a data-quality anomaly, or a query a stakeholder asked for once) without permanently cluttering your DAG with a model nobody downstream should depend on.

What actually goes in git. Everything in the project folder except target/ and dbt_packages/ (both regenerated by dbt deps/dbt run) belongs in version control — including dbt_project.yml and packages.yml. profiles.yml and anything holding a password, API key, or account identifier does not — those are secrets, and belong in environment variables or your orchestrator's secret store instead, exactly as they would for any other application code.

❯ Short version

dbt_project.yml is committed config for the project itself (folders, default materializations). profiles.yml holds warehouse connection secrets and is never committed — dbt Cloud replaces it with UI-managed connections entirely. The standard folder layout splits models into staging/intermediate/marts, alongside seeds, snapshots, macros, and tests; target/ is generated output and always gitignored.

04. Sources & Seeds

Declaring the raw tables you didn't build, and loading the small ones you did

Every dbt project starts by reading data it didn't create. Sources are how you tell dbt about it: a YAML declaration mapping a friendly name to a real, already-existing table (usually one a loader like Fivetran, Airbyte, or a native pipe populated). You reference a source in a model with {{ source('source_name', 'table_name') }} — the same "don't hardcode the path" idea as ref(), extended to raw data.

# models/staging/sources.yml
version: 2

sources:
  - name: shop_raw
    schema: raw
    tables:
      - name: orders
      - name: customers
        loaded_at_field: _fivetran_synced
        freshness:
          warn_after: {count: 12, period: hour}
          error_after: {count: 24, period: hour}

Source freshness — catching a broken loader before your dashboards do

dbt source freshness checks how long ago each source table was last updated (via its loaded_at_field) and compares that gap against the warn_after/error_after thresholds you set. If the upstream loader silently stops working overnight, this is what tells you before a stakeholder notices a dashboard hasn't moved in three days — a cheap, automatable check most teams run at the very start of every production job, before wasting compute rebuilding models on top of stale data.

Seeds solve a different, smaller problem: small, static, rarely-changing datasets that live comfortably as a CSV in your repo — a lookup of country codes to region names, a manually-curated list of holiday dates, an internal mapping table nobody else owns. You put the CSV in your seeds/ folder, run dbt seed, and dbt loads it into the warehouse as a real table you can then ref() exactly like any model. Seeds are for reference data measured in kilobytes, not a substitute for a real ingestion tool — if a dataset changes often or grows large, it belongs behind a source, not a seed.

❯ Short version

A source declares a raw table a loader (Fivetran, Airbyte, etc.) already populated, referenced with source(); it's also where you configure freshness checks that catch a broken loader early. A seed is a small, static CSV committed to the repo and loaded with dbt seed, then referenced with ref() exactly like a model — meant for small reference data, not real ingestion.

05. The Layered Architecture

The folder convention almost every real dbt project converges on

Nothing forces you to organize models any particular way, but almost every production dbt project — including dbt Labs' own recommended structure and the worked example in section 20 — converges on the same three layers, because it maps cleanly onto how a DAG naturally grows.

Staging → Intermediate → Marts

Staging models sit directly on top of sources: one staging model per source table, doing only light, mechanical cleanup — renaming columns to a consistent convention, casting types, standardizing casing — never joins, never business logic. Intermediate models sit between staging and marts: this is where joins, deduplication, and business logic building blocks live, often not meant to be queried directly by anyone outside the project (frequently built as ephemeral, section 06, so they don't even appear as objects in the warehouse). Marts are the final, consumption-ready layer — the dimension and fact tables (or wide, denormalized tables) that BI tools, analysts, and other teams actually query.

Naming convention most teams follow: stg_orders, stg_customers (staging) → int_orders_joined (intermediate) → dim_customers, fct_orders (marts, Kimball-style dimension/fact naming).

Ties into section 20. The Netflix/MovieLens worked example this guide references follows exactly this shape: one src_* staging model per raw MovieLens table, dimension and fact models (dim_movies, dim_users, fct_ratings) built on top of staging, and a final mart joining a fact table against a seed. Section 20 walks through it end to end.

❯ Short version

Staging models do light, 1:1 cleanup on top of sources with zero business logic. Intermediate models hold joins and business-logic building blocks, often not meant for outside consumption. Marts are the final, wide, dimension/fact tables analysts and BI tools actually query. This funnel shape — many messy sources in, a few trustworthy marts out — is the closest thing dbt has to an industry-standard project structure.

06. Materializations

view, table, ephemeral, incremental — and the newer microbatch strategy

A model file is just a SELECT. Its materialization is the setting that decides what dbt actually builds from that SELECT — and it's the single highest-leverage decision you make per model, because it trades off build cost against query cost and freshness.

MaterializationWhat gets builtRebuild costBest for
view (default)A SQL view — no data stored, query re-runs the SELECT every timeFree to "build" (just DDL)Staging models; anything cheap to query and rarely reused
tableA real physical table, fully rebuilt (dropped & recreated) every runFull cost, every runMarts that need fast reads and don't grow unbounded
ephemeralNothing at all — inlined as a CTE into every model that refs itNone — no object existsIntermediate models nobody queries directly
incrementalA table that's only appended to / merged, not fully rebuilt each runCheap — only processes new/changed rowsLarge, append-heavy fact tables (section 07)
 
-- models/marts/dim_customers.sql
{{ config(materialized='table') }}

SELECT customer_id, first_name, last_name, signup_date
FROM {{ ref('stg_customers') }}

Newer addition: microbatch. As of recent dbt Core releases, incremental_strategy: microbatch gives you a purpose-built strategy for very large, time-partitioned fact tables: instead of one big incremental SQL statement, dbt breaks the run into many small, independent per-time-slice batches (e.g. one batch per day), each processed and retried independently. That makes backfills over months of history dramatically more resilient — a single failed batch doesn't take down the whole run, and failed batches can be retried individually. Covered in more depth in section 19.

❯ Short version

view stores nothing and re-runs on every query; table is fully rebuilt every run; ephemeral creates no object at all and is inlined into whatever refs it; incremental only processes new or changed rows on top of an existing table. microbatch is a newer incremental strategy purpose-built for large, time-partitioned tables, splitting a run into small, independently-retryable per-time-slice batches.

07. Incremental Models, Deep Dive

is_incremental() and the strategies for merging new data in

An incremental model's SQL file has to serve two different purposes depending on how it's run: on a full refresh (dbt run --full-refresh, or the model's very first run) it needs to select the entire history; on every run after that, it should select only new or changed rows and merge them into the table that already exists. The is_incremental() macro is how one file does both jobs.

-- models/marts/fct_events.sql
{{
  config(
    materialized='incremental',
    unique_key='event_id',
    incremental_strategy='merge'
  )
}}

SELECT event_id, user_id, event_type, event_time
FROM {{ ref('stg_events') }}

{% if is_incremental() %}
  -- this filter only applies on incremental runs, never on a full refresh
  WHERE event_time > (SELECT MAX(event_time) FROM {{ this }})
{% endif %}

Incremental strategies — how new rows actually get merged in

append just inserts the new rows — fastest, but unsafe if a row you already loaded could show up again with changed values (you'd get a duplicate, not an update). merge (the most common choice, on warehouses that support it) does a real MERGE/upsert keyed on unique_key: an existing row gets updated in place, a new row gets inserted — safe against reprocessing the same time window twice. delete+insert deletes any existing rows matching the new batch's keys, then inserts the new batch — a common fallback on warehouses without a native merge. microbatch (section 06, section 19) is newer still: it doesn't compute a single incremental filter at all — dbt itself decides the time slices and runs one batch per slice.

The classic incremental footgun: schema changes. If an upstream source silently adds, removes, or changes the type of a column, an incremental model's next run tries to merge new-shape rows into an old-shape table. The on_schema_change config decides what happens: ignore (default, often wrong — can silently drop or misalign columns), fail (stop and force a human to look, the safest default for anything important), append_new_columns, or sync_all_columns. Pick this deliberately per model instead of leaving it at the default.

❯ Short version

is_incremental() lets one model file behave differently on a full refresh vs. a normal run — normally filtering to only new/changed rows. The incremental_strategy (append, merge, delete+insert, or the newer microbatch) decides exactly how those new rows get combined with what's already there. Always set on_schema_change deliberately — the default can silently corrupt a table when an upstream schema changes.

08. Jinja & Macros

Why dbt files aren't quite plain SQL, and how to reuse logic across models

Every dbt model is actually a Jinja template that happens to render into SQL. {{ ... }} is an expression that gets replaced with its rendered value ({{ ref('stg_orders') }} becomes a real table path); {% ... %} is a statement — an if, a for loop, a macro call — that controls what gets rendered but produces no output itself. This is what makes is_incremental() (section 07) and looping over column lists (like generating a pivot) possible in something that's ultimately just a SQL file.

Macros — functions for SQL

A macro is a reusable, parameterized chunk of Jinja+SQL, defined once in your macros/ folder and called from any model. The most common motivating example is a repeated pattern you don't want to retype in ten different models — computing a surrogate key from several columns, standardizing a currency conversion, or generating a custom test (a macro is also how you write a custom generic test, section 09).

-- macros/cents_to_dollars.sql
{% macro cents_to_dollars(column_name) %}
    ({{ column_name }} / 100.0)
{% endmacro %}

-- used in a model:
SELECT
    order_id,
    {{ cents_to_dollars('amount_cents') }} AS amount_dollars
FROM {{ ref('stg_orders') }}

You rarely have to write every macro yourself. Packages — declared in packages.yml and installed with dbt deps — are shareable dbt projects made almost entirely of macros and, sometimes, models. dbt_utils (maintained by dbt Labs) is the one nearly every project installs, for things like dbt_utils.generate_surrogate_key() (a stable hash of several columns, commonly used as a primary key for a snapshot or a table with no natural key) and dbt_utils.date_spine() (generate a row per day/week/month over a date range). dbt-expectations ports much of the Python library Great Expectations' test vocabulary into dbt-native generic tests.

# packages.yml
packages:
  - package: dbt-labs/dbt_utils
    version: 1.3.0

❯ Short version

dbt models are Jinja templates that render to SQL: {{ }} for expressions, {% %} for control flow. A macro is a reusable, parameterized snippet of Jinja+SQL you define once and call everywhere. Packages (installed via packages.yml + dbt deps) are shareable bundles of macros and models — dbt_utils is the one almost every project installs.

09. Testing in dbt

Generic tests, singular tests, and unit tests — three different questions, three different tools

A dbt test is fundamentally simple: it's a query, and if that query returns any rows, the test fails. Everything about dbt testing is built on that one idea, applied to three different kinds of questions.

Test typeAnswersDefined whereRuns against
Generic test"Does this column ever violate a simple, reusable rule?" (not_null, unique, accepted_values, relationships)schema.yml, applied to a columnReal data, after the model is built
Singular test"Does this specific, one-off business rule ever get violated?"One .sql file in tests/Real data, after the model is built
Unit test"Given this fake input, does my SQL logic produce the expected output?"tests: block in the model's YAML, with mock rows inlineFabricated rows, not your real data, and not the warehouse's real tables
 
# models/schema.yml — generic tests
models:
  - name: fct_orders
    columns:
      - name: order_id
        tests: [unique, not_null]
      - name: customer_id
        tests:
          - relationships:
              to: ref('dim_customers')
              field: customer_id
      - name: status
        tests:
          - accepted_values:
              values: ['placed', 'shipped', 'delivered', 'cancelled']
-- tests/orders_total_matches_line_items.sql  (singular test)
-- fails if any order's stored total doesn't match the sum of its line items
SELECT o.order_id
FROM {{ ref('fct_orders') }} o
JOIN (
    SELECT order_id, SUM(line_amount) AS computed_total
    FROM {{ ref('fct_order_lines') }}
    GROUP BY order_id
) l ON o.order_id = l.order_id
WHERE ABS(o.order_total - l.computed_total) > 0.01

Unit tests — testing logic without touching real data

Generic and singular tests both run after a model is built, against whatever real data happens to be sitting in your warehouse — great for catching bad data, useless for testing a tricky piece of SQL logic in isolation (you'd need real rows that happen to hit every edge case). A unit test gives a model a handful of hand-written fake input rows and asserts on the exact output rows your SQL should produce from them — the same "arrange, act, assert" idea as a unit test in application code, finally available for SQL. This is the tool for "I refactored this CASE statement, prove it still does the same thing" — not for catching bad production data.

# models/schema.yml — a unit test
unit_tests:
  - name: test_discount_tier_logic
    model: fct_orders
    given:
      - input: ref('stg_orders')
        rows:
          - {order_id: 1, order_total: 40}
          - {order_id: 2, order_total: 150}
    expect:
      rows:
        - {order_id: 1, discount_tier: 'none'}
        - {order_id: 2, discount_tier: 'gold'}

❯ Short version

Every dbt test fails if its query returns any rows. Generic tests (not_null, unique, relationships, accepted_values) apply a reusable rule to a column via YAML. Singular tests are one-off SQL files for a specific business rule. Both run against real, already-built data. Unit tests are different in kind — they feed a model fabricated input rows and assert on the exact output, testing your SQL logic in isolation rather than your production data's health.

10. Snapshots

Capturing change history from a source that only shows you "now"

Most operational source tables only show you the current state — a customers table with a status column shows today's status, with no record of what it was last month. If you need to answer "what was this customer's status on any given date in the past," the source table alone can't tell you, because the row was overwritten in place. A snapshot is dbt's answer: on every run, it compares the current source data to the last snapshot it took, and records any change as a new row — building up exactly the kind of change history relational modeling calls Slowly Changing Dimension Type 2 (SCD Type 2).

-- snapshots/snap_customer_status.sql
{% snapshot snap_customer_status %}

{{
  config(
    target_schema='snapshots',
    unique_key='customer_id',
    strategy='timestamp',
    updated_at='updated_at'
  )
}}

SELECT customer_id, status, updated_at
FROM {{ source('shop_raw', 'customers') }}

{% endsnapshot %}

Two change-detection strategies

The timestamp strategy trusts an updated_at column the source already maintains — simplest and cheapest, but only as reliable as that column actually is. The check strategy instead compares a specific list of columns row-by-row on every run and considers the row changed if any of them differ — more expensive, but works even when the source has no trustworthy update timestamp at all.

❯ Short version

A snapshot captures change history (SCD Type 2) from a source that only shows its current state, by comparing new data against the last snapshot on every run and appending a new row for anything that changed. The timestamp strategy trusts an existing updated_at column; check compares specific columns directly when no trustworthy timestamp exists. The resulting table only grows, with dbt_valid_from/dbt_valid_to marking each version's validity window.

11. Documentation & the DAG

Turning your schema.yml descriptions into a browsable lineage graph

Every description you write in a schema.yml file — for a model, a column, a source — isn't just a comment; it's structured metadata dbt can render into an actual documentation site. dbt docs generate builds that site (reading every model's compiled SQL, every test, every description) and dbt docs serve runs it locally so you can click through it. The centerpiece is the lineage graph: a visual, interactive rendering of the exact DAG built from every ref() and source() in your project (section 02) — click any model and see everything upstream and downstream of it, instantly, instead of grepping through SQL files trying to reconstruct dependencies by hand.

# models/schema.yml
models:
  - name: fct_orders
    description: "One row per order. The core sales fact table used by every revenue dashboard."
    columns:
      - name: order_id
        description: "Primary key. Unique per order."

Exposures — extending the DAG past dbt's own models

An exposure declares something outside dbt — a Looker dashboard, a Tableau workbook, an ML feature pipeline — that depends on specific models, so it shows up as a node in the lineage graph too. That closes an otherwise-invisible gap: without an exposure, "who actually uses fct_orders" is tribal knowledge; with one, it's a documented, visible dependency, which matters enormously once you're deciding whether it's safe to change or deprecate a model (and is one of the building blocks dbt Mesh, section 14, uses to reason about cross-team dependencies).

❯ Short version

Descriptions in schema.yml aren't just comments — dbt docs generate turns them into a real documentation site, centered on an interactive lineage graph built from every ref()/source() in the project. Exposures extend that graph past dbt's own models to represent dashboards, ML pipelines, or anything else downstream that depends on a model, turning "who uses this" from tribal knowledge into a documented dependency.

12. dbt Core, dbt Cloud, and the Fusion Engine

Same modeling language, three very different ways to actually run it

Everything in sections 01-11 — models, ref(), tests, snapshots — is the same regardless of how you run dbt. What differs is the execution layer underneath.

dbt Core vs. dbt Cloud

dbt Core is the free, open-source, MIT-licensed command-line tool. You install it with pip, run dbt run/dbt test/dbt build yourself, and you're responsible for hosting it somewhere and triggering it on a schedule (section 16). dbt Cloud is dbt Labs' commercial, managed platform: a hosted IDE for developing models, a scheduler for production jobs, a hosted docs site, environment/secret management with no local profiles.yml, and features that only exist there (like the original dbt Semantic Layer API and the dbt Cloud CLI). Neither replaces the modeling language you learned in sections 01-11 — dbt Cloud is a management and orchestration layer wrapped around the same core engine.

The Fusion engine. dbt Labs has been building a new execution engine — written in Rust rather than Python — with a real SQL comprehension layer that understands column-level lineage and type information before ever sending a query to the warehouse, instead of dbt's traditional approach of only understanding your project through Jinja templating and the compiled SQL text. The goal is faster local compilation, better editor tooling (real autocomplete and error-checking on your SQL, not just your Jinja), and faster CI feedback. Section 19 covers where this stands as of this guide's most recent update — engine internals are exactly the kind of thing that moves fast, so treat any specific version/availability claim as a snapshot in time, not a permanent fact.

❯ Short version

dbt Core is the free, open-source CLI — you host and schedule it yourself. dbt Cloud is dbt Labs' managed platform, adding a hosted scheduler, IDE, docs site, and secrets management on top of the same core engine. The Fusion engine is a newer, Rust-based execution engine with real SQL comprehension, aimed at faster compilation and better tooling — see section 19 for its current status.

~/dbt ❯ promises a model isn't allowed to break

13. Data Contracts & Model Versioning

Turning "please don't change this column" into something dbt actually enforces

Once a model has consumers outside your own team — a dashboard, another team's dbt project (section 14), a downstream ML pipeline — an innocent-looking change (renaming a column, changing a type from integer to string) can silently break something you'll never hear about until it's a production incident. Data contracts and model versions are dbt's answer: turning an informal promise about a model's shape into something dbt actively checks.

# models/schema.yml
models:
  - name: dim_customers
    config:
      contract:
        enforced: true
    columns:
      - name: customer_id
        data_type: int
        constraints:
          - type: not_null
      - name: email
        data_type: string

What "enforced" actually does

With contract.enforced: true, dbt checks — before building the model — that its SELECT statement's actual output columns, order, and types exactly match what's declared in schema.yml. If a column is missing, extra, reordered, or the wrong type, the run fails immediately with a clear error, instead of silently building a table that no longer matches what downstream consumers expect. This converts "someone changed the model and something broke three hops downstream two weeks later" into "the build failed right here, right now, with a message pointing at the exact mismatch."

Model versioning extends this to changes you make on purpose: if you need to change a contracted model's shape in a breaking way, you publish it as a new version (v2) while the old version (v1) keeps running unchanged for consumers who haven't migrated yet — the same "don't break existing callers" discipline an API would apply to a breaking endpoint change, now available for a table.

❯ Short version

A data contract declares a model's exact columns, order, and types, and dbt checks the model's real output against that declaration before building it — turning a silent breaking change into an immediate, clear build failure. Model versioning lets you ship a breaking change as a new version while the old version keeps serving consumers who haven't migrated, avoiding a forced simultaneous break.

14. dbt Mesh & Multi-Project Architecture

What happens once one dbt project isn't the right unit of ownership anymore

Everything so far has assumed one dbt project, owned by one team. That works until an organization grows past it: a hundred-model project with five teams committing to the same repo turns code review, CI run times, and "who owns this model" into constant friction. dbt Mesh is dbt Labs' architecture for splitting a monolithic project into several smaller, independently owned projects that can still safely reference each other's models — the same "modular monolith to microservices" motivation applied to a dbt project instead of an application.

Cross-project ref() and groups

A model in a producer project can be exposed for other projects to consume; a consumer project then calls ref() across the project boundary exactly like it would within its own project — the mechanics don't change, only the ownership boundary does. Within a single project, groups let you restrict which models can ref() a given model at all (e.g. "only models in the finance group may select from this raw PII model directly"), enforcing team-level ownership boundaries even before you split into separate projects — often the natural first step before adopting full multi-project Mesh.

❯ Short version

dbt Mesh lets an organization split one large dbt project into several independently owned projects that can still reference each other's models via cross-project ref(), instead of forcing every team into one shared repo. Groups give you a lighter-weight version of the same ownership boundary inside a single project, before a full multi-project split is warranted.

~/dbt ❯ defining a metric once, not once per BI tool

15. The Semantic Layer & MetricFlow

Metrics as version-controlled code instead of a chart-by-chart definition

Without a semantic layer, "revenue" gets defined once in a Tableau workbook, slightly differently in a Looker explore, and slightly differently again in an analyst's ad-hoc SQL — three numbers, all called "revenue," none of them guaranteed to agree. dbt's Semantic Layer fixes this by letting you define a metric exactly once, as code, on top of your existing marts, and have every BI tool and query ask that definition for the answer instead of reimplementing it.

Semantic models and metrics — built on MetricFlow

A semantic model declares, on top of an existing mart, which columns are dimensions (things you group or filter by — order status, region, signup month) and which are measures (things you aggregate — order count, revenue). A metric is then built from one or more measures — a simple aggregation, a ratio of two measures, or a calculation derived from other metrics. MetricFlow is the query-building engine underneath all of this: given a metric and a requested set of dimensions, it compiles the correct SQL — joins, aggregation, and all — so nobody hand-writes that SQL per dashboard ever again.

# models/marts/_fct_orders.yml
semantic_models:
  - name: orders
    model: ref('fct_orders')
    defaults:
      agg_time_dimension: order_date
    dimensions:
      - name: order_date
        type: time
      - name: status
        type: categorical
    measures:
      - name: order_total
        agg: sum
      - name: order_count
        agg: count

metrics:
  - name: revenue
    type: simple
    type_params:
      measure: order_total

Once defined, a BI tool with Semantic Layer support (or a direct API/SQL query) asks for revenue grouped by order_date and status, and MetricFlow generates the correct SQL against your actual marts — nobody downstream needs to know or care how revenue is computed, only that it's the same definition everywhere it's used.

❯ Short version

The Semantic Layer lets you define a metric once, as version-controlled code on top of your existing marts, instead of redefining it separately in every BI tool. A semantic model declares dimensions and measures on a mart; a metric is built from those measures; MetricFlow is the engine that compiles a requested metric+dimensions combination into the correct SQL automatically.

~/dbt ❯ not testing the whole warehouse on every PR

17. CI/CD for dbt: Slim CI

Only building what actually changed, against a safe, isolated schema

A naive CI setup for dbt would run the entire project — every model, every test — on every pull request, which gets slow and expensive fast as a project grows past a few hundred models. Slim CI is dbt's answer: build and test only what a given PR actually changed, plus everything downstream of it, instead of the whole DAG.

# CI job step
dbt build --select state:modified+ --defer --state ./prod-artifacts

state:modified+ and --defer

state:modified+ compares the current branch's compiled project against a saved set of artifacts from a previous run (usually production's last successful run) and selects only the models that changed, plus (the trailing +) everything downstream of them — the minimum set of models a given change could possibly have affected. --defer then lets those selected models still resolve ref() calls to their unselected upstream parents' already-built production tables, instead of needing to rebuild the entire upstream DAG in a throwaway CI schema just to satisfy a foreign-key-style dependency. Together, these two flags are what makes a PR touching one model in a thousand-model project buildable in CI in seconds instead of hours.

Environment promotion in practice. Most teams run three tiers: a personal/PR-scoped schema for Slim CI (built fresh, torn down after), a shared staging environment for integration testing, and production. dbt Cloud automates the "compare against production's last state" plumbing Slim CI needs out of the box; a self-hosted dbt Core setup has to build that artifact-passing pipeline itself (typically: persist manifest.json from every successful production run somewhere CI can fetch it from).

❯ Short version

Slim CI builds only what a PR changed plus its downstream dependents (state:modified+), and --defer lets those models resolve unchanged upstream refs to production's already-built tables instead of rebuilding the whole DAG in CI. This is what keeps CI fast as a project grows — without it, every PR would rebuild everything.

18. Performance & Cost Optimization

The handful of decisions that actually move your warehouse bill

Because dbt compiles down to plain SQL the warehouse runs and bills for, every performance lever here is really a warehouse-level lever — dbt's job is making the right lever easy to reach for and consistent to apply across a whole project.

Where the cost actually goes, roughly in order of impact

• Materialization choice (section 06). A table that gets fully rebuilt nightly on top of billions of rows, when it could be incremental, is usually the single biggest avoidable cost in a project — full rebuilds scale with total table size, incremental runs scale with how much actually changed.

• Incremental strategy and predicate pushdown (section 07). An incremental filter that can't be pushed down to prune partitions (e.g. filtering on a computed expression instead of a raw partitioned column) forces a full scan anyway, quietly defeating the entire point of being incremental.

• Ephemeral overuse. Ephemeral models (section 06) are free to store, but the SQL still executes — every time. Making something ephemeral that's ref'd by fifteen downstream models means that same subquery gets computed fifteen times per run instead of once.

• Clustering / partitioning keys. Configuring a warehouse-native clustering key (Snowflake) or partition column (BigQuery) on large marts lets the warehouse skip scanning irrelevant data entirely — a dbt config() setting that maps directly to a warehouse feature, not a dbt-specific concept.

• Warehouse sizing per job. Most warehouses let you assign compute size per query/session; a job dominated by a few huge, wide joins benefits from a bigger warehouse for a shorter time far more than a project full of small, view-heavy staging models does.

Query tagging — the tool that makes any of this diagnosable. dbt can tag every query it sends with metadata (which model, which invocation, which dbt Cloud job) via warehouse-native session/query tags. Without this, "why did the warehouse bill spike Tuesday" means guessing; with it, it means filtering the warehouse's own query history by tag and finding the exact model responsible in minutes.

❯ Short version

Almost every real dbt cost problem traces back to one of: a full-rebuild table that should be incremental, an incremental filter that can't be pushed down to prune data, an over-used ephemeral model getting recomputed many times per run, or a missing clustering/partition key on a large table. Query tagging turns "the warehouse bill spiked" from a guessing game into a five-minute lookup.

19. What's New in dbt (2024-2026)

The Fusion engine, and what dbt Labs has actually confirmed about it

The single biggest thing happening in the dbt ecosystem right now is the dbt Fusion engine — and because engine internals and release timelines change fast, this section sticks closely to what dbt Labs has stated on its own official blog and docs, rather than repeating every performance number floating around online.

What Fusion actually is

dbt Core has always worked by treating your SQL as text: it renders Jinja into a SQL string and hands that string to the warehouse, without ever really "understanding" the SQL itself — no awareness of column-level lineage, types, or syntax errors until the warehouse itself complains. Fusion is a from-scratch rewrite of dbt's execution engine in Rust, built on SQL-compiler technology dbt Labs acquired from SDF Labs, that gives dbt genuine SQL comprehension: real parsing and understanding of your SQL and its column-level lineage across the whole project, multi-dialect, before anything is sent to the warehouse — while keeping the same SQL/YAML/Jinja authoring experience you already know from sections 01-11. dbt Labs launched Fusion as a public beta on May 28, 2025, rolling out warehouse support one adapter at a time, Snowflake first.

Status, honestly: still in preview, not GA. Despite an originally stated ambition to reach general availability within roughly a year of the beta, the most recent tracked release at the time of this guide's research was still labeled a preview build, with dbt Labs continuing to work toward full feature parity with dbt Core before calling it GA. Specific performance-multiplier claims (parsing/compilation speedups) circulate widely but didn't hold up to direct scrutiny of dbt Labs' own materials strongly enough to repeat as settled numbers here — if you need current figures, check docs.getdbt.com/docs/fusion/fusion-releases directly rather than trusting a number in any secondary write-up, including this one.

Licensing, clarified

dbt Labs has explicitly and repeatedly confirmed that dbt Core stays permanently licensed under Apache 2.0 — including future v2.x releases — with no change planned. A separate 2023 licensing action under a Business Source License applied only to a distinct, newly-created "dbt Server" component, never to dbt Core itself, and was designed to auto-convert to Apache 2.0 over time regardless. The Fusion engine's own specific licensing terms are a separate, still-evolving question dbt Labs has framed differently from dbt Core's — worth checking directly before assuming it inherits dbt Core's Apache 2.0 terms.

❯ Short version

The dbt Fusion engine is a from-scratch Rust rewrite (built on acquired SDF Labs SQL-compiler technology) that gives dbt real SQL comprehension and column-level lineage instead of treating SQL as opaque text — same authoring experience, different internals. It launched as public beta in May 2025, rolling out warehouse-by-warehouse, and was still in preview (not GA) as of this guide's research. dbt Core itself remains permanently Apache 2.0 licensed; Fusion's own licensing terms are a distinct, separately-evolving question.

20. Worked Example: A Movie-Ratings dbt Project on Snowflake

Every idea from sections 01-11, seen together in one small, real project

Abstract explanations only click so far. This section walks through the shape of a real, publicly documented dbt project — a widely-watched YouTube tutorial project that transforms the public MovieLens ratings dataset (millions of movie ratings, tags, and metadata) into a small analytics warehouse on Snowflake, presented as a "Netflix-style" portfolio project. It's a genuinely good teaching example precisely because it's small enough to hold in your head, while still touching almost every concept from sections 01-11.

What Snowflake actually is

Snowflake is a cloud data warehouse built around one architectural idea: storage and compute are completely separate. Your data sits once in Snowflake-managed cloud storage; any number of independently-sized virtual warehouses — the actual compute clusters that run queries — can read that same data at once, each scaling up, down, or to zero without touching the data itself or each other. That's why a dbt project typically runs against its own small, dedicated virtual warehouse: it scales independently of whatever warehouse a BI tool queries against at the same time, and you only pay for compute while a warehouse is actually running.

The setup: Snowflake plus a plain dbt Core project

The raw CSVs (movies, ratings, tags, genome scores/tags, cross-platform ID links) get into Snowflake through a standard cloud-to-warehouse loading path: the files are uploaded to an Amazon S3 bucket, Snowflake is pointed at that bucket via an external stage (a named reference to the S3 location plus the IAM role/credentials needed to read it), and a COPY INTO statement bulk-loads each CSV from the stage into a raw table — raw_movies, raw_ratings, raw_tags, and so on. (For data arriving continuously rather than as a one-time file drop, Snowflake's Snowpipe automates that same COPY INTO on every new file landing in the bucket — worth knowing exists, even though a one-time learning dataset doesn't need it.) A dedicated Snowflake role, warehouse, and database are then provisioned specifically for dbt to connect with (mirroring the general permissions story: a scoped, least-privilege service account rather than a personal login, section 12's "who's allowed to run this" concern showing up even in a learning project). From there it's a standard dbt Core CLI project — dbt init, a profiles.yml pointing at that Snowflake account, and the folder layout from section 03.

A few choices in this project are worth calling out because they're realistic, deliberate tradeoffs rather than arbitrary defaults — the same kind of decision section 06 and section 18 discuss in the abstract, made concretely here:

• Staging models default to views, dimension/fact models default to tables. Set once in dbt_project.yml per folder (section 03/06) rather than repeated in every model file — cheap-to-query marts, cheap-to-maintain staging.

• The ratings fact table is incremental, filtered to only pull rows newer than the table's current maximum rating timestamp on every run after the first — the exact is_incremental() pattern from section 07, chosen because a ratings table is by nature large and constantly append-heavy.

• A custom generic test, written as a macro, checks that no column in a given model contains a null by looping over that model's actual columns at compile time (via adapter.get_columns_in_relation) instead of hardcoding a column list — the Jinja-as-templating-language idea from section 08, applied to build a test (section 09) more general than dbt's built-in not_null, which only checks one column at a time.

• A snapshot tracks user tags over time using the timestamp strategy (section 10), keyed on the combination of user, movie, and tag — the standard shape for "this source can change, and I want the history, not just the latest state."

• The final mart is a straightforward left join of the fact table against a small, hand-curated seed of movie release dates (section 04) — a good real-world illustration of exactly when a seed is the right tool: a small, mostly-static reference dataset with no natural upstream source to pipe in through a real loader.

What this project doesn't show — and why that's normal. It's a single-developer, single-environment learning project, so it doesn't need dbt Mesh (section 14), a Semantic Layer (section 15), or Slim CI (section 17) — none of those solve a problem this project actually has. That's not a gap in the tutorial; it's exactly the right call. Reach for those tools when the problem they solve shows up (multiple teams, multiple environments, a slow CI pipeline), not by default just because they exist. Sections 14, 15, and 17 exist for the day this same project grows past one person and one environment.

❯ Short version

This project is a small, real dbt-on-Snowflake pipeline transforming the public MovieLens dataset: raw CSVs → Snowflake sources → staging views → dimension/fact tables (one incremental) → a mart joined against a seed, plus a custom generic test macro and a snapshot for tag history. It's a compact, realistic instance of nearly every idea in sections 01-11, and a useful reference point for how small a "real" dbt project can legitimately be.

21. dbt in Production: A Real Case Study

GitLab's own public playbook for running dbt Core at scale

Sections 16-18 described production dbt patterns in the abstract. GitLab is unusually useful here because, true to how it runs its whole business, it documents its own data platform in a public engineering handbook — making it one of the few named companies with a genuinely verifiable, detailed public account of running dbt in production, rather than a vague "we use dbt" case-study page.

GitLab's setup: self-hosted dbt Core, not dbt Cloud

GitLab runs self-hosted dbt Core — not dbt Cloud's managed jobs — orchestrated by Apache Airflow running as pods on Kubernetes, against a Snowflake warehouse. In production, this isn't one generic "run dbt" job: GitLab maintains several individually hand-written Airflow DAGs, each with its own hardcoded schedule, model selector, and warehouse size — for example, a daily snapshot-refresh DAG, a twice-daily core transformation DAG, a monthly full-refresh DAG, and a DAG validating and syncing Salesforce data roughly every six hours. Each DAG exists because a different part of the business needs a different freshness/cost tradeoff — daily snapshots don't need the same urgency as a twice-daily core refresh, and a monthly full refresh exists specifically to catch anything an incremental strategy (section 07) might have silently missed.

Even sophisticated dbt Core adopters outgrow hand-written orchestration. GitLab's handbook is candid that these DAGs currently exist as individually hand-written Python files — one per pipeline, cadence and configuration hardcoded into each. GitLab publicly documents a "future state, in progress": migrating toward a config-driven orchestration layer, where a schedule and selector are declared in a YAML manifest rather than duplicated as bespoke Python per pipeline. That's a genuinely useful signal for planning your own setup: hand-writing one Airflow DAG per pipeline is a completely reasonable way to start, but expect it to become the thing you refactor once the number of pipelines outgrows what's comfortable to hand-maintain.

On the CI/CD side (section 17), the pattern GitLab and dbt Labs' own official guidance converge on is the same: PRs build only state:modified+ against a deferred, designated production state — not the whole DAG. One specific, well-documented gotcha is worth calling out because it's easy to hit and non-obvious the first time: an incremental model silently full-refreshes inside CI, because is_incremental() (section 07) checks whether the target table already exists — and in a fresh, empty PR-specific schema, it doesn't, so dbt quietly builds the model as if from scratch, defeating the entire point of testing an incremental model's incremental logic. dbt Labs' own documented fix is to run dbt clone (zero-copy, on warehouses that support it) for exactly the incremental models affected by a PR before the build step, cloning their already-built production data into the PR schema so is_incremental() correctly evaluates true — a fix that requires dbt 1.6 or newer.

❯ Short version

GitLab publicly documents running self-hosted dbt Core via hand-written Airflow DAGs on Kubernetes against Snowflake — several separately scheduled DAGs, each tuned to a different pipeline's freshness/cost needs — and is itself migrating toward config-driven orchestration because hand-written per-pipeline DAGs stopped scaling comfortably. Its CI follows the same Slim CI pattern as section 17, plus one specific documented gotcha: incremental models silently full-refresh in a fresh CI schema unless you clone production's data into that schema first.

Where the public record runs out. Beyond GitLab, other companies sometimes cited as running dbt at scale (several fast-growing tech companies and fintechs have mentioned dbt in passing at conferences) don't currently have a public engineering writeup detailed enough to independently verify specific architecture claims — so this guide doesn't repeat unverified numbers about them. If you're evaluating how to run dbt at your own organization, GitLab's public handbook and dbt Labs' own official CI/CD guides (both cited above) are the most solid, checkable public reference points available today — treat any other "company X runs dbt like this" claim you encounter with the same skepticism until you've found its primary source.

22. dbt on Databricks: Adapter & Unity Catalog

Same models, same ref() — a different connection, and one extra level of namespace

Everything in sections 01-19 is adapter-agnostic — the same models, tests, and DAG concepts apply whether you're pointed at Snowflake, BigQuery, or Databricks. What changes is the dbt-databricks adapter's connection setup, and one structural difference that matters from day one: Databricks' Unity Catalog adds a third level of namespace that older warehouses don't have.

# ~/.dbt/profiles.yml — connecting dbt Core to Databricks
analytics:
  target: dev
  outputs:
    dev:
      type: databricks
      host: adb-1234567890123456.7.azuredatabricks.net
      http_path: /sql/1.0/warehouses/abcd1234efgh5678
      token: "{{ env_var('DBT_DATABRICKS_TOKEN') }}"
      catalog: dev_catalog
      schema: analytics_jsmith
      threads: 8

Two-level namespace vs. three-level namespace

A traditional warehouse (or the Hive metastore Databricks used before Unity Catalog) addresses a table with two parts: schema.table. Unity Catalog adds a third: catalog.schema.table. A catalog is the new top-level container — typically one per environment or business domain (dev_catalog, prod_catalog, or finance, marketing) — sitting above the schemas you already know from every other adapter. In dbt terms, what other adapters call the "database" maps directly onto a Unity Catalog catalog.

Because the catalog is a first-class part of the namespace, dbt-databricks lets you set it per-model, per-folder, or project-wide — exactly like +materialized from section 03, just one more knob:

# dbt_project.yml — different catalogs per environment, same models
models:
  analytics:
    +catalog: "{{ 'prod_catalog' if target.name == 'prod' else 'dev_catalog' }}"
    marts:
      +catalog: "{{ 'prod_catalog' if target.name == 'prod' else 'dev_catalog' }}"
      +schema: gold

Catalogs as environment boundaries. Many other adapters isolate dev vs. prod with a schema naming convention (dbt_jsmith vs analytics). On Databricks, it's just as common to isolate them with entirely separate catalogs instead — dev_catalog and prod_catalog — because Unity Catalog's access-control and lineage features are scoped at the catalog level, giving a cleaner permissions boundary than a schema-naming convention alone can.

❯ Short version

The dbt-databricks adapter connects via a SQL Warehouse's http_path plus a token (or OAuth). Unity Catalog adds a third namespace level — catalog.schema.table — where "catalog" maps directly onto what other adapters call "database." Setting +catalog per environment is a common way to isolate dev/prod with a stronger boundary than a schema-naming convention alone provides.

~/dbt ❯ Delta Lake underneath everything

23. Databricks Materializations & Delta Lake

Every table dbt builds on Databricks is a Delta table by default — three extra incremental strategies come with that

dbt-databricks materializes every table and incremental model as a Delta Lake table unless you explicitly ask for something else — the same open table format behind the medallion architecture from the streaming/Kafka side of a data platform. That gives every model ACID transactions, time travel, and schema enforcement for free, and it's also what unlocks three incremental strategies that don't exist on adapters without a transactional table format underneath.

Three Databricks-specific incremental strategies, beyond plain merge

insert_overwrite replaces entire partitions that appear in the new data, leaving every other partition untouched — fast, but only correct if your model is genuinely partitioned by the column you're filtering on. replace_where generalizes that idea: instead of replacing whole partitions, you supply an arbitrary SQL predicate, and Databricks atomically overwrites exactly the rows matching it — useful when your natural "what changed" boundary isn't a clean partition (a specific date range being backfilled, say). Plain merge (section 07) still works identically to every other adapter and remains the right default when neither of the above applies more precisely.

-- models/marts/fct_daily_events.sql — insert_overwrite, partitioned by day
{{
  config(
    materialized='incremental',
    incremental_strategy='insert_overwrite',
    partition_by='event_date'
  )
}}

SELECT event_date, event_id, user_id, event_type
FROM {{ ref('stg_events') }}
{% if is_incremental() %}
  WHERE event_date >= dateadd(day, -3, current_date())   -- reprocess the last 3 partitions
{% endif %}
-- models/marts/fct_backfilled_region.sql — replace_where, an arbitrary predicate
{{
  config(
    materialized='incremental',
    incremental_strategy='replace_where',
    incremental_predicates=["region = 'APAC'"]
  )
}}

SELECT region, event_date, event_id, revenue
FROM {{ ref('stg_events') }}
WHERE region = 'APAC'

Liquid clustering — the newer alternative to partition_by + Z-ORDER. Databricks' liquid clustering replaces the older pattern of choosing a fixed partition_by column and periodically running OPTIMIZE ... ZORDER BY. dbt-databricks exposes it as a model config, liquid_clustered_by, and unlike static partitioning it can be changed later without rewriting the whole table, and clusters on multiple columns without the cardinality pitfalls fixed partitioning has. For new models on recent Databricks runtimes, it's generally the better default over manually managed partitions.

dbt-databricks builds Delta tables by default. Liquid clustering (liquid_clustered_by) is the modern replacement for manually managed partition_by + ZORDER.

24. Python Models on Databricks

A model file that returns a DataFrame instead of running a SELECT

Every model so far has been a .sql file. On a small number of adapters — Databricks among them — dbt also supports Python models: a .py file defining a model() function that returns a DataFrame, materialized into a table exactly like a SQL model would be. It participates in the same DAG, gets the same ref()/source() treatment, and can be tested and documented identically (sections 02 and 09) — only the body swaps SQL for Python.

# models/marts/customer_churn_features.py
def model(dbt, session):
    dbt.config(materialized="table")

    orders = dbt.ref("fct_orders").toPandas() if False else dbt.ref("fct_orders")  # a Spark DataFrame

    from pyspark.sql import functions as F

    features = (
        orders.groupBy("customer_id")
        .agg(
            F.count("order_id").alias("order_count"),
            F.max("order_date").alias("last_order_date"),
            F.datediff(F.current_date(), F.max("order_date")).alias("days_since_last_order"),
        )
    )
    return features

When a Python model earns its complexity

Reach for a Python model when the logic genuinely doesn't fit SQL: calling out to an ML model for batch inference, using a PySpark-specific library (MLlib, a pandas UDF, a third-party feature-engineering package), or a transformation that's a mess of window functions in SQL but a two-line groupBy().agg() in PySpark. For anything expressible as a normal SELECT — joins, aggregations, filters — stick with SQL models; a Python model is strictly more operational overhead for no benefit if the logic doesn't need it.

Python models need a cluster, not just a SQL Warehouse. This is the detail that catches people first: a Databricks SQL Warehouse (what your normal SQL models run against) can't execute arbitrary Python/Spark code — Python models require an actual Databricks cluster (All-Purpose or Job compute) with a Python/Spark runtime. A project mixing SQL and Python models needs both a SQL Warehouse connection for the SQL models and cluster access for the Python ones, configured together in your connection profile.

❯ Short version

A Python model (.py, a model() function returning a DataFrame) participates in the DAG exactly like a SQL model — same ref(), same tests, same docs — but runs on a Databricks cluster rather than a SQL Warehouse, since SQL Warehouses can't execute arbitrary Spark/Python code. Use it only when the logic genuinely doesn't fit a SELECT.

25. Orchestrating dbt on Databricks

Workflows' native dbt task, and Asset Bundles for shipping it as code

Section 16 covered three general orchestration options: dbt Cloud's own scheduler, a general-purpose orchestrator like Airflow, or a scheduled CI job. If you're already on Databricks, there's a fourth, platform-native option worth knowing specifically: Databricks Workflows has a first-class dbt task type — a Job step that runs dbt build (or any dbt command) directly against a SQL Warehouse, with no separate orchestrator to stand up at all.

Databricks Asset Bundles (DABs) — deploying the whole thing as code

A Databricks Asset Bundle is a YAML-defined project (databricks.yml) describing everything a Databricks project needs — jobs, the dbt task's configuration, target workspace per environment — as version-controlled config, deployed with a CLI command (databricks bundle deploy) instead of clicking through the Workflows UI by hand. It plays the same "describe the infrastructure as code, not by hand" role Terraform plays for cloud infra generally, scoped specifically to Databricks jobs, clusters, and now dbt projects.

SQLMesh and Dataform, briefly — and why "dbt" is still the default answer

dbt popularized the "SQL files + ref() + a DAG + tests" model so thoroughly that most of its competitors adopt a strikingly similar shape rather than a different one — the real differences are usually in execution model and tooling philosophy, not the core idea.

Why this guide doesn't spend more space here. Unlike Kafka/Flink/Spark, where genuinely different architectural approaches (event-at-a-time vs. micro-batch) justify a deep three-way comparison, dbt's competitors mostly validate its core idea rather than replace it. The practical decision most teams actually face isn't "dbt vs. something totally different" — it's whether their warehouse, team size, and existing tooling favor dbt's much larger ecosystem and hiring pool, or a smaller tool's specific technical edge (like SQLMesh's environment diffing). For the large majority of teams, dbt's ecosystem size alone settles the question.

28. Common Pitfalls & Best Practices

The mistakes almost every dbt project makes at least once

Most dbt problems in production trace back to a small, recurring set of avoidable mistakes rather than exotic bugs.

Hardcoding a table name instead of using ref()/source(). The moment one model bypasses ref() and selects directly from a hardcoded schema.table path, the DAG (section 02) built from that point becomes incomplete — dbt no longer knows about that dependency, so it can build things in the wrong order, and state:modified+ (section 17) can miss that this model needed rebuilding at all.

Marts referencing marts referencing marts. When final consumption-layer models start depending on each other instead of on intermediate models, the layered architecture from section 05 collapses back into an untangleable web — the entire point of the staging/intermediate/marts split is to keep dependencies flowing in one direction.

Leaving on_schema_change at its default on an important incremental model. As section 07 covers, the default can silently misalign or drop data when an upstream schema changes — decide this setting deliberately per model instead of accepting whatever dbt assumes.

No tests on anything that isn't a primary key. unique and not_null on ID columns is the bare minimum, not the finish line. A test suite that never checks a business rule (order totals reconciling, no negative revenue, accepted values on a status column) only ever catches the easiest, least valuable class of bugs.

Treating dbt run as the production command. As section 16 covers, dbt build interleaves tests with model builds and stops a broken branch of the DAG before it propagates; a schedule that only runs dbt run (skipping tests, or running them as an afterthought) can let bad data reach a mart and a dashboard before anyone notices.

No Slim CI, on a large project. Once a project has more than a couple hundred models, rebuilding everything on every PR (section 17) turns CI into the slowest, most expensive part of the whole development loop — and, ironically, makes people avoid running CI carefully, which defeats its purpose.

❯ Short version

Almost every recurring dbt problem is one of: bypassing ref()/source() and breaking the DAG, letting marts depend on marts instead of respecting the layered architecture, an unconsidered on_schema_change default, tests that only ever check primary keys, running dbt run instead of dbt build in production, or skipping Slim CI once a project has grown past a couple hundred models.

30. Learning Path & Resources

A concrete, hands-on order to actually learn this by doing it

Reading about dbt gets you only partway — models, tests, and the DAG click into place once you've watched them behave on a real (even tiny) warehouse. Here is a practical, ordered path.

• 1. Get any warehouse connection working. A free-tier Snowflake trial, BigQuery's sandbox, or even local DuckDB via the community adapter — the specific warehouse matters far less than just having a working profiles.yml connection.

• 2. Run dbt init, then build one staging model on top of one raw table, using source() (section 04) — get the smallest possible end-to-end loop working before adding complexity.

• 3. Add one dimension/fact pair on top of it (section 05), and add generic tests (not_null, unique, relationships) to both (section 09).

• 4. Convert your fact model to incremental (section 07), then intentionally run a full refresh and a normal incremental run back to back, and look at the compiled SQL for each in target/compiled/ (section 02) — seeing the actual difference matters more than reading about it.

• 5. Write one custom macro and one singular test (sections 08-09) — even something trivial — to get comfortable with Jinja before you need it for something that matters.

• 6. Run dbt docs generate and click through your own lineage graph (section 11) — seeing your project's DAG rendered back at you is usually the point everything from section 02 clicks together.

• 7. Set up Slim CI once you have more than a handful of models (section 17), even on a personal project — the state:modified+/--defer mechanics are much easier to internalize with a small, low-stakes project than as your first exposure on a real team's repo.

• 8. Once you're running this in production, read dbt Labs' own current docs. Materialization defaults, new features, and the Fusion engine's status (section 19) all move fast — docs.getdbt.com beats any guide, including this one, at staying current.

Primary sources this guide leans on for currency. docs.getdbt.com — official reference docs, always the first place to check exact current syntax and defaults. getdbt.com/blog — official announcements, including Fusion engine updates and licensing clarifications. handbook.gitlab.com (Enterprise Data Platform section) — a rare, genuinely detailed public account of running dbt Core in production. The MovieLens/Snowflake tutorial project referenced in section 20 — a compact, complete, real dbt project worth cloning and running yourself rather than only reading about.

❯ Short version

Get one warehouse connection working → one staging model → a dimension/fact pair with tests → convert one model to incremental and compare compiled SQL → write a macro and a singular test → generate and click through your own docs/lineage graph → set up Slim CI → then live in dbt Labs' own current docs once you're in production, since defaults and new features move fast.

∎ The takeaway

dbt's core trick is small and easy to underestimate: turn your SQL transformations into a real, version-controlled project with dependencies, tests, and documentation, and let the warehouse keep doing what it already does well. Everything else in this guide — incremental strategies, Slim CI, contracts, Mesh, the Semantic Layer, the Fusion engine — is dbt Labs and the wider ecosystem building more infrastructure around that one small trick as it scales from one analyst's laptop to a company the size of GitLab.

You do not need to memorize everything in this guide. Come back to it when you hit a real problem: an incremental model that silently full-refreshes in CI, a contract failing on a column you didn't think you'd touched, a DAG that's grown too tangled to reason about. Re-read the relevant section, look at the diagram, and it should click faster the second time.
