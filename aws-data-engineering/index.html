01. Why AWS for Data Engineering

One cloud, a dozen-plus services, and one question that decides which ones you need

AWS didn't design one "data engineering product" — it grew a family of separate services (storage, compute, warehousing, streaming, orchestration) that happen to compose well together, because most of them were built to read and write the same underlying storage layer: Amazon S3. That's the single fact that makes the rest of AWS data engineering learnable instead of overwhelming: almost every service in this guide either writes data into S3, reads data out of S3, or helps you query/govern/move data that's sitting in S3.

The one question that decides your service list

Before reaching for any AWS service, ask: is this data moving (streaming) or sitting still (batch), and does it need to be queried with SQL, transformed with code, or just stored? Batch storage and transformation points toward S3 + Glue. SQL analytics points toward Athena or Redshift. Continuously arriving data points toward Kinesis or MSK. Tying steps together on a schedule or in response to an event points toward Step Functions, Lambda, and EventBridge. This guide is organized in exactly that order — storage first, then compute/ETL, then query, then streaming, then orchestration — because that's the order real architecture decisions actually get made in.

❯ Short version

AWS data engineering isn't one product — it's a family of services that compose around a shared storage layer, Amazon S3. Nearly everything else in this guide either writes to S3, reads from S3, or helps you query/govern/move data that lives there. Picking the right services starts with one question: is the data moving or sitting still, and what do you need to do with it once it lands?

02. AWS Fundamentals for Data Engineers

The baseline every other section in this guide quietly assumes you already have

"The cloud" is not a technology, it's a billing and provisioning model: instead of buying servers, racking them, and running them at whatever utilization you happen to have, you rent compute, storage, and managed services from AWS by the second or by the gigabyte, and you can request more (or release what you have) in minutes instead of months. That single shift — from owned, fixed capacity to rented, elastic capacity — is why almost everything in this guide is "provision nothing, pay for what runs."

Regions and Availability Zones — and why "regional" vs "global" matters

An AWS Region is an independent geographic area (e.g. us-east-1, ap-southeast-1) built from multiple isolated data centers called Availability Zones (AZs) — physically separate enough that one AZ failing doesn't take the others with it, but connected by low-latency links so services can replicate across them. For a data engineer this is concrete, not academic: an S3 bucket, a Redshift cluster, and a Glue job all live in exactly one Region — data residency and compliance requirements (e.g. keeping customer data inside a specific country) are enforced at the Region level, and moving data between Regions costs money and adds latency. IAM (covered in its own section) is the deliberate exception: it has no Region at all — one set of users, roles, and policies applies everywhere the account operates.

# The Region segment in an ARN shows which resources are Region-scoped -- and which aren't
arn:aws:redshift:us-east-1:111122223333:cluster:sales-cluster   # regional resource
arn:aws:s3:::yt-pipeline-silver                                # S3: no Region segment
arn:aws:iam::111122223333:role/GlueETLRole                      # IAM: no Region -- it's global

The Shared Responsibility Model: "of the cloud" vs "in the cloud"

AWS secures everything of the cloud — the physical data centers, the hardware, the hypervisor, and the underlying durability of a service like S3 (a standard, well-known claim: 11 nines, achieved by replicating objects across multiple devices and AZs). The customer is responsible for everything in the cloud — how they configure it. Concretely: if S3 loses an object because of an AWS-side hardware failure, that's AWS's problem, and it essentially never happens. If an S3 bucket is publicly readable because someone left Block Public Access disabled, that's entirely the customer's misconfiguration — AWS built the switch, but flipping it was your decision.

Exam angle
The Shared Responsibility Model shows up as a "whose fault is it" question. A scenario describes a data leak caused by an open S3 bucket or an overly permissive IAM policy and asks who was responsible. The pattern is consistent: durability, hardware failure, and physical/network infrastructure are AWS's job; anything you configured — bucket policies, encryption settings, network access, IAM permissions — is yours, even though it happened inside "AWS's service."

EC2 underneath: why a data engineer should still recognize instance types

Amazon EC2 rents virtual machines by the family, sized for a workload shape: general purpose (m family, balanced CPU/memory), compute-optimized (c family, CPU-heavy transforms), and memory-optimized (r family, large in-memory joins/aggregations). Glue, Lambda, and Athena hide this completely — you never pick an instance type. Amazon EMR (covered in its own section) does not: an EMR cluster is a managed fleet of EC2 instances running Spark/Hadoop, and choosing an EMR node type is choosing an EC2 instance type under a different label.

One AWS account rarely means one AWS account

Every AWS account has a single, all-powerful root user (created with the account, should have MFA enabled, and almost never used day-to-day) plus any number of IAM identities (covered in its own section) for everything else. Real organizations essentially never run everything in one account: AWS Organizations lets a company manage many accounts — separate dev, staging, and prod accounts, or one account per team — under a single consolidated bill and one set of guardrails, so a mistake or a leaked credential in a dev account can't reach production data.

Picking the wrong Region costs money you won't notice until the bill arrives. Data transferred between Regions (e.g. an EC2 instance in us-east-1 reading from a bucket in ap-southeast-1) is billed as inter-Region data transfer, on top of the added latency — an easy mistake when a team defaults to whichever Region they created their first bucket in, without checking where the rest of the pipeline actually runs.

Almost every service in this guide — S3, Glue, Athena, Lambda — bills on consumption: per GB stored, per DPU-hour, per GB scanned, per millisecond executed. There's no fixed server sitting underneath most of the pipeline racking up cost while idle. That's not incidental — it's why AWS's default recommendation for new data engineering workloads is serverless-first: you pay when a pipeline runs, and nights, weekends, and quiet periods with no incoming data cost nothing.

❯ Short version

The cloud is rented, on-demand infrastructure billed by consumption rather than owned. Resources like S3 and Redshift are Regional (with AZs inside them for resilience); IAM is global. AWS secures the cloud's infrastructure ("of the cloud"); you secure how you configure it ("in the cloud"). EC2 quietly underlies EMR, and the six Well-Architected pillars are worth naming even in a pure-data context.

03. AWS Networking for Data Engineers

VPCs, endpoints, and the difference between a free path to S3 and a NAT-Gateway-billed one

Every other section in this guide describes a service that looks "serverless," which tempts you into assuming there's no network to think about. There is: a Glue job with a VPC connection, a Lambda function attached to a VPC, an EMR cluster, and an Amazon Redshift cluster all run inside a private network you're responsible for wiring correctly. Get the wiring wrong and the job either fails outright or quietly routes traffic — and cost — through the public internet when it didn't need to.

Amazon VPC: your own private slice of the AWS network

A VPC (Virtual Private Cloud) is an isolated network inside one Region, defined by a CIDR block (e.g. 10.0.0.0/16) that sets its total address range. A VPC is divided into subnets, each pinned to one AZ, and each subnet is either public (has a route to an Internet Gateway, so resources in it can hold a public IP) or private (no such route). Pipeline compute — a Glue job's VPC connection, an EMR cluster, a Lambda function attached to a VPC, a Redshift cluster — is placed in private subnets almost by default: none of it needs to accept inbound connections from the public internet, and keeping it unreachable from outside is a security requirement in most environments before it's ever a "nice to have."

Internet Gateway vs NAT Gateway — inbound-capable vs outbound-only

An Internet Gateway (IGW) is what makes a subnet "public" — a two-way door: resources with a public IP can be reached from the internet and can reach out to it. A NAT Gateway sits in a public subnet and gives resources in a private subnet a one-way door out: a Lambda function in a private subnet calling an external weather API goes out through the NAT Gateway, which translates its private IP to the NAT Gateway's public IP — but nothing on the internet can open a connection back in. The private resource stays unreachable from outside while remaining able to reach outside.

Exam angle
"A Lambda in a private subnet needs to read from S3 — what's the most cost-effective, most secure way?" is a recurring question shape. The trap answer is "add a NAT Gateway" — it works, but it charges an hourly fee plus per-GB data processing. The correct answer for S3 or DynamoDB specifically is a Gateway VPC Endpoint: free, added as a route-table entry, and the traffic never leaves the AWS network at all.

The Interface Endpoint mechanism has an official name worth remembering for the exam: AWS PrivateLink — it's the underlying technology behind every Interface Endpoint, and it also lets a VPC privately reach a service exposed by a different AWS account or a third-party SaaS product, not only AWS's own services.

// Gateway VPC Endpoint policy: restrict it to exactly one bucket, not "any S3 in the account"
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": "*",
    "Action": ["s3:GetObject", "s3:PutObject"],
    "Resource": "arn:aws:s3:::yt-pipeline-silver/*"
  }]
}

A Gateway Endpoint only helps the subnets whose route table you added it to. Adding an S3 Gateway Endpoint doesn't automatically cover every subnet in the VPC — forgetting to associate it with a private subnet's route table is a common reason a Glue job "can't reach S3" even though the endpoint exists elsewhere in the same VPC.

Route 53 and CloudFront — light-touch for most data pipelines

Amazon Route 53 is AWS's DNS service; its main relevance to a data platform is the private hosted zone, which resolves internal names (e.g. metastore.internal) only inside a VPC — useful for service discovery between pipeline components without hardcoding IPs. Amazon CloudFront is a CDN that caches content at edge locations close to users; a data engineer runs into it mainly when caching an embedded Amazon QuickSight dashboard or an API Gateway-fronted data API at the edge, so repeat requests for the same dashboard don't hit the underlying compute every time.

Connecting many VPCs, and connecting to on-premises

AWS Transit Gateway is a hub-and-spoke router for many VPCs (and accounts) that would otherwise need a tangle of point-to-point peering connections — attach each VPC once to the Transit Gateway instead of pairing every VPC with every other VPC directly. AWS Direct Connect is a dedicated, private network link from an on-premises data center straight into AWS, bypassing the public internet entirely; it's the connection of choice when migrating a large on-premises database or data warehouse into AWS via AWS DMS (covered in its own section), where a stable, high-bandwidth, non-internet link matters more than the extra setup cost.

❯ Short version

Pipeline compute (Glue VPC connections, EMR, Lambda-in-VPC, Redshift) lives in private subnets; a NAT Gateway gives it one-way outbound internet access, while a free Gateway VPC Endpoint gives it a direct, private, no-cost path to S3/DynamoDB that never touches the internet or the NAT Gateway. Security Groups are stateful and resource-level; NACLs are stateless and subnet-level, mostly reserved for explicit deny rules. Transit Gateway connects many VPCs; Direct Connect connects on-premises to AWS over a dedicated line.

04. Amazon S3: The Foundation of Every AWS Data Platform

Object storage, storage classes, and the partitioning scheme that decides how everything downstream performs

Amazon S3 (Simple Storage Service) stores objects — arbitrary blobs of data, from a 1KB JSON file to a multi-terabyte export — inside buckets, addressed by a key (the full "path" string, even though S3 has no real folders — the slashes in a key are just characters that tools render as if they were a folder tree). Every object also carries a storage class, which is really a cost/retrieval-speed tradeoff dial, not a different product.

Storage classRetrievalTypical use in a data pipeline
S3 StandardMillisecondsBronze/silver/gold data actively being read by ETL jobs and queries
S3 Intelligent-TieringMilliseconds (auto-moves objects)Default choice when access patterns are unpredictable — no lifecycle rules to hand-tune
S3 Standard-IA / One Zone-IAMilliseconds, cheaper storage, pricier retrievalOlder partitions still queried occasionally (last quarter's data, not last year's)
S3 Glacier Instant/Flexible/Deep ArchiveMilliseconds (Instant); Flexible Retrieval spans minutes (Expedited) to ~12 hours (Bulk); Deep Archive's Bulk tier can take up to 48 hoursCompliance retention, rarely-touched historical archives

Partitioning — the single biggest lever on query cost and speed

A partition is a folder-like grouping baked into the S3 key itself, almost always by date and sometimes by another high-cardinality-but-not-too-high dimension: s3://bucket/table/region=US/date=2026-04-01/hour=12/file.parquet. Athena, Redshift Spectrum, and Glue can all use partition values in a query's WHERE clause to skip scanning entire folders of irrelevant data — a query for one day's data against a partitioned table might read one 200MB folder instead of a 2TB table. Get partitioning wrong (too fine, too coarse, or missing entirely) and every query downstream pays for it, every single time it runs.

The classic mistake: partitioning by a column nobody filters on. Partitioning by customer_id when every real query filters by date gives you thousands of tiny partitions and zero query speedup — the query planner still has to check every partition's date range. Partition by the column(s) your queries' WHERE clauses actually use, almost always date first, and don't add a second partition column unless a real query pattern needs it.

❯ Short version

S3 stores objects in buckets, addressed by a key; storage classes trade cost against retrieval speed. Partitioning — encoding columns like date/region into the S3 key path — lets query engines skip entire folders of irrelevant data instead of scanning everything, and is the single biggest lever on both query cost and speed. Partition by what your queries actually filter on, not by whatever column feels most "unique."

05. IAM for Data Engineers

The permission model every other service in this guide sits on top of

IAM (Identity and Access Management) isn't a data service, but every single other service in this guide checks with it before doing anything — a Glue job can't read from S3, and a Lambda function can't write to a Glue table, without an IAM identity that explicitly allows it. Understanding IAM's two building blocks up front saves hours of "access denied" debugging later.

IAM roles vs. IAM users, and why data pipelines almost never use a "user"

An IAM user represents a person, with long-lived credentials. An IAM role has no long-lived credentials at all — a service (Lambda, Glue, Step Functions) assumes the role temporarily, gets short-lived credentials scoped to exactly that role's permissions, and those credentials expire automatically. Every service-to-service permission in a real pipeline should be a role, not a user with an access key hardcoded somewhere — the same "least privilege, no long-lived secrets lying around" instinct that shows up everywhere in cloud security.

// A trust policy: WHO is allowed to assume this role (attached to the role itself)
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {"Service": "glue.amazonaws.com"},
    "Action": "sts:AssumeRole"
  }]
}

// A permissions policy: WHAT the role can do once assumed (attached separately)
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["s3:GetObject", "s3:PutObject"],
    "Resource": "arn:aws:s3:::yt-data-pipeline-silver-*/*"
  }]
}

Two policies, two different jobs — a common source of confusion. The trust policy answers "who can become this role" (in the example above: the Glue service itself). The permissions policy answers "what can this role do once it's been assumed" (read/write specific S3 paths). A role with a perfect permissions policy but the wrong trust policy simply can't be assumed by the service that needs it — and the error message for that ("not authorized to perform sts:AssumeRole") looks nothing like a normal permissions error, which is exactly why it trips people up.

❯ Short version

Service-to-service access in a pipeline should always be an IAM role, assumed temporarily, never a user with a long-lived key. A role has two separate policies: a trust policy (who/what can assume it) and a permissions policy (what it can do once assumed) — mixing these two up is the most common source of confusing IAM errors.

06. AWS Security & Encryption for Data Platforms: KMS, Secrets Manager & Macie

Encryption at rest and in transit, credential management, and finding PII nobody knew was there

IAM (see the IAM section) answers "who can call which API." Lake Formation (see the Lake Formation section) answers "which columns and rows within a table." This section covers a third, easily-skipped layer: how data is actually protected once it's sitting on disk or moving over a wire, how a job gets credentials without anyone typing a password into a script, and how you'd ever find out a data lake already contains raw PII that nobody manually reviewed.

Customer managed vs. AWS managed vs. AWS owned keys — who's actually in control

An AWS owned key is the default, invisible encryption many services apply automatically (plain SSE-S3 on an S3 object, for instance) — you never see the key, there's no key policy to manage, and no per-object entry in CloudTrail. An AWS managed key (named aws/s3, aws/redshift, etc.) is created and rotated by AWS inside your account when you turn on default SSE-KMS without specifying your own key — visible in the console, its usage logged, but you can't edit its key policy for fine-grained control. A customer managed key (CMK) is one you create: you own the key policy, choose whether to enable annual automatic rotation, and can grant or deny kms:Decrypt to specific principals. The practical trigger for reaching for a CMK is simple: the moment you need to control who can decrypt this as a decision separate from who has S3 permissions, or you need the ability to disable/schedule deletion of a key for compliance reasons, AWS managed or owned keys can't give you that.

 SSE-S3SSE-KMS
Who manages the keyAWS, entirely — invisible to youYou (customer managed key) or AWS (aws/s3 managed key)
Audit trailNone — no per-object decrypt event anywhereEvery Encrypt/Decrypt/GenerateDataKey call logged to CloudTrail, including caller identity
CostFreeSmall per-request charge per KMS API call, plus ~$1/month per customer managed key
When to chooseDefault for most objects — no regulatory need to prove who decrypted whatAn audit trail is required, decrypt access must be controlled separately from S3 permissions, or a key (not the data itself) needs to be revocable

A third option, SSE-C, has AWS encrypt/decrypt with a key you supply on every single request and never store — rarely used in a pipeline context because Athena, Glue, and Redshift Spectrum have no way to attach your key to every call they make on your behalf. For data in transit, S3 and every AWS API endpoint enforce TLS by default; the pipeline-relevant control is a bucket policy statement that denies any request where aws:SecureTransport is false, closing off plain-HTTP access entirely rather than just discouraging it.

Exam angle
Two independent policies, both must say yes. A question describing an Access Denied on s3:GetObject where the IAM policy and the bucket policy both clearly allow it is almost always pointing at a missing kms:Decrypt grant in the KMS key's own key policy. S3 permissions can never, by themselves, grant access to a customer managed key you don't separately have kms:Decrypt on — re-checking or widening the IAM policy again fixes nothing; the key policy has to name that principal explicitly.

Secrets Manager — the anti-pattern it exists to eliminate

AWS Secrets Manager stores credentials — database passwords, API keys, OAuth tokens — as versioned secret values, retrievable at runtime by any principal with secretsmanager:GetSecretValue on that secret, and can automatically rotate native-integrated secrets (RDS, Redshift, DocumentDB) on a schedule via a managed rotation Lambda. The anti-pattern it replaces: a database password typed directly into a Glue job script argument or a Lambda environment variable. That password now lives in version control history, in the Glue console's job parameters visible to anyone with read access to the job, and in CloudTrail's StartJobRun request parameters — and it never rotates, so a single leak stays valid indefinitely until someone notices by accident.

# Anti-pattern: password embedded directly in the job -- visible in the script,
# in git history, and to anyone with read access to the job definition
DB_PASSWORD = "Sup3rSecret!"   # never do this

# Fix: fetch the secret at runtime; access is controlled purely by the job's IAM role
import boto3
import json

def get_db_credentials(secret_id):
    client = boto3.client("secretsmanager", region_name="ap-southeast-1")
    resp = client.get_secret_value(SecretId=secret_id)
    return json.loads(resp["SecretString"])   # {"username": "...", "password": "..."}

creds = get_db_credentials("prod/redshift/etl-user")
conn = psycopg2.connect(host=DB_HOST, user=creds["username"], password=creds["password"])

Secrets Manager vs. Systems Manager Parameter Store — a real "which one" distinction

Parameter Store (standard tier) is free and stores arbitrary configuration strings — a table name, a feature flag, an S3 prefix, even a SecureString encrypted with KMS — but has no built-in rotation. Secrets Manager charges per secret per month plus per API call, and its whole reason to exist is native automatic rotation for things that must periodically change. The rule of thumb: if it's a credential that should rotate, use Secrets Manager; if it's a config value that doesn't need rotation and isn't worth the extra cost, Parameter Store is the cheaper, simpler fit (see also the Governance section, which cross-references Parameter Store for operational use).

Amazon Macie — finding PII nobody knew was there

Amazon Macie runs machine-learning-based and pattern-based detection (its "managed data identifiers") over S3 objects to flag sensitive data — national ID numbers, card numbers, financial account numbers, credentials embedded in a file — without anyone having to manually open every file that ever landed in a data lake. This matters precisely because a lake ingests from many upstream sources, and nobody reviews every column of every file by hand before it arrives. A Macie job produces findings (severity-rated, per-bucket or per-object) that typically feed an EventBridge rule for alerting, or inform a concrete governance action — for example, adding a column-level policy in Lake Formation (see the Lake Formation section) to hide the column Macie just flagged from everyone except a compliance role.

Macie finds; it doesn't fix. Macie only scans the buckets and accounts you scope a job to, must be run (one-time or on a schedule) to catch newly arrived objects, and a finding is a starting point, not a resolution — nothing about it automatically encrypts, masks, restricts, or deletes the data. Someone (or some automated workflow triggered off the finding) still has to act on it.

AWS Shield and AWS WAF — perimeter defense, not data-layer defense

These two round out the "Security, Identity, and Compliance" exam category and matter mainly for anything in a data platform that's actually internet-facing — a QuickSight-embedded dashboard, an API Gateway endpoint serving gold-layer data, or a public-facing ingestion API. AWS Shield (Standard is automatic and free on every AWS account; Advanced is a paid tier) protects against DDoS attacks at the network and transport layer. AWS WAF is a web application firewall that inspects HTTP(S) requests reaching CloudFront, API Gateway, or an Application Load Balancer, and blocks requests matching rules for common exploits (SQL injection, cross-site scripting) or simple rate-based abuse. Neither one encrypts data or controls who can query a table — that's KMS, IAM, and Lake Formation's job — Shield and WAF exist purely to stop malicious traffic from reaching the endpoint in the first place.

❯ Short version

KMS has three key tiers — AWS owned (invisible), AWS managed (visible, not editable), and customer managed (full control) — and an object encrypted with SSE-KMS under a customer managed key needs its role granted kms:Decrypt in the key policy, completely separate from S3 permissions; missing that grant is a very common real "access denied" bug. Secrets Manager retrieves rotating credentials at runtime instead of hardcoding them; Parameter Store is the cheaper choice for non-rotating config. Macie automatically discovers PII sitting in S3 that nobody manually reviewed, feeding findings into alerting or a Lake Formation policy fix.

07. AWS Lambda for Data Engineering

The right tool for short, event-driven work — and the wrong one for heavy transforms

AWS Lambda runs a function in response to a trigger — an S3 upload, an API call, a Step Functions task, a schedule — without you provisioning or managing any server. You pay per invocation and per millisecond of execution, and it scales from zero to thousands of concurrent invocations automatically. In a data pipeline, Lambda is almost always doing one of three jobs: pulling data from an external API, doing a small, fast transformation, or running a quick validation check.

Lambda's real limits — and why they matter for data work specifically

A single Lambda invocation has a hard 15-minute maximum runtime, up to 10GB of memory, and, on deployment package size, two separate limits depending on how the code is packaged: a zip-based deployment is capped at roughly 250MB unzipped (code plus layers), with a smaller 50MB limit if uploading the zip directly through the console rather than via S3; a container image deployment can go up to 10GB instead. None of that is a problem for "call an API and write JSON to S3" — it's a real problem for "run a multi-gigabyte Spark join," which is exactly why heavy transforms belong in Glue or EMR (sections 09, 11), not Lambda. Reaching for Lambda for genuinely large-data transforms is one of the most common early-career AWS data engineering mistakes.

# A minimal ingestion Lambda -- the shape almost every "pull from an API, land in S3" job takes
import json, boto3, urllib.request
from datetime import datetime

s3 = boto3.client("s3")

def lambda_handler(event, context):
    response = urllib.request.urlopen("https://api.example.com/trending?region=US")
    data = json.loads(response.read())

    key = f"raw_statistics/region=US/date={now:%Y-%m-%d}/hour={now:%H}/data.json"
    s3.put_object(
        Bucket="yt-data-pipeline-bronze-ap-south-1-dev",
        Key=key,
        Body=json.dumps(data),
    )
    return {"status": "success", "records": len(data), "s3_key": key}

08. AWS Glue Data Catalog & Crawlers

S3 has no schema of its own — the Catalog is what gives your files a table structure SQL engines can use

S3 stores files; it has no idea those files are "a table." The Glue Data Catalog is a persistent, central metadata store — for every table, it holds the schema (column names/types), the S3 location, the file format, and the partition scheme. It is not a AWS-Glue-only detail: Athena, Redshift Spectrum, EMR, and Glue ETL jobs all read the same Catalog, which is exactly why defining a table's metadata once, in one place, makes it usable everywhere.

Crawlers — the lazy way to populate the Catalog

A Crawler connects to a data source (an S3 prefix, a JDBC database), inspects a sample of files, infers the schema and partition structure, and writes (or updates) a table definition in the Catalog automatically — no CREATE TABLE statement required. Crawlers are convenient for getting started and for sources whose schema genuinely drifts over time, but they cost real money per run and can occasionally infer a schema you wouldn't have chosen by hand (a column that's usually an integer but has one string value in a rare file becomes a string column for the whole table). Many production teams define Catalog tables explicitly via infrastructure-as-code instead, and reserve Crawlers for genuinely-evolving or exploratory sources.

❯ Short version

The Glue Data Catalog is the shared metadata store (schema, S3 location, format, partitions) that Athena, Redshift Spectrum, EMR, and Glue ETL all read from — define a table once, use it everywhere. Crawlers auto-populate the Catalog by inspecting files, convenient but not free and not always precisely the schema you'd choose by hand; many production tables are instead defined explicitly as code.

09. AWS Glue ETL Jobs

Managed Spark, without managing a cluster

A Glue ETL job is a managed Apache Spark (or plain Python) script AWS runs on infrastructure you never provision or patch — you write the transformation logic, pick how much compute to give it, and Glue handles cluster startup, scaling within the job, and teardown. This is the workhorse for medallion-style bronze→silver→gold transforms in an AWS-native data lake.

DynamicFrame vs. DataFrame — Glue's one real API quirk

Glue's Spark jobs can use a normal Spark DataFrame, but Glue also offers its own DynamicFrame: almost the same idea, but more forgiving about schema inconsistencies across files (a column that's sometimes missing, sometimes a different type) and with built-in helpers for reading straight from a Catalog table. Most real jobs read as a DynamicFrame, convert to a DataFrame immediately for the actual transformation logic (where the much larger Spark ecosystem's functions are available), and convert back only if writing via a Glue-specific sink.

# glue_jobs/bronze_to_silver_statistics.py -- the real shape of a medallion transform
import sys
from awsglue.transforms import *
from awsglue.utils import getResolvedOptions
from awsglue.context import GlueContext
from pyspark.context import SparkContext
from pyspark.sql import functions as F

args = getResolvedOptions(sys.argv, ["bronze_database", "bronze_table", "silver_bucket", "silver_database"])
glueContext = GlueContext(SparkContext.getOrCreate())

bronze_dyf = glueContext.create_dynamic_frame.from_catalog(
    database=args["bronze_database"], table_name=args["bronze_table"]
)
df = bronze_dyf.toDF()

silver_df = (
    df.withColumn("views", F.col("views").cast("long"))
      .withColumn("likes", F.col("likes").cast("long"))
      .withColumn("like_ratio", F.col("likes") / F.col("views"))
      .dropDuplicates(["video_id", "region", "trending_date"])
)

silver_df.write.mode("overwrite").partitionBy("region").parquet(
    f"s3://{args['silver_bucket']}/clean_statistics/"
)

10. AWS Glue DataBrew & Glue Data Quality

A visual data-prep tool for analysts, and a rules engine for enforcing quality automatically

Two Glue features solve two different problems that often get confused. Glue DataBrew is a visual, no-code tool for exploring and cleaning data — point it at a dataset, and it profiles columns, suggests transformations (trim whitespace, fix types, handle nulls), and lets an analyst build a cleaning recipe by clicking, not coding. Glue Data Quality is a rules engine: you declare expectations about a dataset (row counts, null thresholds, value ranges, freshness), and it evaluates them automatically as part of a pipeline, the same job the DQ Lambda plays in the worked example this guide references throughout (section 37).

# A Glue Data Quality ruleset -- DQDL (Data Quality Definition Language)
Rules = [
    RowCount > 10,
    IsComplete "video_id",
    ColumnValues "views" > 0,
    ColumnValues "region" in ["US", "GB", "CA", "DE", "FR", "IN", "JP", "KR", "MX", "RU"],
    Completeness "views" > 0.95

Where this fits relative to a hand-written DQ Lambda

The worked example in this guide's later sections uses a custom Lambda function for data quality checks — a perfectly valid, common approach, especially when checks need custom logic (freshness relative to a specific business rule, cross-table consistency). Glue Data Quality is worth knowing as the managed alternative: less custom code, a declarative rule language, and quality metrics tracked over time in the Glue console automatically — a good default to reach for first, reserving a custom Lambda for checks Glue Data Quality's rule language genuinely can't express.

Glue DataBrew is a no-code visual tool for profiling and cleaning data, aimed at analysts rather than pipeline code. 12. AWS Batch & Containers for Data Workloads

Long-running or massively parallel containerized jobs, and the container building blocks underneath them

AWS Batch is fully managed batch job scheduling and execution: you submit a job — a containerized command — to a job queue, and Batch provisions and scales the underlying compute (EC2 or Fargate) to match queue depth, runs the job, and tears the capacity back down when there's nothing left to run. There's no cluster to keep warm and no server to patch; you think in terms of jobs and queues, not instances.

Batch vs. Lambda vs. Glue — three different "run my code" shapes

Against Lambda (see the Lambda section): Batch has no 15-minute runtime ceiling, so it fits long-running jobs (hours) or bursty, large-scale parallel workloads (thousands of array-job tasks) that would simply time out in Lambda. Against Glue (see the Glue ETL section): a Batch job runs an arbitrary container image and command — any language, any binary, any custom tool — not specifically a Spark job. Reach for Batch when a workload is a custom script or third-party tool that doesn't fit Glue's Spark, Python-shell, or Ray job types at all, rather than trying to force it into a Glue job shape it wasn't built for.

The container building blocks underneath (lighter-tested here than on a DevOps-focused exam, but worth recognizing)

Amazon ECR (Elastic Container Registry) is the private registry where you push the Docker image a Batch job, or a container-image Lambda function, actually runs. Amazon ECS (Elastic Container Service) and Amazon EKS (Elastic Kubernetes Service) are container orchestrators — ECS is AWS's own simpler scheduler for running and scaling containers, EKS is managed Kubernetes for teams that need Kubernetes's ecosystem and portability. A long-running custom streaming consumer, or a lightweight microservice feeding a pipeline, is sometimes run on ECS/EKS/Fargate directly rather than as a Lambda, precisely because it needs to run continuously rather than being invoked per event.

13. Amazon Athena

Serverless SQL over the Glue Catalog — no cluster, no loading, pay per query

Amazon Athena runs standard SQL (built on Trino/Presto) directly against files sitting in S3, using the Glue Data Catalog for schema and location — there's nothing to provision, nothing to load, and no cluster sitting idle between queries. You pay per query, priced by the amount of data actually scanned, which is exactly why storage format and partitioning (section 04) directly control your Athena bill, not just your query latency.

Two levers that change an Athena bill by 10-100x on the same data

Columnar formats (Parquet, ORC) let Athena read only the columns a query actually selects, instead of every column in every row — a SELECT views FROM table against Parquet reads a fraction of what the same query reads against row-oriented CSV/JSON. Partition pruning (section 04) lets Athena skip entire S3 prefixes a query's WHERE clause can't match. Together, these two are why the same logical query can cost cents against a well-organized table and dollars against a poorly organized one holding identical data.

-- Athena reads the Glue Catalog for schema + partition locations -- no data movement
CREATE EXTERNAL TABLE trending_gold (
  video_id string, title string, views bigint, likes bigint, region string
)
PARTITIONED BY (trending_date string)
STORED AS PARQUET
LOCATION 's3://yt-data-pipeline-gold-ap-south-1-prod/trending_analytics/';

-- Only the matching partition + selected columns are scanned and billed
SELECT video_id, views FROM trending_gold
WHERE trending_date = '2026-04-01' AND region = 'US';

Athena vs. a data warehouse — not a replacement, a different entry point. Athena doesn't load or store data itself; it's a query engine on top of files you already have in S3. That makes it excellent for ad hoc analysis, one-off investigations, and BI tools that can point straight at S3 — but a warehouse like Redshift (section 14) still wins for workloads with heavy, repeated joins/aggregations at scale, because a warehouse's storage engine is built and tuned specifically for that, where Athena is bounded by whatever format and layout the underlying S3 files happen to have.

❯ Short version

Athena runs serverless SQL directly against S3 via the Glue Catalog, billed by data scanned. Columnar formats (Parquet/ORC) and partition pruning are the two levers that most directly control cost. It's the natural first tool for ad hoc SQL and BI-over-S3 — reach for Redshift instead once workloads need heavy, repeated, large-scale joins a warehouse engine is built for.

Modern Redshift is usually Redshift Serverless, which bills by compute actually used instead of requiring you to size and pay for a fixed cluster around the clock.

Redshift Spectrum lets a Redshift query reach outside the warehouse and read data straight from S3 (via the Glue Catalog, same as Athena) in the same query as data already loaded into Redshift tables. This is the practical bridge between "data lake" and "data warehouse": keep cold or rarely-joined data in S3 as Parquet, keep hot, frequently-joined data loaded natively in Redshift, and let one Spectrum query span both without a separate ETL step to merge them.

15. AWS Lake Formation

Fine-grained governance on top of the Glue Catalog — column, row, and cell-level access control

IAM (section 05) controls access at the level of an S3 bucket, a Glue table, or an API call — it has no concept of "this analyst can see 8 of a table's 12 columns" or "this role can only see rows where region='US'." AWS Lake Formation sits on top of the Glue Catalog specifically to add that finer grain: column-level, row-level, and (via data filters) cell-level permissions, defined once centrally and enforced consistently across Athena, Redshift Spectrum, EMR, and QuickSight — instead of every consuming service needing its own separate access-control mechanism.

Why this matters specifically for a regulated data platform

A common real requirement: an analytics team should see anonymized/aggregated customer metrics, but never raw PII columns like national ID or account number, even though both live in the same underlying table. Doing this with IAM alone means either duplicating the table (a masked copy for analysts, a full copy for compliance — extra storage, extra pipelines, extra drift risk) or writing custom application-layer masking everywhere the data gets queried. Lake Formation's column-level permissions and data filters solve this once, centrally, at the Catalog layer — every engine that reads the table through Lake Formation sees the same, consistently enforced view.

❯ Short version

Lake Formation adds column-, row-, and cell-level access control on top of the Glue Catalog — permissions IAM alone can't express — enforced consistently across Athena, Redshift Spectrum, EMR, and QuickSight. It's the practical answer to "this role should see this table, but not these PII columns," without duplicating data or hand-rolling masking in every consumer.

16. Amazon Kinesis Data Streams

A durable, ordered, replayable buffer for continuously arriving records

Everything covered so far assumes data that's already sitting still, in a file. Kinesis Data Streams is for data that isn't: clickstreams, IoT telemetry, application logs, change events — records arriving continuously, which one or more consumers need to process, often in near-real-time. A stream is split into shards, each an ordered, independently-scalable sequence of records; a partition key on each record decides which shard it lands on, and records with the same key always land on the same shard, preserving order for that key.

Why "durable and replayable" is the actual point

Records written to a Kinesis stream stay available for a configurable retention window (24 hours by default, up to 365 days) and can be re-read from any point in that window by any number of independent consumers, each tracking its own read position. That's the core difference from a simple pub/sub queue: a stream isn't consumed-and-gone, so a second consumer can be added later, or a failed consumer can rewind and reprocess, without losing or duplicating what already happened upstream.

17. Amazon Data Firehose

The managed, no-code way to land a stream in S3, Redshift, or an OpenSearch index

Amazon Data Firehose (formerly Kinesis Data Firehose) takes records — from a Kinesis stream, direct API calls, or several other AWS sources — and delivers them into a destination like S3, Redshift, or OpenSearch, batching, compressing, optionally transforming (via an attached Lambda), and handling retries automatically. It's the managed answer to "I don't want to write and operate a consumer application just to get streaming data into a bucket."

Kinesis Data Streams vs. Firehose — the question that actually decides which one you need. Do you need custom, low-latency, per-record processing logic, and/or multiple independent consumers reading the same records? Use Data Streams with your own consumer (often a Lambda). Do you just need the data to reliably land in S3/Redshift/OpenSearch, batched sensibly, with no consumer application to write or operate? Use Firehose — and note Firehose can itself read from a Kinesis stream, so the two are often used together, not as alternatives.

❯ Short version

Firehose is managed, no-code delivery of streaming records into S3, Redshift, or OpenSearch — batching, compressing, and retrying automatically, with an optional Lambda for light in-flight transforms. It complements Kinesis Data Streams rather than replacing it: Streams for custom processing and multiple consumers, Firehose for "just land it reliably."

19. Amazon Managed Service for Apache Flink

Continuous, stateful computation over a stream — the piece Kinesis Data Streams and Firehose don't do

Apache Flink is an open-source stateful stream-processing engine: it doesn't just move records (that's Kinesis Data Streams, see that section) or land them somewhere (that's Firehose, see that section) — it runs a continuous computation that keeps state across many events, such as a running count, a session window, or a join between two streams. Amazon Managed Service for Apache Flink (renamed from Kinesis Data Analytics, as this guide notes elsewhere) runs Flink applications without handing you a cluster to size or patch: AWS runs the underlying compute, automatically checkpoints application state to a durable store so a failed task resumes from where it left off instead of from zero, and scales the application's parallelism with load.

Why a Lambda function sitting on a Kinesis stream can't do this

Take "the rolling 5-minute average of views per video" or a fraud check that needs to correlate several events across a time window. A single Lambda invocation is stateless and short-lived: it processes one batch and exits, carrying nothing forward except what you manually write out and reload — typically to DynamoDB — on every single invocation, which is slow and easy to get subtly wrong (missed windows, double counts on retries). Flink keeps the aggregation state in memory, checkpointed for durability, for the entire life of the window, as a first-class part of how you write the application — not something you bolt on yourself.

Exam angle
"Continuous aggregation" or "windowing" in a question stem is a strong signal. If a scenario needs a rolling average, a session window, or correlating events over a time span, a Lambda-on-Kinesis answer choice is almost always the trap — the correct answer is Managed Service for Apache Flink. Plain per-record filtering or enrichment, with no window and no cross-event state, is the signal for the opposite: Lambda is enough, and Flink would be over-engineering.

Windowed state isn't free or automatically bounded. A window keyed on a high-cardinality field (e.g. a raw user_id with millions of distinct values, kept "forever") grows the checkpointed state size and cost right along with it. Bounding window length and key cardinality deliberately is part of designing the Flink application, not something the managed service does for you.

❯ Short version

Apache Flink is a stateful stream-processing engine — it computes continuously over events rather than just moving or delivering them. Managed Service for Apache Flink removes the cluster, handles checkpointing to a durable store, and auto-scales the application. Reach for it over a Kinesis-plus-Lambda consumer specifically when the problem needs a rolling window, a running aggregate, or state correlated across events — something a stateless, short-lived Lambda invocation structurally cannot hold onto.

20. Amazon OpenSearch Service

A managed search-and-analytics engine — the destination when a dashboard needs to search text, not just aggregate columns

Amazon OpenSearch Service is a managed, distributed search-and-analytics engine, forked from the Elasticsearch/Kibana lineage: it indexes documents in near-real-time and supports both full-text search (ranking, fuzzy matching, relevance scoring over free text) and log/metric analytics with interactive dashboards (OpenSearch Dashboards, the Kibana-lineage UI). It answers a different question than a data warehouse — "find the documents that match this text" rather than "aggregate these structured columns."

The data-engineering path in: Firehose as the on-ramp

Amazon Data Firehose can deliver a stream directly into an OpenSearch index as one of its built-in destinations (see the Firehose section) — no consumer application to write, the same batching/retry handling Firehose gives S3 or Redshift. That makes OpenSearch the natural landing spot whenever the actual requirement is "I need to search and explore this data interactively, including free text, not just run SQL aggregates on it" — application logs, clickstream events for ad hoc investigation, or an operational dashboard that engineers query by hand while debugging.

OpenSearch vs. Athena/Redshift — the actual decision point

These aren't competing for the same job. Searching free-text application log messages for a substring like "connection refused", ranked by relevance and explored interactively, is exactly what OpenSearch is built for — Athena and Redshift handle structured rows and don't do text relevance ranking at all. OpenSearch clusters need real capacity planning. Shard count and instance sizing behave more like a traditional database than most other services in this guide — OpenSearch is not a pay-per-query serverless service by default, and under-provisioned shards or nodes show up as slow queries and red cluster health, not an automatic scale-up. (OpenSearch Serverless exists as a newer option that manages capacity for you, trading some of that manual sizing control for a pay-for-what-you-use model.)

Choose OpenSearch for full-text search and operational dashboards; choose Athena/Redshift for structured SQL analytics over large historical data. Unlike most serverless services here, OpenSearch clusters need capacity planning — shards and instance sizing — unless you use OpenSearch Serverless.

21. Amazon SQS & SNS

A queue for one consumer to work through at its own pace, and a topic for fanning out to many

Amazon SQS is a message queue: a producer sends a message, and exactly one consumer (out of however many are polling) picks it up and processes it, deleting it when done — a natural fit for "a backlog of work items to get through," decoupling how fast producers send from how fast consumers can process. Amazon SNS is a pub/sub topic: a message published to a topic is delivered to every subscriber (which can itself be an SQS queue, a Lambda function, an email address, and more) — a natural fit for "several independent systems all need to react to this same event."

The "SNS fan-out to SQS" pattern. A very common combination: publish once to an SNS topic, and give each independent consumer its own SQS queue subscribed to that topic. Every consumer gets every message (SNS's fan-out), but each consumer processes its own queue at its own pace with its own retry/failure handling (SQS's durability) — getting both properties at once, which neither service alone provides.

SQS is a queue — one message goes to one consumer, good for work backlogs. SNS is pub/sub — one message fans out to every subscriber, good for notifying multiple independent systems of the same event. Combining them (SNS topic → multiple SQS queues, one per consumer) is a standard pattern for getting fan-out and per-consumer durability together.

22. Amazon EventBridge

Scheduling a pipeline, and routing events between services, without hand-wiring either

Amazon EventBridge does two related jobs. As a scheduler, a cron or rate expression triggers a target (a Lambda, a Step Functions execution) on a fixed cadence — the most common way a batch pipeline actually starts, replacing what used to be a literal cron job on a server nobody wants to maintain. As an event bus, it routes events from AWS services (an S3 upload, a Glue job state change) or custom application events to one or more targets, based on rules that match on the event's content — a more flexible alternative to wiring services directly together.

Where EventBridge sits relative to the "one Lambda triggers the next" instinct

It's tempting to have a Lambda directly invoke the next Lambda when it finishes. EventBridge (or Step Functions, next section) decouples that: a producer emits an event or runs on schedule without knowing or caring who consumes it, and rules/targets can be added, removed, or changed later without touching the producer's code at all. For anything beyond a single trigger→single action, this decoupling is what keeps a pipeline's services from becoming tightly, invisibly wired together.

❯ Short version

EventBridge schedules pipelines on a cron/rate cadence and routes events between AWS services or custom applications based on rules — the decoupled alternative to hard-wiring one service to directly call the next. It's almost always the first thing in a batch pipeline, and the mechanism behind most "this AWS event triggered that action" behavior.

~/aws-data-eng ❯ the conductor

23. AWS Step Functions

A visual state machine for stitching many services into one reliable, observable pipeline

AWS Step Functions defines a pipeline as a state machine — a sequence (and branching structure) of states, each usually a Task that invokes another AWS service (a Lambda, a Glue job, an ECS task) — with built-in, declarative handling for retries, error branching, parallel execution, and waiting, instead of every one of those concerns being hand-coded inside a monolithic script. It is the piece that turns "several separate AWS services" into "one pipeline with one execution history you can look at."

The state types that show up in almost every real pipeline

Task invokes a service and waits for its result. Wait pauses for a fixed time or until a timestamp (used, for example, to pace a step that depends on an upstream async process — like a Glue Crawler or Catalog update — finishing before the next state reads its result). Parallel runs multiple branches concurrently and waits for all to finish. Choice branches based on a prior state's output — commonly used right after a data-quality check, to route to "continue" or "fail loudly" depending on the result. Retry and Catch attach to any state: Retry re-attempts a failed state automatically (with backoff), and Catch routes a failure to a specific error-handling state instead of failing the whole execution silently.

Step Functions vs. orchestrating from inside a Lambda. It's possible to have one Lambda call several others in sequence and call that "orchestration." The difference that matters in production: Step Functions gives you a visual execution history for every single run (which state failed, with what input, at what time), declarative retry/catch without try/except sprawl, and a 15-minute Lambda timeout stops applying to the pipeline as a whole, since Step Functions itself can run for up to a year. A pipeline with more than two or three steps, or any step that can plausibly fail, benefits from this almost immediately.

❯ Short version

Step Functions defines a pipeline as a visual state machine — Task, Wait, Parallel, Choice, with declarative Retry/Catch — giving every execution a full, inspectable history. It's the standard way to stitch Lambda, Glue, and other AWS services into one reliable, observable pipeline instead of hand-coding orchestration logic inside a single script.

24. Amazon MWAA (Managed Workflows for Apache Airflow)

Orchestration as version-controlled Python DAGs — the choice for teams who already think that way

Amazon MWAA (Managed Workflows for Apache Airflow) is a managed Apache Airflow environment: AWS runs the webserver, scheduler, and worker infrastructure, and handles patching and version upgrades on a schedule you control. You author pipelines as DAGs (Directed Acyclic Graphs) — plain Python files — and upload them to an S3 bucket that MWAA continuously watches; drop a new or changed DAG file in, and the scheduler picks it up automatically, no redeploy step involved.

The real mental-model difference from Step Functions: code vs. configuration

A Step Functions state machine (see the AWS Step Functions section) is a declarative JSON/ASL document — you describe states, and Step Functions calls each AWS service directly through its own SDK integration. An Airflow DAG is imperative Python: you instantiate Operators (an S3KeySensor that waits for a file, a GlueJobOperator that starts a Glue job, a PythonOperator that runs any callable) as task objects, then wire dependencies between them with >>. The pipeline's logic — retries, branching, what runs in parallel — lives in a file that goes through the same code review and version control as any other application code, rather than in a separate state-machine artifact.

When a real team reaches for MWAA instead of Step Functions. The decision is rarely "which is more powerful" — both can express the same DAG-shaped pipeline. 25. AWS Database Migration Service (DMS)

One-time migration, and ongoing CDC replication, out of operational databases

AWS DMS replicates data from a source database (on-premises, RDS, or another cloud) to a target — often S3, for landing operational data into a data lake. It supports a one-time full load of existing data, and, separately, ongoing CDC (change data capture) replication that reads a database's transaction log and streams every subsequent insert/update/delete to the target continuously, without querying or impacting the source database with repeated polling queries.

Why log-based CDC beats "just run a SELECT on a schedule"

A naive alternative — periodically querying a source table for rows changed since last time — needs a reliable updated_at column (not every table has one), misses deletes entirely (a deleted row simply isn't there to select), and adds repeated read load to a live operational database. DMS's log-based CDC reads the database's own transaction log (binlog/WAL/redo log depending on engine), which captures every change, including deletes, with minimal load on the source — the same fundamental technique this guide's dbt-focused sibling covers in depth (collapsing a change log to current state via ROW_NUMBER() ordered by log sequence number).

❯ Short version

DMS moves data out of operational databases into a target like S3 — a one-time full load, plus ongoing CDC replication read from the source's transaction log, which captures deletes and adds minimal load compared to polling with SELECT. It's the standard bridge from "data trapped in a live application database" to "data available in the lake."

26. Amazon RDS, Aurora & Purpose-Built Databases

The operational relational stores a pipeline pulls from via DMS, plus four purpose-built engines for shapes SQL doesn't fit well

Amazon RDS and Amazon Aurora are the two managed relational database services you'll actually find running production workloads, and — together with Amazon DynamoDB (see that section) — they're the databases a real data pipeline most often needs to get data out of, not build analytics on top of directly. Everything else in this guide (S3, Glue, Athena, Redshift) assumes data has already arrived from somewhere; RDS and Aurora, reached via AWS DMS (see the DMS section), are frequently that somewhere. This section also briefly covers four narrower, purpose-built databases — DocumentDB, Neptune, Keyspaces, and MemoryDB for Redis — that the exam expects you to recognize by the specific access pattern each is built for, not by deep operational detail.

Amazon RDS — a managed engine, not a managed schema-less black box

Amazon RDS runs PostgreSQL, MySQL, MariaDB, SQL Server, or Oracle as a managed service: AWS handles patching, automated backups, and — in a Multi-AZ deployment — synchronous replication to a standby in a second Availability Zone that's promoted automatically if the primary fails. What AWS does not take off your plate: you still pick a specific engine and version, and you still schedule and test its upgrades. In a data pipeline, RDS is almost always cast as the source — the classic operational database an application writes to, that a pipeline reads from later, not a place data gets loaded into for analytics.

Amazon Aurora — same SQL, different storage underneath

Amazon Aurora is AWS's own MySQL- and PostgreSQL-compatible engine: wire-compatible with the open-source engines, but built on a distributed, log-structured storage layer that's decoupled from compute and replicated across 3 Availability Zones by default, instead of one synchronously-mirrored standby. That architecture is why AWS's own published guidance claims significantly higher throughput than standard MySQL/PostgreSQL, and faster failover than a typical Multi-AZ RDS deployment — because a newly-promoted Aurora primary picks up the same shared storage instantly, with no standby storage state that needs rebuilding or catching up first. Treat both of those as AWS-reported figures, not independently reproduced benchmarks — the same caution this guide applies to other AWS-published performance claims. Aurora Serverless v2 is the variant that scales compute capacity up and down automatically instead of running on a fixed instance size, useful for spiky or hard-to-size workloads.

# Aurora exposes two different endpoints for one cluster -- they're not interchangeable
mycluster.cluster-xxxxxx.us-east-1.rds.amazonaws.com      # writer endpoint: app traffic goes here
mycluster.cluster-ro-xxxxxx.us-east-1.rds.amazonaws.com    # reader endpoint: point BI/reporting here

The anti-pattern this guide keeps warning about, applied to RDS/Aurora specifically. Pointing a BI tool or a scheduled batch job straight at a live RDS/Aurora primary for repeated analytical queries competes for the exact compute and I/O the production application needs — a slow analytical scan can visibly slow down the app it's supposed to be a harmless bystander to. An Aurora read replica offloads that read traffic to a separate instance; better still, replicate into the lake/warehouse via DMS's log-based CDC (see the DMS section) — which reads RDS/Aurora's own binlog rather than querying live tables at all — and run analytics there instead of against the OLTP database.

Aurora is AWS's own MySQL/PostgreSQL-compatible engine on different, distributed storage — AWS claims higher throughput and faster failover than standard RDS as a result. Both are usually the source a pipeline reads from via DMS's log-based CDC, never a target for repeated live analytical queries — that's what read replicas and the lake/warehouse are for. DocumentDB, Neptune, Keyspaces, and MemoryDB for Redis round out the purpose-built lineup for document, graph, wide-column, and ultra-low-latency workloads respectively.

27. Data Movement Services: AppFlow, Transfer Family, DataSync & Snow Family

Four services, four different sources — match the scenario's origin to the one built for it

AWS DMS (see that section) covers one specific data-movement problem: replicating out of a live database. These four services each cover a different one — the exam (and real production decisions) hinge on matching a scenario's data source to the service built for exactly that source, rather than any of the others that also, technically, "move data."

Amazon AppFlow runs no-code, managed flows that move data directly between SaaS applications — Salesforce, ServiceNow, Zendesk, Slack, and others — and AWS services like S3 or Redshift, on a schedule or in response to an event, using a pre-built connector instead of hand-written API pagination and auth code. It overlaps in spirit with the zero-ETL SaaS integrations covered in the What's New section, but isn't the same thing: those are fully managed replication with no flow to configure at all, while AppFlow still has you define a flow (source, destination, field mapping, schedule) — the right tool when a given SaaS app isn't a zero-ETL-covered source, or when the pipeline needs transformation/filtering a zero-ETL pipe doesn't offer.

AWS Transfer Family provides managed SFTP, FTPS, and FTP endpoints backed by S3 (or Amazon EFS — AWS's managed, NFS-based file system) storage. It exists because a genuinely large share of real inter-company data exchange — especially in finance, healthcare, and government EDI-style exchanges — still happens over these decades-old file-transfer protocols, and a partner or legacy system that only speaks SFTP isn't going to be talked into calling an API instead. Transfer Family gives that partner the exact endpoint it expects, while the file lands in S3 like any other object.

AWS DataSync moves bulk file-based data — as a one-time migration or on an ongoing schedule — between on-premises NFS/SMB file shares and AWS storage (S3, EFS, FSx), or between AWS storage services, with built-in integrity verification on every transfer. It's the online alternative to shipping hardware: fast and managed, appropriate whenever the available network bandwidth can move the dataset in an acceptable amount of time.

The AWS Snow Family (Snowball, Snowcone, Snowmobile) is what you reach for when it can't: AWS ships a physical, ruggedized device to your location, you load data onto it locally, and ship it back for AWS to ingest into S3. The rough intuition for when this beats an online transfer is bandwidth math — moving tens to hundreds of terabytes (or more) over a limited or expensive network connection can take weeks, while the same data written to a device and shipped can land in days, which is why physical shipment remains a real, non-quaint answer for very large one-time migrations.

Two more Migration and Transfer services round out the exam's list, and they solve a different problem again: moving whole servers, not data files. AWS Application Discovery Service scans an on-premises environment (via an agent or agentless collector) to inventory servers, their utilization, and their network dependencies — the assessment step before a migration, answering "what do we even have, and what talks to what" before anyone moves anything. AWS Application Migration Service (MGN) is the lift-and-shift tool that actually rehosts those on-premises (or other-cloud) servers as EC2 instances, replicating the running server's disk state continuously so cutover is a short, low-risk switch rather than a from-scratch rebuild. Neither is data-pipeline-specific — they migrate infrastructure, not tables — but a data engineering team is often on the receiving end of one: a legacy on-premises data warehouse server that Application Migration Service rehosts onto EC2 is frequently the exact source DMS (see that section) then starts replicating from.

28. Amazon CloudWatch

Logs, metrics, and alarms — the observability layer underneath every service in this guide

Amazon CloudWatch collects logs and metrics from nearly every AWS service automatically — a Lambda's print statements, a Glue job's Spark logs, a Step Functions execution's state transitions — and lets you build alarms that fire when a metric crosses a threshold (error count, duration, a custom business metric a pipeline emits itself). For a data pipeline specifically, CloudWatch is usually where you'd notice a job silently stopped running, started taking much longer than usual, or started failing at a higher rate — none of which a pipeline's own success/failure return value necessarily surfaces on its own.

Logs vs. metrics vs. alarms — three layers, not three names for the same thing

Logs are the detailed, textual record of what happened — what you read after something's already gone wrong, to find out why. Metrics are numeric time series (invocation count, duration, error rate) — what you'd graph to see a trend. Alarms watch a metric and act (notify via SNS, trigger a Lambda) when it crosses a threshold — what actually gets a human's attention without them having to go look. A pipeline with only logs, and nobody reading them proactively, is a pipeline that fails silently; alarms are what close that gap.

❯ Short version

CloudWatch is the observability layer under every AWS service — logs for post-incident detail, metrics for trends, alarms for getting notified automatically when something crosses a threshold. A pipeline without alarms wired to at least its failure states and its freshness/latency will fail silently far more often than one with them.

CloudTrail vs. Config — an audit log is not the same thing as a compliance check

Parameter Store (see the Security & Encryption section) holds cheap, non-rotating configuration values. The AWS Well-Architected Tool walks a specific, named workload through a structured review against the six pillars introduced in the Fundamentals section, producing a list of identified risks and improvement items tied to that workload. It's a formal, repeatable review process, not a monitoring service — it doesn't touch running infrastructure, it organizes the review conversation and tracks whether previously flagged risks actually got addressed.

Cost Explorer & Budgets — catching the bill before it arrives

Cost Explorer visualizes and forecasts spend by service, tag, or cost category; AWS Budgets sets a threshold against that spend and alerts proactively (before the bill closes out, not after). The classic data-engineering cost surprise is a runaway Athena query scanning far more data than intended — a missing partition filter turning a few-cent query into one that scans an entire dataset — or an always-on EMR cluster nobody remembered to terminate after a one-off job. A concrete guardrail against the first: cap an Athena workgroup's per-query scan limit, so a bad query fails fast instead of quietly billing for terabytes.

AWS Data Exchange — buying data instead of building a pipeline to get it

AWS Data Exchange lets you subscribe to third-party datasets directly from AWS Marketplace, with many providers delivering updates straight into an S3 bucket in your own account rather than requiring you to build and maintain an ingestion pipeline against an external API. For data that's genuinely commodity (market reference data, demographic datasets, weather history), this is often a faster and cheaper path than acquiring and maintaining the equivalent pipeline yourself.

AWS Backup — one policy instead of a backup script per service

AWS Backup centralizes backup scheduling, retention, and restore across services that would otherwise each need their own backup mechanism — Amazon EBS volumes (the block storage attached to EC2 and EMR nodes), RDS/Aurora databases, DynamoDB tables, EFS file systems, and more, all governed by one reusable backup plan instead of a different script or console setting per service. For a data platform, this matters most for the stateful pieces that aren't just files sitting in S3 (which already has its own versioning/replication story) — an RDS instance behind DMS, or an EBS-backed EMR cluster's local storage, are the kinds of resources a real backup plan actually needs to cover.

Two pitfalls worth calling out directly. CloudTrail records API activity against AWS — it has no idea what happened inside a running Glue job or what it printed; that's still CloudWatch Logs' job (see the CloudWatch section). And AWS Config is enabled per-region — turning it on in ap-southeast-1 says nothing about resources sitting in us-east-1; a genuinely account-wide compliance view needs Config enabled in every region actually in use, or aggregated across an AWS Organization.

❯ Short version

CloudTrail answers who called which API and when; AWS Config answers whether a resource's configuration is compliant right now, continuously; CloudWatch answers what a running service or job actually did. Session Manager gives keyless shell access; Managed Grafana covers ops dashboards; the Well-Architected Tool formalizes a workload review. Cost Explorer and Budgets exist specifically to catch a runaway Athena scan or a forgotten EMR cluster before the bill does, and Data Exchange is the option to buy a dataset instead of building a pipeline to acquire it.

30. Amazon DynamoDB

Not a data-lake service itself — but the operational database most CDC pipelines pull from

Amazon DynamoDB is a fully managed, serverless NoSQL key-value/document database, built for single-digit-millisecond lookups by a known key at massive scale. It shows up in this guide not as an analytics engine — it isn't one — but because a huge share of real-world data pipelines exist specifically to move data out of an operational store like DynamoDB (an application's live order table, a user-profile table) and into the analytics stack this guide otherwise covers.

DynamoDB Streams — the CDC mechanism that's actually built in

DynamoDB Streams captures a time-ordered log of every item-level change (insert/update/delete) in a table, which a Lambda function can subscribe to directly — no separate CDC tool needed, unlike the log-parsing DMS does for relational databases (section 25). This is the standard way a DynamoDB-backed application's data ends up landing in S3 continuously rather than through periodic full-table exports, and it's also the mechanism behind the DynamoDB-to-S3-Tables zero-ETL integration AWS added in 2025 (section 34).

❯ Short version

DynamoDB is a managed NoSQL database for operational, key-based access at scale — not an analytics engine. It matters to a data engineer chiefly through DynamoDB Streams, its built-in item-level change log, which is the standard way DynamoDB-backed application data flows into a data lake via CDC without a separate replication tool.

31. Amazon QuickSight

Where the pipeline's output finally becomes a chart someone in the business actually looks at

Amazon QuickSight is AWS's managed BI tool — dashboards, visuals, and (via its SPICE in-memory engine) fast interactive exploration over data pulled from Athena, Redshift, S3, or a growing set of other sources. Everything else in this guide exists to get data clean, modeled, and queryable; QuickSight is usually the last hop, the layer a non-engineer in the business actually opens.

Exam angle

SPICE — why QuickSight dashboards feel instant even over large tables

SPICE (Super-fast, Parallel, In-memory Calculation Engine) is QuickSight's option to import and cache a dataset in memory rather than re-querying the source engine on every single interaction. A dashboard built on SPICE stays fast even as users filter and drill in repeatedly; the tradeoff is that SPICE data is a snapshot as of its last refresh, not live, so a dashboard needing up-to-the-second data should query the source directly instead — a small, deliberate choice that decides whether "fast" or "fresh" wins for a given dashboard.

❯ Short version

QuickSight is the managed BI layer where a pipeline's gold-layer output finally becomes a dashboard. 32. Machine Learning & AI Services for Data Engineers

SageMaker, Bedrock, Kendra, and API Gateway — recognized well enough for the exam, not operated deeply

The DEA exam includes a handful of ML/AI services, but it tests them from one angle: does the data engineer understand that these services are usually downstream consumers of the pipeline, not something the data engineer builds or tunes? A model is only as good as the data feeding it, and preparing that data — cleaned, joined, correctly typed, on a predictable refresh schedule — is squarely a data engineering job. Training the model, choosing hyperparameters, and serving predictions at scale usually is not.

Amazon SageMaker AI and the Feature Store bridge

Amazon SageMaker AI is AWS's managed ML platform — notebooks for experimentation, managed training jobs that spin up compute and shut it down when done, and hosting/inference endpoints that serve predictions over an API. A data engineer's actual touchpoint is almost never the training loop itself; it's making sure a pipeline's gold-layer output is shaped as clean, well-modeled features — the columns a model actually trains on — and making those features available consistently for both training and real-time inference. That specific bridge has a name: SageMaker Feature Store, which stores versioned feature values in an offline store (for training, backed by S3) and an online store (for low-latency lookups at inference time), so the same feature definition and the same values are used in both places instead of drifting apart.

Amazon Bedrock gives managed API access to foundation models — Anthropic's Claude and others — without provisioning any infrastructure, for building generative AI applications. The data-engineering-relevant angle is retrieval-augmented generation (RAG): a Bedrock-based application answers questions well only if it can retrieve the right internal documents at query time, usually from a vector store holding embedded chunks of a knowledge base. A data engineer may be asked to build the pipeline that chunks source documents, generates embeddings, and loads them into that vector store — the same extract-transform-load discipline this guide covers elsewhere, aimed at a different destination.

Amazon Kendra is a managed enterprise search service: point it at internal documents, wikis, and file shares, and it answers natural-language questions ("what's our expense reimbursement policy?") with ranked, relevant passages, using built-in relevance tuning and connectors so there's little to operate. It covers similar conceptual ground to Amazon OpenSearch Service — both index and search text — but the practical distinction matters: Kendra is a closer-to-turnkey enterprise-search product with minimal configuration, while OpenSearch is a lower-level, far more customizable search and analytics engine that a team operates and tunes itself (and, elsewhere in this guide, shows up as a Firehose delivery destination for streaming data). Reach for Kendra when the requirement is "search our documents"; reach for OpenSearch when the requirement is "build and control a custom search or log-analytics index."

33. Infrastructure as Code & CI/CD for Data Pipelines

Why production pipeline infrastructure gets defined in code instead of clicked together by hand

Hand-clicking through the AWS console to create an S3 bucket, a Glue job, an IAM role, and a Step Functions state machine works fine for a first prototype. It falls apart the moment there's a second environment: nothing guarantees staging matches prod, there's no code review for an infrastructure change the way there is for application code, and recreating an environment from scratch after deleting it means remembering — and re-clicking — every setting by hand. Infrastructure as Code (IaC) fixes this by defining infrastructure as a versioned, reviewable artifact that gets deployed the same way every time.

AWS CloudFormation — AWS's native, declarative IaC

AWS CloudFormation takes a YAML or JSON template describing the desired resources — an S3 bucket, a Glue job, an IAM role, whatever — and deploys or updates them together as a single unit called a stack. It's declarative: you describe the end state, and CloudFormation figures out the create/update/delete calls needed to get there, including rolling back automatically if a deployment fails partway through.

AWS CDK — the same stack, written in a real programming language

Writing every resource by hand in YAML gets repetitive fast — ten pipelines with a similar S3-bucket-plus-Glue-job shape mean ten near-identical, copy-pasted template blocks. The AWS CDK (Cloud Development Kit) lets you define that same infrastructure in Python, TypeScript, or another supported language, using loops, functions, and reusable constructs instead — then cdk synth compiles it down to an ordinary CloudFormation template, and cdk deploy deploys it exactly like any other stack. Teams choose CDK over raw CloudFormation for the same reason they choose code over a spreadsheet: type-checking catches a typo'd property name before deployment, and one function can define twenty near-identical pipelines instead of twenty pasted YAML blocks.

# CDK (Python) -- defines the same CloudFormation stack, in code
bucket = s3.Bucket(self, "GoldBucket",
    versioned=True,
    removal_policy=RemovalPolicy.RETAIN)

glue_job = glue.CfnJob(self, "SilverToGoldJob",
    role=glue_role.role_arn,
    command={"name": "glueetl",
             "scriptLocation": f"s3://{bucket.bucket_name}/scripts/gold.py"})

AWS SAM (Serverless Application Model) is a CloudFormation extension purpose-built for serverless resources specifically — Lambda functions, API Gateway APIs, DynamoDB tables — with a shorthand syntax that needs far fewer lines than the equivalent raw CloudFormation, plus a local-testing CLI (sam local invoke) that runs a Lambda function on a laptop before it's ever deployed. It's narrower in scope than CDK or CloudFormation by design: reach for SAM specifically for serverless application resources, not as a general-purpose IaC replacement for a whole data platform.

The AWS CLI scripts one-off or repeated operations against any AWS service straight from a terminal — aws s3 cp, aws glue start-job-run, aws cloudformation deploy — and it's the tool that actually runs inside most CI/CD deploy scripts, gluing together whatever a pipeline stage needs to do: package a template, kick off a job, check a status.

CI/CD for pipeline code — the same pattern software engineering already uses

A data pipeline's Glue scripts, Lambda functions, and CloudFormation/CDK templates benefit from the same source-control-triggered release process application code does, just aimed at infrastructure and transformation logic instead of a web app. AWS CodePipeline orchestrates the overall release — source, build, test, deploy stages, triggered by a commit. AWS CodeBuild runs the actual build/test commands inside that pipeline, such as linting a Glue script or running unit tests against transformation logic. AWS CodeDeploy handles the deployment step itself onto compute targets like EC2, ECS, or Lambda.

Amazon Q Developer is AWS's generative-AI assistant for writing and explaining code and infrastructure templates inside supported IDEs and the AWS console — useful for drafting a CDK construct or understanding an unfamiliar CloudFormation template faster, but it's a productivity tool sitting alongside this workflow, not a pipeline component itself.

Manual console edits cause state drift. If CloudFormation (directly or via CDK) created an IAM policy, and someone later tweaks that policy by hand in the console — adding a permission, say — the stack's recorded template no longer matches the deployed resource. The next stack update can silently revert that manual change, or fail outright on a conflicting property, because CloudFormation only knows about the state it last wrote, not any state changed outside it.

❯ Short version

CloudFormation deploys declarative YAML/JSON templates as a stack; CDK writes that same infrastructure in a real language and synthesizes it down to CloudFormation, trading copy-paste for loops and reusable constructs; SAM is a narrower, serverless-specific extension with a local-test CLI. CodePipeline/CodeBuild/CodeDeploy apply ordinary CI/CD to infrastructure and Glue/Lambda code changes. Never hand-edit a resource that IaC also manages — it causes drift that the next deploy can silently undo.

35. Worked Example: A Real YouTube Trending Data Pipeline

What each layer actually is, concretely, in this pipeline

Bronze is the ingestion Lambda's raw output — untouched JSON from the YouTube API, partitioned by region/date/hour — plus historical Kaggle CSVs uploaded once for backfill. Silver is the output of two independent transforms running in parallel: a Glue Spark job (bronze_to_silver_statistics) that enforces schema across both the API-JSON and Kaggle-CSV shapes, casts types, dedupes, and derives metrics like like_ratio; and a small Lambda (json_to_parquet) that normalizes the separate category-ID reference data. 36. Worked Example: The Step Functions Execution, State by State

Every Retry, Catch, and Choice in the real state machine, and why each one is there

This is the actual shape of the Step Functions state machine that runs the pipeline from section 35 — the same structure introduced abstractly in section 23's diagram, now traced state by state with the real reasoning behind each retry and branch.

"IngestFromYouTubeAPI": {
  "Type": "Task",
  "Resource": "arn:aws:states:::lambda:invoke",
  "Parameters": { "FunctionName": "...youtube-ingestion-dev", "Payload": {...} },
  "Retry": [{ "ErrorEquals": ["Lambda.ServiceException", "Lambda.TooManyRequestsException"],
              "IntervalSeconds": 30, "MaxAttempts": 3, "BackoffRate": 2 }],
  "Catch": [{ "ErrorEquals": ["States.ALL"], "Next": "NotifyIngestionFailure" }],
  "Next": "WaitForS3Consistency"
}

Reading this state, line by line

The Retry block targets two specific, transient error types — a Lambda service hiccup or a throttling response — and backs off exponentially (30s, 60s, 120s) rather than hammering a rate-limited API immediately again. That specificity matters: retrying every error type, including a genuine bug in the Lambda's code, would just waste three attempts before failing anyway. The Catch block is the safety net for everything Retry doesn't handle (or that still fails after retries exhaust) — States.ALL routes any remaining failure to a dedicated notification state, so the execution never just silently dies with no one told.

"WaitForS3Consistency": { "Type": "Wait", "Seconds": 10, "Next": "ProcessInParallel" }

"ProcessInParallel": {
  "Type": "Parallel",
  "Branches": [
    { "StartAt": "TransformReferenceData", "States": { "TransformReferenceData": {
        "Type": "Task", "Resource": "arn:aws:states:::lambda:invoke", "End": true } } },
    { "StartAt": "RunBronzeToSilverGlueJob", "States": { "RunBronzeToSilverGlueJob": {
        "Type": "Task", "Resource": "arn:aws:states:::glue:startJobRun.sync", "End": true } } }
  ],
  "Catch": [{ "ErrorEquals": ["States.ALL"], "Next": "NotifyTransformFailure" }],
  "Next": "RunDataQualityChecks"
}

Two details worth noticing in this Parallel state. First, the short Wait before it exists as a small, deliberate pacing buffer — giving the ingestion Lambda's write and any Catalog-side bookkeeping a moment to settle before the next stage reads that output (S3 itself has provided strong read-after-write consistency for every request since December 2020, so this isn't working around an S3 consistency gap — it's just cheap insurance against other, less-guaranteed propagation delays). Second, the Glue branch's Resource ends in .sync — that suffix tells Step Functions to wait for the Glue job to actually finish (not just accept the start request) before treating the branch as done, which is what lets ProcessInParallel correctly wait for both the Lambda and the Glue job before moving on.

"EvaluateDataQuality": {
  "Type": "Choice",
  "Choices": [{ "Variable": "$.dq_result.Payload.quality_passed",
                "BooleanEquals": true, "Next": "RunSilverToGoldGlueJob" }],
  "Default": "NotifyDQFailure"
}

The Choice state — this is the actual data-quality gate

This is the mechanism, not a metaphor: the DQ Lambda's boolean result, sitting at $.dq_result.Payload.quality_passed in the execution's accumulated state, is the single value this Choice state branches on. If it's true, Gold aggregation runs. Otherwise, execution falls to Default — NotifyDQFailure — and Gold simply never runs for that execution. Section 37 covers what the DQ Lambda actually checks to produce that boolean.

❯ Short version

The real state machine pairs targeted Retry (specific transient errors, exponential backoff) with a catch-all Catch (nothing fails silently), uses a Wait to dodge an S3 consistency race, uses .sync so a Parallel branch actually waits for a Glue job to finish, and gates Gold behind a Choice state reading one boolean the DQ Lambda produced.

A DQ Lambda producing one boolean, and four distinct failure paths, each telling a human something different

Why four separate SNS notification states, not one

NotifyIngestionFailure, NotifyTransformFailure, NotifyDQFailure, and NotifyGoldFailure each publish to the same SNS topic, but with a distinct Subject and a message built from that specific failure's context ($.error for a hard failure, the full $.dq_result payload for a DQ failure). A single generic "pipeline failed" alert would tell an on-call engineer nothing about where to start looking; these four tell them immediately whether the problem is upstream (ingestion), in a transform, in the data itself (DQ), or in the final aggregation — collapsing what would otherwise be several minutes of digging through Step Functions' execution history into what the alert's subject line already says.

  "Type": "Task",
  "Resource": "arn:aws:states:::sns:publish",
  "Parameters": {
    "TopicArn": "arn:aws:sns:ap-south-1:<account-id>:yt-data-pipeline-alerts-dev",

    "Message.$": "States.JsonToString($.dq_result)"
  },
  "End": true
}

Notice this is a WARNING, not a FAILURE, in its own subject line — and that's deliberate. A DQ gate failing doesn't mean anything crashed; it means the pipeline correctly refused to promote data it couldn't vouch for. That's a meaningfully different signal than a Lambda throwing an exception, and the pipeline's alert wording reflects it — an important habit for any DQ gate: failing loudly and safely is the gate working as designed, not the pipeline breaking.

The DQ Lambda checks row count, null percentage, schema, value ranges, and freshness, producing one pass/fail boolean the Step Functions Choice state branches on. 38. End-to-End Lakehouse Project on AWS: Architecture Overview

The modern shape: one Iceberg-backed copy of the data, queried by both the lake side and the warehouse side

A lakehouse is a specific architectural claim, not a marketing word: a single storage layer — here, Amazon S3 — holding data in an open table format (Apache Iceberg, either as an S3 Tables bucket or as ordinary S3 objects registered as Iceberg tables in the Glue Data Catalog) that both lake-style engines (Spark on EMR or Glue, Athena) and a warehouse engine (Redshift, via Spectrum or native Iceberg support) can read and write. That last part is the whole point: a classic two-tier "lake plus warehouse" architecture keeps two physical copies of the same data — one in S3 for the lake, one loaded into the warehouse's own storage — synced by a scheduled ETL job that can drift, lag, or simply fail one night. A lakehouse deletes that second copy and that sync job entirely. The worked pipeline in the Worked Example: A Real YouTube Trending Data Pipeline section is the other, simpler shape — plain Parquet on S3, no table format, no cross-engine write path — and is worth keeping in mind as the contrast point throughout this section and the next.

Cross-cuttingStep Functions, CloudWatch, KMS + IAMStep Functions (see AWS Step Functions) — or MWAA, noted as its DAG-based alternative in the AWS-Native vs. Cross-Cloud Alternatives section — sequences every layer above on a schedule; CloudWatch (see Amazon CloudWatch) watches for failures and staleness; KMS-encrypted storage plus IAM roles (see IAM for Data Engineers) secure every layer without each one reinventing access control.

Why this earns its complexity over the plain-Parquet medallion pattern

The YouTube pipeline's medallion layers (bronze/silver/gold, all plain Parquet on S3) are simpler to build, and that simplicity is a legitimate choice for a lot of real workloads. Iceberg earns the extra moving parts specifically when a project needs: ACID transactions — a reader querying a table mid-write sees either the old snapshot or the new one, never a half-written mix of files, which plain Parquet on S3 cannot guarantee under concurrent writers; schema evolution without rewriting history — adding, renaming, or widening a column updates the table's schema going forward without touching a single existing data file; time travel — querying the table as of a prior snapshot, covered as an operational tool in the next section; and, biggest of all, one physical copy serving both the lake engines and the warehouse engine, removing the "load into Redshift" step and the drift risk that step always carries.

Iceberg doesn't remove the small-files problem — it just gives you a tool to fix it. Continuous writers (a Firehose-fed streaming ingest job, or frequent small Glue Spark commits) still produce many small underlying data files per Iceberg table, which slows scans just like it would with plain Parquet. Iceberg's answer is periodic compaction (rewriting many small files into fewer large ones via a Spark job or Athena's table-optimization support) — a maintenance step a real project has to actually schedule, not something that happens automatically just because the table format is Iceberg.

❯ Short version

A lakehouse is one Iceberg-backed copy of data on S3 that both lake engines (Athena, Spark/EMR) and a warehouse engine (Redshift) read and write directly — no separate "load into the warehouse" step. The reference shape chains sources → ingestion (Lambda/Glue, DMS, Firehose) → S3+Iceberg storage with Glue Catalog and Lake Formation governance → Glue Spark/EMR transformation → dual serving (Athena + Redshift on the same gold tables) → QuickSight/ML consumption, with Step Functions, CloudWatch, and KMS/IAM running across all of it. It earns its complexity through ACID transactions, safe schema evolution, time travel, and eliminating a duplicate, drift-prone copy.

39. End-to-End Lakehouse Project: Building It Step by Step

Turning the architecture from the previous section into an actual build order, with the reasoning behind each step

The previous section laid out the shape. This one walks it in the order a real project would actually build it — not source code order, but decision order: what has to exist before the next thing can safely land on top of it. Each step below leans on a full section elsewhere in this guide for the mechanics and only explains what's specific to building it this way, as a lakehouse, rather than the plain-Parquet way the earlier worked example used.

Step 1 — Provision storage and governance before any data lands

Create the S3 bucket(s) — or enable an S3 Tables bucket directly — register the Iceberg table format, stand up the target databases in the Glue Data Catalog, and register Lake Formation as the permissions layer on top of that catalog, all before a single ingestion job runs. This ordering matters: retrofitting governance onto a lake that's already accumulated data means auditing every existing table for who can already see what, under whatever ad hoc bucket-policy or IAM rules grew up organically — a much bigger job than declaring permissions once, up front, and having every table inherit them by default from the moment it's created.

Step 2 — Land raw data through three ingestion paths into one bronze table format

Stand up all three ingestion paths from the architecture: a batch Lambda or Glue job for files, a DMS CDC task streaming changes off the operational RDS/Aurora database, and Firehose receiving app events from Kinesis or MSK — and point all three at bronze tables using the same Iceberg table format, not a separate streaming-only store for the Firehose path. That single decision is what makes this a lakehouse rather than two parallel systems bolted together: a batch record and a streaming record that both describe "an event that happened" end up in tables governed by the same catalog, the same Lake Formation rules, and the same downstream Spark code, instead of one path needing its own bespoke query tooling.

Step 3 — Transform bronze → silver → gold with Iceberg-aware Spark code

Write the Glue Spark ETL jobs the same way the medallion pattern always works — bronze cleaned into silver, silver aggregated into gold — but change how the job writes. A plain-Parquet job (like the one behind the YouTube pipeline's silver and gold layers) writes with a raw path, e.g. df.write.mode("overwrite").parquet("s3://bucket/gold/table/"): the job just drops files at a path, and nothing enforces that the next write matches the previous write's schema. An Iceberg job instead writes through the catalog:

# Plain Parquet path write (the shape behind the YouTube pipeline's Silver/Gold jobs)
df.write.mode("overwrite").parquet("s3://bucket/gold/channel_analytics/")

# Catalog-aware Iceberg write (this lakehouse's Silver/Gold jobs)
df.writeTo("glue_catalog.gold.channel_analytics") \
  .using("iceberg") \
  .createOrReplace()

Why that difference is what unlocks ACID and schema evolution

Writing through glue_catalog.gold.channel_analytics means the write goes through Iceberg's transaction machinery: new data files get written, then a single atomic commit swaps the table's metadata pointer to include them, so any concurrent reader either sees the fully-committed old state or the fully-committed new state — never a partial mix. A raw .parquet() path write has no such commit step; a reader hitting that S3 path mid-write can see an inconsistent set of files. The catalog-aware write path is also what lets a later ALTER TABLE ... ADD COLUMN apply going forward without rewriting every historical Parquet file underneath.

Step 4 — Gate the promotion to gold, at two layers

Reuse the exact pattern from the Worked Example: The Data Quality Gate & SNS Alerting Pattern and Worked Example: The Step Functions Execution, State by State sections — a DQ Lambda producing a pass/fail boolean, and a Step Functions Choice state branching on it — sitting between silver and gold in this lakehouse exactly as it does in the YouTube pipeline. Iceberg adds a second, complementary layer underneath that application-level check: because gold tables have a real, registered schema, a write that doesn't match it — a column with the wrong type, say — can be rejected by the table itself at write time, not just eventually noticed by a downstream consumer. The DQ gate catches bad values; Iceberg's schema enforcement catches bad shape, and a real pipeline wants both.

Step 5 — Enforce governance on the finished gold layer

Apply Lake Formation permissions (see AWS Lake Formation) directly on the gold tables once they exist: a column-level filter masking a PII column like customer_email for a role that only needs aggregate metrics, and a row-level filter restricting a regional analyst role to WHERE region = 'ap-south-1' rather than every region in the table. Because these rules live on the table in the catalog rather than inside each consuming query or dashboard, a new consumer — a new QuickSight dashboard, a new Athena workgroup, a data scientist's notebook — inherits the same restrictions automatically, with nothing to re-implement per consumer.

Step 6 — Serve the same gold tables two ways, from one copy

Point Athena at the gold tables for ad hoc SQL and BI-tool connections, and point Redshift (via Spectrum, or native Iceberg table support) at the identical tables for the heavy, repeated analytical joins a provisioned or serverless warehouse engine handles well. This is the step worth pausing on: it is structurally impossible in the YouTube pipeline's plain-Parquet worked example without adding a separate load job to copy gold data into Redshift's own storage first. Here, both engines query the same physical Iceberg table at the same time — there is no second copy to load, refresh, or let drift out of sync with the first.

Step 7 — Orchestrate and observe the whole pipeline

Wrap every step above in a Step Functions state machine (see AWS Step Functions) on a schedule, following the same Retry/Catch/Choice shape traced in the Step Functions worked example section — the state-machine mechanics don't change just because the tables underneath are Iceberg instead of plain Parquet. Add CloudWatch alarms (see Amazon CloudWatch) on job failures and on data freshness, and route failures to SNS, exactly as the DQ gate worked example does. None of this is new for a lakehouse; it's the same operational discipline the rest of this guide already covers, applied to one more pipeline.

Step 8 — Iterate, using time travel as a real operational tool

Because every write to an Iceberg table creates a new snapshot rather than overwriting the old one in place, a bad transformation run — a Glue job with a subtle bug that silvers or golds data incorrectly — can be recovered from by querying the table as of the snapshot immediately before that run, e.g. a FOR VERSION AS OF <snapshot_id>-style clause (exact syntax varies by engine), and re-pointing the table there or re-running downstream logic against that earlier state. That is a genuinely practical recovery path a plain-Parquet table doesn't offer without a separate, manually-maintained backup/restore process.

Time travel is not a permanent backup. Iceberg snapshots expire on a retention policy (configurable, but not infinite) and old data files get cleaned up behind expired snapshots to control storage cost. Treat it as a short-window operational safety net for "I need yesterday's state to fix today's bad run" — not as a substitute for actual backups of anything that needs to be recoverable months later.

Exam angle
"A single copy of data queried by multiple analytics engines, with ACID guarantees and schema evolution" is describing this pattern by definition. When an exam question describes that requirement, the answer is reaching for Iceberg (via S3 Tables or Glue-Catalog-registered tables) plus the Glue Data Catalog and Lake Formation — not "copy the data into Redshift" or "build a separate pipeline per consuming engine." The tell is any phrase like "without duplicating data" or "queried by both Athena and Redshift" sitting next to "consistent" or "no stale copies" — that combination is this section's architecture, not a two-tier lake-plus-warehouse design.

❯ Short version

Build governance and storage first, land all three ingestion paths into the same Iceberg table format, transform with catalog-aware writes (writeTo(...).using("iceberg"), not raw .parquet() paths) to unlock ACID and schema evolution, gate gold behind both a DQ check and Iceberg's own schema enforcement, apply Lake Formation permissions on the finished gold tables, and serve those exact tables from both Athena and Redshift with no load step between them. Step Functions and CloudWatch run it on a schedule; Iceberg time travel gives you a real rollback path when a run goes wrong.

40. AWS Lakehouse vs. Databricks Lakehouse: A Head-to-Head Comparison

Building the same lakehouse a second way — and a fair look at where each platform genuinely wins

The lakehouse architecture sections you just read solve one problem: keep one physical copy of the data, wrapped in a table format that gives every engine ACID transactions, schema evolution, and time travel over that same copy, instead of syncing separate copies into a lake and a warehouse. Databricks solves the exact same problem, with the same underlying ideas — but it draws the platform boundary in a different place. AWS composes many separate, best-of-breed services (S3, Glue Catalog, Lake Formation, Glue/EMR, Athena, Redshift) around a shared storage-and-catalog foundation, each independently priced and independently swappable. Databricks is one unified product — notebooks, job scheduling, a metastore, SQL warehouses, and ML tooling all under one roof — that itself runs on top of a cloud (AWS, Azure, or GCP) rather than being one. Neither is "the lakehouse" — they're two different ways of assembling the same idea, and the AWS-Native vs. Cross-Cloud Alternatives section already flagged Databricks briefly as one such alternative; this section is the dedicated deep-dive on exactly that question.

LayerAWS-native (this guide's build)Databricks equivalent

IngestionGlue Jobs, Lambda, DMS, Data FirehoseAuto Loader for files, plus Delta Live Tables (now part of Databricks Lakeflow) for declarative incremental pipelines

Delta Lake vs. Apache Iceberg — genuinely both good, genuinely not identical

Delta Lake and Apache Iceberg are both open table formats that add ACID transactions, schema evolution, and time travel on top of plain Parquet files — neither is "the real one" and the other a knockoff. Delta Lake originated at Databricks and, despite being open-sourced years ago, is still the format most deeply and natively integrated into the Databricks product itself. Iceberg has broader multi-engine, multi-vendor adoption, and it's what AWS has invested in natively — S3 Tables, Athena, Redshift, and EMR are all built to read and write Iceberg directly, with no translation layer. The gap is narrowing: Delta's UniForm feature can expose a Delta table's metadata as Iceberg (and vice versa), and Databricks itself now supports Unity Catalog acting as an Iceberg REST catalog, so tables built in one ecosystem are increasingly readable from the other without a full rewrite. That said, picking one on day one still meaningfully shapes which tools integrate cleanly out of the box — treat the formats as converging, not yet interchangeable.

Where AWS-native wins. Granular, pay-per-service pricing — you pay AWS for exactly the Glue, Athena, or Redshift compute a job actually consumes, with no platform license layered on top. The deepest possible integration with every other AWS service a team is already running — IAM, VPC, KMS, CloudWatch, and EventBridge are all first-class citizens, not bolted-on connectors. One vendor to manage and negotiate with, not two. And it's proven at genuinely enormous scale: see this guide's Enterprise Validation section's LSEG example.

Where Databricks wins. A meaningfully more unified, polished developer experience — one notebook environment spans ETL, SQL, and ML, instead of context-switching between Glue Studio, Athena's query editor, and SageMaker separately. Unity Catalog's governance model covers data and ML assets — models, features, the works — in one place, where Lake Formation's scope stops at data. Genuine multi-cloud portability: the same Databricks workspace concept runs on AWS, Azure, or GCP, which matters for a company avoiding single-cloud lock-in or already operating across clouds for unrelated reasons. And often a faster path to production for teams without deep AWS-specific expertise already on staff, since Databricks abstracts away more of the cloud plumbing (VPC networking, IAM role wiring) that a fully AWS-native build requires getting right explicitly.

The cost model is structurally different, not just a different price tag

AWS-native is "pay AWS for the compute and storage you use — there's no other bill." Databricks is "pay Databricks (in DBUs) on top of the underlying cloud infrastructure cost it runs on" — a real, structural extra layer, not a rounding error. That said, sticker price per hour isn't the whole picture: a unified platform can lower total cost of ownership for some teams by needing fewer specialized engineers and less glue-code integration work between separately-billed services. Whether that trade nets out in Databricks' favor depends entirely on team size, in-house AWS depth, and how many of those "separate services" a given team would actually need to wire together by hand anyway.

The trap: assuming Databricks means "no AWS bill." Running Databricks on AWS still means paying AWS for the underlying EC2 compute, EBS/S3 storage, and networking Databricks provisions — the DBU charge is layered on top of that infrastructure cost, not a replacement for it. Teams that only budget for the DBU line item are routinely surprised by the infrastructure bill sitting underneath it.

A pragmatic decision rule, not a universal answer. If the team is AWS-only already, has real in-house AWS expertise, and wants to minimize per-service cost with full control over every layer, AWS-native tends to win. If the team is multi-cloud, wants one unified environment spanning data engineering and data science/ML without needing deep cloud-plumbing knowledge on staff, or is already standardized on Databricks elsewhere in the org, Databricks tends to win. This mirrors the honest framing of the AWS-Native vs. Cross-Cloud Alternatives section: the right call depends on the team's actual constraints, not on inertia toward whichever platform is more familiar.

Exam angle
The exam is entirely AWS-native in scope — Databricks-specific product knowledge is not tested. You won't be asked how DBUs are priced or what a Databricks Workflow's retry syntax looks like. But being able to articulate why a real team might still choose Databricks — and to name the actual contrasts (Iceberg vs. Delta, Glue Data Catalog vs. Unity Catalog, Lake Formation vs. Unity Catalog's data-plus-ML scope) — is exactly the architectural judgment the exam's scenario questions probe, even when every answer choice is phrased in pure AWS terms. A question describing a multi-cloud constraint or a team that needs unified data-plus-ML governance is testing whether you recognize when Lake Formation and Glue Catalog are the wrong shape for the problem, not whether you know a competitor's pricing.

❯ Short version

Both platforms solve the same problem — one copy of data, ACID transactions, multi-engine access — but AWS composes many separately-priced, deeply-integrated services around S3 + Iceberg + Glue Catalog, while Databricks is one unified product (Unity Catalog, notebooks, SQL warehouses, ML) that itself runs on top of AWS, Azure, or GCP. Iceberg and Delta Lake are converging but not yet interchangeable. AWS-native wins on granular pricing, AWS-service integration, and single-vendor simplicity; Databricks wins on developer-experience unification, ML-plus-data governance, and multi-cloud portability — and its DBU cost sits on top of, not instead of, the underlying cloud bill.

AWS revised the DEA-C01 exam guide (to v1.1, effective December 2025) to merge its separate "knowledge" and "skills" statements into unified task statements, and to add newly in-scope services: Amazon Aurora, Amazon Q, Amazon Bedrock, Amazon Kendra, AWS Data Exchange, and Amazon S3 Tables — all of which this guide now covers in their own sections. The same revision also renamed Amazon QuickSight to Amazon Quick in AWS's own in-scope services list; if official AWS materials refer to "Amazon Quick," that's this guide's QuickSight section, not a different product.

The exam format itself, briefly

Like other AWS Associate-level exams, DEA-C01 is delivered as 65 scored/unscored questions in a 130-minute window, as multiple choice (one correct answer) and multiple response (select exactly N correct answers, clearly stated in the question) — almost every question is scenario-based: a paragraph describing a company's data, constraints, and requirement, followed by "which solution meets these requirements" rather than a bare definitional question. That format is exactly why this guide leans on worked examples and named exam-angle callouts throughout — recognizing which service a described scenario is pointing at is a bigger part of passing than recalling an isolated fact.

A practical way to use this guide in the final study stretch. Re-read Domain 1's sections first and most carefully (Glue, Kinesis/Firehose/MSK, Lambda, Step Functions/MWAA, EventBridge, DMS/AppFlow) — it's worth more than a third of the exam on its own. Then Domain 2 (S3, Athena, Redshift, Lake Formation, DynamoDB, RDS/Aurora, the Lakehouse Project sections). Domain 3 and 4 are each worth less individually, but skipping them is a mistake precisely because they're concentrated in a handful of this guide's sections (CloudWatch/governance for Domain 3; IAM/Networking/KMS/Secrets Manager for Domain 4) — a few focused re-reads cover a disproportionate share of two domains at once.

46. Learning Path: What to Build Next

This guide is the map — the services only become real once you've built something with them

Reading this guide gives the vocabulary and the mental model; it doesn't yet give the muscle memory. The fastest way to close that gap is to build a small, complete pipeline — end to end, not one service in isolation — and hit the real friction points (an IAM permission that's subtly wrong, a Glue job that's slow because of a partitioning mistake, a Step Functions execution that fails in a way the console's visual debugger makes obvious in seconds) firsthand.

A concrete first project, sized to be finishable

Pick any public API or dataset producing at least daily-changing data. Land raw pulls in S3 (Bronze) via a scheduled Lambda (sections 07, 22). Transform to cleaned, typed, deduplicated Parquet (Silver) with a Glue job (section 09). Add one automated data-quality check before promoting to a Gold aggregate table (sections 10, 37). Orchestrate the whole thing with Step Functions, including at least one retry policy and one SNS failure notification (sections 23, 37). Query the Gold table from Athena (section 13), and, if there's appetite, put a QuickSight dashboard on top (section 31). That's the worked example from sections 35-37, rebuilt with different source data — deliberately, since rebuilding a known-good shape with new data is a far faster way to internalize it than starting from a blank page.

❯ Short version

This guide's value compounds once it's paired with actually building something: a small, real, end-to-end pipeline — ingest, medallion transform, a DQ gate, Step Functions orchestration, a queryable Gold table — following the worked example's shape with different source data. That's where the AWS console's error messages, IAM's rough edges, and Step Functions' execution history actually teach the parts a document can't.
