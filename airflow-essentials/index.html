01. What Is Apache Airflow?

Workflows as code: an orchestrator that schedules, sequences, and watches over the jobs that do the real work.

Apache Airflow is an open-source platform for programmatically authoring, scheduling, and monitoring workflows. It started in 2014 inside Airbnb, where engineers needed a way to manage a growing pile of interdependent batch jobs, and it later became a top-level project of the Apache Software Foundation. The defining idea behind it is "workflows as code": instead of describing a pipeline through a drag-and-drop UI or a pile of shell scripts glued together with cron, you describe it as ordinary Python. That code defines a graph of tasks, their dependencies, and the rules under which they should run — and Airflow takes care of executing that graph reliably, retrying what fails, and recording what happened.

The problem Airflow solves is deceptively familiar to anyone who has operated a data platform: a nightly job depends on three upstream extracts, one of which sometimes runs long; a report must not start until the load finishes; a failure at 3 a.m. needs to page someone, not fail silently until a customer notices. Plain cron has no concept of "wait until job A succeeds" — it only knows "run at this time." Ad hoc scripts that call each other directly work until someone needs to see what ran, when, and why it failed, at which point there is no history to consult. Airflow exists to replace that fragility with a system that understands dependencies, retries, scheduling windows, and execution history as first-class citizens.

Orchestration

Coordinating a set of separate, otherwise-unaware tasks so they run in the correct order, at the correct time, with visibility into success or failure — without hard-coding that coordination logic inside the tasks themselves.

Plain example — a restaurant expediter never cooks a dish; they track which stations have finished, call out what's ready to plate, and flag anything running late. Airflow plays that same role for a fleet of computer programs.

What Airflow Is Not

It is easy to mistake Airflow for a data-processing engine because it sits so close to data pipelines. It is not. Airflow does not move rows, transform records, or hold data in memory for computation — it triggers and tracks the systems that do: a Spark cluster, a warehouse query engine, an S3 copy, a container that runs a training job. If a task in a DAG is slow or under-resourced, the fix is almost never "give Airflow more resources" — it's giving the underlying compute (Spark, the database, the container) more resources. Airflow's own footprint should stay lightweight: scheduling metadata, task state, and thin calls out to other systems.

Typical use cases follow directly from this division of labor: ETL/ELT orchestration (coordinating extracts, transforms, and loads across a warehouse), machine learning pipeline orchestration (feature generation, training, evaluation, and deployment steps that must run in strict order), infrastructure automation (provisioning, teardown, and scheduled maintenance jobs), and cross-system workflows that stitch together APIs, databases, and cloud services that otherwise have no shared scheduler.

A single missed dependency — a report built from a table that hasn't loaded yet — can quietly ship wrong numbers to a business user for days before anyone notices. An orchestrator that tracks dependency chains, retries transient failures automatically, and pages someone the moment a critical path breaks turns that silent risk into a visible, actionable one.

02. DAGs, Tasks &amp; Operators — Core Concepts

The three nouns you need before anything else in Airflow makes sense.

Every Airflow workflow is described in terms of three related but distinct concepts: the DAG (the whole workflow's shape), the task (one node in that shape), and the operator (the reusable template that gives a task its behavior). Getting these three straight — and, just as important, getting the difference between a task and a task instance straight — is the foundation everything else in this guide builds on.

DAG (Directed Acyclic Graph)

A collection of tasks connected by directional dependencies, with no cycles — there is never a path that loops back to a task already completed. In Airflow, a DAG is defined as Python code and represents one workflow: what to run, in what order, and under what scheduling conditions.

Plain example — extract must finish before transform, which must finish before load; nothing leads back from load to extract.

Task vs. Task Instance

A task is a definition — a node in the DAG, declared once in your Python file, describing one unit of work ("extract this table," "run this Spark job"). A task instance is one concrete execution of that task, tied to a specific DAG run and logical date. The same task definition produces a new task instance every time the DAG runs — one for today's schedule, one for yesterday's, one for a manual backfill run last week. The task is the recipe; the task instance is one specific time someone actually cooked it, with its own state (queued, running, success, failed, up_for_retry) and its own log file.

An operator is the reusable, pre-built template that gives a task its actual behavior — it's the class you instantiate to create a task. Airflow and its provider packages ship many: action operators that do something (PythonOperator runs a Python callable, BashOperator runs a shell command, EmailOperator sends an email), transfer operators that move data from one system to another (for example, an S3-to-Redshift or S3-to-database transfer operator shipped in a provider package), and sensor operators that wait for a condition to become true before letting the DAG proceed (covered in depth in Section 11). Most production DAGs mix all three categories.

with DAG(
    dag_id="daily_sales_report",
    start_date=datetime(2026, 1, 1),
    schedule="0 6 * * *",   # recent Airflow versions use `schedule`; older ones use `schedule_interval`
    catchup=False,
    tags=["sales", "reporting"],
) as dag:

        # business logic lives in plain Python
        print(f"Building report for logical date {context['ds']}")

    extract = BashOperator(
        task_id="extract_raw_sales",
        # {{ ds }} is Jinja templating -- Airflow fills it in with this run's
        # logical date (e.g. "2026-07-13") at execution time; see Section 07
        bash_command="extract_sales.sh {{ ds }}",
    )

    build_report = PythonOperator(
        task_id="build_report",
        python_callable=_build_report,
    )

    notify = BashOperator(
        task_id="notify_slack",
        bash_command="notify.sh 'sales report ready'",
    )

    # >> declares the dependency graph, not the order Python statements execute in
    extract >> build_report >> notify

A classic trap: confusing the task with the task instance. When someone says "clear this task," they almost always mean clearing one or more specific task instances — say, the failed runs for the past three days — not deleting the task's definition from the DAG file. The task definition lives in your Python code and changes only when you edit that code; task instances live in the metadata database and multiply with every scheduled or backfilled run.

03. Airflow Architecture: Scheduler, Webserver, Metadata DB, Executor &amp; Triggerer

A small set of cooperating processes built around one shared database — how the pieces actually talk to each other.

Airflow is not one program; it's a small set of cooperating processes that all revolve around a single relational database. Understanding which process does what — and, critically, which of them talk directly to each other versus which only communicate by reading and writing shared state — is the single most useful mental model for operating Airflow in production, debugging a stuck DAG, or reasoning about MWAA later in this guide.

The Scheduler is the heart of the system: it continuously parses DAG files, evaluates each DAG's schedule and each task's dependencies, decides which task instances are now eligible to run, and hands them off to the executor. Since Airflow 2.0's scheduler high-availability work, production deployments can run more than one scheduler process concurrently against the same metadata database for redundancy and throughput, with the database's row-level locking preventing two schedulers from double-dispatching the same task instance.

The Metadata Database — Postgres or MySQL in any real deployment, with SQLite supported only for local experimentation because it can't safely handle concurrent writes — is the single shared source of truth. DAG run state, task instance state, connections, variables, XComs, and pool slot usage all live there. Almost every other component's job description reduces to "read some of this, write some of that."

The Webserver serves the Airflow UI: DAG graphs, run history, logs, Gantt charts, connection and variable management. It is read-mostly against the metadata database — it doesn't schedule or execute anything itself, it just presents what the scheduler and executor have already recorded. The Executor is the component that decides how and where a ready task instance actually runs — locally, on a remote worker fleet, or in a freshly launched container — and is important enough to get its own section (04). The Triggerer, added to support deferrable operators and async-capable sensors, uses a single asyncio event loop to await many long-running conditions at once without occupying a worker slot for each one; it's covered in depth alongside sensors in Section 11.

Executor

The pluggable component that determines how and where a ready task instance runs — as a local subprocess, a job queued to a remote worker pool, or a freshly launched Kubernetes pod. The scheduler decides what should run and when; the executor decides where the work actually happens. Section 04 compares the main executor types.

Example — with LocalExecutor, a retried task becomes a new subprocess on the scheduler's own machine; with KubernetesExecutor, that same retry spins up an entirely new pod.

The DAG File Processor

Parsing every .py file in the DAGs folder into in-memory DAG objects is its own piece of work, distinct from scheduling decisions. On smaller deployments this parsing happens as a thread inside the scheduler process itself. On larger deployments, recent Airflow versions can run the DAG file processor as its own separate process, isolating a slow or buggy DAG file's parsing time from the scheduler's ability to keep making scheduling decisions for everything else.

The metadata database is the true point of coordination in Airflow. The webserver never talks to the scheduler directly; the triggerer never talks to the webserver directly. Almost everything is decoupled through shared, durable state in the database — the one deliberate exception being the scheduler's direct handoff of ready task instances to the executor's queue or API. This is why a healthy, appropriately-sized Postgres or MySQL instance matters as much to Airflow's reliability as the scheduler process itself: if the database is slow, every component that depends on it — which is nearly all of them — feels it.

04. Executors Compared: Sequential, Local, Celery &amp; Kubernetes

One setting that decides whether your DAGs run one at a time on a laptop or across a fleet of machines.

The executor is a single configuration choice — core.executor — but it shapes almost everything about how a deployment scales, fails, and costs money. Airflow ships several built-in executors, and picking the right one is really a decision about how much isolation and horizontal scale a workload needs versus how much operational complexity you're willing to run.

SequentialExecutor is the default out of the box and runs exactly one task at a time, with no parallelism whatsoever. It's paired by default with SQLite, which cannot safely handle concurrent writes — a fitting match, since SequentialExecutor has no concurrency to protect against in the first place. It exists purely so a freshly installed Airflow can run a first DAG without any setup. LocalExecutor is the smallest step up that's actually usable in production: it runs task instances as parallel subprocesses on the same machine as the scheduler, giving real concurrency on a single box, and requires a proper database (Postgres or MySQL) rather than SQLite. It's a solid fit for small-to-medium deployments that don't need to scale beyond one machine's CPU and memory.

CeleryExecutor distributes task instances to a pool of separate worker machines through a message broker — typically Redis or RabbitMQ. Workers pull tasks off the queue and report results back through the metadata database, which means you can add workers horizontally to absorb more load. This is, notably, the execution model AWS MWAA runs under the hood — a managed, auto-scaling Celery-based worker fleet on Fargate, which Section 15 covers in detail. KubernetesExecutor goes further on isolation: it launches a brand-new, ephemeral Kubernetes pod for every single task instance, giving each task its own container, its own resource requests/limits, and no leftover state from a previous task — at the cost of pod-startup latency on every task. A hybrid, CeleryKubernetesExecutor, lets you route specific tasks to Celery workers or to dedicated Kubernetes pods based on the queue name assigned to the task, useful when most tasks are lightweight but a few need heavier isolation.

Choosing an Executor

In practice the decision usually comes down to two questions: how much parallel capacity does the workload need beyond one machine, and how much does task-to-task isolation matter? A team running a modest number of DAGs on one well-sized server rarely needs more than LocalExecutor. A platform running hundreds of DAGs across many teams, with wildly varying resource needs per task, tends to gravitate toward KubernetesExecutor or the Celery/Kubernetes hybrid specifically for that resource isolation.

SequentialExecutor plus SQLite is fine for a five-minute local tutorial and actively dangerous in production: there is no real task concurrency, and SQLite does not safely support the concurrent writes that even a modest production scheduler generates. Deployments that accidentally stay on these defaults tend to discover the problem as silent task queuing delays or corrupted metadata, not a clear error message.

AWS MWAA removes this decision entirely: it runs a managed, auto-scaling Celery-based executor on AWS Fargate (serverless containers with no servers for you to patch — more in Section 15) behind the scenes, scaling worker capacity up and down based on queued task load without requiring you to provision or patch broker or worker infrastructure yourself. Section 15 covers how that managed architecture is put together.

05. Writing DAGs: TaskFlow API, Classic Operators &amp; Dependencies

Two syntaxes, one dependency graph underneath.

Every DAG, no matter how it is written, ultimately produces the same thing: a graph of task instances connected by upstream/downstream dependencies, stored in Airflow's metadata database and executed by the scheduler and executor described in Section 03. What differs is the Python surface you write against. Airflow has historically offered the classic operator style — instantiate an operator class, pass it a task_id and whatever arguments it needs, and wire dependencies explicitly. Airflow 2.0 added the TaskFlow API, a decorator-based layer (@dag, @task) that lets you write plain Python functions and have Airflow turn them into tasks automatically, inferring dependencies from how you call one function's result into another.

Both styles compile down to the same underlying DAG and BaseOperator objects — TaskFlow is sugar on top of PythonOperator, not a separate execution engine. That matters practically: you can freely mix TaskFlow-decorated tasks with classic operators like BashOperator or a provider's S3ToRedshiftOperator in the same DAG, and set dependencies between them with the same >> syntax either way.

Dependency (upstream / downstream)

A dependency declares that one task must reach a terminal state before another is eligible to run. a >> b makes a upstream of b (and b downstream of a); the equivalent explicit calls are a.set_downstream(b) / b.set_upstream(a).

extract >> transform >> load — load runs only after transform succeeds, which runs only after extract succeeds.

Classic operators

In the classic style you import an operator class, instantiate it with a unique task_id inside a with DAG(...) as dag: block, and chain the resulting objects with >>. Data moves between tasks only via explicit XCom calls (see Section 06) — there is no automatic wiring.

from airflow import DAG
from airflow.operators.python import PythonOperator
from datetime import datetime

def extract(**context):
    context["ti"].xcom_push(key="rows", value=1000)

def transform(**context):
    rows = context["ti"].xcom_pull(task_ids="extract", key="rows")
    context["ti"].xcom_push(key="rows", value=rows * 2)

def load(**context):
    rows = context["ti"].xcom_pull(task_ids="transform", key="rows")
    print(f"Loading {rows} rows")

with DAG(
    dag_id="etl_pipeline_classic",
    schedule="@daily",
    start_date=datetime(2026, 1, 1),
    catchup=False,
) as dag:
    extract_task = PythonOperator(task_id="extract", python_callable=extract)
    transform_task = PythonOperator(task_id="transform", python_callable=transform)
    load_task = PythonOperator(task_id="load", python_callable=load)

    extract_task >> transform_task >> load_task

TaskFlow API

The same pipeline in TaskFlow style: functions decorated with @task become tasks automatically when called inside a @dag-decorated function, and passing one function's return value directly to another wires both the dependency edge and the XCom push/pull for you — no manual xcom_push/xcom_pull required.

from airflow.decorators import dag, task
from datetime import datetime

@dag(
    dag_id="etl_pipeline_taskflow",
    schedule="@daily",
    start_date=datetime(2026, 1, 1),
    catchup=False,
)
def etl_pipeline():

    @task
    def extract() -> int:
        return 1000

    @task
    def transform(rows: int) -> int:
        return rows * 2

    @task
    def load(rows: int) -> None:
        print(f"Loading {rows} rows")

    load(transform(extract()))

More Ways to Define Dependencies

>>/<< and set_upstream()/set_downstream() cover the common cases, but three more patterns come up often enough to be worth knowing. Passing a list on either side of >> fans out or fans in without a loop: [extract_a, extract_b] >> transform makes both extracts upstream of transform in one line, exactly like the branching DAG diagrammed above. chain() wires a long, purely linear sequence — or several parallel sequences of equal length — more readably than a long string of >>: chain(a, b, c, d) is equivalent to a >> b >> c >> d. cross_downstream() handles the case a plain list can't: making every task in one group upstream of every task in another group, a full many-to-many fan-out rather than a shared single fan-in/fan-out point.

from airflow.models.baseoperator import chain, cross_downstream

# fan-in: both extracts must succeed before transform runs
[extract_a, extract_b] >> transform

# a long linear sequence, more readable than a >> b >> c >> d >> e
chain(extract, validate, transform, load, notify)

# every task in the first list becomes upstream of every task in the second --
# four edges created here, not just two
cross_downstream([extract_a, extract_b], [transform_a, transform_b])

Dependencies set this way work identically whether the tasks were created with classic operators or TaskFlow-decorated functions — chain() and list-based fan-out/fan-in accept either kind of task object interchangeably, and the same rules apply to tasks inside a Task Group from Section 09.

Templating with Jinja & the Airflow Context

Most operator parameters — bash_command, sql, a file path, an S3 key — are templated fields: Airflow treats their string value as a Jinja template and renders it at task-execution time, not at DAG-parse time. That's why {{ ds }} (used back in Section 02's example) works at all — it isn't Python string interpolation, it's Jinja substitution, filled in from a dictionary of runtime values Airflow calls the context. The context carries the logical date and its variants (ds, ds_nodash, prev_ds, next_ds), the running dag, dag_run, and task_instance (ti) objects, any params passed into the run, and a macros namespace with date-arithmetic helpers — roughly three dozen keys in total.

Classic operators and Jinja-templated fields get the context automatically at render time. Inside a plain Python callable, you access it either by accepting **kwargs and reading keys off it (the pattern used in Section 06's XCom examples, where context["ti"] pulls the task instance), or by calling get_current_context() from anywhere inside a running task, including nested helper functions. In the TaskFlow API, there's a third option: simply name a parameter after a context key (e.g. ds: str) and Airflow injects it automatically — no **kwargs, no explicit lookup.

Airflow context

The dictionary of runtime values — logical date, DAG run, task instance, params, and macros — that Airflow makes available to every task and every templated field, refilled fresh for each run.

Example: bash_command="load.sh {{ ds_nodash }}" renders to load.sh 20260713 for the run whose logical date is 2026-07-13 — the same template text produces a different command every single run, without the DAG author writing any date-formatting code.

from airflow.decorators import task
from airflow.operators.python import get_current_context

@task
def summarize_run(ds: str):              # TaskFlow auto-injects `ds` by parameter name
    print(f"Summarizing logical date {ds}")

@task
def log_task_instance():
    context = get_current_context()      # works anywhere inside a running task
    ti = context["ti"]
    print(f"Running as {ti.task_id}, attempt {ti.try_number}")

Two more knobs are worth knowing once the built-in keys stop being enough. user_defined_macros (a dict passed to the DAG constructor) adds your own names to the macros namespace, and user_defined_filters adds custom Jinja filters ({{ my_value | my_filter }}) — useful for a date format or a lookup your team reuses across many DAGs. And when writing a custom operator (see Section 10), declaring template_fields = ("sql", "bucket_key") as a class attribute tells Airflow which of that operator's __init__ parameters should be treated as Jinja templates and rendered at execution time — without it, a custom operator's parameters are taken as literal strings, and {{ ds }} would show up unrendered in your logs instead of a real date.

When a templated field renders to something unexpected, use airflow tasks render <dag_id> <task_id> <logical_date> (or the "Rendered Template" tab in the UI, next to the Log tab covered in Section 13) to see exactly what Jinja produced for that specific run — far faster than guessing from the DAG source alone.

06. XComs: Passing Data Between Tasks

A message-passing shortcut through the metadata database — built for pointers, not payloads.

XCom (short for "cross-communication") is Airflow's built-in mechanism for one task to hand a small piece of data to another. A task pushes a value — explicitly via ti.xcom_push(key, value), or implicitly by returning a value from its callable, which Airflow pushes under the default key return_value — and a downstream task pulls it, either explicitly via ti.xcom_pull(task_ids=..., key=...) or implicitly by receiving it as a function argument in the TaskFlow API. Under the hood, every pushed value is serialized (pickled or, more commonly today, JSON-encoded depending on the configured serializer/backend) and stored as a row in the metadata database, keyed by DAG, task, run, and XCom key.

That storage mechanism is exactly why XComs have a hard practical size ceiling. They are metadata-database rows, not a data bus. Passing a pandas DataFrame, a large JSON blob, or file contents through XCom means writing that entire payload into the database and rendering it later in the webserver's UI — which can bloat the database, slow down every query that touches the task_instance/xcom tables, and in bad cases make the "Rendered Task Instance" or Grid view painfully slow or unresponsive to load.

The correct pattern for anything beyond a small piece of metadata is: write the actual data to durable, shared storage in the producing task — an S3 object, a warehouse table, a data lake path — and pass only a reference to it (the S3 key, the table name, a row-count) through XCom. The consuming task reads the real data from that storage location using the reference.

XCom

A small, serialized value stored in the metadata database that one task instance pushes and another pulls, scoped to a specific DAG run.

extract writes a CSV to s3://lake/raw/2026-07-13/report.csv and pushes that key; process pulls the key and reads the file from S3 — the file itself never touches XCom.

from airflow.operators.python import PythonOperator

    print(f"Reading {file_path}")

07. Scheduling: Timetables, start_date, Catchup &amp; Backfill

The interval ends before the run begins — and that single fact explains most scheduling confusion.

start_date is not "when the DAG starts running" in wall-clock time — it is the earliest logical date Airflow will ever consider scheduling a run for. The scheduler compares it against the DAG's schedule (formerly schedule_interval) — a cron string, a timedelta, a preset like @daily/@hourly, or in recent Airflow versions a custom Timetable object for schedules that don't fit a fixed interval, such as "every weekday except public holidays" — to decide which intervals exist and which have already been satisfied.

The single most confusing mechanic in Airflow is that a DAG run for a given interval is triggered after that interval has fully elapsed, not at its start. Older Airflow docs and code called the interval's start execution_date, a name many people misread as "when this actually executes." Current terminology is clearer: every run has a data interval start and data interval end, and the run's logical date is conventionally the interval start — but the run itself is only queued once the interval end has passed.

Worked example: a DAG with schedule="@daily" has a run whose data interval is Monday 00:00 → Tuesday 00:00. That run is not triggered at Monday 00:00 — it is triggered at Tuesday 00:00, immediately after Monday's interval closes, because only then does Airflow know that "Monday" is complete and ready to process.

Cron Expressions, Timedeltas & Presets

Before any of the logical-date subtlety above matters, schedule itself needs a value, and the most common one is a five-field cron expression — the same syntax used by Unix cron, inherited wholesale rather than reinvented. Each field is separated by whitespace and constrains one unit of time; a field left as * matches every possible value for that unit.

Cron expression

Five whitespace-separated fields, in this fixed order: minute (0–59), hour (0–23), day of month (1–31), month (1–12), and day of week (0–6, where 0 is Sunday). A field matches every value it's compatible with — * means "any," a comma-separated list like 1,15 means "these specific values," a range like 1-5 means "this inclusive span," and a step like */15 means "every 15 units starting from the field's minimum."

Reading order: 0 6 * * * reads as "minute 0, hour 6, any day of month, any month, any day of week" — 6:00 AM, every single day.

A plain timedelta (e.g. schedule=timedelta(hours=6)) is simpler when the schedule is a fixed interval that doesn't need to land on clean clock boundaries — every 6 hours from whenever start_date falls, rather than specifically at 00:00/06:00/12:00/18:00. Logical date / data interval

The date-time window a DAG run represents. The run's logical date is the interval's start, but the run is not scheduled until the interval's end has passed.

For a @daily DAG, the run labeled 2026-07-13 covers [2026-07-13 00:00, 2026-07-14 00:00) and is triggered at 2026-07-14 00:00.

Catchup vs. backfill

catchup controls what happens when a DAG is deployed (or unpaused) with a start_date that lies in the past. When catchup=True — historically the default — the scheduler will create and run one DAG run for every missed interval between start_date and now, sequentially, as fast as available slots allow. Deploy a new DAG with start_date=datetime(2024, 1, 1) and a daily schedule today, and you'll get roughly 900+ backfill runs queued instantly. Setting catchup=False makes the DAG only run from the most recent completed interval forward.

Backfill is the deliberate counterpart: intentionally running a DAG for a specific past date range, typically via airflow dags backfill -s <start> -e <end> <dag_id> on the CLI, or by manually re-triggering specific past DAG runs from the UI. The difference is intent — catchup is an automatic side effect of scheduling logic; backfill is a manual, deliberate reprocessing action, often used after fixing a bug in a DAG's transformation logic.

Triggering DAG Runs Outside the Schedule

Everything above assumes the schedule itself decides when a DAG runs. Three other mechanisms start a DAG run on demand instead, and all three create the exact same kind of DAG run object the scheduler would have — just with run_type=manual instead of scheduled. Manual triggering is the simplest: the "Trigger DAG" button in the UI, or airflow dags trigger <dag_id> on the CLI, both start a run right now for the current logical date, optionally with a JSON conf payload the run's tasks can read out of the context. The REST API (POST /dags/{dag_id}/dagRuns) does the same thing programmatically, which is how an external system — a webhook, another orchestrator, a CI pipeline — can kick off an Airflow DAG without going near the UI or CLI. TriggerDagRunOperator is the third mechanism: a task, inside DAG A, that triggers a run of DAG B as part of DAG A's own execution — optionally waiting for DAG B to finish (wait_for_completion=True) before DAG A's downstream tasks proceed, and passing a conf dict to configure that specific triggered run.

Don't confuse "triggering a run" with trigger_rule from Section 08 — the words overlap but the concepts don't. Trigger_rule decides whether an already-scheduled task is allowed to start, based on its upstream tasks' states. Triggering (this section) decides whether a DAG run exists at all, independent of any schedule.

from airflow.operators.trigger_dagrun import TriggerDagRunOperator

    task_id="kick_off_report",

    conf={"source": "manual_reprocess"},   # available in the triggered run's context
    wait_for_completion=True,              # block this task until orders_report finishes
    poke_interval=30,
)

Use manual triggering or the REST API for one-off reprocessing and external-system integration; use TriggerDagRunOperator when one pipeline's structure genuinely depends on another DAG's run — it's a heavier-weight coupling than the Dataset-based scheduling covered in Section 12, so prefer a Dataset when you only care that data became available, and reach for TriggerDagRunOperator when you specifically need to control and wait on another DAG's execution.

08. Task Execution Semantics: Retries, Timeouts, Trigger Rules &amp; SLAs

Controlling not just whether a task runs, but how it fails, how long it waits, and what upstream state it demands.

Beyond the dependency graph itself, Airflow gives each task a set of execution controls that decide its resilience and its relationship to failure. retries sets how many times a failed task instance will be automatically re-attempted, paired with retry_delay (a fixed wait between attempts) and optionally retry_exponential_backoff=True (each retry waits longer than the last). execution_timeout is a hard ceiling — if the task instance runs longer than this duration, it is killed and marked failed, independent of whether it would have eventually succeeded.

sla looks similar to a timeout but behaves very differently: it defines an expected maximum duration measured from the start of the DAG run, and if a task is still running (or hasn't started) past that point, Airflow fires an SLA-miss event — logged, and optionally sent to an sla_miss_callback — but it does not stop, kill, or fail the task. The task keeps running to completion or failure on its own terms; SLA is purely an alerting mechanism, not an enforcement one. (Note: the classic sla/sla_miss_callback mechanism described here was removed from core in Airflow 3.0, pending a successor alerting mechanism — check the release notes for your version if you're on 3.0 or later.)

trigger_rule governs whether a downstream task is eligible to run based on the status of its upstream tasks, not merely whether they finished. The default, all_success, requires every upstream task to succeed. Other rules — all_done, one_success, all_failed, none_failed, one_failed — let you build patterns like a cleanup step that must run whichever way upstream tasks land, or a failure-notification task that should fire only when something actually broke.

Trigger rule

The condition, evaluated against upstream task instance states, that determines whether a task is eligible to run.

A notify_failure task set with trigger_rule="one_failed" runs only when at least one upstream task has failed — it stays skipped if everything upstream succeeds.

09. Dynamic Task Mapping &amp; Task Groups

Grouping tasks for humans vs. multiplying tasks for data — two different problems that both look like "lots of boxes"

As DAGs grow, you run into two unrelated needs that beginners often conflate. The first is organizational: a DAG with 40 tasks becomes unreadable in the graph view unless you can collapse related tasks into a single visual node. The second is structural: you frequently don't know how many tasks you need until the DAG is actually running — the number of files in a directory, the number of partitions in a table, the number of rows returned by a query. Section 02 introduced tasks and operators as fixed, hand-written units; this section covers the two mechanisms Airflow provides once a fixed, hand-written task list stops being enough.

Task Groups solve the first problem. A TaskGroup is purely a UI and organizational construct — it changes how tasks are drawn in the Graph view (collapsible into one node with a group_id prefix on each contained task's ID) but has zero effect on scheduling, parallelism, or execution semantics. The tasks inside still run exactly as their own dependencies dictate.

Dynamic Task Mapping, introduced in Airflow 2.3, solves the second problem. Instead of writing process_file_1, process_file_2, process_file_3... by hand, you write one mapped task definition and call .expand() on it; Airflow creates one task instance per element in the collection at runtime, after the upstream data is known. Each mapped instance gets its own log, its own retry behavior, and its own entry in the UI (shown as process_file[0], process_file[1], and so on).

Dynamic Task Mapping

A pattern where a task is expanded into N parallel task instances at runtime, where N is determined by the output of an upstream task rather than fixed at DAG-authoring time.

Example: a list_files task lists a directory and returns ["a.csv","b.csv","c.csv"]. A process_file task calls .expand(filename=list_files.output) — Airflow creates exactly three process_file task instances, one per filename, without the DAG author ever hard-coding the count.

Expand, expand_kwargs, and partial

The plain form, .expand(arg=upstream.output), maps a single keyword argument over a list. When a mapped task needs several parameters that vary together per instance — say a filename and a target bucket — use .expand_kwargs(), which takes a list of dictionaries and maps one dictionary per task instance instead of one scalar per argument. When some arguments should stay fixed across every mapped instance while others vary, call .partial() first to bind the fixed keyword arguments, then chain .expand() for the varying ones — e.g. process_file.partial(bucket="raw-data").expand(filename=files) holds bucket constant across every mapped instance.

# dynamic_mapping_example.py
from airflow.decorators import dag, task
from datetime import datetime

@dag(schedule=None, start_date=datetime(2024, 1, 1), catchup=False)
def process_incoming_files():

    @task
    def list_files() -> list[str]:
        # in practice: os.listdir(), an S3 ListObjects call, etc.
        return ["a.csv", "b.csv", "c.csv"]

    @task
    def process_file(filename: str) -> dict:
        # runs once per mapped instance: process_file[0], [1], [2]...
        return {"filename": filename, "rows": 1000}

    @task
    def summarize(results: list[dict]):
        total = sum(r["rows"] for r in results)
        print(f"processed {len(results)} files, {total} rows total")

    files = list_files()
    processed = process_file.expand(filename=files)
    summarize(processed)

10. Connections, Variables, Hooks &amp; Providers

How Airflow talks to everything that isn't Airflow

Airflow's core distribution is deliberately thin — it ships the scheduler, webserver, and DAG-authoring primitives, but almost none of the integrations needed to actually talk to S3, Snowflake, Postgres, or Slack. Four concepts work together to bridge that gap: Connections hold credentials, Hooks wrap a system's client library using those credentials, Operators package a hook into a ready-to-use task, and Providers are the installable packages that bundle all three for a given system.

A Connection stores everything needed to reach an external system: host, schema, login, password, port, and an arbitrary extra JSON blob for anything system-specific (region, role ARN, SSL mode, and so on). Connections are created through the Airflow UI (Admin → Connections), the CLI (airflow connections add), environment variables (AIRFLOW_CONN_<CONN_ID>), or — in production — a secrets backend. Code never references the credentials directly; it references a conn_id string, and Airflow resolves that string to the actual secret at task-execution time.

A Variable is a simpler key-value store for configuration that isn't a credential: an S3 bucket name, a feature flag, a list of table names. You read it with Variable.get("my_var") in Python or {{ var.value.my_var }} in a Jinja template. Be careful where you call Variable.get() — calling it at the top level of a DAG file (outside a task) runs on every single DAG-file parse cycle, which means every Variable lookup becomes a metadata-database query on every parse. A DAG file with a dozen top-level Variable.get() calls can measurably slow the whole scheduler's parsing loop; prefer reading variables inside task functions, or use the templated Jinja form which is deferred to task execution.

A Hook is the lower-level client wrapper: PostgresHook, S3Hook, SlackHook, and dozens more. A hook takes a conn_id, resolves the Connection, authenticates, and exposes Pythonic methods (hook.run(sql), hook.check_for_key(key, bucket_name)) instead of making you construct a boto3 or psycopg2 client by hand. Most built-in operators are themselves thin wrappers around a hook — and when no existing operator fits your exact logic, you can call a hook directly from inside a PythonOperator or @task function.

Building Custom Operators, Hooks & Sensors

When no existing operator, hook, or sensor — core or provider — fits a piece of logic your team repeats often, Airflow lets you write your own by subclassing the same base classes every built-in component is built from. A custom hook subclasses BaseHook and wraps a new system's client library the same way S3Hook wraps boto3. A custom operator subclasses BaseOperator, taking whatever parameters it needs in __init__ and doing the actual work in an execute(self, context) method — whatever execute() returns becomes that task's XCom return_value, exactly like a PythonOperator's callable. A custom sensor subclasses BaseSensorOperator and implements poke(self, context), returning True once its condition is met, and automatically inherits mode="poke"/"reschedule" from Section 11 for free.

Custom components earn their cost when the same logic needs to be reused, tested, and documented as one named building block across many DAGs or teams — a "load a file into our internal warehouse dialect" operator, say. For a one-off, a plain @task function calling an existing hook (as in the example above) is usually simpler and is not a step down in quality; don't build a custom operator class just to wrap a single call site. Once you do build one, package it as a plugin — a plugins.zip uploaded alongside your DAGs (the same mechanism Section 16 covers for MWAA), or as a proper installable Python package if it needs to be shared and versioned across more than one Airflow environment.

# custom_operators/reconciliation_operator.py
from airflow.models import BaseOperator
from airflow.providers.amazon.aws.hooks.s3 import S3Hook

class S3RowCountCheckOperator(BaseOperator):
    # custom operators are ordinary classes -- __init__ takes the parameters
    # this task needs, execute() does the work when the task actually runs
    def __init__(self, bucket_name, prefix, min_rows, aws_conn_id="aws_default", **kwargs):
        super().__init__(**kwargs)
        self.bucket_name = bucket_name
        self.prefix = prefix
        self.min_rows = min_rows
        self.aws_conn_id = aws_conn_id

    def execute(self, context):
        hook = S3Hook(aws_conn_id=self.aws_conn_id)
        keys = hook.list_keys(bucket_name=self.bucket_name, prefix=self.prefix) or []
        if len(keys) < self.min_rows:
            raise ValueError(f"Expected at least {self.min_rows} files, found {len(keys)}")
        return len(keys)   # becomes this task's XCom return_value

This is also how the provider packages from earlier in this section come to exist in the first place: apache-airflow-providers-amazon is, under the hood, a large, well-tested collection of exactly this kind of subclass — custom hooks and operators someone else already wrote, tested, and packaged so you don't have to.

Making a Custom Operator's Fields Templatable

The S3RowCountCheckOperator above takes plain, literal parameters — fine for min_rows, less useful for prefix if you want it to vary by logical date. Adding template_fields = ("prefix",) as a class attribute tells Airflow to run prefix through Jinja at execution time using the context from Section 05, the same as a built-in operator's bash_command or sql — so a DAG author can pass prefix="daily/{{ ds }}/" into your custom operator and have it resolve per run, without your operator's code needing to know anything about dates.

Once a custom operator, hook, or sensor is packaged, Airflow needs a way to discover it. A plugins.zip dropped into the plugins folder (or synced by MWAA, per Section 16) is picked up automatically at the next scheduler/webserver restart — fine for a single environment. A proper installable package instead registers itself via Python entry points, the same discovery mechanism every provider package uses, which is what lets it be versioned, published, and reused across more than one Airflow deployment without copying files around by hand.

11. Sensors &amp; Deferrable Operators — the Triggerer

Waiting for something to happen without paying for a worker slot the whole time

A huge share of real pipelines start with "wait until X exists" — a file lands, an upstream DAG finishes, an S3 key appears. Airflow's answer is the Sensor, a special operator subclass whose poke() method returns true once its condition is satisfied. Common examples include FileSensor (waits for a file on disk), ExternalTaskSensor (waits for a task in another DAG to reach a given state), and S3KeySensor (waits for an object to exist in S3). Downstream tasks simply depend on the sensor like any other task.

The naive way to wait is expensive. In the classic mode="poke" (the historical default), the sensor task occupies a full worker slot for its entire waiting period, checking the condition on a poke_interval and otherwise just sitting there idle. A sensor waiting three hours for a file ties up one worker slot for three hours, even though it's doing almost no actual work. mode="reschedule" improves this: between checks the sensor releases its worker slot entirely and reschedules itself for the next check, so it only occupies a slot for the brief moment it's actually poking — at the cost of a small scheduling gap between checks.

Deferrable operators, introduced in Airflow 2.2, go a step further: instead of occupying a worker slot at all — even briefly — a deferrable task suspends itself entirely and hands its waiting condition to the Triggerer process. The Triggerer runs a single asyncio event loop that can efficiently watch potentially thousands of deferred conditions concurrently, because each one is typically just an async poll against an API rather than a blocking call. Only once a condition is actually met does the Triggerer hand the task back to a worker to resume. Many provider sensors now ship deferrable versions — sometimes a separate class suffixed Async, sometimes the same class with a deferrable=True parameter in recent provider releases.

Poke mode vs. deferrable

Poke: a sensor waiting six hours for a file occupies one worker slot for the full six hours — if your Celery pool has 16 slots, that sensor alone is 1/16th of your entire execution capacity, doing nothing but sleeping and checking.

Deferrable: the same wait costs zero worker slots for those six hours — the condition lives as one lightweight watch inside the single Triggerer process, and a worker is only claimed for the few seconds needed to actually run the task once the file appears.

Poke-mode sensors are a classic way to silently exhaust a deployment. A handful of long-running poke-mode sensors scheduled at the same time can quietly consume every available worker slot, at which point no other task in any DAG can start — the deployment looks "stuck" with no obvious error, because nothing has actually failed. If you rely on sensors for long waits, default to reschedule mode or a deferrable equivalent rather than the poke default.

12. Datasets &amp; Data-Aware Scheduling

Triggering a DAG when data is actually ready, not when the clock says it probably is

Time-based scheduling, covered in Section 07, assumes you can predict when upstream data will be ready and pick a safe cron time after it. In practice that safety margin is a guess, and guesses cause two failure modes: run too early and you process incomplete or stale data, or pad the buffer so generously that downstream reports are needlessly delayed by hours. Datasets, introduced in Airflow 2.4, offer a different trigger: schedule a DAG to run when the data it depends on is actually updated, expressed as a first-class object rather than an assumed time offset.

A Dataset is identified by a URI string — conventionally something like s3://bucket/cleaned_orders/, though Airflow does not validate or resolve the URI itself; it's an opaque logical identifier. A producing task declares it updates a dataset via the outlets parameter. A consuming DAG sets its schedule (in place of a cron expression) to a list of one or more Dataset objects; Airflow triggers that DAG automatically once every listed dataset has been updated at least once since the consumer's last run. If a downstream DAG depends on three datasets produced by three different upstream DAGs, it waits for all three before running — no cron guessing required.

These dependencies are tracked centrally and are visible in a dedicated Datasets view in the Airflow UI, which shows which DAGs produce and which DAGs consume each dataset — useful for understanding cross-DAG data lineage that would otherwise be invisible outside each DAG's own code. For many cross-DAG dependency cases, this is a substantially lighter-weight alternative to polling with ExternalTaskSensor: no sensor task, no worker slot, and neither DAG needs to know the other's internal task IDs or DAG structure — only the shared dataset URI. Later Airflow releases (Airflow 3.0) broadened this idea into a more general "Asset" concept with additional scheduling capabilities; the core producer/consumer model described here is the foundation that generalization builds on, though you should check the current documentation for the full extent of what changed if you're on a newer release.

Data-aware scheduling (Dataset)

A scheduling mode where a DAG's schedule is a list of Dataset objects instead of a cron expression or timetable; the DAG is triggered once all listed datasets have been updated since its last run.

Example: a clean_orders DAG's final task declares outlets=[Dataset("s3://bucket/cleaned_orders/")]. A separate orders_report DAG sets schedule=[Dataset("s3://bucket/cleaned_orders/")] and runs automatically the moment the cleaning DAG finishes — with no cron time and no direct reference between the two DAGs' task IDs.

It also decouples the two DAGs structurally: the consumer never needs to know the producer's DAG ID or task IDs, only the shared dataset URI, so either DAG can be renamed, split, or restructured internally without breaking the other. 13. Monitoring, Logging, Alerting &amp; the Airflow UI

Reading the Grid and Graph views, finding logs when workers are ephemeral, and wiring failures to a pager.

Airflow's webserver ships with a UI built around two complementary views of the same information. The Grid view (the successor to the old "Tree view" in Airflow 2.x) lays DAG runs out as columns and tasks as rows, so each cell is one task instance's state at one point in run history — it is the fastest way to see "has this task been flaky over the last two weeks" at a glance. The Graph view instead shows the dependency structure of a single specific run, with each node colored by that task instance's current state — it answers "where exactly did today's run stop." Airflow 3.0 reworked the UI's look and navigation substantially, but the underlying concepts — a run-history matrix and a per-run dependency graph — carry over; check the version-specific docs for exact screen names if you're on a newer release.

Both views lean on the same small vocabulary of task instance states, described below. Depending on your Airflow version and executor you'll also find a Gantt view for timing, a Calendar view for spotting missed runs, and a Code view showing exactly what the scheduler parsed.

Task instance state

success — the task's callable ran to completion without raising.

running — a worker currently has the task checked out and is executing it right now.

failed — the task raised (or was killed) and has exhausted all configured retries.

upstream_failed — an upstream dependency failed or was skipped in a way that blocks this task, so it was never attempted.

skipped — a branch operator, trigger rule, or short-circuit decided this task should not run this time.

up_for_retry — the task failed but retries remain; it is waiting out retry_delay before trying again.

queued — the scheduler has handed the task to the executor, which hasn't yet started it on a worker.

Logging: local disk vs. remote logging

Every task instance attempt writes its own log — the task's stdout/stderr plus Airflow's own lifecycle lines (state transitions, retry decisions, exception tracebacks) — to a configured logging location. By default that location is local disk on whichever machine ran the task. That's fine for a single-machine SequentialExecutor setup, but it breaks down the moment workers are ephemeral: with CeleryExecutor autoscaling groups or KubernetesExecutor pods that are torn down right after the task finishes, "local disk" means the log lives on a filesystem that no longer exists by the time you go looking for it.

Production Airflow deployments configure remote logging instead: task logs are streamed to durable blob storage — Amazon S3, Google Cloud Storage, Azure Blob Storage, or similar — as the task runs, and the webserver fetches them from there when you open the Logs tab, regardless of which worker or pod originally produced them.

The remote-logging trap. Teams that move to KubernetesExecutor or scale-to-zero worker pools without configuring remote logging don't notice anything wrong at first — everything looks fine while tasks succeed. The gap shows up only when a task fails, the pod is already garbage-collected, and the Logs tab has nothing to show: the one log you actually need is the one that's already gone. Configure remote logging (an S3 or GCS bucket plus the matching logging connection) before you adopt ephemeral workers, not after the first unexplained outage.

Alerting: callbacks and built-in email hooks

Beyond staring at the UI, Airflow lets you attach arbitrary Python to lifecycle events via callbacks: on_failure_callback, on_success_callback, on_retry_callback, and sla_miss_callback can all be set on an individual task, or in a DAG's default_args so every task inherits them. They receive a context dictionary with the task instance, DAG, execution date, and exception (if any), which is enough to post a Slack message, open a PagerDuty incident, or write a custom metric. For simpler needs, the built-in email_on_failure and email_on_retry task parameters send plain email through a configured SMTP server — no custom code required, but also far less flexible than a callback.

14. Security &amp; Production Best Practices

RBAC, secrets backends, and the authoring habits that keep the scheduler fast and retries safe.

Running Airflow safely in production is two separate problems that get lumped together: securing who can see and do what, and writing DAGs that don't quietly damage the system that runs them. On the access-control side, Airflow's webserver has built-in RBAC (Role-Based Access Control), with default roles — Admin, Op, User, Viewer, Public — of increasing restriction, plus the ability to define custom roles scoped to specific DAGs (e.g. a role that can only view and trigger DAGs in one team's namespace). Credentials shouldn't live in the metadata database's Connections/Variables tables in the first place: a secrets backend — AWS Secrets Manager, HashiCorp Vault, GCP Secret Manager — lets Airflow resolve a connection or variable's value at runtime from an external vault instead, as covered in Section 10. At the network layer, restrict who can reach the webserver at all, and make sure the metadata database is not reachable from anywhere outside the Airflow components that need it.

The other half of "production-ready" has nothing to do with access control and everything to do with how DAGs are written.

Idempotency: the habit that makes retries safe

Airflow will re-run tasks — because of retries, because of a manual clear, because of a backfill — far more often than you plan for. A DAG is idempotent if running it twice for the same logical date leaves the system in the same state as running it once. Idempotency is what turns "just re-run it" from a risky guess into a safe, boring operation.

Idempotency

A DAG run for a given logical date produces the same end result no matter how many times, or in what order, it is re-executed.

Not idempotent: INSERT INTO sales SELECT * FROM staging WHERE ds = '{{ ds }}' — rerunning it duplicates every row.

Idempotent: DELETE FROM sales WHERE ds = '{{ ds }}'; INSERT INTO sales SELECT * FROM staging WHERE ds = '{{ ds }}' — or a native MERGE / INSERT ... ON CONFLICT DO UPDATE — so a rerun overwrites rather than appends.

Keep DAG files cheap and dumb

Top-level code in a DAG file — anything not inside a task function — runs every time the scheduler's file processor parses that file, which by design happens repeatedly, on a short interval, for every DAG file in the folder. An API call, a database query, or a slow computation sitting at the top level of one DAG file doesn't just slow down that DAG: it slows down every parse cycle, which competes for the same scheduler resources that every other DAG depends on to get scheduled. The fix is twofold: keep top-level code to cheap, static definitions (operator instantiation, dependency wiring), and push anything with real work — API calls, transformations, business rules — into task functions or external services, so the DAG file itself stays a thin, "dumb" description of structure. Test that structure the same way you'd test any code: unit-test task logic directly, use dag.test() or the DAG-level testing utilities to run a full DAG locally without a live scheduler, and add a CI check that simply imports/parses every DAG file — catching a broken DAG at review time is far cheaper than catching it when the scheduler silently stops loading it.

The DAG-parse-time trap. This is one of the most common causes of a slow, unresponsive scheduler in production — and one of the hardest for newcomers to see coming, because the offending code looks completely normal: a database lookup to build task lists dynamically, an API call to fetch a config, a Variable.get() call outside a task. Idempotency is the one habit that changes everything. Once every task in a DAG is safe to re-run for the same logical date, retries stop being scary, backfills stop being a weekend project you dread, and "just clear it and rerun" becomes the correct first response to almost any failure instead of a gamble.

15. AWS MWAA: Overview &amp; Architecture

Same Airflow you've been reading about, minus the pager for the infrastructure underneath it.

MWAA — Amazon Managed Workflows for Apache Airflow — is a fully managed AWS service that runs Apache Airflow on your behalf. Instead of standing up and operating the scheduler, webserver, metadata database, and executor yourself, you point MWAA at an S3 bucket containing your DAGs, and AWS provisions, patches, scales, and monitors the Airflow components for you, inside a VPC you designate. The trade you're making is explicit: you give up direct control over — and direct access to — the underlying infrastructure, in exchange for not having to run it.

The rest of this guide's MWAA-specific sections build on this one: Section 16 covers setting up and operating an environment day to day, and Section 17 covers integrating MWAA with the wider AWS ecosystem.

MWAA environment

One complete managed Airflow deployment: tied to exactly one VPC, one S3 bucket (for DAGs, plugins, and requirements), and one set of environment-size and worker-autoscaling settings. Everything Airflow-related — scheduler, workers, webserver, metadata database — lives inside this one environment.

What runs where

MWAA runs each Airflow component as a task on AWS Fargate — serverless containers, so there's no EC2 (AWS's rentable virtual machines) fleet for you to patch — inside private subnets of a VPC you provide. MWAA requires at least two private subnets across two Availability Zones, so an environment survives the loss of a single AZ. The metadata database is an Amazon Aurora (PostgreSQL-compatible) cluster that MWAA creates and manages for you; you cannot connect to it directly, run queries against it, or choose its engine version independently of what MWAA supports. The executor is a managed, Celery-based executor — MWAA does not expose executor choice the way self-managed Airflow does (compare Section 04's executor comparison); worker capacity autoscales up and down within limits you configure, based on task queue depth. Internally, MWAA uses Amazon SQS as the Celery broker between scheduler and workers — this is purely an implementation detail of the managed executor, not a resource you provision, view, or tune directly.

Your side of the contract is the S3 bucket: a dags/ prefix for DAG files, an optional plugins.zip for custom plugins/hooks, and a requirements.txt for extra Python packages. MWAA watches this bucket and syncs changes into the running environment automatically. Finally, the webserver can be exposed via a public endpoint reachable over the internet (behind AWS-managed auth), or, for tighter security postures, a private webserver access mode reachable only from within your VPC or a network connected to it (VPC peering, AWS Direct Connect, or VPN) — see Section 17 for the networking specifics.

16. Setting Up &amp; Operating an MWAA Environment

No servers to SSH into — just an S3 bucket, an IAM role, and a few dials AWS turns for you.

Running Airflow yourself means owning the scheduler process, the webserver process, the metadata database, and whatever executor and broker sit underneath. Running it on MWAA flips that model: AWS owns all of those internals, and what you actually configure is a much smaller, more declarative surface — an S3 bucket layout, an IAM role, a size tier, and a networking choice. Understanding exactly what that surface is (and isn't) is the difference between a smooth MWAA rollout and a confusing week of "why won't my DAG show up."

Every MWAA environment is anchored to a single S3 bucket, and that bucket's layout is not arbitrary — MWAA looks for specific prefixes and object names inside it. Versioning on the bucket is strongly recommended, since it gives you an easy rollback path if a bad DAG file or a broken requirements.txt gets uploaded. MWAA's internal DAG processor polls this bucket on a short interval, so deploying a new or updated DAG is as simple as an aws s3 cp or sync — there is no code to push to a server, no container to rebuild, no restart to trigger for a plain DAG file change.

Two other objects in that same bucket carry more operational weight than a DAG file: plugins.zip, a zip archive of any custom operators, hooks, or UI plugins your DAGs depend on, and requirements.txt, which pins the Python packages installed into the environment. Because both of these affect the Python environment the scheduler and workers run in — not just a single DAG's logic — changing either one triggers MWAA to rebuild and restart the affected components, which briefly interrupts scheduling. AWS publishes a "local-runner" Docker image that reproduces MWAA's constrained installation process, so you can validate a new requirements.txt locally before it ever touches a live environment.

Execution role (MWAA)

An IAM (Identity and Access Management — AWS's system for permissions) role that MWAA assumes on your behalf to read from and write to your S3 bucket, ship logs to CloudWatch, and reach any other AWS service your DAG code touches. An IAM role is a set of permissions a service can "wear" temporarily, rather than a fixed username/password. There is exactly one execution role per environment, and every permission any task in any DAG needs — not just S3 access — must be attached to it.

Example: a task using S3Hook.download_file() to pull a file from a data lake bucket needs s3:GetObject on that bucket in the execution role's policy. Without it, the DAG still parses fine and shows up in the UI — the failure only appears at task-run time, as an access-denied error in the task log, which catches teams off guard because nothing about it is visible at deploy time.

Beyond the bucket, an MWAA environment is sized by an environment class — a small handful of predefined size tiers (generally described as small, medium, and large rather than fixed forever, since AWS revises exact specs over time) that set the baseline capacity of the managed scheduler and webserver. Separately from the class, you configure minimum and maximum worker counts, which bound MWAA's autoscaling of the underlying worker fleet as task load rises and falls — you're tuning capacity, not choosing or operating an executor.

The last major setup decision is networking mode for the webserver. Public mode makes the Airflow UI reachable over the internet, still gated behind AWS IAM permissions and a generated login token mapped to an Airflow RBAC role — not a separate Airflow username/password database the way self-hosted Airflow's own login screen works; private mode restricts access to within your VPC or a network connected to it (Direct Connect, a VPN — virtual private network — or VPC peering). Private is the recommended default for anything handling regulated or sensitive data, since it removes an entire class of internet-facing exposure regardless of how strong Airflow's own auth is. Section 17 covers how private networking interacts with reaching other AWS services via VPC endpoints.

17. MWAA Integrations with the AWS Ecosystem

MWAA rarely moves data itself — it tells other AWS services when to move it.

A useful mental model for MWAA: it is the conductor, not the orchestra. Almost nothing inside a well-designed MWAA DAG does heavy data processing directly — instead, tasks call operators and hooks from apache-airflow-providers-amazon that reach out to purpose-built AWS services which do the actual work, while Airflow handles sequencing, retries, backfills, and alerting around them. This division of labor keeps the scheduler light and lets each service scale independently of the others.

S3 shows up twice: as the DAG-deployment bucket described in Section 16, and independently as a common data lake source or sink that DAGs read from and write to via S3Hook, S3KeySensor, or transfer operators like S3ToRedshiftOperator. Redshift is a frequent landing point for that data, loaded or queried through RedshiftDataOperator or a SQL-executing hook. AWS Glue is one of the most common Airflow-on-AWS patterns: a DAG uses GlueJobOperator to kick off a Glue ETL job (Glue's own Spark environment does the transform) and GlueCrawlerOperator to refresh a Data Catalog afterward — Airflow decides when, Glue does the what. Amazon EMR plays the same orchestrator/executor role for larger Spark or Hadoop jobs, via EmrCreateJobFlowOperator and EmrAddStepsOperator. AWS Lambda is the lightweight end of the spectrum — LambdaInvokeFunctionOperator lets a DAG treat a short-lived function call as a single task step, useful for small transforms or notifications that don't warrant a full cluster.

Two AWS services support the environment rather than the pipeline logic: AWS Secrets Manager, usable as a secrets backend for MWAA's connections and variables so credentials never sit in the metadata database in plain form (see Section 14), and Amazon CloudWatch, where MWAA ships scheduler, webserver, worker, and DAG-processing logs and metrics — it is the primary place to check an MWAA environment's health from outside the Airflow UI itself, and often the first place to look when a DAG that used to appear has silently stopped being parsed. SNS and SQS are common choices for pushing failure alerts out of Airflow or decoupling an event that should eventually trigger a DAG run, and Amazon SageMaker operators let a DAG kick off a training or inference job as one step in a larger pipeline.

"Why not just use Step Functions instead of MWAA?" It genuinely depends on the team and the shape of the workflow. Many real architectures use both — a Step Functions state machine handling one Glue-triggered piece, nested inside a larger multi-day pipeline that Airflow orchestrates end to end. Pick based on how AWS-centric the workflow is and what your team already knows well.

The pattern to remember. Across every integration above, MWAA is the conductor, not the orchestra: the actual data movement and compute happen in S3, Redshift, Glue, EMR, or SageMaker. Airflow's job is sequencing those steps correctly, retrying the ones that fail, and alerting a human when something needs attention.

∎ The takeaway

Airflow's entire value proposition comes down to one idea: workflows as code, coordinated by a scheduler that knows the full dependency graph, executed by workers that can be scaled independently of the scheduler itself. Everything else in this guide — TaskFlow, dynamic task mapping, deferrable operators, datasets — is refinement on top of that one idea, built to make DAGs easier to write correctly and cheaper to run at scale.

MWAA doesn't change any of that. It takes the same scheduler, the same executor model, the same metadata database, and runs them as a managed service inside your VPC — trading the operational work of patching and scaling Airflow yourself for the constraints of a managed environment (fixed environment classes, an S3-based deployment model, AWS-specific networking and IAM). Whether that trade is worth it depends on the same question every technology choice in data engineering comes down to: what does your team gain by not operating this themselves, and what do they give up in flexibility to get it.
