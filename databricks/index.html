01 · Foundations

What Is Databricks?

Databricks is a cloud platform built around one idea: you shouldn't need a separate data warehouse for BI and a separate data lake for AI. It unifies both into a single system — the Lakehouse — running on top of your own cloud storage.

Under the hood, Databricks runs a managed, optimized version of Apache Spark for large-scale data processing, on top of an open storage format called Delta Lake that adds database-like reliability to plain files in cloud storage. Around that core, Databricks adds a workspace (notebooks, jobs, dashboards), a governance layer (Unity Catalog), and specialized engines for SQL analytics (Databricks SQL) and machine learning (MLflow, and Mosaic AI — Databricks' suite for building and serving generative-AI and large-language-model applications).

Definition A Lakehouse is a data architecture that combines the low-cost, flexible storage of a data lake with the reliability, performance, and governance features traditionally found only in a data warehouse.

Why It Exists

Before the Lakehouse pattern, most companies ran two separate systems: a data warehouse for clean, structured, SQL-friendly reporting data, and a data lake for cheap, flexible storage of raw and unstructured data used by data scientists. Keeping both in sync meant duplicated data, duplicated pipelines, and data that drifted out of sync between the two. Databricks' core pitch is that one well-governed copy of the data, stored as open files in your own cloud account, can serve BI, data engineering, and AI/ML workloads at once.

Databricks vs. a Traditional Data Warehouse

Traditional Data WarehouseDatabricks Lakehouse
StorageProprietary, inside the warehouseOpen file formats (Parquet/Delta) in your own cloud storage
Data typesMostly structured, tabularStructured, semi-structured, unstructured, streaming
Primary workloadBI & reportingBI, data engineering, streaming, and ML/AI on one copy of data
ComputeTied to the warehouse engineElastic clusters and serverless SQL/compute, scaled independently of storage

What You Actually Touch, Day to Day

Even though the platform underneath is deep, day-to-day work happens in a small number of surfaces: a notebook or SQL editor for writing code, a cluster or SQL warehouse that actually executes it, a catalog browser for finding and governing tables, and a Workflows screen for scheduling it all. Almost everything you'll do reduces to the same three-line pattern, whatever the specific task:

# 1. Read a governed table by its three-level name
df = spark.read.table("restaurant_prod.silver.orders_clean")

# 2. Transform it with the DataFrame API (or an equivalent SQL query)
daily = df.groupBy("restaurant_id").sum("order_amount")

# 3. Write the result back as a new governed Delta table
daily.write.mode("overwrite").saveAsTable("restaurant_prod.gold.daily_totals")

Every concept in the rest of this guide is really just filling in the details around that pattern: what makes the read/write fast and safe (Delta Lake), where the table lives in the namespace (Unity Catalog), how the transformation runs at scale (Spark), and how the whole thing gets scheduled and monitored (Workflows). Keep this snippet in mind as an anchor — each new concept below attaches to one specific part of it.

The Lakehouse Architecture

Every Databricks concept in this guide fits into one layered picture. Once this diagram makes sense, the rest of the platform is just detail.

The Three Layers, in One Sentence Each

  • Storage: data lives as open files (Parquet, wrapped by Delta Lake) in your own cloud account — the actual bucket is Amazon S3, Azure Data Lake Storage (ADLS), or Google Cloud Storage (GCS) depending on which cloud you use, but Databricks works the same way on top of all three, and never locks data into a proprietary format.
  • Compute: Spark clusters (or serverless compute) read and process that data, scaling up or down independently of how much data you store.
  • Governance: Unity Catalog sits across everything, providing one place to control who can see and touch which data, and to trace where every table came from.

One Copy of Data, Read Three Different Ways

Because every layer in Fig. 1 sits on top of the same open Delta table, the exact same restaurant_prod.gold.daily_sales_by_restaurant table can be queried from whichever engine actually fits the task at hand — with no export step, no sync job, and no second copy that can go stale:

-- From the Databricks SQL editor, or a BI tool connected over JDBC/ODBC
SELECT * FROM restaurant_prod.gold.daily_sales_by_restaurant WHERE sales_date = current_date();

# From a Python notebook, as a distributed Spark DataFrame
df = spark.table("restaurant_prod.gold.daily_sales_by_restaurant")

# From that same notebook, as a local pandas DataFrame for a quick plot
pdf = spark.table("restaurant_prod.gold.daily_sales_by_restaurant").toPandas()

All three read the same underlying Parquet files, are checked against the same Unity Catalog permissions, and see the same transaction-log state — only the engine doing the reading changes. This is the concrete payoff of "one Lakehouse": a data scientist's notebook and an executive's dashboard are never looking at two different, slowly-diverging copies of "sales."

03 · Foundations

Workspace & Compute

The workspace is where people work day to day: notebooks, files, and jobs. Compute is what actually runs the code — and Databricks separates the two so compute can scale up, down, or to zero without losing any work.

Control Plane vs. Data Plane

Databricks runs as two separate halves. The control plane (managed entirely by Databricks) hosts the web UI, notebooks, job scheduler, and metadata. The data plane runs inside your cloud account — the actual clusters that touch your data never leave your network. This split is what lets Databricks manage the platform without ever hosting your raw data itself.

Cluster Types

All-Purpose Clusters

Interactive clusters attached to notebooks for exploration and development. Billed while running; usually shared by a team.

Job Clusters

Created automatically when a scheduled job runs, then torn down when it finishes. Cheaper for production pipelines since you only pay while the job executes.

SQL Warehouses

Compute purpose-built for SQL and BI queries, with fast startup and auto-scaling to handle bursts of dashboard traffic.

Serverless Compute

Databricks manages the underlying infrastructure entirely — no cluster sizing decisions, near-instant start, pay only for what you use.

Configuring Compute

Every cluster — all-purpose or job — boils down to a small set of choices: which Databricks Runtime version, what VM type for the driver and workers, and how many workers (a fixed count, or an autoscaling range). The same shape shows up whether you set it by hand in the UI or declare it as code, e.g. as part of a Workflow task:

new_cluster:
  spark_version: "15.4.x-scala2.12"
  node_type_id: "Standard_DS3_v2"
  autoscale:
    min_workers: 1
    max_workers: 4
  data_security_mode: "SINGLE_USER"

autoscale lets Spark add workers mid-job if a stage is running behind due to data volume, then scale back down once it finishes — useful when a job's input size varies run to run, at the cost of a little scale-up latency versus a fixed-size cluster. A cluster policy, set once by an admin, can restrict which of these fields a given user is even allowed to choose — e.g. capping the maximum node count or pinning a specific runtime version — so cost and compliance limits are enforced before a cluster ever starts, not caught after the fact.

Photon Photon is Databricks' native, vectorized query engine, written to execute Spark SQL and DataFrame operations dramatically faster than standard open-source Spark, while remaining fully compatible with existing Spark code. It's a checkbox on a cluster (or the default on SQL warehouses and serverless compute), not a different API to learn — the same spark.read.table(...) code simply runs faster underneath.
04 · Foundations

Apache Spark on Databricks

Spark is the distributed processing engine underneath almost everything Databricks does. You rarely manage Spark directly — but understanding its basic model makes every other concept click.

  • Distributed by design: a dataset is split into partitions, and each partition is processed in parallel across the machines in a cluster.
  • Lazy evaluation: transformations (filter, select, groupBy) build up a query plan but don't run anything; an action (display, write, count) triggers actual execution.
  • DataFrames: the primary API — a table-like structure with named, typed columns, usable from Python, SQL, Scala, or R with the same underlying engine.
  • Driver & executors: the driver plans the work and coordinates; executors on worker nodes actually run the tasks against their slice of the data.
# Transformations are lazy - nothing runs yet after these three lines
orders = spark.read.table("restaurant_prod.silver.orders_clean")
big_orders = orders.filter("order_amount > 50")
by_restaurant = big_orders.groupBy("restaurant_id").count()

# .explain() prints the query plan Spark has built, without running it
by_restaurant.explain()

# An action - .show(), .collect(), .write... - triggers actual execution
by_restaurant.show()

Between filter and groupBy, Spark is free to reorder, combine, or even skip work entirely, because nothing has executed yet — that flexibility is exactly what "lazy evaluation" buys you. The one operation that's usually the slow part in practice is a shuffle: whenever rows need to be regrouped across partitions — any groupBy, join, or orderBy — Spark has to physically move data between machines over the network, which costs far more than filtering or mapping a partition in place on one machine.

Why This Matters for Everything Else Every Delta Lake write, every Lakeflow pipeline, every Databricks SQL query, and every MLflow training run ultimately compiles down to a Spark job running on a cluster. Learning Spark's mental model (partitions, lazy plans, driver/executors, shuffles) explains why certain operations are fast, why some are slow, and why cluster sizing matters.

Delta Lake

Delta Lake is what turns a folder of plain Parquet files into something that behaves like a real database table — with transactions, history, and reliability guarantees.

What a Plain Data Lake Is Missing

Raw files in cloud storage have no concept of a transaction. If a job writing a table fails halfway through, readers can see a half-written, corrupted result. There's no way to see what a table looked like yesterday, and two jobs writing to the same table at once can corrupt each other's work. Delta Lake fixes all of this by adding a transaction log next to the data files.

What Delta Lake Adds

ACID Transactions

Writes either fully succeed or fully fail — concurrent readers never see a half-written table.

Time Travel

Every version of a table is queryable by version number or timestamp, enabling audits, rollbacks, and reproducible ML training data.

Schema Enforcement & Evolution

Bad writes that don't match the expected schema are rejected by default; controlled schema changes (like adding a column) are explicit and safe.

Upserts with MERGE

The MERGE INTO statement inserts, updates, or deletes rows in one atomic operation — the backbone of most incremental pipelines.

OPTIMIZE & Z-Ordering

Compacts many small files into larger ones and physically co-locates related data, so downstream queries scan far less data.

VACUUM

Permanently removes old, unreferenced data files once they're no longer needed for time travel, controlling storage cost.

In Practice: the Four Statements You'll Use Most

-- 1. Upsert: update matching rows, insert everything else, in one atomic write
MERGE INTO restaurant_prod.silver.orders_clean AS target
USING staged_updates AS source
ON target.order_id = source.order_id
WHEN MATCHED THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *;

-- 2. Time travel: query the table exactly as it looked at a past version or moment
SELECT * FROM restaurant_prod.gold.daily_sales_by_restaurant VERSION AS OF 12;
SELECT * FROM restaurant_prod.gold.daily_sales_by_restaurant TIMESTAMP AS OF '2026-07-10';

-- 3. OPTIMIZE + Z-ORDER: fewer, bigger files, physically sorted by the columns you filter on most
OPTIMIZE restaurant_prod.gold.daily_sales_by_restaurant ZORDER BY (restaurant_id);

-- 4. VACUUM: permanently delete files no longer needed once past versions age out
VACUUM restaurant_prod.gold.daily_sales_by_restaurant RETAIN 168 HOURS;

A useful mental model: MERGE is how data gets into a Delta table incrementally; time travel is how you look backward without extra backup tooling; OPTIMIZE/ZORDER and VACUUM are the two routine maintenance jobs that keep a table fast and cheap as it grows — in production these two are usually scheduled weekly rather than run by hand.

Doing This in the UI Instead of SQL Open any table in Catalog Explorer (left sidebar → Catalog icon → browse to the table) and its History tab lists every version with a timestamp, the operation that created it, and the row counts changed — the UI's read-only view of the same transaction log. The Sample Data tab lets you preview rows without writing a query, and the Details tab shows current size and file count, which is the number to watch before deciding a table needs an OPTIMIZE run.

The Medallion Architecture

Medallion architecture is the standard way to organize Delta tables inside a Lakehouse: data gets progressively cleaner and more trustworthy as it moves through three tiers.

TierContainsTypical Consumer
BronzeRaw, untouched data plus ingestion metadata (source file, load time)Data engineers debugging or replaying a pipeline
SilverCleaned, deduplicated, validated, and joined data at a granular levelAnalysts and data scientists exploring the business
GoldAggregated, business-level tables (daily sales by region, churn features)BI dashboards, executives, production ML models

A Minimal Bronze → Silver → Gold, End to End

-- Bronze: land raw data exactly as it arrived, plus load metadata
CREATE OR REPLACE TABLE main.demo.bronze_orders AS
SELECT *, current_timestamp() AS _ingested_at
FROM read_files('/Volumes/main/demo/landing/orders/', format => 'json');

-- Silver: clean, deduplicate, and enforce basic validity
CREATE OR REPLACE TABLE main.demo.silver_orders AS
SELECT DISTINCT order_id, customer_id, amount, order_ts
FROM main.demo.bronze_orders
WHERE amount > 0;

-- Gold: aggregate to the exact shape a dashboard or model needs
CREATE OR REPLACE TABLE main.demo.gold_daily_revenue AS
SELECT to_date(order_ts) AS sales_date, sum(amount) AS revenue
FROM main.demo.silver_orders
GROUP BY to_date(order_ts);

Nothing above is specific to any particular framework — three plain CREATE TABLE ... AS SELECT statements already form a valid, if very manual, medallion pipeline: each one reads the previous tier and writes the next. What Lakeflow Declarative Pipelines add on top, covered next, is turning these three separately-run statements into one declared, dependency-aware, incrementally-processed pipeline, instead of three tables you have to remember to re-run yourself, in the right order, every time.

Ingestion: Auto Loader & Lakeflow Connect

Getting data into Bronze reliably, incrementally, and without reprocessing everything every run is its own discipline. "Lakeflow" is Databricks' umbrella name for the whole ingestion-to-orchestration toolset, made of three pieces: Connect (ingestion), Declarative Pipelines (transformation), and Jobs (orchestration — the Workflows covered later in this guide).

Auto Loader

Auto Loader incrementally and efficiently processes new data files as they arrive in cloud storage, without you having to track which files have already been processed. It automatically infers and evolves schema, and scales from a handful of files to billions.

(spark.readStream.format("cloudFiles")
    .option("cloudFiles.format", "json")
    .option("cloudFiles.schemaLocation", "/Volumes/restaurant_prod/bronze/landing/_schema")
    .load("/Volumes/restaurant_prod/bronze/landing/incoming/")
    .writeStream
    .option("checkpointLocation", "/Volumes/restaurant_prod/bronze/landing/_checkpoints/files")
    .toTable("restaurant_prod.bronze.menu_photos_metadata")
)

The key is format("cloudFiles") — that's Auto Loader itself, not a raw file reader. It uses cloud-native file notification services under the hood (rather than repeatedly listing the whole directory) to detect new files cheaply even when a folder holds millions of them; cloudFiles.schemaLocation is where it persists the schema it has inferred so far, so a newly-added column in a later file is handled as a controlled schema evolution instead of a failure. Note this is a different ingestion path from Auto Loader's cousin used elsewhere in this guide's project: Lakeflow Connect (Step 2) is for tables inside a database, Auto Loader is for files landing in storage.

Lakeflow Connect (Managed Ingestion, Including CDC)

Auto Loader is great when data already lands as files, but a lot of real source data lives inside operational systems — a SQL database, Salesforce, ServiceNow — that were never designed to hand you files at all. Lakeflow Connect is Databricks' library of managed, largely no-code connectors for exactly that case. Point it at a source and it handles authentication, schema discovery, and incremental refresh for you. For databases specifically, Lakeflow Connect can ingest via Change Data Capture (CDC): instead of re-reading an entire table on every run, it reads the source database's own change log and replays only the inserts, updates, and deletes that actually happened since the last sync — landing an always-up-to-date mirror of that table directly into Bronze.

resources:
  pipelines:
    salesforce_accounts_cdc:
      catalog: main
      target: bronze
      ingestion_definition:
        connection_name: salesforce_prod_connection   # a Unity Catalog Connection, not a raw password
        objects:
          - table:
              source_schema: salesforce
              source_table: Account
              destination_table: accounts

That's the entire pipeline definition — there's no extract loop or pagination logic to write. connection_name is the one piece of indirection that matters: it points at a governed Connection object (created once, holding real credentials) so this file itself never contains a secret. The project later in this guide uses exactly this pattern against Azure SQL instead of Salesforce.

Lakeflow Declarative Pipelines

Lakeflow Declarative Pipelines (the underlying open-source engine is also called Spark Declarative Pipelines, and the product was previously known as Delta Live Tables) let you declare what a table should contain rather than writing the step-by-step logic of how to compute it. You define a chain of tables (Bronze → Silver → Gold) as declarative queries, attach data-quality expectations to each one, and the engine works out execution order, incremental processing, and retries automatically — for both batch and streaming data, using the same syntax.

import dlt
from pyspark.sql.functions import col

@dlt.table
def bronze_events():
    return (spark.readStream.format("cloudFiles")
                 .option("cloudFiles.format", "json")
                 .load("/Volumes/main/demo/landing/"))

@dlt.table
@dlt.expect_or_drop("valid_id", "event_id IS NOT NULL")
def silver_events():
    # bronze_events is defined ABOVE, in this same pipeline, so the short
    # dlt.read_stream("") form resolves it directly - no catalog path needed.
    return (dlt.read_stream("bronze_events")
               .select(col("event_id"), col("event_type"), col("event_ts").cast("timestamp")))

@dlt.table
def gold_event_counts():
    return dlt.read("silver_events").groupBy("event_type").count()

Three @dlt.table functions, one per medallion tier, chained together purely by referencing each other's plain table names — this is what "the engine resolves execution order" means concretely: nowhere in this file does anything say "run bronze_events before silver_events," yet the pipeline can only run them in that order, because silver_events literally can't produce output until dlt.read_stream("bronze_events") has something to read. Contrast this with the project later in this guide, where Bronze is written by a completely separate job outside the pipeline — in that case the code has to switch to spark.readStream.table("catalog.schema.table") instead, since dlt.read_stream only ever looks inside its own pipeline's dataset graph.

Batch and Streaming, Same Code A Lakeflow Declarative Pipeline table definition doesn't need to say whether it's batch or streaming — the same declarative query can run as a one-time backfill or as a continuously updating streaming table, which removes an entire category of duplicated pipeline logic.

Unity Catalog

Unity Catalog is the governance layer that sits above every table, file, model, and dashboard in a Databricks account, giving one consistent way to organize, secure, and audit all of it.

The Namespace Hierarchy

Unity Catalog organizes every asset into four nested levels, from broadest to most specific: a metastore (one per region, the top-level container), a catalog inside it (typically one per environment or business unit, like "restaurant_prod"), a schema inside that catalog (a grouping of related objects, like "gold"), and finally the objects themselves — tables, views, and even registered ML models, all addressed the same way. The diagram below traces that path top to bottom.

What Unity Catalog Gives You

Fine-Grained Access Control

Grant SELECT/MODIFY at the catalog, schema, table, row, or even column level with standard SQL GRANT statements.

Automatic Data Lineage

Every query is tracked, so you can see exactly which upstream tables and columns fed into any downstream table, dashboard, or model.

Unified Auditing

Every access to every table across every workspace is centrally logged for compliance and security review.

Secure Data Sharing

Share live tables with other Databricks accounts — or organizations without Databricks at all — without copying data.

In Practice

-- Create the namespace
CREATE CATALOG IF NOT EXISTS main;
CREATE SCHEMA IF NOT EXISTS main.demo;

-- Grant access with ordinary SQL - no separate "permissions language" to learn
GRANT USE CATALOG, USE SCHEMA ON SCHEMA main.demo TO `analysts`;
GRANT SELECT ON TABLE main.demo.gold_daily_revenue TO `analysts`;

-- Share a live table with another Databricks account, with no copy involved
CREATE SHARE partner_share;
ALTER SHARE partner_share ADD TABLE main.demo.gold_daily_revenue;

Every one of these is a plain SQL DDL statement — Unity Catalog doesn't introduce a separate governance language; it makes CATALOG and SHARE first-class objects that the same familiar CREATE/GRANT/ALTER verbs already know how to operate on. The one thing that has no SQL equivalent is the full lineage graph mentioned above — that's viewed in Catalog Explorer's Lineage tab on any table, rendered automatically from every query Databricks has ever run against it, rather than something you query for directly.

Data Sharing: Delta Sharing & Clean Rooms

Unity Catalog's CREATE SHARE statement from the previous section is the tip of a bigger capability: sharing governed data and AI assets — live, with no copies — both across your own Databricks accounts and with organizations that don't run Databricks at all.

Delta Sharing: An Open Protocol, Not a Databricks-Only Feature

Delta Sharing is an open-source protocol (not a proprietary API) for sharing a live view of a Delta table with a recipient. A provider (you) creates a SHARE containing one or more tables and adds recipients; a recipient reads that share either as another Databricks Unity Catalog metastore (open, governed access with no extra tooling) or, if they run no Databricks at all, via a small open-source Delta Sharing client — pandas, Spark, Power BI, and several other tools can all read a share directly. Because the recipient's own compute runs the query against the provider's storage, the provider never pays for the recipient's read workload, and the recipient always sees the current data — there's no export file that goes stale the moment the source table changes.

Ordinary table GRANTDelta Sharing
RecipientA user/group inside your own metastoreAnother Databricks account, or an organization with no Databricks at all
Access modelDirect SQL/DataFrame access to the tableA versioned, revocable SHARE object the recipient reads via Unity Catalog or an open client
Data movementNone — same metastoreNone — recipient reads live, no export/copy

Databricks Marketplace

Built on top of Delta Sharing, the Databricks Marketplace is a catalog of datasets, notebooks, and AI models that providers publish for other organizations to discover and subscribe to — the same share-without-copying mechanics, with a public discovery layer on top so a recipient doesn't need a direct relationship with the provider before requesting access.

Clean Rooms: Collaborating Without Exposing Raw Data

Delta Sharing answers "let this other party read my table." Clean Rooms answer a stricter question: "let this other party and I compute something together, without either of us ever seeing the other's raw rows." Each participant brings their own governed data into a secure, Unity-Catalog-governed environment; only pre-approved queries or notebooks can run inside it, and only the aggregated result — not either party's underlying data — ever leaves. This is the pattern behind joint marketing measurement between a retailer and an ad platform, or cross-institution analysis in healthcare and finance, where the whole point is that neither side can see the other's individual-level records.

Definition A Clean Room is a governed, multi-party compute environment where each participant's raw data stays inside their own account — only the outputs of jointly agreed-upon queries are shared, never the underlying rows.
Key Idea Delta Sharing, Marketplace, and Clean Rooms are three points on the same spectrum — "share a table," "publish a table for anyone to discover," and "compute jointly without sharing the table at all" — and all three are governed through the same Unity Catalog permissions as everything else in this guide, not a separate security system.

Databricks SQL & AI/BI

Databricks SQL is the platform's data warehouse experience: a SQL editor, dashboards, and alerts, running on the same Gold Delta tables as everything else — no separate warehouse or copy of data required.

  • SQL Warehouses: purpose-built, auto-scaling compute for interactive SQL and BI tools (Tableau, Power BI, Looker connect directly).
  • Dashboards: visualizations built directly on top of SQL queries against Gold tables, shareable across the organization.
  • Alerts: automatic notifications when a query result crosses a defined threshold.
  • AI/BI Genie: a natural-language interface that lets business users ask questions about governed data in plain English and get back a chart or number, grounded in the same Unity Catalog metadata and permissions as everything else.
-- A query a dashboard panel, or an alert, can be built directly from
SELECT restaurant_id, sum(total_sales) AS revenue
FROM restaurant_prod.gold.daily_sales_by_restaurant
WHERE sales_date >= current_date() - INTERVAL 7 DAYS
GROUP BY restaurant_id
ORDER BY revenue DESC;

Any BI tool that speaks standard SQL over JDBC/ODBC — Tableau, Power BI, Looker — can point straight at a SQL warehouse and run this exact query as-is; there's no Databricks-specific driver logic a report author needs to learn. Save the query above as a named query, then attach an alert to it in the UI — e.g. "notify #ops-channel if any restaurant's revenue drops more than 20% week over week" — with no extra code: the alert simply re-runs the same saved query on a schedule and compares the result against a threshold you set.

Key Idea Because BI queries run against the exact same Gold Delta tables used by data engineering and ML, there is no separate "reporting copy" of the data to keep in sync — one governed table, many consumers.

MLflow, Agent Bricks & Generative AI

Databricks gives you a spectrum of paths to "add AI to a pipeline," not just one. MLflow is for when you're training your own model. Mosaic AI Functions are for when a general-purpose foundation model can already do the job with zero training. And Agent Bricks, Vector Search, and Genie are for when the task needs a full production agent rather than one model call — picking the right point on that spectrum matters.

MLflow: Track and Register Your Own Models

Tracking

Every training run automatically logs its parameters, metrics, and output files, so experiments are comparable and reproducible.

Model Registry

A central, versioned store for trained models, with stages (staging, production) that control which version actually serves traffic.

Model Serving

Registered models can be deployed behind a REST endpoint with a few clicks, including large language models via Mosaic AI.

Feature Store

Reusable, governed feature tables (stored as Delta tables in Unity Catalog) shared consistently between training and real-time serving.

import mlflow

with mlflow.start_run():
    model = train_model(X_train, y_train)          # any scikit-learn / XGBoost / PyTorch model
    mlflow.log_param("max_depth", 6)
    mlflow.log_metric("accuracy", model.score(X_test, y_test))
    mlflow.sklearn.log_model(model, artifact_path="model",
                              registered_model_name="restaurant_prod.ml.demand_forecaster")

start_run() opens one tracked experiment run; every parameter and metric logged inside it becomes a row you can compare against every other run later, with none of that bookkeeping done by hand. Passing registered_model_name immediately versions the trained model into Unity Catalog's Model Registry — a governed object addressed the same catalog.schema.name way as a table — from which promoting a specific version to serve real traffic is a UI action (or one API call), not a redeploy of any code.

Mosaic AI Functions: No Training Required

Mosaic AI is Databricks' platform for building on foundation and generative-AI models. Its most immediately useful piece for a data engineer is AI Functions: built-in SQL functions that call a foundation model directly inside a query, right next to your data, with no separate model to train, deploy, or manage.

ai_analyze_sentiment()

Classifies a text column as positive, negative, neutral, or mixed — e.g. scoring customer review text row by row.

ai_classify()

Sorts free text into a set of categories you define, with no labeled training data needed.

ai_summarize()

Condenses long text (support tickets, reviews, transcripts) into a short summary.

ai_query()

Calls any Mosaic AI-served model — a foundation model or your own custom one — from plain SQL for cases the other functions don't cover.

Which One Do I Use? If the task is generic (sentiment, classification, summarization of everyday text), reach for an AI Function first — it's a single line of SQL with nothing to train or maintain. Reach for MLflow and a custom-trained model only when the task is specific to your business and a general-purpose model genuinely can't do it well enough (e.g. forecasting your specific demand pattern from your own historical data).

Agent Bricks: Building Production AI Agents

AI Functions are a single call; a real production assistant usually needs several calls chained together, tool use, and a way to keep quality and cost under control as volume grows. Agent Bricks is Databricks' framework for that next step: point it at your own data and a task description, and it automatically generates domain-specific synthetic evaluation data and task-aware benchmarks, then uses them to tune the agent's quality and cost automatically — rather than you hand-tuning prompts against a handful of examples.

Information Extraction Agent

Turns unstructured documents — emails, PDFs, reports — into structured fields (names, dates, line items).

Knowledge Assistant Agent

A grounded Q&A assistant over your own documents, built to reduce the vague or incorrect answers a plain chatbot gives.

Multi-Agent Supervisor

Coordinates several agents and tools — including Genie spaces and MCP-connected tools — as one system for multi-step tasks.

Native MCP Support

Model Context Protocol lets an agent safely call external APIs, databases, and SaaS tools through a standardized tool interface.

Every agent, tool, and model Agent Bricks touches is still governed through Unity Catalog — ownership, permissions, and lineage work the same way for an AI agent as they do for a table.

Mosaic AI Vector Search

Grounding an LLM's answers in your own documents — retrieval-augmented generation, or RAG — needs a way to find the handful of passages most relevant to a question out of potentially millions. Vector Search is a managed vector database, integrated with Unity Catalog, that automatically keeps its index in sync with a source Delta table — embed a text column once, and new or updated rows flow into the index without a separate pipeline to maintain.

CREATE VECTOR SEARCH ENDPOINT restaurant_reviews_endpoint;

CREATE VECTOR SEARCH INDEX restaurant_prod.gold.reviews_index
  ON TABLE restaurant_prod.gold.customer_reviews
  USING ENDPOINT restaurant_reviews_endpoint
  EMBEDDING_SOURCE_COLUMN review_text
  SYNC MODE TRIGGERED;

Once the index exists, an agent (or an ai_query() call) can search it for the reviews most similar to an incoming question, and pass those specific rows to the model as grounding context — instead of hoping the model already "knows" your data.

Genie Spaces as Agent Tools

The AI/BI Genie from the previous section isn't only a chat box for business users — a Genie space can itself be registered as a callable tool inside a Multi-Agent Supervisor, so an agent can delegate "what were last week's sales by region" to Genie's natural-language-to-SQL engine instead of trying to write that SQL itself. The BI layer and the agent layer share the same underlying governed data and the same permissions, rather than being two disconnected systems.

The Recommended Path Databricks' own guidance for adding AI is a progression, not a single tool: start with an AI Function for a quick answer; move to Agent Bricks once you need consistent quality and cost control at volume; add a Multi-Agent Supervisor when one task needs several tools or agents working together; and reach for a fully custom Agent Framework build only when a requirement genuinely exceeds what the packaged products above can do.

Lakebase: Managed Postgres for OLTP

Every engine covered so far — Spark, Delta Lake, Databricks SQL — is built for scanning and aggregating large volumes of data. None of them is built for what an application needs: reading or writing one specific row, in milliseconds, thousands of times a second. Lakebase is Databricks' answer — a fully managed, Postgres-compatible database for exactly that pattern, living inside the same governed platform instead of a disconnected database service you'd otherwise have to stand up yourself.

OLTP vs. OLAP, in One Line OLTP (what Lakebase is for) means many small, fast, single-row transactions — "update this one order's status." OLAP (what Spark/Delta/Databricks SQL are for) means scanning and aggregating huge volumes of history — "sum every order from the last year." Lakehouse tables are excellent at the second job and a poor fit for the first.

Why It's Not Just "Another Postgres"

Native Unity Catalog Governance

Lakebase tables are governed the same way as Delta tables — same permissions model, same audit trail, no second security system to maintain.

Bidirectional Sync with the Lakehouse

Sync a Gold Delta table into Lakebase for low-latency serving, and/or capture Lakebase's own changes back out as Delta tables for downstream analytics and audit.

Instant, Git-Like Branching

Copy-on-write branching creates an independent copy of a database in seconds — cheap, because unchanged data is shared with the parent rather than duplicated.

Autoscaling to Zero

Compute scales with load automatically within limits you set, and scales to zero — and back — when idle, so a dev branch sitting unused costs nothing.

What You'd Use It For

  • Backend database for a Databricks App — an interactive tool needs fast row lookups a Delta table can't give it.
  • Online feature store — serving a specific customer's features to a model in milliseconds at inference time.
  • State/memory store for an AI agent — conversation history and tool-call state, read and written constantly during a session.
  • Operational reporting — a small, frequently-hit summary table synced from Gold, without adding load to the SQL warehouse.
Know the Boundaries Lakebase is managed Postgres, not unrestricted Postgres: there's no host/OS access and no Postgres superuser (a databricks_superuser role covers routine administration instead), native Postgres logical replication in or out isn't available, and tablespaces aren't supported. Design around these the same way you'd design around any managed database service's constraints.

Workflows & Orchestration

Databricks Workflows schedules and orchestrates everything above: notebooks, Lakeflow pipelines, SQL queries, and ML training runs, as a single dependency-aware Job.

The diagram below reads left to right, then top to bottom: a schedule fires the Job, which runs a chain of tasks — Bronze, then Silver, then Gold — each waiting for the one before it to succeed. Once Gold is ready, two independent tasks (refreshing the dashboard and running a SQL/AI task) run at the same time, since neither depends on the other.

Task Types You Can Mix in One Job

Notebook / Python script

Runs a notebook or a plain .py file on a cluster — a new job cluster per task, or a shared cluster across tasks.

Pipeline

Triggers a Lakeflow Declarative Pipeline (or a Lakeflow Connect ingestion pipeline) and waits for it to finish.

SQL (File / Query / Dashboard / Alert)

Runs a saved .sql file or query, or refreshes a dashboard or alert — always against a SQL warehouse, never a cluster.

Run Job / For each / Condition

Calls another job as a sub-step, loops a task over a list of inputs, or branches based on a prior task's output — for building larger orchestration graphs.

Every depends_on edge is what turns a flat task list into the DAG shown above: a task with no depends_on starts as soon as the job is triggered; a task depending on two others waits for both to succeed (a fan-in, like run_silver_gold_pipeline in the project below); two tasks that depend on the same upstream task but not on each other run in parallel automatically — no extra configuration needed to get that parallelism.

resources:
  jobs:
    nightly_refresh:
      schedule:
        quartz_cron_expression: "0 0 2 * * ?"
        timezone_id: "UTC"
      tasks:
        - task_key: bronze
          notebook_task: { notebook_path: ./01_bronze.py }
        - task_key: silver
          depends_on: [{task_key: bronze}]
          pipeline_task: { pipeline_id: ${resources.pipelines.silver_gold.id} }
        - task_key: gold_alert
          depends_on: [{task_key: silver}]
          sql_task:
            alert: { alert_id: ${var.revenue_drop_alert_id} }
            warehouse_id: ${var.warehouse_id}

This is the same shape the restaurant project's own job uses later in this guide, genericized: a top-level schedule, a flat list of tasks, and depends_on edges connecting them into the DAG in Fig. 7. Whether a task is a notebook, a pipeline, or — as gold_alert shows here — refreshing a SQL alert instead of a dashboard, the dependency mechanics stay identical: name it with a unique task_key, point depends_on at whichever other task_keys must finish first.

Security Essentials

Most day-to-day security in Databricks flows through Unity Catalog, plus a handful of platform-level controls.

  • Identity & access: users and groups (often synced from an identity provider) are granted privileges on catalogs, schemas, and tables via standard SQL grants.
  • Secrets: credentials for external systems (API keys, database passwords) are stored in a managed secret scope and referenced in code, never hard-coded.
  • Network isolation: clusters can be restricted to run entirely inside a private network with no public IP exposure.
  • Row & column-level security: Unity Catalog can mask columns or filter rows dynamically based on who is querying, from the same table.
-- Create a secret scope and store a credential (CLI, since secrets have no dedicated UI page)
databricks secrets create-scope restaurant-scope
databricks secrets put-secret restaurant-scope eventhub-connection-string

# Read it back inside a notebook - the value never appears in code or logs
connection_string = dbutils.secrets.get("restaurant-scope", "eventhub-connection-string")

-- Column masking: analysts see a redacted phone number, engineers see it in full
CREATE FUNCTION restaurant_prod.gold.mask_phone(phone STRING)
RETURN CASE WHEN is_account_group_member('data-engineers')
            THEN phone ELSE '***-***-' || right(phone, 4) END;

ALTER TABLE restaurant_prod.gold.customers
ALTER COLUMN phone SET MASK restaurant_prod.gold.mask_phone;

The masking function above is ordinary SQL and an ordinary Unity Catalog grant — there is no separate "security product" bolted on top; the same GRANT/MASK mechanism used for a whole table works down to a single column, evaluated per query based on who's actually running it.

Set Up Unity Catalog Structure

Every object in the project needs a home in the Unity Catalog namespace before any data lands. Create one catalog for the project, one schema per medallion tier, and a managed volume for streaming checkpoints (the Unity-Catalog-governed replacement for old-style DBFS mount paths).

CREATE CATALOG IF NOT EXISTS restaurant_prod;

CREATE SCHEMA IF NOT EXISTS restaurant_prod.bronze;
CREATE SCHEMA IF NOT EXISTS restaurant_prod.silver;
CREATE SCHEMA IF NOT EXISTS restaurant_prod.gold;

CREATE VOLUME IF NOT EXISTS restaurant_prod.bronze.landing;

-- Grant the data engineering group write access to bronze/silver,
-- and read-only access to gold for the analytics team.
GRANT USE CATALOG ON CATALOG restaurant_prod TO `data-engineers`;
GRANT USE SCHEMA, CREATE TABLE, MODIFY, SELECT ON SCHEMA restaurant_prod.bronze TO `data-engineers`;
GRANT USE SCHEMA, CREATE TABLE, MODIFY, SELECT ON SCHEMA restaurant_prod.silver TO `data-engineers`;
GRANT USE SCHEMA, SELECT ON SCHEMA restaurant_prod.gold TO `analysts`;

Batch-Ingest Reference Data via Lakeflow Connect (CDC)

Restaurant and menu details live in an operational Azure SQL database, not in files — exactly the case Lakeflow Connect is built for. Rather than hand-writing extract logic, you declare an ingestion pipeline that points at the source tables; Databricks handles connecting, reading the change log, and keeping Bronze in sync incrementally from then on.

        connection_name: azure_sql_prod_connection

Stream-Ingest Orders from Azure Event Hub

Order events (including a customer review left after delivery, when one exists) arrive continuously on an Azure Event Hub topic. Event Hub exposes a Kafka-compatible endpoint, so Structured Streaming's built-in Kafka connector reads it directly — no separate connector needed.

from pyspark.sql.functions import current_timestamp, col

connection_string = dbutils.secrets.get("restaurant-scope", "eventhub-connection-string")

kafka_options = {
    "kafka.bootstrap.servers": "restaurant-orders-eh.servicebus.windows.net:9093",
    "kafka.sasl.mechanism": "PLAIN",
    "kafka.security.protocol": "SASL_SSL",
    "kafka.sasl.jaas.config":
        "kafkashaded.org.apache.kafka.common.security.plain.PlainLoginModule required "
        f'username="$ConnectionString" password="{connection_string}";',
    "subscribe": "orders",
    "startingOffsets": "earliest",
}

(spark.readStream.format("kafka").options(**kafka_options).load()
    .withColumn("_ingested_at", current_timestamp())
    .withColumn("json_value", col("value").cast("string"))
    .writeStream
    .option("checkpointLocation",
            "/Volumes/restaurant_prod/bronze/landing/_checkpoints/orders")
    .trigger(processingTime="30 seconds")
    .toTable("restaurant_prod.bronze.orders_raw")
)

The connection string never appears in the code itself — dbutils.secrets.get pulls it from a managed secret scope at runtime, exactly the pattern described in the Security Essentials section earlier. Clean, Join & Validate into Silver with Lakeflow

Silver is where the two Bronze sources actually meet: raw order JSON gets parsed and validated, then joined to the restaurant reference data that Lakeflow Connect landed in Step 2.

Why the Code Says "dlt" Lakeflow's declarative pipelines were originally shipped as a product called Delta Live Tables, and its Python library keeps that original name — every pipeline notebook starts with import dlt. The decorators below (@dlt.table, @dlt.expect_or_drop) are how a Lakeflow Declarative Pipeline is actually written, even though the product itself is now branded Lakeflow. 
import dlt
from pyspark.sql.functions import from_json, col

order_schema = ("order_id STRING, restaurant_id STRING, customer_id STRING, "
                 "quantity INT, order_amount DOUBLE, order_timestamp TIMESTAMP, "
                 "review_text STRING")

# The pipeline this file belongs to targets catalog "restaurant_prod",
# schema "silver" - so, as always in Lakeflow, only the plain table
# name is given to the decorator itself.
@dlt.table(name="orders_clean")
@dlt.expect_or_drop("valid_quantity", "quantity > 0")
@dlt.expect_or_drop("valid_amount", "order_amount > 0")
@dlt.expect("has_restaurant_id", "restaurant_id IS NOT NULL")
def orders_clean():
    orders = (
        dlt.read_stream("restaurant_prod.bronze.orders_raw")
           .select(from_json(col("json_value"), order_schema).alias("o"))
           .select("o.*")
           .dropDuplicates(["order_id"])
    )
    restaurants = dlt.read("restaurant_prod.bronze.restaurants")
    return orders.join(restaurants, "restaurant_id", "left")

The expect_or_drop rules silently remove rows that fail validation and record how many were dropped; expect alone just flags violations for monitoring without dropping the row. Joining against restaurants here — rather than in every downstream query — means every consumer of Silver already has restaurant names and details attached, without re-deriving that join themselves.

Implement This in the Databricks UI (covers Steps 4 & 5 together)

Aggregate into Gold

Gold tables answer specific business questions directly, so dashboards never have to re-derive the same aggregation logic. Here, daily sales are rolled up by restaurant.

from pyspark.sql.functions import to_date, sum, countDistinct

@dlt.table(name="daily_sales_by_restaurant")
def daily_sales_by_restaurant():
    return (
        dlt.read("restaurant_prod.silver.orders_clean")
           .groupBy("restaurant_id", "restaurant_name",
                     to_date("order_timestamp").alias("sales_date"))
           .agg(
               sum("order_amount").alias("total_sales"),
               sum("quantity").alias("total_units"),
               countDistinct("customer_id").alias("unique_customers")
           )
    )

Notice this table is defined with dlt.read (a full, materialized read) rather than dlt.read_stream — Gold aggregates are usually recomputed from all of Silver's history rather than processed incrementally row by row, since the aggregation itself needs the full picture.

Implement This in the Databricks UI This file is added as the second source-code entry in the same pipeline created for Step 4 (see the UI steps there) — there's no separate pipeline to create. Score Review Sentiment with Mosaic AI

Some orders in Silver carry a review_text field — free-text feedback a customer left after delivery. Rather than training a custom classifier, this project scores it with a built-in Mosaic AI Function, directly in SQL.

CREATE OR REPLACE TABLE restaurant_prod.gold.review_sentiment AS
SELECT
    order_id,
    restaurant_id,
    review_text,
    ai_analyze_sentiment(review_text) AS sentiment
FROM restaurant_prod.silver.orders_clean
WHERE review_text IS NOT NULL;

ai_analyze_sentiment calls a Databricks-hosted foundation model for every row and returns a label (positive, negative, neutral, or mixed) — no model to train, register, or serve. The result lands as an ordinary governed Gold table, review_sentiment, sitting right next to daily_sales_by_restaurant for the dashboard to query.

Build the Performance & Sentiment AI/BI Dashboard

With both Gold tables populated, a Databricks SQL dashboard is built directly on top of them — no extraction, no separate reporting database. One panel tracks sales performance, a second tracks sentiment.

-- Panel 1: sales performance
SELECT sales_date, restaurant_id, restaurant_name, total_sales, total_units, unique_customers
FROM restaurant_prod.gold.daily_sales_by_restaurant
WHERE sales_date >= current_date() - INTERVAL 30 DAYS

FROM restaurant_prod.gold.review_sentiment

  5. Drag a stacked-bar or pie widget bound to review_sentiment_mix (category = restaurant_id, value = review_count, color = sentiment).
  6. Click Publish, then copy the dashboard's ID from its URL or the Share dialog into the performance_dashboard_id variable used by Step 6 — or just pick it by name if the Workflow was built in the UI.
  7. Optional: sidebar → Genie → New Genie space, point it at the same restaurant_prod.gold schema, to add the natural-language Q&A layer mentioned above.
9

Govern, Monitor, and Close the Loop

With the pipeline running end to end, the last step is making sure it stays trustworthy and cheap to run.

  • Lineage check: in Unity Catalog's lineage graph, gold.review_sentiment and gold.daily_sales_by_restaurant should both trace back through silver.orders_clean to both Bronze sources — bronze.orders_raw (streaming) and bronze.restaurants (CDC) — confirming the whole chain is connected and auditable.
  • CDC & expectation monitoring: the Lakeflow Connect pipeline reports rows synced per run, and the Declarative Pipeline's event log reports how many rows were dropped by each expect_or_drop rule; a sudden spike in either signals a broken upstream source.
  • Cost control: the Workflow uses job clusters that terminate automatically after each run, and OPTIMIZE is scheduled weekly on the Silver and Gold tables to keep query performance stable as they grow.
  • Access review: Unity Catalog's audit log is periodically reviewed to confirm only the intended groups (data-engineers, analysts) have access to each schema.
Where to Click for Each Check
  1. Lineage: Catalog Explorer → open gold.review_sentiment or gold.daily_sales_by_restaurant → Lineage tab → the graph traces every upstream table and notebook automatically, with no manual tagging required.
  2. CDC & expectations: the pipeline's own page (Workflows → Pipelines) shows an Event log tab with rows processed/dropped per run and per expectation; the Lakeflow Connect pipeline's page shows rows synced per CDC run the same way.
  3. Cost: Compute → a job cluster's detail page shows its lifetime (created at run start, terminated at run end); schedule OPTIMIZE as its own weekly Workflow task, same task type as Step 7's SQL task.
  4. Access: Catalog Explorer → catalog/schema → Permissions tab shows every current grant at a glance; the workspace-level Audit log (under admin settings) shows historical query and access history for compliance review.
Project Complete Streaming orders and batch CDC reference data now flow, unattended, into a governed Gold layer, scored for sentiment by Mosaic AI and visualized on a live Performance & Sentiment dashboard — on a daily schedule, with quality checks, lineage, and access control enforced at every step.

Glossary

ACID
Atomicity, Consistency, Isolation, Durability — the transaction guarantees Delta Lake adds to plain files.
Agent Bricks
Databricks' framework for building production AI agents, using auto-generated synthetic data and benchmarks to tune an agent's quality and cost automatically.
AI Functions
Built-in Mosaic AI SQL functions (ai_analyze_sentiment, ai_classify, ai_summarize, ai_query) that call a foundation model directly in a query, with no model training or deployment required.
Auto Loader
A Databricks feature that incrementally and efficiently ingests new files from cloud storage as they arrive.
Bronze / Silver / Gold
The three tiers of the medallion architecture: raw, cleaned, and business-aggregated data.
CDC (Change Data Capture)
Reading a source database's own change log to replay only the inserts, updates, and deletes since the last sync, instead of re-reading the whole table.
Clean Room
A governed, multi-party compute environment where each participant's raw data stays in their own account — only jointly agreed-upon query outputs are shared.
Databricks Apps
A way to build and host interactive web applications directly inside a Databricks workspace on serverless compute, with inherited auth and governed data access.
Databricks Marketplace
A discovery catalog, built on Delta Sharing, where providers publish datasets, notebooks, and AI models for other organizations to subscribe to.
Delta Lake
An open table format that adds ACID transactions, time travel, and schema enforcement on top of Parquet files.
Delta Sharing
An open-source protocol for sharing a live view of a table with a recipient — another Databricks account or a non-Databricks client — with no data copy involved.
Lakebase
A fully managed, Postgres-compatible OLTP database built into Databricks, governed by Unity Catalog and able to sync bidirectionally with Delta tables.
Lakeflow
Databricks' umbrella name for its ingestion-to-orchestration toolset: Lakeflow Connect (ingestion), Lakeflow Declarative Pipelines (transformation), and Jobs (orchestration).
Lakeflow Connect
Databricks' managed, largely no-code connectors for ingesting from operational databases and SaaS applications, including via CDC.
Lakeflow Declarative Pipelines
Databricks' framework (Python module name dlt, formerly branded Delta Live Tables) for defining batch and streaming data pipelines as a chain of tables with built-in quality checks.
Lakehouse
An architecture combining data lake storage economics with data warehouse-like reliability and performance.
MCP (Model Context Protocol)
A standardized interface that lets an AI agent safely call external APIs, databases, and SaaS tools through a common tool-calling protocol.
MLflow
An open-source platform for tracking ML experiments, registering models, and deploying them to production.
Mosaic AI
Databricks' platform for building on foundation and generative-AI models, including AI Functions, Agent Bricks, Vector Search, and model serving.
OLTP / OLAP
OLTP (online transaction processing) means many small, fast, single-row operations, as in Lakebase; OLAP (online analytical processing) means scanning and aggregating large volumes of history, as in Spark/Delta/Databricks SQL.
Photon
Databricks' native, vectorized execution engine for accelerating Spark SQL and DataFrame workloads.
Time Travel
Querying a Delta table as it existed at a previous version or timestamp, made possible by the transaction log.
Unity Catalog
Databricks' unified governance layer for access control, lineage, and auditing across all data and AI assets.
Vector Search
A managed vector database, integrated with Unity Catalog, that keeps a search index synced with a source Delta table for retrieval-augmented generation (RAG).
Workflow / Job
A scheduled, dependency-aware set of tasks (notebooks, pipelines, SQL, ML) orchestrated by Databricks.
Z-Ordering
A technique that co-locates related data within Delta files to speed up queries that filter on the ordered columns.
