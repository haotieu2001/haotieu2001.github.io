OLTP vs OLAP, In Depth

Two database worlds, built for opposite jobs. OLTP systems answer thousands of tiny questions per second — "place this order," "fetch user 42." OLAP systems answer one enormous question at a time — "what was revenue by region, by month, across three years?" This piece opens both up to the metal: how each one stores data on disk, how a query actually runs through it, every trick used to make it fast, and why a design that's perfect for one is a disaster for the other.

~30 min read · Deep dive · Fundamentals

❯ The one idea (read this first)

A database can lay out a table on disk in one of two ways, and the choice decides everything. Row stores (OLTP — Online Transaction Processing) keep every field of a row glued together, so reading or changing one whole record is a single cheap operation — perfect for apps that constantly touch individual rows. Column stores (OLAP — Online Analytical Processing) keep every value of one column together, so scanning one column across millions of rows is cheap — perfect for reports and aggregations. Each layout makes its own job fast and the other job slow. That single physical decision — group by row, or group by column — is the root of every other difference in this article.

01. The split: transactions vs analytics

Every database is a trade-off. You cannot make one storage layout that is simultaneously the fastest at "change one row" and the fastest at "scan a billion rows." So the industry split into two families, each ruthlessly optimised for one kind of question.

The names are old and a little clunky, but they tell you the purpose:

  • OLTP — Online Transaction Processing. A transaction here means a small unit of business work: place an order, transfer money, update a profile, add an item to a cart. These touch a handful of rows, must finish in milliseconds, and happen constantly — thousands or millions per second. This is the database behind your app. Think PostgreSQL, MySQL, SQL Server, Oracle.
  • OLAP — Online Analytical Processing. An analytical query reads huge swaths of history to compute a summary: total revenue by region, average session length by month, the funnel conversion rate. Each query scans millions to billions of rows, runs for seconds to minutes, and there are relatively few of them. This is the database behind your dashboards and reports. Think BigQuery, Snowflake, Redshift, Spark, ClickHouse, Databricks.

They usually live side by side

This isn't "pick one." A real company runs both: the app writes to an OLTP database all day, and a pipeline (ETL/ELT — Extract, Transform, Load: software that copies and reshapes data) moves that data into an OLAP system where analysts run heavy reports — without slowing down the live app. The rest of this article explains why you can't just point your dashboards at the production database and call it a day.

02. OLTP — the world of single-row work

Picture the database behind an online store. Its whole day is a blizzard of tiny, precise operations, each touching one or a few specific rows:

sql — typical OLTP queries
-- fetch exactly one customer by their id
SELECT * FROM customers WHERE id = 8842;

-- add one new order
INSERT INTO orders (customer_id, total) VALUES (8842, 59.90);

-- update one row's stock count
UPDATE products SET stock = stock - 1 WHERE id = 177;

Notice the shape of these: each one names a specific row (usually by id), reads or writes all of its columns, and must return in a few milliseconds because a human is waiting. The defining traits of an OLTP workload:

  • Point access — find one row (or a small range) by key, not scan everything.
  • Whole-row access — when you read order #500, you typically want all its fields; when you insert, you write all fields at once.
  • Write-heavy & concurrent — thousands of users inserting and updating at the same instant, so the system needs strict correctness guarantees (this is where ACID transactions come from — the guarantee that each transaction is all-or-nothing and survives crashes).
  • Low latency, high throughput — each query is cheap, but there is a flood of them.

Every storage and execution decision an OLTP database makes is in service of those four traits. Let's see how.

03. How OLTP stores data on disk

To make "read or write one whole row" cheap, an OLTP database stores all of a row's columns physically next to each other. This is called row-oriented storage, or a row store. But it doesn't drop loose rows onto the disk — it packs them into fixed-size containers called pages.

The page: the unit of everything

Disks (and the operating system) move data in blocks, not bytes. So databases adopt a fixed page size — typically 8 KB (PostgreSQL) or 16 KB (MySQL/InnoDB) — as the smallest unit they ever read from or write to disk. A table is, on disk, just a long sequence of these pages. Each page holds a small header, a slot directory (pointers to where each row lives inside the page), and then the rows themselves packed in.

This is the key consequence: reading row 8843 means reading the one page it lives on, and that single read brings in all of its columns at once. For OLTP — where you want the whole record — that's ideal. One page read, whole row in hand.

Heap vs clustered: how pages are ordered

There are two common ways the pages of a table are organised:

  • Heap — pages are unordered; new rows go wherever there's free space. Fast to insert, but to find a specific row you'd have to scan every page… unless you have an index (next section). PostgreSQL tables are heaps.
  • Clustered index (index-organised table) — the rows themselves are stored in primary-key order, inside the leaves of a B-tree (the sorted tree structure we unpack in the very next section — for now, just "a structure that keeps rows findable in sorted order"). Looking up by primary key lands you directly on the row. MySQL/InnoDB stores tables this way.

The buffer pool: RAM is where the work happens

Disk is slow, RAM is fast. So an OLTP database keeps a big chunk of memory called the buffer pool (or buffer cache) that holds recently-used pages. Every read and write goes through it:

1
Need a page? If it's already in the buffer pool (a cache hit), use it — no disk touched. This is why a hot row served thousands of times per second never hits the disk.
2
If not (a cache miss), read the page from disk into the buffer pool, then use it. It now stays cached for next time.
3
Writes modify the page in memory and are made durable by first appending to a write-ahead log (WAL) — a sequential journal — so a crash can't lose committed data. The actual page is flushed to disk later, in the background.

Why the write-ahead log matters

Appending one line to the end of a log file is far faster than hunting down and rewriting a random page in the middle of a huge table. The WAL lets the database confirm a transaction as committed the instant the log entry is safely on disk, while the slower page updates happen lazily afterwards. It's the trick that makes durable writes fast — central to OLTP.

04. Indexes — the B-tree behind instant lookups

Row storage makes reading a known row cheap — but how does the database find the right row among millions without scanning every page? The answer, almost always, is a B-tree index. It is the single most important performance structure in OLTP, so it's worth understanding properly.

The problem an index solves

Without an index, WHERE id = 8842 forces a full table scan: read every page, check every row, until you find it. On a 10-million-row table that's hundreds of thousands of page reads for one answer. An index turns that into a handful of reads.

What a B-tree is

A B-tree is a balanced tree kept sorted by the indexed column. The top root node and middle branch nodes hold only signposts ("keys 1–1000 are this way, 1001–2000 that way"); the bottom leaf nodes hold the actual keys in sorted order, each pointing to the row's location on disk. Because the tree is balanced and very wide (each node fans out to hundreds of children), it stays only 3–4 levels deep even for billions of rows.

So a point lookup is: walk root → branch → leaf (a few reads, and the upper levels are almost always cached in the buffer pool), get the row's page address, read that one page. Four reads to find one row in ten million. That's the magic that makes OLTP feel instant.

An index also serves sorted ranges

Because leaves are kept in sorted order and linked to their neighbours, a B-tree is great for WHERE id BETWEEN 8000 AND 8100 or ORDER BY created_at LIMIT 10 too — find the start, then walk sideways along the leaves. The same structure powers equality, ranges, and sorting.

Indexes aren't free

Each index is a separate B-tree that must be kept in sync. Every INSERT/UPDATE/DELETE now also has to update every index on the table — so more indexes mean slower writes and more disk used. In a write-heavy OLTP system you index the columns you actually search on, and no more. It's a deliberate read-speed-vs-write-cost trade.

05. How an OLTP query runs, step by step

Let's trace one query — SELECT * FROM customers WHERE id = 8842 — from the text you type to the row you get back. Every relational database follows the same five stages.

1
Parse. The database checks your SQL is syntactically valid and converts the text into an internal tree of operations. A syntax error stops here.
2
Bind. It resolves every name against the catalog (the database's internal list of tables and columns) — confirming the customers table and id column exist, checking types and your permissions.
3
Plan (optimize). The query optimizer considers the possible ways to get the answer and picks the cheapest, using statistics it keeps about each table (row counts, how many distinct values a column has, data distribution). For id = 8842 it sees a B-tree on id and chooses an index seek over a full scan. This chosen plan is the query plan — you can see it yourself with EXPLAIN.
4
Execute. The engine runs the plan: traverse the B-tree to find the row's page address, fetch that page (from the buffer pool if cached, else disk), and extract the row.
5
Return. The row travels back to the client. For a point lookup the whole round trip is well under a millisecond once the page is cached.

The optimizer is a cost estimator

"Plan" sounds abstract but it's concrete: the optimizer assigns an estimated cost (roughly, how many pages it'll have to read) to each candidate strategy and picks the smallest. If its statistics are stale or wrong, it can pick a bad plan — which is why "run ANALYZE to refresh statistics" is a classic fix for a query that suddenly got slow.

06. Making OLTP queries fast

Given that storage layout and that execution path, here is the full toolbox an engineer reaches for to keep an OLTP database snappy — and why each one works.

TechniqueWhat it doesWhy it helps
IndexingBuild a B-tree on the columns you filter or join by.Turns full scans into O(log n) seeks — the #1 OLTP lever.
Composite indexesOne index over several columns, e.g. (customer_id, created_at).Serves multi-column filters and sorts in one structure, in the right order.
Covering indexIndex that includes all columns a query needs.The answer is read straight from the index — the table page is never touched ("index-only scan").
Clustering (clustered index)Physically store the table's rows in key order (§03), instead of a heap.Adjacent keys sit on adjacent pages, so range scans (BETWEEN, ORDER BY) read a few contiguous pages instead of jumping all over disk. Only one per table.
Bigger buffer poolGive the cache more RAM.More of the working set (the pages your queries actually touch) stays in memory → fewer disk reads.
Fresh statisticsKeep the optimizer's table stats up to date.Good stats → good plans. Stale stats cause bad plan choices.
Connection poolingReuse a fixed set of DB connections.Opening a connection is expensive; pooling avoids that cost under high concurrency.
PartitioningSplit one giant table into smaller chunks by a key (e.g. by month).Queries and maintenance touch only the relevant partition; smaller indexes per chunk.
DenormalizationStore some redundant data together to avoid joins.Fewer joins per request = less work on the hot path (a deliberate trade against write simplicity).
Read replicasCopies of the DB that serve reads only.Spread read load across machines; the primary focuses on writes.
ShardingSplit rows across multiple databases by a key (e.g. user_id ranges).Scales writes out past one machine — each shard owns a slice. The cost: cross-shard queries and transactions get hard, so it's a last resort.

The pattern behind all of them

Every OLTP optimization is doing one of two things: (a) read fewer pages to answer a query (indexes, covering indexes, partitioning, smaller rows), or (b) keep the pages you need in fast memory (bigger buffer pool, replicas, pooling). Hold that lens and the list stops being a list to memorise.

07. Why OLTP collapses under analytics

Here's the crux. The OLTP database is beautifully tuned for "touch a few known rows." Now point an analyst's question at it:

sql — an analytical query on an OLTP database
SELECT city, AVG(total)
FROM   orders                 -- 500 million rows
WHERE  created_at >= '2023-01-01'
GROUP BY city;

This query needs only three columns — city, total, created_at — but it needs them from hundreds of millions of rows. Watch what the row store is forced to do:

− It reads everything, to use almost nothing

Columns are glued together inside each row, inside each page. To get the total column from every row, the engine must read every page of the whole table — dragging along customer_name, shipping_address, notes and every other column it doesn't want. If the three columns it needs are 5% of each row, it still reads 100% of the bytes — 20× more disk I/O (the reads and writes between disk and memory) than the query actually needs.

− The index can't save it

Indexes shine at finding a few rows. This query wants most rows, so the optimizer correctly falls back to a full table scan — the index is useless here. There is no "few rows" to seek to.

− Poor compression

A page holds a jumble of different types — ints, strings, dates, all mixed. Because the page interleaves values from many different columns, there are almost no long runs of similar values for a compressor to latch onto — so a row store compresses far worse than the same data laid out one column at a time. The table stays large on disk, so the scan reads even more bytes.

− It fights the live app for resources

A scan of 500M rows floods the buffer pool with cold analytical pages, evicting the hot rows your customers' requests need. One heavy report can make the whole app slow. And a traditional single-node OLTP database scales mainly by scaling up (a bigger machine) plus read replicas — not by spreading one query across many machines — so there's a hard ceiling on CPU and RAM for that giant scan. (Distributed SQL systems like Spanner and CockroachDB do scale transactions out, but they trade away some of the simplicity that makes a single-node row store so fast.)

None of these are bugs. They're the direct, unavoidable consequence of a layout optimised for the opposite job. Row storage makes whole-row access cheap by making whole-column access expensive — and analytics is all whole-column access. That's the wall. The fix isn't to tune the row store harder; it's to flip the layout 90 degrees.

08. OLAP — the world of big scans

OLAP systems are built for exactly the query that broke the row store. Their workload is the mirror image of OLTP:

  • Few columns, many rows — read 3 columns out of 50, but across billions of rows.
  • Aggregation, not point lookup — SUM, AVG, COUNT, GROUP BY over huge ranges; almost never "fetch row #8842."
  • Read-mostly — data is loaded in big batches (or streamed in), then queried many times; few row-level updates.
  • Throughput over latency — a query running for 5 seconds over a billion rows is fine; what matters is scanning enormous data efficiently and in parallel.

To serve that, OLAP flips the two foundations we just studied: it stores data by column, and it spreads the work across many machines. Let's open both up.

09. How OLAP stores data (columnar & Parquet)

The core move is column-oriented storage: instead of keeping each row's fields together, keep each column's values together. All the city values in one run, all the total values in another, all the created_at values in another.

That one change unlocks three compounding wins:

1
Column pruning (projection pushdown). A query reads only the columns it names. Need 3 of 50 columns? Read ~6% of the data, not 100%. This directly erases the row store's biggest waste. (Pushdown, a word you'll see a lot below, just means pushing an operation — picking columns, applying a filter — down into the storage/scan layer so it happens as early as possible, on as little data as possible.)
2
Brilliant compression. A column holds values of one type, often very repetitive (the city column is the same few city names over and over). Same-type, repetitive data compresses enormously — often 5–10×. Less data on disk = less to read = faster scans and cheaper storage.
3
Vectorized execution. A tight, contiguous run of one column is perfect for the CPU to process in bulk — apply total > 100 to a whole batch of values at once (an approach called SIMD — Single Instruction, Multiple Data: one CPU instruction operating on many values together), instead of row-by-row. More on this in §10.

Parquet: a columnar file, opened up

In the modern data world the data usually lives in files (in cloud object storage like S3), and the dominant columnar file format is Apache Parquet. Understanding its layout makes every OLAP optimization obvious. A Parquet file is organised into a hierarchy:

  • Row group — a horizontal slice of the table (commonly ~128 MB / ~1M rows). Inside a row group, data is stored column-by-column.
  • Column chunk — all the values of one column, for the rows in that row group, stored contiguously and compressed.
  • Data pages — a column chunk is split into pages, each encoded — e.g. dictionary encoding, which replaces repeated values with small integer codes — then squeezed further with a general-purpose compressor (Snappy, Zstd). Encoding exploits the column's repetition; compression shrinks whatever's left.
  • Footer (metadata) — at the end of the file: the schema, and crucially the min and max value of each column chunk, plus null counts. This tiny summary is what powers data skipping.

The min/max stats are the secret weapon

Because the footer records, say, "in row group 7 the created_at column ranges from 2021-03 to 2021-04," a query filtering for created_at >= '2023-01-01' can look at that and skip the entire row group without reading it. These per-chunk min/max summaries are sometimes called zone maps, and reading-the-stats-to-skip-the-data is the heart of OLAP speed.

Partitioning: skipping at the folder level

One level up from the file, OLAP datasets are usually partitioned — physically split into folders by a column, most often a date:

object storage layout
s3://lake/orders/
  date=2023-01-01/  part-0001.parquet  part-0002.parquet
  date=2023-01-02/  part-0001.parquet  …
  date=2023-01-03/  …

Now WHERE date = '2023-01-02' reads only that one folder and ignores the rest entirely — the engine never even lists the other files. This is partition pruning, and it's the cheapest, biggest win in analytics: skip terabytes by reading a folder name.

10. How an OLAP query runs, step by step

Now the payoff. Let's run the same query that broke the row store — SELECT city, AVG(total) FROM orders WHERE created_at >= '2023-01-01' GROUP BY city — but on a columnar, distributed OLAP engine. The same parse → bind → plan front-end from §05 still runs first; what's different — and what makes analytics fast — is the execute stage. It works like a funnel: each stage throws away data so the next stage does less.

1
Plan & partition pruning. The optimizer reads the query, sees the created_at / date filter, and immediately eliminates every date-partition folder that can't match — those files are never opened.
2
Data skipping. For the surviving files, it reads each Parquet footer and uses the per-row-group min/max stats to drop any row group whose range can't contain matching rows. More data gone, unread.
3
Column pruning (projection pushdown). It reads only the city, total, and created_at column chunks from the remaining row groups — the other 47 columns are skipped on disk.
4
Predicate pushdown. The created_at >= '2023-01-01' filter is applied at the scan, as data is decoded — so rows that fail are discarded before any later operator (the aggregation) ever sees them.
5
Vectorized execution. Surviving column data is processed in batches (e.g. 1,024 values at a time) rather than row by row. Working a batch at a time avoids the per-row bookkeeping of the old one-row engine and lets the CPU stay cache-friendly and use SIMD — together, several times faster.
6
Distributed partial aggregation. The data is spread across many worker nodes (this is MPP — Massively Parallel Processing). Each node computes a partial result over its slice — its own running sum and count of total per city.
7
Shuffle & final merge. Partial results are shuffled so all partials for the same city meet on one node, which combines them into the final average. The handful of result rows go back to the client.

Compare the two paths

The row store read 100% of the bytes on one machine. The column store skipped whole folders and row groups, read 3 columns out of 50, filtered at the scan, processed in vectorized batches, and split what remained across many nodes. Same SQL, same answer — but orders of magnitude less I/O and far more parallelism. That is why analytics belongs on OLAP.

11. Making OLAP queries fast

Here's the full optimization toolbox for analytics. Notice the theme is no longer "find the row fast" — it's "read as little data as possible, and process what's left in parallel." The first rows gather the mechanics you just saw in §09–§10 into one reference; the lower rows are extra levers we haven't met yet (compaction, materialized views, caching, bloom filters).

TechniqueWhat it doesWhy it helps
Columnar storageStore each column contiguously (Parquet/ORC).The foundation — enables column pruning, compression, and vectorization.
Partition pruningSplit data into folders by a key; skip folders the filter excludes.Eliminates the most data for the least effort — reads a folder name, skips terabytes.
Data skipping (zone maps)Use per-file/row-group min/max stats to skip chunks that can't match.Avoids reading data even within the files you do open.
Projection pushdownRead only the columns the query references.3 of 50 columns = ~6% of the bytes. The core columnar win.
Predicate pushdownApply WHERE filters at the scan, not after.Discards rows before they reach joins/aggregations — less work downstream.

Right-sized files / compactionMerge many tiny files into fewer ~128MB–1GB files.Avoids the "small files problem" — per-file overhead and metadata thrash.

CachingCache results and hot data (result cache, local SSD cache).Repeated dashboard queries skip the scan entirely.
Bloom filtersCompact per-file "is this value possibly here?" filters.Skip files for equality filters on high-cardinality columns — ones with very many distinct values, like an email or UUID — where the min/max range is too wide to rule anything out.

The mirror image of OLTP

OLTP optimization = "read fewer pages to find a few rows" (indexes, caching). OLAP optimization = "read fewer bytes of a giant scan, then parallelize the rest" (pruning, skipping, columnar compression, MPP). Same goal — do less I/O — but pursued from opposite directions because the workloads are opposite.

12. Side-by-side & when to use which

Everything in this article reduces to one table. Read it top to bottom and notice how the workload (row 1) forces every other choice.

OLTPOLAP
Typical queryTouch a few rows by keyScan/aggregate millions of rows, few columns
Storage layoutRow-oriented (fields of a row together)Column-oriented (values of a column together)
On disk8–16 KB pages, heap or B-treeParquet/ORC files, row groups, partitions
Find data viaB-tree index seekPartition pruning + min/max data skipping
Reads…Whole rows (all columns)Only the columns & row groups needed
WorkloadWrite-heavy, high concurrencyRead-mostly, bulk loads
Scales byMostly up (bigger box) + read replicasScaling out (MPP, many nodes)
LatencySub-millisecond per querySeconds–minutes per query
Optimised forMany tiny operations, fastFew huge scans, efficient
ExamplesPostgreSQL, MySQL, SQL Server, OracleBigQuery, Snowflake, Redshift, Spark, ClickHouse

So which do you use?

  • Powering an application — user accounts, orders, inventory, anything that reads/writes individual records live? OLTP.
  • Powering reports, dashboards, BI (business-intelligence reporting), ML features — anything that crunches large history into summaries? OLAP.
  • Both, which is the norm: run OLTP for the app, copy data into OLAP for analytics, and keep heavy reporting off the production database. The pipeline that moves data between them is the daily work of data engineering.

The line is blurring (HTAP & the lakehouse)

Newer systems try to do both at once — HTAP (Hybrid Transactional/Analytical Processing) databases keep a row store for writes and a column store for reads in sync; lakehouse formats like Delta Lake and Iceberg add transactions and updates on top of columnar files. The two worlds are converging — but the underlying trade-off between row and column layout is physics, and it never fully goes away. Knowing which side of it you're on is the whole game.

That's both worlds, end to end: how a row store packs whole rows into pages and finds them with B-trees; how to tune it, and why it hits a wall under analytics; then how a column store splits data into Parquet files it can skip, read column-by-column, and crunch across many nodes. The through-line worth remembering: row layout makes record access cheap, column layout makes scan access cheap — and that one physical choice decides everything else.

OLTP
Online Transaction Processing — databases for many small, live read/write operations on individual rows (the app's database).
OLAP
Online Analytical Processing — databases for large scans and aggregations over history (the reporting/analytics database).
ACID
The correctness guarantees of a transaction: Atomic (all-or-nothing), Consistent, Isolated, Durable (survives crashes).
ETL / ELT
Extract, Transform, Load — the pipeline that copies and reshapes data from OLTP into OLAP (ELT just reorders the last two steps).
I/O
Input/Output — the reads and writes that move data between disk and memory; the main cost a database tries to minimise.
Row store
Storage that keeps all of a row's columns physically together. Great for whole-row access.
Column store
Storage that keeps all of one column's values together. Great for scanning few columns over many rows.
Page / block
The fixed-size unit (e.g. 8–16 KB) a database reads from and writes to disk. A table is a sequence of pages.
Heap
A table whose pages are unordered; rows go wherever there's space.
B-tree
A balanced, sorted tree index that locates a row in O(log n) — a few hops instead of a full scan.
Clustered index
A table physically stored in primary-key order inside a B-tree's leaves.
Buffer pool
The in-RAM cache of recently used pages; reads/writes pass through it to avoid disk.
Write-ahead log (WAL)
A sequential journal written before pages, making commits both fast and crash-safe.
Query optimizer
The component that chooses the cheapest execution plan using table statistics.
Query plan
The chosen step-by-step strategy for running a query (seen via EXPLAIN).
Full table scan
Reading every page of a table — what happens with no useful index.
Parquet
The dominant open columnar file format: row groups → column chunks → pages, with a stats footer.
Row group
A horizontal slice of a Parquet file (~1M rows), stored column-by-column inside.
Column chunk
All values of one column within one row group, contiguous and compressed.
Footer / metadata
The end of a Parquet file holding the schema and per-chunk min/max stats.
Partitioning
Splitting a dataset into folders by a key (often date) so queries skip whole folders.
Partition pruning
Skipping partition folders that a filter rules out — the cheapest big win in OLAP.
Data skipping / zone maps
Using per-chunk min/max stats to skip data that can't match a filter.
Projection pushdown
Reading only the columns a query references (column pruning).
Predicate pushdown
Applying WHERE filters at the scan, before later operators.
Vectorized execution
Processing columns in batches rather than one row at a time, avoiding per-row overhead.
SIMD
Single Instruction, Multiple Data — one CPU instruction that operates on many values at once; a big part of why batch (vectorized) processing is fast.
Parquet / ORC
The two dominant open columnar file formats; ORC is an alternative to Parquet with the same row-group + column-chunk + stats idea.
Bloom filter
A compact per-file structure answering "is this value possibly here?" — lets a query skip files for equality filters on high-cardinality columns.
MPP
Massively Parallel Processing — splitting a query across many worker nodes that scan in parallel.
Shuffle
Redistributing data across nodes (e.g. by GROUP BY key) so related rows meet to be combined.
Materialized view
A precomputed, stored query result (e.g. a daily rollup) that turns a big scan into a small lookup.
