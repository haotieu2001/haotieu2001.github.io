Docker Essentials for Data Engineers

Docker packages your code and everything it needs to run — the Python version, the system libraries, the environment variables — into a single portable box that behaves identically on your laptop, in CI, and in production. It is the definitive cure for "but it works on my machine." This guide builds the mental model one piece at a time (image vs container, and why that distinction explains almost everything), then makes it concrete with Dockerfiles you can copy — before covering the parts that actually bite: the layer cache, multi-stage builds, volumes, networking, Compose, a fully worked example, and the data-engineering specifics: dbt, Airflow, Spark and reproducible environments.

The one idea (read this first)

Docker solves one problem: your code needs a specific environment, and every machine has a different one. So you write a Dockerfile — a recipe listing the exact base OS, the exact Python, the exact packages — and Docker builds it into an image: a frozen, read-only snapshot of a complete filesystem. You can then run that image to get a container: an isolated process that believes it has the machine to itself. The image is the class; the container is the instance — one image can spawn a thousand identical containers, and throwing one away costs nothing. Because the image contains everything, the thing you tested on your laptop is byte-for-byte the thing that runs in production. That's the whole promise.

01. What Docker is & why it exists

You write a pipeline. It runs beautifully on your laptop. You hand it to a colleague and it explodes: they have Python 3.10, you have 3.12. You get it working, ship it to the server — and it explodes again, because the server is missing a system library that pandas quietly depends on. Every hour spent on that is an hour not spent on data. Docker's answer: stop shipping code, and start shipping the whole environment.

The unit you ship is an image — a complete, frozen filesystem containing your code, your Python, your packages, your system libraries, right down to the OS userland. When that image runs anywhere — a laptop, a CI runner, a Kubernetes node — it brings its world with it. What that buys you:

  • Reproducibility — the environment is declared in a file and rebuilt identically, instead of being assembled by hand and half-remembered. The image you tested is the image that runs.
  • Isolation — project A needs Spark 3.3 and project B needs Spark 3.5, and neither has to know the other exists. No more virtualenv archaeology, no more "which Java is on the PATH?"
  • Portability — the same image runs on your Mac, on a Linux CI runner, and on a production cluster. This is what makes containers the substrate for Airflow, Kubernetes, and nearly every modern data platform.
  • Disposability — containers start in milliseconds and cost nothing to destroy. Spin up a real Postgres for a test, run it, delete it, leave no trace on your machine.

A container is not a tiny virtual machine

This is the misconception that makes everything else confusing. A container does not boot an operating system. It is one ordinary process running directly on your host's kernel, which the kernel has been told to lie to: it sees its own filesystem, its own network, its own process list, and nothing else. That's why containers start in milliseconds and weigh megabytes while VMs take a minute and weigh gigabytes. §03 shows exactly how the trick works.

Docker, containers, and the OCI

"Docker" is a company and a tool; "container" is the underlying idea. The formats are standardised by the OCI (Open Container Initiative), so images are portable across tools: Podman (a daemonless, rootless alternative with a near-identical CLI), containerd (what Kubernetes actually runs), BuildKit (the modern builder Docker uses under the hood). Everything in this guide — the Dockerfile, the image, the commands — works the same on all of them. Learn Docker; you've learned containers.

02. The mental model: image vs container

If you take one thing from this guide, take this. Nearly every Docker question a beginner asks — "where did my data go?", "why is my change not showing up?", "why is it so big?" — dissolves the moment this distinction is solid.

1
An image is a read-only stack of filesystem layers, built from a Dockerfile. It's inert — a file on disk, like a .zip that happens to contain an entire OS userland. Images are shared: run ten containers from one image and the layers exist on disk exactly once.
2
A container is an image plus a thin writable layer, plus a running process. Anything the process writes — a temp file, a log, a downloaded CSV — lands in that writable layer.
3
Deleting a container deletes its writable layer. This is not a bug; it's the entire design. Containers are meant to be cattle, not pets. If you need something to survive, it must live outside the container — in a volume (§14) or a database. Every "Docker ate my data" story is this sentence, learned the hard way.

The two commands beginners confuse

docker build makes an image from a Dockerfile. docker run makes a container from an image. So if you edit your code and re-run docker run without rebuilding, you get the old code — because the image hasn't changed. "Why isn't my change showing up?" is almost always a forgotten build. (In development, bind-mounting your source solves this permanently — see §14.)

03. How it actually works (and how it differs from a VM)

You don't strictly need this section to use Docker — but ten minutes here will save you from a dozen confusing errors later, because it explains why containers behave the way they do.

The isolation is done by two Linux kernel features you should be able to name:

  • Namespaces — what the process can see. Separate namespaces for the filesystem, process list, network stack, hostname and users. Inside its namespace your process is PID 1 and thinks it owns the machine; it simply cannot perceive anything else on the host. This is the "fence."
  • cgroups (control groups) — what the process can use. CPU, memory, and I/O limits. This is what makes --memory=4g real, and it's why a container that exceeds its memory limit is killed by the kernel with exit code 137 — a number you will meet when a Spark job or a pandas read_csv gets greedy.

"Shares the kernel" has two consequences that will bite you

(1) Containers are Linux. On a Mac or Windows machine, Docker quietly runs a small Linux VM and puts your containers inside it. That's why file access to a bind-mounted folder can feel slow on a Mac — it's crossing a VM boundary. (2) Containers are CPU-architecture-specific. An image built on an Apple Silicon Mac is arm64; your cloud runners are almost certainly amd64. Push the first to the second and you get exec format error. The fix is docker build --platform linux/amd64 (or a multi-arch build with buildx). This single gotcha accounts for a truly disproportionate share of data engineers' lost afternoons.

04. The architecture: what actually runs when you type docker run

The word "Docker" hides four separate programs. Knowing which one does what turns most confusing errors into obvious ones — and explains why access to one innocuous-looking socket is the same thing as being root.

Access to /var/run/docker.sock IS root access

The daemon runs as root, and it will happily run any container you ask it to — including one that bind-mounts the host's / and gives you a shell in it. So adding a user to the docker group is exactly the same as giving them passwordless sudo, and mounting the socket into a container ("so it can build images") hands that container the whole machine. This is the single most common way a container escape happens, and it isn't a bug — it's the design. If you need rootless builds, use rootless mode or Podman, which runs containers as your own user with no daemon at all.

What each piece is for

PieceWhat it doesWhy you care
docker (CLI)Turns your command into an HTTP request on a Unix socket.It can talk to a remote daemon (DOCKER_HOST) — "docker" needn't be local.
dockerdImages, builds, volumes, networks, the API.The root-owned part. Restarting it does not kill running containers.
containerdContainer lifecycle: pull, unpack, start, stop.Kubernetes talks to this directly and skips dockerd entirely.
runcMakes the namespaces + cgroups, then execs your process.The actual "containerising" step — and it lasts milliseconds.

05. Namespaces & cgroups — the two kernel features that are containers

There is no such thing as a "container" in the Linux kernel. There are two unrelated features, and a container is just a process that has been started with both of them applied. Namespaces control what a process can see. Cgroups control how much it can use.

bash — see it for yourself
# A container really is just a process on your host. Prove it:
docker run -d --name proof python:3.12-slim sleep 300
ps -ef | grep sleep          # ← there it is, in YOUR process list, as a normal PID

# Inside, the pid namespace makes it think it is PID 1:
docker exec proof ps -ef     # PID 1 = sleep 300. It cannot see your host at all.

# The cgroup is a directory. The limit is a number in a file:
docker run --memory=512m --cpus=0.5 alpine \
  cat /sys/fs/cgroup/memory.max     # → 536870912

Limits are not optional in production

A container with no --memory can consume every byte on the host and take its neighbours down with it. A pandas job that reads a bigger file than usual is the classic trigger. Always set limits — and remember that the process inside sees the host's total RAM unless the library is cgroup-aware, so a JVM or a thread pool sized from "available CPUs" will happily over-allocate.

"root" inside a container is real root — unless you say otherwise

By default the user namespace is not enabled, so UID 0 inside the container is UID 0 on the host. It is only the missing capabilities and the seccomp filter that stop it from doing damage — and --privileged removes both, which is why it is effectively "become root on the host." Run as a non-root USER in your Dockerfile and the question stops mattering.

06. The core loop: build → run → ship

Four commands carry you through almost every day with Docker. Here they are, in the order you'll use them.

the loop
# 1. BUILD an image from the Dockerfile in this directory
docker build -t acme/etl:1.4.0 .
#            └─ tag: name:version      └─ the "build context" (see below)

# 2. RUN a container from that image
docker run --rm -e ENV=dev acme/etl:1.4.0
#          │      └─ set an env var        └─ which image
#          └─ delete the container when it exits (use this constantly)

# 3. SHIP it to a registry so other machines can pull it
docker push acme/etl:1.4.0

# 4. …and anywhere else
docker pull acme/etl:1.4.0   # byte-for-byte the same image

That lonely . is the build context, and it matters

The final argument to docker build is not "where the Dockerfile is" — it's the build context: the directory that gets uploaded to the Docker engine before the build starts. Everything COPY can see must be inside it. Run docker build . in a folder containing a 4 GB data/ directory and Docker will dutifully package up all 4 GB before it builds anything. That's why .dockerignore is not optional (§07) — it's how you keep your data, your .git, and your virtualenv out of the build.

07. The Dockerfile, instruction by instruction

A Dockerfile is a recipe read top to bottom. Each instruction runs in order, and each one that changes the filesystem produces a new layer. That's the whole language — about ten instructions, and you'll use six of them constantly.

Dockerfile — annotated
FROM python:3.12-slim          # the base image you build ON TOP of. Always first.

WORKDIR /app                    # cd into /app (creates it). Every later command runs here.

ENV PYTHONUNBUFFERED=1          # env var, baked into the image AND every container
    # ↑ without this, Python buffers stdout and your logs appear late — or never.
    #   Put it in every Python image you ever write.

RUN apt-get update && apt-get install -y --no-install-recommends \
      libpq-dev gcc \
    && rm -rf /var/lib/apt/lists/*      # ← clean up IN THE SAME RUN (see below)

COPY requirements.txt .         # copy ONE file first — this is deliberate (§08)
RUN pip install --no-cache-dir -r requirements.txt

COPY src/ ./src/                # now the code, which changes far more often

USER 1001                       # drop root. Do this in every image (§19).

EXPOSE 8000                     # pure DOCUMENTATION — it publishes nothing (§15)

ENTRYPOINT ["python", "-m"]      # the fixed part of the command
CMD ["src.pipeline"]           # the DEFAULT argument — overridable at run time

The instructions worth knowing

FROM
The base image to build on. python:3.12-slim, ubuntu:24.04, or scratch (literally nothing). Every Dockerfile starts here, and this one line decides most of your final size and most of your security exposure.
RUN
Executes a command at build time and saves the result as a layer. This is where you install packages. Each RUN is a separate layer, so chain related commands with &&.
COPY
Copies files from the build context into the image. Prefer it over ADD, which also auto-extracts tarballs and fetches URLs — surprising behaviour you rarely want.
WORKDIR
Sets the working directory for everything after it. Use it instead of RUN cd /app, which doesn't persist between instructions.
ENV / ARG
ENV is available at build time and at run time, and is baked into the image. ARG exists only during the build. Neither is a secret store — both are visible in docker history.
CMD
The default command, run when the container starts. Easily overridden: docker run myimage python -c "print(1)" replaces it entirely.
ENTRYPOINT
The command that always runs; anything you pass to docker run becomes its arguments. Use ENTRYPOINT for the executable and CMD for the default arguments — that combination gives you a container that behaves like a CLI tool.
USER
Switches to a non-root user for everything after it. See §19 — the default really is root, and that really is a problem.
EXPOSE
Documents which port the app listens on. It does not publish anything — only docker run -p does that. A genuinely misleading name.
HEALTHCHECK
A command Docker runs periodically to decide whether the container is actually healthy, not merely running. Compose can wait on it (§17).

Two rules that separate good Dockerfiles from bad ones

chain your RUNs — and know why
− BAD — three layers, and the "cleanup" achieves nothing
RUN apt-get update
RUN apt-get install -y gcc
RUN rm -rf /var/lib/apt/lists/*     # the files are STILL in the layer below!

+ GOOD — one layer; the deleted files never exist in any layer
RUN apt-get update \
    && apt-get install -y --no-install-recommends gcc \
    && rm -rf /var/lib/apt/lists/*

Why this matters more than it looks: layers are immutable and additive. Deleting a file in layer 3 doesn't remove it from layer 2 — it just adds a "this file is hidden" marker on top. The bytes are still in the image, still downloaded on every pull, and still readable by anyone with the image. That's a size problem and a security one: a secret you COPY in and then RUN rm is still in the image, forever.

The .dockerignore file — write it first, not last

.dockerignore
.git
.venv
venv/
__pycache__/
*.pyc
data/                # ← your 4 GB of CSVs do NOT belong in the build context
notebooks/
.env                 # ← especially this. never bake secrets into an image.
*.md
.pytest_cache/

Same syntax as .gitignore. It makes builds faster (less to upload), images smaller, and cache hits far more likely — because if .git/ is in the context, every commit changes the context and busts your cache.

08. The layer cache — why instruction order matters

This is the section that turns a two-minute rebuild into a two-second one. Docker caches every layer; understanding when it throws that cache away is the difference between a Dockerfile that's a joy to iterate on and one you dread.

The rule is simple and unforgiving: Docker walks your instructions top to bottom, reusing the cached layer for each one — until it hits an instruction whose inputs changed. From that point on, every single layer below is rebuilt. The cache is invalidated downward, never partially.

the difference, in code
− SLOW — every code edit reinstalls every package
COPY . .                                    # any file changes → cache busted here
RUN pip install -r requirements.txt         # …so this re-runs. 2 minutes. Every build.

+ FAST — the install layer is reused until your dependencies actually change
COPY requirements.txt .                     # only busts when THIS file changes
RUN pip install -r requirements.txt         # cached for weeks at a time
COPY src/ ./src/                            # the fast-changing stuff goes LAST

How to read the cache in build output

During a build, CACHED next to a step means it was reused. The first step without it is where your cache broke — and everything after it is being rebuilt. When a build is mysteriously slow, find that line and ask "what did I change that this step depends on?" Nine times out of ten the answer is a COPY that's too broad, or a missing .dockerignore.

09. BuildKit — how builds actually run now

The old builder walked your Dockerfile one line at a time, top to bottom, and cached whole layers. BuildKit (the default since Docker 23) reads the whole file first, builds a dependency graph, and runs everything it can in parallel. It also unlocks two features you should be using: cache mounts and build secrets.

Cache mounts — keep the pip cache between builds

A normal RUN pip install downloads every wheel again on every cache miss. A cache mount is a directory BuildKit keeps outside the image and re-attaches on each build. It is not a layer, so it never ships — but it makes a cold rebuild feel warm.

Dockerfile — BuildKit features
# syntax=docker/dockerfile:1.7
FROM python:3.12-slim

# The pip download cache survives across builds — it is NOT baked into a layer.
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install -r requirements.txt

# A secret is mounted for ONE command and never touches a layer.
# Compare with ENV/ARG, which are permanently visible in `docker history`.
RUN --mount=type=secret,id=pypi_token \
    pip install --extra-index-url \
      "https://$(cat /run/secrets/pypi_token)@pypi.acme.internal/simple" acme-lib
bash
docker build --secret id=pypi_token,env=PYPI_TOKEN -t acme/etl:1.4.0 .

# Multi-arch — build for the Mac laptops AND the amd64 cluster, in one push:
docker buildx build --platform linux/amd64,linux/arm64 \
  -t acme/etl:1.4.0 --push .

Never pass a secret with ARG or ENV

Both are baked into the image metadata for anyone who runs docker history. Even RUN export TOKEN=… && … && unset TOKEN does not help: the layer's filesystem diff has already been written. --mount=type=secret is the only way to use a credential at build time without shipping it.

10. Images, tags, registries & multi-stage builds

Images have names, versions, and homes. And they have a size problem that multi-stage builds solve elegantly.

Naming and tags

anatomy of an image name
ghcr.io/acme/etl:1.4.0
└──┬───┘ └─┬─┘ └┬┘ └─┬─┘
   │       │    │    └─ TAG — a version. Mutable! Someone can move it.
   │       │    └────── the image name
   │       └─────────── the org / namespace
   └─────────────────── the REGISTRY (default: docker.io — Docker Hub)

# Registries you'll meet: Docker Hub · GitHub (ghcr.io) · AWS ECR · Azure ACR · GCP Artifact Registry

:latest is a lie

latest is not "the newest version" — it is simply the default tag, and it means whatever someone last pushed under that name. It can change under you between your test and your deploy. Never use it in anything that matters: pin python:3.12-slim, not python:latest, and tag your own images with a real version or the Git SHA (acme/etl:1.4.0, acme/etl:a3f9c1). For genuine immutability, pin the digest — python:3.12-slim@sha256:abc123… — which is a content hash and cannot be moved.

Multi-stage builds — leave the toolchain behind

Building Python packages often needs a compiler (gcc), header files, maybe Rust. Running them needs none of that. A multi-stage build lets you use a fat image to build and a slim one to ship, copying only the finished artefacts across.

Dockerfile — multi-stage
# ── stage 1: build the dependencies (needs a compiler) ──
FROM python:3.12 AS builder

WORKDIR /app
COPY requirements.txt .
# install into a self-contained folder we can lift out wholesale
RUN pip install --no-cache-dir --prefix=/install -r requirements.txt

# ── stage 2: the image we actually ship ──
FROM python:3.12-slim

COPY --from=builder /install /usr/local    # ← only the built packages come across
WORKDIR /app
COPY src/ ./src/
USER 1001
ENTRYPOINT ["python", "-m", "src.pipeline"]

Choosing a base image: slim, yes. alpine, probably not.

Alpine Linux images are famously tiny, and for a Go binary they're perfect. For Python data work they are usually a trap. Alpine uses musl instead of glibc, and the pre-compiled wheels on PyPI for numpy, pandas, pyarrow and friends are built for glibc. So on Alpine, pip can't use them — it falls back to compiling from source. Your "smaller, faster" image now takes twenty minutes to build, needs a full toolchain, and can end up bigger. Use python:3.12-slim (Debian-based, glibc, ~120 MB) as your default and only reach for Alpine when you've measured a reason.

11. Inside an image: digests, manifests & multi-arch

An image is not a file. It is a small JSON document that points at other things by cryptographic hash: a config blob, and a list of layer blobs. Everything is content-addressed, which is why layers dedupe across images for free — and why a tag is a much weaker promise than you think.

bash — look inside a real image
docker manifest inspect python:3.12-slim | head -30   # the multi-arch list
docker image inspect acme/etl:1.4.0 --format '{{.RootFS.Layers}}'   # its layer digests
docker history acme/etl:1.4.0                        # which instruction made which layer

# A tag is a moving pointer. A digest is forever:
docker pull python:3.12-slim@sha256:1f5b1a2c…        # ← what prod should do

:latest is a lie, and so is any other tag

A tag is a mutable label the publisher can re-point at any time. python:3.12-slim today and python:3.12-slim next month are different bytes — the base gets security patches, and rightly so. That is fine for development and a genuine hazard for reproducibility: the pipeline that passed CI in March can fail in April with nothing in your repo having changed. Pin by digest for production images, and let a bot (Dependabot, Renovate) raise the bump as a reviewable PR.

12. How the filesystem works: OverlayFS & copy-on-write

Ten containers from one image do not cost ten times the disk. They share the image's layers read-only and stack one thin writable layer on top. The kernel merges the stack with OverlayFS, and the trick it uses — copy-on-write — explains two things that surprise people: why writing a big file inside a container is slow, and why deleting a secret from an image does not delete it.

A secret added in one layer and deleted in the next is still in the image

This is the single most common way credentials leak from a container image:

Dockerfile — the bug
COPY .env /app/.env        # layer 4 — the secret is now in the image, forever
RUN python setup.py && rm /app/.env   # layer 5 — only HIDES it with a whiteout

Anyone who pulls the image can run docker save, untar layer 4, and read the file. The fixes: never COPY the secret in the first place (use --mount=type=secret, §09), keep secrets out of the build context with .dockerignore, and use a multi-stage build so the final stage copies only the artefacts you name.

13. Running containers day to day

Building is half the job. Here's the handful of commands and flags you'll actually type, and the one lifecycle fact that explains most confusion.

the flags that matter
docker run \
  --name etl \                # give it a name (else you get "vibrant_hopper")
  --rm \                      # auto-delete when it exits — use this by default
  -d \                        # detached: run in the background
  -it \                       # interactive + TTY: for a shell you can type into
  -p 8080:80 \                # publish HOST port 8080 → CONTAINER port 80
  -e DB_HOST=postgres \       # set an environment variable
  --env-file .env \           # …or a whole file of them
  -v mydata:/var/lib/data \   # mount a volume (§14)
  --memory=4g --cpus=2 \      # cgroup limits — real, enforced
  acme/etl:1.4.0 \
  --date 2026-07-12           # args appended to the ENTRYPOINT
inspecting what's running
docker ps                     # running containers
docker ps -a                  # …including stopped ones (this is where "exited" ones hide)
docker logs -f etl            # stream its stdout/stderr. Your #1 debugging tool.
docker exec -it etl bash      # get a shell INSIDE a running container. Your #2.
docker inspect etl            # everything: mounts, network, env, exit code
docker stats                  # live CPU/memory — is it about to be OOM-killed?

docker stop etl               # polite SIGTERM, then SIGKILL after 10s
docker rm etl                 # delete the stopped container (and its writable layer!)
docker image ls               # what images do I have?
docker system df              # what is eating my disk? (spoiler: build cache)
docker system prune -a        # reclaim it. Careful: -a removes unused IMAGES too.

A container lives exactly as long as its main process

This one fact explains the most common beginner bug: "my container exits immediately." A container is not a machine that "stays up" — it is a wrapper around PID 1. When that process finishes, the container stops. Full stop. So a container whose command is bash (with nothing to read) exits instantly; a container running a web server stays up because the server never returns. If yours dies on startup, run docker logs <name> — the process almost certainly crashed, and the exit code tells you why (0 = finished normally, 1 = error, 137 = killed, usually out of memory, 139 = segfault).

14. Data: volumes & bind mounts

Containers are ephemeral by design. So the moment you have data you care about — a Postgres database, a model artefact, your source code during development — you need it to live outside the container. There are two ways, and they're for two different jobs.

 Named volumeBind mount
Syntax-v pgdata:/var/lib/postgresql/data-v ./src:/app/src (a path)
Where it livesA location Docker manages for youAn exact folder on your machine
Best forPersistent data: databases, warehouse files, model artefactsDevelopment: your source code, so edits take effect without rebuilding

The development trick that makes Docker pleasant

Bind-mount your source code in development: -v ./src:/app/src. Now the container sees your live files, so you edit in your editor and re-run without rebuilding the image at all. The build-edit-rebuild cycle that makes people hate Docker just disappears. In production you do the opposite — no bind mount, the code is baked into the image, so what you tested is exactly what runs. Compose makes it easy to have both (§17).

15. Networking

Two questions cover almost everything: how do I reach a container from my laptop? and how do containers reach each other?

1
From the host → publish a port. -p 8080:5432 means "traffic to my port 8080 goes to the container's port 5432." Left is host, right is container — get them backwards and you'll stare at a refused connection for ten minutes. Note that EXPOSE in the Dockerfile publishes nothing; only -p does.
2
Container → container: use a user-defined network, and address each other by name. Put containers on the same network and Docker gives you DNS for free: a container named postgres is reachable at the hostname postgres. No IPs, ever. Compose does this for you automatically.

localhost inside a container is not your laptop

The single most common networking bug. Your app's config says DB_HOST=localhost, it works outside Docker, and inside a container it fails with "connection refused" — because localhost now means this container, which has no database in it. Use the service name (DB_HOST=postgres). And if a container genuinely needs to reach a service running on your host machine, the special hostname is host.docker.internal.

16. Networking, deeper: bridges, veth pairs & NAT

The previous section covered the rules you need day to day. This one covers the machinery underneath — because "connection refused" and "port already in use" stop being mysterious once you can picture the virtual cable and the NAT rule.

docker exec airflow ip addr                  # eth0 = 172.18.0.2
docker exec airflow cat /etc/resolv.conf     # nameserver 127.0.0.11 ← Docker's DNS
docker exec airflow getent hosts postgres    # resolves by service name

# The host's view — the NAT rule -p created:
sudo iptables -t nat -L DOCKER -n

Two rules that resolve almost every networking bug

1. Bind to 0.0.0.0, not 127.0.0.1. A server listening on 127.0.0.1 inside a container is reachable only from inside that container — the published port will connect and immediately refuse. This is the classic "works locally, refuses in Docker" bug, and it is a one-word fix in your app's config.

2. Container-to-container traffic ignores -p entirely. Publishing is only for traffic arriving from outside. Two containers on the same user-defined network talk on the container port directly — so DB_HOST=postgres, DB_PORT=5432, whether or not you published 5432. If you find yourself publishing a database port just so your app can reach it, you have the wrong mental model — and you have also exposed your database to the whole machine.

17. Docker Compose — the whole stack in one file

Real data work is never one container. It's a pipeline and a database and an object store and maybe an Airflow scheduler. Compose declares all of them in one YAML file and starts them with one command — and it's how most data engineers use Docker every single day.

docker-compose.yml — a local data stack
services:

  postgres:                                # ← this name IS the hostname (§15)
    image: postgres:16                      # pinned, not :latest
    environment:
      POSTGRES_USER: dev
      POSTGRES_PASSWORD: dev              # fine for LOCAL only
      POSTGRES_DB: warehouse
    volumes:
      - pgdata:/var/lib/postgresql/data     # ← named volume: the data survives (§14)
    healthcheck:                            # "is it actually ready?"
      test: ["CMD-SHELL", "pg_isready -U dev"]
      interval: 5s
      retries: 5

  minio:                                   # an S3-compatible object store, locally
    image: minio/minio:latest
    command: server /data --console-address ":9001"
    ports: ["9000:9000", "9001:9001"]
    volumes: [minio_data:/data]

  pipeline:
    build: .                                # build from the Dockerfile here…
    depends_on:
      postgres:
        condition: service_healthy         # ← wait for the HEALTHCHECK, not just start
    environment:
      DB_HOST: postgres                    # ← the service name. NOT localhost.
      S3_ENDPOINT: http://minio:9000
    volumes:
      - ./src:/app/src                      # ← bind mount: live code editing in dev

volumes:                                   # declare the named volumes
  pgdata:
  minio_data:
the Compose commands
docker compose up -d              # start everything in the background
docker compose up --build         # rebuild images first (after a Dockerfile change)
docker compose ps                 # what's running, and is it healthy?
docker compose logs -f pipeline   # follow one service's logs
docker compose exec postgres psql -U dev warehouse   # a psql shell, instantly
docker compose down               # stop & remove containers + network
docker compose down -v            # …AND delete the volumes. Your database is gone.

depends_on alone does not wait for readiness

A classic and maddening race. Plain depends_on: [postgres] waits only until the Postgres container has started — not until Postgres is accepting connections, which takes a few seconds longer. So your pipeline starts, connects, and dies with "connection refused" roughly one run in three. The fix is exactly what's in the file above: give the dependency a healthcheck, and depend on condition: service_healthy. Now Compose genuinely waits.

18. Docker for data engineers

Generic Docker knowledge is most of the battle. Here's what's specific to our world — where containers show up, and the traps that are ours alone.

Where you'll actually meet containers

  • A reproducible dev environment. The everyday win: your whole stack — Python, dbt, a Postgres, a MinIO standing in for S3 — comes up with docker compose up and behaves identically for every teammate. New joiner productive in an hour, not a week.
  • Airflow. Airflow itself runs in containers, and its DockerOperator / KubernetesPodOperator run your tasks as containers — which is how you let one DAG run a Python 3.10 task and a Python 3.12 task without them fighting. You'll also build a custom Airflow image (base image + your requirements.txt) rather than pip-installing into a running scheduler.
  • dbt. Containerising dbt pins the exact dbt-core and adapter versions, so CI and every laptop resolve identically — a real problem given how quickly adapter versions drift.
  • CI. Build the image once, run the tests inside it, push it if they pass. The artifact you tested is the artifact you deploy — the same "build once, deploy many" principle as CI/CD Essentials §05.
  • Spark & Kubernetes. Spark on K8s ships your job as an image. Understanding containers is understanding how your Spark job gets to the cluster.

The four traps that are specifically ours

(1) The arm64/amd64 mismatch. You build on an Apple Silicon Mac, push, and production says exec format error. Build with --platform linux/amd64, or use buildx for multi-arch. (2) Memory limits are real. Pandas happily allocates until the cgroup kills it — exit code 137, no traceback, no clue. Set --memory deliberately and watch docker stats. (3) Don't bake data into images. An image is code, not a data store; a 6 GB image with a CSV inside is slow to pull and impossible to version sensibly. Mount data, or read it from object storage. (4) Alpine + pandas = pain. As in §10: use slim.

a custom Airflow image — the standard pattern
FROM apache/airflow:2.9.2-python3.11   # pin BOTH airflow and python

# switch to root only to install SYSTEM packages…
USER root
RUN apt-get update \
    && apt-get install -y --no-install-recommends libpq-dev gcc \
    && rm -rf /var/lib/apt/lists/*

# …then switch straight back. Airflow refuses to run as root anyway.
USER airflow

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY --chown=airflow:root dags/ /opt/airflow/dags/

19. Security & best practices

Containers isolate, but they are not a security boundary you should lean on carelessly — and a careless image leaks secrets in a way that's genuinely hard to take back.

  • Don't run as root. Containers run as root by default, and root in the container is root on the host kernel if anything escapes. Add USER 1001 (or a created user) to every image. This is the single highest-value line in this section.
  • Never bake secrets into an image. Not with ENV, not with ARG, not with a COPY .env that you delete later — the layer keeps it (§07), and docker history will show it to anyone who pulls the image. Pass secrets at run time (-e, --env-file, a mounted file, or your orchestrator's secret store), or use BuildKit's --mount=type=secret if you need one during the build.
  • Pin your base images. python:3.12-slim, never python:latest. For real reproducibility, pin the digest (@sha256:…).
  • Keep images small. Multi-stage builds and a good .dockerignore. Fewer packages = smaller attack surface, faster pulls, cheaper storage.
  • Scan them. docker scout cves acme/etl:1.4.0, or Trivy/Grype in CI. Base images accumulate CVEs just sitting there; rebuilding on a fresh base is often the whole fix.
  • Prefer read-only where you can. --read-only plus a tmpfs for scratch space stops a compromised process from rewriting your app.
  • Mind the Docker socket. Mounting /var/run/docker.sock into a container hands it effective root on the host. Sometimes necessary (CI runners), never casual.

Try this once and you'll never bake a secret again

Build an image that does COPY .env . followed by RUN rm .env. The file is gone from the final filesystem — ls proves it. Now run docker history --no-trunc on the image, or simply extract the layer beneath. The secret is right there. Layers are immutable and additive; deletion only hides. This is why an accidentally-committed AWS key in a pushed image must be treated as leaked and rotated, not "deleted in the next build."

20. A worked example: containerising a real pipeline

Now let's put it together: a small ETL project, containerised properly — every file, why it exists, and what happens when you change one line. This is the shape you can copy into a real repo tomorrow.

the repo
etl-pipeline/
├── Dockerfile              # multi-stage, non-root, cache-friendly
├── .dockerignore           # write this FIRST
├── docker-compose.yml      # the local stack: pipeline + postgres + minio
├── requirements.txt        # pinned dependencies
├── src/
│   ├── __init__.py
│   └── pipeline.py         # the actual ETL
└── tests/
    └── test_pipeline.py

Dockerfile — every line earns its place

Dockerfile
# ─────────── stage 1: builder ───────────
FROM python:3.12-slim AS builder

# build-only system deps: gcc is needed to compile some wheels,
# and it will NOT exist in the final image.
RUN apt-get update \
    && apt-get install -y --no-install-recommends gcc libpq-dev \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /app

# ↓ the cache trick from §08: dependencies BEFORE source code
COPY requirements.txt .
RUN pip install --no-cache-dir --prefix=/install -r requirements.txt

# ─────────── stage 2: runtime ───────────
FROM python:3.12-slim

# only the RUNTIME system libs — libpq, but no compiler
RUN apt-get update \
    && apt-get install -y --no-install-recommends libpq5 \
    && rm -rf /var/lib/apt/lists/* \
    && useradd --uid 1001 --create-home appuser      # a real non-root user

ENV PYTHONUNBUFFERED=1 \
    PYTHONDONTWRITEBYTECODE=1                       # no stray .pyc files

WORKDIR /app

COPY --from=builder /install /usr/local            # ← the packages, without the toolchain
COPY --chown=appuser:appuser src/ ./src/

USER appuser                                        # drop root (§19)

ENTRYPOINT ["python", "-m", "src.pipeline"]
CMD ["--date", "today"]                            # default arg; override at run time

Read it against the rules from earlier sections and every choice has a reason. slim, not alpine, because pandas wheels need glibc (§10). Two stages, so gcc — needed to build, useless to run — never ships. requirements.txt copied before src/, so editing a Python file doesn't reinstall pandas (§08). apt-get install and cleanup chained in one RUN, so the package lists never exist in a layer (§07). USER appuser, so a compromise doesn't start as root (§19). ENTRYPOINT + CMD split, so the container behaves like a CLI: docker run acme/etl --date 2026-07-12 just works.

docker-compose.yml — the dev stack

docker-compose.yml
services:
  pipeline:
    build: .
    depends_on:
      postgres: { condition: service_healthy }   # wait for READY, not just started
    environment:
      DB_HOST: postgres                            # service name (§15)
      DB_USER: dev
      DB_PASSWORD: dev
    volumes:
      - ./src:/app/src                              # live-edit in dev; NOT in prod (§14)

  postgres:
    image: postgres:16
    environment:
      POSTGRES_USER: dev
      POSTGRES_PASSWORD: dev
      POSTGRES_DB: warehouse
    volumes: [pgdata:/var/lib/postgresql/data]     # the data outlives `compose down`
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U dev"]
      interval: 5s
      retries: 5

volumes:
  pgdata:

The CI job — test inside the image

.github/workflows/docker.yml
jobs:
  build-test-push:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build the image
        run: docker build --platform linux/amd64 -t acme/etl:${{ github.sha }} .
        #                    ↑ explicit platform — never trust the runner's default

      - name: Run the tests INSIDE the image we just built
        run: |
          docker run --rm --entrypoint pytest acme/etl:${{ github.sha }} -q
        # ↑ overriding the entrypoint runs pytest in the exact prod environment.
        #   If the tests pass here, they pass in production — same bytes.

      - name: Push (only from main, only if the tests passed)
        if: github.ref == 'refs/heads/main'
        run: docker push acme/etl:${{ github.sha }}

This is the detail that makes containerised CI worth doing. Most pipelines install dependencies on the runner and run pytest there — which tests the runner's environment, not your image's. Running the suite with --entrypoint pytest against the freshly built image tests the artefact you're about to deploy. And tagging with the Git SHA means every image is traceable to exactly one commit.

The life of a change

1
You edit src/pipeline.py. In local dev, that's it — the file is bind-mounted, so docker compose restart pipeline picks it up instantly. No rebuild at all.
2
You rebuild anyway (for CI). Docker replays the cache: FROM +, apt-get +, COPY requirements.txt +, pip install + CACHED — the ninety-second layer is untouched, because requirements.txt didn't change. Only COPY src/ and below rebuild. Total: about two seconds.
3
Now you add polars to requirements.txt. That file changed, so its COPY layer busts, and everything below it rebuilds — including pip install. This build takes the full ninety seconds. That's correct and unavoidable: the dependencies genuinely changed. The point of the ordering is that this happens only when dependencies change, not on every code edit.
4
CI builds, tests inside the image, pushes it, and production pulls it. No environment was assembled twice. Nothing was installed on a server by hand. The image that passed the tests is the image that runs.

Why this shape holds up

Notice what you get almost for free: a new teammate runs docker compose up and has a working Postgres and a working pipeline in two minutes; CI tests the real artefact instead of an approximation of it; production runs bytes that were tested, not a rebuild that should be equivalent. Every one of those is a class of bug that simply stops happening.

21. Troubleshooting: the errors you will hit

Almost every Docker problem a data engineer meets is one of these eight. Learn to recognise them and you'll debug in seconds rather than hours.

SymptomWhat's really happeningFix
Container exits immediately, no errorIts PID 1 finished. A container lives exactly as long as its main process (§13)docker logs <name>. If the command was meant to be long-running, it crashed — the logs say why
Exit code 137The kernel OOM-killed it: it exceeded its memory cgroup. Very common with pandasRaise --memory, or process in chunks. Watch docker stats
exec format error in the cloudYou built arm64 on an Apple Silicon Mac; the runner is amd64docker build --platform linux/amd64
connection refused to the databaselocalhost inside a container means that container (§15)Use the service name: DB_HOST=postgres
DB connection fails only sometimes on startupdepends_on waited for started, not ready (§17)Add a healthcheck + condition: service_healthy
My code change isn't showing upYou ran the old image — docker run doesn't rebuild (§02)docker build again, or bind-mount the source in dev
Permission denied on a bind-mounted fileThe container's UID doesn't own the host folderMatch the UIDs (--user $(id -u):$(id -g)), or chown in the image
My disk is fullDangling images, stopped containers, and an enormous build cachedocker system df to see it, docker system prune -a to reclaim it

The universal first move: get inside and look

When a container misbehaves, stop guessing and open a shell in it: docker exec -it <name> bash (or sh on slim images). Now you're standing inside the environment — check the files are where you think, that the env vars are set, that you can reach the database (ping postgres). If the container won't stay up long enough to exec into, override the entrypoint and get a shell instead of the app: docker run -it --entrypoint bash acme/etl:1.4.0. Nine bugs in ten become obvious within thirty seconds of doing this.

22. Quick-reference cheat sheet

docker — keep this handy
── build & run ───────────────────────────────────────
docker build -t name:tag .                 # build (the "." is the CONTEXT)
docker build --platform linux/amd64 -t n .  # …for the cloud, from an M-series Mac
docker run --rm -it name:tag               # run, interactive, auto-delete
docker run -d -p 8080:80 --name web name   # detached, port published, named
docker run --rm --entrypoint bash name     # get a shell instead of the app

── inspect & debug ───────────────────────────────────
docker ps  /  docker ps -a                 # running / all (incl. exited)
docker logs -f <name>                      # follow the logs   ← START HERE
docker exec -it <name> bash                # shell inside it   ← THEN HERE
docker inspect <name>                      # mounts, env, network, exit code
docker stats                               # live CPU / memory

── clean up (do this before you run out of disk) ─────
docker system df                           # what's using space?
docker system prune -a                     # reclaim it (removes unused images too)
docker volume ls  /  docker volume rm X    # volumes are NOT removed by prune

── compose ───────────────────────────────────────────
docker compose up -d --build               # build + start the whole stack
docker compose logs -f <svc>               # one service's logs
docker compose exec <svc> bash             # shell into a service
docker compose down                        # stop & remove (volumes SURVIVE)
docker compose down -v                     # …and DELETE the volumes

── Dockerfile, in the right order ────────────────────
FROM python:3.12-slim          # slim, pinned. not alpine, not :latest
WORKDIR /app
ENV PYTHONUNBUFFERED=1         # or your logs vanish
RUN apt-get update && apt-get install -y X && rm -rf /var/lib/apt/lists/*
COPY requirements.txt .        # ← deps FIRST (cache!)
RUN pip install --no-cache-dir -r requirements.txt
COPY src/ ./src/               # ← code LAST (changes most)
USER 1001                      # ← never ship as root
ENTRYPOINT ["python","-m","src.pipeline"]
CMD ["--date","today"]         # the overridable default arg

── the mental model, in four lines ───────────────────
image      = read-only stack of layers        (the class)
container  = image + thin writable layer      (the instance)
volume     = storage that OUTLIVES containers (your data)
registry   = where images live between machines

＊ Glossary

Image
A read-only, layered snapshot of a filesystem — your code plus everything it needs to run. Inert until you run it.
Container
A running instance of an image: an isolated process, plus a thin writable layer that is destroyed with it.
Layer
The filesystem change produced by one Dockerfile instruction. Immutable, cached, and shared between images. Deleting a file in a later layer only hides it.
Dockerfile
The recipe: an ordered list of instructions Docker executes to build an image.
Build context
The directory sent to the Docker engine at build time (the . in docker build .). COPY can only see inside it. Trim it with .dockerignore.
Layer cache
Docker's reuse of unchanged layers. Invalidated downward from the first changed instruction — which is why instruction order matters so much.
Multi-stage build
Using a fat image to compile and a slim one to ship, copying only artefacts across with COPY --from.
Registry
Where images are stored and shared: Docker Hub, GHCR, ECR, ACR, Artifact Registry.
Tag
A human-readable version label on an image (:1.4.0). Mutable — it can be moved. A digest (@sha256:…) cannot.
Volume
Docker-managed storage that outlives containers. Where your persistent data belongs.
Bind mount
A host directory mapped into a container. Ideal for live-editing source in development.
Namespaces
The kernel feature that decides what a container can see (its own filesystem, processes, network).
cgroups
The kernel feature that decides what a container can use (CPU, memory). Exceeding a memory cgroup gets you exit code 137.
ENTRYPOINT vs CMD
ENTRYPOINT is the executable that always runs; CMD is the default arguments, overridable at docker run.
EXPOSE
Documentation only. It publishes nothing — docker run -p does that.
Compose
A YAML file defining a multi-container stack, plus the network and volumes to wire it together. One command to start it all.
Healthcheck
A command Docker runs to decide if a container is genuinely ready, not merely started. What depends_on: service_healthy waits for.
OCI
The Open Container Initiative — the standard that makes images portable across Docker, Podman, containerd and Kubernetes.
PID 1
The container's main process. When it exits, the container stops. This explains most "why did my container die?" questions.

That's the full toolkit: image vs container, how the isolation actually works, the Dockerfile, the layer cache, multi-stage builds, volumes, networking, Compose, a worked repo you can copy, and the traps that are specifically ours. Bookmark §22 as your day-to-day reference — and remember the through-line: build once, keep the image small, never run as root, and assume anything not on a volume is already gone.
