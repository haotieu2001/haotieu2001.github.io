Imagine a bank's mobile app. Every time a customer taps "transfer," something has to record that event immediately, check it for fraud in under a second, update a balance, and eventually land in a report — all while millions of other customers are doing the same thing at the same time. A normal database that only answers "what is true right now" cannot do this well. You need something built for a constant, never-ending flow of events. That is stream processing.

Two tools show up in almost every real-time data system built today: Apache Kafka, which moves and stores the flow of events, and Spark Structured Streaming, which reads that flow and computes answers from it (counts, joins, alerts, aggregates) as the data arrives. This guide explains both, from the ground up, with a diagram next to almost every idea — no assumption that you already know distributed systems, and every acronym is explained the first time it shows up.

❯ Short version (read this first)

Kafka is a distributed, append-only log. Producers write events to it, it stores them safely and in order, and consumers read them — possibly hours or days later. It is the plumbing: durable, ordered, replayable.

Spark Structured Streaming is a processing engine. It treats the incoming stream as a table that keeps growing one row at a time, and lets you write normal-looking queries (filters, joins, aggregations) that keep producing new results as new rows arrive.

• Kafka — durable, ordered, replayable event storage. Think: the postal system that never loses a letter and keeps every letter it ever delivered, in order, for as long as you ask it to.

• Spark Structured Streaming — turns a live stream into continuously-updated query results. Think: a spreadsheet formula that recalculates itself every time a new row is typed in, forever.

• Apache Flink — a true event-at-a-time engine built from the ground up for streaming (batch is the special case, not the other way around). Think: a factory line that reacts to each part the instant it arrives, instead of waiting to collect a tray of parts first.

Together: apps and services write events into Kafka topics → Spark Structured Streaming or Flink reads those topics, computes something useful, and writes the result to a database, dashboard, or data lake table. Real companies (see section 29) often run more than one of these at once, for different jobs.

New to the words? A topic is a named stream of events in Kafka (like a folder). A partition is one ordered slice of a topic (like one lane on a highway). A broker is one Kafka server. A producer writes events; a consumer reads them. A DataFrame is Spark's table-like data structure. A micro-batch is a small chunk of new data Spark processes every few hundred milliseconds to a few seconds.

01. Why Streaming, Why Now

From "check back tomorrow" to "know right now"

For decades, most data systems worked in batch: data piled up all day, and once a night a job ran, read the whole pile, and produced a report you could look at the next morning. That is batch processing — you wait, then you process everything at once. Stream processing flips this around: instead of waiting for a pile to build up, you process each new piece of data the moment it arrives, or within a few seconds of it arriving.

Think of it as a spectrum, not a hard line. Batch jobs might run once a day and take hours. Micro-batch systems (like Spark Structured Streaming, which you will meet later in this guide) collect a small slice of new data every few seconds and process that slice as if it were a tiny batch. True streaming systems process one event at a time, often within milliseconds of it happening. The diagram below shows where each one sits.

Bounded data vs. unbounded data

Bounded data has a fixed size — a CSV file, a database table as it looks right now, last month's sales. You can count exactly how many rows it has. Unbounded data — a stream — never finishes. New events keep arriving for as long as the system producing them keeps running: every card swipe, every sensor reading, every click. You can never say "that's all of it," because more is always on the way.

Example: a spreadsheet of last year's transactions is bounded. The live feed of transactions happening right now, forever, is unbounded.

Where stream processing shows up in the real world

• Fraud detection — a card swipe must be scored as fraud or not-fraud in under a second, before the transaction completes.

• Ride-hailing pricing — surge pricing recalculates constantly as demand and driver supply change minute to minute.

• IoT sensors — a factory floor with thousands of sensors streaming temperature and vibration readings that must trigger an alarm within seconds of a problem.

• Clickstream personalization — a website adjusts what it shows you based on what you clicked seconds ago, not yesterday.

• Bank transaction monitoring — watching account activity continuously to flag suspicious patterns as they happen, not in next month's audit.

Before Kafka, many systems used a message queue (like ActiveMQ or RabbitMQ) to move data between applications. A message queue is great for "deliver this one message to one worker, then forget it" — but it was not designed to durably keep millions of events per second, in order, for days or weeks, so that many different consumers can each read the same data independently, and even replay it from the beginning if needed. That gap is exactly why Apache Kafka was built: a distributed, durable, replayable log designed for the scale and shape of real-time data. The rest of this guide is about how Kafka does that, and how Spark Structured Streaming processes what comes out of it.

02. Kafka Core Concepts

Topics, partitions, offsets, brokers — the five words you need first

Everything else in Kafka is built from a handful of ideas. Get these right and the rest of the guide will make much more sense.

A Kafka message (record)

Every piece of data written to Kafka is a record with four parts: a key (optional, used to decide which partition it goes to and to group related records), a value (the actual payload — the data you care about), a timestamp (when it was created or received), and headers (optional small metadata tags that travel with the record but sit outside the value — useful when you want to attach information that has nothing to do with the business data itself, like a trace ID for debugging a request across services, without changing the shape of the value your application code actually parses).

Example: key = "customer-482", value = { amount: 250000, currency: "VND" } — this keeps every event for the same customer landing in the same partition, in order.

A topic is a named stream of records — think of it like a named folder that events get dropped into, for example a topic called transactions. A topic is split into one or more partitions: each partition is its own ordered, append-only sequence of records. Kafka only guarantees ordering within a single partition — it makes no promise about the order of records across different partitions of the same topic. Each record's position inside its partition is called its offset, a simple incrementing number starting at 0.

A broker is a single Kafka server process. A group of brokers working together is a cluster. Each partition is copied across multiple brokers for safety — the number of copies is the replication factor. Exactly one of those copies is the leader for that partition (it handles all reads and writes for it); the rest are followers that continuously copy the leader's data so they can take over if the leader's broker fails.

Kafka does not keep data forever by default. Retention controls how long records stay in a partition before they are deleted — either after a fixed amount of time (for example 7 days) or once the partition passes a size limit, whichever comes first. There is a second retention mode called log compaction: instead of deleting old records after a time limit, a compacted topic keeps only the latest record for each key, forever, and quietly removes older records for the same key in the background. Compaction is useful when a topic represents "current state" (like the latest known address for each customer ID) rather than a full history of events.

All of this is visible from the command line — no code required. Creating the 3-partition topic from the diagram above, then looking at what you got, is two commands:

# create a topic with 3 partitions and replication factor 3
kafka-topics.sh --create --topic orders \
  --partitions 3 --replication-factor 3 \
  --bootstrap-server localhost:9092

# describe it — shows each partition's leader, replicas, and ISR
kafka-topics.sh --describe --topic orders \
  --bootstrap-server localhost:9092
# Topic: orders  PartitionCount: 3  ReplicationFactor: 3
#   Partition: 0  Leader: 1  Replicas: 1,2,3  Isr: 1,2,3
#   Partition: 1  Leader: 2  Replicas: 2,3,1  Isr: 2,3,1
#   Partition: 2  Leader: 3  Replicas: 3,1,2  Isr: 3,1,2

03. Kafka Producers

How a record gets from your code into a partition

A producer is any piece of code that sends records into Kafka. You give it a ProducerRecord: a topic name, an optional key, a value, and optionally a specific partition. Everything else — serialization, choosing a partition, batching, compression, retries — is handled by the producer client for you, based on configuration.

producer = KafkaProducer(
    bootstrap_servers="broker1:9092",
    key_serializer=str.encode,
    value_serializer=lambda v: json.dumps(v).encode(),
    acks="all",                 # wait for leader + in-sync replicas (section 05)
    enable_idempotence=True,    # dedupe retries at the broker (see below)
    compression_type="lz4",
)

producer.send(
    "transactions",
    key="customer-482",         # same key -> same partition -> stays in order
    value={"amount": 250000, "currency": "VND"},
)
producer.flush()  # block until this producer's queued records are sent

Serializers

Kafka only stores and moves raw bytes — it does not know about your programming language's objects. A serializer converts your key and value into bytes before sending, and a matching deserializer turns bytes back into objects on the consumer side. The simplest serializer just converts a string to bytes. A very common production choice is Avro — a compact binary format that stores data next to a schema describing its fields, instead of human-readable text like JSON — paired with a Schema Registry — a separate service that stores the exact structure ("schema") of your data, so producers and consumers can check they agree on the shape of a record, and schemas can evolve safely over time.

How does a producer decide which partition a record goes to? If you give it a key, Kafka hashes that key and always sends records with the same key to the same partition — roughly, partition = hash(key) % number_of_partitions — this is how you keep related events (like all events for one customer) in order. If there is no key, Kafka spreads records across partitions using a sticky/round-robin approach to balance load evenly. You can also write a custom partitioner for special cases.

Producers do not send one record at a time over the network — that would be slow. They collect records into small batches, controlled by batch.size (max bytes per batch) and linger.ms (how many extra milliseconds to wait, hoping more records arrive to fill the batch) — plus buffer.memory, the total memory the producer can use to hold unsent batches. Batches can also be compressed (gzip, snappy, lz4, or zstd) before sending, trading a bit of CPU time for much less data over the network and less disk space on the broker.

acks=0Nothing — the producer does not wait for any confirmation at all.Fastest, but a record can be silently lost.

If a send fails or times out, the producer can automatically retry (controlled by retries and an overall delivery.timeout.ms budget). But retries create a new risk: what if the first attempt actually succeeded, and only the acknowledgment got lost — now the retry creates a duplicate record? This is exactly what the idempotent producer fixes.

Idempotent producer

Turning on enable.idempotence=true gives the producer a unique Producer ID (PID) and a per-partition sequence number that increases with every record. If the broker ever sees the same PID and sequence number twice, it knows it is a retried duplicate and simply drops it — the original record is never written twice. This does not cost you anything extra to use and is recommended as the default in modern Kafka.

Idempotent producers stop duplicates from retries on a single partition. They do not, by themselves, make a write across multiple partitions or topics atomic — for that you need Kafka transactions, covered fully in section 05 (right after we cover consumers next).

04. Kafka Consumers

Consumer groups, offsets, and the rebalance dance

A consumer reads records from one or more partitions by repeatedly calling poll() — a loop that says "give me whatever new records are available." Consumers almost always run as part of a consumer group: a named set of consumers that cooperate to read a topic, splitting the partitions between them. Kafka guarantees that within one group, each partition is read by exactly one consumer at a time — this is what makes consumer groups parallel and safe at the same time. It also means your maximum useful parallelism for one group is capped by the number of partitions: 8 partitions can keep at most 8 consumers in that group busy.

consumer = KafkaConsumer(
    "orders",
    bootstrap_servers="broker1:9092",
    group_id="fraud-checkers",   # same group_id = same consumer group
    enable_auto_commit=False,    # we'll commit manually, after processing
    auto_offset_reset="earliest",
)

while True:
    records = consumer.poll(timeout_ms=1000)
    for topic_partition, batch in records.items():
        for record in batch:
            process(record.key, record.value)   # your business logic
    consumer.commit()   # commit AFTER processing -> leans at-least-once

Kafka needs to remember how far each group has read in each partition — this is the committed offset, stored in a special internal topic called __consumer_offsets. You can let the client auto-commit periodically, or commit manually after you finish processing. The timing of that commit changes your delivery guarantee:

Commit after processingOffset was not saved yet, so on restart the same record is read and processed again.At-least-once

The rebalance protocol itself has evolved through three stages (a cluster uses one at a time — newer replaces older, they are not layered on top of each other): the classic "eager" protocol is stop-the-world — every consumer in the group pauses, gives up all its partitions, and waits for a brand-new assignment, even consumers whose partitions were not actually changing; the cooperative-sticky assignor improved this by only moving the partitions that actually need to move, letting everyone else keep working; and Kafka 4.0's KIP-848 protocol goes further still, moving the assignment logic to the broker side for even faster, less disruptive rebalances — see section 11 for the full story.

Separately from all of that, static group membership — setting a stable group.instance.id — is an independent switch you can turn on with any of the three protocols above. It tells Kafka "this is the same consumer coming back after a short restart, don't reassign its partitions yet," which avoids triggering a rebalance at all for brief blips like a deploy or a pod restart.

05. Reliability &amp; Delivery Guarantees

ISR, min.insync.replicas, and the honest trade-offs behind "exactly-once"

Section 02 introduced replication: a partition has one leader and several followers. Not every follower is always fully caught up — maybe one is briefly slow or restarting. The set of replicas that are fully caught up with the leader right now is called the ISR (in-sync replicas). Kafka uses the ISR to decide how safe a write really is.

min.insync.replicas

This broker setting says: "when a producer uses acks=all, at least this many replicas (including the leader) must confirm the write before it counts as successful." A common durable setup is replication.factor=3 with min.insync.replicas=2 — the write survives losing any one broker.

Example: with min.insync.replicas=2, if only the leader is currently in sync (both followers are lagging), a write with acks=all will fail loudly instead of pretending to succeed with just one copy.

What if every in-sync replica dies at once (rare, but it happens — a whole rack or availability zone can fail together)? Kafka faces a hard choice: wait for one of them to come back, or let an out-of-date replica become the new leader so the topic keeps working. Allowing the out-of-date replica to take over is called unclean leader election — and it means any records that only existed on the dead, in-sync replicas are gone for good.

All of this — idempotent producers (section 03) plus transactions — is how Kafka achieves true exactly-once semantics (EOS). A transaction, identified by a transactional.id, lets a producer write to multiple partitions or topics as one atomic unit: either every write in the transaction becomes visible, or none of them do. Consumers opt in by setting isolation.level=read_committed, which means they simply skip over any writes that belong to a transaction that never committed.

producer = KafkaProducer(bootstrap_servers="broker1:9092", transactional_id="txn-writer-1")
producer.init_transactions()

producer.begin_transaction()
try:
    producer.send("partition-a-topic", value=event_a)
    producer.send("partition-b-topic", value=event_b)
    producer.commit_transaction()      # both writes become visible together
except Exception:
    producer.abort_transaction()       # neither write becomes visible

# on the reading side:
consumer = KafkaConsumer(bootstrap_servers="broker1:9092",
                          isolation_level="read_committed")  # skip aborted/uncommitted writes

06. Kafka Cluster Management &amp; KRaft

From "Kafka needs ZooKeeper" to "Kafka runs itself"

For most of Kafka's history, a cluster needed a second distributed system called ZooKeeper running alongside it, just to keep track of cluster metadata — which broker is the controller (the broker responsible for cluster-wide decisions like electing partition leaders), what topics exist, their configs, and access control lists. Running two different distributed systems side by side to operate one Kafka cluster was operationally painful.

KRaft (KIP-500)

KRaft ("Kafka Raft") is Kafka's own built-in replacement for ZooKeeper: a small set of dedicated controller nodes reach agreement on cluster metadata using the Raft consensus algorithm — the same kind of algorithm many distributed databases use — instead of relying on a separate system. In plain terms, "reaching agreement" means: one controller is elected leader, and every metadata change (like "partition 3's leader is now broker 2") is a proposal that the leader sends to the other controllers; the change only becomes official once more than half of all controllers have written it down and confirmed it. That majority rule is exactly why you always run an odd number of controllers (3 or 5) — it guarantees there can never be two different groups that both claim to have a majority at the same time. Real timeline: KRaft appeared as early access in Kafka 2.8, became production-ready from Kafka 3.3, became the default for new clusters from 3.5, and ZooKeeper mode was fully removed in Kafka 4.0 (2024). If you are starting a new cluster today, you are on KRaft, full stop.

A few more operational basics: you can create a topic with a chosen number of partitions and replication factor, and later increase its partition count — but you can never decrease it. Here's the honest nuance: remember partition = hash(key) % number_of_partitions from section 03 — changing that number at all means new records for an existing key can land in a different partition than that key's older records already sit in, whether you go up or down. Kafka accepts that trade-off when you increase (existing records simply stay where they are; only new ones shift), because it's a one-way, additive change with a clear "before" and "after." Decreasing has no such story: Kafka would have to merge existing partitions' data back together, and it simply does not implement any mechanism to do that — so shrinking a topic's partition count isn't offered as an operation at all, not even as something risky you could opt into. Broker settings can be changed dynamically, without a restart. Client quotas let you cap how many bytes per second a specific user or application can produce or consume, so one noisy client cannot starve everyone else on a shared cluster.

# increase partitions (one-way — you can't go back down)
kafka-topics.sh --alter --topic orders --partitions 6 \
  --bootstrap-server localhost:9092

# change a broker config dynamically, no restart needed
kafka-configs.sh --alter --entity-type brokers --entity-name 1 \
  --add-config log.retention.hours=168 --bootstrap-server localhost:9092

# cap a noisy client to 5 MB/s produce, 10 MB/s consume
kafka-configs.sh --alter --entity-type clients --entity-name reporting-service \
  --add-config 'producer_byte_rate=5242880,consumer_byte_rate=10485760' \
  --bootstrap-server localhost:9092

Tiered storage (KIP-405)

Reached general availability around Kafka 3.9. Normally, every byte a broker keeps sits on its own local disk — which gets expensive if you want to retain data for a long time. Tiered storage automatically moves older log segments off to cheap object storage (an S3-style service) while keeping only the newest, most-accessed segments on fast local disk. Brokers need far less local disk, and you can afford much longer retention.

07. Kafka Connect

Moving data in and out of Kafka with configuration, not code

Kafka Connect lets you move data into and out of Kafka using only configuration files — no custom producer or consumer code required. A source connector pulls data from an external system into a Kafka topic (for example, Debezium reading a database's change log). A sink connector pushes data from a Kafka topic out to an external system (for example, writing to Amazon S3 or Elasticsearch).

Connect runs as one or more workers — JVM processes that actually execute connectors. Standalone mode runs a single worker process: simple, but with no fault tolerance, so it is best for local development and testing. Distributed mode runs a cluster of workers that share the load and automatically reassign work if a worker crashes — this is what production deployments use. A single connector definition can be split into multiple parallel tasks (for example, one task per database table), and those tasks get spread across the available workers.

A connector is nothing but a JSON config posted to Connect's REST API — no code to write or deploy. Here's a JDBC source connector, the kind that polls a database table and streams new rows into a topic:

curl -X POST http://connect-worker:8083/connectors -H "Content-Type: application/json" -d '
{
  "name": "orders-jdbc-source",
  "config": {
    "connector.class": "io.confluent.connect.jdbc.JdbcSourceConnector",
    "connection.url": "jdbc:postgresql://db:5432/bank",
    "table.whitelist": "orders",
    "mode": "timestamp+incrementing",
    "topic.prefix": "pg-",
    "tasks.max": "3",
    "transforms": "maskAccount",
    "transforms.maskAccount.type": "org.apache.kafka.connect.transforms.MaskField$Value",
    "transforms.maskAccount.fields": "account_number"
  }
}'

Two more pieces worth knowing: converters control how Connect serializes and deserializes record data as it crosses the boundary — JSON converter and Avro-with-Schema-Registry converter are the two most common choices. Single Message Transforms (SMTs) are small, chainable, declaratively-configured changes applied to one record at a time — masking a sensitive field, renaming a field, or dropping a field — without writing a custom connector. If a record fails to process, it can be routed to a dead letter queue (a separate topic) instead of crashing the whole connector, so you can inspect and fix bad records later without losing them.

08. Kafka Streams &amp; ksqlDB

Processing events without leaving your own application

Kafka Streams is a Java/Scala library you add to your own application — unlike Spark, there is no separate cluster to deploy and operate. You just run your app, and it reads from Kafka, processes, and writes back to Kafka (or elsewhere).

The stream-table duality: KStream vs. KTable

A KStream is an unbounded log of individual events — like a bank's list of every single transaction, one row per swipe. A KTable is a compacted view that only shows the latest value per key — like a bank's current account balances, one row per account. Both can be built from the very same underlying topic; they are just two different ways of looking at it. A GlobalKTable is a KTable that gets fully copied to every instance of your application — handy for small reference/lookup data that every instance needs a full local copy of, so a join against it never needs a network call to another instance (this is the same "keep a local copy so lookups are instant" motivation behind the stream-static joins you'll see in section 18).

A Kafka Streams application is built as a topology — a graph of source nodes (where data enters), processor nodes (where it is transformed), and sink nodes (where results go out). Stateless operations like map, filter, and flatMap need no memory of previous records. Stateful operations like count, aggregate, and join do — they must remember something across many events. Windowed aggregations (tumbling, hopping/sliding, session) work the same conceptual way here as they do in Spark Structured Streaming — see section 16 for the full detail with diagrams, since the two systems share the exact same windowing ideas.

StreamsBuilder builder = new StreamsBuilder();

KStream<String, Order> orders = builder.stream("orders");

KTable<String, Long> countPerCustomer = orders
    .filter((key, order) -> order.getAmount() > 0)   // stateless
    .groupByKey()
    .count();                                        // stateful -> needs a state store

countPerCustomer.toStream().to("order-counts");

KafkaStreams streams = new KafkaStreams(builder.build(), streamsConfig);
streams.start();   // this is your whole app -- no cluster to submit it to

Where does that "memory" actually live? In a local state store, backed by an embedded database called RocksDB, running right inside your application instance. To survive a crash, every change to the state store is also written to an internal changelog topic in Kafka — so if your instance dies and restarts (or moves to another machine), it can rebuild its state store exactly by replaying the changelog. Some deployments also keep a standby replica of the state pre-loaded on another instance, so failover is nearly instant instead of waiting for a full changelog replay.

Kafka Streams also supports exactly-once processing (processing.guarantee=exactly_once_v2), built directly on the Kafka transactions from section 05, and interactive queries, which let an external caller query a running application's local state store directly (for example, to build a simple "what's the current count for this key" API). If writing Java feels like overkill for your use case, ksqlDB gives you the same stream-table ideas through a SQL-like query language, so you can build streaming pipelines without writing or deploying any application code at all — for example, CREATE TABLE balances AS SELECT account_id, LATEST_BY_OFFSET(amount) FROM transactions GROUP BY account_id; creates a KTable-style running balance per account using one SQL statement, no Java project required.

09. Kafka Security &amp; Monitoring

Who is allowed to talk to your cluster, and how do you know it is healthy

Everything we have covered so far assumed a friendly network where any client can connect to any broker and read or write any topic. Real clusters — especially at a bank — cannot work that way. A production Kafka cluster needs three separate protections, and it helps to keep them separate in your head because they solve different problems: encryption (can someone snooping on the network read the bytes?), authentication (is the client really who it claims to be?), and authorization (is that identity allowed to do this specific thing?). On top of that, you need monitoring so you find out about problems before customers do.

Encryption in transit — TLS (Transport Layer Security)

TLS scrambles the bytes travelling between a client and a broker (or between brokers) so that anyone capturing network traffic sees only ciphertext, not your data. Kafka supports TLS on its network listeners the same way a web browser uses HTTPS. Without it, a producer sending a customer's transaction record is putting that record on the wire in plain, readable text.

Example: a listener configured as SSL://broker1:9093 only accepts encrypted connections on port 9093.

TLS answers "is this connection private," but it does not by itself answer "who is on the other end." That is the job of authentication, and Kafka delegates it to SASL (Simple Authentication and Security Layer) — a framework that plugs different login mechanisms into the same network protocol. Kafka supports several SASL mechanisms; you pick one (or mix a few) depending on what identity system your organization already has.

Authentication tells the broker "this connection belongs to user reporting-service." It says nothing about what reporting-service is allowed to do. That is authorization, and in Kafka it is handled by ACLs (Access Control Lists) — simple rules of the shape "this principal can do this operation on this resource." A principal is just the authenticated identity (a user or service account).

ACL (Access Control List)

A rule stored by Kafka's authorizer that grants (or denies) one operation on one resource to one principal. Operations include things like Read, Write, Create, Describe. Resources include topics, consumer groups, and the cluster itself.

Example: "user reporting-service can Read topic transactions" — and nothing else. That same user would be denied if it tried to Write to that topic or Read a different one.

# client-side config combining TLS + SASL/SCRAM (a producer or consumer property file)
security.protocol=SASL_SSL

• Encryption at rest — protecting the log files sitting on a broker's disk. Kafka itself mostly leaves this to the operating system or the cloud provider (disk-level or volume-level encryption), rather than encrypting the log segments itself. As a data engineer, this is usually a checkbox to verify with your infrastructure/security team rather than something you configure in Kafka directly — but for a bank handling customer transaction data, it's exactly the kind of control an auditor or regulator will ask you to prove is turned on.

Once security is in place, the next question is: is the cluster actually keeping up? The single most useful health number in Kafka is consumer lag. Lag tells you how far behind a consumer is, measured in records it has not processed yet.

Consumer lag

Lag = (latest offset written to a partition) − (offset the consumer has committed). It is a count of unprocessed records sitting in that partition, waiting for this consumer.

Example: a partition has written up to offset 1000. The consumer's last committed offset is 940. Lag = 1000 − 940 = 60 records behind.

Best practice: don't just look at lag in the moment — watch its trend. A lag of 5,000 that is shrinking every minute is a consumer catching up after a restart. A lag of 500 that keeps climbing is a consumer that is falling behind for good and needs attention now.

Consumer lag is only one of several signals worth putting on a dashboard. Here are the ones to know first, described in plain English:

Security is layered: TLS encrypts the wire, SASL proves identity, ACLs decide what that identity can touch. 10. Cross-Cluster Replication

Copying whole topics between separate Kafka clusters, not just between brokers in one

Everything so far has been about replication inside one cluster — copies of a partition sitting on different brokers so one broker failing does not lose data. Cross-cluster replication is a different, bigger idea: copying topics between two entirely separate Kafka clusters, which might be running in different data centers, different cities, or even different cloud regions.

There are a few concrete reasons a company sets this up:

• Disaster recovery — if an entire region or data center goes down, you can fail traffic over to a cluster that has a copy of the data.

• Geo-proximity — users in Ho Chi Minh City reading from a cluster in Vietnam is faster than reading from one in another continent.

• Data residency — some data has to legally stay within a country or region. This is genuinely relevant for a bank operating under Vietnamese regulations, and cross-cluster replication is one of the tools teams use to keep a required copy of data within the right jurisdiction — though the specific legal requirements for any given system are a question for your compliance and legal teams, not something to decide from a technical guide.

• Edge aggregation — collecting data from many small clusters (stores, branches, factories, ATMs) into one central cluster for analytics.

MirrorMaker 2 (MM2)

Kafka's standard tool for replicating between clusters. It is built on top of Kafka Connect (covered in the connectors section) — under the hood, MM2 literally is a set of Connect source connectors reading from one cluster and writing to another. Besides the topic data itself, MM2 can also replicate topic configuration, ACLs, and — importantly — consumer group offsets. That last part means a consumer that fails over to the other cluster can resume roughly where it left off, instead of starting from scratch.

• Topic naming — MM2 usually prefixes a replicated topic with the source cluster's alias, so a topic called transactions in a cluster aliased us-east shows up as us-east.transactions on the destination cluster. This stops a replicated topic from clashing with a same-named local topic.

# mm2.properties -- run with: connect-mirror-maker.sh mm2.properties
clusters = hanoi, hcmc
hanoi.bootstrap.servers = hanoi-broker:9092
hcmc.bootstrap.servers  = hcmc-broker:9092

# one-way replication: hanoi -> hcmc (active-passive DR)
hanoi->hcmc.enabled = true
hanoi->hcmc.topics = transactions|orders
hcmc->hanoi.enabled = false

sync.topic.configs.enabled = true
sync.topic.acls.enabled = true
emit.checkpoints.interval.seconds = 30   # keeps consumer offsets translated

There are two common shapes for a cross-cluster setup, and picking the right one matters a lot for how much operational care you will need later.

In an active-passive setup, only the primary cluster ever serves real traffic. The standby cluster exists purely to have a warm, mostly up-to-date copy ready in case the primary fails. It is the simpler of the two patterns to reason about, because data only ever flows in one direction.

Pitfall: active-active needs real care in two places. First, infinite replication loops — if cluster A replicates a topic to B, and B naively replicates it right back to A, the same records circle forever. MM2 guards against this, but the configuration has to be set up correctly, especially topic naming and replication filters. Second, offset translation — offset 940 on cluster A's copy of a topic does not mean the same record as offset 940 on cluster B's copy. This is because each cluster's partition is its own independent log with its own write history — cluster B assigns its own offsets as records arrive from MM2, on its own timeline, which will not line up number-for-number with cluster A's offsets. A consumer failing over from A to B needs an offset that has been translated (MM2 keeps a mapping for this), not just copied as-is.

Active-active lets both clusters serve live traffic but needs care around replication loops and offset translation. Edge aggregation just points several small clusters at one big central one.

11. What's New in Kafka (2024-2026)

The parts of Kafka that did not exist yet when most textbooks were written

Kafka keeps moving, and a few changes landed recently enough that older books and tutorials simply do not mention them. This section is a deliberate "catch-up" — four things anyone working with a modern Kafka cluster (roughly version 3.7 and up, and definitely version 4.0 and up) should know about.

Kafka 4.0 (2024): ZooKeeper is gone. As covered in section 06, Kafka used to depend on ZooKeeper (an external coordination service) to track cluster metadata, and then migrated to its own built-in replacement called KRaft. Kafka 4.0 finishes that migration: ZooKeeper mode has been fully removed. Every Kafka 4.0+ cluster is KRaft-only. If you see a tutorial that starts with "first, start ZooKeeper," it was written for an older version.

The next change fixes a problem every team running consumer groups eventually feels: rebalances used to be disruptive for everyone, even consumers who had nothing to do with the change that triggered the rebalance.

KIP-848 — the next-generation consumer group protocol

A KIP (Kafka Improvement Proposal) is Apache Kafka's formal design-document process for significant changes — this is KIP number 848. The classic rebalance protocol is called eager, and it is stop-the-world: the moment group membership changes (a consumer joins, leaves, or crashes), every consumer in the group gives up all of its partitions and pauses, the group coordinator recomputes the whole assignment from scratch, and only then does everyone resume — even consumers whose own assignment did not actually change. KIP-848 replaces this with a protocol where the group coordinator, running on the broker, computes assignments incrementally and pushes out only the changes that are actually needed. A consumer whose partitions are unaffected just keeps consuming. The result is rebalances that are faster and far less disruptive.

The last major addition is not a tweak to an existing feature — it is a genuinely new way to consume from Kafka.

KIP-932 — "Queues for Kafka" / share groups

Recall from section 04 that in a normal consumer group, each partition is assigned to exactly one consumer at a time — that is how Kafka guarantees order within a partition. A share group breaks that rule on purpose: multiple consumers in a share group can read from the same partition at the same time. Each individual record is handed to one consumer and must be acknowledged by that consumer, record by record — much closer to how a traditional message queue (like RabbitMQ or SQS) works. This trades away strict per-partition ordering guarantees in exchange for being able to spread work across more consumers than you have partitions, which is exactly what you want for workloads like "process this stream of independent jobs as fast as possible" where the order between jobs does not matter.

Example: a topic with 3 partitions, but a workload that needs 12 workers to keep up. A share group can spread records across all 12, regardless of partition count — something a classic consumer group cannot do, since it would leave 9 consumers with no partition at all.

# the only real difference from section 04's consumer code: group.protocol=share
consumer = KafkaConsumer(
    "notification-jobs",
    bootstrap_servers="broker1:9092",
    group_id="notification-workers",
    group_protocol="share",   # opt into share-group semantics (Kafka 4.0+)
)

while True:
    records = consumer.poll(timeout_ms=1000)
    for record in records:
        process(record.value)
        consumer.acknowledge(record)   # per-record ack, not a partition-wide offset commit

Don't confuse the two models. A share group is a deliberate trade: you get to spread work across more consumers than partitions, but you give up the strict per-key ordering that a normal consumer group guarantees within a partition. Use share groups for independent units of work (e.g. "process this notification job"), not for anything where the order of records for the same key matters.

Two more things worth knowing about, more briefly. Tiered storage — the ability to move older log segments off broker-attached disks and into cheaper object storage like S3, while Kafka still serves reads for that data transparently — reached general availability, building on what was introduced as covered in section 06. And more broadly, the wider streaming ecosystem is trending toward what people are starting to call "diskless" or cloud-native Kafka-compatible engines — systems that speak the Kafka protocol but write data directly to object storage instead of broker-attached disks, aiming for lower operating cost at large scale. This is worth knowing as a direction the industry is exploring; it is not (as of this writing) a settled standard, and it does not replace Kafka itself — just something to keep an eye on as the ecosystem matures.

Kafka in 2026 is KRaft-only, rebalances faster, and — for the first time — lets more than one consumer share a single partition's work.

❯ Short version

Kafka 4.0 removed ZooKeeper for good — KRaft only. KIP-848 replaced the old stop-the-world "eager" rebalance with an incremental one where unaffected consumers keep working. KIP-932 introduced share groups, a queue-like consumption model where multiple consumers split the work of one partition, record by record. Tiered storage is now GA, and "diskless" object-storage-native engines are an industry direction worth watching, not a Kafka replacement.

12. Spark Structured Streaming Fundamentals

Treat the stream as a table that never stops growing — everything else follows from that one idea.

So far this guide has been about Kafka — the pipes that move records around and store them durably. Now we get to Spark Structured Streaming, the engine most teams use to actually read those records, transform them, and write results somewhere useful. Before touching any code, there is exactly one mental model you need to internalize, because every other feature in this guide (output modes, watermarks, joins) is just a consequence of it.

The unbounded table

Structured Streaming asks you to imagine the incoming stream not as "events flying past" but as a table — an ordinary table with rows and columns, like something you'd query with SQL — except new rows keep getting appended to the bottom of it, forever. It never finishes loading. That's what "unbounded" means: there is no final row, ever, as long as the stream is running.

Example: a Kafka topic of payment events becomes, from Spark's point of view, a table where each new payment is one more row appended at the bottom, one row (or a batch of rows) at a time.

Why does this matter so much? Because once you think of the stream as a table, you can reuse the exact same DataFrame API that Spark already uses for ordinary, one-shot batch jobs — filter, groupBy, join, select, all of it. A streaming query and a batch query over the same data can share nearly identical code. The only difference is how you start reading and how you start writing: spark.read becomes spark.readStream, and df.write becomes df.writeStream. Everything in between — the transformation logic — stays the same. This is the single biggest reason Structured Streaming feels approachable once you already know batch Spark: you are not learning a new programming model, you are learning two new verbs.

So how does Spark actually turn "an ever-growing table" into something that runs on real machines with finite memory? By default, through micro-batch execution: instead of processing one record the instant it arrives, Spark waits a short moment, grabs whatever new data has shown up since the last check, and processes that whole chunk as one small batch job — reusing the exact same query engine that powers regular, non-streaming Spark SQL (the parts of Spark that plan an efficient execution order and run it fast on real hardware; you don't need to know their internal names to use Structured Streaming, only that streaming gets all the same performance work for free). Then it repeats. A micro-batch might take 100 milliseconds or 10 seconds depending on how you configure it; either way, from the outside it looks continuous, but under the hood it is a fast, tight loop of small batch jobs.

That checkpoint is worth pausing on, because it's what makes a streaming query resilient instead of fragile. A checkpoint is a small amount of state Spark writes to durable storage (usually a directory in cloud storage like S3, or in HDFS — the Hadoop Distributed File System, an older but still common way to store large files across many machines) after every micro-batch: which offsets it has already consumed from the source, and any intermediate aggregation state it's keeping. If the Spark application crashes or gets restarted for a deployment, it reads the checkpoint and picks up exactly where it left off — no reprocessing duplicates, no skipping data, as long as the source supports replay (which Kafka does, via offsets).

Heads-up — continuous processing mode exists too. Spark also ships an experimental continuous processing mode that skips micro-batching entirely and processes records one at a time, giving true millisecond-level latency instead of the small-but-nonzero delay of micro-batches. The catch: it only supports a much smaller set of operations (as of recent Spark versions, it does not support aggregations, and it does not support most kinds of joins) and has seen limited production adoption. We'll compare it side-by-side with micro-batch triggers in §14 and revisit it again in §19. For now, just know it exists as "the other mode" — 95%+ of real Structured Streaming jobs run on micro-batches.

❯ Short version

Structured Streaming = think of the stream as a table that keeps growing, write ordinary DataFrame code against it, and let Spark repeatedly run that code as small batch jobs (micro-batches) — checkpointing its progress after each one so it can resume safely after a restart.

A streaming query is just a batch query that Spark happens to run over and over, forever.

One place the pandas comparison breaks down: if you know pandas, "DataFrame" feels familiar — but a Spark DataFrame is not a pandas DataFrame. A pandas DataFrame lives entirely in one machine's memory and every operation runs immediately. A Spark DataFrame describes work that is lazy (nothing actually runs until you ask for output) and distributed (the real data is split across many machines and processed in parallel). The code often looks similar; what happens underneath is completely different.

13. Sources &amp; Sinks

Every streaming query needs somewhere to read from and somewhere to write to — here's the full menu of both.

Every Structured Streaming query has the same two-ended shape: a source you read from and a sink you write to, with your transformation logic in between. In code, this reads almost like plain English. One thing to know before you look at the example: Kafka stores every record as a raw key and value made of bytes — Kafka itself has no idea if those bytes are JSON, Avro, or anything else. So the very first step after reading from Kafka is almost always to cast those bytes to a string and parse them (here, as JSON) into real, typed columns you can filter and query. That's what the CAST and from_json calls below are doing — turning "meaningless bytes" into "a row with an amount column." The schema argument to from_json is just a description of what fields to expect in that JSON (e.g. "there's a field called amount that's a number"), written once earlier in your code using Spark's StructType. Here's a realistic (simplified) example reading from Kafka and writing to another Kafka topic:

df = (spark.readStream
        .format("kafka")
        .option("kafka.bootstrap.servers", "broker1:9092")
        .option("subscribe", "orders")
        .load())

# df.value is raw bytes here — cast to string, then parse the JSON into real columns
result = df.selectExpr("CAST(value AS STRING) as json") \
           .select(from_json("json", schema).alias("order")) \
           .filter("order.amount > 100")

(result.writeStream
       .format("kafka")
       .option("kafka.bootstrap.servers", "broker1:9092")
       .option("topic", "big-orders")
       .option("checkpointLocation", "/chk/big-orders")
       .start())

Notice the shape: spark.readStream.format(<source type>).option(...).load() starts the pipeline, then any number of ordinary DataFrame transformations, then .writeStream.format(<sink type>).option(...).start() finishes it. .start() is what actually kicks off the micro-batch loop from §12 — everything before it is just building up the plan, nothing runs until you call .start().

Let's go through the sources first — where data comes in:

• Kafka Kafka — the most common real-world source. As shown above, the first thing your query almost always does is deserialize the raw bytes — cast to a string and parse JSON with from_json(), or use an Avro/Schema Registry deserializer — turning raw bytes into proper typed columns.

• Files Spark — point Spark at a directory, and it watches for new files (Parquet, JSON, CSV, text) landing in it, treating each new file as more rows appended to the unbounded table. At small scale this is fine with the plain file source; at large scale, listing a directory over and over gets slow, which is exactly the problem Databricks' Auto Loader solves efficiently — more on that in §19.

• Socket Spark — reads lines of text from a TCP network socket (e.g. via nc -lk 9999 on your laptop). It has no fault tolerance and no offset tracking, so it exists purely for local tutorials and quick experiments.

• Rate Spark — generates fake rows (an incrementing counter plus a timestamp) at a rate you control, e.g. 10 rows/second. Useful for demos, benchmarking, and testing your pipeline logic without needing a real Kafka cluster running.

Pitfall — socket and rate sources in production. Both the socket source and the rate source are explicitly documented by Spark as testing-only. Neither survives a restart cleanly, and socket has no delivery guarantees at all. If you see either one outside a notebook or a CI test, that's a red flag.

Now the sinks — where results go out:

• Kafka Kafka — write your result rows back out as key/value bytes to another topic, commonly the starting point for a downstream consumer or another streaming job.

• Files Spark — write result rows as new files into a directory, typically Parquet, or increasingly a lakehouse table format like Delta Lake or Iceberg that adds ACID transactions on top of files — a database-style guarantee that a write either fully completes or has no effect at all, so concurrent readers and writers never see a half-finished, corrupted result.

• Console Spark — prints the result batch to standard output. Purely for local debugging; never wire this into anything a real user depends on.

• Memory Spark — keeps the result table in an in-memory table you can query with SQL from the same Spark session. Handy for interactive debugging and demos; the data disappears when the process stops.

foreachBatch — the escape hatch

foreachBatch is the sink you reach for whenever the built-in sinks (Kafka, files, console, memory) can't do what you need. Instead of writing rows for you, Spark hands you a plain, ordinary batch DataFrame for each micro-batch — and inside that function, you can do literally anything a regular batch Spark job could do: write to a relational database over JDBC, upsert into a Delta/Iceberg table, write to two or three destinations at once, or even call an external HTTP API for each row.

Example: foreachBatch(lambda df, batch_id: df.write.jdbc(url, "orders_table", mode="append")) — every micro-batch gets appended into a Postgres table using the ordinary batch JDBC writer.

Sources bring the unbounded table's new rows in (Kafka, files, socket, rate); sinks decide where the result table's rows go out (Kafka, files, console, memory); and foreachBatch is the universal escape hatch when none of the built-in sinks fit your target system.

14. Output Modes &amp; Triggers

Output mode decides WHAT gets written each micro-batch. Trigger decides WHEN a micro-batch happens.

These two settings are independent knobs you set on every writeStream, and beginners often mix them up because both affect "when do I see data." Output mode controls which rows of the result table are written to the sink on a given micro-batch. Trigger controls how often a micro-batch happens at all. Let's take output mode first.

Imagine a tiny streaming aggregation: SELECT key, COUNT(*) FROM stream GROUP BY key, tracking a running count per key. Say the result table currently holds three keys — A, B, C — and on this particular micro-batch, key B's count changed from 3 to 4, and key C just appeared for the first time. Here's what each output mode does with that:

• Append — only rows that are brand-new and final get written. This is the default, and it's required for plain non-aggregated streams (like a filter or a map with no groupBy). For windowed aggregations, Append also works, but only once a watermark (§15) guarantees a window's result is final — otherwise Spark would need to "un-write" a row it already emitted, which Append cannot do.

• Update — only the rows whose value actually changed since the last micro-batch get written. This is the natural fit for a continuously-updated aggregate, like a running count or running total per key, where you want the sink to reflect the latest number for each key without resending unchanged rows.

• Complete — the entire result table is rewritten to the sink on every micro-batch, unchanged rows included. This is only realistic when the result is small — a handful of aggregate groups, a small dashboard summary — because the cost of Complete mode scales with the size of the whole result table, not just what changed.

Pitfall — Append and unfinished aggregates don't mix. If you run a windowed aggregation in Append mode without setting a watermark, Spark will refuse to start the query (or, in older versions, silently never emit a window until the whole application stops). Append mode can only emit a row once it is sure that row is final and will never need to change again — and without a watermark, Spark has no way to know that. We'll fix this properly in §15.

Now for triggers — how often a micro-batch fires, independent of what gets written:

• Default (unspecified) — the next micro-batch starts immediately after the previous one finishes. This gives the lowest latency micro-batching can offer, but it also means the query is constantly busy, which can be wasteful if the source only has a trickle of new data most of the time.

• Fixed interval (ProcessingTime trigger) — you specify a period, e.g. trigger(processingTime="30 seconds"), and Spark fires a micro-batch every 30 seconds on the clock, even if the previous batch finished in 2 seconds (it just waits out the rest of the interval). This smooths out load and cost, at the price of slightly higher latency.

• Trigger.AvailableNow — processes everything currently sitting in the source, possibly across several micro-batches, and then stops the query on its own once it's caught up. This superseded the older Trigger.Once (which forced everything into a single giant micro-batch — painful if a lot of data had piled up). AvailableNow is the standard way to run a "streaming" job on a schedule, like a nightly or hourly cron job, getting incremental, checkpoint-tracked processing without paying for a cluster that runs 24/7.

• Continuous trigger — the trigger you'd use to opt into the experimental continuous processing mode from §12: true record-at-a-time, millisecond-level latency, but a much smaller set of supported operations. Full comparison in §19.

counts = df.groupBy("key").count()

(counts.writeStream
       .outputMode("update")                       # only changed rows, per micro-batch
       .trigger(processingTime="30 seconds")        # fire on a schedule, not ASAP
       .format("console")
       .start())

# same query, run once as an incremental "batch" job instead of a 24/7 cluster:
(counts.writeStream
       .outputMode("complete")
       .trigger(availableNow=True)                  # process the backlog, then stop
       .format("console")
       .start())

• Trigger.Once — deprecated, still works, forces all available data into one single micro-batch.

• Trigger.AvailableNow — recommended replacement, splits available data across multiple appropriately-sized micro-batches automatically.

• Why it matters — with a huge backlog, one giant batch can blow past executor memory; AvailableNow avoids that by chunking the catch-up work.

15. Event-Time &amp; Watermarking

Real data doesn't arrive in order — watermarks are how Spark decides when it's safe to stop waiting for stragglers.

Every event carries (at least) two different notions of time, and mixing them up is one of the most common sources of confusing streaming bugs.

Event-time vs. processing-time

Event-time is the timestamp of when something actually happened in the real world — the moment a customer tapped "pay" on their phone. Processing-time is the timestamp of when Spark actually received and started processing that record. In a perfect world these would be identical. In real systems, they are almost never exactly equal — the gap between them is called skew.

Example: a customer taps "pay" at 10:00:00 (event-time), but their phone was offline and only reconnects and sends the event at 10:04:30 (processing-time) — 4 minutes and 30 seconds of skew.

Why does this skew happen so often in practice? A few very concrete, very common reasons: a mobile phone loses signal, buffers every tap locally, and sends a burst of events all at once when it reconnects. A network hiccup causes a retry, delaying delivery by seconds. An upstream microservice is slow under load and only forwards an event minutes after it originally received it. None of these are exotic edge cases — they are just how distributed systems behave, which is why Structured Streaming has first-class support for handling them.

So if events can show up late and out of order, how does Spark ever decide an aggregation window is "done" and safe to emit? This is exactly the problem a watermark solves.

Watermark

A watermark is a moving line on the event-time axis. Concretely, it is defined as: the latest event-time Spark has seen so far, minus a delay threshold you configure. To be precise about scope: this is tracked once per streaming query, across the whole stream — not separately per key and not separately per window. Every key and every window in that query shares the same single watermark line. Any new record whose event-time is older than the current watermark is considered "too late" — for stateful operations like windowed aggregations, Spark is allowed to drop it, because it has already decided that time window is closed and finalized.

Example: df.withWatermark("event_time", "10 minutes") — plain English: "I'll wait up to 10 minutes past the latest event-time I've seen for stragglers to show up. After that, I consider that period closed and finalized."

counts = (df
    .withWatermark("event_time", "10 minutes")   # allow up to 10 min of lateness
    .groupBy("customer_id")
    .count())

# a record with event_time older than (max event_time seen - 10 minutes)
# is considered too late and dropped from this aggregation

Setting the threshold is a genuine trade-off, and there's no single correct number — it depends on how late your real data actually gets and how fast you need answers.

Bigger threshold (e.g. 24 hours). Catches almost all legitimately late data — great if your mobile clients can buffer for a long time offline. The cost: results are finalized much later, and Spark must keep a lot more aggregation state in memory (and spilled to disk) while it waits, since every open window has to stay "open" until the watermark passes it.

Smaller threshold (e.g. 2 minutes). Finalizes results fast and keeps far less state around, which is cheaper and simpler to run. The cost: any record that arrives more than 2 minutes late gets dropped from aggregations, even if it was a perfectly legitimate event.

A good starting point: look at your actual data's real-world lateness distribution (Kafka consumer lag plus client-side buffering behavior) before picking a number, and lean slightly generous at first — you can always tighten the threshold later once you understand your traffic, but data dropped by too small a watermark is gone for good.

❯ Short version

Event-time (when it happened) and processing-time (when Spark saw it) drift apart in real systems. A watermark tracks "latest event-time seen minus your allowed lateness" and lets Spark safely finalize and drop state for windows that are older than that line — bigger threshold = more complete but slower and heavier; smaller = faster and leaner but drops more late data.

A watermark isn't a guarantee no data is late — it's a deadline you set for how long you'll wait before moving on.

16. Windowing

Turning an endless stream into time-bounded buckets you can actually aggregate.

In batch processing, "count orders per customer" is a simple GROUP BY. In streaming, there is a problem: the stream never ends, so "count orders per customer" would have to keep updating forever — you could never print a final number. Windowing solves this by adding a second grouping key: time. Instead of "count per customer," you compute "count per customer, per 10-minute slice of time." Each 10-minute slice is a window, and once enough time has passed (thanks to the watermark you met in the previous section), Spark can treat that window's result as done and move on.

Spark Structured Streaming gives you three window types, all created inside a normal groupBy: tumbling, sliding, and session windows. They differ only in how the window boundaries are drawn — the aggregation logic (count, sum, average, etc.) inside them stays the same.

Tumbling window Spark

A tumbling window is a fixed-size, non-overlapping bucket of time. Every event belongs to exactly one window. Think of it as slicing the timeline into equal pieces, back to back, with no gaps and no overlap.

Example: groupBy(window(col("event_time"), "10 minutes")).count() — one row per distinct 10-minute slice, e.g. 10:00–10:10, 10:10–10:20, and so on.

Sliding window Spark

A sliding window is also fixed-size, but a new window starts more often than the window's own length — the "slide" interval is shorter than the window size. That means windows overlap, and a single event can fall inside more than one window at the same time. Sliding windows are useful for smoothed, rolling metrics, like "average price over the last 10 minutes, refreshed every 5 minutes."

Example: groupBy(window(col("event_time"), "10 minutes", "5 minutes")).avg("price") — the second string is the slide interval.

Session window Spark — since Spark 3.2

A session window has no fixed size at all. Instead, you configure a "gap" — a maximum amount of quiet time. A session stays open as long as new events for that key keep arriving before the gap passes. The moment the gap of inactivity passes with no new events, Spark closes the session and finalizes its result.

Example: groupBy(session_window(col("event_time"), "30 minutes")).count() — group a user's shopping-cart clicks into one session as long as they don't go quiet for more than 30 minutes.

# tumbling: one window per event, no overlap
tumbling = df.withWatermark("event_time", "5 minutes") \
    .groupBy(window("event_time", "10 minutes")) \
    .count()

# sliding: windows overlap because slide (5 min) < window size (10 min)
sliding = df.withWatermark("event_time", "5 minutes") \
    .groupBy(window("event_time", "10 minutes", "5 minutes")) \
    .avg("price")

# session: no fixed size, closes after 30 minutes of inactivity for that key
sessions = df.withWatermark("event_time", "5 minutes") \
    .groupBy("user_id", session_window("event_time", "30 minutes")) \
    .count()

Windows and watermarks are joined at the hip. A windowed aggregation's number for, say, the 10:00–10:10 window is only final once the watermark has passed 10:10. Before that, more late-but-still-acceptable data could still arrive and change the count. If you read a window's result from the output sink before the watermark clears it, you might be reading a number that will still change on a later micro-batch (unless you're using complete output mode, which recomputes everything).

❯ Short version

Tumbling = fixed, no overlap, one window per event. Sliding = fixed, overlapping, one event can hit several windows. Session = no fixed size, driven by an inactivity gap, closes when the gap passes. All three only become "final" once the watermark passes the window's end.

~/streaming ❯ spark: remembering things safely

17. Stateful Processing &amp; Fault Tolerance

How Spark remembers what happened in earlier micro-batches, and survives crashes without losing that memory.

Not every streaming operation needs memory. A simple select or filter can process each row independently and forget it immediately. But a running count, a deduplication check, or a windowed aggregation cannot work that way — they need to remember something from earlier micro-batches. A running count per customer needs to remember "the count so far" for every customer it has ever seen. This remembered information is called state, and any query that needs it is called a stateful query.

State store Spark

The state store is where that "memory" physically lives while the query is running. Spark ships with two implementations you choose between with a config setting:

• Default (in-memory) state store — state is kept as Java/Scala objects in the executor's memory. Fast, but only practical when the total state comfortably fits in RAM.

• RocksDB state store provider — available since Spark 3.2. State is kept in an embedded RocksDB (a fast key-value database library that runs inside the same process, storing data on local disk) database on each task's local disk. This lets state grow far larger than available memory, at the cost of a little extra read/write latency per key.

State living in memory or on local disk is not durable by itself — if an executor dies, that copy of the state dies with it. This is where checkpointing comes in. Structured Streaming periodically writes its progress to a checkpoint location — typically a reliable, durable place like cloud object storage (Amazon S3, Azure ADLS, GCS). Two things get written there:

• Offsets — exactly which records from the source (e.g. which Kafka offsets per partition) have already been processed.

• State snapshot — a durable copy of the state store's current contents.

If the query crashes and restarts — whether from a code bug, a cluster failure, or a planned redeploy — it reads both pieces from the checkpoint location and resumes from exactly that point. It does not reprocess everything from the beginning, and it does not silently skip data either.

Some sinks also rely on a write-ahead log (a durability technique where every intended write is first appended to a simple sequential log before being applied) so that, combined with checkpointed offsets, a sink can achieve exactly-once writes even across restarts. You don't usually configure this directly — it's part of how certain sinks implement their reliability guarantees internally.

Windows and simple aggregations cover a lot of ground, but sometimes you need fully custom logic per key — arbitrary business rules that don't fit a built-in aggregate function. For that, Structured Streaming offers arbitrary stateful processing through two operators (both used after a groupByKey):

• mapGroupsWithState — for each key, you write a function that reads the old state, computes new state, and returns exactly one output row per micro-batch, for every key that had new data.

• flatMapGroupsWithState — the same idea, but your function can return zero, one, or many output rows per key per micro-batch, which is more flexible when you don't always have something to emit.

Both let you set a GroupState timeout: a way of telling Spark "if this key has not seen new data for X amount of time, call my function one more time anyway, with no new input, so I can emit a final result and clean up." Without a timeout, state for a key that goes permanently inactive (e.g. a user who abandons a session and never returns) would sit in the state store forever, slowly growing memory or disk usage.

Unlike the rest of this guide's DataFrame code (filter, groupBy, .avg()), this is a different, more manual style: you write the per-key function body yourself, imperatively, instead of describing the result declaratively. Here's roughly what the shape of that function looks like for a running count per key:

def update_count(key, new_rows, state: GroupState):
    old_count = state.getOption or 0             # read old state (0 if new key)
    new_count = old_count + len(list(new_rows))
    state.update(new_count)                    # write new state
    return (key, new_count)                    # exactly one output row

counts = df.groupByKey(lambda row: row.user_id) \
           .mapGroupsWithState(update_count, GroupStateTimeout.NoTimeout)

That's the whole pattern: read old state in, compute new state, write it back, return your output row(s). Everything else (checkpointing that state, restoring it after a crash) is handled for you by Structured Streaming.

End-to-end exactly-once needs all three legs of the stool. "Exactly-once" is a query-wide property, not something one setting turns on. It requires: (a) a replayable source like Kafka, where you can re-read from a specific offset after a failure; (b) deterministic processing logic, so re-running the same input produces the same output; and (c) an idempotent or transactional sink, so writing the same result twice (which can happen after a retry) doesn't create duplicates. Drop any one of the three, and your pipeline quietly degrades to at-least-once — you may see duplicates after a failure, even if your Spark code looks correct.

~/streaming ❯ spark: combining streams

18. Streaming Joins

Joining a stream with a lookup table is easy. Joining a stream with another stream is a scheduling problem in disguise.

Structured Streaming supports two very different kinds of joins, and the difference between them matters a lot for how you configure your query.

Stream-static join Both / Integration

One side is a streaming DataFrame (unbounded, always growing); the other side is an ordinary static/batch DataFrame (bounded, finite, doesn't grow while the query runs). A common example is enriching a stream of raw transactions with a slow-changing customers or products lookup table. Spark re-reads or re-checks the static side as needed on each micro-batch, but since the static side is bounded, Spark never needs to buffer it waiting for more data — no watermark is required on it.

Example: transactionsStream.join(customersStaticDF, "customer_id").

customers_static = spark.read.format("delta").table("customers")   # bounded, read once (refreshed per batch)

enriched = transactions_stream.join(customers_static, "customer_id")   # no watermark needed on either side

Stream-stream join Spark

Here both sides are live, unbounded streams — for example, matching "order placed" events with "payment confirmed" events. This is genuinely harder than a stream-static join: Spark cannot wait forever hoping a match shows up on the other side, because that would mean buffering unmatched rows forever, growing state without bound. To make this workable, Spark requires a watermark on both streams, plus a time-range join condition — something like "the payment must arrive within 15 minutes of the order." That range tells Spark exactly how long it needs to keep buffering an unmatched row before it's safe to give up on it.

Example: orders.join(payments, expr("payments.order_id = orders.order_id AND payment_time BETWEEN order_time AND order_time + interval 15 minutes")).

orders = orders_stream.withWatermark("order_time", "20 minutes")
payments = payments_stream.withWatermark("payment_time", "20 minutes")

matched = orders.join(
    payments,
    expr("""
        payments.order_id = orders.order_id AND
        payment_time BETWEEN order_time AND order_time + interval 15 minutes
    """),
    "leftOuter"   # unmatched orders still emitted (with null payment) once watermark clears them
)

The join type then decides what happens to a row that never finds a match within that window:

• Inner stream-stream join — an unmatched row is simply never emitted. If a payment never shows up within 15 minutes, that order just silently never appears in the output.

• Outer stream-stream join (left / right / full) — an unmatched row is eventually emitted, with nulls filled in for the missing side, but only once the watermark has advanced far enough that Spark is sure no late match is still on its way.

No watermark, no stream-stream join. If you try a stream-stream join without watermarks on both sides and a time-range condition, Spark will refuse to run the query (or, in older/looser configurations, buffer state without bound until you run out of memory or disk). The time-range condition is not optional decoration — it's the mechanism that tells Spark when it's safe to stop waiting.

19. What's New in Spark Structured Streaming (2024-2026)

The classic textbooks stop around Spark 3.x. Here's what changed since — this is the part that keeps this guide from being a museum piece.

Most Spark books (including the well-known Learning Spark) describe mapGroupsWithState and flatMapGroupsWithState as the way to do custom stateful logic, and full RocksDB snapshots as the way state gets checkpointed. Both are still valid and still ship in Spark — but Spark 4.0 (released 2025) and recent Databricks runtimes added real, documented upgrades on top of them. This section covers five of them.

transformWithState Spark 4.0

This is the modern replacement for mapGroupsWithState / flatMapGroupsWithState from the previous section. It does the same basic job — custom per-key logic across micro-batches — but fixes several rough edges. There's also a Pandas variant, TransformWithStateInPandas, for PySpark users who want to write their state logic with pandas data structures instead of plain Python objects.

• Richer state types — instead of one simple object per key, you can keep several named, independently-typed pieces of state per key (e.g. a value, a list, and a map, all at once).

• Built-in TTL (time-to-live) — you can tell Spark "expire this piece of state automatically after X time" without writing manual timeout bookkeeping yourself.

• Explicit timers — you can register and cancel your own timers per key/state, instead of relying only on one global per-key timeout.

• Graceful schema evolution — changing the shape of your state between app versions (e.g. adding a field) is handled far more gracefully than before, where it could break restarts from an old checkpoint.

class FraudCounter(StatefulProcessor):
    def init(self, handle):
        # named, typed state -- with a TTL, so inactive accounts clean up on their own
        self.count = handle.getValueState("count", IntegerType(), ttl_duration=timedelta(hours=1))

    def handleInputRows(self, key, rows, timer_values):
        new_count = (self.count.get() or 0) + len(list(rows))
        self.count.update(new_count)
        yield Row(account_id=key[0], swipe_count=new_count)

counts = df.groupBy("account_id").transformWithState(
    FraudCounter(), outputMode="update", timeMode="ProcessingTime"
)

RocksDB changelog checkpointing Spark 3.4+

Recall from the previous section that RocksDB lets state grow larger than memory by spilling to local disk. The remaining problem was checkpointing: writing a full snapshot of a large RocksDB state store on every checkpoint is slow, and that slowness shows up as extra latency in every micro-batch. Changelog checkpointing fixes this by writing small, incremental "changelog" records of just what changed since the last checkpoint, and only occasionally writing a full snapshot. The win is in normal operation, not recovery: every micro-batch now writes a small diff instead of a large snapshot, so steady-state checkpointing latency drops noticeably. Recovery after a failure still has to replay the last full snapshot plus whatever changelogs came after it — that part isn't magically faster, which is exactly why Spark keeps taking full snapshots periodically in the background: it bounds how many changelogs would ever need replaying, so a crash doesn't force you to replay an unbounded pile of tiny diffs.

Two patterns from earlier sections have also become much more mainstream in production over the last couple of years:

• Trigger.AvailableNow — recap from an earlier section: this trigger processes exactly whatever data is currently available, as a bounded batch of micro-batches, then stops the query on its own. It has become the go-to pattern for "streaming pipeline that only needs to run periodically" (e.g. every hour via a scheduler), because it reuses all your streaming code and checkpoint logic without needing a query that runs forever.

• Spark Connect for streaming — Spark Connect is a thin client protocol that lets an application talk to a remote Spark cluster over a lightweight gRPC-based connection (gRPC is just a common, efficient way for one program to call functions on another program over a network — think of it as a fast, modern alternative to a REST API), without needing a full Spark driver running locally. Streaming queries can now be started and managed through Spark Connect, which makes it much easier to trigger and monitor streaming jobs from small client applications, notebooks, or services that shouldn't have to embed a full Spark installation.

Lakeflow Declarative Pipelines Databricks, 2024 rebrand of Delta Live Tables

Lakeflow Declarative Pipelines is Databricks' evolution of what used to be called Delta Live Tables (DLT). The idea: instead of writing imperative Structured Streaming code that says exactly how to read, transform, and write each table, you declare — in SQL or Python — what tables you want and whether each one is streaming or batch. The system then figures out how to run it: it automatically manages checkpoints, retries failed runs, and works out the correct dependency order between tables (so a downstream table always runs after the upstream table it reads from is up to date). Under the hood, the streaming tables in a Lakeflow pipeline are still built on Structured Streaming — it's a declarative layer on top, not a replacement engine.

Two more Delta Lake features come up constantly once you pair Structured Streaming with a lakehouse table format:

• Auto Loader — an efficient way to incrementally discover new files landing in cloud storage (e.g. new Parquet or JSON files arriving in an S3 bucket) without listing the entire directory tree on every micro-batch. It tracks which files it has already processed, so it scales to directories with millions of files instead of getting slower over time.

• Change Data Feed (CDF) — a Delta Lake feature that lets you stream out just the row-level changes — inserts, updates, and deletes — that happened to a Delta table, instead of downstream jobs having to re-read the whole table to figure out what changed. This is what makes it practical to chain several Delta tables together in a streaming fashion, each one reacting only to the actual changes in the table before it.

Iceberg is catching up fast. Apache Iceberg — a competing open table format — is increasingly supported as both a streaming source and a streaming sink in Spark Structured Streaming, in much the same way Delta Lake already works. If you work at a shop that standardized on Iceberg instead of Delta, expect the streaming read/write story to keep converging with what's described here for Delta.

20. Flink Architecture &amp; Core Concepts

Built stream-first from day one — batch is just a stream that happens to end

Apache Flink is a distributed engine for processing unbounded and bounded data streams. The key difference from everything you've read so far in this guide: Spark Structured Streaming starts from a batch engine and adds a micro-batch loop on top, so a "stream" is really a fast-arriving sequence of tiny batches. Flink flips that around — its runtime is a true event-at-a-time, pipelined dataflow engine, and a bounded batch job is treated as a special case (a stream that happens to end) rather than the other way around. That single design choice is why Flink typically reaches lower latency than Spark's micro-batches, and why its notion of time, state, and windows (sections 22-23) is considered the most complete of the three engines you've now met.

A Flink program is written against an API (most commonly the DataStream API, section 21, or Table API / Flink SQL, section 24) and compiled into a JobGraph — a graph of operators (source, map, keyBy, window, sink) connected by data streams. That JobGraph is submitted to a Flink cluster, which schedules it, runs it continuously, and keeps it alive indefinitely for a true streaming job.

JobManager vs. TaskManager

A Flink cluster has two kinds of processes. The JobManager is the brain: it accepts submitted jobs, turns the JobGraph into an ExecutionGraph (the JobGraph with parallelism expanded out), schedules tasks onto workers, coordinates checkpoints (section 23), and handles failure recovery. A TaskManager is a worker process that actually executes tasks and holds data in memory/state on local disk; each TaskManager offers one or more task slots, and a slot is the unit of resource a single parallel slice of the job runs in.

Example: think of the JobManager as an air traffic controller and TaskManagers as the actual planes (workers) — the controller decides who flies where and reroutes around problems, but never carries any passengers (data) itself.

A job's parallelism is how many parallel instances each operator runs as. If a keyBy+window stage has parallelism 8, Flink runs 8 copies of that operator, each owning a disjoint slice of the keyspace (exactly like Kafka partitions are divided among consumers, section 04, or how Spark shuffles data across partitions for a groupBy). Operators that don't need a shuffle between them — a map feeding straight into a filter, for instance — get fused into a single task via operator chaining, running in one thread with no network or serialization overhead between them. A shuffle-like operation (any keyBy, rebalance, or explicit repartition) always breaks the chain, because data has to move between parallel instances.

Where Flink shows up in the real world

• Sub-second fraud & risk scoring — payment networks and banks scoring a transaction against dozens of signals before it completes.

• Real-time feature pipelines for ML/recommendation — computing features (click counts, session state) the instant a user acts, so a recommendation model sees fresh signal.

• Ride-hailing ETA and pricing — continuously recomputing estimates as GPS pings and demand signals stream in, event by event, not batch by batch.

• Complex event processing (CEP) — detecting patterns across a sequence of events (e.g. "three failed logins from the same device in 30 seconds") with strict ordering and timing guarantees.

• Large-scale change data capture pipelines — Flink CDC (section 24) reading database change logs from thousands of tables and fanning them out with low, predictable latency.

❯ Short version

Flink is a true event-at-a-time streaming engine, not a micro-batch engine — batch is just a bounded stream. A JobManager (Dispatcher + ResourceManager + one JobMaster per job) coordinates; TaskManagers run the actual work in task slots. Operators chain into one task when there's no shuffle between them, and split into separate stages whenever a keyBy or similar operation forces data to move across the network.

21. The Flink DataStream API

Building a topology by hand — sources, transformations, and low-level process functions

The DataStream API (Java, Scala, or Python) is Flink's general-purpose, code-first API — the rough equivalent of Kafka Streams' KStream/KTable API (section 08), but running on a real cluster instead of embedded inside your own application. A program starts from a StreamExecutionEnvironment, adds a source, chains transformations, and ends with one or more sinks. Nothing actually runs until you call execute() — up to that point you're just building the JobGraph described in section 20.

StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();

DataStream<Transaction> txns = env.fromSource(
    KafkaSource.<Transaction>builder()
        .setBootstrapServers("broker:9092")
        .setTopics("transactions")
        .setValueOnlyDeserializer(new TransactionDeserializer())
        .setStartingOffsets(OffsetsInitializer.latest())
        .build(),
    WatermarkStrategy.<Transaction>forBoundedOutOfOrderness(Duration.ofSeconds(5))
        .withTimestampAssigner((t, ts) -> t.getEventTimeMillis()),
    "transactions-source");

DataStream<Alert> alerts = txns
    .filter(t -> t.getAmount() > 0)                       // stateless
    .keyBy(Transaction::getAccountId)                      // partition by key, like a Kafka partition key
    .window(TumblingEventTimeWindows.of(Time.minutes(2)))  // section 22
    .aggregate(new CountAndFlagAggregator());              // stateful -> needs state, section 23

alerts.sinkTo(KafkaSink.<Alert>builder()
    .setBootstrapServers("broker:9092")
    .setRecordSerializer(alertSerializer)
    .setDeliveryGuarantee(DeliveryGuarantee.EXACTLY_ONCE)  // section 23
    .build());

env.execute("fraud-detection-job");

Most transformations should feel familiar from Kafka Streams and Spark: map/flatMap/filter are stateless, per-record operations; keyBy repartitions the stream by a key, the same idea as a Kafka partition key or a Spark groupBy column; reduce/aggregate combine records per key over a window. Built-in connectors read from and write to the outside world — Kafka, filesystems (including S3/HDFS), JDBC databases, Elasticsearch, and Iceberg/Paimon tables are all officially maintained, playing a similar role to Kafka Connect's source/sink connectors (section 07) but compiled directly into the job rather than run as a separate service.

ProcessFunction — the escape hatch

Every operation you've seen so far (map, filter, windowed aggregate) is really a friendly wrapper around Flink's lowest-level building block: the ProcessFunction. A ProcessFunction gives you direct access to every event, to keyed state (section 23), and to timers — you can say "wake me up at this future timestamp" and Flink will call your code back exactly then, even if no new event arrives. This is what makes patterns like "alert if no follow-up event arrives within 10 minutes" possible without polling. A ProcessFunction can also emit to a side output — a separate, differently-typed stream tapped off the main one, commonly used to route late-arriving events (section 22) or malformed records to their own downstream path instead of dropping them silently.

Example: a session-timeout detector sets a timer for "now + 30 minutes" every time it sees an event for a user; if the timer fires before a newer event resets it, that's the signal the session ended — no polling loop required.

❯ Short version

The DataStream API builds a topology from a StreamExecutionEnvironment: sources → stateless transforms (map/filter) → keyBy → stateful transforms (aggregate/process) → sinks. Built-in connectors cover Kafka, files, JDBC, Elasticsearch, and lakehouse formats. When the built-in operators aren't enough, ProcessFunction gives you raw access to events, state, and timers, plus side outputs for anything you don't want to silently drop.

22. Event Time, Watermarks &amp; Windowing in Flink

The same ideas as sections 15-16, taken further

You already met event-time, processing-time, and watermarks in section 15, and tumbling/sliding/session windows in section 16 — the core ideas are identical here, because Flink's model is what the rest of the industry, including Spark, converged toward. What Flink adds is composability: instead of one bundled window() call, Flink separates windowing into independent pieces you can mix and match, which is a big part of why its event-time semantics are widely considered the most flexible of the three engines in this guide.

WatermarkStrategy, and why "idleness" matters

A WatermarkStrategy tells Flink two things: how to read an event's timestamp out of each record, and how to generate watermarks from the stream of timestamps it sees. The most common built-in, forBoundedOutOfOrderness(Duration), is the same idea as Spark's withWatermark (section 15): "assume events can arrive up to N late, and advance the watermark to max timestamp seen so far minus N." One thing Spark doesn't give you as directly: a source that stops sending anything for a while (a low-traffic partition, an idle Kafka topic) will silently stall the whole watermark, because Flink's watermark is the minimum across every parallel source instance. WatermarkStrategy.withIdleness(Duration) tells Flink to ignore a source instance that's gone quiet, so one idle partition doesn't freeze every window in the whole job.

Windowing itself is split into independent, swappable pieces: a WindowAssigner decides which window(s) an event belongs to (tumbling, sliding, session, or a global window you control entirely by hand); a Trigger decides exactly when a window fires and produces output (by default, when the watermark passes the window's end, but you can also fire early on a count or on a timer); an optional Evictor can remove elements from a window right before or after it fires; and a window function (reduce, aggregate, or process) computes the actual result. Spark bundles roughly the assigner and the "fire at the watermark" trigger into one window() call and doesn't expose the trigger/evictor concept at all — which is simpler to learn, but less flexible when you need something Spark's defaults don't offer, like firing partial, early results before a window officially closes.

Late data is the other place the two engines diverge. Spark drops any row that arrives after the watermark has passed its window (section 15). Flink instead gives you allowed lateness: a grace period after the watermark passes a window's end during which the window stays open and re-fires (updating its result) if more data shows up, and only after that grace period expires does Flink drop the window's state and route any further late arrivals to a side output (section 21) — visible, not silently discarded.

23. State, Checkpoints &amp; Exactly-Once in Flink

How a job remembers things, and how it recovers when a machine dies mid-stream

Any stateful operation — a running count, a window's accumulator, a custom ProcessFunction's timers — needs somewhere to keep that memory. Flink calls this keyed state when it's scoped to the current key in a keyBy'd stream (a ValueState, ListState, or MapState per key, conceptually the same role as Kafka Streams' state store from section 08), and operator state when it isn't tied to a key at all (used less often, mostly inside custom source/sink connectors). State physically lives in a state backend: the default HashMap state backend keeps everything in JVM heap memory, which is fast but limited by available RAM, while the RocksDB state backend spills to local disk via an embedded key-value store — the same RocksDB used by Kafka Streams and Spark's own state store — letting state grow far larger than memory at some cost in per-access latency.

Checkpoints vs. savepoints

A checkpoint is an automatic, periodic, internal snapshot of every operator's state, taken so the job can recover after a crash — you rarely touch these directly. A savepoint is the same underlying mechanism, but triggered manually and kept until you delete it, used for planned operations: upgrading your job's code, changing its parallelism, or migrating it to a different cluster, all while resuming from exactly the state it had before. Think of a checkpoint as an autosave and a savepoint as a save file you choose to keep.

The mechanism behind both is a distributed snapshot algorithm using checkpoint barriers: the JobManager periodically injects a special marker record into every source, and that barrier flows through the dataflow graph alongside the real data, in the same order. When an operator receives a barrier on all of its input streams, it snapshots its own state to durable storage (typically S3 or HDFS) and forwards the barrier downstream. Once every operator, all the way to the sinks, has done this, the checkpoint is complete and globally consistent — it represents one exact, coherent point in the stream's history across the entire job, not just one machine.

Aligned vs. unaligned checkpoints. An aligned checkpoint makes an operator with multiple inputs wait for the barrier to arrive on all of them before snapshotting — simple, but under heavy backpressure (one input much slower than another) this wait can stall checkpoints for a long time. An unaligned checkpoint lets the barrier overtake buffered records already sitting in the input queues, snapshotting that in-flight buffered data as part of the state instead of waiting for it to drain — trading a slightly larger snapshot for checkpoints that complete quickly even under severe backpressure.

None of this matters end-to-end unless the sink cooperates, which is exactly the same "chain of guarantees" idea from section 27's exactly-once discussion. Flink's answer is the two-phase commit pattern: a transactional sink (the Kafka sink with DeliveryGuarantee.EXACTLY_ONCE is the canonical example) opens a transaction, writes data to it as records arrive, and only commits that transaction once the checkpoint that covers those records has been confirmed complete by the JobManager. If the job crashes before the checkpoint completes, the transaction is aborted and none of those writes become visible — so a downstream consumer never sees partial or duplicate output, mirroring exactly how Kafka's transactional producer works (section 05).

❯ Short version

Keyed state lives in a state backend — heap (fast, RAM-limited) or RocksDB (disk-backed, scales further). Checkpoints are automatic snapshots for crash recovery; savepoints are the same mechanism triggered manually for planned upgrades and rescaling. Both work by flowing a barrier through the dataflow graph and snapshotting each operator's state as it passes — aligned barriers wait for all inputs, unaligned barriers overtake buffered data to checkpoint faster under backpressure. End-to-end exactly-once needs a transactional, two-phase-commit sink that only makes writes visible once the covering checkpoint is confirmed.

24. Flink's Table API &amp; Flink SQL

The same declarative idea as ksqlDB and Spark SQL, applied to Flink's engine

Writing raw DataStream code (section 21) for every job is more power than most pipelines need. The Table API (a fluent, DataFrame-like API in Java/Scala/Python) and Flink SQL (plain, mostly-standard SQL) sit on top of the DataStream API and compile down to the exact same runtime — the relationship is the same one Spark SQL has to Spark's underlying execution engine, or ksqlDB has to Kafka Streams (section 08). The central idea is the dynamic table: a table that keeps changing as new events arrive, with a continuous SQL query re-evaluating over it and emitting a changelog stream of inserts, updates, and deletes as the answer changes — conceptually identical to a Kafka Streams KTable (section 08), just expressed in SQL instead of Java.

CREATE TABLE transactions (
    account_id STRING,
    amount DECIMAL(12,2),
    event_time TIMESTAMP(3),
    WATERMARK FOR event_time AS event_time - INTERVAL '5' SECOND
) WITH (
    'connector' = 'kafka',
    'topic' = 'transactions',
    'properties.bootstrap.servers' = 'broker:9092',
    'format' = 'avro-confluent'
);

CREATE TABLE alerts (
    account_id STRING,
    window_start TIMESTAMP(3),
    swipe_count BIGINT
) WITH ('connector' = 'kafka', 'topic' = 'alerts', 'format' = 'json');

INSERT INTO alerts
SELECT account_id, window_start, COUNT(*) AS swipe_count
FROM TABLE(
    TUMBLE(TABLE transactions, DESCRIPTOR(event_time), INTERVAL '2' MINUTES))
GROUP BY account_id, window_start
HAVING COUNT(*) >= 10;

A CREATE TABLE ... WITH (...) statement is Flink SQL's equivalent of a Kafka Connect connector config (section 07): it declares a connector (Kafka, JDBC, filesystem, Iceberg/Paimon), a format, and — critically for streaming — where the watermark comes from, right in the DDL. The same query can run in streaming mode (continuous, unbounded) or batch mode (over a fixed, bounded table) with no code changes, which is the Table API's version of Spark's "one API for batch and streaming" pitch.

Flink CDC. Capturing every row-level change from an operational database (the same job Debezium does for Kafka Connect, section 07) is common enough that it has its own project, Apache Flink CDC, built around a YAML-configured "pipeline connector" that can snapshot and stream changes from an entire database — many tables at once — directly into Flink, with schema evolution handled automatically. Where Debezium-into-Kafka-Connect gives you one topic per table for something else to consume, Flink CDC treats the whole thing as one integrated ingestion + processing job.

25. Flink Deployment, Scaling &amp; Operations

Session vs. application clusters, Kubernetes, and how a job survives failures in production

Flink can run on standalone servers, YARN, or Kubernetes, but in 2024-2026 production environments the overwhelming default is Kubernetes, usually via the Flink Kubernetes Operator — a controller that manages Flink clusters and jobs as native Kubernetes custom resources, the same operational pattern most teams already use for every other stateful service they run.

Session mode vs. Application mode

In session mode, one long-running Flink cluster (a JobManager plus a pool of TaskManagers) stays up and accepts many jobs submitted to it over time — cheaper to keep warm, but jobs share the same JobManager, so a JobManager crash or a resource-hungry neighbor can affect every job on the cluster. In application mode — the recommended default for production since Flink 1.11+ — each job gets its own dedicated JobManager and TaskManagers, spun up specifically to run that one application's main() and torn down when it finishes; this trades a little startup overhead for full isolation between jobs, so a runaway job on one team's pipeline can't take down another's.

The Flink Kubernetes Operator lets you describe a job as a FlinkDeployment custom resource (a YAML file: image, parallelism, checkpoint config, resource requests) and hands the operator the job of reconciling that desired state — starting pods, restarting a crashed JobManager, and rolling out upgrades from a savepoint (section 23) instead of a cold restart. Its most valuable production feature is the built-in autoscaler: it watches each operator's busy-time and backpressure metrics and automatically raises or lowers that operator's parallelism to hit a target utilization, instead of an engineer guessing a fixed parallelism up front and re-tuning it by hand every time traffic shifts.

When something does fail — a TaskManager crashes, a checkpoint times out — the JobManager's restart strategy decides what happens next: fixed-delay retries a fixed number of times with a fixed pause between attempts; failure-rate gives up only if failures exceed a rate threshold over a time window, tolerating occasional blips without exhausting retries; exponential-backoff spaces out retries increasingly to avoid hammering a struggling downstream system. Every restart resumes from the most recent completed checkpoint, so recovery means "replay from the last consistent snapshot," not "start over." For the JobManager itself, high availability (via Kubernetes-native HA or ZooKeeper in older deployments) keeps a standby JobManager ready to take over leadership if the active one dies, so a JobManager crash doesn't take the whole job down with it.

Watching for trouble. The Flink Web UI (bundled with every cluster) shows per-operator backpressure — whether an operator is waiting on a slow downstream neighbor — directly in the job graph, which is usually the fastest way to spot the one slow stage causing an entire pipeline to lag. For anything beyond ad-hoc debugging, Flink's metrics system exports the same signals (checkpoint duration, backpressure, records/sec, state size) to Prometheus/Grafana, the same monitoring stack described for Kafka in section 09.

❯ Short version

Production Flink runs on Kubernetes via the Flink Kubernetes Operator, almost always in application mode (one dedicated cluster per job) rather than session mode, for isolation. The operator's autoscaler tunes parallelism automatically from backpressure metrics. Failures recover by restarting from the latest checkpoint under a configurable restart strategy, and JobManager HA keeps a standby ready so a single JobManager crash doesn't kill the job.

26. What's New in Flink (2024-2026)

Flink 2.0 and the shift toward cloud-native, disaggregated state

Flink's biggest recent shift mirrors what section 11 described for Kafka's tiered storage and section 19 described for Spark's RocksDB changelog checkpointing: separating compute from storage so a cluster can scale and recover without dragging gigabytes of local state around with it. Flink 2.0, released in 2025, is the vehicle for that shift — a major version that removed long-deprecated APIs (including the legacy DataSet batch API, fully folded into the unified DataStream API) and raised the baseline Java version, clearing the way for the state-management changes below.

The centerpiece is disaggregated state management, built around a new state backend called ForSt ("far and fast state," conceptually similar to RocksDB but designed to work efficiently against remote object storage such as S3). Today's RocksDB state backend keeps state on each TaskManager's local disk, which means rescaling a job — say, doubling its parallelism to handle a traffic spike — requires physically redistributing potentially huge amounts of state across machines before processing can resume. A disaggregated backend keeps the state in remote, shared storage instead, so rescaling becomes a matter of reassigning which TaskManager reads which slice of that remote state, not copying gigabytes of RocksDB files over the network. The tradeoff is per-access latency to remote storage, which is where the paired asynchronous state access work comes in: instead of a task blocking on every state read, the runtime can overlap many in-flight state requests, hiding that added latency behind throughput rather than paying for it serially.

On the SQL side, Materialized Tables let you declare a query and a target freshness ("keep this table updated within 5 minutes of the source") and have Flink's SQL Gateway manage the underlying continuous refresh job for you — the same "describe what you want, not how to run it" shift section 19 described for Spark's Lakeflow Declarative Pipelines. And Apache Flink CDC (mentioned in section 24) graduating to a top-level Apache project reflects just how central whole-database change capture has become to real-time architectures — it's no longer a niche integration, it's core infrastructure in its own right.

Flink 2.0 is a cleanup release that unblocks a bigger architectural shift: disaggregated state (the ForSt backend) moves state off local TaskManager disks and into remote storage, making rescaling far cheaper, paired with asynchronous state access to hide the added remote-storage latency. Materialized Tables bring declarative, freshness-based refresh to Flink SQL, and Flink CDC has grown into a top-level project for whole-database change capture.

27. Putting It Together: Kafka + Spark Architecture

Everything you've learned so far, wired into one real pipeline

Up to now you've mostly seen Kafka and Spark Structured Streaming discussed as separate pieces: partitions, offsets, and consumer groups on one side; readStream, windows, and checkpoints on the other. In a real production system, they are almost always used together, plus a lakehouse table format like Delta Lake or Apache Iceberg on the far end. This section walks through one realistic end-to-end pipeline, hop by hop, and then shows how the "exactly-once" guarantee you've heard about actually survives across all of those hops — or doesn't, if you get one link wrong.

Here is the shape of a typical real-time pipeline, described as a sequence of five steps:

• 1. Produce — Applications and services (a checkout service, a mobile app's event tracker, an IoT sensor gateway) write events into one or more Kafka topics as they happen.

• 2. Read — A Spark Structured Streaming job opens a readStream against those topics, subscribing continuously instead of running once and stopping.

• 3. Transform — The job cleans, filters, reshapes, and enriches each micro-batch of records. A common enrichment step is a stream-static join: joining the fast-moving stream against a small, slower-changing reference table (a product catalog, a customer segment table) that lives in a regular batch table.

• 4. Write — The job calls writeStream to land results somewhere durable: a lakehouse table (Delta/Iceberg), or back into a different Kafka topic so other downstream consumers can pick up the processed data.

• 5. Serve — BI dashboards, ad-hoc SQL, and ML models read the lakehouse tables, usually hours or even just seconds behind the original event, depending on how the pipeline is tuned.

If you've seen the medallion architecture before, this should look familiar: bronze tables hold events almost exactly as Kafka delivered them (just enough parsing to make them queryable), silver tables hold cleaned, deduplicated, enriched versions of the same data, and gold tables hold business-level aggregates that are cheap and fast for a dashboard to query. Not every pipeline needs all three layers, but it's a useful default shape to reach for.

Now for the part that trips people up: what does "exactly-once" actually mean across a pipeline with three separate systems (producer, Kafka, Spark) all failing and retrying independently? The honest answer is that exactly-once isn't one feature — it's a chain of four separate guarantees, each covered earlier in this guide, that all have to hold at the same time. This is the same idea section 17 called "the three legs of the stool" (replayable source, deterministic processing, idempotent sink) — here we're just splitting the "replayable source" leg into its two real pieces, the producer and the Kafka log, so you can see exactly where in a Kafka+Spark pipeline each leg actually lives:

• Hop 1 — the producer is idempotent (retrying a send never creates a duplicate) or wrapped in a Kafka transaction, so a network retry doesn't double-write the event.

• Hop 2 — the Kafka log is durable and replicated: once a write is acknowledged by enough in-sync replicas (ISR), it survives a broker crash.

• Hop 3 — the Spark checkpoint records the exact offsets processed in every micro-batch, so a restarted query resumes from precisely where it stopped, never re-reading or skipping records.

• Hop 4 — the sink must itself be idempotent or transactional. A very common pattern is a Delta MERGE run inside foreachBatch, keyed so that re-running the exact same micro-batch a second time (say, after a crash mid-write) produces the same end state instead of double-counting rows.

Exactly-once is a chain, and a chain is only as strong as its weakest link — one hop that's merely "at-least-once" turns the whole pipeline into at-least-once.

One more pattern worth knowing before you move on: fan-out. A single Kafka topic can be read independently by any number of different consumer groups, each for a completely different purpose, without any of them affecting the others or even knowing the others exist. For example, the same orders topic might be read by a fraud-detection Spark job computing real-time risk scores, and separately by a plain archival job that just copies raw events into a bronze Delta table for later analysis — two totally independent consumer groups, two totally independent jobs, one topic.

Worked example: the card-swipe fraud check from section 01

Here is roughly how the concepts in this guide combine to build the "score a card swipe in under a second" scenario this guide opened with: (1) a payments service produces one event per swipe to a card-swipes topic, keyed by account_id (section 03) so every swipe for the same account lands in order in the same partition; (2) Spark reads that topic and does a stream-static join (section 18) against a slowly-changing blocklist table of known-bad merchants/devices; (3) in parallel, a windowed aggregation (section 16) with a short watermark (section 15) keeps a rolling count of swipes per account over the last few minutes, to catch "10 swipes in 2 minutes" patterns; (4) both signals feed a rule inside foreachBatch (section 13) that writes any flagged swipe to an alerts Kafka topic and a Delta table for audit; (5) a separate downstream service consumes alerts and actually blocks or flags the transaction. None of this is new — it is sections 03, 13, 15, 16, and 18 combined into one pipeline.

swipes = (spark.readStream.format("kafka")
    .option("subscribe", "card-swipes")
    .load()
    .select(from_json(col("value").cast("string"), swipe_schema).alias("s")).select("s.*")
    .withWatermark("event_time", "2 minutes"))

blocklist = spark.read.format("delta").table("known_bad_merchants")   # static, section 18
flagged_by_merchant = swipes.join(blocklist, "merchant_id")            # stream-static join

rolling_counts = (swipes.groupBy("account_id", window("event_time", "2 minutes"))
    .count())                                                          # windowed aggregation, section 16

def write_alerts(batch_df, batch_id):
    risky = batch_df.filter("count >= 10")
    (risky.write.format("delta").mode("append")
        .saveAsTable("alerts"))                                        # idempotent-ish sink, keyed by account+window

(rolling_counts.writeStream
    .foreachBatch(write_alerts)
    .option("checkpointLocation", "/chk/fraud-alerts")
    .start())

❯ Short version

A real pipeline is: producers → Kafka topics → Spark readStream/transform/writeStream → bronze/silver/gold lakehouse tables → BI/ML. Exactly-once only holds end-to-end if every one of the four hops (producer, Kafka, Spark checkpoint, sink) is individually safe. One Kafka topic can safely feed many independent consumers doing completely different jobs at once.

28. Kafka Streams vs Spark Structured Streaming vs Flink

Three different tools for stream processing, and when each one actually wins

You now know three ways to process a stream in real depth: Kafka Streams (a library), Spark Structured Streaming (a cluster engine), and Apache Flink (sections 20-26) — a third major cluster engine you'll run into constantly in job postings, conference talks, and architecture diagrams. This section is the payoff for having learned all three: a direct, side-by-side comparison so you can pick the right one for a given job instead of defaulting to whichever your team already happens to know.

• Kafka Kafka Streams — recap

A Java/Scala library you add to your own application's code — there is no separate cluster to deploy or operate. It reads from and writes to Kafka topics directly. It shines when your data already lives in Kafka and your per-event logic is relatively simple (filter, map, join two topics, keep a running count), and it's especially popular for microservices that want to react to events without bolting on a whole separate processing platform.

• Spark Spark Structured Streaming — recap

A full cluster compute engine — a driver plus a pool of executors, the same engine batch Spark jobs use. It's the right choice when you need heavy transformations, joins against large batch or reference datasets, tight integration with ML pipelines, or when your team is already running Spark for batch work and wants one engine and one skill set for both.

• Both / Integration Apache Flink — the third engine

Flink is a true event-at-a-time streaming engine: it processes each record the instant it arrives, rather than collecting records into small batches first. That gives it the lowest typical latency of the three, and its event-time and windowing semantics are widely considered the most mature and flexible available today. This makes Flink a strong, and increasingly popular, choice for the lowest-latency use cases — sub-second fraud detection, real-time bidding, live risk scoring — where even Spark's fast micro-batches add more inherent delay than the use case can tolerate. This isn't a knock on Spark: for the vast majority of analytics and ETL-style streaming workloads, Spark's micro-batch model is fast enough, simpler to reason about, and comes with a much bigger existing batch/ML ecosystem around it. Pick based on your actual latency budget, not on which engine sounds newest.

Back to this guide's opening example. Section 01 opened with "score a card swipe as fraud in under a second." That's a real latency budget — worth writing down explicitly before you pick a tool. If your actual requirement is a few hundred milliseconds to a couple of seconds, Spark Structured Streaming's micro-batches comfortably fit. If it's truly sub-second with hard tail-latency limits (a payment that must be approved or declined before the terminal times out), that's the situation where it's worth asking whether Flink — or a non-streaming synchronous lookup outside this guide's scope entirely — is the better fit, rather than assuming the tool your team already knows is automatically the right one.

29. Streaming in Production: Real-World Case Studies

Not "how it works," but what LinkedIn, Netflix, Uber, and DoorDash actually run

Everything so far has explained how Kafka, Spark Structured Streaming, and Flink work. This section is about what happens when you point those tools at a real business with real traffic: which engine companies actually picked, why, and what broke along the way. Every claim below is drawn from a company's own engineering blog, a peer-reviewed paper by its own engineers, or an official conference talk — not secondhand summaries — and each was independently checked against its primary source before being included here. Where the public record has real gaps (and it does — see the callout at the end of this section), that's stated plainly instead of papered over with a plausible-sounding guess.

The one pattern that shows up everywhere below. Every verified case study in this section uses Kafka as the durable transport layer and Flink as the compute engine — not Spark Structured Streaming. That doesn't mean Spark has no place in real companies (it's everywhere in batch ETL and ML pipelines), but for the specific job of "process a Kafka stream with low latency and strict correctness at extreme scale," the verified public record in this section points overwhelmingly at Kafka+Flink. Take that as a data point for section 28's comparison, not a verdict — plenty of companies run Spark Structured Streaming successfully; they simply don't dominate the public case-study literature the way Netflix, Uber, and DoorDash's Flink writeups do.

Netflix — Keystone, the Real-Time Distributed Graph, and the ads pipeline

Netflix's streaming platform, Keystone, splits into two production services: a Kafka-based Data Pipeline that routes events, and Stream Processing as a Service (SPaaS), which runs custom streaming applications with Apache Flink as the processing engine. Each Flink job gets its own isolated Flink cluster running on Netflix's Titus container runtime, coordinating through ZooKeeper and checkpointing to S3 — the same JobManager/TaskManager and checkpoint-to-durable-storage model from sections 20 and 23, just running one dedicated cluster per job the way section 25 described application mode.

A newer system, the Real-Time Distributed Graph (RDG), ingests Kafka topics that each produce up to roughly 1 million messages per second. Netflix's engineers initially built one large Flink job to consume every source topic at once — and found that uneven throughput between topics made the whole job unstable and hard to tune. They re-architected to a 1:1 mapping of one Flink job per Kafka topic, trading "more jobs to operate" for "each job is simple enough to actually tune" — a concrete, publicly documented instance of the operator-parallelism tradeoffs discussed in section 20. Separately, Netflix's ads event pipeline runs two dedicated Flink jobs: Ads Metrics (raw events → dimensions/metrics → Apache Druid) and Ads Sessionizer (stitches every event for one ad into a single "Ad Session," the base every downstream ads report is built on).

Historical data point (2016, not current scale): Netflix's Kafka fleet ran 36 clusters with 4,000+ broker instances, split into "Fronting Kafka" (collection) and "Consumer Kafka" (routing/consumption) roles. At that time Netflix deliberately chose availability over strict durability at ingestion — producers configured with acks=1 and non-blocking sends — accepting a measured under-0.01% daily data loss rather than paying the latency/backpressure cost of stronger guarantees. That's a real, documented tradeoff, not a mistake: for Netflix's use case at the time, a rare dropped viewing event mattered far less than the ingestion tier falling behind.

Uber — Kafka + Flink + Pinot, and why they picked Flink over Storm and Spark

Uber's own account of this decision is unusually direct: an internal 2016 comparison found Flink recovered from a backlog of millions of buffered messages in about 20 minutes, where Apache Storm took several hours to do the same, and equivalent Spark jobs used 5-10x more memory than the same workload running on Flink. That benchmark — not a preference for whichever tool was newest — is why Flink became the default compute layer across Uber's real-time stack, alongside Kafka for transport and Apache Pinot for low-latency OLAP serving.

The clearest worked example is Uber's real-time, exactly-once ad event processing pipeline: three Flink jobs (Aggregation, Attribution, and Union-and-Load) deployed across two regions for resiliency, processing hundreds of millions of ad events per week. Exactly-once here is built from the same pieces as section 23's two-phase-commit pattern, tuned for their latency budget: Kafka transactional writes, a Flink consumer reading in read_committed mode, and a checkpoint interval shortened to 2 minutes (Flink's default is 10). Because their correctness contract says an event only counts as "processed" once its covering checkpoint commits, that 2-minute interval is a direct, deliberate trade of latency for correctness — precisely the tension section 23 describes. On top of that, a UUID generated in the Aggregation job doubles as a deduplication key, paired with upsert support Uber added to Pinot's real-time ingestion, as a second line of defense against double-counting.

At the scale of Uber's overall real-time infrastructure — trillions of messages and petabytes of data per day — the guarantees loosen deliberately: Kafka and Flink run at-least-once by default, and full exactly-once (via primary-key upsert) is reserved for a small set of critical use cases at the Pinot layer, not applied universally. Operationally, Uber also caps individual Kafka clusters below roughly 150 nodes each and federates many such clusters together, rather than growing one cluster without bound — the same "spread the blast radius" instinct behind Kafka's replication design in section 05.

DoorDash — Iguazu, and exactly-once all the way into the warehouse

DoorDash's Iguazu platform picked Kafka as its pub/sub layer specifically to unify data from many heterogeneous sources at high throughput, and picked Flink for its layered API (both the DataStream API and Flink SQL from sections 21 and 24), low-latency event-time processing, built-in fault tolerance, and ready-made connectors to Kafka, Redis, Elasticsearch, and S3. Over roughly two years, Iguazu scaled to process hundreds of billions of events per day at a 99.99% delivery rate.

Their exactly-once-to-the-warehouse design is a clean, concrete example of the checkpoint-plus-transactional-sink chain from section 23: Flink's file sink writes Parquet files to S3 as part of the checkpoint itself, so a file only becomes visible once its covering checkpoint is confirmed complete, guaranteeing consistency before Snowpipe loads that data into Snowflake. It's the same "nothing partial ever becomes visible" guarantee Uber gets from Kafka transactions, applied to a filesystem sink instead of a message broker.

Source: DoorDash Engineering Blog, "Building Scalable Real-Time Event Processing with Kafka and Flink" (2022).

LinkedIn — Brooklin, moving data across more than just Kafka

LinkedIn (where Kafka itself was born) built Brooklin, a distributed streaming system for moving data reliably across multiple different data stores and messaging systems at once — Espresso (LinkedIn's own Oracle-backed store), Kafka, Azure Event Hubs, and AWS Kinesis among them — covering both streaming-bridge (move events from system A to system B continuously) and change-data-capture use cases. It's a useful reminder that "streaming infrastructure" at this scale isn't only "Kafka plus a compute engine" — it also includes the connective tissue that gets data between systems that were never built to talk to each other directly, playing a role adjacent to Kafka Connect (section 07) but across more than just Kafka.

Honest caveat: the verified public record for Brooklin covers its purpose and architecture category well, but doesn't include hard scale numbers (throughput, cluster size, latency) the way the Netflix and Uber sources do — so treat this entry as thinner evidence than the other three.

Source: LinkedIn Engineering Blog, "Brooklin: Open Source Data Streaming for Everyone" (2019).

❯ Short version

Every verified production case study here — Netflix, Uber, DoorDash — runs Kafka for durable transport and Flink as the compute engine, not Spark Structured Streaming. All three get end-to-end exactly-once the same way section 23 describes it in theory: a shortened checkpoint interval plus a transactional sink (Kafka transactions or a checkpoint-integrated file write), trading some latency for correctness. At extreme scale, even these companies default to at-least-once and reserve exactly-once for the specific paths that truly need it. Public, verifiable case studies for Alibaba, ByteDance, and bank-specific fraud systems are genuinely scarce — that gap is real, not an oversight in this guide.

30. Common Pitfalls &amp; Best Practices

The mistakes almost everyone makes at least once — so you don't have to

Most streaming incidents aren't caused by exotic bugs — they're caused by a handful of well-known mistakes that are easy to avoid once you know they exist. Here are the ones that come up again and again, split into Kafka-side and Spark-side pitfalls, followed by a short list of habits that prevent most of them.

Kafka Pitfalls

Picking the wrong number of partitions. Too few partitions caps your maximum parallelism — if a topic has 4 partitions, at most 4 consumers in a group can ever be doing work at once, no matter how many more you add. Too many partitions adds overhead (more file handles, more replication traffic, slower leader elections) without any benefit if you never have enough consumers to use them. Worse: you can only increase a topic's partition count later, never decrease it, so under-provisioning is much easier to fix than over-provisioning.

A hot partition from a bad key choice. If you key messages by a low-cardinality field like country, and 80% of your traffic is from one country, that one partition (and the one consumer reading it) gets overloaded while the other partitions sit nearly idle. The fix is usually keying by something with much higher cardinality and a flatter distribution, like user_id or order_id.

Ignoring consumer lag until it's a production incident. Lag (how far behind the latest offset a consumer group is) creeps up quietly for hours or days before anyone notices a dashboard is stale — and by the time someone does notice, there can be millions of unprocessed messages to catch up on.

Forgetting to set min.insync.replicas. Without it, a producer using acks=all can still end up with data written to only one replica if the others are temporarily down, silently trading away the durability you thought you had for a bit more availability.

Unbounded topic retention. A topic with no retention limit (or one set far too generously) keeps every message forever, and broker disks fill up slowly and quietly until one day they don't have room for anything new.

Spark Pitfalls

Unbounded state growth from a missing watermark. A stateful aggregation (like a running count per key) with no watermark set never knows it's safe to forget old keys, so its state store grows forever, eventually exhausting memory or disk.

Changing what the checkpoint location means. If you restart a query with a structurally different set of transformations (new aggregation columns, a different join, a different watermark column) but point it at the same checkpoint folder, recovery can silently break or produce corrupted results — the checkpoint's saved state no longer matches what the new query expects.

A trigger interval that's too aggressive. If you set a 5-second trigger but each micro-batch actually takes 8 seconds to process, the query falls permanently behind — batches queue up faster than they can be cleared, and lag only grows.

No plan for schema drift. A producer team changes their message shape (renames a field, changes a type) without warning, and your Spark job's parsing logic breaks or — worse — silently produces nulls or garbage instead of failing loudly.

Treating late data as an afterthought. Deciding how late an event is allowed to arrive and still count (your watermark threshold) is a real design decision with real trade-offs, not a default you can skip and fix later — skipping it usually means silently dropping or double-processing data.

Best Practices

Default to idempotent producers. There's almost never a good reason to leave this off; it prevents duplicate writes from retries at essentially no cost.

Monitor consumer lag from day one. Treat it as a first-class metric with alerts, not something you only look at once something else has already gone wrong.

Always set an explicit watermark on streaming aggregations. Even a generous one (say, 24 hours) is far better than none, because "none" means "grow forever."

Test your failure/recovery path before production. Kill the job mid-run, restart it, and confirm it resumes correctly from checkpoint with no duplicated or missing results — don't assume it works, prove it.

Version or namespace your checkpoint location on structural changes. Something as simple as checkpoints/orders-agg-v2/ instead of reusing v1's folder avoids the silent-corruption pitfall above entirely.

# a sane default recipe that avoids most of the pitfalls above in one place

# producer config
acks=all
enable.idempotence=true
min.insync.replicas=2          # paired with replication.factor=3 on the topic

# streaming query
df.withWatermark("event_time", "24 hours")   # generous default beats "none" every time
  .writeStream
  .option("checkpointLocation", "checkpoints/orders-agg-v2")   # bump the vN on structural changes
  .start()

32. Learning Path &amp; Resources

A concrete, hands-on order to actually learn this by doing it

Reading about Kafka, Spark Structured Streaming, and Flink will only get you so far — the concepts (partitions, offsets, watermarks, checkpoints) click into place much faster once you've seen them with your own eyes on your own laptop. Here is a practical, ordered path. Each step builds on the last, and none of them require anything more than a laptop and some free disk space.

• 1. Run a single-node Kafka broker locally in KRaft mode (no ZooKeeper needed in modern Kafka). Create a topic, then use the command-line producer and consumer tools to send and read a few messages — watch the offsets and partitions change with your own eyes.

# format storage and start a single-node KRaft broker (run once)
kafka-storage.sh random-uuid
kafka-storage.sh format -t <uuid-from-above> -c config/kraft/server.properties
kafka-server-start.sh config/kraft/server.properties

# in another terminal: create a topic, then produce and consume
kafka-topics.sh --create --topic practice --bootstrap-server localhost:9092
kafka-console-producer.sh --topic practice --bootstrap-server localhost:9092
kafka-console-consumer.sh --topic practice --from-beginning --bootstrap-server localhost:9092

• 2. Write a tiny producer and a tiny consumer in whichever language you're comfortable with, sending and receiving real messages against that broker instead of the command-line tools.

• 3. Start a small Spark Structured Streaming job that reads from that same topic and does the simplest possible thing — count records per key — writing results to the console sink so you can watch them update.

• 4. Add a tumbling window and a watermark to that same job and watch how the results and their timing change.

• 5. Add a stream-static join against a small lookup table (a tiny CSV or Delta table of, say, product names) so you see enrichment in action.

• 6. Re-implement the same simple counting job in Kafka Streams or ksqlDB as a comparison exercise, and notice how differently it feels: no cluster to submit to, a different API shape, but the same underlying Kafka concepts.

• 6.5. Re-implement it a third time using Flink's DataStream API or Flink SQL (sections 20-24) against the same topic. This is where the differences from sections 20-26 stop being abstract: you'll feel the event-at-a-time model, the separate WatermarkStrategy, and the checkpoint/savepoint split directly.

• 7. Once you're running one of these in production, go deep on that engine's official docs. Defaults, configuration names, and best practices shift with almost every release, and no guide (including this one) can track that faster than the source.

Reference sources. This guide draws heavily on two O'Reilly books — Kafka: The Definitive Guide, 2nd Edition, and Learning Spark: Lightning-Fast Data Analytics, 2nd Edition — plus the official documentation, which is the best place to check for the exact current syntax and defaults:

• kafka.apache.org — official Kafka documentation, configuration reference, and KIP (Kafka Improvement Proposal) index.

• spark.apache.org/docs/latest/structured-streaming-programming-guide.html — the official Structured Streaming programming guide, updated with every Spark release.

• nightlies.apache.org/flink/flink-docs-stable — the official Flink documentation, including the DataStream API, Table API/SQL, and the Kubernetes Operator docs.

❯ Short version

Run Kafka locally → write a producer/consumer → read it from a simple Spark job → add windowing/watermarks → add a join → try the same thing in Kafka Streams, then in Flink → then go deep on whichever engine you actually ship. Once you're in production, official docs beat any book or guide, because defaults change every release.

∎ The takeaway

Kafka, Spark Structured Streaming, and Flink solve overlapping but distinct halves of the same problem. Kafka makes sure every event is captured, kept in order, and never lost — it is the durable, replayable backbone. Spark Structured Streaming and Flink both turn that endless flow of events into continuously updated answers — counts, joins, alerts, tables — but they make different tradeoffs to get there: Spark leans on its micro-batch model and its shared batch/ML ecosystem, while Flink leans on true event-at-a-time processing and the most flexible event-time semantics of the three. Section 29 shows how real companies actually combine these pieces, not just how each one works in isolation.

You do not need to memorize everything in this guide. Come back to it when you hit a real problem: a consumer that will not stop rebalancing, a Spark job whose state keeps growing, a watermark that is dropping data you did not expect. Re-read the relevant section, look at the diagram, and it should click faster the second time.
