Terraform Essentials for Data Engineers

Terraform lets you describe your cloud — buckets, warehouses, clusters, permissions — as text files you keep in Git, and then builds it for you. No more clicking through a console and hoping you remember what you did. This guide builds the mental model one piece at a time (the three worlds: your config, Terraform's state, and reality), then makes it concrete with HCL you can copy, before covering the parts that actually bite: state and locking, the dependency graph, count vs for_each, modules, environments, and running Terraform in CI/CD — finishing with how data engineers really use it: S3, IAM, Databricks, Snowflake and Airflow.

The one idea (read this first)

Terraform is infrastructure as code. Instead of clicking "Create bucket" in a web console, you write a text file that says "a bucket named raw-data should exist", and Terraform makes reality match. It is declarative: you describe the destination, not the driving directions. To pull that off, Terraform keeps a state file — its private notebook recording which real cloud resource belongs to which line of your code. Every run, it compares three things: your config (what you want), the state (what it thinks it built) and the real cloud (what's actually there). The difference is called a plan — a preview you read before anything changes — and apply executes it. Master state and you've mastered Terraform; almost every horror story is really a state story.

01. What Terraform is & why IaC exists

Picture setting up a data platform the old way. You open the AWS console and click: a bucket here, an IAM role there, a Glue job, a Databricks workspace, a few permissions you half-remember. It works. Six months later you need the same thing in a second region — and nobody can reconstruct exactly what was clicked. That's ClickOps, and it doesn't survive contact with a growing team. Infrastructure as code (IaC) replaces the clicking with a text file you can review, version, and re-run.

Terraform is the most widely used IaC tool. You write .tf files describing the infrastructure you want; Terraform talks to the cloud's API and makes it so. What you get in return:

  • Repeatable — the same config produces the same infrastructure, in dev, in prod, in a new region, on a new AWS account. No snowflake environments that only one person understands.
  • Reviewable — infrastructure changes arrive as pull requests. "You're deleting the prod bucket" becomes a comment on a diff instead of an outage.
  • Auditable — Git history tells you who changed the cluster size, when, and why.
  • Disposable — spin up a full environment for a test and tear it down in one command. Infrastructure stops being precious.

Declarative, not imperative — the single most important idea

A shell script is imperative: "create this, then create that." Run it twice and you get two buckets — or an error. Terraform is declarative: your file says "this bucket should exist", and Terraform figures out whether to create it, change it, or leave it alone. Run it a hundred times and you still have exactly one bucket. That property is called idempotence, and it's what makes the whole thing safe to re-run. You describe the destination; Terraform computes the route.

Where Terraform sits next to the tools you know

Terraform manages infrastructure — the buckets, roles, warehouses and clusters your pipelines run on. It is not a replacement for dbt (which transforms data inside the warehouse) or for Airflow (which schedules your pipelines). A clean split: Terraform creates the Snowflake warehouse and the role; dbt builds the tables in it; Airflow runs dbt on a schedule. Terraform is also the IaC mentioned in CI/CD Essentials §10 — this guide is that bullet point, fully unpacked.

Terraform, OpenTofu, and the licence change

In 2023 HashiCorp changed Terraform's licence from open-source (MPL) to the BUSL (a "source-available" licence with restrictions on competing commercial use). The community forked the last open version into OpenTofu, now stewarded by the Linux Foundation. For everything in this guide the two are interchangeable — same HCL, same commands (tofu plan instead of terraform plan), same providers. If your company is licence-sensitive, you'll meet OpenTofu; the concepts transfer 1:1.

02. The mental model: three worlds

Everything Terraform does is a comparison between three things. Once you can name all three and say which command touches which, Terraform stops being mysterious — and every confusing error message suddenly has an obvious home.

1
Your config — the .tf files you write and commit. This is the desired state: the world as you want it. It's the only one of the three you edit by hand.
2
The state — a JSON file Terraform maintains, mapping each resource in your code (aws_s3_bucket.raw) to the real thing it created (arn:aws:s3:::acme-raw-prod). Without it Terraform has no idea that the bucket in your config is the bucket in the cloud — it would try to create a second one. This is Terraform's memory.
3
Reality — what's actually running in AWS/Azure/GCP/Databricks right now, including anything a colleague changed by hand at 2am during an incident.

Drift, defined

Drift is when reality (3) no longer matches the state (2) — because someone changed something outside Terraform. At the start of every plan, Terraform refreshes: it calls the cloud API and re-reads each resource it knows about, so its picture is current. That's why a plan can show changes you didn't make — it's not being weird, it's telling you someone clicked in the console and it intends to put things back the way your code says. The fix is almost always to bring the change into code, not to fight the plan.

03. The core loop: init → plan → apply

Terraform has dozens of subcommands, but day to day you use four. The rhythm never changes: edit the code, preview the diff, approve it, repeat.

The four commands, in order

1
terraform init — run once when you start (and again whenever you add a provider or module). It downloads the providers (the plugins that know how to talk to AWS, Databricks, Snowflake…), fetches any modules, and wires up the backend where state will live. It creates a .terraform/ folder and a .terraform.lock.hcl file. Safe to re-run.
2
terraform plan — the dry run. Terraform refreshes its view of reality, compares it to your config, and prints exactly what it would do. It changes nothing. This is the single most important habit in Terraform: read the plan before every apply.
3
terraform apply — does it for real. By default it shows you the plan again and waits for you to type yes. In CI you skip the prompt with -auto-approve, or (better) hand it a saved plan file so it executes exactly the diff a human already reviewed.
4
terraform destroy — deletes everything in this configuration's state. Genuinely useful for throwaway dev environments and CI test runs. In production, this is the command that ends careers — see §15 for the guardrail (prevent_destroy).

How to read a plan

The plan output uses four symbols. Learn them and you can review any infrastructure PR:

terraform plan — the four symbols
  + create         # a new resource will be made — usually harmless
  ~ update in-place # an attribute changes, the resource survives — usually fine
  - destroy        # the resource will be DELETED — read this line twice
-/+ replace        # destroy, THEN create a new one — the dangerous one:
                   #   the resource gets a new ID, and any data on it is gone

Plan: 2 to add, 1 to change, 0 to destroy.   # ← the line everyone skims. Don't.

-/+ replace is the line that ruins weekends

Some attributes cannot be changed on a live resource — a bucket's name, an instance's availability zone, a database's engine. Change one in your config and Terraform can't "update" it; it must destroy and recreate. For a stateless app server, fine. For an S3 bucket holding 4 TB of raw data, or a production database, that's a data-loss event. Terraform tells you why with a line like # forces replacement right next to the offending attribute. Whenever you see -/+ or "forces replacement" in a plan against a stateful resource, stop and think.

04. HCL: the language, block by block

Terraform files are written in HCL (HashiCorp Configuration Language) — a config language that reads like a cross between JSON and a normal programming language. It's built from blocks. There are only about eight you'll ever use, and this section is all of them.

Every block follows the same shape: a type, zero or more labels, and a body in braces.

the anatomy of a block
resource  "aws_s3_bucket"  "raw"  {      # type · label 1 (kind) · label 2 (your name)
  bucket = "acme-raw-data-prod"          # arguments: key = value
}
#          └─ together, the two labels form the ADDRESS: aws_s3_bucket.raw
#             That address is how you refer to it everywhere else.

A complete, real file

Here is a small but genuinely usable configuration — a raw-data bucket and an IAM role your ETL job can assume. Every block type is labelled; the rest of the section explains each one.

main.tf
# ── 1. TERRAFORM block: settings for Terraform itself ──
terraform {
  required_version = ">= 1.6"
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"            # any 5.x, but not 6.0 — see §05
    }
  }
  backend "s3" {                     # where the STATE lives — see §06
    bucket       = "acme-tfstate"
    key          = "data-platform/prod.tfstate"
    region       = "eu-west-1"
    use_lockfile = true            # state locking — see §06
  }
}

# ── 2. PROVIDER block: configure the plugin that talks to AWS ──
provider "aws" {
  region = var.region
}

# ── 3. VARIABLE block: an input you can change per environment ──
variable "env" {
  type        = string
  description = "Which environment this is (dev / prod)"
}
variable "region" {
  type    = string
  default = "eu-west-1"
}

# ── 4. LOCALS block: named values computed once, reused below ──
locals {
  name_prefix = "acme-${var.env}"          # "acme-prod"
  common_tags = {
    Environment = var.env
    ManagedBy   = "terraform"              # tag everything: see §15
  }
}

# ── 5. DATA block: READ something that already exists (never creates) ──
data "aws_caller_identity" "current" {}   # "who am I?" → account id

# ── 6. RESOURCE block: the thing Terraform actually creates ──
resource "aws_s3_bucket" "raw" {
  bucket = "${local.name_prefix}-raw"      # "acme-prod-raw"
  tags   = local.common_tags
}

resource "aws_iam_role" "etl" {
  name = "${local.name_prefix}-etl-role"
  tags = local.common_tags

  assume_role_policy = jsonencode({        # build JSON without writing JSON
    Version = "2012-10-17"
    Statement = [{
      Effect    = "Allow"
      Action    = "sts:AssumeRole"
      Principal = { Service = "glue.amazonaws.com" }
    }]
  })
}

# ── 7. OUTPUT block: publish a value for humans & other configs ──
output "raw_bucket_name" {
  value       = aws_s3_bucket.raw.bucket   # ← referencing by ADDRESS
  description = "Where the ingestion job should land files"
}

The blocks you'll actually write

terraform { }
Settings for Terraform itself: the minimum version, which providers you need (and their versions), and the backend where state is stored. One per configuration.
provider "aws" { }
Configures a plugin — region, credentials, endpoints. You can have several (AWS and Databricks and Snowflake in one config), and even two of the same kind via alias (e.g. two AWS regions).
resource "type" "name" { }
The only block that creates things. The first label is the kind of thing (defined by the provider); the second is your name for it, used to reference it elsewhere. Address: type.name.
data "type" "name" { }
A read-only lookup of something that already exists — an AMI id, the current account number, an existing VPC someone else manages. Data sources never create or change anything. Address: data.type.name.
variable "name" { }
An input. Lets one config serve dev and prod by changing values, not code. See §08.
output "name" { }
A published result — printed after apply, and readable by other configurations. The way a module hands values back to its caller.
locals { }
Named intermediate values — like a const in a normal language. Computed once, referenced as local.name. Great for killing copy-paste.
module "name" { }
Calls a reusable bundle of resources. See §10.

resource vs data — the distinction beginners miss

A resource is something Terraform owns: it created it, it tracks it in state, and if you delete the block, Terraform deletes the real thing. A data source is something Terraform merely looks at: it's owned by someone else (another team, another Terraform config, or a human), and deleting the block deletes nothing. Rule: if you'd be sad when terraform destroy removes it, make sure it's not in your config as a resource by accident.

File layout — Terraform doesn't care, but humans do

Terraform loads every .tf file in the directory and glues them together — file names are meaningless to it, and order doesn't matter (it works out dependencies itself, see §07). But there's a near-universal convention worth following, because every Terraform repo you'll ever join uses it:

the conventional layout
.
├── main.tf           # the resources (the bulk of it)
├── variables.tf      # all variable blocks (the inputs)
├── outputs.tf        # all output blocks (the results)
├── providers.tf      # terraform{} + provider{} blocks
├── terraform.tfvars  # the actual values  ← NEVER commit secrets here
└── .terraform.lock.hcl  # provider version lock  ← DO commit this

05. Providers, versions & the lock file

Terraform itself knows nothing about AWS. It's a small engine that reads HCL, builds a graph, and calls plugins. Those plugins are providers — and there are thousands of them, which is the real reason Terraform won.

A provider is the translator between Terraform's generic idea of "a resource" and one specific API. hashicorp/aws knows how to create an S3 bucket; databricks/databricks knows how to create a job cluster; snowflakedb/snowflake knows how to create a warehouse and grant a role. They all come from the Terraform Registry, and terraform init downloads them.

a multi-provider data platform
terraform {
  required_providers {
    aws        = { source = "hashicorp/aws",          version = "~> 5.0" }
    databricks = { source = "databricks/databricks",  version = "~> 1.50" }
    snowflake  = { source = "snowflakedb/snowflake",  version = "~> 0.95" }
  }
}

# two AWS regions at once — the SAME provider, twice, distinguished by alias
provider "aws" {
  region = "eu-west-1"                   # the default one
}
provider "aws" {
  alias  = "us"                          # the extra one
  region = "us-east-1"
}

resource "aws_s3_bucket" "dr_copy" {
  provider = aws.us                       # ← this one lands in us-east-1
  bucket   = "acme-prod-raw-dr"
}

Version constraints — pin them, or be surprised

Providers ship breaking changes. If you don't pin a version, a colleague running init next month can pull a newer provider that renames an attribute — and suddenly the plan wants to replace your bucket. The ~> operator (the "pessimistic constraint") is the usual answer:

ConstraintMeansUse when
version = "5.31.0"Exactly this. Nothing else.Maximum paranoia (prod, regulated)
version = "~> 5.31"≥ 5.31, < 6.0 — patches and minors, no majorThe sane default
version = "~> 5.31.0"≥ 5.31.0, < 5.32.0 — patches onlyConservative
version = ">= 5.0"Anything newer. Includes 6.0's breaking changes.Almost never — this is how you get ambushed

Commit .terraform.lock.hcl

Your constraint says "any 5.x". The lock file records which exact 5.x you resolved to, plus checksums. Commit it, and every teammate — and CI — gets byte-identical providers, so "works on my machine" can't happen. It's the same idea as package-lock.json or poetry.lock. Upgrade deliberately with terraform init -upgrade, and review the resulting plan.

06. State: the part that trips everyone up

If you only deeply learn one section of this guide, make it this one. State is where Terraform's power comes from, and it's where every genuinely bad day originates.

What state actually is

A JSON file. For every resource in your config, it records the real-world ID the cloud handed back, plus a snapshot of that resource's attributes at last apply. That's it. It exists to answer one question Terraform otherwise couldn't: "the bucket described on line 12 of my code — which real bucket is that?"

terraform.tfstate (trimmed)
{
  "resources": [{
    "type": "aws_s3_bucket",
    "name": "raw",                              // ← your code's address: aws_s3_bucket.raw
    "instances": [{
      "attributes": {
        "id": "acme-prod-raw",                   // ← the REAL thing in AWS
        "arn": "arn:aws:s3:::acme-prod-raw"
      }
    }]
  }]
}

State is not a backup — and it is not a secret store

Two consequences people learn the hard way. (1) Delete the state and Terraform forgets it owns anything — your infrastructure keeps running, but Terraform's next plan will cheerfully try to create it all again (and fail on name collisions). (2) State contains every attribute in plaintext, including generated database passwords and access keys, even if you marked them sensitive in your config. So: never commit .tfstate to Git, always encrypt the bucket it lives in, and lock down who can read it. Anyone who can read state can read your secrets.

Remote state & locking — the first thing to set up on a team

By default, state is a file on your laptop. That's fine for ten minutes of learning and catastrophic for a team: your colleague has a different copy, and if you both apply at once you corrupt each other's work. So teams put state in a remote backend — a shared bucket — with a lock so only one apply can run at a time.

an S3 backend with locking
terraform {
  backend "s3" {
    bucket       = "acme-tfstate"              # a bucket you create ONCE, by hand
    key          = "data-platform/prod.tfstate" # the path — one key per environment!
    region       = "eu-west-1"
    encrypt      = true                       # state holds secrets → encrypt it
    use_lockfile = true                       # native S3 locking (Terraform ≥ 1.10)
    # older setups used: dynamodb_table = "tf-locks"
  }
}

Historically, S3 couldn't lock, so everyone paired the bucket with a small DynamoDB table that acted as the lock. Recent Terraform versions can lock using S3 itself (use_lockfile), and the DynamoDB approach is being retired — but you will absolutely still meet it in existing repos, so recognise it. Azure Blob and GCS have locking built in; HCP Terraform (the hosted service, formerly Terraform Cloud) does state, locking, and a plan UI for you.

Working with state directly

Occasionally you must reach into state. These commands are safe to read with, and require care to write with:

state commands
terraform state list                      # every resource Terraform is tracking
terraform state show aws_s3_bucket.raw    # everything it knows about one resource

# ── renaming / restructuring (no cloud changes, just bookkeeping) ──
terraform state mv aws_s3_bucket.raw aws_s3_bucket.landing   # old way

# ── forget a resource WITHOUT deleting it in the cloud ──
terraform state rm aws_s3_bucket.raw      # Terraform stops managing it; bucket survives

# ── adopt an existing, hand-made resource INTO Terraform ──
terraform import aws_s3_bucket.raw acme-prod-raw   # old way

Modern Terraform prefers declarative equivalents you commit to Git, so the change is reviewable in a PR instead of being a command someone typed once on their laptop:

the modern, reviewable way
# adopt an existing bucket that was created by hand (Terraform ≥ 1.5)
import {
  to = aws_s3_bucket.raw
  id = "acme-prod-raw"       # next plan shows it being ADOPTED, not created
}

# rename a resource without destroying it (Terraform ≥ 1.1)
moved {
  from = aws_s3_bucket.raw
  to   = aws_s3_bucket.landing   # without this, Terraform would DESTROY + CREATE
}

# stop managing something without destroying it (Terraform ≥ 1.7)
removed {
  from = aws_s3_bucket.legacy
  lifecycle { destroy = false }   # drop from state, leave the bucket alone
}

Never hand-edit the state file

It's JSON, so it's tempting. Don't. It has internal serial numbers and checksums, and a malformed edit can make Terraform lose track of resources — which means the next apply tries to recreate live infrastructure. Use the state subcommands or the import/moved/removed blocks instead. And enable versioning on your state bucket, so a corrupted state can be rolled back to yesterday's copy. That single checkbox has saved more platforms than any other.

07. The dependency graph

You never tell Terraform what order to do things in. It works that out — by reading the references in your code and building a graph. Understanding this explains both why Terraform is fast and why it sometimes needs a nudge.

The rule is simple: if resource A mentions resource B's address, A depends on B. That's an implicit dependency, and it's how 95% of ordering happens.

implicit vs explicit dependencies
# IMPLICIT — the normal way. The reference IS the dependency.
resource "aws_glue_job" "etl" {
  name     = "daily-etl"
  role_arn = aws_iam_role.etl.arn        # ← Terraform now KNOWS: role first, job second

  command {
    script_location = "s3://${aws_s3_bucket.raw.bucket}/jobs/etl.py"
  }                                     # ← and the bucket before the job, too
}

# EXPLICIT — the escape hatch, for dependencies Terraform can't SEE.
resource "aws_glue_job" "etl2" {
  # …the job needs the IAM policy ATTACHED before it can run, but nothing in
  # this block references the policy — so Terraform can't infer the ordering.
  depends_on = [aws_iam_role_policy.etl]
}

depends_on is a code smell — usually

If you find yourself reaching for depends_on, first check whether you can express the relationship as a reference instead — pass aws_iam_role.etl.arn rather than hard-coding the ARN string. References make the graph correct automatically and keep values in sync. Save depends_on for genuinely invisible dependencies (IAM permissions that must exist before an API call succeeds is the classic one). Overusing it serialises your graph and makes applies slow.

08. Variables, outputs, locals & expressions

A config with hard-coded values can build exactly one environment. Variables are what let the same code build dev, staging and prod — the Terraform equivalent of "build once, deploy many."

Variables: typed, validated inputs

variables.tf
variable "env" {
  type        = string
  description = "dev | staging | prod"

  validation {                                # catch typos at PLAN time, not 3am
    condition     = contains(["dev", "staging", "prod"], var.env)
    error_message = "env must be dev, staging or prod."
  }
}

variable "warehouse_size" {
  type    = string
  default = "XSMALL"                          # a default makes it optional
}

variable "snowflake_password" {
  type      = string
  sensitive = true                             # hides it from CLI output & logs…
}                                              # …but NOT from the state file (§06)

# richer types: a whole object, not just a string
variable "buckets" {
  type = map(object({
    versioning    = bool
    retention_days = number
  }))
  default = {
    raw     = { versioning = true,  retention_days = 365 }
    curated = { versioning = true,  retention_days = 90  }
    scratch = { versioning = false, retention_days = 7   }
  }
}

Where values come from — and who wins

Terraform gathers variable values from several places. When two disagree, the last one wins:

#SourceExample
1 (lowest)The default in the variable blockdefault = "XSMALL"
2Environment variables prefixed TF_VAR_export TF_VAR_env=prod
3terraform.tfvars (auto-loaded)env = "prod"
4*.auto.tfvars (auto-loaded, alphabetical)prod.auto.tfvars
5 (highest)-var / -var-file on the command lineterraform apply -var-file=prod.tfvars

Locals & expressions — the "programming" bits

HCL isn't a full programming language (no loops you can write by hand, no functions you can define), but it has enough expression power to kill nearly all copy-paste.

the expressions you'll use constantly
locals {
  # string interpolation
  prefix = "acme-${var.env}"                          # "acme-prod"

  # conditional (ternary) — the if/else of HCL
  cluster_size = var.env == "prod" ? 10 : 2

  # merging maps — shared tags plus per-thing extras
  tags = merge(var.common_tags, { Owner = "data-platform" })

  # a FOR expression — transform a collection
  bucket_names = [for name, cfg in var.buckets : "${local.prefix}-${name}"]
  # → ["acme-prod-raw", "acme-prod-curated", "acme-prod-scratch"]

  # filter while transforming
  versioned = { for k, v in var.buckets : k => v if v.versioning }

  # safe defaults — try() swallows errors, coalesce() picks the first non-empty
  region = coalesce(var.region, "eu-west-1")
}

# useful built-ins you'll meet: jsonencode() · file() · templatefile()
#   toset() · lookup() · length() · upper() · cidrsubnet() · timestamp()

Outputs — how a config talks to the outside

outputs.tf
output "raw_bucket_arn" {
  value       = aws_s3_bucket.raw.arn
  description = "Pass this to the Airflow connection"
}

output "db_password" {
  value     = random_password.db.result
  sensitive = true          # prints as <sensitive>; read with `terraform output -raw db_password`
}

09. count, for_each, lifecycle & dynamic

Meta-arguments are settings you can add to any resource, regardless of type. Four of them matter, and one of them — the choice between count and for_each — is the most consequential small decision in Terraform.

Making many things: count vs for_each

Both create multiple copies of a resource from one block. They differ in how Terraform names each copy in state — and that naming is everything.

count vs for_each in code
# ── count: indexed by POSITION → aws_s3_bucket.data[0], [1], [2] ──
resource "aws_s3_bucket" "data" {
  count  = length(var.bucket_list)              # ["raw","clean","curated"]
  bucket = "acme-${var.bucket_list[count.index]}"
}                                              # remove "clean" → everything shifts 

# ── for_each: keyed by NAME → aws_s3_bucket.data["raw"], ["curated"] ──
resource "aws_s3_bucket" "data" {
  for_each = toset(["raw", "clean", "curated"])
  bucket   = "acme-${each.key}"                 # each.key = "raw"
}                                              # remove "clean" → only that one goes 

# ── for_each over a MAP: each.key AND each.value ──
resource "aws_s3_bucket" "zone" {
  for_each = var.buckets                       # the map(object) from §08
  bucket   = "acme-${var.env}-${each.key}"
  tags     = { RetentionDays = each.value.retention_days }
}

# ── the one good use of count: an optional resource ──
resource "aws_cloudwatch_dashboard" "monitoring" {
  count = var.env == "prod" ? 1 : 0        # exists in prod only
}

lifecycle — overriding Terraform's instincts

the lifecycle block
resource "aws_s3_bucket" "raw" {
  bucket = "acme-prod-raw"

  lifecycle {
    prevent_destroy = true          # ← REFUSE to delete this, ever. Any plan that
                                     #   would destroy it errors out instead.
                                     #   Put this on every prod data store. Today.

    ignore_changes = [tags["LastScan"]] # a security tool edits this tag; stop
                                     #   fighting it on every plan

    create_before_destroy = true    # when replacing: build the new one FIRST,
  }                                  #   then remove the old → no downtime gap
}

dynamic — generating repeated blocks

Some resources contain repeated sub-blocks (an S3 lifecycle rule, a security-group ingress rule). for_each makes many resources; dynamic makes many blocks inside one resource.

dynamic blocks
resource "aws_s3_bucket_lifecycle_configuration" "raw" {
  bucket = aws_s3_bucket.raw.id

  dynamic "rule" {                       # generate one `rule { }` per entry
    for_each = var.buckets
    content {
      id     = "expire-${rule.key}"       # `rule` = the block label
      status = "Enabled"
      expiration { days = rule.value.retention_days }
    }
  }
}

Don't get clever

Nested dynamic blocks driven by nested for expressions over conditional locals are possible, and they produce configs nobody — including future you — can read or safely change. HCL is a configuration language, not a programming one. When a block gets hard to follow, the answer is usually a module (§10), not more cleverness.

10. Modules — write it once

A module is just a folder of .tf files that you call from somewhere else, passing values in and getting values out. It's Terraform's function. If you've ever copy-pasted a block of resources between dev/ and prod/, you wanted a module.

modules/storage/ — the module itself
# modules/storage/variables.tf — its INPUTS
variable "env"  { type = string }
variable "size" { type = string, default = "small" }

# modules/storage/main.tf — the RESOURCES (no provider block! see below)
resource "aws_s3_bucket" "this" {
  bucket = "acme-${var.env}-raw"
  lifecycle { prevent_destroy = true }
}

resource "aws_s3_bucket_versioning" "this" {
  bucket = aws_s3_bucket.this.id
  versioning_configuration {
    status = var.env == "prod" ? "Enabled" : "Suspended"
  }
}

# modules/storage/outputs.tf — what it hands BACK
output "bucket_arn" { value = aws_s3_bucket.this.arn }
envs/prod/main.tf — calling it
module "storage" {
  source = "../../modules/storage"       # local path…
  env    = "prod"                          # …and the inputs
  size   = "large"
}

# reference a module's output as module.<name>.<output>
resource "aws_glue_job" "etl" {
  command { script_location = "${module.storage.bucket_arn}/etl.py" }
}

# modules can also come from the REGISTRY or from Git — always pin a version:
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"   # the public registry
  version = "5.8.1"                          # ← pin it, like a provider
  name    = "data-platform"
}

The rules that keep modules sane

(1) No provider blocks inside a module — providers are configured by the root module (the directory you run terraform in) and inherited. A provider block in a shared module makes it impossible to use in a multi-region setup. (2) No hard-coded environment names — take them as inputs, or the module only works once. (3) Keep them small. A module that creates "the whole platform" is just your config with extra steps; a module that creates "a governed S3 bucket with versioning, encryption and a retention policy" is reusable. (4) Version them if they're shared across repos, and never point source at a Git branch — pin a tag, or a colleague's merge changes your infrastructure.

11. Environments: workspaces vs directories

You need dev, staging and prod. Terraform gives you two ways to do it, and the one that sounds purpose-built is usually the wrong choice.

Option A: terraform workspace — usually not what you want

Workspaces give one configuration several separate state files: terraform workspace new prod, then terraform apply writes to a prod-specific state. Same code, different state. Tempting — but they share one backend, one set of credentials, and one provider config, and the only thing distinguishing prod from dev is a shell variable you can forget to set. The failure mode is applying dev changes to prod because you were in the wrong workspace, and nothing in the code makes that visible in a PR.

Option B: a directory per environment — the standard

Give each environment its own folder, its own backend key, and its own .tfvars. All the shared logic lives in modules, so there's no duplication of substance — just a thin, explicit, greppable entry point per environment. This is what most serious teams do.

the layout that scales
.
├── modules/                    # all the real logic — reusable, environment-agnostic
│   ├── storage/
│   ├── warehouse/
│   └── orchestration/
└── envs/
    ├── dev/
    │   ├── main.tf             # calls the modules with dev inputs
    │   ├── backend.tf          # key = "data-platform/dev.tfstate"
    │   └── terraform.tfvars    # cluster_size = 2
    └── prod/
        ├── main.tf             # the SAME modules, prod inputs
        ├── backend.tf          # key = "data-platform/prod.tfstate"  ← separate state!
        └── terraform.tfvars    # cluster_size = 20

# you run Terraform from inside one env directory:
cd envs/prod && terraform init && terraform plan

Why the extra folders are worth it

Separate directories mean separate state files, which means a mistake in dev physically cannot touch prod's state. They also let prod use different credentials, a different AWS account, and a stricter CI job — and they make "what's actually deployed to prod?" a question you answer by reading one small file. Workspaces are genuinely good for ephemeral environments (a temporary copy per pull request); for the long-lived dev/staging/prod ladder from CI/CD Essentials §05, use directories.

12. Terraform in CI/CD

Running terraform apply from a laptop is how small teams start and how outages begin. The mature pattern: humans review plans, machines run applies. Nobody needs prod credentials on their laptop, and every infrastructure change arrives as a reviewed pull request.

.github/workflows/terraform.yml
name: Terraform
on:
  pull_request:                    # every PR → plan only
  push:
    branches: [main]               # merge to main → apply

permissions:
  contents: read
  id-token: write                  # ← required for OIDC (no stored AWS keys!)
  pull-requests: write             # ← to post the plan as a comment

jobs:
  plan:
    runs-on: ubuntu-latest
    defaults: { run: { working-directory: envs/prod } }
    steps:
      - uses: actions/checkout@v4
      - uses: aws-actions/configure-aws-credentials@v4
        with:                                    # short-lived creds, minted per run
          role-to-assume: arn:aws:iam::123456789012:role/gha-terraform
          aws-region: eu-west-1
      - uses: hashicorp/setup-terraform@v3
      - run: terraform fmt -check -recursive   # style gate (fast, fails first)
      - run: terraform init
      - run: terraform validate                # syntax + type checking
      - run: terraform plan -out=tfplan        # ← SAVE the plan…
      - uses: actions/upload-artifact@v4       # …and keep it for the apply job
        with: { name: tfplan, path: envs/prod/tfplan }

  apply:
    needs: plan
    if: github.ref == 'refs/heads/main'     # never apply from a PR
    runs-on: ubuntu-latest
    environment: production                 # ← the human approval gate
    defaults: { run: { working-directory: envs/prod } }
    steps:
      - uses: actions/checkout@v4
      - uses: actions/download-artifact@v4
        with: { name: tfplan, path: envs/prod }
      - run: terraform init
      - run: terraform apply -auto-approve tfplan   # ← apply THAT plan, exactly

Why -out=tfplan matters more than it looks

If your apply job just runs terraform apply -auto-approve, it computes a brand-new plan at apply time — which may differ from the one your reviewer approved (someone else merged, or the cloud drifted in between). Saving the plan to a file and applying that file guarantees the machine does exactly what a human signed off on. It's the Terraform version of "build once, deploy many" from CI/CD Essentials §05.

Three more habits worth adopting on day one:

  • Use OIDC, not stored cloud keys. As in CI/CD Essentials §11: let the workflow exchange its GitHub identity for a short-lived AWS role, so there's no permanent secret to steal. The plan job can assume a read-only role; only the apply job gets write access.
  • Post the plan on the PR. A plan nobody reads is theatre. Comment it onto the pull request so the diff is reviewed next to the code change that caused it.
  • Add static checks. terraform fmt -check (formatting), terraform validate (syntax), tflint (bad practice), and a security scanner like tfsec or checkov (public buckets, unencrypted volumes) — all cheap, all before the plan.

The rule: Terraform manages the container, not the contents

Terraform should create the bucket, not the files in it. The warehouse and the schema, not the tables inside them. The Databricks job, not the notebook's logic. Why: Terraform's model is "this resource should look exactly like my config," and data is constantly changing outside of Terraform. Point it at data and every plan shows phantom drift — or worse, offers to "fix" a table by replacing it. Terraform builds the platform; dbt builds the tables; Airflow runs the pipelines. Keep the seams clean and each tool is easy. Blur them and you get a plan that wants to drop your warehouse.

Data-specific traps

Renaming a bucket forces replacement — and Terraform will happily plan to delete the one holding your raw data. prevent_destroy on every production data store, no exceptions. Snowflake grants can fight you: if people also grant privileges by hand, Terraform will revoke them on the next apply — decide who owns grants and stick to it. Cluster autoscaling looks like drift: if something outside Terraform changes a worker count, use ignore_changes rather than fighting it every plan.

14. A worked example: a real repo, end to end

Everything so far has been pieces. This section assembles them into one small-but-complete data-platform repo — the layout, every file, and why each line is where it is. Copy this shape and you'll have a codebase that still makes sense when it's ten times bigger.

The tree

Here is the whole thing. Read it top to bottom: modules hold the logic, envs hold the wiring, and nothing else exists.

the repo
data-platform/
│
├── modules/                      # ── THE LOGIC. Written once, environment-agnostic. ──
│   ├── data-lake/
│   │   ├── versions.tf           # which providers this module needs (no config!)
│   │   ├── variables.tf          # its inputs
│   │   ├── main.tf               # the buckets, the IAM role, the lifecycle rules
│   │   └── outputs.tf            # what it hands back to the caller
│   ├── etl-job/                  # same four files: a Glue/Databricks job
│   └── warehouse/                # same four files: Snowflake WH + roles + grants
│
├── envs/                         # ── THE WIRING. One thin root module per environment. ──
│   ├── dev/
│   │   ├── backend.tf            # where DEV's state lives
│   │   ├── providers.tf          # provider config + version pins
│   │   ├── main.tf               # calls the modules with dev inputs
│   │   ├── outputs.tf            # re-publish the handful of values humans need
│   │   ├── terraform.tfvars      # dev's values (small, boring, no secrets)
│   │   └── .terraform.lock.hcl   # committed — exact provider versions
│   └── prod/                     # the SAME six files, different values
│
├── .github/workflows/terraform.yml   # plan on PR, apply on merge (§12)
└── .gitignore                        # .terraform/  *.tfstate  *.tfstate.backup

The five rules this layout encodes

(1) Roots are thin. An envs/*/main.tf should be almost boring — module calls and values, no resources. (2) Modules are dumb. They know nothing about "prod"; they take inputs. (3) State is split per environment, so a dev mistake physically cannot reach prod. (4) Providers are configured only in roots — never inside a module. (5) Everything is pinned — provider versions, module versions, and a committed lock file.

The module, file by file

Each module gets the same four files. That sameness is the point: once you've read one module, you can navigate any of them.

modules/data-lake/versions.tf — what it needs, not how to reach it

modules/data-lake/versions.tf
terraform {
  required_version = ">= 1.6"
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = ">= 5.0"          # a RANGE, not a pin — see the note below
    }
  }
}

Two things to notice, and both are subtle. First: there is no provider "aws" { } block here. The module declares that it needs the AWS provider, but the configuration of that provider (which region, which credentials) is inherited from whichever root calls it. Put a provider block in a shared module and you can never use it twice in one config — no multi-region, no multi-account. Second: a module declares a permissive range (>= 5.0), while a root declares a tight constraint (~> 5.31). The module is saying "I work with anything from 5.0 up"; the root decides what actually gets installed. If a module pinned an exact version, two modules could demand different versions and your config would be unresolvable.

modules/data-lake/variables.tf — the module's contract

modules/data-lake/variables.tf
variable "env" {
  type        = string
  description = "Environment name — used in every resource name"
  validation {
    condition     = can(regex("^(dev|staging|prod)$", var.env))
    error_message = "env must be dev, staging or prod."
  }
}

variable "zones" {
  description = "One bucket per zone, with its retention policy"
  type = map(object({
    retention_days = number
    versioning     = bool
  }))
}

variable "is_protected" {
  type        = bool
  default     = false          # safe default: dev can be destroyed…
  description = "Refuse to destroy these buckets (turn on in prod)"
}

variable "tags" {
  type    = map(string)
  default = {}                    # the root injects Owner / CostCentre / ManagedBy
}

This file is the module's public API — treat it like a function signature. Every variable is typed and described, because the description is what shows up when a colleague runs terraform-docs on it. Note is_protected defaults to false: defaults should always be the safe, cheap, dev answer, so that forgetting to set something gives you a small dev bucket rather than an undeletable production one. Prod then opts in explicitly.

modules/data-lake/main.tf — the resources

modules/data-lake/main.tf
locals {
  prefix = "acme-${var.env}"                    # one place that decides naming
  tags   = merge(var.tags, { Module = "data-lake" })
}

# one bucket per zone — keyed by NAME, so adding/removing a zone
# never disturbs the others (§09)
resource "aws_s3_bucket" "zone" {
  for_each = var.zones                        # { raw = {...}, curated = {...} }

  bucket = "${local.prefix}-${each.key}"       # acme-prod-raw
  tags   = local.tags

  lifecycle {
    prevent_destroy = true                    # see the trap below
  }
}

resource "aws_s3_bucket_versioning" "zone" {
  for_each = { for k, v in var.zones : k => v if v.versioning }

  bucket = aws_s3_bucket.zone[each.key].id     # ← the reference builds the graph (§07)
  versioning_configuration { status = "Enabled" }
}

resource "aws_s3_bucket_lifecycle_configuration" "zone" {
  for_each = var.zones
  bucket   = aws_s3_bucket.zone[each.key].id

  rule {
    id     = "expire"
    status = "Enabled"
    expiration { days = each.value.retention_days }
  }
}

# the role the ETL job assumes to read/write the lake
resource "aws_iam_role" "etl" {
  name = "${local.prefix}-etl"
  tags = local.tags
  assume_role_policy = jsonencode({
    Version   = "2012-10-17"
    Statement = [{
      Effect    = "Allow"
      Action    = "sts:AssumeRole"
      Principal = { Service = "glue.amazonaws.com" }
    }]
  })
}

The prevent_destroy trap you'll hit in about a week

The code above hard-codes prevent_destroy = true, which protects prod — and also makes terraform destroy on your dev environment fail, which is infuriating. The obvious fix is prevent_destroy = var.is_protected… and that doesn't work. The lifecycle block is evaluated before variables are resolved, so it accepts only literal values — no var., no local., no conditionals. This is one of Terraform's genuinely sharp edges. The real options: (a) accept it and never destroy dev (use -target or delete the state), (b) keep a separate module variant for protected data, or (c) rely on the deletion_protection-style attribute the provider itself offers, which is a normal argument and can take a variable. Know this before you design around it.

modules/data-lake/outputs.tf — the return values

modules/data-lake/outputs.tf
output "bucket_names" {
  description = "Zone name → bucket name, for the ETL job to consume"
  value       = { for k, b in aws_s3_bucket.zone : k => b.bucket }
  # → { raw = "acme-prod-raw", curated = "acme-prod-curated" }
}

output "etl_role_arn" {
  description = "Assume this role to read the lake"
  value       = aws_iam_role.etl.arn
}

Outputs are how one module talks to the next without either knowing about the other. The etl-job module doesn't hard-code a bucket name or an ARN — the root passes it in from this module's output. That single habit is what lets you rename a bucket in one place and have everything downstream follow.

The environment root, file by file

Now the wiring. This is the part a reviewer reads when they ask "what does prod actually look like?" — so it should be small enough to read in a minute.

envs/prod/backend.tf
terraform {
  backend "s3" {
    bucket       = "acme-tfstate"
    key          = "data-platform/prod.tfstate"   # ← dev uses .../dev.tfstate
    region       = "eu-west-1"                    #   DIFFERENT KEY = separate state = safety
    encrypt      = true
    use_lockfile = true
  }
}

The one line that matters is key. Dev and prod point at the same bucket but different objects, so they have separate state files and separate locks — an apply in dev can't corrupt or block prod. (Backend blocks can't use variables, which is exactly why this is a per-environment file rather than something clever and shared.)

envs/prod/providers.tf
terraform {
  required_version = ">= 1.6"
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.31"      # the ROOT pins tightly (the module only ranged)
    }
  }
}

provider "aws" {
  region = "eu-west-1"

  default_tags {              # ← tag EVERY resource, automatically
    tags = {
      Environment = "prod"
      ManagedBy   = "terraform"
      Repo        = "acme/data-platform"
    }
  }
}

default_tags is one of the highest-value five lines in this whole repo. Every resource the AWS provider creates gets these tags without you writing them anywhere else — which means six months from now, when something expensive appears in the bill, you can answer "who owns this and is it Terraform-managed?" instantly. ManagedBy = terraform in particular tells the next engineer not to fix it by hand.

envs/prod/main.tf — the whole of prod, in one readable file
# ── the lake ──
module "lake" {
  source = "../../modules/data-lake"

  env          = var.env
  zones        = var.zones          # comes from terraform.tfvars, below
  is_protected = true               # prod opts IN to protection
  tags         = { CostCentre = "data-eng" }
}

# ── the job that runs on it ──
module "etl" {
  source = "../../modules/etl-job"

  env          = var.env
  role_arn     = module.lake.etl_role_arn       # ← module → module, via outputs
  source_bucket = module.lake.bucket_names["raw"]
  target_bucket = module.lake.bucket_names["curated"]
  workers      = 8                              # dev passes 1
}

# ── the warehouse the analysts query ──
module "warehouse" {
  source = "../../modules/warehouse"

  env            = var.env
  warehouse_size = "LARGE"                    # dev passes XSMALL
  readers        = ["ANALYST", "BI_TOOL"]
}

That's the entire production platform — three module calls and about twenty meaningful lines. Notice there is not a single resource block here. Notice too how module.lake.etl_role_arn flows into the etl module: the two modules never reference each other, only the root wires them together, which is what keeps them independently reusable. And because those are real references, Terraform derives the ordering itself — the lake is built before the job that needs it (§07).

envs/prod/terraform.tfvars — the values, and nothing else
env = "prod"

zones = {
  raw     = { retention_days = 3650, versioning = true  }   # 10y — the source of truth
  curated = { retention_days = 730,  versioning = true  }   # 2y — rebuildable, but precious
  scratch = { retention_days = 7,    versioning = false }   # a week — genuinely disposable
}

# NOTE: no passwords, no keys, no tokens. Ever. Those come from the
# secret store at apply time — this file is committed to Git. (§15)

Compare with envs/dev/terraform.tfvars, which is the same file with env = "dev" and retention measured in days rather than years. That diff is the entire difference between your environments — and it fits on a screen, which is exactly the property you want.

The life of a change

Now watch it work. Suppose the analytics team asks for a new features zone in the lake. Here is every step, and the thing to notice is how little of it involves Terraform knowledge at all:

1
Edit one file. Add features = { retention_days = 365, versioning = true } to envs/prod/terraform.tfvars. That's it — no new resources, no new module, no copy-paste. The for_each in the module does the rest.
2
Open a PR. CI runs fmt, validate, then terraform plan, and posts the diff as a comment (§12).
3
Read the plan. This is the moment the whole design pays off — because of for_each, the plan touches only the new bucket. Nothing else moves:
the plan a reviewer sees
  # module.lake.aws_s3_bucket.zone["features"] will be created
  + resource "aws_s3_bucket" "zone" {
      + bucket = "acme-prod-features"
      + tags   = { Environment = "prod", ManagedBy = "terraform", … }
    }
  # module.lake.aws_s3_bucket_versioning.zone["features"] will be created
  # module.lake.aws_s3_bucket_lifecycle_configuration.zone["features"] will be created

Plan: 3 to add, 0 to change, 0 to destroy.
#            ↑ exactly what you asked for, and NOTHING ELSE.
#   The existing raw/curated/scratch buckets aren't even mentioned —
#   that is what "keyed by name" buys you. With `count`, this same
#   change could have shown "3 to add, 0 to change, 2 to destroy". (§09)
4
Merge. CI applies the saved plan — the exact diff that was reviewed — behind the production environment's approval gate. The bucket exists. The PR is the audit trail.

When to grow beyond this shape

This layout comfortably carries a team to a few hundred resources per environment. The signals that you've outgrown it: plans take minutes (Terraform refreshes everything in state), or unrelated teams keep colliding on the lock. The fix is not a cleverer directory tree — it's splitting state further, by blast radius: envs/prod/networking/, envs/prod/data-lake/, envs/prod/warehouse/, each with its own backend key and its own plan. Where one needs a value from another, publish an output and read it with a terraform_remote_state data source. Same rules, smaller blast radius, faster plans.

15. Security, safety & the classic traps

Terraform holds the keys to your entire cloud, and its state file holds your secrets in plaintext. A short list of habits keeps that from becoming a headline.

Security

  • Never hard-code secrets in .tf files. They go to Git, and Git is forever. Inject them as variables from your CI secret store, or better, read them from a secrets manager at apply time (data "aws_secretsmanager_secret_version") so the value never lives in your repo.
  • Treat state as a secret. It contains generated passwords and keys in plaintext regardless of sensitive = true. Encrypt the state bucket, enable versioning, and restrict read access to the CI role and a couple of humans.
  • Prefer OIDC over long-lived cloud keys in CI — short-lived credentials minted per run, nothing permanent to leak.
  • Least privilege for the plan job. Planning only needs read. Give the PR job a read-only role and reserve write permissions for the apply job on main.
  • Scan before you apply. tfsec / checkov catch public buckets, unencrypted volumes and over-broad IAM in CI, before they exist.
  • .gitignore the right things — .terraform/, *.tfstate, *.tfstate.backup, and any *.tfvars holding secrets. Do commit .terraform.lock.hcl.

The traps everyone falls into once

learn these the easy way
− Running apply from a laptop against prod
  → nobody has prod credentials; CI applies, humans review (§12)

− Local state on a team
  → remote backend + locking, on day one (§06)

− Ignoring the plan and typing "yes" on autopilot
  → read every line that says destroy or replace. Every time.

− Using count for named things, then deleting one from the middle
  → for_each, keyed by name (§09)

− Hand-editing terraform.tfstate to "fix" something
  → state mv / import / moved blocks; and turn on bucket versioning (§06)

− Unpinned provider and module versions
  → ~> constraints + a committed lock file (§05)

− No prevent_destroy on the bucket holding 4 TB of raw data
  → add it now, before you read the next section (§09)

− One giant state file for the whole company
  → split by blast radius: per environment, per platform area (§11)

Split state by blast radius

As a config grows, every plan gets slower (Terraform refreshes everything in state) and every apply gets scarier (one bad change can touch anything). So teams split state along boundaries that rarely change together: per environment first (dev/prod), then per area (networking · storage · warehouse · orchestration). Each gets its own state file and its own plan. Where one needs a value from another, publish it as an output and read it with a terraform_remote_state data source — or, more loosely coupled, look it up by name/tag with a normal data source.

16. Quick-reference cheat sheet

The commands and blocks you'll reach for most, in one place.

terraform — keep this handy
── the loop ──────────────────────────────────────────
terraform init                    # providers + modules + backend (re-run when they change)
terraform init -upgrade           # deliberately bump provider versions
terraform fmt -recursive          # auto-format (run before every commit)
terraform validate                # syntax & type check — no cloud calls
terraform plan                    # preview. Changes NOTHING. Free. Do it often.
terraform plan -out=tfplan        # save the plan…
terraform apply tfplan            # …and apply exactly that
terraform apply -auto-approve     # no prompt (CI only)
terraform destroy                 # tear it all down (dev only, please)

── targeting & debugging ─────────────────────────────
terraform plan -target=aws_s3_bucket.raw   # just one resource (escape hatch, not a habit)
terraform plan -refresh-only               # "has anything drifted?" — no config changes
terraform apply -replace=aws_glue_job.etl  # force-recreate one thing (replaces old `taint`)
terraform output -raw bucket_name          # a value, unquoted, for scripts
terraform console                          # a REPL to try expressions
TF_LOG=DEBUG terraform plan                # when the provider is misbehaving

── state ─────────────────────────────────────────────
terraform state list              # everything Terraform tracks
terraform state show ADDR         # all known attributes of one resource
terraform state rm ADDR           # forget it (does NOT delete in the cloud)
terraform import ADDR real-id     # adopt an existing resource (or use an import block)
terraform force-unlock LOCK_ID    # only when a run died mid-apply. Be sure.

── the blocks ────────────────────────────────────────
terraform {}   # versions · required_providers · backend
provider {}    # how to reach a cloud (region, creds, alias)
resource {}    # CREATES and owns a thing        → addr: type.name
data {}        # READS a thing it doesn't own    → addr: data.type.name
variable {}    # an input   (var.name)
output {}      # a result   (module.mod.name)
locals {}      # a computed value (local.name)
module {}      # call a reusable folder of resources
import{} moved{} removed{}   # declarative state surgery, reviewable in a PR

── meta-arguments (usable on any resource) ───────────
for_each = toset([...]) / var.map   # many, keyed by NAME  ← prefer this
count    = var.enabled ? 1 : 0      # many, keyed by INDEX ← "zero or one" only
depends_on = [other.thing]          # explicit order (rarely needed)
provider   = aws.us                 # use an aliased provider
lifecycle { prevent_destroy = true · ignore_changes = [...] · create_before_destroy = true }

── the plan symbols ──────────────────────────────────
  + create   ~ update in-place   - destroy   -/+ REPLACE (destroy then create!)

＊ Glossary

Infrastructure as code (IaC)
Defining infrastructure in version-controlled text files instead of clicking in a console — so it's reviewable, repeatable and auditable.
Declarative
You describe the desired end state; the tool works out the steps. (Imperative = you write the steps.) This is why Terraform is safe to re-run.
Idempotent
Running the same apply twice changes nothing the second time — the destination is already reached.
HCL
HashiCorp Configuration Language — the block-based syntax Terraform files are written in.
Provider
The plugin that knows how to talk to one API (aws, databricks, snowflake). Downloaded by terraform init.
Resource
A thing Terraform creates and owns. Address: type.name. Delete the block and Terraform deletes the real thing.
Data source
A read-only lookup of something Terraform does not own. Creates nothing, deletes nothing.
State
Terraform's record of which real resource corresponds to which block in your code, plus its last-known attributes. Stored in terraform.tfstate. Contains secrets in plaintext.
Backend
Where state is stored. local by default; on a team, a remote backend (S3, Azure Blob, GCS, HCP Terraform) with locking.
State lock
A mutex the backend hands out so only one apply runs at a time — without it, two concurrent applies corrupt state.
Drift
When reality no longer matches state, because something was changed outside Terraform (usually by hand, during an incident).
Plan
The preview diff between your config, the state, and reality. Read-only. The thing you review before every apply.
Apply
The command that actually changes infrastructure — and the only one that does.
Replace (-/+)
Terraform must destroy and recreate a resource because an unchangeable attribute changed. The plan line to fear on stateful resources.
Dependency graph
The order of operations Terraform derives from the references in your code. Independent resources are built in parallel.
depends_on
An explicit dependency, for ordering Terraform can't infer from references. Use sparingly.
Module
A reusable folder of .tf files, called with inputs and returning outputs. Terraform's function.
Root module
The directory you actually run terraform in. Where providers are configured; the only place a backend is declared.
Registry
The public catalogue of providers and modules (registry.terraform.io).
Lock file
.terraform.lock.hcl — pins the exact provider versions and checksums. Commit it.
count vs for_each
Two ways to make many copies: by index (fragile — removing one renumbers the rest) or by name (stable). Prefer for_each.
prevent_destroy
A lifecycle rule that makes Terraform refuse any plan that would delete the resource. Put it on production data stores.
Workspace
Multiple state files for one configuration. Good for ephemeral per-PR environments; a poor fit for the dev/prod ladder — use directories.
OIDC
Short-lived, verifiable identity a CI run presents to a cloud, replacing long-lived stored keys.
OpenTofu
The open-source (MPL) fork of Terraform, created after the 2023 licence change; a drop-in replacement for everything here.

That's the full toolkit: the three worlds, the plan/apply loop, HCL end to end, state and locking, the graph, for_each, modules, environments, CI/CD, the data-platform specifics, and a worked repo you can copy. Bookmark §16 as your day-to-day reference — and remember the through-line: read the plan, respect the state, pin your versions, and never let a machine destroy what you can't rebuild.
