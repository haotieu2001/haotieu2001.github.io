01. Why Python runs data engineering

Python is the glue and the control plane of the modern data stack — it does not crunch the data, it tells the things that crunch data what to do.

Here is the uncomfortable truth a lot of newcomers miss: Python is a genuinely mediocre number cruncher. It was never designed to join two 200-million-row tables or group-by a terabyte of events. It is interpreted (every line is dispatched by the CPython virtual machine at runtime), a single process is throttled by the Global Interpreter Lock so pure-Python code cannot use more than one core at a time for CPU-bound work, and everything it touches lives in the RAM of one machine. Try to sort a dataset bigger than memory in a plain Python list and the kernel's OOM killer ends the conversation.

So why is Python the lingua franca of data engineering? Because the job is not crunching — it is moving, shaping, and orchestrating. The heavy lifting is pushed down to purpose-built engines written in C++/Rust/Java: Spark, DuckDB, Polars, and the warehouses (Snowflake, BigQuery). Python is the driver's seat: it pulls from REST APIs with requests, parses JSON, reads config, wires DAGs together in Airflow / Dagster / Prefect, and hands the actual compute to an engine. Internalise one mental model and most architecture decisions become obvious:

The one mental model

Python drives, engines crunch. Your Python code should be a thin, well-tested orchestration layer. The moment you find yourself writing a manual for-loop to aggregate millions of rows, stop — that work belongs in SQL, DuckDB, Polars, or Spark.

Python drives: pull from an API, hand a DataFrame to the engine

The canonical pattern below is 90% of an extract step: Python owns the network I/O and the shaping, then hands a frame straight into DuckDB, which does the crunching in native code.

python
# Python DRIVES: pull JSON from an API, shape it, let DuckDB crunch it.
import requests, duckdb, pandas as pd

def pull_orders(base: str, page_size: int = 500) -> pd.DataFrame:
    rows, page = [], 1
    while True:
        r = requests.get(f"{base}/orders", params={"page": page, "size": page_size}, timeout=30)
        r.raise_for_status()                 # fail loud on 4xx/5xx
        batch = r.json()["data"]
        if not batch:
            break                            # no more pages
        rows.extend(batch)
        page += 1
    return pd.DataFrame(rows)              # shaping = Python's job

df = pull_orders("https://api.example.com/v1")

# Crunching = the engine's job. DuckDB reads the DataFrame in place.
top = duckdb.sql("""
    SELECT region, SUM(amount) AS revenue
    FROM df
    GROUP BY region
    ORDER BY revenue DESC
    LIMIT 10
""").df()
print(top)

The same shape applies to a database source — Python opens the connection and describes the query, the database does the work, and you get a frame back:

python
import pandas as pd
from sqlalchemy import create_engine

engine = create_engine("postgresql+psycopg://user:pw@warehouse:5432/prod")

# The WHERE/GROUP BY runs IN Postgres. Python just receives the result.
df = pd.read_sql(
    "SELECT date, SUM(amount) AS rev FROM sales WHERE date >= '2026-01-01' GROUP BY date",
    engine,
)

The RAM ceiling

A Python list or a pandas DataFrame lives entirely in RAM, and pandas often needs several times the file size in working memory. pd.read_csv("50gb.csv") on a 16 GB laptop does not run slowly — it dies. When data outgrows memory, stream it (Section 04), push it into DuckDB/Polars with lazy execution, or offload to Spark/the warehouse. "It worked on the sample" is how OOM incidents are born.

02. The four structures: list, dict, set, tuple

Pick the right container and half your performance problems never happen — the wrong one turns an O(n) job into O(n squared).

Four built-ins carry nearly every data pipeline. Knowing not just what they do but their cost model is what separates a job that finishes in seconds from one that hangs.

list — the dynamic array

A list is a growable array of references. Appending and indexing are O(1) (amortised for append), but membership testing (x in mylist) scans element by element — O(n). Great for ordered batches of records you iterate once; terrible as a lookup table.

dict — the hash map, the DE workhorse

A dict is a hash table: average O(1) insert, lookup and delete. Since Python 3.7 it also preserves insertion order. This is the single most important structure for a data engineer — it is how you build in-memory lookups, join keys, and grouping accumulators without an engine.

set — dedup, membership, set algebra

A set is a dict without values: O(1) membership and automatic deduplication, plus real set algebra (| union, & intersection, - difference). Perfect for diffing two datasets — which IDs are new, which disappeared.

tuple — immutable and hashable

A tuple is a fixed, immutable sequence. Because it is immutable it is hashable, so a tuple can be a dict key or a set member — the standard trick for a composite key like (region, product). Tuples also power multiple return values and unpacking.

python
# list: an ordered batch of records you scan once
events = [{"user": 1, "amt": 10}, {"user": 2, "amt": 7}]

# dict: an O(1) lookup table (the join-side of a small dimension)
user_region = {1: "APAC", 2: "EMEA"}
for e in events:
    e["region"] = user_region.get(e["user"], "UNKNOWN")   # enrich = dict lookup

# set: diff yesterday vs today to find new / churned ids
yesterday = {1, 2, 3}
today     = {2, 3, 4}
new_ids    = today - yesterday      # {4}
churned    = yesterday - today      # {1}

# tuple: a composite key for a set / dict
seen = set()
key = ("APAC", "widget")     # hashable because it's a tuple
seen.add(key)
print(key in seen)          # True, O(1)

The collections toolkit

collections gives you specialised structures that remove boilerplate and speed common DE patterns:

  • defaultdict — auto-creates a default value, so grouping needs no "if key not in dict" dance.
  • Counter — frequency counting and top-N in one line.
  • deque — a double-ended queue with O(1) appends/pops at both ends; ideal for a rolling window or a bounded buffer.
  • namedtuple — a lightweight, immutable record with named fields (readability without a class).
python
from collections import defaultdict, Counter, deque, namedtuple

# group amounts by region WITHOUT checking membership first
by_region = defaultdict(list)
for e in events:
    by_region[e["region"]].append(e["amt"])

# frequency + top-N in one shot
paths = ["/a", "/b", "/a", "/a", "/c"]
print(Counter(paths).most_common(2))   # [('/a', 3), ('/b', 1)]

# rolling buffer of the last 3 latencies (auto-drops the oldest)
window = deque(maxlen=3)
for ms in (12, 18, 9, 30):
    window.append(ms)
print(list(window))                # [18, 9, 30]

Row = namedtuple("Row", ["region", "amt"])
r = Row("APAC", 10)
print(r.region, r.amt)          # named fields, immutable

Two traps that bite everyone

1. x in a_list is O(n). Filtering a million rows against a list of allowed IDs is O(n·m) — potentially billions of comparisons. Convert the lookup side to a set first and it becomes O(n). This single change routinely turns a 20-minute job into a 2-second one.
2. The mutable default argument. def f(x, acc=[]) creates acc once and shares it across every call — state leaks between runs. Use acc=None then acc = acc or [] inside (covered in Section 05).

03. Comprehensions & generator expressions

Comprehensions are how Pythonic transforms are written — a filtered map in one readable, fast line — and their lazy sibling lets you stream the same logic without ever building a list.

A comprehension builds a collection from an iterable in a single expression. It reads top-to-bottom like the sentence "give me X for each item where condition", and it runs faster than the equivalent for-loop because the iteration happens in C rather than as interpreted bytecode. There is a comprehension for each of the core containers.

python
records = [{"id": 1, "amt": 10, "ok": True},
           {"id": 2, "amt": 0,  "ok": False},
           {"id": 3, "amt": 7,  "ok": True}]

# list comprehension: map + filter
amounts = [r["amt"] for r in records if r["ok"]]        # [10, 7]

# dict comprehension: build a lookup in one line
by_id = {r["id"]: r for r in records}

# set comprehension: distinct valid ids
valid_ids = {r["id"] for r in records if r["ok"]}

# conditional EXPRESSION inside a comprehension (ternary, not a filter)
labels = ["hi" if r["amt"] > 5 else "lo" for r in records]

# nested: flatten a list of batches into rows
batches = [[1, 2], [3, 4]]
flat = [x for batch in batches for x in batch]   # [1,2,3,4]

Note the two ways if appears. As a filter it goes after the loop (... for r in records if r["ok"]). As a ternary expression it goes before the loop ("hi" if x > 5 else "lo" for x in ...) — that form must have an else.

Generator expressions: same syntax, lazy execution

Swap the square brackets for parentheses and you get a generator expression. It looks identical but builds nothing up front — it yields items one at a time as they are consumed. This is the difference between allocating a list of a million integers and streaming through them one by one. When the generator is the sole argument to a function you can even drop the extra parentheses.

python
# LIST comp: builds the whole list in RAM, then sums it
total_eager = sum([len(line) for line in open("huge.log")])

# GENERATOR expr: streams line by line, holds one line at a time
total_lazy = sum(len(line) for line in open("huge.log"))

# chain lazy steps: parse -> filter -> count, nothing materialised
lines   = (l.rstrip() for l in open("events.jsonl"))
errors  = (l for l in lines if '"level":"ERROR"' in l)
n_errors = sum(1 for _ in errors)

List-comp vs. gen-exp: the memory tradeoff

A list comprehension [...] materialises every element in RAM at once — use it when you need the collection again (index it, len it, iterate twice). A generator expression (...) is lazy and O(1) in memory — use it when you consume the result once, especially over a file or API stream. Rule of thumb for DE: if the source could be large, reach for the parentheses. If you only ever call sum(), any(), max() or a single loop on it, a generator is strictly better.

Know when to stop

A comprehension with three nested loops and two conditionals is a write-only riddle. When it no longer reads like a sentence, use a plain for-loop — clarity beats cleverness in code your on-call teammate has to debug at 3am. Also remember a generator is single-use: once consumed it is exhausted, and iterating it again yields nothing.

04. Generators & iterators: streaming without OOM

Generators let you process a file bigger than RAM one record at a time — this is the single most important Python technique for a data engineer, and it is what stops your pipeline from crashing with Out-Of-Memory.

A generator function is any function containing yield. Calling it does not run the body — it returns a generator object, a lazy iterator. Each time you ask for the next value the function runs up to the next yield, hands back that value, and freezes its entire state (local variables, position) until you ask again. The consequence for data engineering is enormous: you can iterate a 50 GB log while only ever holding a single line in memory.

python
def read_lines(path: str):
    # opening a file is ALREADY lazy — iterating yields one line at a time
    with open(path) as f:
        for line in f:
            yield line.rstrip("\n")

gen = read_lines("huge.log")   # nothing read yet
first = next(gen)               # now ONE line is read

Chaining generators into a streaming pipeline

Because each stage is lazy, you can stack generators into a pipeline where data flows through one record at a time — parse, filter, transform — and memory stays flat regardless of file size. yield from delegates to a sub-iterator, which is handy for fanning multiple files into one stream.

python
import json

def parse(lines):
    for line in lines:
        yield json.loads(line)

def only_paid(records):
    for r in records:
        if r["status"] == "paid":
            yield r

def all_files(paths):
    for p in paths:
        yield from read_lines(p)   # delegate: fan many files into one stream

# the pipeline: memory is O(1), file size can be terabytes
pipeline = only_paid(parse(all_files(["a.jsonl", "b.jsonl"])))
for rec in pipeline:
    ...   # write to warehouse, one record at a time

Batching a stream with itertools

You rarely insert one row at a time — you batch. itertools is the standard-library toolkit for iterators: islice (take/skip without materialising), chain (concatenate iterables), groupby (group a sorted stream), and on 3.12+ batched (chunk into fixed sizes). Here is the streaming-plus-batching pattern that underpins most loaders:

python
from itertools import islice

def batched(iterable, n: int):
    # backport for < 3.12; on 3.12+ use itertools.batched
    it = iter(iterable)
    while chunk := tuple(islice(it, n)):
        yield chunk

# stream a huge file, load 1000 rows per round-trip to the DB
rows = parse(read_lines("huge.jsonl"))
for chunk in batched(rows, 1000):
    bulk_insert(chunk)      # only 1000 records in RAM at any moment

Generator gotchas

A generator is exhausted after one pass — you cannot loop it twice or call len() on it. If you need the data again, materialise with list(gen) (and accept the memory cost). itertools.groupby only groups consecutive keys, so sort first or you will get fragmented groups. And do not silently swallow the file handle: opening inside the generator with with guarantees it closes when the generator is fully consumed or garbage-collected.

05. Functions as first-class citizens

In Python a function is just another object you can pass around, return, and store — which is exactly what lets you build reusable, composable transforms instead of copy-pasting logic.

Functions are first-class: you can assign one to a variable, put it in a list or dict, pass it as an argument, and return it from another function. A higher-order function is one that takes or returns a function. This is the foundation of pluggable pipelines — a generic runner that accepts whichever transform you hand it.

python
# functions are objects: store them, dispatch on them
def to_upper(s): return s.upper()
def strip_ws(s): return s.strip()

CLEANERS = {"upper": to_upper, "strip": strip_ws}   # a registry

def clean(value: str, steps: list[str]) -> str:
    for name in steps:
        value = CLEANERS[name](value)   # look up a fn and call it
    return value

print(clean("  hi ", ["strip", "upper"]))   # "HI"

Flexible signatures: *args, **kwargs, keyword-only

*args captures extra positional arguments as a tuple; **kwargs captures extra keyword arguments as a dict. A bare * in the signature forces everything after it to be passed by name — invaluable for config-heavy functions where positional args would be unreadable. The same */** operators unpack a sequence/dict back into arguments at the call site.

python
def connect(host: str, *, port: int = 5432, timeout: int = 30, **opts):
    # port & timeout are keyword-only (after the bare *)
    return {"host": host, "port": port, "timeout": timeout, **opts}

cfg = {"port": 6543, "sslmode": "require"}
conn = connect("warehouse", **cfg)     # unpack a dict into kwargs
# connect("warehouse", 6543)  ->  TypeError: port is keyword-only

lambda, map/filter, and why comprehensions usually win

A lambda is a small anonymous function, useful as a throwaway key= for sorting. map and filter apply a function lazily across an iterable, but in idiomatic Python a comprehension is usually clearer and just as fast — reserve map/filter for when you already have a named function to apply.

python
rows = [{"id": 3, "amt": 10}, {"id": 1, "amt": 50}]
rows.sort(key=lambda r: r["amt"], reverse=True)   # sort by amount desc

# map/filter (lazy) ...
ids = map(lambda r: r["id"], rows)
# ... but a comprehension reads better and is preferred:
ids = [r["id"] for r in rows]

from functools import partial
# closures & partial: bake in config to build a reusable transform
def rounder(digits):
    def _round(x): return round(x, digits)   # closes over `digits`
    return _round

round2 = rounder(2)
to_cents = partial(round, ndigits=2)   # same idea, stdlib style
print(round2(3.14159), to_cents(3.14159))   # 3.14 3.14

The mutable default argument trap

Default arguments are evaluated once, when the function is defined — not per call. So def f(x, acc=[]) shares one list across every invocation, and state from run 1 leaks into run 2. In a long-running worker this is a silent, growing bug. The fix is the sentinel pattern:
def f(x, acc=None): then acc = acc if acc is not None else []. The same applies to {} and any mutable default.

06. Decorators: retry, timing, logging

A decorator wraps a function to add behaviour — retries, timing, logging — without touching the function's body, which is exactly the cross-cutting glue a pipeline needs everywhere.

A decorator is a callable that takes a function and returns a new function, usually one that calls the original with some extra behaviour around it. The @name syntax above a def is just sugar for fn = name(fn). Always wrap the inner function with functools.wraps so the wrapper keeps the original's name, docstring and signature — otherwise every decorated function shows up as wrapper in tracebacks and logs.

python
import functools, time

def timing(fn):
    @functools.wraps(fn)                # preserve name/docstring
    def wrapper(*args, **kwargs):
        t0 = time.perf_counter()
        try:
            return fn(*args, **kwargs)
        finally:
            dt = time.perf_counter() - t0
            print(f"{fn.__name__} took {dt:.3f}s")
    return wrapper

@timing
def extract():
    time.sleep(0.1)
    return "done"

Decorators with arguments: a factory returning a decorator

To parameterise a decorator (say, "retry up to 5 times") you add one more layer: an outer function that takes the arguments and returns a decorator. Three nested levels — config, function, call — but the pattern is always the same. Here is the workhorse every DE writes eventually: @retry with exponential backoff for a flaky API or database call.

python
import functools, time

def retry(max_attempts: int = 3, backoff: float = 0.5, exc=Exception):
    def decorator(fn):
        @functools.wraps(fn)
        def wrapper(*args, **kwargs):
            for attempt in range(1, max_attempts + 1):
                try:
                    return fn(*args, **kwargs)
                except exc as e:
                    if attempt == max_attempts:
                        raise                       # give up, re-raise
                    sleep = backoff * 2 ** (attempt - 1)   # exponential
                    print(f"attempt {attempt} failed: {e!r}; retry in {sleep}s")
                    time.sleep(sleep)
        return wrapper
    return decorator

@retry(max_attempts=5, backoff=0.5)
def fetch(url: str):
    import requests
    r = requests.get(url, timeout=10)
    r.raise_for_status()
    return r.json()

Stacking decorators

Decorators stack bottom-up: the one nearest the def wraps first, the topmost wraps last and therefore runs its "before" code first. Read them as an onion from the inside out.

python
def log(fn):
    @functools.wraps(fn)
    def wrapper(*a, **k):
        print(f"→ {fn.__name__}")
        return fn(*a, **k)
    return wrapper

@timing          # outermost: runs last around the call
@retry(max_attempts=3)
@log             # innermost: closest to the function
def pull(url): ...
# equivalent to: pull = timing(retry(3)(log(pull)))

Decorator gotchas

Forgetting @functools.wraps silently breaks introspection, logging and some frameworks that read fn.__name__. A naive @retry that catches a bare Exception will happily retry a ValueError from bad data — retry only the transient, retryable errors (timeouts, 5xx, connection resets), and never blindly retry a non-idempotent write. Finally, exponential backoff without a cap or jitter can stampede a recovering service; in production add a max delay and a little randomness.

07. Classes & OOP, properly

You won't write classes to model cats and dogs. You'll write them to model the moving parts of a pipeline — a connector to a source, a config object, a reusable transformation step — so the code stays testable and doesn't rot.

Object-oriented programming has one core idea: bundle state (data) together with the behaviour (functions) that acts on it, behind a clean interface. A function that takes inputs and returns outputs with nothing to remember should stay a function. But the moment something needs to hold on to state between calls — an open database connection, a running row-count, a set of credentials, a buffer of rows waiting to be flushed — a class starts to earn its keep. That is exactly why the canonical DE example is a connector: it holds a live connection and exposes read() / write() methods that use it.

The honest rule: don't reach for a class too early

A stateless transform — "take this DataFrame, return a cleaned one" — is just a function. Wrapping it in a class with a single method and no state is over-engineering. Ask: is there state to keep, or a contract several implementations must share? If neither, write a function (or a dataclass, §08, if you only need to hold data).

Anatomy: class, __init__, and self

A class is a blueprint; an instance is one object built from it. __init__ is the initialiser — it runs right after the instance is created and sets up its starting state. self is the instance itself, handed to every method automatically as the first argument. When you write conn.read(), Python calls read(conn) under the hood — self is conn. That is the whole trick; once self stops being mysterious, classes get easy.

python
class SQLConnector:
    """Holds a connection and knows how to run queries against it."""

    def __init__(self, dsn: str, timeout: int = 30):
        self.dsn = dsn              # instance attribute: unique to THIS object
        self.timeout = timeout
        self.conn = None          # not connected yet — lazy
        self._rows_read = 0       # leading _ = "internal, don't touch from outside"

    def connect(self) -> None:
        # imagine a real driver here (psycopg, etc.)
        self.conn = open_connection(self.dsn, timeout=self.timeout)

    def read(self, sql: str) -> list[dict]:
        if self.conn is None:
            raise RuntimeError("call connect() first")
        rows = self.conn.execute(sql).fetchall()
        self._rows_read += len(rows)   # state that persists across calls
        return rows

db = SQLConnector("postgres://…")   # __init__ runs now
db.connect()
rows = db.read("SELECT * FROM orders LIMIT 10")

Notice what the class buys you: db remembers its dsn, its live conn, and a running _rows_read — the read method uses and updates that state. A bare function could not, without you passing everything in and out every single call.

Instance attributes vs class attributes

An instance attribute (set on self) belongs to one object. A class attribute (set in the class body) is shared by every instance. Class attributes are perfect for constants — a default, a registry, a schema version. They are a classic footgun when they are mutable, because every instance mutates the same shared object.

python
class Loader:
    source_type = "postgres"     # class attribute: shared, fine because it's immutable
    seen_files = []             # DANGER: shared mutable list

    def __init__(self, name):
        self.name = name        # instance attribute: per-object

a, b = Loader("a"), Loader("b")
a.seen_files.append("f1.csv")
print(b.seen_files)             # ['f1.csv'] — b sees a's file! same list

The shared-mutable-class-attribute trap

If each instance needs its own list/dict/set, initialise it in __init__ (self.seen_files = []), never in the class body. It's the same bug as the mutable default argument (§05), and it bites hardest when one long-lived object leaks state into the next run.

The three method flavours: instance, @classmethod, @staticmethod

Regular methods take self and work on one instance. A @classmethod takes the class (cls) instead of an instance — its killer use in DE is the alternate constructor: a tidy from_config() / from_uri() that builds and returns an instance. A @staticmethod takes neither; it is just a plain function that logically lives on the class (a helper you don't want floating in the module).

python
class SQLConnector:
    def __init__(self, host, port, user):
        self.host, self.port, self.user = host, port, user

    @classmethod
    def from_uri(cls, uri: str) -> "SQLConnector":
        # parse "host:port/user" then build an instance via cls(...)
        host, rest = uri.split(":")
        port, user = rest.split("/")
        return cls(host, int(port), user)   # cls, not SQLConnector → subclasses work too

    @staticmethod
    def mask(secret: str) -> str:
        return secret[:2] + "***"       # pure helper, no self/cls needed

db = SQLConnector.from_uri("db.internal:5432/etl")   # reads much better than SQLConnector(...)

@property: a computed attribute with a method behind it

@property lets a method be accessed like an attribute — no parentheses. Use it for values derived from state, or to guard access, without forcing callers to remember which things are fields and which are methods. Add a setter only when assignment should be allowed (and validated).

python
class Batch:
    def __init__(self, rows):
        self._rows = rows

    @property
    def size(self) -> int:
        return len(self._rows)      # computed on access, always correct

    @property
    def is_empty(self) -> bool:
        return self.size == 0

b = Batch([{...}, {...}])

print(b.is_empty)  # False

The __eq__ / __hash__ pact

If you define __eq__, Python sets __hash__ to None — the object becomes unhashable and can't go in a set or be a dict key. If two objects are equal they must hash equal, so define both together (or make the type frozen with a dataclass, §08, which does it for you). Rule: equal objects, equal hashes — never one without the other.

Inheritance and super()

Inheritance models an is-a relationship: a PostgresConnector is a Connector. The subclass inherits the parent's methods and can add or override them; super() calls back up to the parent's version (most often super().__init__(...) so the parent still sets up its part of the state). When you look up an attribute, Python walks the MRO (method resolution order) — child first, then parents — and uses the first match.

python
class Connector:
    def __init__(self, name: str):
        self.name = name
        self.reads = 0

    def read(self, query):
        raise NotImplementedError       # subclasses must define this

class PostgresConnector(Connector):     # PostgresConnector is-a Connector
    def __init__(self, name, dsn):
        super().__init__(name)          # let the parent set name + reads
        self.dsn = dsn                  # then add our own state

    def read(self, query):              # override with the real behaviour
        self.reads += 1
        return run_pg(self.dsn, query)

Abstract base classes: writing a contract

Raising NotImplementedError only fails when the method is called. An abstract base class (from the abc module) fails at instantiation if a required method is missing — you can't even build an incomplete connector. This is the standard DE pattern: define one Connector interface — every source (Postgres, S3, Mongo, an API) must implement read and write — and the rest of your pipeline can treat them interchangeably.

python
from abc import ABC, abstractmethod

class Connector(ABC):
    @abstractmethod
    def read(self, query: str) -> list[dict]: ...
    @abstractmethod
    def write(self, rows: list[dict]) -> None: ...

class S3Connector(Connector):
    def read(self, query):  return read_from_s3(query)
    def write(self, rows): write_to_s3(rows)

# Connector()          → TypeError: can't instantiate abstract class
# a half-built subclass missing write() → TypeError at construction, not in prod
src: Connector = S3Connector()      # the pipeline just needs "a Connector"
def run(src: Connector, dst: Connector):   # works for ANY connector — polymorphism
    dst.write(src.read("…"))

Composition over inheritance

Inheritance is the right tool for a genuine is-a with shared behaviour. For almost everything else, prefer composition — has-a. A Pipeline is not a special kind of Connector; it has a source, a destination, and a list of steps. Composition keeps pieces swappable and testable, and it sidesteps the brittle, deep hierarchies that inheritance tempts you into.

python
class Pipeline:
    def __init__(self, source: Connector, sink: Connector, steps: list):
        self.source = source     # has-a source
        self.sink = sink         # has-a sink
        self.steps = steps       # has-a list of transform callables

    def run(self, query: str):
        rows = self.source.read(query)
        for step in self.steps:   # each step is just a function/callable
            rows = step(rows)
        self.sink.write(rows)

Class, function, or dataclass? A quick decision

Function — stateless input→output (most transforms). Dataclass (§08) — you only need to hold structured data (a config, a record, a schema). Class — you have state and behaviour (a connector, a buffered writer), or a contract multiple implementations must share (an ABC). When in doubt, start with a function and let a real need for state pull you toward a class.

08. Dataclasses & Pydantic

Stop passing bare dicts around your pipeline — give every record and config a typed shape, and decide up front whether you want cheap structure (dataclass) or real runtime validation (Pydantic).

A data engineer moves records for a living, and the worst bugs come from records that look right but aren't: a missing key, a string where a number should be, a config typo that only blows up at 3am. @dataclass and Pydantic both replace the "just use a dict" habit with explicit, self-documenting types — they differ on when the shape is checked.

Dataclasses: structure without boilerplate

The @dataclass decorator inspects the class annotations and auto-generates __init__, __repr__ and __eq__ for you. You write the fields once; you get a constructor, a readable repr for logs, and value equality for free.

python
from dataclasses import dataclass, field

@dataclass
class Record:
    id: int
    name: str
    tags: list[str] = field(default_factory=list)  # NOT tags=[]

r = Record(1, "orders")
print(r)               # Record(id=1, name='orders', tags=[]) — free __repr__
print(r == Record(1, "orders"))  # True — free value __eq__

The mutable-default trap

Never write tags: list = [] — that single list is shared across every instance, so appending to one record mutates them all. Use field(default_factory=list) so each instance gets a fresh list. This is the same shared-mutable footgun as a mutable default argument.

The flags that matter for DE

  • frozen=True — makes instances immutable and hashable. Perfect for a PipelineConfig you never want mutated mid-run, and it can be a dict key or set member.
  • order=True — generates __lt__/__gt__ etc. so instances sort by field order (handy for sorting records or watermarks).
  • slots=True (3.10+) — stores fields in __slots__ instead of a per-instance __dict__: less memory and faster attribute access, which adds up when you materialise millions of records.
  • __post_init__ — runs right after the generated __init__; use it for validation or derived fields.
python
from dataclasses import dataclass

@dataclass(frozen=True, slots=True)
class PipelineConfig:
    source_bucket: str
    batch_size: int = 1000
    retries: int = 3

    def __post_init__(self):
        if self.batch_size <= 0:
            raise ValueError("batch_size must be positive")

cfg = PipelineConfig(source_bucket="s3://raw", batch_size=500)
# cfg.batch_size = 10   -> FrozenInstanceError: can't reassign a frozen field
seen = {cfg}              # hashable because frozen -> usable in a set / dict key
print(cfg.retries)        # 3

Dataclass vs namedtuple vs plain class vs dict

A namedtuple is immutable, indexable and very light, but you can't add methods cleanly and every field is positional. A plain class gives full control but you hand-write all the dunder methods. A raw dict has zero schema — any key, any type, silent typos. The dataclass is the sweet spot for internal records: typed fields, real methods, tiny boilerplate.

Pydantic: validation at the boundary

Dataclasses trust you. The moment data crosses a boundary you don't control — an HTTP payload, a Kafka message, a CSV from a partner — you want runtime checks. Pydantic's BaseModel validates and coerces types on construction and raises ValidationError with a precise, machine-readable list of what failed. It's the de-facto standard for validating external input (FastAPI is built on it).

09. Type hints and why teams enforce them

Type hints don't do anything at runtime — their entire payoff is static analysis, editor autocomplete and pipelines that document themselves.

This is the single most misunderstood feature for newcomers: annotating path: str does not make Python reject a non-string. The interpreter records the hints in __annotations__ and otherwise ignores them. The value comes from tools that read them before your code runs — mypy and pyright — plus IDE autocomplete and the plain readability of a signature that tells the next engineer exactly what flows through it.

Modern syntax (3.11+)

  • Built-in generics: list[int], dict[str, int], tuple[str, ...] (3.9+ — no more typing.List).
  • Union with |: str | None instead of Optional[str], int | str instead of Union[int, str] (3.10+).
  • Callable[[int], str] for functions, TypedDict for dict-shaped records, Literal["append", "overwrite"] for fixed choices, Final for constants.
  • TypeVar/Generic for reusable generic containers, and Protocol for structural (duck) typing that a checker can still verify.
python
from typing import Protocol, TypedDict, Literal, Final

MAX_ROWS: Final = 1_000_000

class Row(TypedDict):        # a dict with a known shape
    id: int
    name: str

class Sink(Protocol):        # structural: anything with write() fits
    def write(self, rows: list[Row]) -> int: ...

def flush(rows: list[Row],
          sink: Sink,
          mode: Literal["append", "overwrite"] = "append") -> int | None:
    if not rows:
        return None
    return sink.write(rows)   # mypy knows write() takes list[Row]

Why DE teams gate CI on mypy

A pipeline that silently coerces or drops the wrong type doesn't crash — it writes bad data, and you find out days later from a broken dashboard. Running mypy --strict as a required CI check catches whole classes of bugs (passing None where a value is required, wrong argument order, a function that forgets a return path) before code reaches production. It also turns every signature into living documentation: a new engineer reads def extract(cfg: PipelineConfig) -> Iterator[Row] and instantly knows the contract without running anything.

Type hints lie at runtime

A str hint won't stop an int. def f(x: str): return x + "!" called as f(5) raises a TypeError from the +, not from the annotation. If you need real enforcement at a boundary, validate with Pydantic (section 08) — hints alone are checked only by tools you choose to run.

10. Errors, retries & logging

Production pipelines don't fail gracefully by accident — you design it in with specific exceptions, chained context, retries with backoff, and structured logs instead of print.

Handle the exceptions you expect, and only those

The try/except/else/finally shape gives you four slots: try for the risky call, except for specific failures, else for the success path (runs only if no exception), and finally for cleanup that runs no matter what. Catch specific exception types — a bare except: swallows KeyboardInterrupt and hides the real cause.

Never bare-except, never silently swallow

except: pass is how a pipeline "succeeds" while writing zero rows. Catch the narrowest exception you can name, log it, and either re-raise or route the record to a dead-letter store. Silence is the enemy of debuggability.

Custom exceptions and chaining

Define your own exception types (class ExtractError(Exception)) so callers can catch your failures distinctly from a stray KeyError. When you translate a low-level error into a domain one, use raise NewError(...) from e — the chain preserves the original traceback so you keep the root cause. Python idiomatically prefers EAFP ("easier to ask forgiveness than permission" — just try it and catch) over LBYL ("look before you leap" — check first), because the check-then-act pattern races and misses edge cases.

python
import logging, time, json

log = logging.getLogger("extract")

class ExtractError(Exception):
    """Raised when a source record cannot be pulled."""

def fetch(record_id: int) -> dict:
    if record_id == 13:
        raise ConnectionError("socket reset")
    return {"id": record_id}

def extract(record_id: int, retries: int = 3) -> dict:
    err: Exception | None = None       # hold the cause; `as e` is cleared per-loop
    for attempt in range(1, retries + 1):
        try:
            return fetch(record_id)          # EAFP: just try it
        except ConnectionError as e:      # specific, not bare
            err = e                          # keep it — `e` is out of scope after the block
            wait = 2 ** attempt              # exponential backoff
            log.warning("fetch failed",
                        extra={"record_id": record_id,
                               "attempt": attempt, "retry_in": wait})
            time.sleep(wait)
    # out of retries -> translate to a domain error, keep the cause
    raise ExtractError(f"gave up on {record_id}") from err

Logging, not print

print goes to stdout with no level, no timestamp and no structure — useless once your job runs on a scheduler. The logging module gives you severity levels (DEBUG/INFO/WARNING/ERROR/CRITICAL), a configured handler and — for pipelines — structured/JSON logs that your log platform can index and alert on. Emit machine-parseable fields (record id, attempt, latency), never a naked string.

python
import logging, json

class JsonFormatter(logging.Formatter):
    def format(self, r: logging.LogRecord) -> str:
        payload = {"level": r.levelname, "msg": r.getMessage(),
                   "logger": r.name}
        if r.exc_info:
            payload["error"] = self.formatException(r.exc_info)
        return json.dumps(payload)

h = logging.StreamHandler()
h.setFormatter(JsonFormatter())
logging.basicConfig(level=logging.INFO, handlers=[h])

try:
    extract(13)
except ExtractError:
    logging.getLogger("extract").exception("dead-letter")  # logs full chain

Wrap those two ideas together for a robust extract: retry transient failures with backoff, log every attempt as structured JSON, and after the last retry push the record to a dead-letter queue instead of crashing the whole run. Make the operation idempotent (safe to re-run — e.g. upsert by key, not blind insert) so a retry never double-writes.

11. Context managers & files

A context manager guarantees cleanup runs — even when the body raises — which is exactly what you want for file handles, DB transactions and temporary credentials.

The with protocol

Any object with __enter__ and __exit__ is a context manager. with obj as x: calls __enter__ (its return becomes x), runs the block, then calls __exit__ on the way out — normally or via an exception. __exit__ receives the exception info, so it can commit on success and roll back on failure. That "cleanup always happens" guarantee is why with open(...) beats a manual open()/close(): if the body raises between them, a manual close() never runs and you leak the handle.

Why with open(...) wins

It closes the file on normal exit and on exception, flushing buffered writes so you don't end up with a half-written, corrupt output file. Manual close() only runs if you reach it — one early raise or return and the handle leaks.

A DB transaction manager

The generator style with @contextmanager lets you write setup, yield, teardown as one function — everything before the yield is __enter__, everything after (in a finally) is __exit__. This is the cleanest way to express commit-on-success / rollback-on-exception.

python
from contextlib import contextmanager

@contextmanager
def transaction(conn):
    cur = conn.cursor()
    try:
        yield cur                 # hand the cursor to the with-block
        conn.commit()             # reached only if no exception
    except Exception:
        conn.rollback()           # body raised -> undo everything
        raise                     # re-raise so the caller still sees it
    finally:
        cur.close()              # always release the cursor

with transaction(conn) as cur:
    cur.execute("insert into staging values (%s)", (1,))
    # if this line raises, the insert above is rolled back

For several resources at once, with a() as x, b() as y: nests them (closed in reverse order). When the number of resources is dynamic — say, opening one file per partition — use contextlib.ExitStack to enter them in a loop and guarantee all are closed.

Files the modern way: pathlib

Skip string-concatenated paths. pathlib.Path gives you the / operator to join, .glob() to walk, and convenience readers/writers. It's OS-independent and far harder to get wrong than os.path.join spaghetti.

python
from pathlib import Path

root = Path("/data/landing")
for f in sorted(root.glob("orders/*.csv")):   # walk matching files
    if not f.exists():
        continue
    text = f.read_text(encoding="utf-8")     # open+read+close in one call
    print(f.name, len(text.splitlines()), "lines")

out = root / "processed" / "orders.done"       # / joins path parts
out.parent.mkdir(parents=True, exist_ok=True)
out.write_text("ok", encoding="utf-8")

12. JSON, CSV, datetime

Serialization is where quiet data corruption lives — mangled unicode, quoted commas split in half, and timezone-naive timestamps that drift your whole warehouse by hours.

JSON and the streaming standard: JSON Lines

json.loads/json.dumps convert between text and Python objects. Pass ensure_ascii=False to keep unicode readable (otherwise "Hà Nội" becomes \u... escapes), and a default= callable to serialize types json doesn't know (like datetime or Decimal). For pipelines, prefer JSON Lines / NDJSON — one JSON object per line — so you can stream and process a huge file record-by-record without loading it all, and append cheaply. A single giant JSON array forces you to parse the whole thing at once.

python
import json
from datetime import datetime, timezone
from decimal import Decimal

def encode(o):
    if isinstance(o, datetime): return o.isoformat()
    if isinstance(o, Decimal):  return str(o)   # keep precision as text
    raise TypeError(type(o))

rows = [{"city": "Hà Nội", "ts": datetime.now(timezone.utc),
         "amount": Decimal("19.99")}]

# write NDJSON: one object per line, stream-friendly
with open("out.ndjson", "w", encoding="utf-8") as fh:
    for r in rows:
        fh.write(json.dumps(r, default=encode, ensure_ascii=False) + "\n")

CSV: use the module, never split(',')

The csv module handles the ugly parts of the format: quoted fields, embedded commas and newlines, and configurable delimiter/quoting. DictReader yields each row as a dict keyed by header; DictWriter writes them back. Naive line.split(',') shreds a value like "Hanoi, Vietnam" into two columns — a classic pipeline bug that only shows up on the one row that had a comma.

python
import csv, io

raw = 'id,city\n1,"Hanoi, Vietnam"\n'       # note the comma inside quotes

print(raw.splitlines()[1].split(","))       # ['1', '"Hanoi', ' Vietnam"']  WRONG

for row in csv.DictReader(io.StringIO(raw)):
    print(row)                            # {'id': '1', 'city': 'Hanoi, Vietnam'}  right

datetime: store UTC, always

A naive datetime has no timezone — comparing or storing it is ambiguous and a frequent source of off-by-hours bugs. Make everything timezone-aware and store UTC everywhere in your pipeline; convert to local time only for display. Parse with strptime, format with strftime, prefer ISO-8601 on the wire, and convert to/from epoch seconds when a system needs a number.

python
from datetime import datetime, timezone

naive = datetime.now()               # no tz -> ambiguous, avoid
now = datetime.now(timezone.utc)     # aware, in UTC -> correct

iso = now.isoformat()               # '2026-07-16T03:00:00+00:00'
parsed = datetime.strptime("2026-07-16 03:00:00+0000",
                          "%Y-%m-%d %H:%M:%S%z")   # aware
epoch = int(now.timestamp())      # -> UTC epoch seconds
back = datetime.fromtimestamp(epoch, timezone.utc)

The naive-datetime trap & JSON float precision

Mixing naive and aware datetimes raises TypeError, and a naive value silently assumes the server's local zone — run the same job in two regions and your data disagrees. Rule: everything UTC, aware, from ingest to storage. Separately, JSON has no decimal type, so json.loads("0.1") gives a lossy binary float; for money use Decimal and serialize it as a string.

13. Talking to databases & APIs

The two ends of most pipelines — a database and an HTTP API — each have one non-negotiable rule: parameterise every query, and put a timeout on every request.

Databases: cursors and parameterised queries

Python's DB-API gives a uniform shape across drivers: a connection, a cursor, execute/executemany, then fetchall/fetchone. Always pass values as parameters, never format them into the SQL string — f-string SQL is how you get SQL injection and broken quoting. Use executemany for bulk inserts, manage the connection with with, and for real workloads use a SQLAlchemy engine so you get connection pooling instead of reconnecting per query.

Parameterise — or get injected

f"select * from t where id = {user_input}" lets an attacker pass 1; drop table t. Use execute(sql, params) and let the driver bind safely. The same habit also fixes quoting/escaping bugs on legitimate data (names with apostrophes, etc.).

APIs: sessions, timeouts, retries, pagination

Use a requests.Session to reuse the underlying TCP connection across calls (much faster over many pages). Always pass timeout= — a request without one can hang forever and freeze your whole pipeline. Mount an HTTPAdapter with a urllib3 Retry to auto-retry transient failures and honor 429 rate-limit backoff, then loop until pagination is exhausted, batching writes to the DB.

python
import requests
from requests.adapters import HTTPAdapter
from urllib3.util import Retry
import sqlalchemy as sa

def make_session() -> requests.Session:
    s = requests.Session()
    retry = Retry(total=5, backoff_factor=0.5,
                  status_forcelist=(429, 500, 502, 503),
                  respect_retry_after_header=True)
    s.mount("https://", HTTPAdapter(max_retries=retry))
    return s

def pull(session, base: str):
    page = 1
    while True:
        r = session.get(base, params={"page": page},
                        timeout=10)          # NEVER omit timeout
        r.raise_for_status()               # turn 4xx/5xx into an exception
        batch = r.json()["items"]
        if not batch:
            break                          # pagination exhausted
        yield batch
        page += 1

engine = sa.create_engine("postgresql://.../warehouse", pool_size=5)
ins = sa.text("insert into orders (id, amt) values (:id, :amt)")

with make_session() as sess, engine.begin() as conn:
    for batch in pull(sess, "https://api.example.com/orders"):
        conn.execute(ins, batch)          # parameterised bulk insert

This is the shape of a resilient connector: one pooled DB engine, one retrying HTTP session, a generator that pages until empty, and a single transactional bulk insert per batch so a mid-run failure rolls back cleanly rather than leaving half a page committed.

14. pandas & the vectorised mindset

pandas is the workhorse for single-machine tabular work — the trick to using it well is to stop thinking in row loops and start thinking in whole columns.

A DataFrame is a table: named columns, each of which is a Series (a typed, labelled array backed by NumPy or Arrow). The reason pandas is fast is that operations on a Series push down into compiled C/NumPy loops that run over the entire column at once. This is the vectorised mindset: express your transform as arithmetic and boolean logic over columns, and let the library iterate. The moment you write a Python-level for loop over rows, you throw that away and pay interpreter overhead per row.

The vectorised mindset vs row iteration

Compare computing a fee column three ways. All three produce the same numbers; only the first is acceptable in a pipeline.

python
import pandas as pd
import numpy as np

df = pd.DataFrame({
    "amount": [100.0, 250.0, 80.0, 1200.0],
    "tier":   ["gold", "silver", "silver", "gold"],
})

# GOOD — vectorised: whole-column arithmetic, runs in C
df["fee"] = df["amount"] * 0.02

# GOOD — vectorised conditional (np.where instead of a Python if)
df["fee"] = np.where(df["tier"] == "gold", df["amount"] * 0.01,
                                                df["amount"] * 0.02)

# SLOW — .apply(axis=1) calls a Python fn per row
df["fee"] = df.apply(lambda r: r["amount"] * 0.01, axis=1)

# SLOWEST — iterrows materialises a Series per row; avoid in pipelines
for i, r in df.iterrows():
    df.at[i, "fee"] = r["amount"] * 0.01

iterrows is a trap

.iterrows() and .apply(axis=1) can be 10–100x slower than the vectorised form and also lose dtypes (every row becomes an object Series). Treat them as a last resort. If you truly need per-row Python logic, prefer itertuples() over iterrows(), or push the work to NumPy.

Reading data: csv vs parquet

read_csv parses text and must guess dtypes; read_parquet reads a columnar binary format that already stores dtypes, compresses well, and lets you read only the columns you need. For DE work, parquet almost always wins on both speed and size — CSV is what you accept at the ingest edge, parquet is what you keep.

python
# Declare dtypes up front so read_csv doesn't guess (faster, less memory)
df = pd.read_csv(
    "events.csv",
    dtype={"user_id": "int32", "tier": "category"},
    parse_dates=["ts"],
    usecols=["user_id", "tier", "ts", "amount"],
)

df.to_parquet("events.parquet", compression="zstd")
# Later: read back only two columns, no dtype guessing needed
sub = pd.read_parquet("events.parquet", columns=["tier", "amount"])

dtypes & memory

Memory is the first thing that kills a pandas job. Two cheap wins: downcast numeric columns to the smallest type that fits, and convert low-cardinality strings to category (which stores each distinct value once plus small integer codes).

python
before = df.memory_usage(deep=True).sum()

df["amount"]  = pd.to_numeric(df["amount"], downcast="float")
df["user_id"] = pd.to_numeric(df["user_id"], downcast="integer")
df["tier"]    = df["tier"].astype("category")

after = df.memory_usage(deep=True).sum()
print(f"saved {(1 - after/before):.0%}")

Filtering with boolean masks and .loc

You select rows by building a boolean Series and indexing with it. Use .loc[mask, cols] for the combined row+column selection — it is explicit and it is the way to avoid the chained-assignment trap.

python
mask = (df["amount"] > 100) & (df["tier"] == "gold")
big_gold = df.loc[mask, ["user_id", "amount"]]

# Assign into a subset the RIGHT way — single .loc, no chaining

15. From pandas to PySpark

PySpark gives you the same select/filter/join/groupBy vocabulary as pandas — but distributed across a cluster and lazily evaluated, so an optimiser can rewrite your plan before anything runs.

A Spark DataFrame is not held in your process — it is a description of a computation over data partitioned across many executors. Because of that, PySpark splits every operation into two kinds:

  • Transformations (select, filter, withColumn, groupBy, join) are lazy: they build up a logical plan and return immediately without touching data.
  • Actions (count, collect, write, show) trigger execution: only now does Spark run the accumulated plan.

Laziness is the whole point. Because Spark sees the entire chain before executing, the Catalyst optimiser can push filters down to the scan, prune columns, and reorder joins — exactly the "you describe what, the engine decides how" contract you know from SQL. You are not writing an execution recipe; you are declaring intent.

read → filter → groupBy → agg → write

Here is a full mini-pipeline. Notice it is entirely Python, and that no data moves until write. Everything above write is just plan construction.

python
from pyspark.sql import SparkSession
from pyspark.sql import functions as F

spark = SparkSession.builder.appName("fees").getOrCreate()

result = (
    spark.read.parquet("s3://bucket/events/")   # transformation (lazy)
         .filter(F.col("amount") > 0)             # transformation
         .withColumn("fee", F.col("amount") * 0.02)  # transformation
         .groupBy("tier")                       # transformation
         .agg(F.sum("fee").alias("total_fee"),
              F.count("*").alias("n"))
)

result.write.mode("overwrite").parquet("s3://bucket/out/")  # ACTION

Why Python UDFs are slow

Spark's engine runs on the JVM. A plain Python UDF forces every row to cross the JVM↔Python boundary — serialize out, run your Python, serialize back — which is slow and opaque to Catalyst. Prefer built-in pyspark.sql.functions, which execute natively on the JVM. When you genuinely need Python logic, use a pandas (vectorised) UDF, which ships whole Arrow batches at once instead of row-by-row.

python
# BEST — native expression, no Python boundary at all
df = df.withColumn("fee", F.when(F.col("tier") == "gold",
                                    F.col("amount") * 0.01)
                             .otherwise(F.col("amount") * 0.02))

# If you MUST use Python, a pandas_udf batches rows via Arrow
from pyspark.sql.functions import pandas_udf
import pandas as pd

@pandas_udf("double")
def net(amount: pd.Series, fee: pd.Series) -> pd.Series:
    return amount - fee          # vectorised, whole batch

df = df.withColumn("net", net("amount", "fee"))

collect() can OOM the driver

collect() pulls every row from all executors into the single driver process — on a large DataFrame that is an instant out-of-memory crash. Use show(20) or take(n) to peek, write to persist, and only collect() when you already know the result is tiny. Always keep the actions-vs-transformations distinction in your head: it explains both when work happens and where the data lands.

When do you move here from pandas? When the data no longer fits one machine, or when you need fault-tolerant distributed I/O over a data lake. Polars and DuckDB are the single-machine middle ground; Spark is where you go when one machine is no longer enough.

16. Concurrency & the GIL

Python can wait on thousands of things at once but can only compute one thing at a time — understanding that split is the key to picking threads, processes, or asyncio correctly.

CPython has a Global Interpreter Lock (GIL): only one thread executes Python bytecode at any instant. The consequence is the single most misunderstood fact in Python performance — threads do not give you parallel CPU. But the GIL is released while a thread waits on the OS (network, disk), so threads absolutely help I/O-bound work: while one thread blocks on a socket, another runs.

WorkloadBottleneckUse
I/O-bound: API calls, downloads, DB querieswaiting on network/diskthreads or asyncio
CPU-bound: parsing, hashing, number crunchingPython bytecode / GILmultiprocessing
CPU-bound at scale in a pipelineone machine's corespush to Spark
Very high concurrency I/O (10k+ sockets)thread overheadasyncio

The three tools:

  • threading — shared memory, cheap, great for I/O-bound fan-out; no CPU parallelism.
  • multiprocessing — separate interpreters, each with its own GIL, so real CPU parallelism; you pay a serialization cost to move data between processes.
  • asyncio — one thread, one event loop, async/await cooperative scheduling; best when you have huge numbers of concurrent I/O operations and want to avoid thread overhead.

In practice you rarely touch these raw — concurrent.futures gives you ThreadPoolExecutor and ProcessPoolExecutor, one clean high-level API where you swap one word to switch between I/O and CPU strategies.

Parallel downloads with ThreadPoolExecutor

The classic DE task: hit an API for many keys, or download many files. It is I/O-bound, so threads win. map preserves input order; as_completed lets you handle results as they land.

python
from concurrent.futures import ThreadPoolExecutor, as_completed
import urllib.request

urls = [f"https://api.example.com/v1/items/{i}" for i in range(50)]

def fetch(url: str) -> bytes:
    with urllib.request.urlopen(url, timeout=10) as r:
        return r.read()

results = {}
with ThreadPoolExecutor(max_workers=8) as pool:
    futures = {pool.submit(fetch, u): u for u in urls}
    for fut in as_completed(futures):
        url = futures[fut]
        try:
            results[url] = fut.result()
        except Exception as e:
            print(f"failed {url}: {e}")   # one bad call won't sink the batch

print(f"got {len(results)} of {len(urls)}")

To convert this to CPU-bound work — say, parsing each downloaded blob — you would change ThreadPoolExecutor to ProcessPoolExecutor and nothing else, because separate processes each get their own GIL and run on separate cores.

The "threads make Python faster" myth

Adding threads to a CPU-bound pure-Python loop makes it the same speed or slower — the GIL serialises the bytecode and you just added context-switching overhead. Threads help only when the work spends its time waiting. For heavy compute, use processes, native/NumPy code that releases the GIL, or — in a real pipeline — push the work to Spark instead of fighting the GIL on one box.

18. Big-O intuition

You don't need proofs — you need to read a solution and instantly know its order of growth, because that's the number the interviewer is checking against the input size.

Big-O answers one question: as the input n gets big, how does the work grow? Ignore constants, keep the dominant term. A senior engineer reads a loop the way you read a road sign — no arithmetic, just pattern recognition. Here are the five you must recognise on sight.

The five orders, with everyday analogies

O(1) — constant. The work never changes with n. Looking a word up in a Python dict is O(1): the hash points straight at the slot. Grabbing nums[i] from a list is O(1). It's like knowing someone's phone number by heart — the size of the phone book doesn't matter.

O(log n) — logarithmic. Each step throws away half the remaining work. Binary search in a sorted array is the canonical case: 1 million items, ~20 comparisons. Like finding a name in a physical phone book by repeatedly splitting it in half.

O(n) — linear. One pass, touch each element a constant number of times. Summing a list, or scanning a log file line by line. Reading every page of a book once.

O(n log n) — linearithmic. The cost of a good sort (Python's Timsort), and of most "sort then sweep" solutions. Think: split the deck, sort halves, merge. If a Medium problem feels like it needs order, this is usually the target.

O(n²) — quadratic. A loop inside a loop over the same data. Comparing every pair of records. Fine for n = 100, a disaster for n = 100000 — that's 10 billion operations. This is the smell you learn to hunt down.

python
# Same problem, two orders. The interviewer wants the second one.
def has_pair_slow(nums, target):        # O(n^2) — nested loop
    for i in range(len(nums)):
        for j in range(i + 1, len(nums)):
            if nums[i] + nums[j] == target:
                return True
    return False

def has_pair_fast(nums, target):        # O(n) — one pass + a set
    seen = set()
    for x in nums:
        if target - x in seen:        # membership is O(1)
            return True
        seen.add(x)
    return False

Space complexity, briefly

Same idea for memory: how much extra space does your solution allocate as n grows? A hashmap of every element is O(n) space; two pointers walking an array in place is O(1) space; recursion adds O(depth) stack frames. In DE terms this is the difference between streaming a file line by line (O(1)) and loading it into a list (O(n)) — the interviewer often cares about space precisely because production data doesn't fit in RAM.

The practical rule

If n is large and you just wrote a loop nested inside another loop over the same data, stop. There is almost always a dict/set that turns it into one pass, or a sort that lets you sweep once. Nested-loop-over-big-n is the #1 reason a working solution still fails the round.

19. Arrays & two pointers

The Python list is a dynamic array, and the two-pointer technique is how you squeeze an O(n²) array problem down to O(n) with O(1) extra space.

A list stores elements contiguously and over-allocates so that append is amortised O(1). Indexing is O(1); inserting or deleting at the front is O(n) because everything shifts. Slicing a[i:j] copies — great for safety, but O(k) in time and space, so don't slice inside a hot loop when an index would do.

python
a = [10, 20, 30, 40]
b = a[1:3]              # COPY -> [20, 30]; a is untouched
a[1:3] = [99]         # in-place splice -> [10, 99, 40]
a.reverse()             # in-place, O(n), returns None
c = a[::-1]             # reversed COPY

The two-pointer technique

Two indices walk the array instead of one, so a job that looks like "check every pair" becomes a single sweep. Two flavours dominate interviews:

Opposite-ends — one pointer at the start, one at the end, moving toward each other. Requires the array be sorted (or that you're matching from both sides). Shows up in sorted two-sum, container-with-most-water, valid palindrome. Fast/slow — two pointers moving at different speeds through the same array, used for in-place rewriting (move zeros, dedup) and cycle detection.

Worked: Two Sum II — sorted input

Problem: given a sorted array and a target, return the 1-based indices of the two numbers that add up to it.

python
def two_sum_sorted(nums: list[int], target: int) -> list[int]:
    lo, hi = 0, len(nums) - 1
    while lo < hi:
        s = nums[lo] + nums[hi]
        if s == target:
            return [lo + 1, hi + 1]
        if s < target:
            lo += 1          # need a bigger sum
        else:
            hi -= 1          # need a smaller sum
    return []

assert two_sum_sorted([2, 7, 11, 15], 9) == [1, 2]

Time O(n), space O(1). The sortedness is what lets a single number tell you which pointer to move.

Worked: Container With Most Water

Problem: heights h[i] are vertical lines; pick two to hold the most water (width × min height).

python
def max_area(h: list[int]) -> int:
    lo, hi = 0, len(h) - 1
    best = 0
    while lo < hi:
        area = (hi - lo) * min(h[lo], h[hi])
        best = max(best, area)
        if h[lo] < h[hi]:      # the short wall caps us; move it in
            lo += 1
        else:
            hi -= 1
    return best

assert max_area([1, 8, 6, 2, 5, 4, 8, 3, 7]) == 49

Time O(n), space O(1). The insight: moving the taller wall can never help (width shrinks, height still capped by the shorter wall), so always move the shorter one — which is why one pass suffices.

In-place trio worth memorising

Reverse: swap a[lo], a[hi] and step both inward. Move zeros: a slow pointer marks the next non-zero slot while a fast pointer scans. Dedup sorted array: slow pointer writes, fast pointer reads; write only when a[fast] != a[slow]. All O(n) time, O(1) space.

Worked: Two Sum (the complement trick)

Problem: return indices of the two numbers that sum to target (unsorted input).

python
def two_sum(nums: list[int], target: int) -> list[int]:
    seen = {}                        # value -> index
    for i, x in enumerate(nums):
        if target - x in seen:       # have we seen the complement?
            return [seen[target - x], i]
        seen[x] = i
    return []

assert two_sum([3, 2, 4], 6) == [1, 2]

Time O(n), space O(n). One pass; the dict replaces the inner loop of the O(n²) brute force. This is the template — the same "remember what you've seen" idea powers dozens of variants.

Worked: Group Anagrams

Problem: group words that are anagrams of one another.

python
from collections import defaultdict

def group_anagrams(words: list[str]) -> list[list[str]]:
    groups = defaultdict(list)
    for w in words:
        key = tuple(sorted(w))    # canonical form; tuple is hashable
        groups[key].append(w)
    return list(groups.values())

group_anagrams(["eat", "tea", "tan", "ate", "nat"])
# [['eat','tea','ate'], ['tan','nat']]

Time O(n · k log k) (k = word length, from sorting each word), space O(n · k). The trick is choosing a canonical key that every anagram maps to.

Worked: Subarray Sum Equals K

Problem: count contiguous subarrays whose elements sum to k.

python
from collections import defaultdict

def subarray_sum(nums: list[int], k: int) -> int:
    count = 0
    prefix = 0
    seen = defaultdict(int)         # prefix-sum -> how many times seen
    seen[0] = 1                    # empty prefix
    for x in nums:
        prefix += x
        count += seen[prefix - k]    # subarrays ending here that sum to k
        seen[prefix] += 1
    return count

assert subarray_sum([1, 1, 1], 2) == 2

Time O(n), space O(n). A running prefix sum plus a count-map: if prefix - k was seen before, every earlier occurrence closes a valid subarray. This "prefix sum + hashmap" combo is a genuine step up in difficulty and a favourite for the harder Medium slot.

Keys must be hashable

A list can't be a dict/set key — it's mutable, so TypeError: unhashable type: 'list'. Convert to a tuple first (that's why Group Anagrams uses tuple(sorted(w))). str, int, frozenset and tuples of hashables are fine.

22. Sorting, searching & binary search

Sorting is O(n log n) and often the cheapest way to impose order that unlocks a linear sweep; binary search is O(log n) and the interviewer's favourite way to check whether you can write a loop without an off-by-one.

Sorting in Python

sorted(x) returns a new list; x.sort() mutates in place and returns None. Both use Timsort — O(n log n), and stable (equal elements keep their original order), which lets you sort by multiple keys in passes. Use key= to sort by a computed field and reverse=True for descending.

# Top-K by sorting — fine, but a heap (see §27) is better at O(n log k)
top3 = sorted(counts.items(), key=lambda kv: -kv[1])[:3]

Worked: the binary-search template

Problem: find target in a sorted array, return its index or -1. Memorise this exact skeleton — the <= and the mid ± 1 updates are what keep it from looping forever or missing the last element.

python
def binary_search(nums: list[int], target: int) -> int:
    lo, hi = 0, len(nums) - 1      # inclusive bounds
    while lo <= hi:                     # note: <=
        mid = (lo + hi) // 2
        if nums[mid] == target:
            return mid
        if nums[mid] < target:
            lo = mid + 1
        else:
            hi = mid - 1
    return -1

Time O(log n), space O(1). For most real code, reach for the standard library instead:

python
from bisect import bisect_left, bisect_right

The deeper skill interviewers probe is binary search on the answer: when the answer is a number in a range and "is X feasible?" is monotonic (false, false, …, true, true), binary-search the answer space instead of the array. Same template, the "array" is just the candidate answers.

Worked: Search in Rotated Sorted Array

Problem: a sorted array was rotated at an unknown pivot; find target in O(log n).

python
def search_rotated(nums: list[int], target: int) -> int:
    lo, hi = 0, len(nums) - 1
    while lo <= hi:
        mid = (lo + hi) // 2
        if nums[mid] == target:
            return mid
        if nums[lo] <= nums[mid]:                # left half is sorted
            if nums[lo] <= target < nums[mid]:
                hi = mid - 1
            else:
                lo = mid + 1
        else:                                   # right half is sorted
            if nums[mid] < target <= nums[hi]:
                lo = mid + 1
            else:
                hi = mid - 1
    return -1

assert search_rotated([4, 5, 6, 7, 0, 1, 2], 0) == 4

Time O(log n), space O(1). Each step, one half is guaranteed sorted; decide whether the target lies inside that sorted half and discard the other. This is the model "modified binary search" the round is testing for.

Worked: Merge Intervals (sort then sweep)

Problem: merge all overlapping intervals.

python
def merge(intervals: list[list[int]]) -> list[list[int]]:
    intervals.sort(key=lambda iv: iv[0])   # sort by start
    out = [intervals[0]]
    for start, end in intervals[1:]:
        if start <= out[-1][1]:            # overlaps the last kept interval
            out[-1][1] = max(out[-1][1], end)
        else:
            out.append([start, end])
    return out

merge([[1, 3], [2, 6], [8, 10], [15, 18]])
# [[1, 6], [8, 10], [15, 18]]

Time O(n log n) (dominated by the sort), space O(n) for the output. Sorting first is what turns "compare every pair of intervals" into one linear sweep — the recurring "impose order, then walk once" move.

23. Stacks & queues

A stack is last-in-first-out (undo history); a queue is first-in-first-out (a task backlog) — and picking the right container is the difference between O(1) and O(n) per operation.

A stack is just a Python list: append to push, pop to pop off the end — both O(1). A queue needs efficient removal from the front, so use collections.deque: append on the right, popleft on the left, both O(1) at either end.

python
from collections import deque

stack = []
stack.append(1); stack.append(2)   # push
stack.pop()                          # 2  (LIFO), O(1)

q = deque()
q.append("job1")                   # enqueue at right, O(1)
q.popleft()                       # 'job1' (FIFO), O(1)

Never use list.pop(0) for a queue

list.pop(0) and list.insert(0, x) shift every remaining element — O(n) each, so a "queue" built on a list is secretly O(n²). This is the classic performance bug in BFS solutions. Use deque and it's O(1). Same reason BFS over a graph or level-order tree traversal always reaches for deque.

Worked: Valid Parentheses

Problem: is a string of brackets ()[]{} balanced and correctly nested?

python
def is_valid(s: str) -> bool:
    pairs = {")": "(", "]": "[", "}": "{"}
    stack = []
    for ch in s:
        if ch in pairs:                      # a closing bracket
            if not stack or stack.pop() != pairs[ch]:
                return False
        else:                                # an opening bracket
            stack.append(ch)
    return not stack                       # must be empty at the end

assert is_valid("()[]{}") is True
assert is_valid("(]") is False

Time O(n), space O(n). The stack remembers the most recent unmatched opener — exactly what LIFO is for.

The monotonic-stack pattern

A monotonic stack keeps its contents sorted (increasing or decreasing) by popping anything that would break the order as you push. It answers "next greater / next smaller element" questions in O(n) instead of O(n²) — you touch each element at most twice (one push, one pop).

Worked: Daily Temperatures (next greater element)

Problem: for each day, how many days until a warmer temperature? (0 if none.)

python
def daily_temperatures(temps: list[int]) -> list[int]:
    ans = [0] * len(temps)
    stack = []                          # indices, temps decreasing
    for i, t in enumerate(temps):
        while stack and temps[stack[-1]] < t:   # t is warmer than stack top
            j = stack.pop()
            ans[j] = i - j              # days waited for day j
        stack.append(i)
    return ans

daily_temperatures([73, 74, 75, 71, 69, 72, 76, 73])
# [1, 1, 4, 2, 1, 1, 0, 0]

Time O(n), space O(n). The stack holds days still waiting for a warmer one; when a warmer day arrives it resolves all of them at once. The same skeleton solves "Next Greater Element" — just store values instead of gaps.

Recognising a monotonic-stack problem

Phrases like "next greater", "next warmer", "nearest smaller to the left", "largest rectangle in histogram", or "how far until…" are the tell. If you're about to write a nested loop scanning forward for the next bigger thing, a monotonic stack collapses it to O(n).

24. Linked lists

You will almost never touch a linked list in production data code, but interviewers love them because they force you to reason about pointers — so learn the three tricks that solve 90% of the questions.

A linked list is a chain of nodes where each node holds a value and a reference to the next node. There is no contiguous block of memory and no index: to reach the 5th element you follow four pointers. The whole model is one tiny class.

python
# The one data structure the whole section is built on.
class ListNode:
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next          # singly linked: one pointer forward

A singly linked list has one pointer per node (next). A doubly linked list adds a prev pointer, which buys O(1) deletion when you already hold the node — that is exactly why Python's own collections.OrderedDict and LRU caches use one under the hood. In DE data itself they are rare: columnar files, Arrow buffers and DataFrames are all contiguous arrays because scanning contiguous memory is far faster. Linked lists show up in interviews, not in your Parquet.

The dummy-head trick

Half of linked-list bugs are special-casing the head. Allocate a throwaway dummy node in front, build off dummy.next, and return dummy.next at the end. Now "insert at head" and "insert in the middle" are the same code path.

Reverse a linked list — iterative and recursive

Problem: given the head of a singly linked list, return the head of the reversed list. Walk the list carrying three pointers — prev, cur, and a saved nxt — and flip each link as you pass it.

python
def reverse_list(head: ListNode | None) -> ListNode | None:
    prev = None
    cur = head
    while cur:
        nxt = cur.next        # 1. save the rest of the list
        cur.next = prev       # 2. flip this node's pointer backward
        prev = cur            # 3. advance prev
        cur = nxt             # 4. advance cur
    return prev              # prev is the new head

# Recursive version: reverse the tail, then hook head onto the end.
def reverse_rec(head: ListNode | None) -> ListNode | None:
    if head is None or head.next is None:
        return head                 # base case: empty or single node
    new_head = reverse_rec(head.next)
    head.next.next = head          # make the next node point back to us
    head.next = None              # and drop our old forward link
    return new_head

Merge two sorted lists — the dummy head in action

Problem: given the heads of two sorted linked lists, splice them into one sorted list. This is the linked-list twin of the merge step in merge sort, and a real echo of merging sorted runs when you spill-and-merge on disk.

python
def merge_two(a: ListNode | None, b: ListNode | None) -> ListNode | None:
    dummy = ListNode()      # throwaway head — no special case for the first node
    tail = dummy
    while a and b:
        if a.val <= b.val:
            tail.next = a
            a = a.next
        else:
            tail.next = b
            b = b.next
        tail = tail.next
    tail.next = a or b       # attach whatever list still has nodes
    return dummy.next

Fast/slow pointers — cycle detection & the middle

Two pointers, one moving twice as fast as the other, is Floyd's tortoise-and-hare. If the list has a cycle the fast pointer eventually laps the slow one and they collide; if it terminates cleanly, fast runs off the end. The same trick parked at the end finds the middle node in one pass — no length pre-count.

Problem: return True if the linked list contains a cycle.

python
def has_cycle(head: ListNode | None) -> bool:
    slow = fast = head
    while fast and fast.next:
        slow = slow.next          # one step
        fast = fast.next.next     # two steps
        if slow is fast:          # they met → there is a loop
            return True
    return False                 # fast reached the end → no loop

def middle(head: ListNode | None) -> ListNode | None:
    slow = fast = head
    while fast and fast.next:
        slow = slow.next
        fast = fast.next.next
    return slow                  # when fast hits the end, slow sits at the middle

If you can reverse a list, merge two lists, and detect a cycle from memory, you can improvise almost every Medium linked-list question. Reach for the dummy head whenever the answer's head might change, and fast/slow whenever the question mentions "middle", "cycle", or "nth from the end".

25. Recursion, trees & BFS/DFS

Trees are recursion made visible: almost every tree question is "do something at this node, then recurse into the children" — and the two ways to walk a tree, DFS and BFS, are the traversals interviewers expect you to write cold.

Recursion works because each call gets its own frame on the call stack: local variables, the return address, the lot. A recursive function needs a base case (when to stop) and a recursive case (a smaller version of the same problem). Miss the base case and you blow the stack.

python
import sys
# CPython caps recursion around 1000 frames to protect the C stack.
sys.setrecursionlimit(10_000)   # raise it — but see the warning below

Deep recursion is a footgun

Raising the limit lets you recurse deeper, but each frame still costs real C-stack memory and you can segfault the interpreter. For a linked list or a skewed tree of a million nodes, convert the recursion to iteration with an explicit stack. Python has no tail-call optimisation, so recursion depth is a hard ceiling, not a suggestion.

A binary tree node mirrors the list node, with two child pointers instead of one:

python
class TreeNode:
    def __init__(self, val=0, left=None, right=None):
        self.val = val
        self.left = left
        self.right = right

DFS — go deep first

Depth-first search dives to the bottom of one branch before backing up. The three orders differ only in when you visit the node relative to its children: pre-order (node, left, right), in-order (left, node, right — sorted output for a BST), post-order (left, right, node). Recursion is the natural expression; an explicit stack is the iterative form interviewers sometimes ask for.

python
# Recursive in-order (gives sorted values for a BST).
def inorder(root: TreeNode | None) -> list[int]:
    if root is None:
        return []
    return inorder(root.left) + [root.val] + inorder(root.right)

# Iterative pre-order with an explicit stack — no recursion limit to hit.
def preorder(root: TreeNode | None) -> list[int]:
    if root is None:
        return []
    out, stack = [], [root]
    while stack:
        node = stack.pop()
        out.append(node.val)
        if node.right:            # push right first so left pops first
            stack.append(node.right)
        if node.left:
            stack.append(node.left)
    return out

Problem: return the maximum depth (root-to-deepest-leaf node count) of a binary tree. This is DFS distilled to one line of recurrence.

python
def max_depth(root: TreeNode | None) -> int:
    if root is None:
        return 0                                  # base case: empty subtree has depth 0
    return 1 + max(max_depth(root.left), max_depth(root.right))

BFS — level by level with a deque

Breadth-first search visits every node at depth d before any node at depth d+1. You need a FIFO queue — use collections.deque, whose popleft() is O(1) (a plain list's pop(0) is O(n)). The trick for "give me each level as its own list" is to snapshot len(q) at the top of each loop: that count is exactly one level.

Problem: return the node values grouped level by level, top to bottom.

python
from collections import deque

def level_order(root: TreeNode | None) -> list[list[int]]:
    if root is None:
        return []
    out, q = [], deque([root])
    while q:
        level = []
        for _ in range(len(q)):     # snapshot: exactly one level's worth
            node = q.popleft()
            level.append(node.val)
            if node.left:
                q.append(node.left)
            if node.right:
                q.append(node.right)
        out.append(level)
    return out

Validate a BST — DFS carrying bounds

Problem: decide whether a binary tree is a valid binary search tree. The classic trap is comparing a node only to its immediate children — that misses violations deeper down. The fix: pass down the open interval (low, high) each node must fall inside, tightening it as you descend.

python
def is_valid_bst(root: TreeNode | None) -> bool:
    def check(node, low, high):
        if node is None:
            return True                      # empty subtree is valid
        if not (low < node.val < high):    # must sit strictly inside the interval
            return False
        return (check(node.left,  low, node.val) and
                check(node.right, node.val, high))
    return check(root, float('-inf'), float('inf'))

Every one of these traversals is O(n) time — each node is visited once. Space is O(h) for DFS (the recursion/stack depth, h = tree height) and O(w) for BFS (the widest level). These are the easy-to-medium tree problems interviewers actually reach for; know max-depth, level-order and validate-BST cold and you have covered the bulk of them.

26. Sliding window & prefix sums

When a problem asks for the best contiguous run — longest, shortest, max-sum subarray or substring — a sliding window turns the naive O(n²) scan of every subarray into one clean O(n) pass.

A window is a pair of indices left and right bounding a contiguous slice. Two flavours: a fixed-size window (length k is given — slide it one step at a time) and a variable-size window (grow the right edge greedily, and shrink the left edge only when a constraint breaks).

Fixed window — max sum of size K

Problem: find the maximum sum of any contiguous subarray of length k. Compute the first window's sum once, then slide: add the entering element, subtract the leaving one. No re-summing.

python
def max_sum_k(nums: list[int], k: int) -> int:
    window = sum(nums[:k])          # first window, O(k)
    best = window
    for i in range(k, len(nums)):
        window += nums[i] - nums[i - k]   # slide: +entering, -leaving
        best = max(best, window)
    return best

Variable window — the expand/contract template

This is the pattern worth memorising. March right across the array to expand; whenever the window violates the constraint, advance left to contract until it is valid again. Each index enters and leaves the window at most once, so it is O(n) despite the nested loop.

Problem: length of the longest substring without repeating characters. A "last seen index" dict lets left jump straight past the previous occurrence instead of crawling.

python
def length_of_longest(s: str) -> int:
    last: dict[str, int] = {}
    left = best = 0
    for right, ch in enumerate(s):
        if ch in last and last[ch] >= left:
            left = last[ch] + 1          # jump past the duplicate
        last[ch] = right
        best = max(best, right - left + 1)
    return best

Prefix sums — O(1) range queries after an O(n) build

A prefix-sum array pre where pre[i] = sum of the first i elements lets you answer "sum of the slice [i, j)" as pre[j] - pre[i] in constant time. The same idea powers cumulative aggregates in SQL window functions and pandas cumsum.

Problem: count the number of contiguous subarrays whose elements sum to exactly k. Combine a running prefix sum with a hashmap of how many times each prefix value has been seen — a single pass, no window needed.

python
from collections import defaultdict

def subarray_sum(nums: list[int], k: int) -> int:
    seen = defaultdict(int)
    seen[0] = 1                 # empty prefix sums to 0, seen once
    running = count = 0
    for x in nums:
        running += x
        count += seen[running - k]   # a prior prefix that makes the gap == k
        seen[running] += 1
    return count

Good to know for stronger DE roles

Sliding window and prefix sums are not everyday DE tools, but they read as fluency in one-pass streaming thinking — exactly the mindset for processing an unbounded event stream where you cannot re-scan. Worth having in the back pocket for the stronger interview loops.

27. Heaps & priority queues

When the question is "the top K", "the k closest", or "merge k sorted streams", a heap gives you the extreme element in O(log n) without keeping everything sorted — and Python ships one in the standard library.

A binary heap keeps the smallest (or largest) element instantly reachable while everything else stays only loosely ordered. Python's heapq operates on a plain list and is always a min-heap: the smallest element sits at index 0.

There is no max-heap in Python

heapq is min-only. For max-heap behaviour, push the negated value (heappush(h, -x)) and negate on the way out, or push a tuple with a negated key like (-count, item) so the largest count pops first. Tuples compare lexicographically, so put your sort key first.

Kth largest — a bounded min-heap of size k

Problem: return the k-th largest element in an array. Keep a min-heap holding only the k biggest seen so far; its root is the smallest of those k, which is exactly the k-th largest overall. Popping whenever the heap exceeds k caps memory at O(k) — the streaming-friendly move when the array does not fit in memory.

python
import heapq

def find_kth_largest(nums: list[int], k: int) -> int:
    heap: list[int] = []
    for x in nums:
        heapq.heappush(heap, x)
        if len(heap) > k:          # keep only the k largest
            heapq.heappop(heap)     # drop the smallest of them
    return heap[0]                # root = smallest of the top k = kth largest

Top K frequent — Counter + heap

Problem: return the k most frequent elements. Count with Counter, then let nlargest pull the k highest counts — it runs a size-k heap internally, so it is O(n log k), cheaper than a full sort when k is small.

python
import heapq
from collections import Counter

def top_k_frequent(nums: list[int], k: int) -> list[int]:
    counts = Counter(nums)
    # nlargest by count; each item is a (value, count) pair
    top = heapq.nlargest(k, counts.items(), key=lambda kv: kv[1])
    return [value for value, _count in top]

The other heap classics follow the same shape: k closest points (push (-dist, point) onto a size-k heap), merge k sorted lists (push the head of each list, pop the min, push its successor), and a streaming running median — a max-heap of the low half plus a min-heap of the high half, rebalanced each insert so the median is always one or two peeks away. That two-heap trick is the go-to for "latest N events" and online percentile dashboards.

Good to know for stronger / infra roles

Heaps underpin real infra: priority scheduling, k-way merges when a sort spills to disk, rate-limiter and expiry timers. If you are aiming at platform/infra-flavoured DE roles, being fluent with heapq tuples and the two-heap median pays off.

28. Basic 1D dynamic programming

You do not need to be a DP wizard — you need enough not to freeze when a simple one shows up, and to recognise the "DP smell" fast enough to reach for a recurrence instead of brute force.

Dynamic programming applies when a problem has two properties: overlapping subproblems (the same smaller computation recurs many times) and optimal substructure (the best answer is built from the best answers to subproblems). The method: write the recurrence first — "the answer at step n in terms of earlier steps" — then either memoise the recursion or fill a table bottom-up.

Climbing Stairs — the Fibonacci in disguise

Problem: you can climb 1 or 2 steps at a time; how many distinct ways to reach step n? Ways to reach step n = ways to reach n-1 (then one step) + ways to reach n-2 (then two). That recurrence is Fibonacci, so two rolling variables suffice — O(n) time, O(1) space.

python
def climb_stairs(n: int) -> int:
    a, b = 1, 1              # ways to reach step 0 and step 1
    for _ in range(n):
        a, b = b, a + b        # roll forward one step
    return a

House Robber — max over a choice with reuse

Problem: maximum money robbable from houses in a row, without robbing two adjacent houses. At each house you choose: skip it (keep the previous best) or rob it (its value plus the best up to two houses back). Take the max.

python
def rob(nums: list[int]) -> int:
    prev, cur = 0, 0          # best up to i-2 and i-1
    for x in nums:
        prev, cur = cur, max(cur, prev + x)   # skip vs rob-this-house
    return cur

Coin Change — memoization and tabulation side by side

Problem: fewest coins that sum to a target amount, or -1 if impossible. The recurrence: best(a) = 1 + min(best(a - c)) over every coin c. Here is the top-down memoised form and the bottom-up 1D table — same recurrence, two encodings.

python
from functools import lru_cache

# Top-down: recursion + automatic cache.
def coin_change_memo(coins: list[int], amount: int) -> int:
    @lru_cache(maxsize=None)
    def best(a: int) -> float:
        if a == 0:
            return 0
        if a < 0:
            return float('inf')
        return min((1 + best(a - c) for c in coins), default=float('inf'))
    ans = best(amount)
    return ans if ans != float('inf') else -1

# Bottom-up: fill a 1D table, dp[a] = fewest coins for amount a.
def coin_change_tab(coins: list[int], amount: int) -> int:
    INF = amount + 1                 # sentinel bigger than any real answer
    dp = [0] + [INF] * amount
    for a in range(1, amount + 1):
        for c in coins:
            if c <= a:
                dp[a] = min(dp[a], dp[a - c] + 1)
    return dp[amount] if dp[amount] != INF else -1

@lru_cache is the fastest way to turn any recursive solution into a memoised one — decorate, and repeated calls with the same arguments return instantly. Tabulation trades that convenience for tighter control and no recursion depth. Both are O(amount × len(coins)) time.

The DP smell

Reach for DP when a problem says count the number of ways, or asks for a min/max over a sequence of choices, and those choices reuse overlapping subresults. "How many ways to…", "fewest/most … to reach…", "longest/shortest subsequence…" are the tell. If a greedy pick can be undone by a later choice, greedy fails and DP is your tool.

29. Graphs & topological sort

Graphs are the one DSA topic that maps directly onto your day job — every DAG in Airflow or Dagster is a graph, and the order tasks run in is a topological sort, cycle detection and all.

Represent a graph as an adjacency list: a dict mapping each node to the list of nodes it points to. It is compact for the sparse graphs DE deals with and gives O(1) access to a node's neighbours.

python
# A tiny directed graph as dict[node, list[node]].
graph: dict[str, list[str]] = {
    "extract":  ["clean", "join"],
    "clean":    ["join"],
    "join":     ["load"],
    "load":     [],
}

You traverse a graph exactly like a tree — BFS with a queue, DFS with a stack or recursion — with one addition: a seen set, because unlike a tree a graph can revisit nodes and loop forever without it.

BFS shortest path on an unweighted graph

Problem: fewest edges (hops) from a start node to a goal node. On an unweighted graph BFS is the shortest-path algorithm: because it expands by distance, the first time you reach the goal you are on a shortest route. (For weighted edges you would reach for Dijkstra, which swaps the plain queue for a min-heap keyed by cumulative distance.)

python
from collections import deque

def shortest_path(graph: dict[str, list[str]], start: str, goal: str) -> int:
    q = deque([(start, 0)])       # (node, hops so far)
    seen = {start}
    while q:
        node, dist = q.popleft()
        if node == goal:
            return dist               # first arrival = shortest
        for nb in graph[node]:
            if nb not in seen:
                seen.add(nb)
                q.append((nb, dist + 1))
    return -1                        # goal unreachable

Topological sort — exactly how an orchestrator orders a DAG

A topological sort produces a linear order where every task appears before everything that depends on it. This is precisely what Airflow and Dagster do to schedule a pipeline: they topologically sort the task DAG and run each task only once its upstreams are done. Kahn's algorithm makes it concrete — count each node's in-degree (number of dependencies), start with the zero-dependency nodes, and peel them off one at a time.

Problem: given a pipeline DAG, return a valid task order — and detect a dependency cycle.

python
from collections import deque

def topo_sort(graph: dict[str, list[str]]) -> list[str]:
    indeg = {u: 0 for u in graph}
    for u in graph:                # count dependencies per node
        for v in graph[u]:
            indeg[v] += 1

    q = deque(u for u in graph if indeg[u] == 0)   # no upstreams → ready
    order = []
    while q:
        u = q.popleft()
        order.append(u)
        for v in graph[u]:          # u is done → relax its downstreams
            indeg[v] -= 1
            if indeg[v] == 0:
                q.append(v)

    if len(order) != len(graph):   # some node never hit in-degree 0
        raise ValueError("cycle detected — this is not a valid DAG")
    return order

# >>> topo_sort(graph)  ->  ['extract', 'clean', 'join', 'load']

A cycle means there is no valid order

If Kahn's algorithm finishes with fewer nodes emitted than the graph holds, some nodes never reached in-degree zero — they depend on each other in a loop. That is a cycle, and a cyclic dependency graph is not a DAG: there is no order in which every task runs after its upstreams. This is exactly the error an orchestrator throws when you accidentally wire task A → B → A.

Stream a big file with a generator

python
def stream(path: str):
    with open(path) as f:          # constant memory: one line at a time
        for line in f:
            yield line.rstrip("\n")

Retry decorator skeleton

python
import time, functools

def retry(tries=3, delay=1.0):
    def deco(fn):
        @functools.wraps(fn)
        def wrap(*a, **kw):
            for i in range(tries):
                try:
                    return fn(*a, **kw)
                except Exception:
                    if i == tries - 1: raise
                    time.sleep(delay * 2 ** i)   # exponential backoff
        return wrap
    return deco

Context-managed transaction

python
from contextlib import contextmanager

@contextmanager
def transaction(conn):
    cur = conn.cursor()
    try:
        yield cur
        conn.commit()          # commit on clean exit
    except Exception:
        conn.rollback()        # roll back on any error
        raise
    finally:
        cur.close()

＊. Glossary

The core vocabulary of this guide, defined crisply — skim it to check you can explain each term in a sentence.

GIL (Global Interpreter Lock)
A CPython lock that lets only one thread execute Python bytecode at a time, so threads help I/O-bound work but not CPU-bound work — for parallel compute use processes or push down to an engine.
Generator
A function using yield that produces values lazily, one at a time, in constant memory — the backbone of streaming a file too big for RAM.
Iterator
Any object with __next__ that yields items until StopIteration; a for loop is sugar over calling next() on one.
Lazy evaluation
Computing a value only when it is actually needed rather than up front — how generators, itertools, and Spark transformations avoid materialising whole datasets.
Comprehension
Concise syntax to build a list, set, or dict from an iterable, e.g. [x*2 for x in xs if x]; a generator expression uses parentheses and stays lazy.
Decorator
A callable that wraps a function to add behaviour (retry, timing, caching) without editing it, applied with @name above the def.
Closure
An inner function that captures variables from the enclosing scope and keeps them alive after the outer function returns — the mechanism decorators are built on.
Context manager
An object usable in a with block that guarantees setup and teardown (open/close, commit/rollback) even when an exception fires; built via __enter__/__exit__ or @contextmanager.
Dataclass
A class decorated with @dataclass that auto-generates __init__, __repr__, and equality from typed fields — ideal for config and records.
Type hint
An annotation such as x: int or -> list[str] that documents intent and lets tools like mypy catch bugs; not enforced at runtime.
Protocol
A typing construct for structural (duck) typing: any object with the right methods satisfies it, no inheritance required.
Hashmap
A key-value structure with average O(1) lookup, insert, and delete; Python's dict is one. Underlies counting, grouping, and joins.
Hashable
An object with a stable __hash__ (typically immutable — str, int, tuple) so it can be a dict key or set member; a list is not.
Set
An unordered collection of unique hashable items with O(1) membership and fast union/intersection — the go-to for dedup and "have I seen this?".
Deque
A double-ended queue (collections.deque) with O(1) append and pop at both ends — the right FIFO queue and BFS frontier.
Two pointers
An array technique using two indices moving toward or with each other to solve pair/partition problems in one O(n) pass instead of nested loops.
Sliding window
A moving sub-range over a sequence whose bounds expand/contract to maintain a property — how you compute rolling metrics in O(n).
Prefix sum
A precomputed array of cumulative totals so any range sum is answered in O(1) as pre[j] - pre[i].
BFS (breadth-first search)
Level-by-level graph/tree traversal driven by a queue; finds shortest paths in unweighted graphs.
DFS (depth-first search)
Traversal that goes as deep as possible before backtracking, driven by recursion or a stack; used for cycle detection and topological order.
Topological sort
A linear ordering of a DAG's nodes where every edge points forward — the ordering that runs task dependencies, e.g. an Airflow DAG.
DAG (directed acyclic graph)
A directed graph with no cycles; the model for pipelines and Spark job plans, and what topological sort operates on.
Memoization
Caching a function's result per input so repeated calls are instant; in Python often just @functools.lru_cache.
Dynamic programming
Solving a problem by combining answers to overlapping subproblems, stored to avoid recomputation; 1D DP carries a running best across one array.
Big-O
Notation for how runtime or memory grows with input size n — the order (constant, log, linear, n²) matters far more than the exact formula.
Vectorisation
Expressing an operation over a whole array/column at once so it runs in fast native code, instead of a Python-level loop — the pandas/NumPy performance mindset.
DB-API
Python's standard database interface (PEP 249): connect, cursor, execute, fetch*, commit — the shape every SQL driver follows.
Idempotency
A property where running an operation again yields the same result — critical for safe pipeline retries and backfills (e.g. upsert, not blind insert).
