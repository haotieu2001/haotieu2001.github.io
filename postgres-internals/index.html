How Postgres Stores Your Data

Every query you write eventually becomes the same physical act: read some 8 KB pages off a disk. An index is a device for reading fewer of them. A partition is a device for skipping them entirely. VACUUM exists because Postgres never overwrites a row in place. This page opens up the engine — the process model (why you need a connection pooler), the WAL, pages, tuples, the heap, MVCC, TOAST, B-trees, BRIN, partitioning — and then shows you how to read EXPLAIN and which monitoring queries actually tell you something.

PostgreSQL 18 storage · indexes · partitions every claim verified on a live server
~50 min read · Reference · Internals

The one idea (read this first)

A Postgres table is a pile of 8 KB pages, in no particular order. That pile is called the heap, and the database can only ever read whole pages — never a single row. So every performance question in this document reduces to one question: how many pages did we have to touch? A sequential scan touches all of them. An index touches a handful and then jumps to the exact pages holding the rows it wants. A partition lets the planner discard whole files without looking. And because Postgres uses MVCC — an UPDATE never overwrites a row, it writes a new version and leaves the old one behind as garbage — that pile of pages keeps accumulating dead rows, which is why VACUUM exists and why bloat is the most common self-inflicted wound in a Postgres database.

Two facts about the running system sit underneath all of that. Postgres forks a whole OS process for every connection, so connections are expensive and a pooler is not optional. And it never writes your change to the data file when you ask — it writes it to a sequential log, the WAL, and returns. Durability comes from the log; the data pages catch up later.

How this page was checked

Every number, plan and behaviour below was produced by running it on a live PostgreSQL 18.4 server against a one-million-row table. The EXPLAIN output is real output, not a sketch. Where a claim is version-specific — and skip scan is brand new in PG 18 — that's called out.

01. First: half these words mean something else here

Most SQL-internals material online is written about SQL Server. Bring its vocabulary to Postgres and you will ask for things that do not exist.

This is not a pedantic point. I ran every one of these against a live PostgreSQL 18 server, and CREATE CLUSTERED INDEX, CREATE NONCLUSTERED INDEX, CREATE COLUMNSTORE INDEX, CREATE FILTERED INDEX and CREATE FILEGROUP each came back with a flat syntax error. Some of these concepts have a Postgres equivalent under a different name; one of them has no equivalent at all. Start here, and the rest of the page will make sense.

The single most important line in that table

PostgreSQL has no clustered index. In SQL Server, a clustered index is the table — the rows live in the index's leaf pages, physically sorted by the key. In Postgres that is simply not a thing that can happen. The table is always an unordered heap, and every index is always a separate structure holding pointers back into it. If you have been reasoning about Postgres performance using a mental model where "the primary key sorts the table on disk" — it doesn't, and §05 explains what that changes.

02. The architecture: processes, memory, and the WAL

Before the storage layout, the shape of the running system. Postgres is a family of cooperating OS processes sharing one block of memory — and it forks a whole new process for every connection.

That last fact is not a detail. Postgres is process-based, not thread-based. Connect, and the postmaster fork()s a dedicated backend process that lives until you disconnect. It has its own memory, its own file descriptors, and costs several megabytes. This is the single most consequential architectural fact for anyone running Postgres at scale, and we come back to what it means in a moment.

Here is what was actually running on the live server:

text
postgres: checkpointer                     ← writes dirty pages at a checkpoint
postgres: background writer                ← trickles dirty pages out continuously
postgres: walwriter                        ← flushes the WAL buffer to disk
postgres: autovacuum launcher              ← starts vacuum workers (§07)
postgres: logical replication launcher     ← replication
postgres: io worker 0                      ← NEW in PG 18: asynchronous I/O
postgres: io worker 1
postgres: io worker 2
postgres: client backend                  ← ONE OF THESE PER CONNECTION

The write path: why COMMIT is both fast and durable

This is the central mechanism of the whole engine. When you write a row, Postgres does not go and update the data file. It writes the change to the WAL (Write-Ahead Log) — a sequential, append-only journal — and modifies the page in RAM. Your COMMIT returns as soon as the WAL is safely on disk. The actual data page may not be written for minutes.

Why do it this way? Because the WAL is sequential and the data pages are scattered. Writing one 200-byte record to the end of a log file is enormously cheaper than fsyncing a dozen random 8 KB pages across the disk. So Postgres buys durability at sequential-write prices and defers the expensive random writes to a background process that can batch them. If the server is killed, the WAL is replayed on restart and every committed change comes back. That is crash recovery, and it is the same log that gets shipped to replicas.

Memory: the one setting that will OOM your server

SettingDefault (measured)What it isScope

work_mem is not a total. It multiplies.

This is the classic way to take a Postgres server down. work_mem is the limit for a single sort or hash node — and one query can easily have five of them, and you can have a hundred connections. So work_mem = 256MB does not mean "256 MB". It means 256 MB × 5 nodes × 100 connections = 128 GB, and your server dies. Keep work_mem modest globally (4–32 MB), and raise it per session for the one big reporting query that genuinely needs it: SET work_mem = '512MB'; before it, and only for it.

Why you need a connection pooler

Come back to the process model. Every connection is a forked OS process consuming several megabytes and a scheduler slot. A web application with 500 connections is asking Postgres to run 500 processes — and Postgres will do it, badly. Throughput collapses past a few hundred, because the machine spends its time context-switching rather than working.

The fix is a connection pooler — PgBouncer is the standard one. It keeps a small pool of real backend connections (say 20) and multiplexes thousands of client connections across them. Rule of thumb: the useful number of active backends is roughly cores × 2–4, not "however many the app opens". If you are running Postgres behind an application server and you do not have a pooler, that is usually the first thing to fix — before any index.

New in PostgreSQL 18: asynchronous I/O

Those io worker processes in the list above did not exist before PG 18. Historically a backend that needed a page from disk blocked while the kernel fetched it — one page at a time, idling. PG 18 adds a real async I/O subsystem (io_method, defaulting to worker with three helper processes; on modern Linux kernels you can set io_uring instead). Backends now queue read requests and keep working while the I/O happens in parallel underneath.

It benefits exactly the operations a data engineer cares about — sequential scans, bitmap heap scans and vacuum — and reported gains on io_uring reach 2–3× on read-heavy scans. It also explains why effective_io_concurrency now defaults to 16 instead of 1. If you are on PG 17 or older, none of this exists and every disk read is a stall.

And the read path, in one paragraph

A query asks for page 14,285. Postgres looks in shared_buffers: if it's there, that's a buffer hit — the shared hit= number in every EXPLAIN (BUFFERS) output you'll see later. If not, it reads from the OS page cache or the disk (read=), and evicts something else to make room. That is the entire reason the cache hit ratio (§19) matters: at 99% your working set is in RAM and Postgres is fast; at 80% you are going to disk constantly, and no index will save you — you have a memory problem, not a query problem.

03. The storage hierarchy: cluster to tuple

Six nested layers sit between "a Postgres server" and "one row on a disk". Only the bottom two do any real work.

Two of these deserve a moment. A tablespace is Postgres's answer to SQL Server's file group: a named directory you can put relations into, which is how you put hot partitions on NVMe and cold ones on cheap disk. And the 1 GB segmentation is worth knowing because it surprises people: a 50 GB table is not one enormous file, it is fifty files called 16385, 16385.1, 16385.2 and so on.

sql
-- Where does a table physically live? Ask Postgres.
SELECT pg_relation_filepath('events');      --  base/5/16385
SHOW data_directory;                        --  /var/lib/postgresql/18/main
SHOW block_size;                             --  8192   ← always. Set at compile time.

-- Tablespaces: put a relation on a specific disk.
CREATE TABLESPACE fast LOCATION '/mnt/nvme/pg';
CREATE TABLE hot_events (…) TABLESPACE fast;

04. Inside an 8 KB page

The page is the atom. Postgres cannot read half a page, and it cannot read one row — it reads 8192 bytes or nothing.

You can see all of this from SQL. The ctid is a real, queryable column on every table:

sql
SELECT ctid, id, user_id FROM events ORDER BY id LIMIT 3;
--  ctid  | id | user_id
# -------+----+---------
--  (0,1) |  1 |    7415     ← page 0, slot 1
--  (0,2) |  2 |    6721
--  (0,3) |  3 |    2694

-- How big is the table, really? (1,000,000 rows, measured)
SELECT pg_size_pretty(pg_relation_size('events')) AS heap,       -- 112 MB
       pg_relation_size('events') / 8192           AS pages,      -- 14,286
       1000000.0 / (pg_relation_size('events')/8192) AS per_page;   -- 70.0 rows/page

Why "70 rows per page" is the number that explains everything

A million rows at ~70 per page is 14,286 pages. A sequential scan must read all 14,286 — that is the cost of the table. An index lookup that finds 109 matching rows must read a handful of index pages plus, in the worst case, 107 heap pages, because those 109 rows are scattered across the heap in insertion order, roughly one per page. That ratio — how many pages do I touch to get the rows I want — is the only thing the planner is really thinking about, and it's why an index that matches 30% of the table is worse than no index at all.

05. The heap — and the clustered index that isn't

In Postgres, every table is a heap: an unordered pile of pages. Rows go wherever there is room. There is no second option.

This is the deepest structural difference from SQL Server, and it changes how you reason. In SQL Server, a clustered index is the table — the rows are stored inside the index's leaf pages, physically sorted by the key, and a lookup by that key lands you directly on the data. In Postgres the table is always a heap, and every index — including the one behind your PRIMARY KEY — is a separate structure off to the side whose leaves hold ctid pointers back into it. Finding a row by primary key means walking the B-tree and then jumping into the heap.

What about CLUSTER?

Postgres does have a CLUSTER command, and it is a trap for anyone arriving from SQL Server. It physically rewrites the table once, sorted in an index's order. It does not create a clustered index, and — this is the crucial part — Postgres does not maintain that order. The very next INSERT goes wherever there is free space, and the table drifts back out of order. It also takes an ACCESS EXCLUSIVE lock, so the table is unreadable while it runs.

sql
CLUSTER events USING idx_created_at;   -- one-off physical sort. NOT maintained.
                                       -- locks the table completely while it runs.

-- What it's actually good for: raising "correlation" so BRIN and range scans work well.
-- Check correlation (1.0 = perfectly ordered on disk, 0 = random):
SELECT attname, correlation FROM pg_stats
WHERE tablename = 'events' AND attname IN ('created_at', 'user_id');
--  created_at |  1.00   ← inserted in time order, already sorted. BRIN will fly.
--  user_id    |  0.01   ← random. Only a B-tree can help here.

That correlation statistic is one of the most useful and least-known numbers in Postgres. It tells you how closely a column's logical order matches its physical order on disk — and it is the single input that decides whether BRIN (§13) will be brilliant or useless.

06. MVCC: an UPDATE is a DELETE plus an INSERT

Postgres never overwrites a row. An UPDATE writes a brand-new row version somewhere else and leaves the old one lying there, dead. This one design decision explains bloat, VACUUM, and half your performance problems.

The mechanism is MVCC — Multi-Version Concurrency Control. Every tuple carries two hidden system columns: xmin, the transaction that created it, and xmax, the transaction that deleted it. A reader looking at the table only sees tuples whose xmin has committed and whose xmax hasn't. That is how Postgres gives you a consistent snapshot without readers ever blocking writers, and it is genuinely excellent. The price is that the dead versions physically remain in the pages until something cleans them up.

Here is the proof, run on a live server. Watch the ctid:

sql
SELECT ctid, xmin, id FROM events WHERE id = 1;
--   ctid  | xmin | id
--  (0,1)  | 752  |  1        ← page 0, slot 1

UPDATE events SET status = 'x' WHERE id = 1;

SELECT ctid, xmin, id FROM events WHERE id = 1;
--     ctid    | xmin | id
--  (14285,51) | 757  |  1     ← the row MOVED. New page, new transaction id.

SELECT n_live_tup, n_dead_tup FROM pg_stat_user_tables WHERE relname = 'events';
--  n_live_tup | n_dead_tup
--    1000000  |     1        ← and left a corpse behind at (0,1)

HOT updates: the optimisation that saves you

If Postgres had to update every index on every UPDATE, write amplification would be brutal. So there's an escape hatch: a HOT update (Heap-Only Tuple). If the new version fits on the same page and you did not modify any indexed column, Postgres writes the new tuple on that page and chains the old line pointer to it — and touches no index at all. The indexes still point at the original slot, which now redirects.

Two things follow, and both are actionable. Don't index columns you update constantly — a single index on a hot column disables HOT for every update to that table's row. And leave room on the page: fillfactor tells Postgres to leave a percentage of each page empty on insert, precisely so that later updates have somewhere local to land.

sql
-- Leave 15% of every page free so UPDATEs can stay on-page (HOT) instead of
-- migrating to a new page and dirtying every index.
ALTER TABLE events SET (fillfactor = 85);
VACUUM FULL events;   -- needed to actually apply it to existing pages

-- Are your updates HOT? n_tup_hot_upd should be close to n_tup_upd.
SELECT relname, n_tup_upd, n_tup_hot_upd,
       round(100.0 * n_tup_hot_upd / nullif(n_tup_upd, 0), 1) AS hot_pct
FROM pg_stat_user_tables WHERE relname = 'events';

07. VACUUM, autovacuum, and bloat

VACUUM is the garbage collector for dead tuples. If it falls behind, your table grows without bound and every scan gets slower — this is the most common self-inflicted wound in Postgres.

Plain VACUUM walks the table, finds tuples no transaction can still see, and marks their space reusable. Note the word: reusable, not returned. The file on disk does not shrink. I measured it — a 117 MB table with 50,000 dead tuples was still exactly 117 MB after VACUUM. The space is now free for future inserts, which is usually what you want, but the disk usage does not drop.

What it doesLockReturns disk to the OS?
VACUUMMarks dead tuple space reusable; updates the visibility map (§12)None — runs alongside trafficNo
VACUUM ANALYZEThe above, plus refreshes planner statisticsNoneNo
VACUUM FULLRewrites the whole table compactlyACCESS EXCLUSIVE — table unusableYes
autovacuumA daemon that runs VACUUM for you, automaticallyNoneNo

Never run VACUUM FULL on a live production table

It takes an ACCESS EXCLUSIVE lock for its entire duration — nothing can read or write the table, not even a SELECT — and it needs enough free disk to hold a second full copy while it rewrites. On a big table that is a multi-hour outage. If you genuinely need to reclaim space with the table online, use pg_repack, which does the same job without the long lock. Better still: fix the autovacuum settings so the bloat never accumulates.

Why autovacuum falls behind (and the one setting to change)

Autovacuum triggers when dead tuples exceed autovacuum_vacuum_threshold + autovacuum_vacuum_scale_factor × live_rows. The defaults are 50 and 0.2 — so a table is vacuumed once it has more dead rows than 20% of its live rows. On a thousand-row table that's fine. On a 100-million-row table, that means waiting for 20 million dead tuples before autovacuum even starts, by which point the damage is done and the vacuum itself is enormous.

sql
-- The fix for a big, hot table: a much smaller scale factor, so it vacuums often
-- and cheaply instead of rarely and catastrophically.
ALTER TABLE events SET (
    autovacuum_vacuum_scale_factor = 0.01,   -- 1% instead of 20%
    autovacuum_vacuum_threshold    = 1000
);

-- Is it keeping up? dead_pct creeping up over time = autovacuum is losing.
SELECT relname, n_live_tup, n_dead_tup,
       round(100.0 * n_dead_tup / nullif(n_live_tup + n_dead_tup, 0), 1) AS dead_pct,
       last_autovacuum
FROM pg_stat_user_tables
WHERE n_dead_tup > 1000
ORDER BY dead_pct DESC;
--  events | 1000000 | 49859 |  4.7 | 2026-07-14 16:30:45

Rule of thumb: above ~20% dead tuples, vacuum is overdue. And a last_autovacuum that is NULL or ancient on a busy table is an alarm, not a curiosity.

The other job VACUUM does — and the one that can take you down

Vacuum also freezes old transaction IDs. Postgres transaction IDs are 32-bit and wrap around, so tuples older than ~2 billion transactions must be marked "frozen" (visible to everyone) before the counter laps them. If autovacuum is so far behind that this can't happen, Postgres will refuse to accept new writes to protect your data — the infamous transaction ID wraparound shutdown. It is rare, it is entirely preventable, and it is always caused by ignoring autovacuum for months.

08. TOAST: oversized values

A tuple cannot span pages. So what happens when you store a 100 KB JSON blob in an 8 KB page? TOAST — The Oversized-Attribute Storage Technique.

When a row exceeds roughly 2 KB (a quarter of a page), Postgres starts moving its biggest variable-length columns out. But it does two things, in order, and most people only know about the second one: it compresses first, and only if that isn't enough does it move the value out-of-line into a hidden side table.

Here is that in action, measured — two rows, both about 100 KB of text, but one compresses and one doesn't:

sql
CREATE TABLE wide(id int, big text);
INSERT INTO wide VALUES (1, repeat('a', 100000));                    -- compressible
INSERT INTO wide VALUES (2, (SELECT string_agg(md5(random()::text),'')
                              FROM generate_series(1,3200)));       -- random = incompressible

SELECT id, pg_column_size(big) AS stored, length(big) AS chars FROM wide;
--  id | stored  | chars
--   1 |   1,156 | 100,000    ← compressed 86× and stayed INLINE
--   2 | 102,400 | 102,400    ← would not compress, so it was pushed OUT

SELECT pg_size_pretty(pg_relation_size('wide'))          AS main_heap,   -- 8192 bytes
       pg_size_pretty(pg_relation_size(reltoastrelid))     AS toast_table  -- 104 kB
FROM pg_class WHERE relname = 'wide';

Row 1 shrank from 100,000 bytes to 1,156 and never left the page. Row 2 couldn't compress, so it was chopped into chunks and stored in a separate TOAST table — and the main heap stayed tiny at one page, holding only a pointer.

Why a data engineer should care about TOAST

TOAST is usually invisible and usually helpful. It becomes a problem in exactly one situation: you SELECT * on a table with a big jsonb or text column, and you didn't need that column. Every row now costs an extra trip to the TOAST table plus a decompression. Selecting only the columns you need is not a style preference here — it is the difference between reading one page and reading hundreds. Note also that pg_relation_size() excludes the TOAST table, so a table that looks small can be hiding gigabytes; use pg_total_relation_size() to see the truth.

09. B-tree: how an index finds a row

A B-tree is a sorted, balanced tree of 8 KB pages. Balanced means every leaf is the same distance from the root — so every lookup costs the same handful of page reads, no matter where the row is.

Three or four levels is typical even for enormous tables, because each internal page holds hundreds of keys. A billion-row table might have a B-tree of depth 4: read the root, read an internal page, read another, read a leaf. Four page reads to locate any row out of a billion — that is the whole magic, and it is why an index turns a 14,286-page scan into a 4-page lookup.

10. The scan types, and reading EXPLAIN

There are four ways Postgres can get rows out of a table. Knowing which one you got — and why — is most of query tuning.

All four outputs below are real, from EXPLAIN (ANALYZE, BUFFERS) on the same one-million-row table. Watch the Buffers line: that is the page count, and the page count is the cost.

1 · Seq Scan — read every page

text
-- No index on user_id. 109 rows out of a million.
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM events WHERE user_id = 42;

Gather (actual rows=109 loops=1)
  Workers Launched: 2
  Buffers: shared hit=13762 read=524              ← 14,286 pages. The whole table.
  -> Parallel Seq Scan on events (actual rows=36 loops=3)
       Filter: (user_id = 42)
       Rows Removed by Filter: 333297            ← threw away 99.99% of what it read

Rows Removed by Filter is the tell. Reading 333,297 rows to keep 36 is the definition of a missing index. (Note it wasn't slow — Postgres parallelised it across 3 workers. Seq scans are fine when you actually want most of the table.)

2 · Index Scan / Bitmap Heap Scan — walk the tree, then fetch

text
-- Now: CREATE INDEX idx_user ON events(user_id);
Bitmap Heap Scan on events (actual rows=109 loops=1)
  Recheck Cond: (user_id = 42)
  Heap Blocks: exact=107                          ← 107 heap pages for 109 rows!
  Buffers: shared hit=107 read=3                  ← 110 pages, down from 14,286
  -> Bitmap Index Scan on idx_user (actual rows=109 loops=1)
       Index Cond: (user_id = 42)
       Buffers: shared read=3                       ← the index itself cost 3 pages

From 14,286 pages to 110. But look at Heap Blocks: exact=107 — 107 pages to fetch 109 rows, nearly one page per row. That's the red arrows from the B-tree diagram: those users are scattered randomly across the heap, so almost every row lives on its own page. This is exactly the cost that an index-only scan eliminates.

The difference between an Index Scan and a Bitmap Heap Scan is worth a sentence, because both appear constantly. An Index Scan walks the tree and fetches heap rows one at a time, in index order — great when you want a handful of rows. A Bitmap Heap Scan first collects all the matching ctids into an in-memory bitmap, sorts them by page number, and then reads each heap page once, in physical order — which turns random I/O into sequential I/O. Postgres chooses it when there are too many matches for one-at-a-time fetching to be worth it.

3 · Bitmap Heap Scan on a range — the middle ground

text
SELECT * FROM events WHERE user_id BETWEEN 42 AND 300;   -- 25,921 rows (2.6%)

Bitmap Heap Scan on events (actual rows=25921 loops=1)
  Heap Blocks: exact=11988                        ← 11,988 of 14,286 pages!
  Buffers: shared hit=11780 read=234

This is the lesson that surprises people. Matching only 2.6% of the rows still required reading 84% of the pages — because 26,000 randomly-scattered rows land on almost every page in the table. At this point the index has bought you almost nothing, and a little further along the planner will correctly abandon it and seq-scan instead. An index is only useful when the rows it finds are few, or physically clustered together.

4 · Index-Only Scan — never touch the heap at all

text
SELECT user_id FROM events WHERE user_id = 42;   -- only needs the indexed column

Index Only Scan using idx_user on events (actual rows=109 loops=1)
  Index Cond: (user_id = 42)
  Heap Fetches: 0                                 ← it never opened the heap
  Buffers: shared hit=4                           ← FOUR pages. Down from 14,286.

Four pages. The index contained every column the query asked for, so Postgres answered entirely from the tree. This is the best outcome available, and §12 explains the one condition it depends on — a condition that quietly fails more often than you'd think.

Plan nodePages read (measured)Postgres picks it when…

How to actually read an EXPLAIN

sql
EXPLAIN                          -- the plan and the planner's GUESSES. Doesn't run.
EXPLAIN (ANALYZE)                -- RUNS it. Now you get actual rows and actual time.
EXPLAIN (ANALYZE, BUFFERS)       -- ← always use this. Buffers = pages = the real cost.

The three numbers that matter, in order

1. Buffers. Pages read. This is the actual work. Ignore cost=… — it's an arbitrary unit for comparing plans, not a time.
2. Estimated rows vs actual rows. If the planner guessed 10 and got 500,000, it chose the wrong plan for a reason: its statistics are stale. Run ANALYZE. This is the single most common cause of a "suddenly slow" query.
3. Rows Removed by Filter. Rows read and then thrown away. A big number here is an index that doesn't exist or isn't being used.

And read the plan inside-out, bottom-up: the most indented node runs first and feeds its parent. loops=3 means that node ran three times, and the actual rows shown is the average per loop — so multiply before you panic.

11. How Postgres joins tables

The last section covered how rows come out of one table. Real plans are mostly about combining two — and there are exactly three algorithms for it.

Every join node in every EXPLAIN you will ever read is a Nested Loop, a Hash Join, or a Merge Join. They all produce identical rows. They differ enormously in what they cost, and the planner chooses between them from its statistics — which is why a bad row estimate so often shows up as "it picked the wrong join".

1 · Nested Loop — and the loops= number everyone misreads

For each row on the outer side, go and find the matching rows on the inner side. If the inner side has an index, that inner lookup is cheap and this is the best possible join for a small number of outer rows. Real output:

text
SELECT u.name, e.amount FROM users u JOIN events e ON e.user_id = u.id WHERE u.id = 42;

Nested Loop (actual rows=109 loops=1)
  Buffers: shared hit=4 read=4                    ← 8 pages. Nothing.
  -> Index Scan using users_pkey on users u (actual rows=1 loops=1)
       Index Cond: (id = 42)                       ← ONE outer row
  -> Index Only Scan on events e (actual rows=109 loops=1)
       Index Cond: (user_id = 42)                  ← the inner side is indexed +

loops=N means the numbers next to it are per loop

This is the single most misread thing in EXPLAIN. In a real plan I captured, the inner node said actual rows=100 loops=2435. That does not mean 100 rows — it means the node ran 2,435 times and returned ~100 rows each time. The real total is 243,554 rows. Always multiply rows × loops before you decide a node is cheap. A nested loop with a huge outer side and no index on the inner is N × M — and it will run until the heat death of the universe while the plan looks innocent.

2 · Hash Join — the data engineer's workhorse

Build an in-memory hash table from the smaller side, then scan the bigger side once and probe the hash for each row. Both tables are read exactly once, so it scales beautifully to large joins with no useful index — which describes most analytical queries. It only works for equality conditions.

text
-- Same join, but with no index on events.user_id:
SELECT count(*) FROM users u JOIN events e ON e.user_id = u.id WHERE u.country = 'US';

Hash Join (actual rows=81184 loops=3)
  Hash Cond: (e.user_id = u.id)
  -> Parallel Seq Scan on events e (actual rows=333333 loops=3)
  -> Hash                                          ← builds the table from `users` (the small side)
       Buckets: 4096  Batches: 1  Memory Usage: 152kB   ← Batches: 1 = fits in RAM +

The line to look at is Batches. Batches: 1 means the hash table fit in work_mem and everything happened in memory. Batches: 8 means it did not — Postgres partitioned the join and spilled to disk, and it is now many times slower. That is not a join problem; it is a work_mem problem (§02), and the fix is to raise work_mem for that one query.

3 · Merge Join — cheap only if the sorting is already done

Sort both inputs by the join key, then walk the two sorted streams in lockstep like a zip. Very efficient if the inputs are already sorted — say, because both are being read in index order. If they are not, Postgres must sort them first, and the sort is the whole cost. Here is a forced merge join, and look what it did:

text
Merge Join (actual rows=333316 loops=3)
  Merge Cond: (e.user_id = u.id)
  Buffers: shared hit=1349 read=13828, temp read=1471 written=1475
  -> Sort (actual rows=333333 loops=3)             ← it had to sort a million rows

temp read=1471 written=1475 is the tell: those are temporary files on disk. The sort did not fit in work_mem, so it spilled — writing ~1,400 blocks out and reading them back. Whenever you see temp in a plan, something overflowed memory and went to disk, and that is almost always the slowest thing in the query.

AlgorithmHow it worksPlanner picks it whenThe failure mode
Nested LoopFor each outer row, look up the innerFew outer rows and the inner side is indexedNo index on the inner side → N × M
Hash JoinHash the small side, probe with the big oneLarge joins, no useful index. Equality onlyBatches > 1 → hash spilled to disk
Merge JoinSort both sides, walk them togetherBoth inputs already sorted; also handles rangesThe Sort spills → temp read/written

How to debug a bad join, in order

1. Compare estimated vs actual rows on the join node. Off by 100× or more? The planner chose this algorithm on bad information — run ANALYZE and re-check before touching anything else. This is the fix far more often than people expect.
2. Nested loop with a big loops=? The inner side needs an index.
3. Batches > 1 or temp read/written? It spilled. SET work_mem higher for that query only (§02 — it multiplies).
4. Only as a last resort, and only to diagnose, disable an algorithm to see what the planner would do otherwise: SET enable_nestloop = off;. Never leave that in production code — you are lying to the planner, not fixing it.

12. Index-only scans and the visibility map

An index-only scan is the fastest thing Postgres can do. It also silently doesn't happen, for a reason that has nothing to do with your index.

Here's the problem. The index holds the key and the ctid — but it does not hold visibility information. It cannot tell you whether a given row version is dead, or belongs to an uncommitted transaction. Normally Postgres resolves that by fetching the heap tuple and checking xmin/xmax. But that fetch is exactly what an index-only scan is trying to avoid.

The escape is the visibility map: one bit per heap page, meaning "every tuple on this page is visible to everyone". VACUUM sets those bits. If the bit is set, Postgres can trust the index and skip the heap entirely. If it isn't set, it must go to the heap anyway — and your index-only scan quietly degrades into an ordinary one.

sql
-- Is the visibility map current? relallvisible should be close to relpages.
SELECT relname, relpages, relallvisible,
       round(100.0 * relallvisible / nullif(relpages,0), 1) AS pct_all_visible
FROM pg_class WHERE relname = 'events';
--  events | 14286 | 14286 | 100.0    ← perfect. Index-only scans available.

-- Is a supposed index-only scan secretly hitting the heap? Look for Heap Fetches.
EXPLAIN (ANALYZE, BUFFERS) SELECT user_id FROM events WHERE user_id = 42;
--   Heap Fetches: 0        + genuinely index-only
--   Heap Fetches: 109      − it's lying to you — VACUUM the table

"Index Only Scan" in the plan does not mean it skipped the heap

The node is named for what it attempted, not what it achieved. Always read the Heap Fetches line. A high number means the visibility map is stale and you are paying full price with extra steps. The fix is not a better index — it's VACUUM, or an autovacuum that keeps up.

13. The other index types: GIN, GiST, BRIN, Hash

Postgres ships six index methods. B-tree is the default and covers ~90% of cases. But one of the others is a superpower for data engineers, and it is almost always overlooked.

I asked the live server what it has:

sql
SELECT amname FROM pg_am WHERE amtype = 'i';
--  brin · btree · gin · gist · hash · spgist
MethodUse it forNotes
btreeEquality and ranges on anything sortable. =, <, >, BETWEEN, ORDER BYThe default. If unsure, this
brinHuge tables where the column tracks physical order — timestamps in an append-only logAstonishingly small. See below

BRIN: the index that costs almost nothing

A BRIN (Block Range INdex) doesn't store a row-level entry at all. It stores, for each block range (128 pages by default), just the minimum and maximum value in that range. To answer WHERE created_at > X it checks each summary and skips the ranges that can't possibly match.

That makes it microscopic. Measured on the same one-million-row table, both indexing created_at:

BRIN is the data engineer's index

Think about what a typical DE fact table looks like: append-only, ordered by time, enormous, and queried by time range. That is precisely BRIN's ideal case — the physical order already matches the logical order, so the block summaries are tight. You get range-scan pruning for a rounding error of disk space and almost no write overhead. If your events table is 500 GB and you're indexing created_at with a B-tree, you are spending gigabytes to buy what BRIN would give you for megabytes.

The one condition: correlation. Check it with SELECT correlation FROM pg_stats WHERE attname='created_at'. Near 1.0 (or −1.0) → BRIN will fly. Near 0 → BRIN is worthless, because every block range will contain the full spread of values and nothing can be skipped.

sql
CREATE INDEX idx_brin ON events USING brin(created_at);        -- 24 kB
CREATE INDEX idx_gin  ON events USING gin(payload jsonb_path_ops);  -- for jsonb @> queries
CREATE INDEX idx_fts  ON docs   USING gin(to_tsvector('english', body));

14. Unique, partial, covering, expression, multicolumn

Same B-tree, five different ways of aiming it. These are where most of the practical wins live.

Unique — a constraint that happens to be an index

sql
CREATE UNIQUE INDEX idx_email ON users(email);
ALTER TABLE users ADD CONSTRAINT uq_email UNIQUE (email);  -- same thing, built on an index
-- A PRIMARY KEY is exactly this + NOT NULL. The uniqueness IS enforced by the B-tree.

Partial — SQL Server calls this a "filtered index"

Index only the rows you actually query. If 90% of your orders are completed and every query hunts for the pending ones, indexing all of them is waste. Measured on our table, a partial index was half the size of the full one:

sql
CREATE INDEX idx_pending ON events(user_id) WHERE status = 'pending';
--  full B-tree on user_id  : 6,960 kB
--  partial (status=pending): 3,496 kB   ← half the size, and faster to maintain

-- The planner only uses it if your WHERE clause PROVABLY implies the index's WHERE.
SELECT … WHERE user_id = 42 AND status = 'pending';   -- + uses it
SELECT … WHERE user_id = 42;                          -- − cannot use it

-- The killer use: index only the NOT NULL rows of a mostly-NULL column.
CREATE INDEX idx_churn ON users(churned_at) WHERE churned_at IS NOT NULL;

Covering — INCLUDE extra columns to buy an index-only scan

The whole point of INCLUDE is to make an index-only scan possible for a query that needs a column it doesn't search on. The included columns ride along in the leaf pages only — they don't bloat the internal navigation pages, and they don't have to be sortable.

sql
CREATE INDEX idx_cov ON events(user_id) INCLUDE (amount);

SELECT user_id, amount FROM events WHERE user_id = 42;
--  Index Only Scan using idx_cov     ← measured
--    Heap Fetches: 0                   the heap was never opened

Expression — index the computed thing, not the column

This is the fix for the sargability problem: if you must filter on LOWER(email), then index LOWER(email). The index and the query now match exactly.

sql
CREATE INDEX idx_lower_email ON users(LOWER(email));
SELECT * FROM users WHERE LOWER(email) = 'a@b.com';    -- + now uses the index

CREATE INDEX idx_day ON events(date_trunc('day', created_at));

Multicolumn — and the PG 18 change that rewrites the old advice

In a multicolumn index, column order matters. The classic rule — the leftmost prefix rule — says an index on (a, b) can serve WHERE a = … and WHERE a = … AND b = …, but is useless for WHERE b = … alone, because the tree is sorted by a first and b's values are scattered across the whole index.

PostgreSQL 18 (September 2025) softened that with skip scan. If the leading column has few distinct values, the planner can now hop through each distinct prefix in turn and search within it — using the index even though your query never mentions the leading column. I tested it, and it works:

text
-- Only index present: idx_multi (status, user_id).  status has just 3 distinct values.
-- The query filters ONLY on user_id — the SECOND column.
SELECT count(*) FROM events WHERE user_id = 42;

Index Only Scan using idx_multi on events (actual rows=109 loops=1)
  Index Cond: (user_id = 42)          ← used the 2nd column WITHOUT the 1st
  Heap Fetches: 0
  Index Searches: 7                   ← the tell: 7 tree descents, one per prefix

Does skip scan mean column order stopped mattering? No.

Skip scan only pays when the leading column has few distinct values — here status had 3, so 7 tree descents covered it. If the leading column were user_id with 10,000 distinct values, skipping would mean 10,000 descents and the planner would rightly seq-scan instead. The rule still stands: put the column you filter on most selectively, and most often, first. Skip scan is a safety net that rescues a badly-ordered index on a low-cardinality prefix — not a licence to stop thinking. (And on PG 17 or older, none of this exists: that query is a plain Seq Scan.)

15. Columnstore: the thing Postgres doesn't have

This is the one item on the list with no Postgres answer — and understanding why leads straight to why your lakehouse exists.

Postgres is row-oriented, all the way down. A page holds whole tuples, with every column of a row sitting together. That's ideal for OLTP: fetch one customer, get all their fields in one page read. It is close to the worst possible layout for analytics. Run SELECT AVG(amount) FROM events and Postgres must read all 14,286 pages — every column of every row — in order to use one number from each. You paid to read the payload, the status, the timestamps, and threw them all away.

A columnstore stores each column separately. The same query reads only the amount column's storage — a fraction of the bytes — and because a column holds one data type with lots of repetition, it compresses 10× or better. That is why analytical engines are all columnar.

Rowstore (Postgres heap)Columnstore (Parquet, Delta)
LayoutAll columns of a row, togetherAll values of a column, together
SELECT * WHERE id = 5One page readMust reassemble from every column
SELECT AVG(x) over 1B rowsReads every column of every rowReads one column, compressed
CompressionPoor — mixed types on a pageExcellent — one type, repetitive
Good forOLTP: point reads, updatesOLAP: scans, aggregates

This is the whole reason the lakehouse exists

Core Postgres has no columnstore, and the extensions that add one (Citus columnar, Hydra) are real but niche. The industry's actual answer was to stop trying — and to put the analytical copy of the data somewhere columnar instead: Parquet files, and Delta Lake or Iceberg on top of them. When you write a Delta table in Databricks, or a Spark job reads Parquet and skips 95% of the file via column pruning and min/max statistics, that is a columnstore doing exactly what Postgres cannot.

And notice that Delta's file-level min/max skipping is the same idea as BRIN (§13), just applied to files instead of block ranges. Same trick, different scale. Postgres is the right tool for the transactional source; the lakehouse is the right tool for the analytical copy. Knowing why is what the OLTP-vs-OLAP distinction actually means.

16. Partitioning: RANGE, LIST, HASH

Partitioning splits one logical table into many physical ones. The payoff is not "smaller tables" — it's that the planner can throw entire partitions away without opening them.

Postgres has had proper declarative partitioning since version 10. You declare a partition key and a strategy, then attach child tables. Inserts route automatically; queries see one table.

sql
CREATE TABLE ev_part (
    id bigint, created_at timestamptz NOT NULL, user_id int, amount numeric
) PARTITION BY RANGE (created_at);

CREATE TABLE ev_2024_01 PARTITION OF ev_part
    FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');   -- lower INCLUSIVE,
CREATE TABLE ev_2024_02 PARTITION OF ev_part                -- upper EXCLUSIVE
    FOR VALUES FROM ('2024-02-01') TO ('2024-03-01');
StrategySplit byUse forPruning works on
RANGEA range of valuesTime-series. One partition per month/dayRanges and equality
LISTAn explicit set of valuesRegion, tenant, countryEquality and IN
HASHhash(key) % modulusSpreading write load evenlyEquality only — never ranges

Partition pruning — the actual point

Here is a query filtered on the partition key, measured. Note what is absent from the plan:

text
SELECT count(*) FROM ev_part
WHERE created_at >= '2024-02-01' AND created_at < '2024-03-01';

Aggregate
  -> Seq Scan on ev_2024_02            ← ONLY February. Jan and Mar never opened.
       Buffers: shared hit=345

-- Now the SAME table, but filtering on a NON-partition-key column:
SELECT count(*) FROM ev_part WHERE user_id = 5;

Aggregate
  -> Append
       -> Bitmap Heap Scan on ev_2024_01   ← every
       -> Bitmap Heap Scan on ev_2024_02   ← single
       -> Bitmap Heap Scan on ev_2024_03   ← partition

No partition key in the WHERE clause = no pruning = you made things worse

This is the mistake that turns partitioning from a win into a regression. If your query doesn't filter on the partition key, Postgres must scan every partition — and you've now added per-partition planning overhead on top of the same work. Choose the partition key to match how you actually query, which for a fact table is almost always the time column. And beware the subtle version: wrapping the key in a function (WHERE date_trunc('day', created_at) = …) can defeat pruning, exactly as it defeats an index.

When partitioning is worth it — and when it isn't

Partitioning is not a general speed-up, and reaching for it too early is common. The genuine wins are: dropping old data instantly (DROP TABLE ev_2023_01 is a metadata operation — versus a DELETE of 100 million rows, which generates 100 million dead tuples and a vacuum storm); vacuum that can keep up, because each partition is small enough to process; and pruning. Below roughly 50–100 GB, with no data-retention requirement, plain indexes are usually simpler and just as fast.

17. Partitioning and indexing together

They compose. Partitioning cuts the table down to the right slice; the index finds the rows inside that slice.

Create an index on the parent and Postgres automatically creates and maintains a matching index on every partition, including ones you add later. I verified the cascade — three partitions, one CREATE INDEX, three indexes:

sql
CREATE INDEX ON ev_part(user_id);     -- on the PARENT

SELECT c.relname AS partition,
       (SELECT count(*) FROM pg_index i WHERE i.indrelid = c.oid) AS indexes
FROM pg_class c JOIN pg_inherits inh ON inh.inhrelid = c.oid
WHERE inh.inhparent = 'ev_part'::regclass;
--  ev_2024_01 | 1
--  ev_2024_02 | 1     ← each partition got its own local index, automatically
--  ev_2024_03 | 1

These are local indexes: each one covers only its own partition. Postgres has no global index across partitions — which is precisely what makes DROP TABLE ev_2023_01 instant, since there is no giant shared index to surgically remove entries from.

The design that actually works for a fact table

Partition by time (RANGE on created_at, monthly or daily) so old data drops instantly and vacuum stays sane. Index by access pattern — a B-tree on the columns you filter by within a time window (user_id, status). Then BRIN the time column inside each partition if the partitions are still large, since the data is append-ordered and BRIN costs nothing. And put the current month's partition on a fast tablespace and the archive on cheap disk — ALTER TABLE … SET TABLESPACE per partition. That's the whole recipe, and it's the same shape as partitioning a Delta table by date.

sql
-- A unique constraint on a partitioned table MUST include the partition key.
-- Postgres cannot enforce global uniqueness without a global index — and it has none.
CREATE TABLE ev_part (id bigint, created_at timestamptz,
    PRIMARY KEY (id, created_at))          -- ← created_at is forced into the PK
PARTITION BY RANGE (created_at);

That last constraint catches people out and is worth internalising: you cannot have a globally-unique id on a partitioned table unless the partition key is part of the key. It is a direct consequence of there being no global index — and it is the same trade-off, for the same reason, that you meet when partitioning tables in a lakehouse.

18. The optimization playbook

A loop, not a checklist. Measure, find the worst node, change one thing, measure again. Guessing at indexes is how you end up with forty of them and a slow database.

Rules for index design

1

Index what you filter and join on, not what you select

Columns in WHERE, JOIN … ON and ORDER BY are index candidates. A column that only ever appears in the SELECT list is not — unless you're adding it via INCLUDE to buy an index-only scan.

2

Keep the column bare — or index the expression

WHERE date_trunc('day', created_at) = '2024-03-01' cannot use an index on created_at. Either rewrite it as a half-open range (>= '2024-03-01' AND < '2024-03-02'), or build an expression index on date_trunc('day', created_at). Both work; the range is usually better because one index then serves every granularity.

3

Selectivity first in a multicolumn index

Put the column that eliminates the most rows — and that you filter on most often — first. PG 18's skip scan can rescue a bad ordering when the leading column has few distinct values, but don't design around the safety net.

4

Build indexes CONCURRENTLY in production

A plain CREATE INDEX takes a lock that blocks all writes for the duration. On a big table that is an outage. CREATE INDEX CONCURRENTLY does not — it's slower and can't run in a transaction, and it can leave an INVALID index if it fails (just drop and retry), but it doesn't take your application down.

5

Delete the indexes nobody uses

Every index is a tax on every write, and a redundant index is pure loss: an index on (a, b) already serves queries on a alone, so a separate index on (a) is dead weight. §19 has the query that finds them.

sql
-- The single highest-value thing to check when a query "suddenly got slow".
-- Stale statistics → the planner guesses wrong → it picks a terrible plan.
ANALYZE events;

-- Building an index without taking the table down:
CREATE INDEX CONCURRENTLY idx_user ON events(user_id);

-- For a skewed column, tell Postgres to keep more detailed statistics.
-- (default 100 buckets; up to 10,000 for a really lumpy distribution)
ALTER TABLE events ALTER COLUMN status SET STATISTICS 1000;
ANALYZE events;

19. Monitoring: the queries to actually run

Six queries. Every one of these produced real output on the live server, and each answers a question you will genuinely be asked.

1 · What is slow? (the only place to start)

sql
-- pg_stat_statements is an extension. Enable it — it is the single most useful
-- diagnostic in Postgres. Requires a restart:
--   shared_preload_libraries = 'pg_stat_statements'   (in postgresql.conf)
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

SELECT substr(query, 1, 60) AS query,
       calls,
       round(total_exec_time::numeric, 1) AS total_ms,
       round(mean_exec_time::numeric, 2)  AS avg_ms,
       shared_blks_read                    AS pages_from_disk
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;

Sort by total_exec_time, not mean. A 20 ms query run two million times is a far bigger problem than a 4-second report that runs nightly, and only the total will show you that.

2 · Which tables are missing an index?

sql
SELECT relname, seq_scan, idx_scan, n_live_tup
FROM pg_stat_user_tables
WHERE n_live_tup > 10000
ORDER BY seq_scan DESC;
--  relname | seq_scan | idx_scan | n_live_tup
--  events  |       16 |       11 |    1000000

A big table with a high seq_scan count and a low idx_scan count is the classic missing-index signature. (Small tables legitimately seq-scan forever — the planner knows a 200-row table fits in one page and an index would be slower.)

3 · Which indexes are dead weight?

sql
SELECT relname AS table, indexrelname AS index,
       idx_scan AS times_used,
       pg_size_pretty(pg_relation_size(indexrelid)) AS size
FROM pg_stat_user_indexes
WHERE idx_scan = 0
  AND indexrelid NOT IN (SELECT conindid FROM pg_constraint WHERE conindid > 0)
ORDER BY pg_relation_size(indexrelid) DESC;
--  events | idx_never_used | 0 | 21 MB     ← never used once. Costing every write.

Two cautions before you drop anything. The counters are cumulative since the last stats reset, so a fresh server will show everything as unused — let it run for a full business cycle first. And the query deliberately excludes constraint-backing indexes, because dropping the index behind a UNIQUE or PRIMARY KEY would drop the constraint.

4 · Is it bloated?

sql
SELECT relname, n_live_tup, n_dead_tup,
       round(100.0 * n_dead_tup / nullif(n_live_tup + n_dead_tup, 0), 1) AS dead_pct,
       last_autovacuum, last_autoanalyze
FROM pg_stat_user_tables
WHERE n_dead_tup > 1000
ORDER BY dead_pct DESC;
--  events | 1000000 | 49859 | 4.7 | 2026-07-14 16:30:45 | …

Above ~20% dead, autovacuum is losing. Above 50%, half your I/O is spent reading garbage.

5 · Is it in RAM?

sql
SELECT round(100.0 * sum(heap_blks_hit)
             / nullif(sum(heap_blks_hit) + sum(heap_blks_read), 0), 2) AS cache_hit_pct
FROM pg_statio_user_tables;
--  99.86     ← measured. You want > 99%.

Below 95% on an OLTP workload means your working set doesn't fit in shared_buffers and you're going to disk constantly. That is a memory problem, not a query problem, and no index will fix it.

6 · What is blocking what?

sql
-- When everything hangs: find who is blocking whom.
SELECT pid, pg_blocking_pids(pid) AS blocked_by,
       left(query, 60) AS query, state,
       now() - query_start AS duration
FROM pg_stat_activity
WHERE cardinality(pg_blocking_pids(pid)) > 0;

-- Long-running transactions are the silent killer: an idle-in-transaction session
-- holds back the vacuum horizon, so dead tuples CANNOT be cleaned anywhere.
SELECT pid, state, now() - xact_start AS open_for, left(query, 50)
FROM pg_stat_activity
WHERE state = 'idle in transaction'
ORDER BY xact_start;

The bug that looks like a vacuum problem but isn't

If one connection sits idle in transaction for hours, VACUUM cannot remove any tuple newer than that transaction's snapshot — anywhere in the database, not just in the tables it touched. Autovacuum runs, reports success, and reclaims nothing, while your tables bloat relentlessly. It's usually an application that opened a transaction and forgot to commit. Check pg_stat_activity before you spend a day tuning autovacuum that was never the problem.

             COMMIT returns at the WAL. The data page follows minutes later.

MEMORY       shared_buffers        ~25% RAM   shared, global (the page cache)
             work_mem              4-32 MB    PER SORT NODE, PER CONNECTION
                                              → it MULTIPLIES. This is how you OOM.
             maintenance_work_mem  raise it   faster CREATE INDEX / VACUUM
             effective_cache_size  ~75% RAM   planner HINT — allocates nothing

BACKGROUND   checkpointer · background writer · walwriter · autovacuum launcher
PG 18        async I/O (io_method = worker | io_uring) → faster seq scans,
             bitmap heap scans and vacuum. Did not exist before PG 18.

# ── SQL SERVER → POSTGRES ──────────────────────────────────────────────
clustered index    → does not exist       filtered index → PARTIAL index
non-clustered idx  → just "an index"        file group     → TABLESPACE
columnstore index  → does not exist       (use Parquet / Delta instead)

# ── THE FOUR SCANS (measured on 1M rows / 14,286 pages) ────────────────
Seq Scan          14,286 pages   no index, or you want most of the table
Index Scan        few pages      few rows, needs columns not in the index
Bitmap Heap Scan  110 … 11,988   medium match count; sorts ctids first
Index Only Scan   4 pages        index has every column AND pages all-visible

# ── EXPLAIN ────────────────────────────────────────────────────────────
EXPLAIN (ANALYZE, BUFFERS) …          ← always. Read bottom-up.
  Buffers:                  pages read = the real cost
  estimated vs actual rows: wildly off → run ANALYZE
  Rows Removed by Filter:   big → missing index
  Heap Fetches:             > 0 on an Index Only Scan → VACUUM
  cost=…                    NOT a time. Ignore it.

# ── THE THREE JOINS ────────────────────────────────────────────────────
Nested Loop  for each outer row, look up the inner
             + few outer rows + INDEX on inner    − no index → N × M = death
             loops=N → the rows shown are PER LOOP. MULTIPLY.
Hash Join    hash the small side, probe with the big one · EQUALITY ONLY
             + big joins, no index — the DE workhorse
             Batches > 1 = spilled to disk → raise work_mem
Merge Join   sort both sides, walk in lockstep · handles ranges
             + both sides ALREADY sorted          temp read/written = spilled

debug order: 1) est vs actual rows off? → ANALYZE
             2) big loops= ? → index the inner side
             3) Batches/temp ? → work_mem (per query!)

# ── INDEX TYPES ────────────────────────────────────────────────────────
btree   the default · equality + ranges + ORDER BY
brin    HUGE append-only tables, time column · 21 MB → 24 kB · needs correlation≈1
gin     jsonb, arrays, full-text
gist    geometry, ranges (PostGIS)      hash  equality only — rarely worth it

# ── INDEX VARIANTS ─────────────────────────────────────────────────────
CREATE UNIQUE INDEX …                            constraint + index in one
CREATE INDEX … WHERE status='pending'          PARTIAL — half the size
CREATE INDEX … (user_id) INCLUDE (amount)       COVERING → index-only scan
CREATE INDEX … (LOWER(email))                  EXPRESSION → makes it sargable
CREATE INDEX … (a, b)                            leftmost prefix rule
                                                  (PG18 skip scan relaxes it
                                                   IF `a` is low-cardinality)
CREATE INDEX CONCURRENTLY …                      no write lock. Use in prod.

# ── VACUUM ─────────────────────────────────────────────────────────────
VACUUM        marks space reusable · does NOT shrink the file · no lock
VACUUM FULL   rewrites compactly · returns disk · ACCESS EXCLUSIVE = OUTAGE
autovacuum    default fires at 20% dead — far too late on a big table
              ALTER TABLE t SET (autovacuum_vacuum_scale_factor = 0.01);
idle-in-transaction → blocks vacuum DATABASE-WIDE. Check pg_stat_activity.

# ── PARTITIONING ───────────────────────────────────────────────────────
PARTITION BY RANGE (created_at)   time-series · prunes ranges AND equality
PARTITION BY LIST  (region)       prunes equality / IN
PARTITION BY HASH  (user_id)      spreads writes · prunes EQUALITY ONLY
no partition key in WHERE  →  NO PRUNING · every partition scanned
index on the parent        →  cascades to every partition (local indexes)
DROP TABLE old_partition   →  instant. This is the real reason to partition.
unique key                 →  MUST include the partition key

# ── MONITORING · the six ───────────────────────────────────────────────
what's slow      pg_stat_statements     ORDER BY total_exec_time DESC
missing index    pg_stat_user_tables    high seq_scan on a big table
unused index     pg_stat_user_indexes   idx_scan = 0
bloat            pg_stat_user_tables    n_dead_tup / n_live_tup > 20%
cache hit        pg_statio_user_tables  want > 99%
blocking         pg_stat_activity       pg_blocking_pids(pid)

Glossary

Backend process
The OS process the postmaster forks for your connection. Postgres is process-based, not thread-based, so every connection costs several megabytes and a scheduler slot — which is why a connection pooler exists.
Connection pooler
PgBouncer and friends: keeps a small pool of real backends and multiplexes many client connections across it. Necessary because throughput collapses past a few hundred backends. Useful active backends ≈ cores × 2–4.
WAL
Write-Ahead Log. A sequential, append-only journal of every change. Postgres writes here first and returns — the data pages are written later. Durability, crash recovery and replication all come from this one file stream.
Checkpoint
The moment the checkpointer process flushes accumulated dirty pages from shared_buffers out to the data files, so the WAL before that point is no longer needed for recovery.
shared_buffers
Postgres's own cache of 8 KB pages, shared by every backend. Set to roughly 25% of RAM. A buffer hit here is the shared hit= figure in EXPLAIN (BUFFERS).
work_mem
Memory allowed for one sort or hash operation, per connection. It is not a total — one query may have several such nodes and you may have hundreds of connections, so it multiplies. The classic way to OOM a server.
Asynchronous I/O
New in PostgreSQL 18. Backends queue read requests and keep working instead of blocking on each page fetch (io_method = worker by default, or io_uring on modern Linux). Speeds up sequential scans, bitmap heap scans and vacuum.
Page (block)
8192 bytes. The unit of I/O — Postgres never reads less. A table's size is just its page count. Set at compile time via block_size.
Heap
The unordered pile of pages that is a Postgres table. Rows go wherever there is free space. Every Postgres table is a heap; there is no alternative.
Tuple
One version of a row. Because of MVCC, a single logical row may have several tuples at once, only one of which is visible to you.
ctid
A tuple's physical address: (page, slot). It is what index leaves store, and it changes when the row is updated — so never store it as a key.
Line pointer
A 4-byte slot at the start of a page pointing at a tuple further down it. The indirection is what lets Postgres move a tuple within a page without touching any index.
MVCC
Multi-Version Concurrency Control. An UPDATE writes a new tuple and marks the old one dead rather than overwriting. Readers never block writers; the cost is garbage.
xmin / xmax
Hidden columns on every tuple: the transaction that created it and the transaction that deleted it. Together they decide whether you can see this version.
Dead tuple
A row version no transaction can see any more, still physically occupying space. Cleaned up by VACUUM.
Bloat
Space consumed by dead tuples. Above ~20% it means autovacuum is not keeping up, and every scan is reading garbage.
VACUUM
Marks dead-tuple space reusable and updates the visibility map. Does not return disk to the OS — only VACUUM FULL does that, and it locks the table completely.
Visibility map
One bit per heap page: "every tuple here is visible to everyone." Set by VACUUM. Index-only scans are impossible without it — this is why a bloated table gets no index-only scans.
HOT update
Heap-Only Tuple: an update that fits on the same page and touches no indexed column, so no index is modified at all. The reason to leave fillfactor headroom and to not over-index hot columns.
fillfactor
The percentage of each page filled on insert. Below 100 leaves room for future HOT updates to land locally.
TOAST
The Oversized-Attribute Storage Technique. For values that make a row exceed ~2 KB: compress first, then move out-of-line into a hidden side table. Excluded from pg_relation_size().
B-tree
The default index: a sorted, balanced tree of pages. Every leaf is the same depth, so every lookup costs the same few page reads. Leaves hold key → ctid.
BRIN
Block Range INdex. Stores only min/max per 128-page range, so it is tiny (24 kB where a B-tree was 21 MB). Brilliant on append-only time-ordered data; worthless if the column doesn't correlate with physical order.
Correlation
A pg_stats figure from −1 to 1: how closely a column's logical order matches its physical order on disk. Near 1 → BRIN and range scans excel. Near 0 → only a B-tree helps.
Partial index
CREATE INDEX … WHERE … — indexes only a subset of rows. SQL Server calls this a filtered index. Only usable when the query's WHERE provably implies the index's.
Covering index
An index carrying extra columns via INCLUDE, so a query can be answered from the index alone. The columns ride in the leaf pages only.
Index-only scan
Answering a query from the index without touching the heap. Requires the index to hold every needed column and the visibility map to be current. Check Heap Fetches: 0.
Nested Loop
For each outer row, look up matches on the inner side. The best join when there are few outer rows and the inner side is indexed; a catastrophe (N × M) when it isn't.
Hash Join
Build a hash table from the smaller input, then scan the larger one and probe it. Reads both tables once, so it scales well to big joins with no index — the workhorse of analytical queries. Equality conditions only. Batches > 1 means it spilled to disk.
Merge Join
Sort both inputs by the join key and walk them in lockstep. Cheap only if the inputs are already sorted; otherwise the sort dominates and may spill (temp read/written).
loops=
How many times a plan node ran. The actual rows beside it is the average per loop, so the true total is rows × loops. The most misread number in EXPLAIN.
Bitmap heap scan
Collects all matching ctids, sorts them by page, then reads each heap page once — turning random I/O into sequential. Chosen when there are too many matches for a plain index scan.
Skip scan
New in PostgreSQL 18. Lets a multicolumn index be used even when the query omits the leading column, by hopping through each distinct prefix. Only viable when that leading column has few distinct values.
Tablespace
A named location on disk you can place tables, indexes and partitions into. Postgres's equivalent of SQL Server's file group. How you put hot data on NVMe and cold data on cheap disk.
Declarative partitioning
Splitting a table into child tables by RANGE, LIST or HASH. The win is pruning (skipping partitions entirely) and instant DROP of old data — not smaller tables per se.
Partition pruning
The planner discarding whole partitions it can prove cannot match. Requires the partition key in the WHERE clause. Without it, partitioning makes things slower, not faster.
Sargable
A predicate an index can be used for — which requires the column to stand bare on one side. WHERE date_trunc('day', d) = … is not sargable; a half-open range is.
pg_stat_statements
The extension that records execution statistics per query. The first place to look when something is slow. Sort by total_exec_time, not mean.
