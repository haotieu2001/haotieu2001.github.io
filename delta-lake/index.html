Why every example runs on Databricks

Delta Lake is open source and runs on any Apache Spark. I use Databricks for the examples for four simple reasons:

  • Delta is the default format there — nothing to install or configure.
  • Unity Catalog (Databricks' built-in governance) gives you storage, permissions and table names that work right away.
  • New features appear here first — for example deletion vectors, liquid clustering and type widening (each explained in its own section below).
  • Managed Spark + Photon (Databricks' fast query engine) means you get a working cluster in a couple of clicks and fast queries.

When a feature only exists on Databricks (not in open-source Delta), I point it out in the text.

❯ The one idea (read this first)

Delta Lake = Parquet data files + a transaction log (a folder called _delta_log/). The log is an ordered list of "add this file / remove that file" notes. To read the table, Delta replays the log to work out which files count right now. That one idea gives you safe writes (ACID), time travel, safe writing by many jobs at once, and cheap updates and deletes. Almost every feature below comes back to it.

01. Why Delta Lake exists

A plain data lake (files in S3 / ADLS) is cheap, grows without limit, and holds any kind of data. But on its own it has three problems that make it unsafe to trust as your main copy of data:

  1. No ACID — if a job stops half-way through writing, it leaves broken, half-written files, and the next reader gets bad data.
  2. No UPDATE / MERGE / DELETE — Parquet files cannot be changed, so "change one row" means read everything, change it, and write it all back.
  3. No schema rules — the same data can arrive in different shapes. One day a row has name and email; the next it has full_name with email inside a nested contact field. These mismatched shapes pile up, so reading the data correctly becomes a guessing game.

Delta Lake fixes all three by adding a transaction layer on top of Parquet, and also adds time travel, batch + streaming in one table, and full history for audits.

02. ACID, the foundation

ACID is four promises a plain lake can't give:

  • Atomicity — a write fully happens or has no effect. No half-written tables after a crash.
  • Consistency — the table always follows the rules you set. For example, if you say price must be greater than 0, Delta rejects any write that breaks that rule, so the table never ends up in a bad state.
  • Isolation — a reader always sees one fixed version of the table (a snapshot), never your half-done write. Readers get this snapshot from the log; writers avoid clashing with each other using optimistic concurrency (§07).
  • Durability — after a change is committed, it survives crashes, because it was written to the transaction log before the system confirmed it.

Delta writes a change in two steps: first it writes the new data files, then it makes them live with a single, all-or-nothing commit — one numbered JSON file in _delta_log. If a crash happens before that commit is written, the new data files are simply never referenced, so readers still see the previous version and there is no half-finished state. That two-step pattern is how the log gives you atomicity and durability.

03. The transaction log (_delta_log)

A Delta table on disk is a folder of part-*.parquet data files plus a _delta_log/ folder. Every change adds one JSON file, named with leading zeros (00000000000000000000.json, then ...0001.json, and so on). The first commit has four parts:

JSON · _delta_log/00000000000000000000.json
{
  "commitInfo": { "operation": "CREATE TABLE", "userName": "hao@..." },   // what + who
  "metaData":   { "schemaString": "...", "partitionColumns": [] },        // schema + partitions
  "protocol":   { "minReaderVersion": 1, "minWriterVersion": 2 },         // format version
  "add": {                                                                // a data file is added
    "path": "part-00000-...snappy.parquet",
    "size": 59210,
    "stats": "{\"numRecords\":5,\"minValues\":{\"customer_id\":1},\"maxValues\":{\"customer_id\":5}}"
  }
}

The action types are add (register a data file, with its size and per-column smallest/largest values), remove (mark a file as removed — a tombstone), metaData, and protocol. Those smallest/largest values are the engine's key advantage: if a file's range can't match your filter, Delta skips the whole file without opening it. This is called data skipping (§17).

Note: when you turn on deletion vectors (§05), Delta raises the protocol version (reader 3 / writer 7). The simple 1/2 above is a basic table with no advanced features yet.
PySpark · peek at the raw log yourself
loc = spark.sql("DESCRIBE DETAIL invoices").select("location").first()[0]
display(spark.read.json(loc + "/_delta_log/00000000000000000000.json"))   # see commit 0's actions

04. DML & computing the current state

INSERT adds a new Parquet file and a new commit. UPDATE and DELETE are smarter — with deletion vectors on (commonly enabled on Databricks; explained next in §05), they avoid rewriting whole files:

SQL
INSERT INTO invoices VALUES (6,'INV006',4,12.0,'tech',DATE'2026-01-04');  -- adds 1 file
UPDATE invoices SET quantity = 10 WHERE customer_id = 1;                  -- writes a deletion vector + 1 tiny file
DELETE FROM invoices WHERE customer_id = 2;                               -- writes/updates a deletion vector only

What "compute the current state" means

To answer SELECT *, Delta works out the file list from the log: it adds and removes files from a list — it does not add up numbers. Start with an empty list, go through the commits in order, add a file on add, and drop it on remove. The files left over are the live files. A file that was added and later removed disappears. Then Delta reads those live files and hides any rows marked by a deletion vector. No big rewrite happens for an update or delete — it's just notes in the log.

05. Copy-on-Write vs Merge-on-Read (deletion vectors)

Parquet files can't be changed, so there are two ways to "change" one:

 Copy-on-Write (DV off)Merge-on-Read (DV on)
What a delete doesRewrites the whole file without those rowsWrites a tiny deletion-vector map; the file is not touched
Write costHigh (rewrite many MB)Low (write a few KB)
Read costLowestA little higher (hide DV rows while reading)
Best forMany reads, few writesTables updated often / change data
SQL
-- Merge-on-Read (default on recent Databricks Runtime)
CREATE OR REPLACE TABLE invoices_mor
  TBLPROPERTIES ('delta.enableDeletionVectors' = 'true')
AS SELECT * FROM invoices;

DELETE FROM invoices_mor WHERE customer_id = 5;
DESCRIBE HISTORY invoices_mor;   -- metrics: numRemovedFiles=0, a deletion vector added, numDeletedRows=1 (no rewrite!)

How a read uses the deletion vector

When you later run SELECT *, Delta opens the data file as normal, but the deletion vector is a small map of the deleted row positions — a row position is just the row's number inside that file (row 0, 1, 2, …); say rows 5 and 88. The engine drops those rows before giving you results — so the deleted rows never reach you, even though they are still physically inside the file.

An UPDATE under Merge-on-Read is really a delete + insert: the old row's position is marked in the deletion vector, and the new value is written to a small new file. So a reader sees the new row from the small file, and the deletion vector hides the old one — you see the row once, never twice. The big file is never rewritten. Over time these maps and tiny files build up; OPTIMIZE + VACUUM (§16, §19) clean them up.

Why Databricks? Databricks auto-enables deletion vectors on most new tables (created through Databricks SQL or a recent runtime) — not on every existing Delta table. In open-source Delta you turn them on with the table property shown above. Note: enabling them is a one-way upgrade of the table's protocol version.

06. Checkpoints & log compaction (keeping the log fast)

Millions of changes = millions of tiny JSON files, and reading them all would be slow. Delta adds two helpers:

  • Compacted JSON — every few commits are joined into one file (it just copies the notes together; it does not apply them). It still lists every change — just in fewer files — which saves the cost of opening many small files and makes the next checkpoint cheaper to build.
  • Checkpoint Parquet — every so often, Delta saves a summary of the live state so far into one *.checkpoint.parquet. To read the table, it loads the latest checkpoint, then applies only the newer JSON commits — it does not replay the whole history.
PySpark
for _ in range(50):                       # 50 small inserts to build up the log
    spark.sql("""INSERT INTO invoices (customer_id,invoice_no,quantity,price,category,invoice_date)
               VALUES (99,'X',1,1.0,'tech',current_date())""")

# list the log folder to see compacted JSON + a *.checkpoint.parquet
loc = spark.sql("DESCRIBE DETAIL invoices").select("location").first()[0]
display(dbutils.fs.ls(loc + "/_delta_log"))
Platform note: open-source Delta saves a checkpoint every 10 commits; Databricks relaxed it to every 100 commits (DBR 11.1+) for faster log listing. You almost never need to change this — just know that reads use checkpoint + newer JSON, not the whole history.

07. Optimistic concurrency (safe writes by many jobs)

Many jobs may try to write to the same table at the same time. Delta keeps them safe without locking the table. "Optimistic" means each job assumes there will be no clash, does its work, and only checks at the very end.

Here is the idea with two jobs. Say the table is at version 4:

  1. Job A and Job B both start, and both read version 4.
  2. Each one writes its new data files (not visible yet), then tries to create the next commit file: ...0005.json.
  3. The storage rule is "create this file only if that name does not exist yet" — so only one job can succeed. Say Job A wins; the table is now version 5.
  4. Job B's attempt to create ...0005.json fails, because the file already exists. Now Job B knows it worked on old data, so it reads the new version 5 and tries again as ...0006.json.

That single "create the file only if it doesn't exist" step is what picks the winner — no separate lock is needed.

Who does the retry? For most operations Delta retries by itself. If the two jobs really changed the same data and it can't be combined, the losing job stops with a ConcurrentAppendException, and you re-run it.

Readers are never blocked — a reader just sees the latest finished version. And two jobs writing to different files both succeed at once; only two writing to the same files make one retry. This is why Delta works very well when there are many reads and few clashes.

08. Time travel & RESTORE

Because every version is kept, you can read or return to any earlier point:

SQL
SELECT * FROM invoices VERSION AS OF 0;
SELECT * FROM invoices TIMESTAMP AS OF '2026-06-01 09:35:00';
-- no exact match? Delta returns the latest version that existed BEFORE that time.

RESTORE TABLE invoices TO VERSION AS OF 0;   -- one-line undo; recorded as a new RESTORE version
PySpark
df = spark.read.option("versionAsOf", 1).table("demo.delta_db.invoices")
# or .option("timestampAsOf", "2026-06-01")

Great for audits ("what did this report show last quarter?"), repeating old results, and fast recovery from a bad load. One warning: VACUUM (§19) can delete the files an old version needs, which limits how far back you can go.

09. MERGE — the command you'll use most

MERGE applies a batch of changes in one step: update matching rows, insert new ones, and (optionally) delete rows that are gone. It's the most-used command in production, so it gets its own section.

SQL · the full shape
MERGE INTO invoices AS t                              -- target (the table)
USING updates       AS s                              -- source (the new data)
ON t.customer_id = s.customer_id                      -- how to match rows
WHEN MATCHED AND s.quantity = 0 THEN DELETE           -- match + condition → delete
WHEN MATCHED                    THEN UPDATE SET *      -- match → update all columns
WHEN NOT MATCHED                THEN INSERT *          -- new key → insert
WHEN NOT MATCHED BY SOURCE      THEN DELETE;           -- in table but not in source → delete (full sync)

First make a small updates source so you can run it:

SQL
CREATE OR REPLACE TEMP VIEW updates AS SELECT * FROM VALUES
  (1,'INV001',9,10.0,'clothing',DATE'2026-02-01'),   -- existing key → will UPDATE
  (7,'INV007',2,30.0,'tech',    DATE'2026-02-01')    -- new key → will INSERT
AS s(customer_id,invoice_no,quantity,price,category,invoice_date);

MERGE INTO invoices AS t USING updates AS s
ON t.customer_id = s.customer_id
WHEN MATCHED     THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *;
The big lesson: MERGE matches columns by name, while INSERT … SELECT matches by position. With INSERT, if the source columns are in a different order, you can silently write quantity into customer_id — wrong data, and no error. So for production loads where the source might change, prefer MERGE. (More on this in §10.) Performance tip: make the ON condition narrow (include the date/partition or a key range) so MERGE doesn't scan the whole table — and remember that with deletion vectors off, MERGE rewrites every file it touches. Also: if two source rows match the same target row, MERGE fails ("multiple source rows matched") — remove duplicates from the source first (keep the latest row per key).

SCD Type 2 (Slowly Changing Dimension — a table that keeps the history of changes) is also built on MERGE: close the old row (set an end date / is_current = false) and insert a new current row. MERGE's conditions make that one statement.

10. Schema enforcement (Delta checks before it writes)

Delta is "schema-on-write": it checks data against the table's columns before writing. It checks in steps:

  1. Safe type conversion first — a number written as text, '99499' → INT, works; 'ABC' → INT fails.
  2. Then NULL and CHECK rules — a NULL in a NOT NULL column, or a value that breaks a CHECK, is rejected.
  3. Then the structure — the set of columns must match (an extra column in an INSERT fails).

Any failure rejects the whole write and leaves the table unchanged (atomicity again). Remember the key rule from §09: INSERT matches by position, MERGE matches by name. Use MERGE when the source order or column set might change.

11. Schema evolution (changing the shape, no rewrite)

Add columns, widen types, or change nested fields without rewriting the data.

SQL
ALTER TABLE invoices ADD COLUMNS (payment_method STRING);   -- add a column by hand

SET spark.databricks.delta.schema.autoMerge.enabled = true; -- session setting (this query/session)
-- now an INSERT/MERGE with a new column auto-adds it; old rows show NULL
SQL · type widening (Delta 3.2+ / DBR 15.4)
ALTER TABLE invoices SET TBLPROPERTIES ('delta.enableTypeWidening' = 'true');
ALTER TABLE invoices ALTER COLUMN customer_id TYPE BIGINT;   -- INT → BIGINT, no data rewrite
PySpark
from pyspark.sql.functions import lit
df2 = df.withColumn("note", lit("hello"))   # df (the invoices table) + one new column

(df2.write.mode("append").option("mergeSchema","true")
    .saveAsTable("demo.delta_db.invoices"))   # the new "note" column is added; old rows show NULL
Why Databricks? Type widening is a preview in Delta 3.2 (included in DBR 15.4 LTS; older 14.3 has Delta 3.1 and can't widen) and fully supported in Delta 4.0. On Databricks you just pick the runtime from a dropdown; on your own Spark you manage the Delta library (jar) versions yourself.

12. Renaming & dropping columns (column mapping)

A common beginner mistake: you try to rename or drop a column and get an error. That's because, by default, the data files use the column names directly. Turn on column mapping first, then it works without rewriting data:

SQL
ALTER TABLE invoices SET TBLPROPERTIES (
  'delta.columnMapping.mode' = 'name',
  'delta.minReaderVersion' = '2', 'delta.minWriterVersion' = '5'
);
ALTER TABLE invoices RENAME COLUMN invoice_no TO invoice_id;
ALTER TABLE invoices DROP COLUMN payment_method;

Column mapping lets Delta track columns by a hidden id instead of by name, so renames and drops are just metadata changes. (After a DROP, the old data stays in the files until you run REORG TABLE invoices APPLY (PURGE) and then VACUUM.) Turning column mapping on is a one-way change — you can't easily turn it off.

13. Constraints, generated & identity columns

SQL
-- CHECK rule: every future write must satisfy it, or it's rejected
ALTER TABLE invoices ADD CONSTRAINT pos_price CHECK (price > 0);

-- Generated column: Delta computes it for you (good for a date partition key)
CREATE TABLE events (
  event_ts   TIMESTAMP,
  event_date DATE GENERATED ALWAYS AS (CAST(event_ts AS DATE))
);

-- Identity column: an automatic ID (a "surrogate key" — a system-created id, not a real value from the data)
CREATE TABLE dim_customer (
  id   BIGINT GENERATED ALWAYS AS IDENTITY,
  name STRING
);
SQL · see a generated value appear
INSERT INTO events (event_ts) VALUES (TIMESTAMP'2026-01-02 09:00:00');
SELECT * FROM events;   -- event_date is filled in automatically as 2026-01-02

Why care: NOT NULL and CHECK rules reject invalid data at write time, so bad rows never enter the table. A generated event_date means nobody can forget to fill the partition key (which would mess up your file layout). An identity column gives you clean IDs without a separate counter object.

14. Idempotent writes (safe to run again)

Idempotent means re-running the same load does not create duplicates — important when a job retries after a failure. Two common ways:

  • MERGE on a key — because MERGE updates the matching row instead of inserting a new one, re-running a backfill is naturally safe.
  • Transaction tags for appends — give a write an app id + a version, and Delta skips it if that version was already written:
PySpark
(df.write.format("delta").mode("append")
   .option("txnAppId", "daily_invoices_load")
   .option("txnVersion", 20260601)           # re-running with the same version is ignored
   .saveAsTable("demo.delta_db.invoices"))

15. Partitioning — and when not to

Partitioning splits a table into folders by a column (often a date), so a query that filters on that column reads only the matching folders.

PySpark
df.write.partitionBy("invoice_date").saveAsTable("demo.delta_db.invoices_by_date")

Simple tips:

  • Do partition large tables by a low-variety column you filter on often — usually a date. Aim for partitions around 1 GB.
  • Don't partition by a high-variety column (like customer_id with millions of values) — you get millions of tiny folders and files (§16).
  • Don't over-partition small tables — the folders cost more than they save.
  • Modern advice: for many tables, skip folder-based ("Hive") partitioning and use liquid clustering (§18) instead — it picks the layout for you and you can change it later.

16. The small-file problem & OPTIMIZE

Thousands of tiny files = thousands of open/close steps = wasted compute. (Imagine a 300-page book saved as 300 one-page PDFs — you'd spend all your time opening files.) OPTIMIZE packs small files into a few large ones — it fits small pieces together like packing boxes (this is called "bin-packing") — toward a target file size.

SQL
OPTIMIZE invoices;                          -- whole table
OPTIMIZE invoices WHERE invoice_date = '2026-01-03';   -- just one partition (cheap, common)

Three ways to reduce small files, often used together:

WayHow it worksUse it when
Manual OPTIMIZEYou run OPTIMIZE on demandBatch pipelines with a scheduled step after loading
Optimized WriteReshuffles each partition into one well-sized writeMany workers would each write a tiny file
Auto CompactionAuto-packs after a write passes a minimum number of small filesStreaming / 5-minute loads between OPTIMIZE runs
PySpark
(df.write.mode("overwrite").partitionBy("category")
   .option("optimizeWrite","true")            # e.g. hundreds of tiny files → a few right-sized ones
   .saveAsTable("demo.delta_db.orders"))

# or set it on the table, for streaming tables:
# TBLPROPERTIES ('delta.autoOptimize.optimizeWrite'='true','delta.autoOptimize.autoCompact'='true')
About the target size: Databricks auto-tunes the target file size by how big the table is — about 256 MB for most tables, rising toward ~1 GB only for very large (10 TB+) tables. You rarely set it by hand.
How often should you run it? A common pattern is OPTIMIZE daily and VACUUM weekly. On Databricks, Predictive Optimization can run OPTIMIZE and VACUUM for you automatically, so you don't have to schedule them.
Why Databricks? optimizeWrite and autoCompact are Databricks features (and Photon makes OPTIMIZE fast). Open-source Delta has OPTIMIZE but not these automatic behaviours, so you'd build the equivalent yourself.

17. Z-Order & data skipping

Data skipping uses each file's smallest/largest values to avoid reading files that can't match your filter. It only helps if file ranges don't overlap. Z-Order sorts and re-packs the data so similar values are stored together, which reduces overlap:

SQL
OPTIMIZE invoices ZORDER BY (customer_id);
OPTIMIZE invoices WHERE invoice_date = '2026-01-03' ZORDER BY (customer_id);

In a test, a customer_id = 201 query went from reading all ~255 files (~785 ms) to reading one file (~144 ms — about 5× faster) after ZORDER BY (customer_id). OPTIMIZE and Z-Order work together: OPTIMIZE packs small files, Z-Order groups similar data. By default Delta keeps min/max stats only for the first 32 columns, so put the columns you filter on early in the table and avoid very long text columns at the front. To confirm skipping actually happened, compare the number of files read (in the Spark UI or query metrics) before and after.

For two or more columns, there's no single sort order. Z-Order combines the values of both columns, so rows that are similar in both columns are stored near each other. A filter on either column can then still skip files. (You don't need the math — the goal is simply "less overlap, so more files can be skipped.") Z-Order is an expensive sort, so run it now and then (often nightly, limited to the latest partition).

18. Liquid clustering (the modern layout)

Folder-based partitioning ("Hive" partitioning, from an older big-data system) and Z-Order both make you decide the layout column in advance; if your filter pattern changes later, you must rewrite the data. Liquid clustering lets you change the clustering keys at any time and keeps the data balanced — similar file sizes, a good file count, and no single folder that's far too big.

PySpark / SQL
df.write.clusterBy("invoice_date","customer_id").saveAsTable("demo.delta_db.sales_lc")
-- SQL: CREATE TABLE sales_lc CLUSTER BY (invoice_date, customer_id) AS SELECT ...
-- change the keys later: ALTER TABLE sales_lc CLUSTER BY (category);

It merges small files and splits oversized ones so every Spark task gets an even amount of work — this fixes skew (uneven data, where one giant folder slows down the whole job). Pick your most-filtered 1–4 column(s). Don't combine it with Z-Order or Hive partitioning — it decides the layout itself.

Why Databricks? Liquid clustering is a newer Delta feature. On Databricks you get the table syntax and the background re-packing is run for you; on open-source Delta support is more limited and you'd schedule the optimization yourself.

19. VACUUM & retention

DELETE/UPDATE/OPTIMIZE only tombstone (mark as removed) old files; VACUUM actually deletes them from storage.

SQL
VACUUM invoices;                       -- removes tombstoned files older than the retention window (default 7 days)
-- lab only — never in production:
SET spark.databricks.delta.retentionDurationCheck.enabled = false;
VACUUM invoices RETAIN 0 HOURS;

Two things people get wrong:

  • VACUUM saves storage but does NOT make queries faster — Delta decides which files to read from the log, with or without tombstones.
  • VACUUM limits time travel — once a version's files are deleted, you can't go back to it. Set the retention window to at least the longest time any process (audit, reconciliation, re-processing) might need old data. Also run VACUUM after OPTIMIZE.

20. Converting Parquet / CSV / JSON to Delta

PySpark
# A) rewrite as Delta — gives fresh files + good sizes + stats (more work, moves data)
spark.read.parquet("/path/raw").write.format("delta").mode("overwrite").save("/path/delta")

# B) in-place — keeps the existing Parquet files, just adds a _delta_log (instant, no data moved)
from delta.tables import DeltaTable
DeltaTable.convertToDelta(spark, "parquet.`/path/raw`")
# if the Parquet is partitioned, pass the partition schema:
# DeltaTable.convertToDelta(spark, "parquet.`/path/raw`", "invoice_date DATE")

# CSV/JSON: read into a DataFrame, then write as delta
spark.read.option("header", True).csv("/path/file").write.format("delta").save("/path/delta")

Which to pick? Use B (in-place) for a quick migration with no data movement. Use A (rewrite) when you also want well-sized files and good statistics from the start.

22. Cloning: shallow & deep

Cloning copies a table into its own new table. There are two kinds, and the difference is simple: a shallow clone copies only the log (the list of files), so it shares the source's data files; a deep clone also copies the data files, so it stands completely on its own.

SQL
CREATE TABLE sales_shallow SHALLOW CLONE invoices;   -- copies only the log; shares the source's data files (instant, cheap)
CREATE TABLE sales_deep    DEEP CLONE   invoices;   -- copies the log AND the data (independent, self-contained)
CREATE TABLE sales_v0      SHALLOW CLONE invoices VERSION AS OF 0;   -- clone the table as it looked at version 0

Re-cloning the same target—Copies only what changed (incremental)
Best forQuick, one-time analysis or experimentsDisaster-recovery copies; filling dev/uat from prod

Both clones start at their own version 0. A deep clone also keeps the source's partitioning, constraints and table properties — a plain CTAS (CREATE TABLE AS SELECT) does not.

Shallow-clone warning: a shallow clone still points at the source's files. So VACUUM on the source will not delete a file the clone still uses — and if you force it, the clone breaks. Use a deep clone when the copy must be fully independent.

23. Streaming with Delta (one table, batch + streaming)

A Delta table is both a batch table and a streaming source/sink (a source is read-from, a sink is written-to) — the same table, no separate system.

The streaming examples use a bronze source and a silver target — create them like the invoices table in §00, and create a checkpoint Volume once: CREATE VOLUME demo.delta_db.chk;. (A Volume is a Unity Catalog place to store files.)
PySpark
from delta.tables import DeltaTable

stream = spark.readStream.table("demo.delta_db.bronze")     # read a Delta table as a stream

(stream.writeStream
   .option("checkpointLocation", "/Volumes/demo/delta_db/chk/silver")  # tracks progress for exactly-once (no duplicates, no misses)
   .trigger(availableNow=True)      # process all new data once, then stop (or processingTime="5 minutes" to keep running)
   .toTable("demo.delta_db.silver"))
PySpark · upsert each micro-batch with MERGE
def upsert(batch_df, _):
    (DeltaTable.forName(spark, "demo.delta_db.silver").alias("t")
       .merge(batch_df.alias("s"), "t.customer_id = s.customer_id")
       .whenMatchedUpdateAll().whenNotMatchedInsertAll().execute())

(stream.writeStream.foreachBatch(upsert)
   .option("checkpointLocation", "/Volumes/demo/delta_db/chk/up").start())

The checkpoint location stores how far the stream has read. On Unity Catalog, use a Volume path (/Volumes/...) for it.

Why Databricks? Databricks adds Auto Loader (spark.readStream.format("cloudFiles")) for cheap incremental file ingestion, and Delta Live Tables for pipelines you define by describing the result. Both build on this same Delta streaming base.
Two streaming gotchas: (1) updates or deletes on a streaming source table will stop the stream — add .option("skipChangeCommits","true") to ignore them. (2) Use maxFilesPerTrigger or maxBytesPerTrigger to limit how much each micro-batch reads.

24. Change Data Feed (CDF)

CDF makes Delta produce the exact row-level changes between versions — so later steps (silver, gold) can process only what changed instead of re-reading everything.

SQL
ALTER TABLE invoices SET TBLPROPERTIES ('delta.enableChangeDataFeed' = 'true');

SELECT * FROM table_changes('invoices', 5);   -- changes since version 5
-- extra columns you get back:
--   _change_type    = insert | update_preimage (row before) | update_postimage (row after) | delete
--   _commit_version = which commit
--   _commit_timestamp

This is the clean way to do incremental ETL (Extract, Transform, Load) and to keep SCD tables up to date, instead of recomputing everything. Note: CDF only records changes made after you enable it.

PySpark · consume the feed downstream
changes = (spark.read.format("delta")
   .option("readChangeFeed", "true").option("startingVersion", 5)
   .table("demo.delta_db.invoices"))
# keep only the latest row per key, then MERGE those changes into a gold table
# inserts/updates -> upsert; _change_type = 'delete' -> delete

25. Sharing & UniForm (working with other tools)

  • Delta Sharing — an open way to share a live table with another team or company without copying it; they read your latest data through a secure connection.
  • UniForm — Delta also writes Iceberg (and Hudi) metadata next to its own log, so Iceberg/Hudi readers can read your Delta table. It means choosing Delta does not block other engines.
SQL
-- UniForm: also expose this table as Iceberg
ALTER TABLE invoices SET TBLPROPERTIES ('delta.universalFormat.enabledFormats' = 'iceberg');

-- Delta Sharing: publish a share, then add a table to it (Unity Catalog)
CREATE SHARE sales_share;
ALTER SHARE sales_share ADD TABLE demo.delta_db.invoices;
Important limit: UniForm works in one direction only — other engines should only read. It can also become unreliable with the newest features such as deletion vectors. So it's a good safety option, not a perfect automatic solution.

26. What's new in Delta 4.0 (2025)

  • Variant type (preview) — store free-form, JSON-like data efficiently, without first splitting it into columns.
  • Type widening — safe column type changes (int → bigint) with no data rewrite and a history of the change (§11).
  • Coordinated commits (preview) — a "commit coordinator" (such as Unity Catalog) lets several engines/clouds write to one table safely, and prepares for changes across multiple tables at once.
  • Delta Kernel & Delta Connect — a small, portable library so non-Spark tools can read Delta correctly, plus a client/server way to use Delta.

Common errors & quick fixes

Error you might seeWhat it means / fix
ConcurrentAppendException (or similar)Two jobs changed the same files. Delta already retries; if it keeps failing, write to different partitions, or serialize the jobs.
"schema mismatch detected"Your write has different columns than the table. Add the column first, or use MERGE / mergeSchema (§10–§11).
FileNotFound on time travelVACUUM deleted that old version's files (§19). Increase the retention window if you need to go back further.
"table or view not found"You're in a new session — re-run USE demo.delta_db; or use the full name demo.delta_db.invoices.
Can't RENAME/DROP a columnTurn on column mapping first (§12).

27. Cheat sheet — when to use what

QuestionAnswer
Update a copy from a source?MERGE on a key (safe by name, idempotent)
Table updated often / change data?Deletion vectors (Merge-on-Read) — auto-enabled on new Databricks tables
Read-heavy, rarely updated?Copy-on-Write is fine (deletion vectors off)
Big table, filtered by date?Partition by date (≈1 GB partitions) or liquid clustering
Fast filtering on another key?Z-Order that key (within the partition) — or use liquid clustering
Filter pattern keeps changing?Liquid clustering (change keys anytime)
Too many small files?OPTIMIZE (+ optimizeWrite / autoCompact for streaming)
Free disk / not keeping old versions?VACUUM (but it limits time travel — set retention first)
Must keep data if table dropped?External table (with LOCATION)
Continuous data?Streaming (readStream/writeStream) + foreachBatch MERGE
Send only what changed downstream?Change Data Feed
Recover a bad load / audit?Time travel + RESTORE

Glossary

Every term used in this article, defined in one place. Skim it first if you're new, or jump back here whenever a word feels rushed.

Parquet
The common file type for analytics data. Delta stores its data as Parquet files.
Transaction log
The ordered list of changes in _delta_log/. Sometimes called a "ledger" (a record book).
Commit
One change written to the log as one numbered file (e.g. ...0001.json). Also called a "version".
ACID
Four safety promises: a write fully happens or not at all (atomic), rules are kept (consistent), readers don't see half-done work (isolated), and committed data survives crashes (durable).
Tombstone
A note that marks a file as "no longer used". The file stays on disk until VACUUM deletes it.
Deletion vector (DV)
A tiny file that marks which rows inside a data file are deleted, so Delta doesn't have to rewrite the whole file. It's a "bitmap" (a compact map of row positions).
Checkpoint
A saved summary of the log so far, so reading doesn't replay every old commit.
MERGE / upsert
One command that updates a row if it exists and inserts it if it doesn't. "Upsert" = update-or-insert.
OPTIMIZE / compaction
Packing many small files into a few right-sized ones.
Partition
Splitting a table into folders by a column value (often a date) so queries can skip folders.
Idempotent
Safe to run again. Re-running the same load does not create duplicates.
Streaming / micro-batch
Processing data continuously in small chunks ("micro-batches") instead of one big run.
Source / sink
A "source" is where data is read from; a "sink" is where it is written to (the destination).
Medallion (bronze→silver→gold)
A common pattern of table layers: raw → cleaned → business-ready.
Metastore / Unity Catalog
The service that stores table names, locations and permissions. Unity Catalog is Databricks' version.
Protocol version
The lowest Delta version a tool needs to read or write the table. Turning on a new feature can raise it — a one-way change.
Volume (Unity Catalog)
A place in Unity Catalog to keep files (for example, streaming checkpoints). You use it with a /Volumes/... path.
SCD
Slowly Changing Dimension — a table that keeps the history of changes.
ETL
Extract, Transform, Load — moving data and reshaping it into clean tables.

Putting it together. In real work you'll use a few of these very often: MERGE for safe, repeatable loads; deletion vectors for cheap updates; partitioning or liquid clustering plus a scheduled OPTIMIZE for fast reads; VACUUM for cleanup; time travel for audits and recovery; and Change Data Feed to pass only what changed to the next layer. Learn the transaction log once, and the rest follows from it.
